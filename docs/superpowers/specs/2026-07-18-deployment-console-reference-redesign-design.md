# 배포 콘솔 시안 기반 재설계

## 목표

첨부 시안의 파란 강조색, 네이비 실행 패널, 요약 중심 정보 구조를 SketchCatch 배포 모달의 `배포` 화면과 `배포 이력` 화면에 적용한다. 사용자는 현재 단계, 실행 중인 작업, 실제 변경량, 다음 행동을 한눈에 확인할 수 있어야 한다.

이 변경은 배포 동작을 새로 만들지 않는다. 기존 Terraform Plan, 승인, Apply, Destroy, Rollback, 로그, Output 조회와 안전 게이트를 그대로 사용하고 표시 구조와 상호작용만 재구성한다.

## 범위

### 포함

- 배포 모달의 `배포` 화면 전체
- 같은 화면 아래의 `배포 이력` 영역
- 검증, 승인, 배포 단계별 상태 표현
- 진행 중 작업과 진행률 표현
- 배포 설정과 검증 요약의 정보 위계
- 배포 이력 KPI, 필터, 테이블, 선택 상세
- 데스크톱, 태블릿, 모바일 반응형 처리
- 키보드 탐색, 초점 표시, 상태 텍스트와 접근성 속성

### 제외

- Terraform Plan, 승인, Apply, Destroy, Rollback API 변경
- Deployment, ApplicationRelease, BuildEnvironment 공유 타입 또는 DB 변경
- 배포 상태 판정 규칙 변경
- 새로운 배포 지표의 저장 또는 집계 API 추가
- CI/CD 화면 재설계
- 실제 AWS, Terraform 또는 Git 실행

## 설계 원칙

1. 시안과 유사한 파란색 강조 체계와 네이비 실행 패널을 사용한다.
2. 시안의 숫자를 복제하지 않고 현재 클라이언트에 존재하는 실제 데이터만 표시한다.
3. 사용자가 지금 해야 할 주요 행동은 단계마다 하나만 가장 강하게 강조한다.
4. 실행 취소, 승인 취소, Destroy처럼 위험하거나 보조적인 작업은 주요 CTA와 시각적으로 분리한다.
5. 색상만으로 상태를 구분하지 않고 아이콘과 상태 문구를 함께 제공한다.
6. 기존 모달의 focus trap, Escape 닫기, 확인 대화상자 우선 닫기 동작을 유지한다.

## 화면 구조

### 1. 배포 화면 헤더

배포 화면 상단에는 방패 아이콘, 단계에 맞는 제목, 설명을 표시한다. 검증 단계의 기본 제목은 `배포 검증`이며, 승인된 Plan 이후 Board가 바뀐 경우 `변경사항 재검증`으로 바뀐다.

헤더 아래에는 다음 다섯 값을 한 줄 KPI 영역으로 표시한다.

| 항목 | 실제 데이터 | 값이 없을 때 |
| --- | --- | --- |
| 현재 상태 | Direct Deployment flow와 현재 Deployment 상태 | `검증 필요` |
| Terraform Plan | 진행 중이면 계산된 진행률, 완료되면 Plan 상태 | `실행 전` |
| 변경 적용 범위 | `selectedDeployment.scope` 또는 선택한 scope | 선택값 표시 |
| 예상 변경 수 | `planSummary`의 생성·변경·삭제 합계 | `계산 전` |
| 빌드 환경 | Repository 검증 및 BuildEnvironment 상태 | 대상이 아니면 `해당 없음` |

KPI 영역은 정보를 요약할 뿐 클릭 가능한 제어가 아니다. 작은 화면에서는 두 열 또는 한 열로 줄바꿈한다.

### 2. 단계 표시

`검증 → 승인 → 배포`의 기존 세 단계와 상태 계산을 유지한다. 각 단계는 번호 또는 완료 아이콘, 단계명, 짧은 상태 문구를 가진다.

- 현재 단계: 파란색 원과 연결선
- 완료 단계: 초록색 완료 아이콘
- 대기 단계: 중립 회색
- 오류 또는 차단: 빨간색 아이콘과 문구
- 접근 가능한 단계만 버튼으로 이동할 수 있으며 대기 단계는 비활성화한다.

### 3. 진행 패널

Plan, Apply, Destroy 또는 Rollback 작업이 시작 중이거나 실행 중이면 단계 표시 아래에 네이비 진행 패널을 노출한다.

- 현재 작업명과 구체적인 설명
- `DeploymentProgressBar`가 계산하는 실제 진행률
- 최근 로그를 기반으로 한 현재 세부 단계
- 가능한 경우 `실행 취소` 보조 버튼

진행 패널은 별도의 가짜 애니메이션이나 임의 진행률을 만들지 않는다. 진행 정보가 없으면 패널을 숨기고 일반 상태 요약을 표시한다.

### 4. 배포 설정

설정 영역은 실행 타깃 결정 방식과 현재 Plan 정보를 나란히 보여준다.

- 실행 타깃 결정 방식: 기존 `SelectMenu`와 설명을 유지한다.
- 현재 범위: 자동 감지 결과 또는 사용자가 선택한 범위를 표시한다.
- Terraform Plan: 실행 전, 생성 중, 승인 대기, 승인 완료 상태를 표시한다.
- 세부 Plan 정보는 기존 `PlanSummaryRows` 또는 접힌 상세 영역으로 확인한다.

설정 변경은 기존과 같이 검증을 다시 요구하며 자동으로 Plan이나 Apply를 실행하지 않는다.

### 5. 검증 요약

검증 요약은 다음 네 영역으로 구성한다.

1. 설정 상태: 저장된 배포 기준선과 현재 Board의 차이
2. 빌드 환경: Repository checkout 검증과 BuildEnvironment 상태
3. 변경 내용: 생성·변경·삭제 합계와 세부 분포
4. 실행 준비: 현재 단계에서 다음 단계로 이동 가능한지 여부

Repository 권한 실패, 배포 타깃 누락, Pre-Deployment Check 실패는 각 요약 영역 안에 숨기지 않고 바로 아래에 기존 복구 링크와 메시지를 유지한다. High finding은 검토 정보로 표시하되 기존 제품 정책대로 severity만으로 Plan 승인을 새로 차단하지 않는다.

### 6. 단계별 CTA

- 검증: `저장 후 검증 실행` 또는 `새 변경사항 검증`
- 승인: `Plan 승인`
- 배포: `배포 실행`

주요 CTA는 오른쪽 하단에 파란색으로 표시한다. 승인 취소, 실행 취소, Destroy Plan, Rollback Plan 등은 흰색 또는 위험색 보조 버튼으로 분리한다. 비활성화 상태에는 사용자가 해결해야 할 선행 조건을 텍스트로 제공한다.

## 배포 이력 화면

### 1. 이력 요약 KPI

이력 헤더 아래에 다음 네 값을 표시한다.

| 항목 | 계산 기준 |
| --- | --- |
| 전체 배포 | 화면에 로드된 성공 또는 정리 완료 이력 수 |
| 완료 | `SUCCESS`와 `DESTROYED` 이력 수 |
| 전체 변경 수 | 이력의 `planSummary` 생성·변경·삭제 합계 |
| 평균 실행 시간 | 시작·완료 시각을 신뢰할 수 있는 이력만 평균 |

현재 `Deployment` 데이터로 평균 시간을 신뢰성 있게 계산할 수 없으면 해당 KPI를 `집계 전`으로 표시한다. 값이 없는 상태를 임의의 `00:00:00`으로 만들지 않는다.

### 2. 필터

다음 필터를 제공한다.

- 전체
- 완료
- 변경 없음

필터는 클라이언트에 이미 로드된 이력만 대상으로 하며 서버 상태를 변경하지 않는다. 선택된 필터는 텍스트, 테두리, `aria-pressed`로 함께 표현한다.

### 3. 이력 테이블

테이블 열은 `상태`, `실행 시각`, `실제 변경`, `실행 범위`, `상세`로 구성한다.

- 성공한 Destroy는 `정리 완료`로 표시하며 삭제 수를 오류색으로 표현하지 않는다.
- 선택된 행은 파란 테두리와 옅은 파란 배경으로 강조한다.
- 상세 버튼은 최소 44px 클릭 영역을 가진다.
- 정렬 기능이 실제로 제공되지 않으면 정렬 아이콘을 표시하지 않는다.
- 이력이 없으면 빈 테이블 대신 다음 배포를 안내하는 빈 상태를 표시한다.

### 4. 선택 상세 패널

선택된 이력의 오른쪽 패널은 네이비 요약 헤더와 흰색 상세 정보 영역으로 나눈다.

네이비 헤더에는 상태, 실행 시각, `배포 완료된 버전` 또는 `정리 완료된 버전`, 실제 변경 결과 문장을 표시한다. 흰색 영역에는 다음 정보를 표시한다.

- 실행 범위
- 변경 내용
- 앱 릴리즈와 상태
- 버전 ID
- 실행 시간: 신뢰할 수 있는 시각 값이 있을 때만 표시
- 요청자: 실제 데이터가 있을 때만 표시
- 빌드 환경 또는 Repository 검증 상태

Deployment ID, Terraform artifact, Commit, digest, Build artifact, provider revision은 기본 화면에서 접힌 `기술 정보`에 둔다. 서비스 URL이 있으면 `배포된 서비스 열기`를 제공한다.

### 5. 리소스, Output, 로그

기존 상세 조회 상태와 API를 재사용한다. `리소스와 Output`, `전체 로그`는 선택 상세 아래의 접힌 영역에 배치해 기본 화면의 정보 밀도를 낮춘다. 로딩, 실패, 빈 결과를 각각 다른 문구로 표시한다.

## 컴포넌트 경계

`DirectDeploymentScreen`은 데이터 조회, 선택 상태, API 호출과 안전 게이트를 계속 소유한다. 표시 책임은 다음 단위로 분리한다.

- 배포 헤더와 KPI 요약
- 단계 표시
- 실행 진행 패널
- 배포 설정
- 검증 요약
- 단계별 CTA
- 이력 KPI와 필터
- 이력 테이블
- 선택 상세 패널

작은 표시 컴포넌트는 같은 기능 폴더에 두고 필요한 값과 이벤트만 props로 받는다. API 호출을 새 표시 컴포넌트 안으로 옮기지 않는다.

## 데이터 흐름

1. `DirectDeploymentScreen`이 기존 API로 AWS 연결, BuildEnvironment, Deployment, ApplicationRelease, 로그, 리소스와 Output을 불러온다.
2. 기존 selector와 상태 계산 함수가 현재 단계, 가능한 동작, 진행 상태를 결정한다.
3. 요약 컴포넌트가 실제 상태를 사람이 읽기 쉬운 라벨과 숫자로 변환한다.
4. 사용자가 기존 액션을 누르면 현재 handler가 Plan, 승인, Apply, Destroy 또는 Rollback 요청을 실행한다.
5. 갱신된 Deployment 데이터가 헤더, 단계, 진행 패널과 이력 화면에 동시에 반영된다.

## 오류와 경계 상태

- AWS 연결이 없으면 검증 CTA를 비활성화하고 기존 연결 안내를 유지한다.
- Repository 권한 실패 시 대상 Repository, 요청 commit, 실제 checkout 결과와 복구 링크를 표시한다.
- Plan 데이터가 없으면 변경 수를 0으로 단정하지 않고 `계산 전`으로 표시한다.
- 진행률을 계산할 수 없으면 indeterminate 상태 문구를 사용한다.
- 상세 조회 실패는 선택 행을 해제하지 않고 상세 패널에서 재시도 가능한 오류로 표시한다.
- 새 이력이 도착해도 사용자가 과거 이력을 보고 있으면 현재 선택을 보존한다.
- 긴 ID와 hash는 한 줄 말줄임과 전체 값 `title`을 함께 제공한다.

## 반응형

- 1200px 이상: 배포 설정·검증 요약을 넓게 사용하고, 이력 테이블과 상세 패널을 나란히 표시한다.
- 768~1199px: 배포 본문은 한 열, 이력 상세는 테이블 아래로 이동한다.
- 767px 이하: KPI와 검증 요약은 한 열 또는 두 열, 단계 설명은 축약하며 모든 CTA는 전체 너비로 표시한다.
- 이력 테이블은 최소 열 너비를 유지한 채 수평 스크롤을 허용한다.
- 모바일 전체 화면 모달, 닫기 버튼, 본문 스크롤과 focus trap 동작을 유지한다.

## 접근성

- 본문 텍스트 대비는 4.5:1 이상, 큰 텍스트와 아이콘은 3:1 이상을 목표로 한다.
- 모든 아이콘 버튼에 이름을 제공한다.
- 현재 단계는 `aria-current="step"`, 선택 이력은 `aria-current` 또는 `aria-pressed`로 전달한다.
- 진행률은 `progress` 또는 `aria-valuenow`가 있는 상태 요소로 제공한다.
- 상태 변화와 오류는 기존 `role="status"`, `role="alert"` 경계를 유지한다.
- 모든 클릭 대상은 최소 44px을 확보하고 명확한 focus-visible 스타일을 제공한다.
- 모션을 추가할 경우 `prefers-reduced-motion`에서 전환을 제거한다.

## 검증

1. 표시 계약 테스트로 헤더 KPI, 단계, 진행 패널, 검증 요약, 이력 필터·테이블·상세 구조를 확인한다.
2. 기존 배포 액션 순서, 상태 전이, 취소, 승인, Destroy, Rollback 테스트가 계속 통과해야 한다.
3. 긴 텍스트, 빈 이력, 로딩, 실패, 변경 없음, 성공 Apply와 성공 Destroy 상태를 확인한다.
4. 데스크톱과 모바일에서 모달 스크롤, 고정 닫기 버튼, 표 수평 스크롤과 CTA 배치를 시각 확인한다.
5. `pnpm harness:check`, `pnpm lint`, `pnpm typecheck`, `pnpm build`를 실행한다.
6. 실제 Terraform Plan, Apply, Destroy, AWS 또는 Git 작업은 검증 과정에서 실행하지 않는다.

## 완료 기준

- 배포 화면과 배포 이력이 첨부 시안의 정보 위계와 파란색·네이비 시각 언어를 일관되게 사용한다.
- 실제 데이터가 없는 값을 꾸며내지 않는다.
- 기존 배포 안전 게이트와 모든 실행 동작이 유지된다.
- 데스크톱과 모바일에서 핵심 정보와 주요 CTA를 쉽게 찾을 수 있다.
- 관련 테스트와 필수 저장소 검사가 통과한다.
