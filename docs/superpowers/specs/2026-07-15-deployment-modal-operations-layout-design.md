# 배포 모달 운영형 레이아웃 설계

## 목적

이 화면은 사용자가 3초 안에 현재 배포 단계, 지금 확인할 설정, 최근 실행 결과, 다음 행동을 파악하게 한다. 기존 Direct Deployment와 CI/CD API, 상태 관리, 승인 및 실행 로직은 그대로 유지하고 표시 구조와 문구만 재구성한다.

## 범위와 경계

- `DeploymentConsoleShell`의 모달과 `DirectDeploymentScreen`의 정보 구조, 반응형 레이아웃, 상태 표현, 접근성을 개선한다.
- CI/CD 데이터 로직은 변경하지 않고 공통 모달 헤더와 콘텐츠 너비 체계만 맞춘다.
- 현재 작업에서 제거된 AWS 연결 Select와 오른쪽 배포 컨텍스트 패널은 복원하지 않는다.
- 검증 요청은 기존에 자동 선택된 `selectedAwsConnectionId`를 계속 사용한다.
- API, shared type, 데이터 모델, 배포 승인 규칙, Terraform 실행 동작은 변경하지 않는다.
- 저장소가 TypeScript/TSX를 사용하므로 기존 `.tsx` 파일을 유지하고 JavaScript 파일을 새로 섞지 않는다.

## 모달과 상단 탐색

- 모달 콘텐츠는 최대 1280px 안에서 중앙 정렬하며 데스크톱 좌우 여백은 32px로 둔다.
- 상단에는 `배포`, `CI/CD`, 닫기 버튼을 한 행에 배치한다.
- 활성 탭은 하단 강조선, 약한 중립 배경, 굵기 변화로만 표시한다. 가장 강한 검정색은 실제 실행 CTA에 남긴다.
- 닫기 버튼은 우측 끝에서 최소 40×40px 클릭 영역, `aria-label`, `:hover`, `:focus-visible` 상태를 제공한다.

## Direct Deployment 단계 표시

- Stepper 높이는 72~88px, 단계 원은 28~32px로 줄인다.
- 완료 단계는 체크 아이콘과 성공 텍스트, 현재 단계는 가장 강한 테두리와 텍스트, 예정 단계는 약한 중립색으로 구분한다.
- 단계 버튼은 기존 단계 선택 동작과 `aria-current="step"`을 유지한다.
- 모바일에서는 연결선과 보조 문구가 겹치지 않도록 보조 문구를 숨기거나 축약한다.

## 본문 정보 구조

- Stepper 아래에 `N단계`, 단계별 제목, 한 문장 설명을 한 번만 표시한다.
- 1200px 이상은 `minmax(0, 1fr) 320px`, 32px 간격의 2열 구조를 사용한다.
- 1200px 미만은 상태 카드를 본문 아래로 보내 단일 열로 전환한다.
- 오른쪽 영역은 화면 전체 높이가 아니라 데이터 높이만 차지하는 독립 카드다. 충분히 긴 화면에서 카드 자체에만 제한적으로 sticky를 적용할 수 있다.

## 검증 단계

- `배포 설정` 카드에는 `실행 타깃 결정 방식` label, 선택값만 보이는 Select, 별도 helper text를 표시한다.
- Select의 긴 설명은 펼쳐진 option에서만 보이고 닫힌 trigger에는 `자동 감지` 같은 실제 선택값만 표시한다.
- 검증 요약은 `설정 상태`, `변경 범위`, `Terraform Plan`을 가까운 3열로 배치하고 768px 이하에서는 세로 목록으로 바꾼다.
- 상태는 한국어 텍스트와 아이콘을 함께 사용하는 Badge로 표시한다. `infrastructure` 같은 시스템 값은 중립 code 스타일로 표시한다.
- 안내 문구와 `취소`, `저장 후 검증 실행`을 검증 요약 바로 아래에 배치한다.
- `취소`는 모달을 닫고, Primary CTA는 기존 `runDeploymentReviewStep`을 호출한다.
- 로딩 중에는 버튼을 비활성화하고 `aria-busy`와 진행 문구를 제공한다. 실행할 수 없는 이유는 인접 안내 문구로 표시한다.

## 최근 실행 결과 카드

- 승인 전 Plan 또는 안전 검사 실패는 `최근 검증 결과`, 승인 이후 실행은 `최근 배포 결과`, 구분할 수 없는 경우는 `최근 실행 결과`로 표시한다.
- `FAILED` 같은 내부 상태값을 그대로 노출하지 않고 `실패`처럼 한국어로 변환한다.
- 실패 상태는 아이콘, 약한 실패 배경의 Badge, 텍스트를 함께 사용하되 카드 전체를 빨간색으로 채우지 않는다.
- 범위, 실패 단계, 차단 상태를 실제 데이터가 있을 때 표시하고 실행 시각과 실패 원인도 값이 있을 때만 표시한다.
- 상세 동작은 선택된 실행의 기존 기록·로그 영역으로 이동한다.
- 실행 데이터가 없으면 작은 빈 상태를 표시하며 임시 원인이나 시각을 만들지 않는다.

## 배포 이력과 실행 세부정보

- 하단의 접힌 고정 `Deployment History` 바를 제거하고 메인 콘텐츠 아래 일반 `배포 이력` 섹션을 둔다.
- 이력 건수는 실제 `ApplicationRelease`만 계산한다. 실패한 검증 실행은 릴리즈 건수로 세지 않는다.
- 릴리즈가 없으면 `배포 이력 0건`과 첫 배포 완료 후 표시된다는 빈 상태를 보여준다.
- 실제 릴리즈는 최근 데이터 순서를 유지해 표시한다.
- Direct Deployment 실행 선택, Resource, Output, 로그는 `실행 세부정보`로 분리해 검증 실행과 릴리즈 이력의 의미를 섞지 않는다.

## 반응형과 접근성

- 1440px, 1280px, 1024px, 768px에서 가로 스크롤 없이 재배치한다.
- 작은 화면에서는 필드와 버튼이 전체 너비를 사용할 수 있고 액션 영역이 잘리지 않아야 한다.
- 모든 Select는 시각적 label과 접근 가능한 이름을 가진다.
- 최근 결과 상태 변경은 결과 컨테이너의 `role="status"`와 `aria-live="polite"`로 전달하고, 실행 오류는 기존 `role="alert"`를 유지한다.
- 포커스 표시를 제거하지 않으며 최소 클릭 영역과 WCAG AA 대비를 지킨다.
- 상태는 색상만으로 전달하지 않는다.
- 기존 전환 효과는 `prefers-reduced-motion`에서 제거한다.

## 테스트와 완료 기준

- 소스 회귀 테스트로 헤더, 컴팩트 Stepper, 단일 Select label/helper, 2열/단일 열 전환, 상태 Badge, 일반 이력 섹션, 한국어 문구를 확인한다.
- 상태 매핑 helper가 필요하면 순수 함수 테스트를 먼저 작성한다.
- 관련 Web 테스트를 실행하고 `pnpm harness:check`, `pnpm lint`, `pnpm typecheck`, `pnpm build`, `git diff --check`를 완료한다.
- 가능하면 실제 Workspace 배포 모달을 1440px, 1280px, 1024px, 768px에서 열어 overflow, 포커스, 빈 상태, 실패 상태를 확인한다.
