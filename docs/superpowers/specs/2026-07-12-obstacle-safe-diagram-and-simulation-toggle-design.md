# 장애물 회피 다이어그램과 시뮬레이션 토글 설계

## 목표

- AI 또는 템플릿이 생성한 다이어그램의 정렬은 유지한다.
- 화살표가 시작·도착 리소스를 제외한 다른 리소스의 시각 경계를 통과하지 않게 한다.
- Live Observation의 AI 시뮬레이션을 기본 ON 토글로 접고 펼칠 수 있게 한다.
- 시뮬레이션 결과는 병목, 장애, 비용 정보만 표시한다.

## 범위

### 다이어그램 경로

- 기존 `smoothstep` 엣지 표현과 상·하·좌·우 핸들 모델을 유지한다.
- 생성 또는 정규화 시 가능한 핸들 조합의 경로를 비교한다.
- 시작·도착 노드를 제외한 리소스의 시각 경계와 겹치는 후보에는 경로 길이보다 우선하는 차단 점수를 부여한다.
- 장애물을 피하는 후보가 있으면 반드시 해당 후보를 선택한다.
- 동일 점수에서는 기존 방향성, 경로 길이, 엣지 혼잡도 순으로 결정해 결과를 결정적으로 유지한다.
- Area 노드는 경로 장애물에서 제외하고 실제 Resource 노드만 회피 대상으로 본다.
- 리소스 뒤로 선을 숨기는 방식은 사용하지 않는다. 관계를 오해할 수 있기 때문이다.

이번 변경은 자유형 웨이포인트 편집기나 범용 A* 라우터를 추가하지 않는다. 현재 정렬과 기존 엣지 계약 안에서 우회 가능한 핸들을 강제하는 데 집중한다.

### AI 시뮬레이션

- 섹션 명칭은 `AI 시뮬레이션`으로 표시한다.
- `AI 시뮬레이션` 제목 행에 이진 토글을 추가한다.
- 초기값은 ON이다.
- OFF는 결과 영역만 숨긴다. 계산 요청, 계산 결과, 오류 상태는 유지한다.
- 다시 ON으로 전환하면 API를 재호출하지 않고 기존 결과를 즉시 표시한다.
- 트래픽 다이어그램, Live Observation polling/stream, 목업 애니메이션은 토글과 독립적으로 계속 동작한다.
- 결과 컴포넌트에서는 `요청 흐름` 섹션을 제거한다.
- 펼쳐진 결과 콘텐츠의 첫 항목은 기존 요약 문장이다.
- 남는 섹션은 `병목 후보`, `장애 대응`, `비용·다음 검토` 세 개다.
- 표시 순서는 `요약 문장 → 병목 후보 → 장애 대응 → 비용·다음 검토 → LLM 설명`이다.

## 컴포넌트 경계

### 엣지 라우팅

- `workspace-ai-diagram-adapter.ts`가 생성·정규화 시 경로 후보를 평가한다.
- 기존 리소스 시각 경계 계산 함수를 사용해 보드에 보이는 아이콘과 라벨 영역을 기준으로 충돌을 판단한다.
- 경로 평가 로직은 순수 함수로 유지해 중간 리소스가 있는 수평·수직 배치 테스트에서 직접 검증한다.

### 시뮬레이션 표시

- `LiveObservationModal.tsx`가 토글 상태를 소유한다.
- `WorkspaceAiDesignSimulationResult`는 요약 문장을 가장 먼저 렌더링한 뒤 세 결과 그룹을 렌더링한다.
- 결과 데이터 계약과 API DTO는 변경하지 않는다. `requestFlow` 데이터가 응답에 남아 있어도 UI에서 표시하지 않는다.

## 상태 흐름

1. 사용자가 시뮬레이션 모달을 연다.
2. 기존 동작대로 설계 시뮬레이션 요청과 관측 애니메이션이 자동 시작된다.
3. 토글은 ON 상태로 결과 또는 로딩 상태를 보여준다.
4. 사용자가 OFF로 바꾸면 결과 컨테이너만 접힌다.
5. 비동기 요청이 OFF 상태에서 완료되어도 결과는 상태에 저장된다.
6. 사용자가 다시 ON으로 바꾸면 저장된 결과가 즉시 렌더링된다.

## 오류 처리

- 시뮬레이션 요청 실패 상태도 토글과 함께 숨기고 복원한다.
- 토글 전환은 추가 API 요청을 만들지 않는다.
- 우회 가능한 핸들 조합이 없는 엣지는 기존 최소 충돌 점수 경로를 사용한다. 이 경우에도 렌더링을 중단하지 않는다.

## 테스트

- 중간 리소스가 직선상에 있을 때 선택된 경로가 해당 리소스 경계를 통과하지 않는지 검증한다.
- 장애물이 없을 때 기존의 짧은 좌우 경로가 유지되는지 검증한다.
- Area 노드는 장애물로 취급하지 않는지 검증한다.
- 시뮬레이션 토글이 기본 ON인지 검증한다.
- OFF 후 다시 ON으로 전환해도 `runAiDesignSimulation`이 추가 호출되지 않는 구조인지 검증한다.
- 결과 컴포넌트에 `요청 흐름`이 없고 병목, 장애, 비용 섹션만 남는지 검증한다.
- 관련 집중 테스트 후 `pnpm harness:check`, `pnpm lint`, `pnpm typecheck`, `pnpm build`를 실행한다.

## 비범위

- 사용자가 웨이포인트를 직접 편집하는 기능
- 모든 임의 배치에 대한 범용 최단 경로 탐색
- API 응답에서 `requestFlow` 필드를 제거하는 계약 변경
- AI 시뮬레이션의 재실행 버튼 추가
