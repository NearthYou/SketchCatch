# 장애물 회피 다이어그램과 시뮬레이션 토글 설계

## 목표

- AI 또는 템플릿이 생성한 다이어그램의 정렬은 유지한다.
- 화살표가 시작·도착 리소스를 제외한 다른 리소스의 시각 경계를 통과하지 않게 한다.
- Live Observation의 AI 시뮬레이션을 기본 ON 토글로 접고 펼칠 수 있게 한다.
- 시뮬레이션 결과는 병목, 장애, 비용 정보만 표시한다.
- 관측 헤더는 `LIVE OBSERVATION`과 `실시간 트래픽 관측`만 표시한다.
- AI 시뮬레이션을 펼쳐도 트래픽 부하 액션은 항상 접근 가능하게 유지한다.
- 트래픽 전송과 scale-out을 발표에서 즉시 알아볼 수 있는 단계적 애니메이션으로 복원한다.
- Capacity는 실제 최대 용량에 맞게 확장하되 개별 슬롯은 최대 8개까지만 표시한다.

## 범위

### 다이어그램 경로

- 기존 `smoothstep` 엣지 표현과 상·하·좌·우 핸들 모델을 유지한다.
- 생성 또는 정규화 시 가능한 핸들 조합의 경로를 비교한다.
- 시작·도착 노드를 제외한 리소스의 시각 경계와 겹치는 후보에는 경로 길이보다 우선하는 차단 점수를 부여한다.
- 장애물을 피하는 후보가 있으면 반드시 해당 후보를 선택한다.
- 동일 점수에서는 기존 방향성, 경로 길이, 엣지 혼잡도 순으로 결정해 결과를 결정적으로 유지한다.
- Area 노드는 경로 장애물에서 제외하고 실제 Resource 노드만 회피 대상으로 본다.
- 리소스 뒤로 선을 숨기는 방식은 사용하지 않는다. 관계를 오해할 수 있기 때문이다.

이번 변경은 자유형 웨이포인트 편집기나 범용 A* 라우터를 추가하지 않는다. 현재 정렬과 기존 엣지 계약 안에서 우회 가능한 핸들을 강제하는 데 집중한다.

### AI 시뮬레이션

- 섹션 명칭은 `AI 시뮬레이션`으로 표시한다.
- `AI 시뮬레이션` 제목 행에 이진 토글을 추가한다.
- 초기값은 ON이다.
- OFF는 결과 영역만 숨긴다. 계산 요청, 계산 결과, 오류 상태는 유지한다.
- 다시 ON으로 전환하면 API를 재호출하지 않고 기존 결과를 즉시 표시한다.
- 트래픽 다이어그램, Live Observation polling/stream, 목업 애니메이션은 토글과 독립적으로 계속 동작한다.
- 결과 컴포넌트에서는 `요청 흐름` 섹션을 제거한다.
- 펼쳐진 결과 콘텐츠의 첫 항목은 기존 요약 문장이다.
- 남는 섹션은 `병목 후보`, `장애 대응`, `비용·다음 검토` 세 개다.
- 표시 순서는 `요약 문장 → 병목 후보 → 장애 대응 → 비용·다음 검토 → LLM 설명`이다.

### 관측 헤더와 트래픽 부하 액션

- 헤더의 eyebrow는 `LIVE OBSERVATION`을 유지한다.
- 제목은 `실시간 트래픽 관측`으로 줄인다.
- 프로젝트명, `실제 배포 근거를 15분 동안 읽기 전용으로 관측`, `오토 스케일링 관측` 문구는 제거한다.
- 트래픽 부하 액션은 스크롤되는 본문과 AI 시뮬레이션 결과 영역 밖의 고정 Control Rail에 둔다.
- Control Rail은 실제 관측 세션이 없더라도 렌더링한다.
- 실제 관측 세션에서는 기존 `+90초 부하`가 실제 audience URL로 트래픽을 보낸다.
- 개발 목업 상태에서는 같은 액션이 목업 압력 단계를 즉시 진행시켜 버튼 동작을 확인할 수 있게 한다.
- AI 시뮬레이션 토글은 트래픽 부하 액션의 표시·실행 상태에 영향을 주지 않는다.

### 트래픽 및 Capacity 애니메이션

- 모달을 열면 개발 목업은 첫 트래픽 burst를 즉시 시작하고 이후 단계도 자동 진행한다.
- 실제 관측에서는 `acceptedEventCount`가 증가할 때마다 분석된 메인 경로의 각 연결 구간에 트래픽 점을 다시 생성한다.
- 트래픽 부하 버튼을 누르면 실제 세션과 개발 목업 모두 즉시 새 burst를 발생시켜 사용자가 반응을 확인할 수 있게 한다.
- 트래픽 점은 리소스 뒤를 통과하거나 마지막 Capacity 슬롯을 넘어가지 않는다.
- scale-out은 `공간 확보 → STARTING 슬롯 확장 → 강조 펄스 → RUNNING 전환` 순서로 표시한다.
- STARTING 슬롯은 크기와 투명도를 함께 변화시키고, RUNNING 전환 시 한 번의 강한 테두리 펄스와 상태 색상 전환을 사용한다.
- `prefers-reduced-motion`에서는 크기 이동을 제거하고 상태 색상과 텍스트 변경만 유지한다.

### Capacity 슬롯 정책

- 필요한 슬롯 수는 다이어그램의 capacity-unit 수, 현재 용량, desired capacity, max capacity 중 가장 큰 값을 기준으로 계산한다.
- 개별 Capacity 슬롯은 최대 8개까지 표시한다.
- 최대 용량이 8개를 넘으면 8개 슬롯 뒤에 `+N` 요약 표시를 추가한다.
- 슬롯 수가 늘어나면 Capacity 영역과 전체 경로의 최소 너비가 함께 늘어나며, 모달 폭을 넘으면 기존 가로 스크롤을 사용한다.
- 슬롯이 추가되더라도 앞선 메인 경로 리소스의 위치와 간격은 변하지 않는다.
- 미실행 슬롯은 `EXPECTED`, 생성 중은 `STARTING`, 정상 실행은 `RUNNING`, 관측 불가는 `UNAVAILABLE`로 구분한다.

## 컴포넌트 경계

### 엣지 라우팅

- `workspace-ai-diagram-adapter.ts`가 생성·정규화 시 경로 후보를 평가한다.
- 기존 리소스 시각 경계 계산 함수를 사용해 보드에 보이는 아이콘과 라벨 영역을 기준으로 충돌을 판단한다.
- 경로 평가 로직은 순수 함수로 유지해 중간 리소스가 있는 수평·수직 배치 테스트에서 직접 검증한다.

### 시뮬레이션 표시

- `LiveObservationModal.tsx`가 토글 상태를 소유한다.
- `WorkspaceAiDesignSimulationResult`는 요약 문장을 가장 먼저 렌더링한 뒤 세 결과 그룹을 렌더링한다.
- 결과 데이터 계약과 API DTO는 변경하지 않는다. `requestFlow` 데이터가 응답에 남아 있어도 UI에서 표시하지 않는다.

### Control Rail

- `LiveObservationModal.tsx`가 실제 세션과 개발 목업 중 현재 실행 대상을 결정한다.
- Control Rail은 모달 본문 바깥에서 고정되어 AI 결과가 길어져도 화면에서 밀려나지 않는다.
- 실제 부하와 목업 부하는 동일 버튼을 공유하지만, 실제 AWS 상태와 목업 상태 표시는 기존 구분을 유지한다.

### Capacity 표시 모델

- `live-observation-diagram.ts`가 snapshot의 capacity 값을 바탕으로 표시 슬롯과 초과 개수를 계산한다.
- `LiveObservationDiagramMap.tsx`는 계산된 슬롯 배열을 그대로 렌더링하고 슬롯 수에 비례해 최소 너비를 결정한다.
- 최대 8개 제한은 표시 전용이며 실제 capacity 관측 데이터와 AWS 동작을 변경하지 않는다.

## 상태 흐름

1. 사용자가 시뮬레이션 모달을 연다.
2. 기존 동작대로 설계 시뮬레이션 요청과 관측 애니메이션이 자동 시작된다.
3. 토글은 ON 상태로 결과 또는 로딩 상태를 보여준다.
4. 사용자가 OFF로 바꾸면 결과 컨테이너만 접힌다.
5. 비동기 요청이 OFF 상태에서 완료되어도 결과는 상태에 저장된다.
6. 사용자가 다시 ON으로 바꾸면 저장된 결과가 즉시 렌더링된다.
7. 사용자는 AI 결과의 펼침 상태와 무관하게 고정 Control Rail에서 트래픽 부하를 실행하거나 중단한다.
8. scale-out이 시작되면 새 Capacity 슬롯이 STARTING으로 확장되고, 완료 snapshot에서 RUNNING으로 전환된다.

## 오류 처리

- 시뮬레이션 요청 실패 상태도 토글과 함께 숨기고 복원한다.
- 토글 전환은 추가 API 요청을 만들지 않는다.
- 실제 세션이 없고 개발 목업도 아닌 환경에서는 부하 버튼을 비활성화한다.
- 우회 가능한 핸들 조합이 없는 엣지는 기존 최소 충돌 점수 경로를 사용한다. 이 경우에도 렌더링을 중단하지 않는다.
- `maxCapacity`가 없거나 관측 불가인 경우 다이어그램의 capacity-unit 수와 현재 snapshot 용량으로 슬롯 수를 결정한다.

## 테스트

- 중간 리소스가 직선상에 있을 때 선택된 경로가 해당 리소스 경계를 통과하지 않는지 검증한다.
- 장애물이 없을 때 기존의 짧은 좌우 경로가 유지되는지 검증한다.
- Area 노드는 장애물로 취급하지 않는지 검증한다.
- 시뮬레이션 토글이 기본 ON인지 검증한다.
- OFF 후 다시 ON으로 전환해도 `runAiDesignSimulation`이 추가 호출되지 않는 구조인지 검증한다.
- 결과 컴포넌트에 `요청 흐름`이 없고 병목, 장애, 비용 섹션만 남는지 검증한다.
- 헤더에 `LIVE OBSERVATION`과 `실시간 트래픽 관측`만 남고 제거 대상 설명이 없는지 검증한다.
- AI 결과가 펼쳐진 상태에서도 Control Rail과 트래픽 부하 버튼이 렌더링되는지 검증한다.
- 개발 목업 상태의 부하 버튼이 목업 압력 단계를 진행시키고 실제 traffic API를 호출하지 않는지 검증한다.
- 모달 초기 개발 목업과 부하 버튼 클릭 모두 새 트래픽 burst를 생성하는지 검증한다.
- scale-out 단계가 `EXPECTED → STARTING → RUNNING`으로 전환되는지 검증한다.
- max capacity 2, 8, 12에서 각각 2개, 8개, `8개 + +4` 표시가 만들어지는지 검증한다.
- Capacity 슬롯 증가가 메인 경로 stage 수와 위치 계산을 변경하지 않는지 검증한다.
- 관련 집중 테스트 후 `pnpm harness:check`, `pnpm lint`, `pnpm typecheck`, `pnpm build`를 실행한다.

## 비범위

- 사용자가 웨이포인트를 직접 편집하는 기능
- 모든 임의 배치에 대한 범용 최단 경로 탐색
- API 응답에서 `requestFlow` 필드를 제거하는 계약 변경
- AI 시뮬레이션의 재실행 버튼 추가
