# 실시간 관측 흐름 애니메이션 설계

## 목적

Live Observation의 메인 트래픽 경로를 실제 관측 상태와 구분되는 명확한 모션으로 표현한다. 트래픽이 없을 때도 화면이 멈춘 것처럼 보이지 않게 하되, 실제 요청이 발생한 것처럼 오해시키지 않는다. ECS Fargate Task의 Scale-out과 Scale-in도 현재 Provider 관측값 변화에 맞춰 나타나고 사라지는 과정으로 보여준다.

이 변경은 Live Observation 화면의 표현만 바꾼다. AWS 요청 생성, 관측 세션 시작, Terraform, 배포 리소스, API 계약은 변경하지 않는다.

## 선택한 접근

현재 `LiveObservationFocusedFlow`의 Architecture 기반 경로와 반응형 DOM 구조를 유지하고 상태별 CSS 애니메이션을 강화한다. 단일 SVG 경로로 전체 화면을 재구현하거나 이전 EC2/ASG 중심 Signal Map을 복원하지 않는다.

이 접근은 현재 ECS 경로 의미를 보존하면서 수정 범위를 Focused Flow, 모델, 스타일, 회귀 테스트로 제한한다.

## 동작 설계

### 유휴 상태

- 모든 연결선에 낮은 대비의 흐름 애니메이션을 계속 표시한다.
- 유휴 흐름은 시스템이 관측 가능한 상태임을 나타낼 뿐 요청을 의미하지 않는다.
- 실제 요청 공, 노드 도착 파동, 요청 수 증가는 표시하지 않는다.

### 실제 트래픽

- 공개 요청 수 또는 새로운 CloudWatch RequestCount가 증가했을 때만 트래픽 공을 생성한다.
- 한 번에 최대 5개의 공을 보여주며 나머지 요청 수는 기존 집계 근거에 남긴다.
- 각 공은 CloudFront에서 시작해 연결선을 따라 ALB, Listener, Target Group, ECS Service, Fargate Task 순서로 이동한다.
- 공이 노드에 도착할 때 짧은 링 피드백을 표시한다.
- 실제 이벤트가 끝나면 공과 도착 피드백은 제거되고 유휴 흐름만 남는다.

### Fargate Scale-out

- 화면에는 Architecture 기준 최소 Task와 Provider가 보고한 `running` 또는 `desired` 수만 표시한다.
- `max` 값만으로 흐린 가상 Task 슬롯을 미리 렌더링하지 않는다.
- `desired > running`인 새 Task는 확대·페이드인되며 `STARTING` 상태로 표시한다.
- Provider가 `running` 증가를 보고하면 해당 Task가 초록색 도착 파동과 함께 `RUNNING`으로 전환된다.

### Fargate Scale-in

- 이전 snapshot에 존재하던 Task가 다음 snapshot에서 감소하면 제거 대상으로 표시한다.
- 제거 Task는 약 600ms 동안 축소·페이드아웃된 뒤 DOM에서 제거한다.
- Scale-in 애니메이션은 실제 `running`/`desired` 감소에서만 시작하며 유휴 상태에서 반복하지 않는다.

## 레이아웃

- Focused Flow의 고정 `52vh` 높이를 제거하고 실제 경로 콘텐츠 높이에 맞춘다.
- 경로 아래의 의미 없는 빈 공간을 제거한다.
- `전체 Architecture 보기` disclosure와 `LiveObservationDiagramMap` 렌더링을 제거한다.
- `WorkspaceDesignAnalysisPanel`과 설계 분석 영역을 Live Observation에서 제거한다.
- 관측 세션이 활성화되면 요청률, 오류율, 지연, 가용성, 실제 Task 수와 로그 근거는 기존처럼 경로 아래에 표시한다.

## 구성 요소 경계

- `LiveObservationFocusedFlow`: 유휴 흐름, 실제 요청 burst, Task 진입·전환·퇴장 표현을 담당한다.
- `live-observation-diagram.ts`: 표시할 실제/희망 Task 수와 Task 상태를 Provider snapshot에서 계산한다.
- `live-observation-diagram-particles.ts`: 요청 공의 구간별 지연과 수명을 계산한다.
- `LiveObservationModal`: Focused Flow와 관측 근거만 조합하고 Architecture/설계 분석 UI를 더 이상 렌더링하지 않는다.
- `workspace.module.css`: 유휴 연결선, 요청 공, 노드 도착, Task Scale-out/Scale-in 애니메이션을 정의한다.

프런트엔드는 AWS SDK나 Terraform 실행 로직을 포함하지 않는다. 모든 상태는 기존 Live Observation snapshot과 배포 Architecture 조회 결과만 사용한다.

## 접근성

- `prefers-reduced-motion: reduce`에서는 반복 흐름, 이동 공, 확대·축소 애니메이션을 중지한다.
- 모션을 중지해도 `STARTING`, `RUNNING` 상태 라벨과 Task 수 변화는 즉시 반영한다.
- 애니메이션 요소는 `aria-hidden`을 유지하고 실제 상태는 기존 텍스트와 관측 근거로 제공한다.

## 테스트

- 유휴 상태에는 연결선 흐름이 존재하지만 실제 요청 공은 생성되지 않는 계약을 검증한다.
- 실제 요청 증가에서만 burst가 생성되고 최대 5개로 제한되는 기존 회귀를 유지한다.
- Provider `max`만으로 비활성 Task 슬롯을 만들지 않는 모델 회귀를 추가한다.
- `desired > running`이 `STARTING`, `running` 증가가 `RUNNING`으로 계산되는지 검증한다.
- Task 감소가 Scale-in 퇴장 상태를 만들고 제한 시간 뒤 제거되는 순수 상태 계산을 검증한다.
- Live Observation modal에서 Architecture disclosure와 설계 분석이 제거됐는지 검증한다.
- 브라우저에서 유휴 흐름, 실제 트래픽 공을 위한 DOM 상태, Scale-out/Scale-in 전환, 불필요한 하단 공백 제거를 확인한다.

## 범위 밖

- 가짜 요청 또는 데모용 요청 수 생성
- 사용자 승인 전 관측 세션이나 부하 테스트 자동 시작
- Live Observation API, SSE, capability token, Redis Store 변경
- AWS Auto Scaling 정책이나 Terraform 변경
- Deployment Output URL 자동 호출
