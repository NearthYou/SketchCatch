# PR #20에서 내가 알아야 할 내용

> 이 문서는 과거 PR #20을 다시 설명할 때 보는 참고 문서다.
> 현재 최신 실행 화면은 `/workspace/ai`가 아니라 실제 Workspace 안의 AI 패널이다.

## 한 줄 요약

이번 PR은 gg AI 파트의 1차 구현이다.

핵심은 **사용자 입력을 받아 Architecture Draft를 만들고, 배포 전 점검과 Terraform 설명을 fallback 방식으로 제공하는 것**이다.

외부 LLM이 실제로 붙은 PR은 아니다. 지금은 API, shared type, fallback rule, 프론트 연결 화면을 먼저 만들어서 팀원이 붙을 수 있는 기준을 세운 PR이다.

## 이번 PR의 핵심 흐름

```text
사용자 입력
→ AI API route
→ deterministic fallback / rule engine
→ ArchitectureJson, finding, checklist, 설명 응답
→ /workspace/ai 데모 화면에서 확인
```

여기서 중요한 말은 `ArchitectureJson`이다.

`ArchitectureJson`은 그냥 아무 JSON이 아니라, 팀이 같이 쓰기로 한 인프라 설계도 JSON이다.

```ts
type ArchitectureJson = {
  nodes: ResourceNode[];
  edges: ResourceEdge[];
};
```

- `nodes`: VPC, EC2, RDS, S3 같은 Resource 목록
- `edges`: Resource 사이의 연결선

즉 AI가 말로만 “이런 구조가 좋다”고 하는 게 아니라, 보드가 바로 열 수 있는 형태로 설계 초안을 반환하는 게 핵심이다.

## 구현된 API

### 1. 자연어 기반 Architecture Draft

API:

```text
POST /api/ai/architecture-draft
```

역할:

- 자연어 요청을 받아 Architecture Draft를 만든다.
- 결과는 `architectureJson`으로 반환된다.
- 현재는 LLM이 아니라 keyword 기반 fallback이다.

현재 fallback 기준:

```text
db / database / 데이터베이스 / rds / 백엔드
→ DB 포함 백엔드 구조

api / 서버 / ec2
→ API 서버 구조

그 외
→ 정적 웹사이트 구조
```

팀원이 물어보면 이렇게 말하면 된다.

> 지금은 실제 LLM이 의도를 깊게 이해하는 단계가 아니라, API 계약과 보드 연결을 검증하기 위한 deterministic fallback입니다. 다음 단계에서 자연어 의도 분류를 고도화할 예정입니다.

### 2. GitHub 기반 Architecture Draft

API:

```text
POST /api/ai/github-architecture-draft
```

역할:

- public GitHub repo URL을 받는다.
- README, package metadata, Dockerfile, compose file 정도만 evidence로 본다.
- 전체 코드를 분석하지 않는다.

중요한 점:

- GitHub는 지금 핵심 경로가 아니다.
- “GitHub 코드만 보고 정답 인프라를 만든다”라고 말하면 안 된다.
- 지금은 보조 evidence 또는 후순위 기능에 가깝다.

팀원이 물어보면 이렇게 말하면 된다.

> GitHub 링크는 메인 기능이 아니라 보조 evidence입니다. 이번 PR의 핵심은 자연어 요구사항을 ArchitectureJson으로 바꾸는 계약을 만든 것입니다.

### 3. 배포 전 점검

API:

```text
POST /api/ai/pre-deployment-check
```

역할:

- `ArchitectureJson`을 받아 비용, 보안, 설정 누락 finding을 만든다.
- 결과로 summary, cost estimate, findings, checklist를 반환한다.

중요한 응답:

```ts
type AiPreDeploymentAnalysisResult = {
  summary: string;
  totalMonthlyEstimate: MoneyEstimate & {
    pricingAssumption: string;
  };
  resourceCostEstimates: ResourceCostEstimate[];
  findings: CheckFinding[];
  checklist: ChecklistItem[];
};
```

예시 finding:

- Security Group이 SSH `0.0.0.0/0`을 허용하면 high security finding
- RDS가 있으면 cost finding
- EC2에 `instanceType`이 없으면 configuration finding

팀원이 물어보면 이렇게 말하면 된다.

> AI가 위험을 마음대로 판단하는 게 아니라, rule engine이 finding을 만들고 AI layer는 그걸 사용자에게 설명하는 구조입니다.

### 4. Terraform 오류 설명

API:

```text
POST /api/ai/terraform-error-explanation
```

입력:

```ts
{
  stage: "validate" | "export" | "plan" | "apply";
  rawMessage: string;
  relatedResourceId?: string;
}
```

역할:

- Terraform 오류 원문을 쉬운 말로 바꿔준다.
- 권한, 인증, region/resource, quota, syntax, dependency, unknown으로 분류한다.

중요한 점:

- 1차 제공에서는 `validate`, `export`가 더 중요하다.
- `plan`, `apply`는 2차 제공에서 ck 배포 파트와 연결될 때 본격적으로 쓰인다.

팀원이 물어보면 이렇게 말하면 된다.

> 지금 오류 설명 DTO는 1차 validate/export와 2차 plan/apply를 같이 담을 수 있게 미리 열어둔 것입니다.

### 5. Terraform Preview 설명

API:

```text
POST /api/ai/terraform-preview-explanation
```

역할:

- Terraform 코드 문자열을 받아 어떤 Resource가 있는지 설명한다.
- open SSH나 RDS 비용 같은 위험도 같이 찾는다.

중요한 점:

- AI가 Terraform 최종본을 만드는 기능이 아니다.
- Terraform 생성은 sw 파트의 결정론적 생성기가 담당하는 게 맞다.
- gg AI는 “이 코드가 뭘 만드는지”, “어디가 위험한지”를 설명한다.

## shared type에서 중요한 것

### ResourceType

현재 AI가 만들 수 있는 Resource type:

```ts
type ResourceType =
  | "VPC"
  | "SUBNET"
  | "EC2"
  | "RDS"
  | "S3"
  | "SECURITY_GROUP"
  | "CLOUDFRONT"
  | "LAMBDA"
  | "UNKNOWN";
```

팀원과 맞춰야 하는 이유:

- jh 보드는 이 문자열로 노드를 그린다.
- sw Terraform 생성기는 이 문자열로 Terraform resource를 만든다.
- gg 분석은 이 문자열로 비용/보안 rule을 적용한다.

예를 들어 누가 `SecurityGroup`, `security_group`, `SECURITY-GROUP`처럼 다르게 쓰면 연결이 깨진다.

### CheckFinding

```ts
type CheckFinding = {
  id: string;
  category: CheckFindingCategory;
  severity: "low" | "medium" | "high";
  resourceId?: string;
  title: string;
  description: string;
  recommendation: string;
};
```

중요한 필드:

- `resourceId`: 보드의 `nodes[].id`와 연결된다.
- `severity`: 위험도를 보여준다.
- `recommendation`: 사용자가 뭘 해야 하는지 알려준다.

팀원이 물어보면 이렇게 말하면 된다.

> finding은 그냥 경고 문구가 아니라 특정 Resource에 붙일 수 있는 분석 결과입니다.

### ChecklistItem

```ts
type ChecklistItem = {
  id: string;
  label: string;
  status: "pass" | "warning" | "fail";
  relatedFindingIds: string[];
};
```

역할:

- 사용자가 배포 전 확인해야 할 항목을 보여준다.
- finding 여러 개를 checklist 하나에 연결할 수 있다.

## 팀원별로 연결되는 부분

### jh 보드 파트

gg가 주는 것:

- `architectureJson`
- `findings[].resourceId`

jh가 쓰는 방식:

- `architectureJson.nodes`로 보드 노드를 그린다.
- `architectureJson.edges`로 연결선을 그린다.
- `finding.resourceId`로 위험 배지를 특정 노드에 붙인다.

말할 것:

> AI 응답에서 보드가 꼭 봐야 하는 건 `architectureJson`입니다. metadata는 설명용이라 보드 필수 입력이 아니어야 합니다.

### sw Terraform 파트

gg가 주는 것:

- `ArchitectureJson`
- Resource별 config
- Terraform Preview 설명 API

sw가 쓰는 방식:

- `ArchitectureJson`을 원천으로 Terraform 코드를 생성한다.
- AI가 만든 자유 Terraform 코드를 원천으로 삼으면 안 된다.

말할 것:

> Terraform 생성의 원천은 AI가 아니라 `ArchitectureJson`입니다. AI는 생성된 코드 설명과 위험 분석을 맡습니다.

### ck 배포 파트

gg가 주는 것:

- Pre-Deployment Check 결과
- Terraform 오류 설명 API

ck가 쓰는 방식:

- 배포 전 blocked 여부나 warning 표시 참고
- Plan/Apply 실패 시 `{ stage, rawMessage, relatedResourceId? }` 형태로 gg API 호출

말할 것:

> raw log 전체를 AI에 넘기기보다, masking된 최소 오류 메시지만 넘기는 구조가 안전합니다.

### ys 플랫폼 파트

gg가 주는 것:

- optional AI summary
- findings/checklist

ys가 쓰는 방식:

- 프로젝트 상세 화면이나 dashboard에서 AI 분석 결과를 보여줄 수 있다.
- 프로젝트 목록에는 필수로 넣지 않아도 된다.

말할 것:

> 프로젝트 목록이 AI 분석 저장 정책에 묶이지 않도록, AI 요약은 optional로 보는 게 안전합니다.

아래는 아직 구현 완료라고 말하면 안 된다.

## 리뷰나 발표 때 자주 나올 질문

### Q. 지금 실제 AI가 붙어 있나요?

아니요.

지금은 외부 LLM 없이도 동작하는 fallback과 rule engine이 먼저 구현되어 있습니다. 이렇게 한 이유는 API key, timeout, rate limit 때문에 데모가 깨지는 걸 막기 위해서입니다.

### Q. 그럼 AI 파트라고 보기 어렵지 않나요?

이번 PR은 AI 기능의 “외부 provider 호출”보다 “AI 응답 계약과 안전한 fallback 구조”를 먼저 만든 것입니다.

LLM은 이후에 자연어 의도 분류와 설명 생성 쪽으로 붙일 수 있습니다.

### Q. GitHub 링크 기능은 왜 약하게 뒀나요?

코드만 보고 인프라 정답을 알 수 없기 때문입니다.

같은 백엔드 코드도 EC2, ECS, Lambda, Kubernetes 등 여러 방식으로 배포할 수 있습니다. 그래서 GitHub 링크는 보조 evidence로만 두고, 핵심은 사용자의 자연어 요구사항입니다.

### Q. 왜 AI가 Terraform을 직접 만들지 않나요?

위험하기 때문입니다.

Terraform은 실제 비용과 리소스 생성으로 이어질 수 있으므로, 최종 생성은 결정론적 생성기가 맡고 AI는 설명과 위험 분석을 맡는 구조가 안전합니다.

### Q. 왜 TDD를 했나요?

AI API는 팀원들이 같이 쓰는 계약이라, 응답 모양이 깨지면 연결이 전부 깨집니다.

그래서 최소한 API contract test를 먼저 잡아두는 게 안전합니다.
