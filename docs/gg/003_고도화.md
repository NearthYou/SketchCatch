# AI 고도화 고려사항

이 문서는 지금 당장 만들지 않는 기능을 잊지 않기 위한 문서다.

쉽게 말하면:

> "이번 MVP에서는 못 하지만, 나중에 하면 좋은 것들"을 적어두는 곳이다.

## 1. 기록 원칙

- 지금 만들 것은 `002_MVP범위_gg.md`에 적는다.
- 나중에 만들 것은 이 문서에 적는다.
- 지금 안 한다고 영원히 버리는 것은 아니다.
- 일정 때문에 뒤로 미룬 이유도 같이 적는다.
- 실제 AWS 비용, 보안, 배포와 연결되는 기능은 항상 안전장치를 같이 생각한다.

| 단계 | 의미 |
| --- | --- |
| 1차 | 자연어 설계, 비용/보안 점검, Terraform 설명, 기본 시뮬레이션 |

## 3. 자연어 설계 고도화

1차에서는 대표 시나리오만 지원한다.

예:

- 정적 웹사이트
- API 서버
- DB 포함 백엔드

나중에는 더 많은 조건을 받을 수 있다.

예:

- region
- runtime
- database 종류
- traffic 수치
- 예산 금액
- 보안 수준
- 가용성 수준

지금 미루는 이유:

- 조건을 많이 받을수록 타입과 UI가 커진다.
- sw Terraform 변환과도 더 많이 맞춰야 한다.
- 먼저 작은 입력이 안정적으로 돌아가야 한다.

## 4. AI 수정 제안 고도화

1차에서는 `ArchitectureSuggestion` 타입으로 구조화된 수정 제안까지만 만든다.

이번에 하는 것:

- `CheckFinding.recommendation`은 짧은 권장 문장으로 둔다.
- 별도 `ArchitectureSuggestion`에서 대상 Resource, 제안 행동, 예상 영향을 표현한다.
- 제안은 화면에 보여주기만 하고 자동 적용하지 않는다.

나중에는 사용자가 이렇게 말할 수 있게 한다.

```text
비용 줄여줘
보안 강화해줘
트래픽 더 버티게 바꿔줘
```

AI는 바로 적용하지 않는다.

먼저 바뀔 내용을 보여준다.

```text
추가될 리소스
삭제될 리소스
변경될 리소스
예상 비용 변화
위험 변화
```

사용자가 확인해야 적용한다.

이번에 미루는 것:

- 제안을 Architecture Board에 자동 적용
- Resource diff 실제 생성
- Terraform 코드 자동 재생성
- 사용자의 자연어 수정 명령을 받아 설계를 직접 변경

지금 미루는 이유:

- 자동 수정은 위험하다.
- 보드, Terraform, 비용 분석이 모두 같은 변경을 이해해야 한다.
- 먼저 "제안"과 "적용"의 경계를 분리해야 한다.

## 5. 운영 조건 구조 반영

이번에는 운영 조건을 화면에서 고르게 만들었고, 일부 안전한 설정만 `ArchitectureJson.config`에 반영했다.

이번에 반영한 것:

- 낮은 예산 + RDS 조합이면 비용 warning을 보여준다.
- 보안 우선순위가 높으면 S3 public access 차단 설정을 넣는다.
- 보안 우선순위가 높으면 RDS public 접근 차단 설정을 넣는다.
- 보안 우선순위가 높으면 Security Group 기본 ingress를 비워 둔다.

하지만 리소스 개수나 큰 구조까지 자동으로 바꾸는 일은 아직 하지 않는다.

운영 조건:

- 예산
- 트래픽
- 보안 우선순위

이 값들은 설명만 바꾸면 부족하다.

나중에는 `ArchitectureJson` 안의 리소스 목록, 연결선, 설정값이 더 크게 바뀔 수 있어야 한다.

예:

```text
예산 낮게
→ 작은 EC2, 작은 RDS, 비싼 리소스 제외

예산 보통
→ 조금 더 큰 인스턴스나 안정성 옵션 고려

트래픽 작음
→ 단일 EC2, 단순 구조

트래픽 보통
→ ALB, Auto Scaling 같은 확장 구조 후보

보안 기본
→ 최소 보안 설정

보안 높음
→ private subnet, DB public 접근 차단, 보안 그룹 더 제한
```

아직 미루는 것:

- ALB 자동 추가
- Auto Scaling 자동 추가
- NAT Gateway 자동 추가
- Multi-AZ RDS 자동 추가
- private subnet/public subnet 정교한 분리
- 인스턴스 크기 자동 변경

아직 미루는 이유:

- 구조가 바뀌면 jh 보드, sw Terraform 생성, ck 배포 흐름까지 같이 맞춰야 한다.
- 리소스가 늘어나면 비용 계산과 보안 규칙도 같이 늘어난다.
- 지금은 공통 지원 리소스 안에서 안전한 config만 먼저 반영하는 것이 더 안전하다.

## 6. 비용/트래픽 시뮬레이션 고도화

1차에서는 `Design Simulation`의 최소 세로 조각을 만든다.

이번에 할 것:

- Resource와 Edge를 보고 요청 흐름을 만든다.
- 병목 후보를 Resource와 연결한다.
- 단일 EC2, 단일 RDS, public Resource 같은 구조적 실패 시나리오를 설명한다.
- 비용 압박과 트래픽 가정을 쉬운 말로 설명한다.

단, 실제 성능 측정이나 실제 AWS 지표 분석은 하지 않는다.

1차 이후에는 더 정량적인 추정으로 고도화한다.

예:

- RDS는 비용이 커질 수 있음
- 작은 EC2는 병목이 될 수 있음
- CloudFront는 트래픽 비용이 생길 수 있음

나중에는 사용자가 값을 바꾸면 다시 계산하게 한다.

예:

```text
월 방문자 1만 명
월 방문자 10만 명
예산 50달러
예산 100달러
```

지금 미루는 이유:

- 실제 가격과 성능을 정확히 보장하기 어렵다.
- 먼저 "설계 검토용 추정"이라는 선을 지켜야 한다.

## 8. AI 결과 저장 기준

이번에는 gg AI가 DB에 직접 저장하지 않는 방향으로 정리했다.

지금 gg AI는 이렇게 동작한다.

```text
요청 받음
→ 계산
→ 응답 반환
```

나중에 저장할 수 있는 후보:

- AI 분석 결과 요약
- AI finding
- AI checklist
- AI 요청 로그
- 외부 AI 사용량
- 외부 AI 비용

하지만 저장 주체를 잘 정해야 한다.

기본 원칙:

```text
gg AI는 계산/설명 담당
저장은 그 결과를 실제로 쓰는 파트가 담당
```

예:

```text
ArchitectureJson 저장
→ jh 보드 또는 프로젝트 저장 흐름

Terraform 파일 저장
→ sw Terraform 또는 공통 asset 흐름

Plan/Apply 분석 결과 저장
→ ck 배포 이력 흐름

프로젝트 목록 AI 요약 저장
→ ys 플랫폼 흐름
```

지금 미루는 이유:

- AI 결과를 무조건 저장하면 DB schema가 커진다.
- 같은 `ArchitectureJson`으로 다시 계산할 수 있는 결과는 MVP에서 꼭 저장하지 않아도 된다.
- 저장 위치를 잘못 잡으면 gg AI가 다른 파트의 책임까지 가져가게 된다.

## 11. LLM provider 고도화

1차에서는 fallback이 먼저다.

즉 OpenAI API가 없어도 최소 기능은 돌아가야 한다.

LLM 보강 결과는 기존 rule 결과를 덮어쓰지 않는다.

예:

```text
Check Finding 원본
→ 코드가 만든 위험 근거

LLM 보강 설명
→ 사용자가 이해하기 쉬운 요약과 다음 행동
```

나중에는 사용자 화면에서 raw Check Finding 노출을 줄이고, LLM 보강 설명을 더 앞에 보여줄 수 있다.

하지만 API와 내부 판단 근거에서는 Check Finding을 바로 제거하지 않는다.

이유:

- `ArchitectureSuggestion`이 어떤 Check Finding에서 나왔는지 추적해야 한다.
- LLM이 실패하거나 이상한 문장을 만들 때 원래 rule 판단으로 돌아가야 한다.
- 팀원 연동에서 `findings.resourceId`가 Resource 연결 기준으로 쓰인다.

지금 MVP에서는 LLM 보강 결과를 공통 필드 하나로 통일한다.

예:

```ts
llmEnhancement: {
  target: "design_simulation",
  summary: string,
  highlights: string[],
  nextActions: string[],
  fallbackUsed: boolean,
  fallbackReason?: string
}
```

이 구조는 빠르게 붙이고, 프론트가 같은 방식으로 처리하기 위한 기준이다.

MVP 화면에서는 `llmEnhancement`를 패널 맨 위의 공통 요약 박스로 크게 보여주지 않는다.

대신 관련 결과 항목 옆에 작은 `AI 설명` 영역으로 붙인다.

이 UI는 `/workspace/ai` 데모 화면뿐 아니라 실제 `/workspace` 화면에도 연결한다.

단, 실제 `/workspace`에서는 설명을 보여주는 데까지만 한다.

LLM 설명을 근거로 Architecture Board의 Resource를 자동 수정하는 것은 하지 않는다.

예:

- `Design Simulation`: 요청 흐름, 병목, 장애 시나리오, 비용 압박 영역 근처
- `Pre-Deployment Check`: Check Finding 또는 `ArchitectureSuggestion` 항목 근처
- Terraform 오류 설명: 오류 설명 결과 근처
- `Architecture Draft`: 선택된 시나리오나 warning 근처

다만 MVP API 구조는 여전히 공통 `llmEnhancement` 하나로 유지한다.

`findingId`별 설명 배열처럼 항목과 LLM 설명을 1:1로 매핑하는 구조는 고도화로 미룬다.

`fallbackUsed: true`일 때 사용자 화면에는 작은 `기본 설명 사용` 표시만 보여준다.

에러처럼 크게 보여주지 않는다.

이유:

- LLM 보강 실패는 API 실패가 아니다.
- 기본 rule 설명은 정상 결과다.
- 시연이나 디버깅 중에는 OpenAI 설명이 아니라 fallback 설명이라는 사실을 알 수 있어야 한다.

`fallbackReason`은 화면 중심 문구로 노출하지 않는다.

필요하면 개발자가 응답 JSON이나 로그에서 확인한다.

`fallbackUsed`는 LLM 호출 전체 실패만 뜻하지 않는다.

아래 상황도 `fallbackUsed: true`로 본다.

- API key 없음
- provider timeout
- LLM 응답 검증 실패
- summary는 성공했지만 `nextActions`만 rule 기본값으로 대체
- 일부 필드라도 rule 기반 기본값으로 대체

`fallbackReason`은 optional이다.

화면에 꼭 보여주기 위한 값은 아니고, 개발자가 왜 fallback이 섞였는지 확인하기 위한 값이다.

`fallbackReason`은 안전한 enum 값만 반환한다.

허용 값:

```text
missing_api_key
timeout
rate_limited
invalid_request
auth_error
provider_error
invalid_response
```

원본 OpenAI 에러 메시지는 API 응답에 넣지 않는다.

서버 로그에도 원본 OpenAI 에러 메시지를 그대로 남기지 않는다.

OpenAI 실패 시 서버 로그에 남길 값:

- `target`
- `fallbackReason`
- 짧은 safe message

서버 로그에 남기지 않을 값:

- 원본 OpenAI 에러 메시지
- 원본 요청 payload
- API key
- 민감할 수 있는 값

이번에 하지 않는 것:

- 원본 OpenAI 에러 메시지를 서버 로그에 그대로 남기기
- 요청 payload 전체를 로그에 남기기
- LLM 실패 로그를 아예 남기지 않기

예:

```text
llm enhancement fallback target=design_simulation reason=timeout
```

이유:

- 에러 메시지에 내부 설정이나 요청 정보가 섞일 수 있다.
- 사용자에게 필요한 것은 원인 분류이지 raw provider 메시지가 아니다.
- 디버깅에는 safe enum만으로도 충분한 1차 구분이 가능하다.
- 전체 실패 로그 저장과 정교한 secret masking은 별도 보안 기준이 필요하다.

테스트에서는 실제 OpenAI를 호출하지 않는다.

각 AI API route마다 아래 흐름을 fake OpenAI client로 확인한다.

- LLM 성공
- provider 실패 또는 timeout으로 fallback
- JSON 파싱 실패나 Zod 검증 실패 같은 invalid response fallback

확인해야 할 핵심은 아래다.

- fallback이어도 rule 기반 결과가 사라지지 않음
- fallback이어도 API 전체 실패로 바뀌지 않음
- `fallbackUsed`와 `fallbackReason`이 약속한 값으로 들어감
- 원본 provider 에러 메시지가 응답에 노출되지 않음

프롬프트는 공통 schema와 기능별 prompt builder로 나눈다.

지금 MVP 기준:

```text
공통 llmEnhancement schema
+ target별 prompt builder
```

이유:

- 응답 모양은 프론트가 쉽게 처리하도록 하나로 유지한다.
- `Design Simulation`, `Pre-Deployment Check`, Terraform 오류 설명, `Architecture Draft`는 설명해야 할 근거가 다르다.
- 하나의 공통 프롬프트만 쓰면 기능별로 중요한 말이 빠질 수 있다.
- 기능별 schema까지 전부 나누면 MVP 프론트와 테스트가 너무 커진다.

이번에 하지 않는 것:

- 모든 기능이 하나의 공통 프롬프트를 공유하는 구조
- 기능별 schema와 prompt builder를 전부 따로 두는 구조

나중에 고도화할 때는 기능별 상세 필드가 생기는 대상부터 schema를 분리한다.

예:

```text
Pre-Deployment Check findingExplanations 추가
→ preDeploymentCheck 전용 schema 추가 검토

Design Simulation requestFlowNarrative 추가
→ designSimulation 전용 detail schema 추가 검토
```

LLM prompt에는 MVP에서 한국어 설명 기준을 넣는다.

프롬프트에는 쉬운 한국어로 설명하라는 기준을 넣는다.

기준:

- 사용자가 바로 이해할 수 있는 짧은 문장
- 어려운 클라우드 용어는 필요할 때만 사용
- 전문 용어를 쓸 때는 앞뒤 문맥으로 뜻을 알 수 있게 설명
- 영어 응답이나 한영 혼합 응답이어도 언어만으로는 fallback 처리하지 않음
- 언어가 섞였더라도 JSON schema, 길이, 금지 문장 검증을 통과하면 그대로 표시

이번에 하지 않는 것:

- 사용자의 입력 언어를 따라가는 다국어 응답
- 영어로 받은 뒤 프론트에서 번역하는 구조
- 영어 또는 한영 혼합 응답이라는 이유만으로 fallback 처리하는 구조
- 한국어로 바꾸기 위해 OpenAI에 재요청하는 구조

나중에 고도화할 때는 사용자 설정이나 브라우저 언어를 기준으로 설명 언어를 고르는 방식을 검토한다.

또는 한국어 품질이 중요해졌을 때 아래 방식을 다시 검토한다.

- 언어 불일치 감지 후 fallback
- 언어 불일치 감지 후 1회 재요청
- 한국어 품질 검사 기준 추가

OpenAI에는 원본 전체를 넘기지 않는다.

대신 LLM 보강용 summary payload를 따로 만든다.

목표:

- 토큰 낭비 줄이기
- 불필요한 정보 노출 줄이기
- 테스트하기 쉬운 입력 구조 만들기
- LLM이 설명에 필요한 근거만 보게 하기

기능별 summary payload 후보:

| 대상 | OpenAI에 넘길 최소 정보 |
| --- | --- |
| `Design Simulation` | target, Resource 요약, requestFlow 요약, bottlenecks 요약, failureScenarios 요약, costPressure 요약 |
| `Pre-Deployment Check` | finding id/category/severity/resourceId, checklist status/title, suggestion action/targetResourceId/expectedImpact |
| Terraform 오류 설명 | stage, sanitized rawMessage, relatedResourceId |
| `Architecture Draft` | selectedScenario, assumptions, warnings, Resource id/type 요약 |

전달하지 않을 것:

- 원본 요청/응답 전체
- 불필요한 metadata
- 저장 정보
- 내부 식별자
- 민감할 수 있는 값

Terraform 오류의 `rawMessage`는 그대로 OpenAI에 넘기지 않는다.

먼저 masking/sanitize한 뒤 summary payload에 넣는다.

masking 후보:

- AWS account id
- access key 형태 문자열
- ARN 일부
- 긴 토큰처럼 보이는 문자열
- 로컬 파일 경로
- 환경변수 값처럼 보이는 문자열

LLM 보강 실패는 API 실패로 보지 않는다.

기준:

```text
rule 기반 분석 성공 + LLM 보강 실패
→ HTTP 성공 응답
→ llmEnhancement.fallbackUsed = true

rule 기반 분석 자체 실패
→ API 실패
```

이유:

- LLM은 핵심 판단이 아니라 보강 역할이다.
- API key 문제나 timeout 때문에 기본 분석 흐름이 깨지면 안 된다.
- 시연 중에도 fallback 결과가 안정적으로 보여야 한다.

지금 MVP에서는 AI API 호출마다 LLM 보강을 기본으로 시도한다.

요청 body에 `includeLlmEnhancement` 같은 on/off 옵션은 두지 않는다.

LLM 보강은 rule 분석이 끝난 뒤 같은 API 요청 안에서 실행한다.

응답 흐름:

```text
API 요청
→ rule 기반 분석
→ LLM 보강용 summary payload 생성
→ OpenAI 호출
→ 검증과 부분 fallback
→ rule 결과 + llmEnhancement 함께 응답
```

프론트는 rule API와 LLM API를 따로 호출하지 않는다.

프론트 loading은 기존 API 실행 버튼 loading 하나로 처리한다.

결과 영역 안에 별도 `AI 설명 생성 중` 상태를 만들지 않는다.

이번에 하지 않는 것:

- rule 분석 결과를 먼저 응답하고 LLM을 나중에 별도 API로 요청
- 프론트가 rule API와 LLM API를 따로 호출
- 백그라운드 작업으로 LLM 보강만 나중에 갱신
- polling으로 LLM 보강 결과 조회
- 결과 영역 안에 `AI 설명 생성 중` 상태 따로 추가
- LLM 전용 loading state 추가

나중에 LLM 응답 시간이 제품 사용성을 많이 해치면 별도 LLM 보강 API나 streaming 표시를 다시 검토한다.

이유:

- 프론트 호출 흐름을 단순하게 유지한다.
- 사용자는 같은 API에서 항상 보강 설명을 기대할 수 있다.
- LLM이 느리거나 실패하는 문제는 10초 timeout과 fallback으로 처리한다.

timeout 기준:

```text
LLM provider가 10초 안에 유효한 응답을 주지 않음
→ HTTP 성공 응답
→ llmEnhancement.fallbackUsed = true
→ llmEnhancement.fallbackReason = "timeout"
```

OpenAI 호출 실패 시 MVP에서는 재시도하지 않는다.

기준:

```text
OpenAI 1회 호출 실패
→ 바로 fallback
```

재시도하지 않는 대상:

- timeout
- rate limit
- provider error
- auth error
- invalid request
- invalid response

이유:

- 재시도를 넣으면 화면 응답이 늦어진다.
- 10초 timeout 뒤 재시도까지 하면 시연 흐름이 끊길 수 있다.
- LLM은 핵심 판단이 아니라 설명 보강이므로 실패해도 기본 설명으로 충분히 이어갈 수 있어야 한다.

이번에 하지 않는 것:

- timeout 또는 rate limit만 1회 재시도
- 모든 OpenAI 실패를 1회 재시도
- exponential backoff
- retry queue

나중에 실제 사용량이 늘어나고 응답 안정성이 더 중요해지면 rate limit과 일시적 provider error만 짧게 재시도하는 방식을 검토한다.

API key가 없으면 provider를 호출하지 않고 바로 fallback한다.

```text
OPENAI_API_KEY 없음
→ provider 호출 생략
→ llmEnhancement.fallbackUsed = true
→ llmEnhancement.fallbackReason = "missing_api_key"
```

LLM 응답 검증은 너무 약하게 두지 않는다.

OpenAI 응답은 자연어 덩어리로 받지 않는다.

반드시 `llmEnhancement` JSON 형식으로 요청한다.

JSON 파싱과 공통 응답 envelope 검증은 반드시 통과해야 한다.

그다음 `summary`, `highlights`, `nextActions`는 필드별로 검증하고, 이상한 필드만 rule 기본값으로 대체할 수 있다.

```text
JSON 파싱 실패
→ 전체 fallback

공통 응답 envelope 검증 실패
→ 전체 fallback

일부 필드 검증 실패
→ 정상 필드는 사용
→ 이상한 필드만 rule 기본값으로 대체
→ llmEnhancement.fallbackUsed = true
→ llmEnhancement.fallbackReason = "invalid_response"
```

부분 fallback 기준:

- `summary`가 비어 있거나 너무 길면 rule 기본 summary로 대체
- `highlights` 중 빈 문자열이나 너무 긴 항목은 제거
- `highlights`가 전부 제거되면 rule 기본 highlights로 대체
- `nextActions` 중 빈 문자열이나 너무 긴 항목은 제거
- `nextActions`가 전부 제거되면 rule 기본 nextActions로 대체
- 금지 문장이 들어간 필드는 해당 필드를 rule 기본값으로 대체

rule 기본값은 서버의 target별 fallback builder가 만든다.

fallback builder는 기존 rule 결과만 근거로 기본 문구를 만든다.

사용하지 않는 입력:

- 원본 사용자 요청
- 원본 API 요청 전체
- target 이름만 보고 만든 고정 문구

target별 fallback builder:

- `Design Simulation` fallback builder
- `Pre-Deployment Check` fallback builder
- Terraform 오류 설명 fallback builder
- `Architecture Draft` fallback builder

프론트는 빈 값을 보고 임시 문구를 채우지 않는다.

프론트는 서버가 준 `llmEnhancement`를 표시하고, `fallbackUsed: true`이면 작은 `기본 설명 사용` 표시만 붙인다.

이번에 하지 않는 것:

- 프론트에서 빈 값일 때 임시 문구 채우기
- LLM 공통 모듈에 generic 기본 문구 하나만 두기
- 모든 target에 같은 fallback 문구 사용
- 원본 사용자 요청까지 다시 섞어서 fallback 문구 만들기
- target 이름만 보고 고정 문구 만들기

이유:

- target마다 사용자가 궁금해하는 내용이 다르다.
- `Design Simulation`의 기본 설명과 Terraform 오류 설명의 기본 설명은 같은 문구로 처리하기 어렵다.
- 프론트가 fallback 문구를 만들면 API 응답 계약이 흐려진다.
- 원본 사용자 요청을 다시 섞으면 민감 정보나 프롬프트 주입 위험이 커진다.
- target 이름만 쓰면 사용자가 실제로 받은 분석 결과와 설명이 멀어진다.

그릴링 종료 후 구현 계획은 `docs/gg/007_LLM보강구현계획_gg.md`로 분리한다.

GitHub issue #51에는 구현 단위와 완료 조건만 남기고, 세부 설계와 보류 항목은 gg 문서를 기준으로 본다.

전체 fallback 기준:

- JSON 파싱 실패
- `llmEnhancement` 객체 자체가 없음
- `target`이 요청한 target과 다름
- 필드별 대체 뒤에도 사용자에게 보여줄 수 있는 설명이 남지 않음

지금 MVP 기준:

```text
summary
→ 1~300자

highlights
→ 최대 5개
→ 각 120자 이하
→ 빈 문자열 제거

nextActions
→ 최대 5개
→ 각 120자 이하
→ 빈 문자열 제거
```

최소 금지 문장:

- 배포 가능 보장
- 비용 없음 보장
- 보안 안전 보장

이런 문장이 들어오면 LLM 응답을 그대로 쓰지 않고 fallback한다.

나중에는 금지 문장 목록과 검증 규칙을 더 늘릴 수 있다.

나중에 LLM 비용이나 응답 속도가 문제가 되면 API별 on/off 옵션을 다시 검토한다.

나중에는 공통 필드는 유지하고, API별 특화 필드를 추가한다.

예:

```ts
llmEnhancement: {
  target: "design_simulation",
  summary: string,
  highlights: string[],
  nextActions: string[],
  fallbackUsed: boolean,
  fallbackReason?: string,
  detail?: {
    requestFlowNarrative?: string,
    failureScenarioNarrative?: string,
    costRiskNarrative?: string
  }
}
```

즉 기본 방향은 아래 2층 구조다.

```text
1층: llmEnhancement
= 모든 AI API 공통 요약

2층: targetSpecificDetail
= API별 특화 설명
```

기능별 고도화 후보:

| 대상 | 나중에 추가할 수 있는 특화 설명 |
| --- | --- |
| `Pre-Deployment Check` | finding별 설명, 심각도 이유, 무시 가능 여부, 수정 우선순위 |
| `Design Simulation` | 요청 흐름 설명, 병목 이유, 실패 시나리오 설명, 비용 압박 원인, 트래픽 증가 시 변화 |
| Terraform 오류 설명 | 에러 원인, 관련 Terraform Resource, 관련 AWS 권한, 수정 예시 코드, 재시도 전 checklist |
| `Architecture Draft` | 추천 이유, 대안 구조 비교, 비용형/보안형/성능형 설명 |

예를 들면 `Pre-Deployment Check`는 나중에 이렇게 확장할 수 있다.

```ts
{
  findings: [...],
  llmEnhancement: {
    target: "pre_deployment_check",
    summary: "보안 위험이 큽니다.",
    highlights: ["SSH 전체 오픈", "RDS Public"],
    nextActions: ["SSH 제한", "RDS Private 이동"],
    fallbackUsed: false
  },
  findingExplanations: [
    {
      findingId: "finding-001",
      whyItMatters: "외부 공격 위험이 커집니다.",
      priorityReason: "배포 전 반드시 수정해야 합니다."
    }
  ]
}
```

지금 MVP에서는 OpenAI 전용 구현으로 시작한다.

예:

- `openAiClient`
- `createOpenAiEnhancement`
- `OPENAI_API_KEY`
- `OPENAI_MODEL`

이름과 설정은 OpenAI 기준으로 써도 된다.

OpenAI 호출은 `fetch`로 직접 호출하지 않고 공식 SDK를 추가해서 구현한다.

이유:

- 나중에 streaming 응답을 붙일 수 있다.
- 나중에 tool calling / function calling을 붙일 수 있다.
- structured output 기반 JSON 응답을 더 안정적으로 받을 수 있다.
- timeout, rate limit, invalid request 같은 OpenAI 에러 타입을 나눠 fallback 처리할 수 있다.

따라서 이번 작업에서는 `apps/api`에 OpenAI SDK runtime dependency를 추가한다.

모델명은 `OPENAI_MODEL` 환경변수로 바꿀 수 있게 한다.

`OPENAI_MODEL`이 없으면 코드의 기본 모델을 사용한다.

나중에는 provider를 바꿀 수 있게 추상 계층을 만든다.

예:

- OpenAI
- Gemini
- Claude

그때는 아래처럼 generic 이름으로 바꿀 수 있다.

```text
openAiClient
→ llmProvider

createOpenAiEnhancement
→ createLlmEnhancement
```

지금 미루는 이유:

- 여러 provider를 추상화하는 것보다 OpenAI로 실제 LLM 보강 흐름을 먼저 검증하는 것이 중요하다.
- provider 연결보다 안정적인 응답 구조와 fallback 규칙이 먼저다.
- 발표 중 API key 문제나 timeout 때문에 흐름이 깨지면 안 된다.

## 15. 지원 리소스 확장

- EC2
- RDS
- S3

- CloudFront

나중에 추가 후보:

- ALB
- NAT Gateway
- IAM
- Lambda
- DynamoDB
- ECR
- ECS
- EKS

추가할 때는 한 번에 하나씩 해야 한다.

- 보안 규칙
- 설정 검증
