  - AI Dock을 Architecture 전용으로 둘지, Deployment 판단 보조까지 맡길지는 두 레벨이 안정된 뒤 결정한다.

- Reverse Engineering 화면 재구성
  - 이번 작업은 일반 Workspace의 Resource/Terraform과 Deployment 위자드 분리가 범위다.

- 새 DB migration
  - 현재 Deployment, Plan 승인, Direct Apply, Git/CI/CD handoff 계약을 재사용한다.

- 비용·보안 분석 규칙과 Deployment Wizard 승인·실행 흐름 변경
  - 지원 전 Resource는 계속 확정 분석과 배포 대상에서 제외한다.

## 2026-07-19 - AWS 가져오기 권한 복구에서 보류하는 항목

- 임의 이름 AWS Role까지 가져오기 권한 복구 대상으로 확대
  - 이번에는 현재 검증 가능한 `SketchCatchTerraformExecutionRole`과 `SketchCatchTerraformExecutionRole-...` 이름만 지원한다.
  - 임의 이름을 허용하려면 연결별 Role allowlist, SketchCatch 운영 IAM Policy, 잘못된 Role 접근 방지 검증을 함께 설계해야 한다.

- 가져오기 권한 복구 때 전체 배포 권한까지 최신화
  - 이번에는 기존 배포 Policy를 그대로 두고, 부족한 읽기 Policy만 별도로 추가한다.
  - 전체 연결 업그레이드가 필요하면 변경 권한을 따로 설명하고 별도 승인을 받는 흐름으로 다룬다.

- Reverse Engineering 전용 Role 분리
  - 읽기와 배포 권한을 완전히 나누는 장점은 있지만 Role 두 개를 저장하고 이전하는 변경이 필요하다.
  - Workspace·팀 공유 AWS 연결과 권한 모델을 설계할 때 다시 검토한다.

- Role 접속이나 Stack 상태만으로 가져오기 권한 갱신을 성공 처리
  - 실제 읽기 권한이 없어도 성공으로 보일 수 있어 채택하지 않는다.
  - 운영상 빠른 상태 확인이 따로 필요하면 사용자 성공 판정이 아닌 내부 참고 정보로만 사용한다.

- Resource Explorer와 모든 확장 reader까지 성공해야만 가져오기를 허용
  - 계정에 Resource Explorer가 설정되지 않았거나 확장 정보 하나가 제한된 경우에도 주요 Resource 가져오기를 막아 채택하지 않는다.
  - 주요 reader가 성공하면 가져오기를 허용하고 확장 reader 제한은 누락 가능성과 설정 방법을 별도로 안내한다.

- STS와 기본 inventory 조회만으로 가져오기 준비 완료 처리
  - 실제 주요 Resource reader의 AccessDenied를 발견하지 못해 채택하지 않는다.

- SketchCatch 서버가 원래 연결 Stack이나 배포 Policy를 자동 갱신
  - 기존 companion 가져오기 권한 Stack은 환경설정의 명시적 승인 뒤 서버 갱신을 허용하지만, 그 권한을 원래 Stack까지 넓히지 않는다.
  - 전체 배포 권한 최신화가 필요하면 별도 범위와 승인을 다시 설계한다.

- 기존 연결 Role에 자기 companion Policy를 변경할 IAM 권한을 추가
  - 별도 Role은 필요 없지만 연결 Role이 자신의 권한을 바꿀 수 있어 채택하지 않는다.
  - CloudFormation 전용 service Role에 companion Policy 수명주기만 맡긴다.

- CloudFormation service Role을 Policy Stack 안에서 함께 생성
  - service Role은 대상 Stack보다 먼저 존재해야 하고 연결된 뒤 제거할 수 없어 같은 Stack 수명주기로 안전하게 정리하기 어렵다.
  - 연결별 Manager Stack에서 service Role을 먼저 만들고 Policy Stack을 나중에 생성한다.

- AWS 계정 전체가 공유하는 Import Access Manager Stack과 service Role
  - AWS Resource 수는 줄지만 한 연결의 관리 권한과 실패가 다른 연결에 번질 수 있고 마지막 연결까지 cleanup을 미뤄야 해 채택하지 않는다.

- companion Stack 갱신 전에 CloudFormation Change Set을 따로 보여주는 2단계 승인
  - AWS가 계산한 diff를 더 자세히 볼 수 있지만 사용자가 같은 읽기 권한 변경을 두 번 승인해야 해 이번 기본 흐름에는 사용하지 않는다.
  - Settings API는 쉬운 변경 설명과 한 번의 명시적 승인 뒤 즉시 갱신한다.

- 서버가 Change Set만 만들고 AWS Console에서 실행하는 혼합 방식
  - 수동 Template 업로드는 없앨 수 있지만 다시 Console로 이동해야 해 선택하지 않는다.
  - 원래 Stack이나 배포 권한을 변경하는 더 큰 작업에는 별도 승인 방식으로 다시 검토할 수 있다.

- 수동 Role에 IAM Policy를 사용자가 직접 복사·설치하는 방식을 기본 흐름으로 사용
  - Role을 잘못 선택하거나 Policy 버전과 정리 책임이 흩어질 수 있어 채택하지 않는다.
  - 모든 기존 Role은 Role을 소유하지 않는 별도 가져오기 권한 Stack으로 지원한다.

- 수동 Role을 새 SketchCatch 관리 Role로 자동 교체
  - 기존 배포와 다른 시스템이 사용하는 Role을 바꿀 수 있어 채택하지 않는다.
  - 연결 Role 마이그레이션은 별도 승인·재검증 설계가 생기기 전까지 진행하지 않는다.

- 수동 Role 연결 삭제 때 읽기 Policy Stack을 남겨 두기
  - 사용하지 않는 권한과 Stack이 남기 때문에 채택하지 않는다.
  - AWS 정리가 실패하면 metadata를 버리지 않고 `AWS 권한 정리 필요` 재시도 기록을 유지한다.

- SketchCatch 서버가 Policy Stack을 자동 삭제
  - AWS Console에서 사용자가 권한 제거를 직접 승인하는 경계와 충돌해 채택하지 않는다.

- 기존 연결 Stack을 최신 전체 Template으로 갱신해 읽기 Policy를 추가하는 방식
  - 실제 배포된 기존 Template을 저장하지 않아 배포 Policy와 다른 Stack Resource까지 바뀔 수 있으므로 채택하지 않는다.

- 기존 Stack Template을 읽는 선행 권한을 먼저 추가한 뒤 부분 병합하는 방식
  - 권한 추가를 위해 또 다른 선행 권한 절차가 필요하고 복구 흐름이 길어져 채택하지 않는다.

- 사용자의 `삭제 완료` 확인만 믿고 connection metadata를 삭제하는 방식
  - AWS에 읽기 Policy나 Stack이 남아 있어도 알 수 없어 채택하지 않는다.

- Stack 삭제 알림을 위한 Lambda 또는 custom callback Resource를 고객 AWS에 추가하는 방식
  - 삭제 확인은 더 정확하지만 읽기 Policy 외의 실행 Resource와 운영 경로가 생겨 이번 범위에서는 채택하지 않는다.

## 2026-07-19 - 자동 정리 범위에서 채택하지 않은 제한안

- 위치·기존 영역 크기·연결선만 바꾸고 Resource 크기와 표시 Group을 고정하는 방식
  - 사용자가 선택한 자동 정리 범위보다 좁아 채택하지 않는다.

- Resource 크기까지 바꾸되 표시 Group 생성·병합은 금지하는 방식
  - 복잡한 보드를 주제별로 정리하려는 요구를 충족하지 못해 채택하지 않는다.

## 2026-07-19 - 표시 프레임에서 채택하지 않은 소속 방식

- 표시 프레임에 `presentationOrigin` 전용 metadata를 추가하는 방식
  - 출처는 명확하지만 공통 타입·저장 API·복원 schema를 넓혀야 해 이번에는 기존 Design Group 형식과 ID prefix를 사용한다.

- 모든 Design Group을 Board Auto Arrange 소유로 취급
  - 사용자가 직접 만든 Group도 자동 병합·삭제될 수 있어 채택하지 않는다.

- 표시 프레임마다 둘러싼 Resource ID 목록을 저장하는 방식
  - 단순한 배경 장식에 별도 소속 생명주기가 생기고, Resource 이동 때 숨은 소속을 계속 맞춰야 해 채택하지 않는다.

- 기존 `parentAreaNodeId`를 표시 프레임 소속에도 재사용하는 방식
  - VPC·AZ·Subnet의 실제 containment와 화면 장식을 구분할 수 없어 채택하지 않는다.

- 사용자가 직접 관리하는 실제 Group 기능
  - 향후 별도 기능으로 검토할 수 있지만, 이번 Board Auto Arrange의 장식용 표시 프레임에는 포함하지 않는다.

- Resource 이동에 맞춰 표시 프레임을 실시간으로 조절하는 방식
  - 표시 프레임에 Resource 소속이 없는데도 주변 geometry로 숨은 소속을 계속 추론해야 해 채택하지 않는다.

- Board를 편집할 때마다 표시 프레임을 제거하거나 다시 생성하는 방식
  - 사용자가 요청하지 않은 화면 변화와 History가 반복될 수 있어 채택하지 않는다.

## 2026-07-19 - 자동 정리 결과를 품질 측정값으로 제한하는 방식

- Compiler 점수가 가장 높은 변경안 하나만 사용자에게 보여주는 방식
  - 화면 선호를 내부 점수 하나로 대신하게 되므로 채택하지 않는다. 서로 다른 안전한 정리안은 사용자가 직접 비교한다.

- 원본을 변경 후보와 함께 점수 경쟁에 넣는 방식
  - 안전한 시각 변경이 있어도 원본이 선택돼 결과가 숨겨질 수 있으므로 사용하지 않는다. 원본은 비교 기준으로만 둔다.

- 모바일에서 후보 카드를 세로로 모두 나열하는 방식
  - 후보 설명은 읽기 쉽지만 보드 비교 전에 화면을 너무 많이 차지해 사용하지 않는다.

- 모바일에서 후보를 전체 화면 carousel로만 넘기는 방식
  - 현재 후보는 크게 보이지만 앞선 후보와 원본을 기억해서 비교해야 하므로 사용하지 않는다.

- 측정한 배치 문제가 줄어든 후보만 사용자에게 제공하는 방식
  - 측정 규칙이 사용자의 화면 크기와 시각적 선호를 전부 대신할 수 없어 채택하지 않는다.

- 측정상 개선이 없거나 일부 항목이 나빠지면 적용 버튼을 막는 방식
  - 사용자가 원본과 결과를 직접 비교해 선택하는 권한을 제한하므로 채택하지 않는다.

- 내부 품질 점수로 자동 정리 성공 여부를 표현하는 방식
  - 숫자만으로 무엇이 달라졌는지 이해하기 어려워 사용하지 않는다. 실제 좋아진 점과 나빠진 점을 쉬운 문장으로 설명한다.
