# Architecture Board Compiler 설계

## 한 줄 결론

현재 Web의 좌표 중심 자동 배치 엔진을, 29개 사용 가능한 Template 사례를 근거로 Resource·관계·설정·소속·영역·좌표·크기·z-index·연결선까지 다시 구성할 수 있는 하나의 깊은 Module로 확장한다.

Compiler는 틀린 결과도 제안할 수 있다. 다만 현재 Board를 조용히 덮어쓰지 않고 변경 전후와 진단을 함께 반환하며, 실제 적용은 사용자가 한 번 승인한다.

## 설계 당시 기준선

현재 `apps/web/features/workspace/automatic-diagram-layout.ts`는 이미 다음 기능을 가진다.

- primary flow와 support lane 분리
- Area 자식 우선 배치와 부모 크기 계산
- 반복 AZ·Subnet 정렬
- 여섯 가지 배치 후보 생성과 최저 점수 후보 선택
- node·Area 겹침, 포함관계 위반, 역방향 edge, edge 교차, 빈 공간, canvas 비율 평가
- 같은 입력에 같은 결과를 내는 결정론적 실행

그러나 현재 Interface는 좌표 중심이며 다음 한계가 있다.

- AI `ArchitectureJson → DiagramJson` 변환 경로에서만 직접 사용한다.
- 이미 정해진 `parentAreaNodeId`와 Resource 구성을 주로 신뢰한다.
- 후보와 간격이 코드에 고정되어 있고 Template 사례를 지식으로 사용하지 않는다.
- 사용자가 실행하는 Board `자동 정리`와 Template 품질 검토의 공통 Module이 아니다.
- Resource 추가·삭제, 관계·설정 변경을 하나의 제안으로 설명하지 못한다.

Template gallery는 SketchCatch 작성 Template 6개와 Brainboard Template 24개로 구성된다. Brainboard 한 개는 capture 실패 상태이므로 Compiler가 실제로 사용할 수 있는 Board 사례는 총 29개다.

## 확정한 권한

Architecture Board Compiler는 다음 작업을 모두 제안할 수 있다.

- Resource 추가·삭제·교체
- 관계 추가·삭제·방향 변경
- Terraform parameter와 참조 변경
- VPC·AZ·Subnet·Security Group 등 소속 재판단
- presentation Area와 Group 생성·삭제
- node 위치·크기·정렬·z-index 변경
- edge handle·경로·label 변경
- 사용자 요구사항, 기존 배포 상태, Provider·Terraform 유효성과 충돌하는 결과 제안

이 권한은 **제안 생성 권한**이다. Compiler는 실제 Deployment를 실행하지 않으며, 결과가 유효하다고 보장하지 않는다. 현재 Board 적용에는 기존 `User-Accepted Change` 계약을 유지한다.

## Module과 Seam

외부 Seam은 한 함수로 유지한다.

```ts
function compileArchitectureBoard(
  input: ArchitectureBoardCompilationInput
): ArchitectureBoardCompilationProposal;
```

호출자가 알아야 할 입력은 현재 Architecture와 선택적인 Board 상태, 호출 출처뿐이다.

```ts
type ArchitectureBoardCompilationInput = {
  architecture: ArchitectureJson;
  currentDiagram?: DiagramJson;
  trigger: "ai-draft" | "board-auto-organize" | "reverse-engineering" | "template-review";
};
```

결과는 수정된 Architecture와 Board, 변경 내역, 품질, 진단, 근거를 함께 가진다.

```ts
type ArchitectureBoardCompilationProposal = {
  architecture: ArchitectureJson;
  diagram: DiagramJson;
  changes: readonly ArchitectureBoardCompilationChange[];
  diagnostics: readonly ArchitectureBoardCompilationDiagnostic[];
  quality: {
    before: ArchitectureBoardCompilationQuality;
    after: ArchitectureBoardCompilationQuality;
    compilationDistance: number;
  };
  provenance: {
    compilerVersion: string;
    candidateId: string;
    referenceTemplateIds: readonly string[];
  };
};
```

Template corpus, 후보 수, 점수 가중치, Provider별 role mapping, edge router는 Interface에 노출하지 않는다. 모두 Implementation 안에 둔다. Compiler는 순수 계산 결과를 반환하며 DB 저장이나 Board mutation을 직접 수행하지 않는다.

첫 구현 Seam은 기존 geometry·edge routing 호출자가 모여 있는 `apps/web/features/architecture-board-compiler`에 둔다. API나 worker가 같은 계산을 직접 호출해야 하는 실제 두 번째 Adapter가 생길 때만 shared package 추출을 검토한다.

## 내부 처리 순서

1. **입력 정규화**
   Resource identity, 관계, Terraform reference, 기존 geometry를 비교 가능한 graph로 바꾼다.

2. **유사 사례 검색**
   29개 사례에서 Resource role, containment, 반복 구조, flow가 가까운 Template을 찾는다.

3. **Architecture 후보 생성**
   Resource·관계·설정·소속의 유지, 변경, 추가, 삭제 후보를 만든다. 원본도 반드시 후보에 포함한다.

4. **presentation 구조 생성**
   Region, AZ, Group, Security Group scope 같은 시각 영역과 parent 구조 후보를 만든다.

5. **geometry 후보 생성**
   영역 깊이, 주 흐름, support lane, 반복 구조를 기준으로 위치·크기·z-index 후보를 만든다.

6. **edge routing**
   최종 node와 Area가 정해진 뒤 handle과 장애물 회피 경로를 계산한다.

7. **평가와 선택**
   시각 품질, 구조 일관성, 진단, Compilation Distance를 합산해 최저 비용 후보를 선택한다.

8. **제안 생성**
   변경 전후 diff, 경고, 사용한 사례와 Compiler version을 결과에 포함한다.

## 29개 Template에서 추출할 지식

절대 x/y 좌표를 외우지 않고 상대 규칙을 추출한다.

| 범주 | 추출 정보 |
| --- | --- |
| Resource | provider type, semantic role, 표시 크기, caption footprint |
| 포함관계 | parent chain, Area depth, padding, child density |
| 정렬 | 형제 순서, 행·열 중심선, 간격 분포, 반복 구조 |
| 흐름 | entry에서 compute·data로 이어지는 rank와 방향 |
| 보조 구조 | IAM, observability, delivery Resource의 lane과 연결 대상 거리 |
| layering | Area 깊이와 Resource 종류별 z-index 순서 |
| edge | 시작·끝 side, waypoint, 길이, 교차와 장애물 회피 |
| viewport | 전체 bounds, aspect ratio, 외부 여백, 공백률 |

Source Template은 그대로 보존하고, 위 특성을 정규화한 versioned knowledge artifact를 생성한다. Compiler는 매 실행마다 29개 원본 파일을 직접 순회하지 않고 이 artifact를 사용한다.

첫 버전은 별도 AI model 파인튜닝을 하지 않는다. 29개는 독립 모델을 학습시키기에는 작고 AWS 편향이 강하므로, 사례 검색과 데이터에서 계산한 분포를 결정론적 후보 생성·점수화에 사용한다. 이후 사례가 충분히 늘면 후보 생성 Adapter를 추가할 수 있지만 Compiler Interface는 바꾸지 않는다.

## 품질 점수와 Compilation Distance

최종 점수는 낮을수록 좋다.

```text
totalPenalty = visualPenalty
             + structuralPenalty
             + semanticDiagnosticPenalty
             + compilationDistance
```

`visualPenalty`는 기존 자동 배치 평가 항목을 재사용·확장한다.

- node·caption·Area 겹침
- parent 경계 위반
- edge가 node·caption·Area title을 통과한 길이
- edge 교차·역방향·전체 길이
- 반복 구조 정렬 오차
- support lane 침범
- canvas 면적·공백률·세로 비율

`semanticDiagnosticPenalty`는 잘못된 Terraform 참조, Provider 제약 위반, 명시 요구사항 충돌을 점수에 반영하지만 후보를 탈락시키는 hard gate로 사용하지 않는다.

`Compilation Distance`의 기본 비용 순서는 다음과 같다.

```text
위치 < 크기 < 시각적 소속 < 관계 < 설정 < Resource 추가 < Resource 삭제
```

따라서 모든 변경은 가능하지만, 단순히 교차와 겹침을 없애려고 Resource를 전부 삭제한 빈 Board가 최적해가 되지는 않는다. 정확한 가중치는 public Interface가 아니라 versioned Compiler policy로 관리한다.

## 적용 기능

| 기능 | Compiler가 붙는 지점 | 사용자에게 보이는 결과 |
| --- | --- | --- |
| AI Architecture Draft | AI graph를 Board preview로 만들기 직전 | 정리된 Board와 전체 변경 제안 |
| Board 자동 정리 | 사용자가 명시적으로 실행 | 현재 Board와 Compiler 결과 비교 후 한 번에 적용 |
| Reverse Engineering | provider graph를 Practice Architecture Draft로 만든 직후 | 읽기 쉬운 import Board와 진단 |
| Template 검토 | source fixture를 gallery에 게시하기 전 또는 수동 검토 시 | 원본을 보존한 compiled variant와 품질 report |

사용자가 node를 끌 때마다 전체 Compiler를 실행하지 않는다. 권한 범위와 실행 시점을 분리해, 명시적 실행이나 AI Draft 생성 시점에만 전체 제안을 만든다. 적용 직후에는 기존 Board history로 한 번에 되돌릴 수 있어야 한다.

Brainboard의 `source-exact` fixture는 학습 근거이자 원본 증거이므로 Compiler가 덮어쓰지 않는다. Template 개선 결과는 별도 compiled variant로 비교한 뒤 승인된 결과만 gallery definition에 반영한다.

## 기존 코드의 이동 방향

- `automatic-diagram-layout.ts`의 후보 생성과 품질 평가는 Compiler Implementation 내부로 이동한다.
- `automatic-diagram-layout-provider-mapping.ts`는 Provider Adapter 성격의 내부 mapping으로 유지한다.
- `workspace-ai-diagram-adapter.ts`는 Compiler를 호출하고 결과를 Board DTO로 넘기는 얇은 Adapter가 된다.
- Board 자동 정리, Reverse Engineering, Template review도 같은 Interface를 호출한다.
- 기존 세부 함수 테스트는 Compiler Interface를 통과하는 행동 테스트로 대체한다. 독립적인 obstacle router처럼 별도 Module로 가치가 있는 테스트만 유지한다.

## 검증 기준

### 데이터 검증

- gallery 30개, 사용 가능한 사례 29개, 실패 evidence 1개를 자동 확인한다.
- knowledge artifact를 다시 생성해도 같은 hash가 나와야 한다.
- node·Area·edge·viewport 특성 누락과 비정상 수치를 보고한다.

### Compiler 검증

- 같은 입력과 Compiler version은 완전히 같은 제안을 반환한다.
- 원본 후보를 항상 포함한다.
- Resource 전체 삭제 후보가 시각 점수만으로 선택되지 않는다.
- 모든 변경은 `changes`에 표현되고, 모든 충돌은 `diagnostics`에 표현된다.
- Compiler 호출만으로 현재 Board, DB, IaC Preview가 변경되지 않는다.

### 사례 기반 검증

- 29개 중 하나를 제외하고 나머지 28개로 knowledge artifact를 만든 뒤, 제외한 Template의 상대 배치 특성을 얼마나 복원하는지 leave-one-out으로 확인한다.
- `docs/diagram-layout-reference/failure` 사례가 기존 엔진보다 겹침·교차·영역 이탈·canvas 낭비를 줄이는지 비교한다.
- 기존 6개 authored Template과 23개 사용 가능한 Brainboard Template의 before/after screenshot과 score report를 함께 남긴다.
- 숫자 점수만 최적화하지 않도록 대표 결과를 사람이 pairwise review한다.

### 통합 검증

- AI Draft, Board 자동 정리, Reverse Engineering, Template review가 같은 Compiler version과 Interface를 사용한다.
- diff preview, 승인, 취소, undo가 한 transaction 단위로 동작한다.
- invalid proposal을 승인해도 Pre-Deployment Check와 Deployment Safety Gate는 우회되지 않는다.

## 구현 순서

1. 29개 Template의 정규화 extractor와 knowledge artifact 생성
2. 기존 자동 배치 엔진을 새 Compiler Interface 뒤로 이동하고 회귀 결과 고정
3. Architecture·presentation 후보와 Compilation Distance 추가
4. AI Draft와 Board 자동 정리에 연결
5. Reverse Engineering과 Template review에 연결
6. leave-one-out, failure fixture, screenshot 비교로 점수와 후보 조정
7. canonical `docs/data-models.md`와 shared type을 실제 구현 계약에 맞춰 갱신

## 실현 가능성과 한계

기존 후보 생성, edge routing, 품질 평가, 29개 Board fixture가 이미 있어 구현 가능성은 높다. 그러나 29개 사례는 AWS 중심이고 일부 Brainboard 원본의 시각 품질도 완벽하지 않다. 따라서 처음부터 모든 Architecture를 예쁘게 만든다고 보장할 수 없으며, 의미 변경 후보는 특히 틀릴 수 있다.

성공 기준은 한 번에 정답을 만드는 것이 아니다. 같은 입력에 재현 가능한 제안과 근거를 만들고, 기존 엔진보다 측정 가능한 시각 품질을 높이며, 잘못된 변경도 숨기지 않고 diff와 diagnostics로 드러내는 것이다.

## 구현 상태 (2026-07-15)

- `architecture-board-compiler/v3`가 29개 usable Template와 1개 unavailable evidence로 만든 versioned knowledge artifact를 점수와 근거 사례 선택에 사용한다. 가장 가까운 사례의 sibling/vertical gap은 제한된 spacing profile로 geometry 후보군에 추가된다. 단, baseline보다 겹침·경계 이탈·edge 관통/교차·역방향·support lane 침범을 늘리는 profile 후보는 선택하지 않는다.
- 원본, presentation, semantic 후보를 같은 비용 모델로 비교한다. semantic 후보는 containment edge와 Terraform reference를 함께 보며, 모든 edge add/remove와 viewport 변경을 `changes`에 남긴다.
- semantic operation이 있어도 `original` 후보는 요청 전 source graph를 보존한다. 요청 graph는 `requested-original`에서 시작하며, 명시 operation이 있으면 source original을 선택해 요청을 무음 취소하지 않는다. AI Draft처럼 같은 node ID/type으로 config·관계 label만 달라진 입력도 다시 materialize한다.
- `source-exact` Template는 원본을 보존한다. 명시적 자동 정리는 별도 `catalog-normalized` compiled variant를 제안하고, 잠긴 node·사용자 변수는 유지한다.
- AI Draft, Board 자동 정리, Reverse Engineering, Template 검토가 같은 Compiler Interface와 version을 호출한다. Reverse Engineering은 scan finding·분석 제외·provider error를 external diagnostic signal로 함께 전달한다. AI와 Repository 시작은 변경·진단·점수 요약을 보여 준 뒤 `Board에 적용`을 누를 때만 저장한다.
- Template 검토 결과는 source fingerprint와 함께 별도 approval registry에 보관한다. 사람이 승인해 gallery/start에 반영할 수 있는 것은 presentation·geometry·edge-routing만이며, Resource·관계·설정·containment가 섞인 proposal은 Board-only variant로 승인할 수 없다. stale source나 범위를 벗어난 approval은 authored source를 사용한다.
- knowledge artifact와 evidence report는 각각 `pnpm architecture-board-knowledge:check`, `pnpm architecture-board-evidence:check`로 검증한다. report는 source/compiled Diagram SHA-256 fingerprint를 남기고, `compiler-evidence-review.json`은 실제 Board 전후 WebP와 사람 pairwise 판정을 위한 8개 pending 대기열만 만든다. 개발 전용 `/dev/compiler-evidence?templateId=<id>&stage=before|after`는 이 대기열을 실제 Board renderer로 열어 캡처할 수 있게 하며, pending을 승인이나 시각 품질 보장으로 해석하지 않는다.
- 전체 feature의 AI Chat 검증과 최종 aggregate 검증은 별도 001/통합 기록에서 관리한다.
