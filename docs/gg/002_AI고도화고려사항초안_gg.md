# AI 고도화 고려사항

이 문서는 지금 당장 만들지 않는 기능을 잊지 않기 위한 문서다.

쉽게 말하면:

> "이번 MVP에서는 못 하지만, 나중에 하면 좋은 것들"을 적어두는 곳이다.

## 1. 기록 원칙

- 지금 만들 것은 `001_AI파트1차제공범위초안_gg.md`에 적는다.
- 나중에 만들 것은 이 문서에 적는다.
- 지금 안 한다고 영원히 버리는 것은 아니다.
- 일정 때문에 뒤로 미룬 이유도 같이 적는다.
- 실제 AWS 비용, 보안, 배포와 연결되는 기능은 항상 안전장치를 같이 생각한다.

## 2. 제공 단계

| 단계 | 의미 |
| --- | --- |
| 1차 | 자연어 설계, 비용/보안 점검, Terraform 설명, 기본 시뮬레이션 |
| 1.5차 | 설계 버전 비교, 수정 제안 비교, AI 분석 요약 저장 |
| 2차 | AWS 연결, Plan/Apply/Destroy 오류 설명 |
| 2.5차 | 실제 인프라와 설계 차이 확인, 비용 알림 |
| 3차 이후 | 조직 정책, 승인 흐름, 템플릿 마켓, Kubernetes, 멀티클라우드 |

## 3. 자연어 설계 고도화

1차에서는 대표 시나리오만 지원한다.

예:

- 정적 웹사이트
- API 서버
- DB 포함 백엔드

나중에는 더 많은 조건을 받을 수 있다.

예:

- region
- runtime
- database 종류
- traffic 수치
- 예산 금액
- 보안 수준
- 가용성 수준

지금 미루는 이유:

- 조건을 많이 받을수록 타입과 UI가 커진다.
- sw Terraform 변환과도 더 많이 맞춰야 한다.
- 먼저 작은 입력이 안정적으로 돌아가야 한다.

## 4. AI 수정 제안 고도화

1차에서는 AI가 "이렇게 고치면 좋다" 정도만 제안한다.

나중에는 사용자가 이렇게 말할 수 있게 한다.

```text
비용 줄여줘
보안 강화해줘
트래픽 더 버티게 바꿔줘
```

AI는 바로 적용하지 않는다.

먼저 바뀔 내용을 보여준다.

```text
추가될 리소스
삭제될 리소스
변경될 리소스
예상 비용 변화
위험 변화
```

사용자가 확인해야 적용한다.

지금 미루는 이유:

- 자동 수정은 위험하다.
- 보드, Terraform, 비용 분석이 모두 같은 변경을 이해해야 한다.

## 5. 운영 조건 구조 반영

이번에는 운영 조건을 화면에서 고르게 만들었고, 일부 안전한 설정만 `ArchitectureJson.config`에 반영했다.

이번에 반영한 것:

- 낮은 예산 + RDS 조합이면 비용 warning을 보여준다.
- 보안 우선순위가 높으면 S3 public access 차단 설정을 넣는다.
- 보안 우선순위가 높으면 RDS public 접근 차단 설정을 넣는다.
- 보안 우선순위가 높으면 Security Group 기본 ingress를 비워 둔다.

하지만 리소스 개수나 큰 구조까지 자동으로 바꾸는 일은 아직 하지 않는다.

운영 조건:

- 예산
- 트래픽
- 보안 우선순위

이 값들은 설명만 바꾸면 부족하다.

나중에는 `ArchitectureJson` 안의 리소스 목록, 연결선, 설정값이 더 크게 바뀔 수 있어야 한다.

예:

```text
예산 낮게
→ 작은 EC2, 작은 RDS, 비싼 리소스 제외

예산 보통
→ 조금 더 큰 인스턴스나 안정성 옵션 고려

트래픽 작음
→ 단일 EC2, 단순 구조

트래픽 보통
→ ALB, Auto Scaling 같은 확장 구조 후보

보안 기본
→ 최소 보안 설정

보안 높음
→ private subnet, DB public 접근 차단, 보안 그룹 더 제한
```

아직 미루는 것:

- ALB 자동 추가
- Auto Scaling 자동 추가
- NAT Gateway 자동 추가
- Multi-AZ RDS 자동 추가
- private subnet/public subnet 정교한 분리
- 인스턴스 크기 자동 변경

아직 미루는 이유:

- 구조가 바뀌면 jh 보드, sw Terraform 생성, ck 배포 흐름까지 같이 맞춰야 한다.
- 리소스가 늘어나면 비용 계산과 보안 규칙도 같이 늘어난다.
- 지금은 공통 지원 리소스 안에서 안전한 config만 먼저 반영하는 것이 더 안전하다.

## 6. 비용/트래픽 시뮬레이션 고도화

1차에서는 아주 기본적인 추정만 한다.

예:

- RDS는 비용이 커질 수 있음
- 작은 EC2는 병목이 될 수 있음
- CloudFront는 트래픽 비용이 생길 수 있음

나중에는 사용자가 값을 바꾸면 다시 계산하게 한다.

예:

```text
월 방문자 1만 명
월 방문자 10만 명
예산 50달러
예산 100달러
```

지금 미루는 이유:

- 실제 가격과 성능을 정확히 보장하기 어렵다.
- 먼저 "설계 검토용 추정"이라는 선을 지켜야 한다.

## 7. 설계 버전 관리

나중에는 설계 버전을 비교할 수 있어야 한다.

예:

```text
버전 1: 저비용 구조
버전 2: 보안 강화 구조
버전 3: 트래픽 대비 구조
```

비교할 것:

- 추가된 리소스
- 삭제된 리소스
- 바뀐 연결선
- 예상 비용 변화
- 위험 변화

지금 미루는 이유:

- 버전 저장 구조가 필요하다.
- 프로젝트 저장, 보드 저장, Terraform Preview 저장과 연결된다.

## 8. AI 결과 저장 기준

이번에는 gg AI가 DB에 직접 저장하지 않는 방향으로 정리했다.

지금 gg AI는 이렇게 동작한다.

```text
요청 받음
→ 계산
→ 응답 반환
```

나중에 저장할 수 있는 후보:

- AI 분석 결과 요약
- AI finding
- AI checklist
- AI 요청 로그
- 외부 AI 사용량
- 외부 AI 비용

하지만 저장 주체를 잘 정해야 한다.

기본 원칙:

```text
gg AI는 계산/설명 담당
저장은 그 결과를 실제로 쓰는 파트가 담당
```

예:

```text
ArchitectureJson 저장
→ jh 보드 또는 프로젝트 저장 흐름

Terraform 파일 저장
→ sw Terraform 또는 공통 asset 흐름

Plan/Apply 분석 결과 저장
→ ck 배포 이력 흐름

프로젝트 목록 AI 요약 저장
→ ys 플랫폼 흐름
```

지금 미루는 이유:

- AI 결과를 무조건 저장하면 DB schema가 커진다.
- 같은 `ArchitectureJson`으로 다시 계산할 수 있는 결과는 MVP에서 꼭 저장하지 않아도 된다.
- 저장 위치를 잘못 잡으면 gg AI가 다른 파트의 책임까지 가져가게 된다.

## 9. GitHub 링크 기반 초안

GitHub 링크만 보고 정답 인프라를 만드는 것은 어렵다.

같은 코드도 여러 방식으로 배포할 수 있다.

예:

- EC2
- ECS
- Lambda
- Kubernetes

그래서 1차에서는 GitHub 링크를 핵심으로 두지 않는다.

나중에는 README, Dockerfile, package metadata를 더 잘 읽어서 참고 자료로 쓸 수 있다.

지금 미루는 이유:

- 코드 분석보다 요구사항 해석이 더 중요하다.
- private repo는 권한 문제가 있다.
- monorepo는 서비스 구분이 어렵다.

## 10. LLM provider 고도화

1차에서는 fallback이 먼저다.

즉 OpenAI API가 없어도 최소 기능은 돌아가야 한다.

나중에는 provider를 바꿀 수 있게 만든다.

예:

- OpenAI
- Gemini
- Claude

지금 미루는 이유:

- provider 연결보다 안정적인 응답 구조가 먼저다.
- 발표 중 API key 문제나 timeout 때문에 흐름이 깨지면 안 된다.

## 11. 대화형 AI 보조

나중에는 사용자가 보드나 Terraform Preview를 보면서 질문할 수 있다.

예:

```text
왜 RDS가 들어갔어?
이 구조에서 비용이 많이 나오는 부분은 어디야?
보안을 올리려면 뭘 바꿔야 해?
```

지금 미루는 이유:

- 챗봇을 만들면 제품 중심이 흐려질 수 있다.
- 먼저 버튼 기반 작업 흐름이 안정적이어야 한다.

## 12. Terraform 보조 고도화

1차에서는 Terraform 코드를 설명한다.

나중에는 Terraform plan 결과도 더 잘 설명한다.

예:

- 생성될 리소스
- 수정될 리소스
- 삭제될 리소스
- 위험한 변경

지금 미루는 이유:

- Plan/Apply는 ck 배포 파트와 연결된다.
- Terraform 실행은 gg 책임이 아니다.

## 13. 보안 검증 고도화

1차에서는 쉬운 규칙부터 본다.

예:

- SSH 전체 공개
- RDS 공개 접근
- S3 public access
- 너무 넓은 권한

나중에는 더 전문적인 기준을 붙일 수 있다.

예:

- AWS Well-Architected
- CIS Benchmark
- IAM least privilege

지금 미루는 이유:

- 보안 기준은 잘못 말하면 위험하다.
- MVP에서는 명확한 규칙만 다룬다.
