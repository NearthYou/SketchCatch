# 000. 그릴링 결정 메모

## 이번에 정한 것

새 프로젝트에서 Reverse Engineering을 고르면, 첫 결과는 **새 프로젝트 보드 후보**로 만든다.

기존 프로젝트를 바로 덮어쓰지 않는다.

## 왜 이렇게 정했나

Reverse Engineering은 AWS에 이미 만들어져 있는 리소스를 읽어서 보드에 보여주는 기능이다.

그런데 사용자가 새 프로젝트를 만들면서 Reverse Engineering을 눌렀다면, 아직 이 프로젝트에는 기존 설계도가 없다.

그래서 첫 결과는 이렇게 보여주는 게 자연스럽다.

```text
AWS에서 읽어온 리소스
→ 새 프로젝트에서 확인할 수 있는 보드 미리보기
→ 사용자가 확인
→ 적용 또는 취소
```

## 중요한 기준

사용자가 확인하기 전까지는 기존 프로젝트를 바꾸지 않는다.

읽어온 리소스가 많아도 바로 저장하지 않는다.

먼저 미리보기로 보여주고, 사용자가 적용을 눌렀을 때만 보드에 반영한다.

## 팀원에게 설명할 말

gg 쪽 Reverse Engineering은 AWS 리소스를 읽어서 바로 저장하는 기능이 아니다.

먼저 보드 후보를 만들고, 사용자가 확인한 뒤 적용하는 흐름이다.

그래서 다른 파트는 Reverse 결과를 받을 때 바로 최종 설계도로 보지 말고, **검토 전 후보 상태**로 생각하면 된다.

## 이번 결정에서 제외한 것

기존 프로젝트에 Reverse 결과를 바로 합치는 기능은 이번 기준에서 제외한다.

그 기능은 나중에 고도화로 다룬다.

예를 들면 이런 기능이다.

- 기존 보드와 AWS 실제 상태 비교
- 기존 보드에 없는 리소스만 추가
- AWS에는 없고 보드에만 있는 리소스 표시
- 충돌이 있을 때 사용자에게 선택하게 하기
