# 001. 작업 계획

## 목표

0706 피드백 기준으로 gg 담당 기능의 방향을 다시 잡는다.

이번 문서는 구현자가 다음 작업을 이어서 할 때 헷갈리지 않도록 기준과 현재 반영 상태를 같이 정리하는 문서다.

## 전체 흐름

gg 담당 범위는 크게 두 가지다.

```text
1. Reverse Engineering
2. Template 사용 흐름
```

두 기능 모두 핵심은 같다.

```text
사용자가 직접 복잡한 인프라를 처음부터 만들지 않게 도와준다.
```

Reverse Engineering은 이미 AWS에 있는 것을 읽어와서 시작하게 해준다.

Template은 미리 준비된 구조를 골라서 시작하게 해준다.

## 마일스톤 1. Reverse 시작 흐름 정리

### 목표

새 프로젝트를 만들 때 Reverse Engineering을 선택할 수 있게 한다.

### 사용자가 보는 흐름

```text
새 프로젝트 만들기
→ AI 또는 Reverse Engineering을 크게 보여줌
→ 빈 보드는 작은 보조 선택지로 보여줌
→ Reverse Engineering 선택
```

### 중요한 규칙

Reverse Engineering을 선택했다고 바로 프로젝트를 만들지 않는다.

먼저 AWS를 읽고, 후보를 보여준다.

프로젝트는 사용자가 후보를 고르고 `적용`을 눌렀을 때 만든다.

### AWS Role이 없을 때

AWS Role 연결이 없으면 환경설정으로 보낸다.

Reverse 화면 안에서 Role 연결 UI를 새로 만들지 않는다.

### 완료 기준

- 새 프로젝트 시작 방식에 Reverse Engineering이 들어간다.
- 새 프로젝트 시작 화면에서 빈 보드는 메인 카드가 아니라 작은 보조 선택지로 둔다.
- AWS Role이 없으면 환경설정으로 이동하는 흐름이 정해진다.
- 스캔만 하고 프로젝트가 바로 생성되지 않는 기준이 지켜진다.

## 마일스톤 2. Reverse 후보 만들기

### 목표

AWS에 여러 인프라가 섞여 있어도 사용자가 고를 수 있는 후보를 만든다.

### 사용자가 보는 흐름

```text
기존 AWS 가져오기
→ AWS 리소스 읽기
→ 여러 보드 후보 표시
→ 사용자가 하나 선택
```

### 후보를 만드는 기준

후보는 하나만 만들지 않는다.

아래 기준을 보고 여러 가능성을 만든다.

```text
VPC 기준
region 기준
tag 기준
리소스 연결 관계 기준
```

예를 들면 이렇게 보여줄 수 있다.

```text
1번 후보: VPC 안에 EC2와 RDS가 있는 백엔드 구조
2번 후보: S3 중심 정적 웹사이트 구조
3번 후보: 테스트용 EC2 단독 구조
```

### 후보 표시 방식

후보는 글 목록보다 보드 미리보기를 우선한다.

사용자는 작은 보드 그림을 보고 맞는 후보를 고른다.

### 완료 기준

- 스캔 결과가 하나의 보드로 바로 고정되지 않는다.
- 여러 후보를 만들 수 있는 구조가 정해진다.
- 후보는 보드 미리보기 중심으로 보여준다.

## 마일스톤 3. Resource Explorer 제한 표시

### 목표

AWS를 다 읽지 못한 상황을 숨기지 않는다.

### 기본 방향

Resource Explorer를 쓸 수 있으면 넓게 찾는다.

Resource Explorer를 못 쓰면 주요 AWS API로 가능한 만큼 읽는다.

그리고 화면에 제한을 알려준다.

### 사용자에게 보여줄 말

아래처럼 쉽게 보여준다.

```text
전체 발견 범위가 제한됐습니다.
주요 리소스는 직접 조회했지만, 일부 서비스는 누락될 수 있습니다.
```

### 완료 기준

- Resource Explorer 실패가 조용히 무시되지 않는다.
- 사용자가 전체 스캔이 제한됐다는 사실을 알 수 있다.
- 못 읽은 서비스나 권한 부족이 결과 화면에 표시된다.

## 마일스톤 4. Template 보관함 열기

### 목표

보드에서 Template을 큰 모달로 열 수 있게 한다.

### 사용자가 보는 흐름

```text
보드 화면
→ Template 버튼
→ 큰 모달 열림
→ Template 검색 또는 선택
```

### 중요한 규칙

템플릿 메뉴에서 보는 목록과 보드에서 보는 목록은 같은 데이터를 바라본다.

다만 보여주는 방식만 다르다.

```text
템플릿 메뉴: 페이지
보드: 큰 모달
```

### 완료 기준

- 보드에서 Template을 큰 모달로 열 수 있다.
- Template 페이지와 보드 모달이 같은 Template 목록을 사용한다.
- 검색, 필터, 정렬, 태그를 붙일 수 있는 구조로 둔다.

## 마일스톤 5. Template 적용하기

### 목표

Template을 선택하면 보드가 해당 Template 구조로 바뀌게 한다.

### 사용자가 보는 흐름

```text
Template 선택
→ 현재 보드 자동 백업
→ Template 구조로 보드 덮어쓰기
```

### 중요한 규칙

확인 팝업은 띄우지 않는다.

항상 덮어쓴다.

대신 덮어쓰기 직전에 자동 백업을 남긴다.

### 완료 기준

- Template 선택 시 현재 보드에 섞어 넣지 않는다.
- Template 구조로 보드를 덮어쓴다.
- 덮어쓰기 직전 자동 백업을 남긴다.

## 현재 반영된 것

이번 브랜치에서 반영한 것은 아래와 같다.

- 새 프로젝트 시작 화면에서 `AI로 시작`, `Reverse Engineering으로 시작`, `빈 보드로 시작`을 고를 수 있다.
- `빈 보드로 시작`은 작은 보조 선택지로 둔다.
- 새 프로젝트에서 Reverse를 고르면 오른쪽 패널에서 기존 AWS를 읽을 수 있다.
- 검증된 AWS Role 연결이 없으면 환경설정으로 이동할 수 있게 안내한다.
- 새 프로젝트 Reverse는 스캔만으로 프로젝트를 바로 만들지 않는다.
- AWS를 먼저 읽고, 여러 보드 후보를 보여준다.
- 사용자가 후보를 고르고 적용하면 그때 프로젝트를 만든다.
- 적용한 후보는 새 프로젝트 draft와 Architecture Snapshot으로 저장한다.
- Template은 보드에서 큰 모달로 열 수 있다.
- Template을 선택하면 현재 보드를 Template 구조로 덮어쓴다.

## 더 좋아지면 할 것

아래는 현재 흐름을 더 보기 좋게 만드는 고도화 항목이다.

- Reverse 후보 미리보기 UI를 더 보기 좋게 만들기
- Resource Explorer 상태 확인을 더 명확하게 보여주기
- 자동 백업을 서버 기준으로 더 안전하게 남기기
- Template 저장 권한 정리

gg 문서는 `docs/gg` 아래에만 둔다.

다른 사람 문서는 건드리지 않는다.
