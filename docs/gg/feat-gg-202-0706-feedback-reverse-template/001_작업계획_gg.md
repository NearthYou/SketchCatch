# 001. 작업 계획

## 목표

0706 피드백을 기준으로 gg 담당 기능을 다시 정리한다.

이번 브랜치에서는 바로 큰 구현을 밀어붙이기보다, 팀원이 이해할 수 있는 기준을 먼저 잡는다.

## gg 담당 범위

gg가 신경 써야 하는 큰 흐름은 두 가지다.

```text
1. Reverse Engineering
2. Template
```

## 1. Reverse Engineering

### 제품에서 원하는 모습

사용자는 AWS Role을 한 번 연결한다.

그 다음부터는 버튼 한 번으로 기존 AWS 리소스를 읽어올 수 있어야 한다.

```text
AWS Role 연결
→ 새 프로젝트 만들기
→ Reverse Engineering 선택
→ 기존 AWS 가져오기
→ 보드 미리보기
→ 적용 또는 취소
```

### 이번에 정리할 것

- Reverse Engineering이 새 프로젝트 생성 흐름에 어떻게 붙는지 정리
- 기존 프로젝트를 바로 덮어쓰지 않는 기준 정리
- AWS에서 읽어온 결과를 보드 후보로 다루는 기준 정리
- 못 읽은 리소스나 권한 부족을 숨기지 않는 기준 정리

### 나중에 구현할 것

- 새 프로젝트 생성 모달에서 Reverse Engineering 선택
- 기본 버튼 이름을 `기존 AWS 가져오기`로 단순화
- 기본값은 전체 스캔
- 리전 선택과 리소스 필터는 고급 설정으로 접기
- 스캔 후 바로 저장하지 않고 미리보기 먼저 보여주기

## 2. Template

### 제품에서 원하는 모습

Template은 마켓처럼 사고파는 기능이 아니다.

팀 프로젝트에서는 일단 여러 설계 예시를 한곳에서 모아 보고, 빠르게 찾아서 적용하는 기능이 더 중요하다.

```text
Template 모아보기
→ 검색
→ 필터
→ 정렬
→ 태그
→ 보드에서 불러오기
```

### 이번에 정리할 것

- Template 공유 마켓은 뒤로 미루기
- Template 모아보기 기준 정리
- 검색, 필터, 정렬, 태그 기준 정리
- 보드 페이지와 Template 페이지가 같은 Template 데이터를 보게 하는 기준 정리

### 나중에 구현할 것

- Template 저장
- Template 검색
- Template 필터
- Template 정렬
- Template 태그
- 보드에서 Template 불러오기
