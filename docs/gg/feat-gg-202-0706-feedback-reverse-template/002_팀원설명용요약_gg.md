# 002. 팀원 설명용 요약

## 이 문서를 왜 만들었나

0706 피드백 이후 gg 담당 기능 방향이 조금 바뀌었다.

팀원에게 설명할 때는 아래처럼 말하면 된다.

```text
gg 쪽은 기존 AWS 가져오기와 Template 사용 흐름을 맡는다.
핵심은 사용자가 새 프로젝트를 더 쉽게 시작하게 만드는 것이다.
```

새 프로젝트 시작 방식은 세 가지로 보인다.

```text
AI로 시작
Reverse Engineering으로 시작
빈 보드로 시작
```

다만 `빈 보드로 시작`은 아주 작게 가져간다.

빈 보드는 프로젝트 이름을 넣고 비어 있는 보드를 여는 정도면 된다.

화면에서는 큰 카드가 아니라 `::after` 같은 보조 라벨 버튼처럼 둔다.

사용자는 누를 수 있지만, 핵심 선택지처럼 크게 보이지 않게 한다.

이번 방향의 중심은 `AI로 시작`과 `Reverse Engineering으로 시작`이다.

Reverse Engineering을 새 프로젝트 시작으로 옮기는 가장 큰 이유는 오른쪽 패널 토글을 줄이기 위해서다.

작업 화면 오른쪽에는 이미 Resource, Terraform, Issues, Deploy 같은 모드가 있다.

여기에 Reverse Engineering까지 계속 넣으면 버튼이 많아져서 화면이 복잡해진다.

그래서 팀원에게는 이렇게 설명하면 된다.

```text
Reverse Engineering은 오른쪽 패널에 계속 붙어 있는 기능이 아니라,
기존 AWS를 가져와 새 프로젝트를 시작하는 입구로 옮긴다.
그리고 기존 작업 화면 오른쪽 패널이 아니라 별도 전체 화면에서 진행한다.
```

## Reverse Engineering은 뭐 하는 기능인가

Reverse Engineering은 이미 AWS에 만들어져 있는 리소스를 읽어와서 보드 후보로 보여주는 기능이다.

쉽게 말하면 이거다.

```text
내 AWS에 뭐가 만들어져 있는지 읽는다.
그걸 SketchCatch 보드에서 볼 수 있는 구조로 바꾼다.
사용자가 맞는 구조를 고른다.
그때 프로젝트를 만든다.
```

## Reverse Engineering 흐름

최종 흐름은 아래처럼 생각하면 된다.

```text
새 프로젝트 만들기
→ Reverse Engineering 선택
→ Reverse Engineering 전용 전체 화면으로 이동
→ 기존 AWS 가져오기 버튼 클릭
→ AWS Role 없으면 환경설정으로 이동
→ AWS Role 있으면 기존 AWS 가져오기
→ 최대 3개의 보드 후보 보기
→ 맞는 후보 선택
→ 적용
→ 프로젝트 생성
→ Architecture Board로 이동
```

중요한 점은 **스캔했다고 바로 프로젝트를 만들지 않는다**는 것이다.

사용자가 후보를 고르고 적용해야 프로젝트가 생긴다.

Reverse Engineering 전용 화면에서 중간에 그만두고 싶으면 `작업 화면으로 돌아가기`가 아니라 `시작 방식 다시 선택`을 누른다.

이 버튼은 `/workspace`가 아니라 `/workspace/new`로 보낸다.

이유는 Reverse Engineering 전용 화면이 기존 작업 화면에서 잠깐 들어온 화면이 아니기 때문이다.

새 프로젝트를 만들기 전에 시작 방식을 고르다가 들어온 화면이라서, 돌아갈 곳도 새 프로젝트 시작 방식 선택 화면이다.

## Reverse Engineering 첫 화면은 어떻게 보이나

전용 전체 화면에 들어가면 설정부터 크게 보여주지 않는다.

가장 크게 보여야 하는 것은 아래 버튼이다.

```text
기존 AWS 가져오기
```

기본은 전체 스캔이다.

region이나 가져올 Resource 종류는 고급 설정으로 접어둔다.

사용자는 먼저 "가져오기"를 누르고, 필요할 때만 설정을 연다.

## 왜 후보가 여러 개인가

한 AWS 계정에는 여러 프로젝트 리소스가 섞여 있을 수 있다.

예를 들면 이런 식이다.

```text
실제 서비스용 VPC
테스트용 EC2
개인 S3 버킷
예전 실습용 RDS
```

이걸 아무 기준 없이 하나의 보드에 다 펼치면 너무 복잡하다.

그래서 가능한 구조 후보를 최대 3개까지 보여준다.

3개보다 많으면 고르기 어렵고, 1개만 보여주면 우리가 정답을 단정하는 꼴이 된다.

여기서 후보는 Resource 하나가 아니다.

후보는 VPC 하나도 아니다.

후보는 AWS에서 읽어온 전체 Resource를 어떻게 묶어볼지에 대한 해석이다.

쉽게 말하면 각 후보에는 전체 Resource가 다 들어간다.

다만 Resource끼리 어떤 묶음으로 보이는지가 후보마다 다르다.

예를 들면 이런 식이다.

```text
구조 후보 1:
- 쇼핑몰 백엔드
- 관리자 API
- 정적 웹사이트

구조 후보 2:
- 쇼핑몰 백엔드 + 관리자 API
- 정적 웹사이트

구조 후보 3:
- VPC 중심 묶음
- S3 정적 웹사이트 묶음
```

기본은 Resource를 빼고 보여주는 것이 아니다.

전체 Resource를 유지한 상태에서 묶는 방법을 다르게 보여주는 것이다.

사용자는 그중에서 “이 묶음이 우리 인프라를 제일 잘 설명한다” 싶은 것을 고르면 된다.

즉, Reverse Engineering은 전체 AWS를 읽기는 한다.

하지만 사용자에게 아무 묶음 없이 전체를 한 번에 펼쳐 보여주는 것이 목표는 아니다.

먼저 묶음 후보로 나눠서 보여주고, 사용자가 맞는 묶음을 고르게 한다.

묶음 후보를 만드는 기준은 이렇게 설명하면 된다.

```text
배포에 필요한 관계를 먼저 본다.
VPC/Subnet 같은 포함 관계를 본다.
ALB, EC2, RDS처럼 실제 연결 관계를 본다.
tag와 이름은 힌트로만 쓴다.
정답이 애매하면 여러 후보를 보여주고 사용자가 고르게 한다.
```

예를 들어 `service=shop` tag가 붙은 EC2가 있어도, EC2만 쇼핑몰 백엔드로 보면 안 된다.

그 EC2가 들어 있는 Subnet, VPC, 연결된 Security Group도 같이 봐야 한다.

그래야 나중에 Terraform import나 배포 흐름으로 이어질 때 필요한 Resource가 빠지지 않는다.

## Resource Explorer는 뭐라고 설명하면 되나

Resource Explorer는 AWS 안의 리소스를 넓게 찾는 검색 기능이다.

아주 쉽게 말하면 AWS 리소스 검색창이다.

우리 기능에서는 이렇게 쓴다.

```text
Resource Explorer로 넓게 찾는다.
중요한 리소스는 각 AWS API로 자세히 읽는다.
```

만약 Resource Explorer를 못 쓰면, 가능한 주요 API로만 읽는다.

대신 화면에 이렇게 알려줘야 한다.

```text
전체 발견 범위가 제한됐습니다.
일부 리소스는 누락될 수 있습니다.
```

## Template은 뭐 하는 기능인가

Template은 미리 만들어둔 인프라 구조를 보드에 바로 넣는 기능이다.

예시는 아래와 같다.

```text
S3 정적 웹사이트
EC2 API 서버
ALB + ASG + RDS 구조
```

사용자는 Template을 골라서 보드를 빠르게 시작할 수 있다.

## Template 흐름

보드에서는 Template을 큰 모달로 연다.

흐름은 아래와 같다.

```text
보드에서 Template 버튼 클릭
→ 큰 모달 열림
→ Template 선택
→ 현재 보드 자동 백업
→ Template 구조로 보드 덮어쓰기
```

확인 팝업은 띄우지 않는다.

대신 덮어쓰기 직전에 자동 백업을 남긴다.

## 팀원에게 꼭 말해야 할 부분

Reverse Engineering 결과는 바로 최종 프로젝트가 아니다.

먼저 보드 후보이고, 사용자가 적용해야 프로젝트가 된다.

Template은 현재 보드에 섞어 넣는 기능이 아니다.

Template을 선택하면 보드를 Template 구조로 덮어쓴다.

AWS Role 연결은 Reverse 화면에서 새로 처리하지 않는다.

AWS Role이 없으면 환경설정으로 보낸다.

## 이번에 안 하는 것

이번 문서 정리에서 아래는 깊게 정하지 않는다.

- Template 저장 권한
- Template 공유
- Template 마켓
- 기존 보드와 Reverse 결과 병합
- 여러 Reverse 후보를 합쳐서 하나로 만드는 기능
