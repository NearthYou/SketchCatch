# Reverse Engineering 구현 계획

이 문서는 Reverse Engineering을 어떤 순서로 만들지 정리한 문서다.

쉽게 말하면:

```text
기존 AWS 상태 읽기
→ SketchCatch가 이해하는 Resource 목록으로 바꾸기
→ 보드 후보로 보여주기
→ 비용/보안/가져오기 제안을 붙이기
→ 사용자가 확인하고 적용하기
```

## 1. 이번 브랜치의 목표

이번 브랜치는 `Provider Adapter` 기반 Reverse Engineering을 만든다.

사용자는 프로젝트 안에서 AWS scan을 시작할 수 있어야 한다.

scan은 AWS를 읽기만 한다.

```text
한다
→ AWS Resource 조회
→ scan 상태 저장
→ scan 로그 저장
→ Practice Architecture 후보 생성
→ finding 생성
→ import suggestion 생성

하지 않는다
→ AWS Resource 생성
→ AWS Resource 수정
→ AWS Resource 삭제
→ terraform import 실행
→ terraform apply 실행
→ GitHub PR 자동 생성
```

## 2. 중요한 용어

```text
Reverse Engineering Scan
→ AWS를 읽은 작업 기록

Discovered Resource
→ AWS에서 발견한 Resource

Reverse Engineering Draft
→ scan 결과로 만든 보드 후보

Unknown Resource
→ AWS에서 발견했지만 아직 SketchCatch가 정식 타입으로 못 바꾼 Resource

Architecture Snapshot
→ 사용자가 적용해서 저장한 최종 보드 상태
```

## 3. 마일스톤 1: 공통 타입과 DB 저장 뼈대

먼저 저장할 모양을 고정한다.

만들 것:

- shared type
- DB schema
- migration
- status 값
- scan 결과 payload
- scan log payload
- scan 실패 payload
- Unknown Resource 표시 payload

완료 기준:

- `packages/types`에서 API와 프론트가 같은 타입을 볼 수 있음
- `apps/api` DB schema에 scan 기록을 저장할 수 있음
- migration이 schema와 맞음
- 민감한 값이 shared type에 들어가지 않음
- 관련 타입 테스트 또는 schema 테스트 통과

커밋 기준:

```text
Feat: Reverse Engineering 저장 계약 추가
```

## 4. 마일스톤 2: Provider Adapter와 AWS read-only 조회

AWS를 직접 읽는 경계를 만든다.

만들 것:

- Provider Adapter 인터페이스
- fixture adapter
- AWS read-only adapter
- AWS 응답을 `DiscoveredResource[]`로 바꾸는 코드
- 실패한 조회를 scan error summary로 바꾸는 코드
- 민감 정보 masking

완료 기준:

- 테스트에서는 실제 AWS를 부르지 않음
- fake AWS gateway로 성공/부분 실패를 검증함
- VPC, Subnet, Internet Gateway, Route Table, Security Group, EC2, RDS, S3 조회 흐름이 있음
- 조회 실패를 `없음`으로 착각하지 않음
- 로그와 에러에 secret이 남지 않음

커밋 기준:

```text
Feat: AWS read-only Provider Adapter 추가
```

## 5. 마일스톤 3: scan 작업 실행, 상태, 로그, 취소

scan을 한 번의 API 요청으로 끝내지 않고 작업으로 관리한다.

만들 것:

- scan 시작
- scan 상태 조회
- scan 로그 조회
- scan 취소
- scan 삭제
- `queued`, `running`, `completed`, `failed`, `cancelled` 상태 처리

완료 기준:

- scan 시작 시 `scanId`를 받을 수 있음
- 실행 중 로그를 볼 수 있음
- 취소 요청 후 가능한 지점에서 `cancelled`가 됨
- 삭제된 scan은 상세 조회할 수 없음
- 삭제해도 사용자가 적용한 최종 보드는 지우지 않음

커밋 기준:

```text
Feat: Reverse Engineering scan 작업 API 추가
```

## 6. 마일스톤 4: Reverse Engineering Draft 생성

AWS에서 읽은 Resource를 보드 후보로 바꾼다.

만들 것:

- `DiscoveredResource[]`를 `ArchitectureJson` 후보로 변환
- VPC, Subnet 같은 포함 관계 표시
- Unknown Resource 카드 생성
- 현재 보드와 비교할 수 있는 데이터
- 중복 판단 기준

완료 기준:

- `providerResourceId`가 같으면 같은 Resource로 본다
- 이름만 같으면 같은 Resource로 단정하지 않는다
- Unknown Resource는 보드에 흐리게 표시할 수 있는 데이터가 있음
- 기존 보드를 바로 덮어쓰지 않음
- 사용자가 적용하기 전까지 후보 상태로 남음

커밋 기준:

```text
Feat: Reverse Engineering Draft 생성 추가
```

## 7. 마일스톤 5: finding과 import suggestion

scan 결과에 비용/보안/가져오기 정보를 붙인다.

만들 것:

- 비용 finding
- 보안 finding
- 분석 제외 항목
- import suggestion
- Terraform block draft
- Git/CI/CD handoff-ready metadata

완료 기준:

- 지원 Resource는 finding을 계산함
- Unknown Resource는 분석 제외로 표시함
- Unknown Resource에는 import command를 만들지 않음
- 지원 Resource는 import command와 Terraform block draft를 제공함
- High risk가 있어도 Reverse Engineering 적용 자체를 막지 않음

커밋 기준:

```text
Feat: Reverse Engineering 분석 결과 추가
```

## 8. 마일스톤 6: API 연결과 검증

route와 검증을 붙인다.

만들 것:

- Zod schema
- Fastify route
- project 접근 확인
- response DTO 정리
- route 테스트

완료 기준:

- 잘못된 요청은 `400`으로 막음
- 없는 프로젝트나 권한 없는 프로젝트는 막음
- 응답 타입이 shared type과 맞음
- API 테스트 통과
- typecheck 통과

커밋 기준:

```text
Feat: Reverse Engineering API 연결
```

## 9. 마일스톤 7: 마무리 문서와 공유 스크립트

구현이 끝나면 팀원에게 설명할 문서를 만든다.

만들 것:

- 구현한 기능 학습 문서
- feat 커밋 설명 스크립트
- 팀원 연결 지점 설명
- PR 본문

주의:

- 프론트 관련 feat 커밋은 설명 스크립트에서 제외한다.
- 백엔드, 타입, DB, Provider Adapter, 분석 흐름 위주로 설명한다.

커밋 기준:

```text
Docs: Reverse Engineering 구현 설명 정리
```

## 10. 검증 계획

작업 중 자주 돌릴 것:

```text
pnpm --filter @sketchcatch/types typecheck
pnpm --filter @sketchcatch/api typecheck
pnpm --filter @sketchcatch/api exec tsx --test <관련 테스트 파일>
```

마지막에 돌릴 것:

```text
pnpm harness:check
pnpm lint
pnpm typecheck
pnpm build
```

실제 AWS Resource 생성, 수정, 삭제는 하지 않는다.

## 11. 구현 전에 기억할 것

```text
AI가 알아서 바꾸는 기능이 아니다.
AWS를 읽고 후보를 만든다.
사용자가 확인해야 보드에 반영된다.
```

그리고 이 브랜치는 제품 기능을 작게 보이게 만들기 위한 브랜치가 아니다.

처음 구현 단위가 있더라도 최종 제품 방향은 유지한다.
