# 리소스 관계 쉽게 보기

이 문서는 사람이 읽기 편한 설명 문서다.

정확한 구현 규칙은 아래 명세서를 기준으로 한다.

```text
003_리소스간관계정리_gg.md
```

이 문서는 그 명세서를 이해하기 쉽게 풀어쓴 버전이다.

## 1. 왜 리소스 관계가 중요할까

Reverse Engineering은 기존 AWS에 있는 리소스를 읽어와서 보드에 보여주는 기능이다.

그런데 AWS에는 리소스가 정말 많다.

예를 들면 이런 것들이 있다.

```text
VPC
Subnet
EC2
RDS
S3
Security Group
Route Table
Internet Gateway
IAM Role
CloudWatch Log Group
KMS Key
```

이걸 전부 그냥 아이콘으로 뿌리면 사용자는 이렇게 느낀다.

```text
뭐가 중요한 거지?
서버는 어디 있지?
DB는 어디 있지?
왜 선이 이렇게 많지?
이게 한 서비스야, 여러 서비스야?
```

그래서 리소스끼리 어떤 관계인지 정리해야 한다.

관계를 정리해야 보드가 사람이 볼 수 있는 구조가 된다.

## 2. 관계는 크게 3개만 먼저 기억하면 된다

우리 내부 규칙에서는 관계를 우선 세 가지로 본다.

```text
contains
connects_to
depends_on
```

이름이 어려워 보이지만 뜻은 간단하다.

| 관계 | 쉬운 뜻 | 예시 |
| --- | --- | --- |
| `contains` | 안에 들어감 | VPC 안에 Subnet |
| `connects_to` | 연결되어 있음, 사용함 | EC2가 Security Group 사용 |
| `depends_on` | 먼저 필요함, 참조함 | Route Table이 Internet Gateway 참조 |

## 3. contains는 박스 안에 넣는 관계다

`contains`는 제일 중요하다.

뜻은 이거다.

```text
큰 리소스 안에 작은 리소스가 들어간다.
```

예:

```text
VPC 안에 Subnet이 있다.
Subnet 안에 EC2가 있다.
Subnet 안에 RDS가 있다.
```

보드로 보면 이런 느낌이다.

```text
VPC
┌─────────────────────────────┐
│ Subnet                      │
│ ┌─────────────────────────┐ │
│ │ EC2        RDS          │ │
│ └─────────────────────────┘ │
└─────────────────────────────┘
```

이 관계는 화면 배치에 직접 영향을 준다.

즉:

```text
contains 관계가 있으면
자식 리소스를 부모 컨테이너 안에 넣어야 한다.
```

## 4. connects_to는 선으로 연결하는 관계다

`connects_to`는 안에 들어간다는 뜻이 아니다.

뜻은 이거다.

```text
서로 연결되어 있다.
또는 한 리소스가 다른 리소스를 사용한다.
```

예:

```text
EC2가 Security Group을 사용한다.
RDS가 Security Group을 사용한다.
Internet Gateway가 VPC에 붙어 있다.
CloudFront가 S3를 origin으로 사용한다.
```

중요한 차이가 있다.

```text
EC2는 Subnet 안에 들어간다.
→ contains

EC2는 Security Group을 사용한다.
→ connects_to
```

Security Group은 EC2를 담는 박스가 아니다.

Security Group은 EC2의 출입 규칙이다.

그래서 EC2를 Security Group 안에 넣으면 틀린 그림이 된다.

## 5. depends_on은 기본 화면에서 숨기는 관계다

`depends_on`은 사용자가 구조를 이해하기 위한 관계라기보다, 시스템이 처리 순서를 알기 위한 관계에 가깝다.

뜻은 이거다.

```text
이 리소스는 저 리소스를 먼저 필요로 한다.
```

예:

```text
Subnet을 만들려면 VPC가 먼저 있어야 한다.
Route Table route가 Internet Gateway를 참조한다.
Lambda Permission은 Lambda가 있어야 의미가 있다.
```

이 관계를 기본 보드에 전부 선으로 그리면 너무 복잡해진다.

그래서 기본 화면에서는 숨기고, 아래 기능에서 쓰는 게 좋다.

```text
Terraform 생성 순서
Terraform import 제안
배포 전 점검
오류 설명
상세 패널
```

## 6. 어떤 관계가 화면 배치에 영향을 줄까

정리하면 이렇다.

| 관계 | 화면 배치에 영향 있음? | 기본 화면에 보이나? |
| --- | --- | --- |
| `contains` | 있음 | 강하게 보임 |
| `connects_to` | 거의 없음 | 중요한 것만 선으로 보임 |
| `depends_on` | 없음 | 기본 화면에서는 숨김 |

제일 중요한 규칙:

```text
자동 배치와 자동 크기 조절은 contains만 기준으로 한다.
```

다시 말하면:

```text
connects_to 때문에 리소스를 컨테이너 안에 넣으면 안 된다.
depends_on 때문에 리소스를 컨테이너 안에 넣으면 안 된다.
```

## 7. 컨테이너 리소스가 뭘까

컨테이너 리소스는 다른 리소스를 안에 담는 큰 박스다.

현재 기준으로 제일 중요한 컨테이너는 이 둘이다.

```text
VPC
Subnet
```

VPC는 Subnet을 담는다.

Subnet은 EC2나 RDS 같은 리소스를 담는다.

```text
VPC
└─ Subnet
   ├─ EC2
   └─ RDS
```

나중에 컨테이너처럼 다룰 수 있는 후보도 있다.

```text
Security Group
DB Subnet Group
Availability Zone
Region
```

하지만 지금 당장 핵심은 VPC와 Subnet이다.

## 8. 자식 리소스가 뭘까

자식 리소스는 컨테이너 안에 들어가는 리소스다.

예:

```text
Subnet은 VPC의 자식이다.
EC2는 Subnet의 자식이다.
RDS는 Subnet 또는 DB Subnet Group 쪽 자식으로 볼 수 있다.
```

자식인지 판단할 때는 `contains`를 먼저 본다.

예:

```text
EC2 contains Subnet
```

뜻은:

```text
EC2가 Subnet 안에 있다.
```

조금 헷갈릴 수 있는데, 관계 배열에서는 보통 현재 리소스가 부모를 가리킨다.

그래서 EC2 입장에서 보면:

```text
나는 이 Subnet 안에 있어.
```

라고 말하는 형태다.

## 9. 파라미터는 관계를 찾는 힌트다

파라미터는 리소스의 설정값이다.

예를 들어 EC2에는 이런 값들이 있을 수 있다.

```json
{
  "instanceType": "t3.micro",
  "subnetId": "subnet-1234",
  "securityGroupIds": ["sg-1234"]
}
```

여기서 의미가 다 다르다.

| 파라미터 | 뜻 | 관계 |
| --- | --- | --- |
| `instanceType` | 서버 크기 | 관계 아님 |
| `subnetId` | 어느 Subnet 안에 있는지 | contains |
| `securityGroupIds` | 어떤 Security Group을 쓰는지 | connects_to |

즉:

```text
파라미터 안에 ID가 있다고 무조건 선을 그리면 안 된다.
이 ID가 포함 관계인지, 연결 관계인지, 그냥 설정값인지 구분해야 한다.
```

## 10. 자동 배치는 뭘 해야 할까

자동 배치는 리소스 위치를 보기 좋게 정리하는 기능이다.

기준은 `contains`다.

예를 들어 이런 관계가 있으면:

```text
Subnet contains VPC
EC2 contains Subnet
RDS contains Subnet
```

보드는 이렇게 정리되어야 한다.

```text
VPC
┌────────────────────────────┐
│ Subnet                     │
│ ┌────────────────────────┐ │
│ │ EC2        RDS         │ │
│ └────────────────────────┘ │
└────────────────────────────┘
```

자동 배치가 해야 하는 일:

```text
자식 리소스를 부모 컨테이너 안에 넣기
자식끼리 겹치지 않게 놓기
컨테이너 안쪽에 적당한 여백 두기
VPC 안에 Subnet, Subnet 안에 EC2/RDS처럼 중첩 구조 유지하기
```

자동 배치가 하면 안 되는 일:

```text
Security Group을 EC2의 부모 박스로 만들기
depends_on 선을 보고 리소스를 안에 넣기
사용자가 직접 옮긴 위치를 마음대로 덮어쓰기
```

## 11. 사용자가 직접 옮긴 위치는 어떻게 할까

기본 원칙은 간단하다.

```text
사용자가 직접 옮긴 위치는 유지한다.
```

자동 배치가 덮어써도 되는 경우는 제한적이다.

덮어써도 되는 경우:

```text
처음 Reverse Engineering 미리보기를 만들 때
처음 AI Draft를 만들 때
사용자가 자동 정리 버튼을 직접 눌렀을 때
새 리소스를 추가해서 빈 공간에 넣어야 할 때
```

덮어쓰면 안 되는 경우:

```text
사용자가 이미 직접 옮긴 기존 리소스
사용자가 잠근 리소스
보드를 그냥 다시 불러오는 상황
```

즉 자동 배치는 사용자를 도와야지, 사용자가 정리한 보드를 망가뜨리면 안 된다.

## 12. 자동 크기 조절은 뭘 해야 할까

자동 크기 조절은 컨테이너 크기를 자식에 맞춰 바꾸는 기능이다.

예를 들어 Subnet 안에 EC2가 하나 있을 때:

```text
Subnet
┌────────────┐
│ EC2        │
└────────────┘
```

EC2가 하나 더 추가되면:

```text
Subnet
┌──────────────────────┐
│ EC2        EC2       │
└──────────────────────┘
```

Subnet이 커져야 한다.

그리고 Subnet이 커졌으면, 그 Subnet을 담고 있는 VPC도 필요하면 커져야 한다.

```text
VPC
┌──────────────────────────────┐
│ Subnet                       │
│ ┌──────────────────────────┐ │
│ │ EC2          EC2         │ │
│ └──────────────────────────┘ │
└──────────────────────────────┘
```

핵심 규칙:

```text
컨테이너는 자기 자식을 모두 감싸야 한다.
중첩 컨테이너는 안쪽부터 계산한다.
작아질 수는 있지만 최소 크기보다 작아지면 안 된다.
사용자가 직접 키운 컨테이너는 마음대로 줄이면 안 된다.
```

## 13. RDS는 왜 조심해야 할까

RDS는 EC2처럼 Subnet 하나에 딱 들어간다고 보기 어렵다.

보통 RDS는 여러 Subnet을 묶어서 사용한다.

예:

```text
RDS subnetIds = [subnet-a, subnet-b]
```

이때 잘못 만들면 이런 문제가 생긴다.

```text
subnet-a에도 RDS가 보임
subnet-b에도 RDS가 보임
결과적으로 RDS가 두 개처럼 보임
```

그러면 안 된다.

규칙:

```text
RDS 노드는 하나만 만든다.
대표 위치 하나에만 배치한다.
다른 Subnet과의 관계는 상세 패널에 보여준다.
```

나중에 `DB Subnet Group`을 정식 리소스로 만들면 더 정확하게 표현할 수 있다.

## 14. 왜 지금 다이어그램이 정신없어 보일까

보통 이유는 이거다.

```text
AWS에서 읽은 리소스를 너무 많이 보여준다.
Route Table, Security Group, IAM, CloudWatch 같은 보조 리소스가 섞인다.
contains, connects_to, depends_on이 같은 선처럼 보인다.
컨테이너 안에 들어갈 리소스와 선으로만 연결할 리소스가 섞인다.
같은 유형의 Subnet이 이름만 다르고 구분이 안 된다.
```

즉 사용자가 못 보는 게 아니다.

화면이 너무 많은 정보를 한 번에 보여주는 것이다.

## 15. 기본 화면은 어떻게 가야 할까

기본 화면은 AWS 전체 목록이 아니다.

기본 화면은 사람이 구조를 이해하는 화면이어야 한다.

기본 화면에 강하게 보여줄 것:

```text
VPC
Subnet
EC2
RDS
S3
ALB
CloudFront
Lambda
중요 트래픽 관계
위험 finding이 있는 리소스
```

기본 화면에서 숨기거나 상세로 뺄 것:

```text
Route Table 상세
Route Table Association
Security Group rule 상세
IAM Role
IAM Policy
KMS Key
CloudWatch Log Group
CloudWatch Alarm
UNKNOWN 대량 리소스
```

## 16. 여러 인프라가 섞여 있으면 어떻게 구분할까

AWS 계정 하나에 서비스가 여러 개 있을 수 있다.

예:

```text
쇼핑몰 서비스
관리자 서비스
테스트 서비스
```

AWS가 이걸 자동으로 딱 나눠주지는 않는다.

하지만 리소스 관계를 보면 어느 정도 묶을 수 있다.

```text
리소스 = 점
관계 = 선
```

서로 연결된 리소스끼리 하나의 묶음 후보로 볼 수 있다.

잘 구분되는 경우:

```text
VPC가 다름
Subnet이 다름
Security Group이 다름
RDS가 다름
태그가 다름
이름 prefix가 다름
```

구분이 어려운 경우:

```text
같은 VPC를 같이 씀
같은 Subnet을 같이 씀
같은 Security Group을 같이 씀
공용 Route Table이나 NAT Gateway를 같이 씀
태그가 없음
이름 규칙도 없음
```

그래서 제품에서는 이렇게 가는 게 좋다.

```text
1. 관계로 1차 묶기
2. VPC/Subnet 기준으로 2차 묶기
3. 태그 기준으로 3차 묶기
4. 이름 prefix 기준으로 4차 묶기
5. 사용자가 직접 묶음 수정
```

## 17. 팀원에게 이렇게 설명하면 된다

짧게 말하면:

```text
리소스 관계는 전부 같은 선이 아니다.
contains는 박스 안에 넣는 관계다.
connects_to는 선으로 연결하는 관계다.
depends_on은 기본 화면에서 숨기고 분석에 쓰는 관계다.
자동 배치와 자동 크기 조절은 contains만 기준으로 한다.
```

조금 더 길게 말하면:

```text
Reverse Engineering은 AWS 리소스를 다 긁어오는 기능이지만,
보드에는 사람이 이해할 수 있는 구조만 강하게 보여줘야 한다.

그래서 VPC, Subnet 같은 컨테이너는 박스로 보여주고,
EC2, RDS 같은 자식 리소스는 그 안에 자동 배치한다.

Security Group, Route Table, IAM 같은 정보는 중요하지만
기본 화면에 전부 선으로 그리면 너무 복잡해진다.
그래서 상세 패널이나 별도 View로 빼는 게 맞다.
```

## 18. 내가 기억할 것

이것만 기억하면 된다.

```text
contains = 안에 넣기
connects_to = 선으로 연결하기
depends_on = 기본 화면에서는 숨기기
```

그리고 하나 더:

```text
보드는 AWS 원본 전체를 보여주는 곳이 아니다.
보드는 사람이 이해할 수 있게 정리해서 보여주는 곳이다.
```
