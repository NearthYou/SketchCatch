# 현재 피알 코드 이해 문서

이 문서는 현재 피알 코드를 이해하기 위한 문서다.

쉽게 말하면:

> 기존 에이에스더블유에스에 만들어져 있던 자원들을 읽어서, 스케치캐치 보드에서 볼 수 있는 설계 후보로 바꾸는 기능이다.

여기서 중요한 점은 이거다.

```text
읽기만 한다.
바로 바꾸지 않는다.
사용자가 적용 버튼을 눌러야 보드에 들어간다.
```

## 1. 사용자가 보는 흐름

사용자는 작업 화면 오른쪽 패널에서 기존 에이에스더블유에스 읽어오기를 쓴다.

화면에서 하는 일은 이 순서다.

```text
1. 프로젝트를 고른다.
2. 에이에스더블유에스 연결을 고른다.
3. 가져올 자원 종류를 고른다.
4. 스캔 시작을 누른다.
5. 결과를 미리 본다.
6. 새 보드로 열기 또는 현재 보드에 추가를 누른다.
```

여기서 스캔 시작을 눌러도 바로 현재 보드가 저장되지는 않는다.

먼저 화면에 미리보기만 뜬다.

## 2. 프론트에서 시작되는 곳

가장 먼저 볼 파일은 여기다.

```text
앱스 웹 피처스 워크스페이스 리버스엔지니어링패널
```

실제 파일:

```text
apps/web/features/workspace/ReverseEngineeringPanel.tsx
```

이 파일은 오른쪽 패널 전체를 관리한다.

하는 일은 크게 다섯 가지다.

```text
스캔 조건 관리
스캔 시작
스캔 결과 기다리기
스캔 기록 열기
보드에 적용
```

### 스캔 시작 버튼을 누르면

스캔 시작 버튼을 누르면 `runScan`이 실행된다.

이 함수가 하는 일은 이렇다.

```text
1. 이전 미리보기를 지운다.
2. 서버에 스캔 시작 요청을 보낸다.
3. 서버가 준 스캔 번호를 받는다.
4. 결과가 아직 없으면 계속 조회한다.
5. 결과가 오면 보드 미리보기를 만든다.
```

중요한 점:

```text
여기서는 현재 보드를 바로 저장하지 않는다.
미리보기만 띄운다.
```

### 적용 버튼을 누르면

사용자가 새 보드로 열기 또는 현재 보드에 추가를 누르면 `applyScanResult`가 실행된다.

이 함수가 하는 일은 이렇다.

```text
1. 스캔 결과를 보드 모양으로 바꾼다.
2. 보드에 리버스 엔지니어링 출처를 남긴다.
3. 화면의 보드를 바꾼다.
4. 아키텍처 스냅샷을 저장한다.
```

여기서 저장되는 출처는 이런 의미다.

```text
이 보드는 어떤 스캔에서 시작됐는가
이 보드는 어떤 설계 후보에서 시작됐는가
```

이 출처가 있어야 나중에 추적할 수 있다.

## 3. 프론트와 서버가 주고받는 곳

프론트에서 서버를 부르는 함수는 여기 있다.

```text
apps/web/features/workspace/api.ts
```

이 파일에는 여러 요청 함수가 같이 들어 있다.

리버스 엔지니어링에서 중요한 요청은 이런 것들이다.

```text
스캔 시작
스캔 상세 조회
스캔 로그 조회
스캔 취소
스캔 삭제
아키텍처 스냅샷 저장
```

쉽게 말하면 프론트는 서버에 이렇게 말한다.

```text
이 프로젝트에서
이 에이에스더블유에스 연결로
이 자원 종류들을
읽어와 줘
```

서버는 이렇게 답한다.

```text
스캔 번호를 먼저 줄게.
결과가 생기면 다시 조회해서 가져가.
```

## 4. 서버에서 요청을 받는 곳

서버 라우트는 여기다.

```text
apps/api/src/routes/reverse-engineering.ts
```

이 파일은 주소별로 요청을 받는다.

중요한 주소는 이렇다.

```text
스캔 시작
프로젝트의 스캔 목록 보기
스캔 하나 자세히 보기
스캔 로그 보기
스캔 취소하기
스캔 삭제하기
```

여기서 먼저 확인하는 것은 두 가지다.

```text
로그인한 사용자인가
요청 값이 맞는 모양인가
```

예를 들면 프로젝트 번호가 이상하면 여기서 막힌다.

이전에 화면에서 `잘못된 유유아이디` 오류가 났던 것도 이쪽 검증과 관련 있다.

## 5. 서버의 실제 일 처리

서버에서 실제 일을 처리하는 파일은 여기다.

```text
apps/api/src/reverse-engineering/reverse-engineering-service.ts
```

이 파일은 스캔 작업의 중심이다.

하는 일은 이렇다.

```text
1. 프로젝트가 내 것인지 확인한다.
2. 에이에스더블유에스 연결이 검증된 연결인지 확인한다.
3. 스캔 기록을 만든다.
4. 스캔 시작 로그를 남긴다.
5. 실제 에이에스더블유에스 읽기를 실행한다.
6. 성공하면 결과를 저장한다.
7. 실패하면 실패 이유를 저장한다.
```

중요한 점은 서버가 스캔을 바로 끝내지 않고, 먼저 `실행 중` 상태의 스캔 기록을 만든다는 것이다.

그래서 프론트는 스캔 번호를 먼저 받고, 결과가 나올 때까지 계속 물어본다.

## 6. 실제 에이에스더블유에스를 읽는 곳

실제로 에이에스더블유에스에 물어보는 파일은 여기다.

```text
apps/api/src/reverse-engineering/aws-reverse-engineering-gateway.ts
```

이 파일은 클라우드포메이션을 읽는 게 아니다.

직접 에이에스더블유에스 조회 기능을 호출한다.

현재 읽는 자원은 이렇다.

```text
브이피시
서브넷
인터넷 게이트웨이
라우트 테이블
보안 그룹
이시투
알디에스
에스쓰리
```

흐름은 이렇다.

```text
1. 검증된 에이에스더블유에스 연결을 가져온다.
2. 임시 권한을 만든다.
3. 자원 종류별 조회 요청을 보낸다.
4. 받은 결과를 내부 기록 모양으로 바꾼다.
```

중요한 점:

```text
만들기, 수정, 삭제는 하지 않는다.
읽기만 한다.
```

## 7. 왜 클라우드포메이션이 아닌가

기존 에이에스더블유에스 자원은 꼭 클라우드포메이션으로 만든 것이 아닐 수 있다.

예를 들면:

```text
콘솔에서 직접 만든 서버
테라폼으로 만든 네트워크
명령어로 만든 저장소
옛날에 사람이 수동으로 만든 보안 그룹
```

이런 자원은 클라우드포메이션 스택에 없을 수 있다.

그래서 우리 기능은 클라우드포메이션 스택을 읽는 방식이 아니라, 서비스별 조회 기능으로 직접 읽는다.

## 8. 읽은 결과를 보드 후보로 바꾸는 곳

에이에스더블유에스에서 읽은 원본을 스케치캐치가 이해하는 모양으로 바꾸는 파일은 여기다.

```text
apps/api/src/reverse-engineering/aws-provider-adapter.ts
```

이 파일은 번역기 역할이다.

예를 들면 이렇게 바꾼다.

```text
에이에스더블유에스 브이피시
→ 스케치캐치 브이피시 노드

에이에스더블유에스 서브넷
→ 스케치캐치 서브넷 노드

에이에스더블유에스 보안 그룹
→ 스케치캐치 보안 그룹 노드
```

또 관계도 만든다.

예를 들면:

```text
서브넷은 브이피시 안에 있다.
이시투는 서브넷 안에 있다.
보안 그룹은 이시투에 붙어 있다.
인터넷 게이트웨이는 브이피시에 붙어 있다.
```

이 관계가 보드의 선으로 이어진다.

## 9. 보드에 들어가는 데이터

서버가 최종으로 만드는 중요한 결과는 이거다.

```text
아키텍처 제이슨
```

쉽게 말하면 보드가 열 수 있는 설계도다.

안에는 크게 두 가지가 있다.

```text
노드
선
```

노드는 자원 하나를 뜻한다.

예:

```text
브이피시 노드
서브넷 노드
이시투 노드
에스쓰리 노드
```

선은 자원 사이의 관계를 뜻한다.

예:

```text
서브넷이 브이피시 안에 있다.
이시투가 서브넷에 있다.
보안 그룹이 이시투에 붙어 있다.
```

## 10. 스캔 결과 전체 모양

공통 타입은 여기 있다.

```text
packages/types/src/index.ts
```

리버스 엔지니어링 결과에는 이런 정보가 들어간다.

```text
스캔 정보
발견한 자원 목록
보드가 열 수 있는 설계도
수정 가능한 설계 후보
위험 찾기 결과
분석 제외 목록
테라폼 가져오기 제안
스캔 오류 목록
```

각각의 뜻은 이렇다.

```text
스캔 정보
언제, 어떤 프로젝트에서, 어떤 연결로 읽었는지

발견한 자원 목록
에이에스더블유에스에서 실제로 찾은 자원들

보드가 열 수 있는 설계도
화면에 미리보기로 띄울 보드 데이터

수정 가능한 설계 후보
사용자가 적용 전에 이름, 설명, 위치를 바꿀 수 있는 후보

위험 찾기 결과
열린 에스에스에이치 같은 위험 안내

분석 제외 목록
아직 우리 자원 타입으로 해석하지 못한 것들

테라폼 가져오기 제안
기존 자원을 테라폼 관리 대상으로 가져오기 위한 명령어 초안

스캔 오류 목록
특정 자원 종류를 읽다가 실패한 이유
```

## 11. 보드 비교와 적용

보드 비교와 적용은 여기서 한다.

```text
apps/web/features/workspace/reverse-engineering-board-application.ts
```

이 파일은 두 가지를 비교한다.

```text
현재 보드
에이에스더블유에스에서 읽어온 설계 후보
```

비교 결과는 이런 식으로 나뉜다.

```text
추가 후보
변경 후보
삭제 후보
중복 후보
확인 필요
```

가장 중요한 비교 기준은 실제 에이에스더블유에스 자원 번호다.

예:

```text
vpc로 시작하는 번호
subnet으로 시작하는 번호
i로 시작하는 서버 번호
sg로 시작하는 보안 그룹 번호
```

이 번호가 같으면 같은 자원으로 본다.

이 번호가 없으면 테라폼 이름표를 보조 기준으로 쓴다.

## 12. 새 보드로 열기와 현재 보드에 추가 차이

새 보드로 열기는 현재 보드를 스캔 결과 후보로 바꾼다.

쉽게 말하면:

```text
기존 화면을 새로 읽어온 설계도로 교체
```

현재 보드에 추가는 기존 보드를 유지하면서, 없는 자원만 붙인다.

쉽게 말하면:

```text
현재 보드는 그대로 두고
에이에스더블유에스에는 있는데 보드에는 없는 자원만 추가
```

삭제 후보가 있다고 해서 자동 삭제하지 않는다.

## 13. 스캔 기록 저장

스캔 기록 저장 테이블은 여기서 만들었다.

```text
apps/api/drizzle/0023_reverse_engineering_scans.sql
```

저장되는 것은 크게 두 가지다.

```text
스캔 기록
스캔 로그
```

스캔 기록에는 이런 것이 들어간다.

```text
어떤 프로젝트에서 실행했는지
어떤 에이에스더블유에스 연결을 썼는지
어떤 리전을 읽었는지
어떤 자원 종류를 읽었는지
성공했는지 실패했는지
결과 데이터
```

스캔 로그에는 이런 것이 들어간다.

```text
스캔 시작
스캔 완료
스캔 실패
```

## 14. 스캔 삭제의 의미

스캔 삭제는 에이에스더블유에스 자원을 삭제하는 것이 아니다.

스캔 삭제는 우리 데이터베이스에 저장된 스캔 기록을 숨기는 것이다.

즉:

```text
에이에스더블유에스 실제 자원 삭제 아님
보드에 이미 적용된 자원 삭제 아님
스캔 기록 삭제에 가까움
```

그래서 스캔을 삭제해도 이미 보드에 반영한 설계는 남아 있다.

대신 화면에는 이런 안내가 뜰 수 있다.

```text
이 보드는 리버스 엔지니어링 스캔에서 시작됐지만, 원본 스캔 기록은 삭제됐다.
```

## 15. 취소의 의미

취소는 실행 중인 스캔에만 의미가 있다.

취소 요청이 들어오면 서버는 스캔 상태를 취소로 바꾼다.

중요한 점:

```text
취소된 스캔은 나중에 완료 결과로 덮어쓰지 않게 막는다.
```

이 처리가 없으면 이런 문제가 생길 수 있다.

```text
사용자가 취소를 눌렀다.
그런데 뒤늦게 에이에스더블유에스 응답이 왔다.
서버가 완료로 저장해버렸다.
```

그래서 완료 저장 전에 한 번 더 취소 상태인지 확인한다.

## 16. 민감 정보 처리

리버스 엔지니어링은 에이에스더블유에스 계정을 다룬다.

그래서 화면과 로그에 계정 번호가 그대로 나가면 안 된다.

현재 코드는 계정 번호 같은 값을 이런 식으로 가린다.

```text
316875069960
→ 3168********
```

또 원본 권한 정보는 저장하지 않는다.

저장하지 않는 것:

```text
접근 키
비밀 키
세션 토큰
```

## 17. 팀원에게 설명할 때 한 문장

팀원이 "이 기능 뭐야?"라고 물으면 이렇게 말하면 된다.

```text
선택한 에이에스더블유에스 연결과 리전에서 기존 자원을 읽고, 그 결과를 보드에서 볼 수 있는 설계 후보로 보여주는 기능입니다. 바로 적용하지 않고 미리보기로 보여준 뒤, 사용자가 적용 버튼을 눌러야 보드와 스냅샷에 저장됩니다.
```

팀원이 "클라우드포메이션 읽는 거야?"라고 물으면 이렇게 말하면 된다.

```text
아니요. 클라우드포메이션 스택을 읽는 방식이 아니라, 에이에스더블유에스 서비스별 조회 기능으로 직접 읽습니다. 콘솔에서 직접 만든 자원이나 테라폼으로 만든 자원도 읽기 위해서입니다.
```

팀원이 "실제 에이에스더블유에스가 바뀌어?"라고 물으면 이렇게 말하면 된다.

```text
아니요. 이 기능은 읽기 전용입니다. 실제 자원 생성, 수정, 삭제, 테라폼 가져오기 실행은 하지 않습니다.
```

팀원이 "보드에는 언제 반영돼?"라고 물으면 이렇게 말하면 된다.

```text
스캔 직후에는 미리보기만 뜹니다. 새 보드로 열기나 현재 보드에 추가를 눌렀을 때만 실제 보드에 반영되고, 그때 가져온 스냅샷도 저장합니다.
```
