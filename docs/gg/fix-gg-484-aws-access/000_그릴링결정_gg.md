# AWS 가져오기 복구와 자동 정리 그릴링 결정

## 목적

- 기존 AWS 연결의 권한이 오래되어 일부 Resource만 가져오는 문제를 안전하게 복구한다.
- 자동 정리 결과를 사용자가 바로 이해할 수 있는 말로 설명한다.
- 이미 가져온 결과와 사용자의 승인 경계를 보존한다.

## 이미 정해진 기준

- 일부 AWS 서비스를 읽지 못해도 성공한 Resource는 계속 보여준다.
- 사용자가 승인하기 전에는 AWS 권한, Project, Board를 바꾸지 않는다.
- 권한 복구는 새 AWS 연결을 만들지 않고 기존 연결을 갱신한다.
- 자동 정리는 Resource, 연결 관계, 설정을 바꾸지 않는다.
- 자동 정리 전 원본을 먼저 보여주고, 사용자가 요청한 뒤 미리보기를 보여준다.

## 결정 1. 자동 정리 설명

### 결정

- 점수나 변경 개수를 첫 정보로 보여주지 않는다.
- 실제로 달라진 항목을 쉬운 문장으로 최대 3개 보여준다.
- Resource의 화면 표시 이름을 사용한다.
- 각 문장은 무엇이 바뀌었고 화면에 어떤 영향이 생겼는지 함께 설명한다.
- 나빠진 항목이 있으면 좋아진 항목보다 먼저 보여준다.
- 측정 근거가 없는데 더 읽기 좋아졌다고 단정하지 않는다.
- 더 많은 변경은 `그 외 N곳 정리`처럼 보조 정보로만 보여준다.
- `Resource, 설정, 연결 관계는 바뀌지 않았습니다.`라는 안전 안내를 항상 보여준다.

### 문구 예시

- `Lambda를 API Gateway 가까이 옮겼습니다.`
- `Subnet을 VPC 영역 안으로 정리했습니다.`
- `Resource를 가리던 연결선을 정리했습니다.`
- `연결선 교차가 1곳 늘었습니다. 원본과 비교해 주세요.`

### 피해야 할 표현

- 의미를 설명하지 않는 점수
- `N개 변경`만 있는 요약
- 후보 ID, Compiler 버전, Template ID 같은 내부 정보
- 사용자가 화면에서 찾을 수 없는 내부 Resource ID

## 결정 2. AWS 권한 관리 위치

### 결정

- AWS Role 생성과 권한 갱신은 환경설정의 AWS 연결 기능이 담당한다.
- Reverse Engineering은 AWS Resource 조회만 담당한다.
- Reverse Engineering에서 권한 부족을 발견해도 권한을 직접 바꾸지 않는다.
- 성공한 Resource는 계속 보여주고, 짧은 안내와 AWS 연결 설정으로 가는 버튼을 제공한다.
- 사용자가 AWS에서 갱신을 승인하고 돌아오면 같은 연결로 다시 가져온다.

### 이유

- Resource 조회와 계정 권한 변경을 한 화면에서 섞지 않는다.
- 사용자가 AWS 권한이 언제 바뀌는지 분명히 알 수 있다.
- Reverse Engineering의 부분 결과를 권한 복구 때문에 잃지 않는다.

## 결정 3. 기존 연결의 권한 갱신 범위

> 갱신 범위 결정은 유지한다. 사용자가 AWS Console에서 직접 갱신한다는 실행 방식만 결정 16이 대체한다.

### 선택

- 모든 기존 연결은 원래 CloudFormation Stack과 AWS Role을 그대로 유지한다.
- 원래 Stack은 업데이트하지 않는다.
- 연결별 Manager Stack과 Policy Stack을 사용하며 Policy Stack이 같은 Role에 Reverse Engineering용 읽기 Policy만 연결한다.
- 기존 Terraform 배포 Policy는 변경하지 않는다.
- 새 `AwsConnection`이나 Reverse Engineering에 직접 사용하는 두 번째 연결 Role을 만들지 않는다. 결정 23의 CloudFormation service Role은 권한 Stack 관리에만 사용한다.
- AWS 연결 담당자는 Manager Stack 생성을 AWS Console에서 승인하고, Policy Stack 생성·갱신은 환경설정에서 승인한다.
- 신규 AWS 연결도 하나의 환경설정 온보딩 흐름 안에서 Role Stack과 Import Access Stack Pair를 준비한다.

### 이유

- 기존 Stack의 정확한 배포 Template을 보관하지 않아도 배포 권한을 확실히 보존할 수 있다.
- 기존 연결과 배포 기록을 그대로 유지한다.
- Reverse Engineering이 두 연결 Role을 선택하는 DB와 API 변경 없이 복구할 수 있다.
- 사용자가 누른 `AWS 연결 권한 업데이트`와 실제 변경 범위를 일치시킨다.

### 주의할 점

- 추가하는 Policy는 읽기 전용이지만, 같은 Role에는 기존 배포 권한도 있다.
- 따라서 이 Role 전체를 `읽기 전용 Role`이라고 부르면 안 된다.
- 가져오기와 배포를 완전히 분리한 전용 Role은 이번 범위에 포함하지 않는다.

## 결정 4. 권한 갱신 완료 기준

### 선택

- Role을 맡을 수 있다는 사실만으로 완료 처리하지 않는다.
- CloudFormation Stack의 갱신 상태나 Policy 연결만으로도 완료 처리하지 않는다.
- 갱신된 같은 Role로 지원하는 가져오기 기능의 제한된 읽기 요청을 실제 실행한다.
- 필요한 읽기 요청이 성공해야 `AWS 구조를 가져올 준비가 됐습니다`라고 표시한다.
- 확인 과정에서는 AWS Resource를 생성·변경·삭제하지 않는다.

### 결과 구분

- 읽기 성공: 권한 갱신 완료 후 같은 연결로 다시 가져오기
- 권한 부족: 읽지 못한 AWS 서비스와 권한 갱신 필요 안내
- Resource Explorer 미설정: 권한 부족과 구분해 기능 설정 필요 안내
- AWS 일시 오류: 연결 실패로 확정하지 않고 다시 확인 안내

### 안전 경계

- 가져오기 읽기 확인 실패는 기존 AWS 연결의 배포용 `verified` 상태를 `failed`로 바꾸지 않는다.
- 가져오기 준비 상태와 배포 연결 상태를 별도로 다룬다.
- 원본 AWS 오류, ARN, Request ID, API action은 사용자 화면에 노출하지 않는다.

## 결정 5. 가져오기 권한 Stack 진입 방식

> 기존 기록: 이 단일 Stack 진입 방식은 결정 16·23·24의 Manager/Policy Stack 흐름이 대체한다.

### 선택

- 환경설정에서 변경되는 권한과 유지되는 권한을 먼저 설명한다.
- 사용자가 `AWS에서 확인`을 누르면 정확한 가져오기 권한 Stack의 Create 또는 Update 화면을 새 탭으로 연다.
- 짧은 수명의 CloudFormation Template URL로 읽기 Policy와 연결 대상 Role을 미리 채운다.
- 사용자가 필요하면 SSO로 로그인하고 AWS Console에서 변경 내용을 직접 검토·승인한다.
- SketchCatch는 사용자를 대신해 `CreateStack`이나 `UpdateStack`을 호출하지 않는다.

### 정상 흐름

1. 환경설정에서 `AWS 연결 권한 업데이트` 선택
2. `가져오기 읽기 권한만 추가`, `배포 권한 변경 없음` 확인
3. 정확한 가져오기 권한 Stack의 Create 또는 Update 화면 열기
4. AWS에서 변경 사항 확인 및 승인
5. SketchCatch로 돌아와 업데이트 확인
6. 같은 Role로 실제 읽기 확인
7. 성공하면 같은 연결로 다시 가져오기

### 예외 흐름

- AWS 바로가기를 만들 수 없을 때만 Template 다운로드와 단계별 안내를 제공한다.
- 원래 연결 Stack은 찾거나 업데이트하지 않는다.
- 다른 Role에 Policy를 붙이지 않도록 Role 이름과 connection ID를 Template에 고정한다.
- 이미 Stack update가 진행 중이면 중복 갱신을 시작하지 않는다.

## 결정 6. 복구를 지원할 기존 Role 범위

### 선택

- 최신 SketchCatch 관리 Role만이 아니라 결정 15의 이름 규칙으로 검증된 모든 Role을 복구 대상으로 삼는다.
- 오래된 공통 이름 Role과 이름 규칙을 만족하는 사용자 직접 생성 Role도 가져오기 읽기 권한 복구 경로를 제공한다.
- 사용자가 직접 만든 Role 자체를 SketchCatch 소유로 바꾸거나 삭제해서는 안 된다.

### 해결된 충돌

- 기존 Stack을 부분적으로 업데이트할 수 없으므로 최신 전체 Template을 적용하면 배포 권한까지 바뀔 수 있다.
- 모든 기존 Role에 같은 별도 Stack 방식을 사용해 원래 Stack의 소유 방식과 Template version에 의존하지 않는다.

## 결정 7. 가져오기 읽기 Policy 소유 방식

> 기존 기록: 사용자가 Stack 갱신을 AWS Console에서 직접 승인한다는 부분은 결정 16이 대체한다.

### 선택

- Role의 기존 소유 방식은 바꾸지 않는다.
- 모든 기존 verified 연결은 결정 24의 Import Access Stack Pair를 사용한다.
- Policy Stack은 정확한 기존 Role 이름에 읽기 Policy를 연결하지만 기존 Role Resource를 만들거나 소유하지 않는다.
- 사용자가 SSO로 AWS Console에 들어가 Manager Stack 생성을 승인하고, Policy Stack 생성·갱신은 환경설정에서 승인한다.
- 같은 connection ID를 유지하고 새 `AwsConnection`이나 두 번째 연결 Role을 만들지 않는다. CloudFormation service Role은 결정 23·24의 별도 관리 Resource다.

### 갱신과 삭제

- 읽기 권한이 바뀌면 Policy Stack만 갱신한다.
- Policy Stack을 삭제하면 읽기 Policy만 Role에서 분리·삭제한다.
- Role, 원래 연결 Stack, 기존 배포 Policy는 삭제하거나 변경하지 않는다.
- 실제 읽기 확인이 성공해야 가져오기 준비 완료로 본다.

### 모든 기존 연결의 최종 규칙

- 원래 연결 Stack과 Role은 그대로 유지
- 연결별 Manager Stack에서 service Role과 제어 Policy 관리
- Policy Stack에서 읽기 Policy만 관리
- Manager Stack은 AWS Console 승인, Policy Stack은 환경설정 승인 후 실제 읽기 확인

## 결정 8. 연결 삭제와 가져오기 Policy 정리

### 선택

- 연결 삭제 전에 제거되는 읽기 Policy와 보존되는 Role·배포 권한을 보여준다.
- 사용자를 정확한 Policy Stack 삭제 화면으로 보내 AWS에서 직접 승인을 받는다.
- SketchCatch 서버는 사용자를 대신해 `DeleteStack`을 호출하지 않는다.
- Stack 삭제를 확인한 뒤 연결 metadata 정리를 완료한다.

### 실패 처리

- Policy Stack 삭제가 끝나지 않으면 연결을 활성 상태로 다시 사용하지 않는다.
- `AWS 권한 정리 필요` 상태와 재시도 정보를 남긴다.
- 사용자는 같은 Stack 삭제 흐름을 나중에 다시 이어갈 수 있다.
- Role, 원래 연결 Stack, 기존 배포 Policy는 모든 경우에 보존한다.

### 금지

- Policy Stack을 남긴 채 연결 metadata만 조용히 삭제하지 않는다.
- SketchCatch가 AWS Console 승인 없이 Policy Stack을 자동 삭제하지 않는다.
- Policy Stack 정리 실패를 이유로 Role이나 원래 연결 Stack을 삭제하지 않는다.

## 결정 9. 자동 정리가 바꿀 수 있는 시각 범위

### 선택

- Resource와 기존 영역의 위치를 바꿀 수 있다.
- 기존 VPC·AZ·Subnet 영역과 이미 작성된 표시 프레임의 크기를 조절할 수 있다.
- 개별 Resource 카드와 설명용 Design 요소의 크기를 제한 안에서 조절할 수 있다.
- 연결선 경로와 Resource 사이 간격·정렬을 바꿀 수 있다.
- 표시만을 위한 프레임을 만들고, 합치고, 비어 있으면 제거할 수 있다.
- 표시용 제목과 경계를 추가할 수 있다.

### 절대 유지

- AWS Resource 종류·개수·이름·설정
- Resource 관계와 연결 방향
- 실제 VPC·AZ·Subnet 소속
- 모든 기존 부모 영역 값
- Terraform 내용
- 사용자가 명시한 Architecture 의미

## 결정 10. 표시 프레임의 저장 방식

### 선택

- 화면을 주제별로 나누는 요소의 이름은 `Group`이 아니라 `표시 프레임`으로 한다.
- 표시 프레임은 제목과 배경 사각형을 가진 `design` 요소로 저장한다.
- 표시 프레임은 자신의 ID, 제목, 위치, 크기, 스타일만 저장한다.
- 프레임 안에 보이는 Resource 목록이나 별도의 소속 ID는 저장하지 않는다.
- 프레임은 Resource의 부모가 아니며, Resource의 부모 영역 값도 프레임을 가리키지 않는다.
- 프레임이 Resource를 둘러싼 모양은 화면 표현일 뿐 Architecture 소속을 뜻하지 않는다.
- 자동 정리는 현재 배치를 보고 자동 생성 표시 프레임을 만들고, 합치고, 크기를 바꾸고, 제거할 수 있다.

### 절대 영향이 없는 곳

- 실제 VPC·AZ·Subnet containment
- Resource 관계와 설정
- Terraform 생성과 Provider graph
- 비용·보안 검사와 배포
- Reverse Engineering의 가져온 원본

### 이유

- 기존 부모 영역 값을 재사용하면 화면 정리가 실제 AWS 소속 변경처럼 저장될 수 있다.
- 별도의 표시용 소속 데이터를 만들면 사용자가 단순한 배경 프레임을 옮길 때마다 숨은 소속도 관리해야 한다.
- 소속 없는 Design 요소로 두면 자동 정리가 Architecture 의미를 바꾸지 않는다는 계약이 분명해진다.

### 코드와 맞춰야 할 점

- 기존 `DiagramNode.kind = "design"` 표현을 사용한다.
- 표시 프레임 자신과 프레임이 둘러싼 Resource 어디에도 표시용 소속 값을 추가하지 않는다.
- 자동 정리 diff는 표시 프레임 추가·변경·삭제를 시각 변경으로 취급한다.
- 의미 비교와 Terraform 변환은 표시 프레임을 Architecture 소속으로 해석하지 않는다.

## 결정 11. 수동 편집 뒤 표시 프레임 동작

### 선택

- 자동 정리를 적용한 뒤 사용자가 Resource를 옮겨도 표시 프레임은 자동으로 따라가지 않는다.
- Resource 이동만으로 표시 프레임의 위치나 크기를 몰래 바꾸지 않는다.
- 사용자는 표시 프레임을 직접 옮기거나, 크기를 바꾸거나, 삭제할 수 있다.
- 사용자가 다음 자동 정리를 요청하면 그 시점의 Board 배치를 기준으로 프레임을 다시 계산한다.
- 다음 자동 정리는 모든 표시 프레임의 위치·크기를 조절할 수 있지만, 자동 생성 표시 프레임만 병합·삭제할 수 있다. 적용 전 미리보기와 승인을 다시 거친다.

### 이유

- 표시 프레임에는 Resource 소속 정보가 없으므로 어떤 Resource를 따라가야 하는지 숨겨서 추론하지 않는다.
- Board를 편집할 때 사용자가 요청하지 않은 자동 변경과 History가 생기지 않는다.
- 프레임 정리는 `자동 정리`를 다시 요청한 순간에만 일어나므로 동작을 예측할 수 있다.

## 결정 12. 측정상 좋아지지 않은 자동 정리 결과

### 선택

- Resource·관계·설정·containment를 보존한 시각 변경이 하나라도 있으면 항상 미리보기를 제공한다.
- 겹침·영역 이탈·연결선 문제 같은 측정값이 좋아지지 않아도 `이 정리 사용` 버튼을 막지 않는다.
- 일부 측정값이 나빠져도 결과를 숨기거나 자동으로 원본으로 바꾸지 않는다.
- 품질 측정값은 후보를 차단하는 점수가 아니라 사용자가 비교할 수 있는 설명으로만 사용한다.
- 실제 시각 변경이 하나도 없을 때만 미리보기를 열지 않고 `바꿀 배치가 없습니다.`라고 안내한다.

### 설명 규칙

- 좋아진 점과 나빠진 점을 숨기지 않고 쉬운 문장으로 최대 3개 보여준다.
- 확인된 개선이 없으면 `확인된 배치 문제는 줄지 않았습니다. 원본과 비교해 선택해 주세요.`라고 표시한다.
- 내부 품질 점수, 후보 ID, Compiler version은 보여주지 않는다.
- `Resource, 설정, 연결 관계는 바뀌지 않았습니다.`라는 안전 안내는 항상 유지한다.

### 이유

- 측정 규칙이 사용자의 모든 시각적 선호를 대신할 수 없다.
- 같은 Diagram도 화면 크기와 사용 목적에 따라 사용자가 더 읽기 좋다고 판단할 수 있다.
- 원본과 결과를 직접 비교하고 명시적으로 적용하는 최종 결정은 사용자에게 둔다.

## 결정 13. 기존 연결 Stack을 업데이트하지 않는 방식으로 수정

### 선택

- 모든 기존 verified 연결은 원래 CloudFormation Stack을 업데이트하지 않는다.
- 연결별 Import Access Stack Pair를 사용하고 Policy Stack이 같은 Role에 읽기 Policy만 연결한다.
- 원래 Stack, Role, 배포 Policy, connection ID는 그대로 유지한다.
- 새 연결 Role이나 새 `AwsConnection`을 만들지 않는다. 결정 23의 CloudFormation service Role은 권한 관리에만 사용한다.

### 수정 이유

- CloudFormation Stack update는 전체 Template을 기준으로 변경 대상을 계산한다.
- SketchCatch는 기존 계정에 실제 설치된 전체 Template을 저장하지 않는다.
- 최신 전체 Template을 기존 Stack에 적용하면 읽기 권한 외의 배포 Policy와 다른 Resource까지 바뀔 수 있다.
- 별도 Stack을 사용하면 원래 Stack 내용을 몰라도 가져오기 읽기 Policy만 안전하게 관리할 수 있다.

## 결정 14. 가져오기 권한 Stack 삭제 완료 기준

> 읽기 권한 제거 확인은 유지한다. Manager Stack과 제어 권한 정리는 결정 24가 추가한다.

### 선택

- 사용자가 AWS Console에서 가져오기 권한 Stack 삭제를 직접 승인한다.
- 사용자가 SketchCatch로 돌아와 `삭제를 완료했습니다`를 선택한다.
- SketchCatch는 같은 Role로 제한된 가져오기 읽기 확인을 다시 실행한다.
- 추가했던 가져오기 권한이 더 이상 사용되지 않는 것이 확인돼야 connection metadata 삭제를 완료한다.
- 권한이 계속 남아 있거나 결과를 확실히 판단할 수 없으면 연결을 삭제하지 않는다.
- 확인되지 않은 연결은 비활성 `AWS 권한 정리 필요` 상태로 남기고 다시 확인할 수 있게 한다.

### 결과 구분

- 권한 제거 확인: connection metadata 정리 완료
- 가져오기 읽기 계속 성공: 권한이 남아 있을 수 있으므로 정리 미완료
- AWS 일시 오류: 삭제 성공으로 간주하지 않고 다시 확인
- Role 접속 실패: Stack 삭제 완료의 증거로 사용하지 않고 다시 확인

### 절대 유지

- 배포용 `verified` 상태를 cleanup 결과 때문에 `failed`로 바꾸지 않는다.
- Role, 원래 연결 Stack, 기존 배포 Policy를 삭제하거나 변경하지 않는다.
- 사용자의 완료 확인만으로 connection metadata를 먼저 삭제하지 않는다.

## 결정 15. 이번 범위에서 지원하는 Role 이름

### 선택

- 현재 AWS 연결 검증이 허용하는 Role 이름 규칙을 그대로 사용한다.
- `SketchCatchTerraformExecutionRole` 고정 이름과 `SketchCatchTerraformExecutionRole-...` 접미사 이름을 지원한다.
- 이름 규칙을 만족하면 CloudFormation으로 만든 Role과 사용자가 직접 만든 Role을 구분하지 않고 같은 Stack pair 방식으로 지원한다.
- 임의 이름 Role을 새로 허용하지 않는다.

### 이유

- 현재 verified 연결을 보존하면서 SketchCatch 운영 서버의 IAM AssumeRole 허용 범위를 넓히지 않는다.
- 임의 Role 이름을 허용하려면 연결별 allowlist, 운영 IAM Policy 변경, 잘못된 Role 접근 방지 검증을 별도로 설계해야 한다.
- 이번 작업의 목적은 기존 연결에 부족한 가져오기 읽기 권한을 안전하게 보완하는 것이며 Role 신뢰 경계를 넓히는 것이 아니다.

## 결정 16. 기존 Policy Stack의 갱신 주체

### 선택

- 이미 생성된 Import Access Policy Stack의 버전 갱신은 SketchCatch 서버가 CloudFormation API로 직접 수행한다.
- 사용자는 환경설정에서 실제로 바뀌는 읽기 권한을 확인하고 명시적으로 승인한다.
- 서버의 변경 권한은 해당 connection ID와 Role에 연결된 Policy Stack에만 한정한다.
- 원래 연결 Stack, Role Resource, 기존 배포 Policy는 이 API 경로에서 조회 대상으로만 취급하고 변경하지 않는다.
- Manager Stack 최초 생성과 연결 삭제 시 두 Stack 제거는 현재 정한 AWS Console 승인 흐름을 유지한다.

### 결과

- 기존 Policy Stack 업데이트를 위해 사용자가 Template을 내려받아 AWS Console에 다시 올릴 필요가 없다.
- 이전 결정의 `SketchCatch는 UpdateStack을 호출하지 않는다`는 제한은 Policy Stack 갱신에 한해서 폐기한다.
- 사용자 승인 방식과 CloudFormation 변경 실패·재시도 상태는 다음 결정에서 구체화한다.

### 금지

- 원래 연결 Stack이나 배포 Policy를 같은 API로 갱신하지 않는다.
- 다른 connection의 Policy Stack을 이름만으로 선택하지 않는다.
- 사용자 승인 없이 백그라운드에서 권한을 최신화하지 않는다.

## 결정 17. Policy Stack 갱신 승인 방식

### 선택

- 환경설정은 사용자가 이해할 수 있는 말로 추가·변경되는 가져오기 조회 범위와 유지되는 항목을 먼저 보여준다.
- 사용자가 `가져오기 권한 업데이트`를 한 번 누르면 Settings API가 기존 Policy Stack을 즉시 갱신한다.
- CloudFormation Change Set을 미리 만들거나 AWS Console에서 두 번째 승인을 받지 않는다.
- 승인 화면에는 `원래 Stack 변경 없음`, `Role 유지`, `배포 권한 변경 없음`을 항상 표시한다.

### 실행 안전 조건

- 승인 정보는 connection ID, 정확한 Stack pair 식별자, 기존 Role ARN, service Role ARN, 현재·목표 Template version과 Policy hash에 묶는다.
- 승인 뒤 실행 시 Stack 식별자·상태·현재 version이 달라졌으면 업데이트하지 않고 새 확인을 요구한다.
- 같은 목표 version을 중복 요청해도 한 번만 갱신한다.
- Stack이 다른 작업 중이면 병렬 갱신하지 않고 현재 작업이 끝난 뒤 다시 시도하게 한다.
- 갱신 완료 뒤 실제 bounded read probe를 통과해야 가져오기 준비 완료로 표시한다.
- 실패하면 배포용 `verified` 상태와 원래 Stack·Role·배포 Policy를 그대로 유지하고 안전한 재시도 상태를 보여준다.

### 표시하지 않는 정보

- raw IAM JSON
- CloudFormation 내부 Resource 논리 ID
- AWS Request ID와 내부 Policy hash
- 사용자가 해석하기 어려운 권한 개수만의 요약

## 결정 18. 가져오기 준비 완료에 필요한 읽기 범위

### 선택

- 주요 보드 Resource reader가 성공하면 AWS 구조 가져오기를 시작할 수 있다.
- 주요 reader는 EC2 기반 VPC·Subnet·Internet Gateway·Route Table·Security Group·Instance, S3, RDS, Load Balancer, ECS, CloudFront다.
- Resource Explorer, Resource Groups Tagging API, IAM, KMS, CloudWatch·Logs, API Gateway, Lambda와 AMI 같은 확장 reader 실패는 주요 가져오기를 막지 않는다.
- 확장 reader 일부가 실패하면 `가져오기 가능 · 일부 확장 정보 제한`으로 표시하고 누락될 수 있는 항목을 쉬운 이름으로 보여준다.
- Resource Explorer가 계정에 설정되지 않은 상태는 권한 부족과 구분해 `추가 탐색 기능 설정 필요`로 안내한다.

### 상태 구분

- 주요 reader 모두 성공, 확장 reader도 성공: `가져오기 준비됨`
- 주요 reader 모두 성공, 확장 reader 일부 실패 또는 미설정: `가져오기 가능 · 일부 확장 정보 제한`
- 주요 reader 중 하나 이상 AccessDenied: `가져오기 권한 업데이트 필요`
- 주요 reader의 일시 AWS 오류: `잠시 후 다시 확인`
- Role 접속 실패: 가져오기 준비 상태를 확정하지 않고 AWS 연결 확인 안내

### 안전 경계

- 빈 목록 응답은 읽기 성공으로 취급하며 Resource가 없다는 이유로 실패 처리하지 않는다.
- 확장 reader 실패를 숨기지 않지만 전체 가져오기를 막는 오류로 과장하지 않는다.
- 가져오기 준비 상태는 배포용 connection `verified`와 별도로 저장하고 표시한다.
- 실제 Reverse Engineering에서도 reader별 실패를 부분 결과와 함께 유지한다.

## 결정 19. 자동 생성 표시 프레임 식별 방식

### 선택

- 기존 Diagram 형식을 그대로 사용한다.
- 표시 프레임은 `kind: "design"`, `type: "design_group"`, `metadata.presentationCatalogItemId: "design-group"`로 저장한다.
- Board Auto Arrange가 새로 만든 프레임 ID만 `board-auto-frame:`으로 시작한다.
- 새 metadata 필드나 DB column을 추가하지 않는다.

### 자동 정리 규칙

- `board-auto-frame:` ID가 있는 프레임만 다음 자동 정리에서 자동 병합·삭제할 수 있다.
- 사용자가 직접 만든 Design Group은 자동 정리가 위치와 크기를 제안할 수 있지만 병합·삭제하거나 자동 생성 프레임으로 바꾸지 않는다.
- 프레임 제목이나 현재 위치로 자동 생성 여부를 추측하지 않는다.
- 자동 생성 프레임도 사용자가 직접 이동·크기 조절·삭제할 수 있다.
- 자동 생성 ID는 저장·복원 뒤에도 유지되지만 Resource membership을 뜻하지 않는다.

### 이유

- 기존 직렬화 형식과 저장 API를 그대로 사용한다.
- 사용자 작성 Group을 자동 정리 소유로 오인해 제거하는 일을 막는다.
- 표시 프레임에 Resource 소속 목록이나 Architecture 의미를 추가하지 않는다.

## 결정 20. 자동 정리 후보 제공 방식

### 선택

- Resource·관계·설정·containment를 보존한 서로 다른 시각 후보가 여러 개면 사용자에게 여러 정리안을 제공한다.
- 원본은 후보 점수 경쟁에 넣지 않고 모든 정리안의 비교 기준으로만 사용한다.
- 내부 품질 측정값은 안전한 변경 후보끼리 순서를 정하고 차이를 설명할 때만 사용한다.
- 후보 전환만으로 Board를 바꾸지 않으며, 사용자가 선택한 정리안에 `이 정리 사용`을 눌렀을 때만 적용한다.
- 안전한 시각 후보가 하나면 하나만 보여주고, 하나도 없을 때만 `바꿀 배치가 없습니다.`라고 안내한다.

### 후보 정리 규칙

- geometry와 edge route가 사실상 같은 후보는 하나로 합친다.
- 각 후보는 자체 시각 diff와 좋아진 점·나빠진 점 설명을 가진다.
- 내부 후보 ID, 품질 점수, Compiler version은 사용자에게 보여주지 않는다.
- 후보 개수와 데스크톱·모바일 선택 UI는 다음 결정에서 확정한다.

## 결정 21. 데스크톱 자동 정리 후보 갤러리

### 선택

- 서로 다른 안전한 정리안을 최대 3개 제공한다.
- 안전한 정리안이 1개나 2개뿐이면 있는 만큼만 보여주며 비슷한 가짜 후보를 만들지 않는다.
- 데스크톱 미리보기 상단에 각 정리안의 전체 보드 thumbnail을 갤러리로 나란히 보여준다.
- thumbnail에는 `정리안 A`, `정리안 B`, `정리안 C`처럼 쉬운 이름과 대표 변경 한 줄을 표시한다.
- thumbnail을 선택하면 같은 화면 아래의 큰 비교 영역이 해당 후보로 바뀐다.
- 큰 비교 영역은 원본과 선택한 정리안을 나란히 보여주고, 선택한 후보의 변경 설명과 `이 정리 사용`을 제공한다.
- thumbnail 선택과 후보 전환은 Board를 저장하지 않는다.

### 선택 근거

- 사용자가 전체 구도가 다른 후보를 먼저 빠르게 훑을 수 있다.
- 작은 thumbnail만으로 적용하지 않고 큰 원본 비교를 거쳐 Resource 위치와 연결선을 확인할 수 있다.
- 한 장씩 넘기는 방식보다 후보 차이를 기억에 의존하지 않는다.

## 결정 22. 모바일 자동 정리 후보 비교

### 선택

- 화면 상단에 최대 3개의 후보 thumbnail을 가로 갤러리로 보여준다.
- 후보가 화면 폭을 넘으면 갤러리만 가로로 스크롤하며 전체 화면을 옆으로 밀지 않는다.
- 선택한 후보는 테두리와 `선택됨` 상태로 분명히 표시한다.
- 갤러리 아래에는 `원본`과 선택한 `정리안`을 바꾸는 두 칸 전환 버튼을 둔다.
- 모바일에서는 두 보드를 좁게 나란히 놓지 않고 같은 전체 폭에서 하나씩 보여준다.
- 원본과 정리안 전환은 같은 비교 viewport와 zoom을 유지해 위치 차이를 알아보기 쉽게 한다.
- 변경 설명과 `이 정리 사용` 버튼은 선택한 정리안 아래에 둔다.
- 후보 선택, 가로 스크롤, 원본 전환은 Board를 저장하지 않는다.

### 데스크톱과 동일한 계약

- 최대 3개의 서로 다른 안전한 후보만 제공한다.
- 내부 점수 대신 대표 변경 한 줄과 상세 설명을 사용한다.
- 사용자가 선택한 후보 하나만 명시적 적용으로 저장한다.
- 적용 전까지 원본 Board와 History는 변하지 않는다.

## 결정 23·24로 해결된 권한 충돌

- 현재 `SketchCatchTerraformExecutionRole` Policy에는 `cloudformation:UpdateStack`이 없다.
- Settings API가 companion Stack을 직접 갱신하려면 Role에 Stack 관리 권한을 추가하거나 별도 CloudFormation service Role이 필요하다.
- 기존 Role이 service Role 없이 `AWS::IAM::Policy`를 갱신하면 정확한 Stack의 `UpdateStack`뿐 아니라 정확한 Role Policy를 바꾸는 IAM mutation 권한도 필요하다.
- 따라서 companion Stack이 `읽기 Policy만 소유한다`는 결정과 서버 직접 갱신 결정은 현재 그대로는 동시에 만족할 수 없다.
- 결정 23의 전용 service Role과 결정 24의 Stack pair 분리로 기존 Role의 자기 권한 변경을 피한다.

## 결정 23. Policy Stack 갱신 실행 Role

### 선택

- 기존 연결 Role에 자기 Policy를 변경하는 CloudFormation 실행 권한을 추가하지 않는다.
- `cloudformation.amazonaws.com`만 맡을 수 있는 전용 CloudFormation service Role을 사용한다.
- service Role은 Policy Stack 읽기 Policy의 생성·version 갱신·분리·삭제에 필요한 최소 IAM action만 가진다.
- 기존 연결 Role은 Reverse Engineering과 배포 연결에 계속 사용하고 service Role은 연결·조회·배포에 사용하지 않는다.
- Settings API는 정확한 Policy Stack과 service Role ARN을 함께 검증한 뒤에만 갱신한다.

### AWS 수명주기 제약

- CloudFormation service Role은 대상 Stack을 만들기 전에 존재해야 한다.
- Stack에 연결된 service Role은 이후 모든 작업에 사용되며 Stack에서 제거할 수 없다.
- 따라서 service Role을 자신이 실행할 Policy Stack 안에 함께 넣고 같은 Stack 삭제로 정리하는 방식은 사용하지 않는다.
- 공식 근거: https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/using-iam-servicerole.html

### 유지되는 안전 경계

- 원래 연결 Stack, 기존 연결 Role, 배포 Policy는 갱신하지 않는다.
- service Role에 일반 AWS Resource 생성·변경 권한을 주지 않는다.
- 사용자 승인 없는 Policy Stack 갱신을 허용하지 않는다.
- service Role과 Policy가 모두 제거됐는지 확인되기 전에는 connection metadata를 삭제하지 않는다.

## 결정 24. 연결별 Manager Stack과 Policy Stack 분리

### 선택

- AWS 연결마다 가져오기 권한용 CloudFormation Stack을 두 개 사용한다.
- `Import Access Manager Stack`은 CloudFormation service Role과 기존 연결 Role에 붙는 최소 제어 Policy를 소유한다.
- `Import Access Policy Stack`은 Reverse Engineering 읽기 Managed Policy만 소유하고 같은 기존 연결 Role에 연결한다.
- 두 Stack, service Role, 제어 Policy, 읽기 Policy는 모두 connection ID prefix로 이름을 고정한다.
- 계정 전체가 공유하는 service Role이나 Manager Stack은 사용하지 않는다.

### 생성 순서

1. 환경설정이 정확한 Manager Stack Quick Create 화면을 연다.
2. 사용자가 AWS Console에서 service Role과 최소 제어 Policy 생성을 검토·승인한다.
3. SketchCatch가 같은 연결 Role로 Manager Stack 출력과 ownership을 검증한다.
4. 사용자가 환경설정에서 추가될 읽기 범위를 확인하고 `가져오기 권한 추가`를 누른다.
5. Settings API가 service Role ARN을 지정해 정확한 Policy Stack을 생성한다.
6. Policy Stack 완료 뒤 주요·확장 bounded read probe를 실행한다.

### 갱신 순서

- Settings API는 Policy Stack만 갱신한다.
- Manager Stack, service Role, 제어 Policy는 Template contract version이 그대로면 갱신하지 않는다.
- Policy Stack의 Role ARN, target Role, connection tag, 현재·목표 Template version이 모두 일치해야 갱신한다.
- 사용자의 한 번 명시적 승인 뒤 사전 Change Set 없이 실행한다.

### 삭제 순서

1. 환경설정이 Policy Stack과 Manager Stack, 제거될 권한, 보존될 원래 Role·Stack·배포 Policy를 보여준다.
2. 사용자가 정확한 Policy Stack을 AWS Console에서 먼저 삭제한다.
3. 같은 Role의 bounded read probe로 가져오기 읽기 권한 제거를 확인한다.
4. 사용자가 정확한 Manager Stack을 AWS Console에서 삭제한다.
5. STS Role 확인이 성공한 상태에서 SketchCatch의 해당 Policy Stack 제어 권한이 더 이상 없는지 제한된 확인을 실행한다.
6. 읽기 권한 또는 제어 권한이 남아 있거나 결과가 불확실하면 비활성 `AWS 권한 정리 필요` 상태를 유지한다.

### 절대 유지

- 원래 AWS 연결 Stack과 기존 연결 Role은 두 Stack의 소유 Resource가 아니다.
- 기존 Terraform 배포 Policy와 배포용 `verified` 상태를 변경하지 않는다.
- 한 연결의 service Role로 다른 연결의 Policy Stack을 만들거나 갱신하지 않는다.
- Manager Stack을 Policy Stack보다 먼저 삭제하도록 안내하지 않는다.

### 이유

- service Role을 Policy Stack보다 먼저 만들 수 있다.
- Policy Stack 삭제가 끝날 때까지 service Role이 살아 있어 CloudFormation이 읽기 Policy를 정상 제거할 수 있다.
- 마지막에 Manager Stack을 삭제해 service Role과 제어 Policy를 함께 정리할 수 있다.
- 연결별 격리로 한 연결의 갱신·삭제 실패가 다른 연결에 번지지 않는다.

## 결정 25. Reverse Engineering 원본과 자동 정리 실행 시점

### 선택

- AWS에서 읽은 현재 지원 Resource와 관계를 source-exact 방식으로 먼저 보여준다.
- 원본 변환은 화면에 표시하기 위한 결정론적 최초 좌표만 계산한다.
- 일반 AI 변환이 Region·AZ, 기본 설정, 요약 edge, containment, Terraform 이름을 새로 추론하지 않는다.
- 현재 지원 Resource가 관계 없이 단독으로 발견돼도 원본에서 조용히 버리지 않는다.
- 여러 페이지 중 뒤쪽 조회가 실패해도 앞에서 읽은 결과를 유지한다.
- 사용자가 `자동 정리 해보기`를 선택하기 전에는 Architecture Board Compiler를 실행하지 않는다.
- 원본 확인, 자동 정리 실행, 선택한 정리안 적용을 서로 다른 사용자 결정으로 유지한다.
- 후보 생성과 전환은 Project·Board·Terraform을 저장하지 않으며 `이 정리 사용`만 기존 저장 흐름으로 이어진다.

### 이유

- 자동 정리가 먼저 실행되면 AWS에서 읽은 구조와 SketchCatch가 바꾼 화면을 구분하기 어렵다.
- 일반 AI 변환은 화면 보조를 위해 Architecture 의미를 더할 수 있어 가져온 원본 계약에 맞지 않는다.
- 부분 조회 결과와 단독 Resource를 보존해야 권한 부족 상태에서도 사용자가 실제로 가져온 범위를 확인할 수 있다.

## 결정 26. Stack pair 정리는 소유 artifact를 기준으로 확인

### 선택

- Manager Stack은 service Role·제어 Policy 외에 두 Stack과 소유 IAM Resource만 확인하는 읽기 전용 확인 Policy를 가진다.
- 다른 Manager Resource가 확인 Policy에 의존하게 해 삭제할 때 확인 Policy가 마지막에 제거되도록 한다.
- Policy Stack 삭제 뒤에는 정확한 Stack 부재와 읽기 Managed Policy의 분리·삭제를 먼저 확인한다.
- bounded read probe는 안전 신호로 기록하지만 다른 Policy가 같은 읽기 권한을 주는 경우 정리를 막는 유일한 기준으로 사용하지 않는다.
- Manager Stack 삭제 뒤에는 기존 Role 접속이 성공하는 상태에서 service Role·제어 Policy·확인 Policy 제거를 확인한다.
- 삭제 전 전체 contract와 attachment를 검증한 경우에만 마지막 확인 권한의 AccessDenied를 제거 신호로 사용한다.
- 일부 소유 artifact가 남거나 결과가 불확실하면 `AWS 권한 정리 필요`를 유지한다.
- 정리가 끝나기 전에는 원래 connection row를 물리적으로 삭제하지 않는다.

### 대체되는 부분

- 결정 14와 결정 24에서 bounded read·control probe 성공 여부만으로 정리를 판정하던 부분을 대체한다.
- 다른 AWS Policy가 같은 유효 권한을 제공해도 SketchCatch가 소유한 Stack과 Policy가 정확히 제거됐으면 해당 단계 정리를 완료할 수 있다.

## 결정 27. Policy Stack 변경 경로를 immutable Template과 전체 Manager 검증으로 제한

### 선택

- 기존 연결 Role은 정확한 Policy Stack, service Role, connection tag, `AWS::IAM::ManagedPolicy` Resource type, 서버 소유 immutable Template URL만 사용할 수 있다.
- `TemplateBody`와 다른 Template URL은 허용하지 않는다.
- service Role은 connection 전용 읽기 Managed Policy와 정확한 target Role attachment만 관리한다.
- Manager Stack 검증은 tag·output뿐 아니라 Template hash, service Role trust·권한, 제어·확인 Policy 문서와 attachment를 확인한다.
- Manager contract version 변경은 Settings API가 자동 갱신하지 않고 사용자가 AWS Console에서 명시적으로 승인한다.
- Policy 승인값은 10분 동안 한 번만 사용할 수 있고 connection·Stack pair·Role·version·Policy 내용에 묶는다.
- connection별 작업 lease와 operation ID로 중복·병렬 Stack 작업을 막는다.

### 이유

- 정확한 Stack과 service Role만 제한해도 임의 Template을 허용하면 더 강한 Managed Policy를 만들 수 있다.
- Manager Stack drift를 tag와 output만으로는 발견할 수 없다.
- Manager 권한 변경은 Policy 읽기 범위 변경보다 큰 보안 경계이므로 AWS Console 승인을 유지한다.

## 결정 28. 자동 정리는 다중 후보와 저장 revision을 함께 검증

### 선택

- 후보에서 허용된 geometry와 edge route만 뽑아 원본 의미 위에 다시 조립한 뒤 의미 동일성을 확인한다.
- 자동 표시 프레임은 `kind`, `type`, catalog ID, ID prefix full tuple이 모두 맞아야 자동 소유로 취급한다.
- 표시 프레임은 drop parent, containment 추론, 실제 area auto-size에서 제외한다.
- 잠긴 자동 프레임은 이동·크기 변경·병합·삭제하지 않는다.
- 다중 후보 계약은 후보별 Diagram, visual diff, 사용자 finding, 원본 fingerprint를 가진다.
- apply는 preview session과 선택 후보 하나를 지정하며 후보 전환은 Board를 바꾸지 않는다.
- 적용 때 현재 fingerprint와 ProjectDraft expected revision을 모두 확인하고 서버 compare-and-swap으로 한 번만 저장한다.
- AI 설계·Template의 구조 생성 자체를 visual-only로 제한하지 않고, 각 기능의 별도 `자동 정리` 단계만 같은 Adapter를 사용한다.
