# Reverse Engineering 정보 표시 정리

## 목적

- AWS 구조를 처음 읽은 사용자가 보드와 다음 행동을 바로 이해하게 한다.
- 개발용 설명과 같은 의미의 반복 문장을 기본 화면에서 없앤다.
- 필요한 정보는 숨기지 않고, 사용자가 원할 때만 열어 본다.

## 결정 0. Reverse Engineering 시작과 Workspace 진입

- 프로젝트 생성에서 `Reverse Engineering으로 시작`을 선택하면 AWS 구조를 읽는다.
- 읽은 구조는 먼저 미리보기로만 보여준다.
- 사용자가 `보드에 적용`을 눌러야 Workspace Board가 생성되고 Workspace로 들어간다.
- Workspace에는 `기존 AWS 가져오기`와 다시 스캔하는 새로고침 기능을 두지 않는다.

## 결정 1. 가져오기 완료 직후의 기본 정보

- 기본으로 `리소스 N개 · 연결 N개`만 보여준다.
- 주 행동은 `보드에 적용`이다. 사용자가 이를 눌러야 가져온 구조로 Workspace Board를 만든다.
- `보기 좋게 정리`는 원본을 더 읽기 좋게 보고 싶을 때만 쓰는 보조 행동이다. 정리 후에도 사용자가 `보드에 적용`을 눌러야 한다.
- `상세 정보`는 필요한 정보를 모달에서 여는 보조 진입점이다.
- 서비스별 개수, 읽지 못한 서비스, Terraform 관리 가능 여부, 내부 처리 설명은 기본 화면에 두지 않는다.

## 결정 2. 자동 정리 진행 상태

- 사용자가 `보기 좋게 정리`를 누르면 버튼 안에서만 즉시 `정리하는 중…` 상태와 작은 로딩 아이콘을 보여준다.
- 처리 중에는 같은 요청을 다시 보낼 수 없게 한다.
- 보드 위에 별도 안내를 띄우거나, 보드를 덮는 로딩 화면을 띄우지 않는다.
- 완료되면 원본과 정리안을 비교하는 화면으로 전환한다.

## 결정 3. 정보 공개 단계

### 항상 보이는 정보

- 리소스 수와 연결 수
- 현재 보드가 원본인지 정리안인지
- 지금 할 수 있는 다음 행동 하나

### 사용자가 열어 볼 정보

- 서비스별 리소스 수
- 읽지 못했거나 일부 정보가 부족한 서비스
- 현재 수정·배포할 수 없는 리소스와 짧은 이유
- 자동 정리에서 달라진 위치와 연결선

### 상세 정보의 첫 화면

- 상세 정보 모달은 `네트워크`, `서버`, `저장소`처럼 사용자가 이해할 수 있는 큰 분류와 개수를 먼저 보여준다.
- 일부 제한이나 읽기 실패가 있으면 구조 요약 아래에서 짧게 알린다.
- 개별 리소스 목록과 검색은 그 다음 단계에서 제공한다.

### 상세 정보 진입

- 기본 보드 화면에는 `상세 정보` 진입점 하나만 둔다.
- 상세 정보는 화면을 덮는 큰 모달에서 연다. 오른쪽 패널에 길게 늘어놓지 않는다.
- 상세 화면을 닫으면 사용자는 같은 보드 위치와 원본·정리안 상태로 돌아온다.

### 사용자 화면에서 숨길 정보

- 내부 기능 이름과 처리 단계
- Terraform 변환 규칙과 관리 판정 근거
- Provider 요청, ARN, Resource ID, 오류 코드
- 같은 뜻을 반복하는 안내 문장

## 결정 4. 문구 정리 원칙

- 한 화면에서 같은 행동을 설명하는 문장은 한 번만 쓴다.
- 제목은 기능 이름, 본문은 사용자가 얻는 결과만 말한다.
- `가져오기`, `불러오기`, `읽기`는 화면 문맥에 따라 섞지 않고, 기존 AWS 구조를 화면에 표시하는 행동은 `가져오기`로 통일한다.
- `보드 후보`, `가져온 원본`, `가져온 구조`가 같은 상태를 가리킬 때는 `가져온 구조`로 통일한다.

## 결정 5. 여러 인프라의 화면상 분리

- 같은 AWS 계정에서 읽은 Resource도 인프라별 표시 프레임으로 나누어 보인다.
- `Project`, `Service`, `Environment` AWS Tag를 가장 먼저 묶음 기준으로 사용한다.
- Tag가 없을 때만 VPC와 실제 연결 관계를 보조 기준으로 사용한다.
- 표시 프레임은 화면 정리용이며 AWS 소속, Resource 관계, Terraform 관리 범위를 바꾸지 않는다.
- 사용자는 프레임의 이름·위치·구성을 수정하거나 삭제할 수 있다.

### Tag가 없거나 서로 다른 경우

- Tag가 없는 Resource는 연결된 인프라 프레임에 넣는다.
- 연결할 인프라가 없으면 `공통 리소스` 프레임에 넣는다.
- 표시 이름은 `서비스명 · 환경` 형식을 우선 사용한다.

## 결정 6. 자동 정리의 인프라 프레임 보존

- 자동 정리는 인프라 프레임 안의 Resource 배치만 정리한다.
- 자동 정리는 Resource를 다른 인프라 프레임으로 옮기지 않는다.
- 자동 정리는 인프라 프레임의 위치·크기를 바꾸거나, 프레임을 합치고 쪼개고 지우지 않는다.
- `공통 리소스`도 하나의 인프라 프레임으로 같은 보호를 받는다.
