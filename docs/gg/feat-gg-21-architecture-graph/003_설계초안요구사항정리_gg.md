# 설계 초안 요구사항 정리

이 문서는 내가 맡은 설계 초안 기능을 한 번에 이해하기 위한 문서다.

쉽게 말하면:

```
사용자가 원하는 것을 적고 선택한다.
→ 우리 쪽에서 안전한 설계 초안을 만든다.
→ 왜 그렇게 만들었는지 설명한다.
→ 배포 전에 조심할 점을 알려준다.
```

---

## 1. 왜 필요한가

사용자는 처음부터 클라우드 자원을 하나씩 고르기 어렵다.

예를 들면 사용자는 보통 이렇게 말한다.

```
데이터베이스가 포함된 백엔드 서버를 만들고 싶어요.
```

그런데 이 말만으로 바로 설계를 만들면 위험하다.

이유는 간단하다.

- 사용자가 원하는 용도가 정확하지 않을 수 있다.
- 비용을 낮게 하고 싶은지 알 수 없다.
- 트래픽이 작은지 보통인지 알 수 없다.
- 보안을 기본으로 할지 더 강하게 할지 알 수 없다.
- 지금 만들 수 없는 큰 요구사항이 섞여 있을 수 있다.

그래서 자연어만 받지 않는다.

자연어와 선택값을 같이 받는다.

---

## 2. 사용자가 입력하는 것

사용자는 아래 값을 입력하거나 고른다.

```
무엇을 만들고 싶나요?
데이터베이스가 포함된 백엔드 서버를 만들고 싶어요.

용도
정적 웹사이트 / 서버 / 데이터베이스 포함 백엔드 / 잘 모르겠음

예산
낮게 / 보통

트래픽
작음 / 보통

보안 우선순위
기본 / 높음
```

여기서 중요한 점:

- 사용자가 버튼으로 고른 값은 자연어보다 우선한다.
- 사용자가 잘 모르겠음을 고르면 자연어를 보고 우리가 고른다.
- 지금 만들기 어려운 요구는 실패시키지 않고 경고로 알려준다.

---

## 3. 우리가 돌려주는 것

우리는 크게 네 가지를 돌려준다.

### 설계도 자료

보드가 열 수 있는 설계도 자료다.

여기에는 이런 정보가 들어간다.

- 어떤 자원이 있는지
- 자원 이름은 무엇인지
- 자원끼리 어떻게 연결되는지
- 각 자원에 필요한 기본 설정은 무엇인지

예를 들면 데이터베이스 포함 백엔드는 이런 자원을 가질 수 있다.

- 가상 네트워크
- 서브넷
- 서버
- 데이터베이스
- 보안 그룹

### 선택 이유

왜 이 설계가 나왔는지 설명한다.

예를 들면:

```
입력에 데이터베이스라는 말이 있어서 데이터베이스 포함 백엔드로 판단했습니다.
```

### 경고

조심해야 할 내용을 알려준다.

예를 들면:

```
입력 문장과 선택한 용도가 다릅니다.
선택한 용도를 우선해서 초안을 만들었습니다.
```

또는:

```
이 요구사항은 지금 자동 초안 범위를 벗어납니다.
기본 설계에서 시작한 뒤 보드에서 직접 수정하세요.
```

### 배포 전 점검 결과

배포 전에 조심해야 할 점을 알려준다.

예를 들면:

- 보안 그룹이 너무 열려 있음
- 데이터베이스 비용이 커질 수 있음
- 정리 계획이 없으면 비용이 남을 수 있음
- 설정이 부족함

---

## 4. 꼭 지킬 원칙

### 설계도 자료가 중심이다

우리 결과의 핵심은 설계도 자료다.

설명, 점수, 경고는 부가 정보다.

보드나 다른 파트는 설계도 자료만 있어도 동작할 수 있어야 한다.

### 마음대로 복잡한 구조를 만들지 않는다

지금 지원하는 설계는 세 가지다.

- 정적 웹사이트
- 서버
- 데이터베이스 포함 백엔드

이 범위를 넘어가는 요청은 경고로 처리한다.

예를 들면 아래 요구는 지금 범위 밖이다.

- 여러 지역에 배포
- 쿠버네티스 사용
- 대규모 서비스 자동 설계
- 금융권 수준 보안 자동 설계
- 여러 클라우드 동시 지원

### 사용자가 직접 고른 값이 더 강하다

사용자가 문장에는 데이터베이스를 적었더라도, 버튼으로 정적 웹사이트를 골랐다면 정적 웹사이트를 만든다.

대신 조용히 넘어가지 않는다.

문장과 선택값이 다르다는 경고를 같이 보여준다.

### 실제 배포는 하지 않는다

이 기능은 설계 초안을 만들고, 위험을 설명하는 역할이다.

실제 배포는 다른 파트가 맡는다.

---

## 5. 다른 파트와 연결되는 부분

### 보드 파트

보드 파트는 우리가 만든 설계도 자료를 받아서 화면에 보여준다.

그래서 자원 이름표와 연결 정보가 깨지면 안 된다.

### 테라폼 변환 파트

테라폼 변환 파트는 설계도 자료를 보고 코드를 만든다.

그래서 자원 설정 이름을 마음대로 바꾸면 안 된다.

### 배포 파트

배포 파트는 배포 전에 점검 결과를 확인할 수 있다.

위험이 크면 사용자에게 먼저 보여줘야 한다.

### 플랫폼 파트

플랫폼 파트는 프로젝트 목록이나 상세 화면에서 우리 결과를 보여줄 수 있다.

하지만 지금은 모든 분석 결과를 저장하라고 강제하지 않는다.

---

## 6. 확인해야 하는 것

이 기능이 끝났다고 말하려면 아래를 확인해야 한다.

- 자연어와 선택값을 넣으면 설계도 자료가 나온다.
- 직접 고른 용도가 자연어보다 우선한다.
- 자연어와 선택값이 다르면 경고가 나온다.
- 지금 만들 수 없는 요구는 실패가 아니라 경고로 처리된다.
- 예산, 트래픽, 보안 우선순위가 설명과 설정에 반영된다.
- 배포 전 점검에서 비용, 보안, 설정 관련 결과가 나온다.
- 외부 인공지능 연결이 없어도 기본 결과가 나온다.
- 보드, 테라폼 변환, 배포 파트가 받을 수 있는 모양을 유지한다.
- 작업 화면에서 직접 눌러볼 수 있다.

---

## 7. 이번 범위가 아닌 것

이번에는 아래를 하지 않는다.

- 실제 배포
- 실제 배포 적용
- 테라폼 코드 전체 생성 책임
- 깃허브 저장소 전체 분석
- 쿠버네티스 자동 설계
- 여러 지역 자동 배포 설계
- 여러 클라우드 지원
- 실제 비용 보장
- 모든 분석 결과 데이터베이스 저장
- 최종 보드 화면 완성
- 설계 자동 수정
- 자동 롤백

이 내용들은 필요하면 고도화 문서에 따로 남긴다.

---

## 8. 내가 설명할 때 한 줄 요약

```
내 파트는 사용자의 자연어와 선택값을 받아서,
보드가 열 수 있는 안전한 설계 초안을 만들고,
배포 전에 비용과 보안 위험을 쉽게 설명하는 역할이다.
```
