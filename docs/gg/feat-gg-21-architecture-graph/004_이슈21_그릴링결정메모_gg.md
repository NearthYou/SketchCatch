# 이슈 21 그릴링 결정 메모

이 문서는 이슈 #21에서 결정한 내용을 다시 볼 때 쓰는 참고 메모다.

현재는 `feat-gg-21-architecture-graph` 브랜치 폴더 안에 보관한다.

## 1. 인프라 그래프 MVP 범위

결정: **A. MVP에서는 `ArchitectureJson`만 만든다.**

이슈 #21의 "인프라 그래프"는 이번 브랜치에서는 새 타입 `InfrastructureGraph`가 아니라, Architecture Board가 바로 열 수 있는 `ArchitectureJson`으로 제한한다.

구현 기준:

- 자연어 Requirement Prompt를 분석해 `ArchitectureJson.nodes`와 `ArchitectureJson.edges`를 만든다.
- 예산, 트래픽, 보안 우선순위 같은 조건은 그래프 구조에 직접 섞지 않는다.
- 조건 해석 결과는 `metadata.assumptions` 또는 `metadata.explanations`에 둔다.
- `ResourceNode.config`에는 Terraform 생성기가 오해할 수 있는 임의 조건값을 넣지 않는다.
- 후속 Design Simulation이나 AI 수정 제안에서 더 큰 그래프 타입이 필요하면 별도 이슈에서 확장한다.

이유:

- jh Architecture Board가 바로 열 수 있는 형태가 우선이다.
- sw Terraform 변환 파트가 `ArchitectureJson`을 원천 입력으로 쓰기로 했기 때문에 새 wrapper 타입을 바로 만들면 연결 비용이 커진다.
- 이번 이슈는 자연어 요구사항을 보드 가능한 그래프로 바꾸는 1차 제공에 집중한다.

## 2. Requirement Prompt 입력 형식

결정: **자연어 입력 + 제한된 선택값을 함께 받는다.**

화면 입력 형태:

```text
무엇을 만들고 싶나요?
[ DB가 포함된 백엔드 API 서버를 만들고 싶어요 ]

용도 선택
[ 정적 웹사이트 ] [ API 서버 ] [ DB 포함 백엔드 ] [ 잘 모르겠음 ]

예산
[ 낮게 ] [ 보통 ]

트래픽
[ 작음 ] [ 보통 ]

보안 우선순위
[ 기본 ] [ 높음 ]
```

API 입력 초안:

```ts
type CreateArchitectureDraftRequest = {
  prompt: string;
  scenarioHint: "auto" | "static_site" | "api_server" | "backend_with_db";
  budgetLevel: "low" | "normal";
  trafficLevel: "small" | "normal";
  securityPriority: "basic" | "high";
};
```

구현 기준:

- `prompt`는 사용자가 원하는 Practice Architecture를 자유롭게 설명하는 입력이다.
- `scenarioHint`는 template 선택 가드레일이다.
- `scenarioHint: "auto"`는 화면의 "잘 모르겠음"에 대응한다.
- `budgetLevel`, `trafficLevel`, `securityPriority`는 MVP에서 Resource를 과하게 자동 튜닝하는 값이 아니다.
- 선택값은 우선 `metadata.assumptions`, `metadata.explanations`, fallback 안내, Pre-Deployment Check 설명에 반영한다.
- `budgetLevel: "low"`라고 해서 AI가 임의로 고급 Terraform 설정을 만들지는 않는다.
- `securityPriority: "high"`라고 해서 보안 적합성을 보장한다고 말하지 않는다.

이유:

- 자연어 하나만 받으면 사용자의 애매한 표현을 너무 많이 추측해야 한다.
- 선택값이 있으면 MVP 지원 범위를 UI와 API에서 동시에 제한할 수 있다.
- 팀 발표 때 "아무 AWS 인프라나 자동 설계"가 아니라 "지원 시나리오 안에서 Practice Architecture 초안 생성"이라고 설명하기 쉽다.
- 나중에 고도화할 때 runtime, region, databaseType, availabilityLevel 같은 선택값을 추가하기 쉽다.

## 3. `scenarioHint: "auto"` 선택 규칙

결정: **B. 시나리오별 confidence 점수로 template을 고른다.**

`scenarioHint`가 `"auto"`일 때는 Requirement Prompt를 보고 각 지원 시나리오에 점수를 부여한다.

지원 시나리오:

- `static_site`
- `api_server`
- `backend_with_db`

MVP 점수 규칙 초안:

```ts
type ScenarioScore = {
  scenario: "static_site" | "api_server" | "backend_with_db";
  score: number;
  reasons: string[];
};
```

- DB 단서가 있으면 `backend_with_db` 점수 증가
  - 예: `db`, `database`, `데이터베이스`, `rds`, `postgres`, `mysql`, `백엔드`
- API/서버 단서가 있으면 `api_server` 점수 증가
  - 예: `api`, `서버`, `server`, `ec2`, `express`, `spring`
- 정적 웹사이트 단서가 있으면 `static_site` 점수 증가
  - 예: `정적`, `static`, `웹사이트`, `프론트`, `frontend`, `react`, `next`
- DB 단서와 API/서버 단서가 같이 있으면 `backend_with_db`를 우선한다.
- 모든 점수가 0이면 `static_site`를 fallback으로 선택한다.

구현 기준:

- 점수 계산은 LLM 자유 추론이 아니라 deterministic keyword scoring으로 시작한다.
- 같은 입력은 항상 같은 시나리오와 같은 점수 결과를 반환해야 한다.
- 선택된 시나리오와 점수 근거는 metadata에 남긴다.
- 점수가 낮거나 fallback이면 metadata에 "지원 시나리오와 정확히 일치하지 않아 기본 template으로 시작했다"는 설명을 포함한다.
- 점수는 사용자에게 "정답률"처럼 말하지 않는다. `confidence`는 내부 선택 안정성 또는 안내 문구용으로만 사용한다.

이유:

- 키워드 우선순위 하나보다 고도화 여지가 있다.
- 그래도 deterministic하게 만들 수 있어 API 테스트가 가능하다.
- 나중에 LLM 의도 분류를 붙일 때도 기존 scoring 결과와 비교하거나 fallback 기준으로 활용할 수 있다.

## 4. 지원하지 않는 Requirement Prompt 처리

결정: **B. fallback은 하되, 범위 초과 warning을 metadata에 명확히 남긴다.**

MVP에서 바로 지원하지 않는 요구사항 예:

- 멀티 리전
- Kubernetes/EKS
- 금융권/의료 수준 규제 준수
- 대규모 트래픽 자동 설계
- 운영 자동화/CI/CD
- 실제 비용 정확도 보장
- 실제 보안 적합성 보장
- 회사 내부 시스템 전체 설계

처리 방식:

- Architecture Draft 생성을 완전히 막지는 않는다.
- 가장 가까운 지원 시나리오 template을 선택한다.
- `metadata.assumptions` 또는 `metadata.explanations`에 범위 초과 항목을 명시한다.
- 지원하지 않는 요구사항을 만족했다고 말하지 않는다.
- UI에서는 "기본 초안으로 시작" 또는 "수동 수정 필요" 안내를 보여줄 수 있게 한다.

예시 문구:

```text
입력에 MVP 자동 초안 범위를 벗어난 요구사항이 포함되어 있습니다.
이번 초안은 지원 가능한 기본 Practice Architecture template으로 시작하며,
멀티 리전, Kubernetes, 규제 준수 설계는 수동 검토가 필요합니다.
```

구현 기준:

- unsupported keyword 목록을 별도로 둔다.
- unsupported keyword가 감지되어도 HTTP error로 실패시키지 않는다.
- 대신 응답 metadata에 warning 성격의 explanation을 추가한다.
- 지원 범위 밖이라는 이유로 Terraform 실행 가능성이나 보안 보장을 말하지 않는다.

이유:

- 데모 흐름을 끊지 않으면서도 과장된 자동 설계처럼 보이지 않게 한다.
- 사용자는 보드에서 기본 초안을 보고 수정할 수 있다.
- 팀 발표에서 "가드레일 안에서 시작점을 제안한다"는 메시지를 유지할 수 있다.

## 5. 선택값이 `ArchitectureJson`에 미치는 영향

결정: **C. 선택값에 따라 구조까지 바꾸되, 지원 Resource와 미리 정한 구조 선택지 안에서만 허용한다.**

입력 선택값:

```ts
budgetLevel: "low" | "normal";
trafficLevel: "small" | "normal";
securityPriority: "basic" | "high";
```

구조 변경 허용 기준:

- 구조 변경은 `ArchitectureJson.nodes`와 `ArchitectureJson.edges` 안에서만 한다.
- 새 wrapper 타입 `InfrastructureGraph`는 만들지 않는다.
- 지원하지 않는 Resource를 임의로 만들지 않는다.
- Terraform 생성기가 아직 처리하지 못할 가능성이 높은 Resource는 추가하지 않는다.
- 변경 이유는 metadata에 남긴다.

허용하는 구조 변경 초안:

- `scenarioHint: "static_site"`
  - 기본: S3 + CloudFront
  - `securityPriority: "high"`: CloudFront 사용 이유와 S3 public access 주의 설명 강화
  - `trafficLevel`은 구조 변경보다 설명/assumption 중심으로 처리
- `scenarioHint: "api_server"`
  - 기본: VPC + Subnet + EC2 + Security Group
  - `securityPriority: "high"`: SSH open warning을 더 강하게 남기고, Security Group config를 더 보수적인 기본값으로 둔다.
  - `trafficLevel: "normal"`: MVP에서는 ALB를 자동 추가하지 않는다. ALB는 후순위 variant 후보로 둔다.
- `scenarioHint: "backend_with_db"`
  - 기본: VPC + Subnet + EC2 + RDS + Security Group
  - `budgetLevel: "low"`: RDS 비용 warning을 metadata와 Pre-Deployment Check에서 강하게 표시한다.
  - `securityPriority: "high"`: DB 공개 접근 금지와 Security Group 연결 설명을 강화한다.

아직 자동 구조 변경으로 하지 않는 것:

- ALB 자동 추가
- NAT Gateway 자동 추가
- Multi-AZ RDS 자동 추가
- private subnet/public subnet 정교한 분리
- EKS/Kubernetes 추가
- 멀티 리전 추가

이유:

- 선택값이 실제 초안에 영향을 줘야 사용자가 "내 조건이 반영됐다"고 느낀다.
- 하지만 이번 MVP에서 구조 변경을 무제한으로 열면 jh 보드, sw Terraform 변환, ck 배포 흐름과 충돌할 수 있다.
- 따라서 구조 변경은 미리 정한 구조 선택지 안에서만 허용한다.

## 6. Resource id 생성 규칙

결정: **Resource type + 역할로 만든다.**

여기서 정하는 것은 template id가 아니라 `ArchitectureJson.nodes[].id`다.

예:

```text
vpc-main
subnet-public
ec2-api
rds-primary
sg-api
s3-site
cloudfront-site
```

구현 기준:

- Resource id는 사람이 읽을 수 있어야 한다.
- Resource id는 Resource 종류와 역할이 드러나야 한다.
- 같은 초안 안에서 id가 겹치면 안 된다.
- `CheckFinding.resourceId`, 비용 분석, AI 수정 제안은 이 id로 Resource를 가리킨다.
- template id와 Resource id를 섞어 설명하지 않는다.

이유:

- 경고나 비용 설명을 특정 Resource에 붙이기 쉽다.
- 보드, Pre-Deployment Check, AI 수정 제안이 같은 Resource를 같은 이름으로 가리킬 수 있다.
- 랜덤 id보다 디버깅과 발표 설명이 쉽다.

## 7. `scenarioHint` 직접 선택과 prompt 충돌 처리

결정: **A. 사용자가 직접 고른 `scenarioHint`를 우선한다.**

예:

```json
{
  "prompt": "정적 웹사이트 만들고 싶어요",
  "scenarioHint": "backend_with_db"
}
```

이 경우 `backend_with_db` 초안을 만든다.

구현 기준:

- `scenarioHint`가 `"static_site"`, `"api_server"`, `"backend_with_db"` 중 하나면 해당 값을 우선한다.
- `scenarioHint: "auto"`일 때만 자연어 점수 계산으로 시나리오를 고른다.
- 직접 선택한 값과 prompt 단서가 충돌하면 생성은 계속한다.
- 충돌 사실은 `metadata.explanations`에 남긴다.
- 사용자에게 "선택한 용도를 우선해 초안을 만들었다"는 식으로 설명한다.

예시 문구:

```text
선택한 용도는 DB 포함 백엔드였지만, 입력 문장에는 정적 웹사이트 단서가 포함되어 있습니다.
선택한 용도를 우선해 초안을 만들었습니다.
```

이유:

- 사용자가 버튼으로 명시적으로 선택한 값을 존중한다.
- 입력 문장이 애매해도 사용 흐름이 끊기지 않는다.
- 충돌은 숨기지 않고 설명으로 남긴다.

## 8. Architecture Draft metadata 확장

결정: **B. metadata를 조금 확장한다.**

현재 기본 metadata:

```ts
type AiResultMetadata = {
  source: AiResultSource;
  confidence: AiConfidence;
  assumptions: string[];
  explanations: string[];
};
```

확장 초안:

```ts
type ArchitectureScenario = "static_site" | "api_server" | "backend_with_db";

type ScenarioScore = {
  scenario: ArchitectureScenario;
  score: number;
  reasons: string[];
};

type GuardrailWarning = {
  code: string;
  message: string;
};

type AiResultMetadata = {
  source: AiResultSource;
  confidence: AiConfidence;
  assumptions: string[];
  explanations: string[];
  selectedScenario?: ArchitectureScenario;
  scenarioScores?: ScenarioScore[];
  guardrailWarnings?: GuardrailWarning[];
};
```

metadata에 남길 것:

- 최종 선택된 용도
- `auto`일 때 시나리오별 점수
- 점수 근거
- 지원 범위 밖 요구사항
- prompt와 직접 선택한 용도가 충돌했는지
- fallback이 발생했는지

구현 기준:

- UI가 꼭 써야 하는 핵심 결과는 여전히 `architectureJson`이다.
- metadata는 설명과 경고 표시를 돕는 부가 정보다.
- shared type인 `packages/types/src/index.ts` 수정이 필요할 수 있다.
- metadata가 없어도 Architecture Board가 깨지면 안 된다.

이유:

- 사용자가 왜 이 초안이 나왔는지 이해할 수 있다.
- auto 선택 결과를 설명할 수 있다.
- 지원하지 않는 요구사항을 조용히 무시하지 않는다.
- 나중에 UI에서 warning badge나 선택 근거를 보여주기 쉽다.

## 9. `/workspace/ai` 데모 UI 반영 범위

결정: **B. `/workspace/ai` 데모 UI도 같이 바꾼다.**

이번 이슈에서 바꿀 UI 입력:

- 자연어 입력
- 용도 선택
  - 정적 웹사이트
  - API 서버
  - DB 포함 백엔드
  - 잘 모르겠음
- 예산
  - 낮게
  - 보통
- 트래픽
  - 작음
  - 보통
- 보안 우선순위
  - 기본
  - 높음

구현 기준:

- `/workspace/ai`는 팀 내부 확인용 데모 UI다.
- `/workspace`는 실제 Architecture Board 작업대 화면이다.
- 실제 최종 Architecture Board UI 확정은 jh/ys 파트와 연결될 때 다시 맞춘다.
- 이번 브랜치에서는 새 API 입력 형식을 사람이 눌러 테스트할 수 있게 만든다.
- UI는 선택값을 API body에 그대로 보낸다.
- 결과 화면에는 `selectedScenario`, `scenarioScores`, `guardrailWarnings`를 확인할 수 있게 한다.

이유:

- API만 바꾸면 팀원이 기능을 확인하기 어렵다.
- 선택 입력이 실제로 API에 들어가는지 바로 볼 수 있다.
- 발표나 PR 리뷰 때 "자연어 + 선택값" 흐름을 설명하기 쉽다.
