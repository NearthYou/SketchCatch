## 전체 흐름 먼저 보기

이번 브랜치의 실제 기능 흐름은 아래 순서로 이어진다.

1. `/workspace/ai`에서 gg AI 데모 화면을 연다.
2. Requirement Prompt 또는 Source Repository로 Architecture Draft를 만든다.
3. 생성된 Infrastructure Graph를 Pre-Deployment Check에 보낸다.
4. Pre-Deployment Check는 Check Finding, checklist, Architecture Suggestion을 반환한다.
5. 같은 Infrastructure Graph를 Design Simulation에 보낸다.
6. Design Simulation은 요청 흐름, 병목 후보, 장애 노출, Cost Risk 압박을 추정한다.
7. 화면은 결과를 보여주되 Architecture Board를 자동 수정하거나 Terraform을 실행하지 않는다.

중요한 경계는 하나다. 이번 작업은 “사용자가 검토할 수 있는 분석 정보”를 추가한 것이지, 실제 AWS 실행이나 자동 수정 기능을 추가한 것이 아니다.

## `d2c5b39` Feat: Pre-Deployment Check 수정 제안 추가

### 왜 필요한가

기존 Pre-Deployment Check는 Check Finding과 checklist를 보여줬다. 이 상태에서는 사용자가 “문제가 있다는 건 알겠는데, 그래서 다음에 무엇을 봐야 하지?”를 직접 해석해야 했다.

이 커밋은 Check Finding을 기반으로 `ArchitectureSuggestion`을 함께 내려준다. 즉, 문제 관찰에서 끝나지 않고 “검토할 수정 방향”까지 붙인다.

### 공유 타입 변화

`packages/types/src/index.ts`에 아래 타입이 추가됐다.

- `ArchitectureSuggestionAction`
- `ArchitectureSuggestionCostImpact`
- `ArchitectureSuggestionQualityImpact`
- `ArchitectureSuggestionExpectedImpact`
- `ArchitectureSuggestion`

`AiPreDeploymentAnalysisResult`에는 `suggestions` 필드가 추가됐다.

`ArchitectureSuggestion`은 대략 이런 의미를 가진다.

- `id`: 제안 자체의 ID
- `findingId`: 어떤 Check Finding에서 나온 제안인지
- `targetResourceId`: 어떤 Resource를 봐야 하는지
- `action`: `modify_resource`, `manual_review` 같은 검토 행동
- `expectedImpact`: 비용, 보안, 신뢰성에 어떤 영향이 예상되는지
- `explanation`: 사용자가 읽을 설명

### API 동작 변화

`apps/api/src/services/aiPreDeploymentAnalysis.ts`의 `analyzePreDeployment` 결과가 아래처럼 확장됐다.

1. Resource별 Check Finding 생성
2. Check Finding 기반 checklist 생성
3. Check Finding 기반 Architecture Suggestion 생성
4. 세 결과를 한 응답에 포함

중요한 함수는 `createSuggestionForFinding`이다.

category별 제안 규칙:

- `security`: SSH 접근 범위 제한, `modify_resource`, security 개선
- `cost`: 연습 비용 낮추기 검토, `manual_review`, cost 감소 가능
- `configuration`: 필수 Resource 설정 입력, `modify_resource`, reliability 개선
- `permission`, `network`, `performance`, `availability`: 자동 방향을 정하지 않고 `manual_review`

### 안전 경계

`ArchitectureSuggestion`은 이름 그대로 제안이다. Architecture Board의 Resource를 직접 수정하지 않는다.

예를 들어 Security Group이 SSH를 `0.0.0.0/0`으로 열고 있으면 아래처럼 말한다.

- 대상 Resource: 해당 Security Group
- 원본 Check Finding: open SSH finding
- 제안 행동: `modify_resource`
- 예상 영향: security 개선
- 설명: 관리용 CIDR만 남기라는 검토 문장

하지만 실제 `ArchitectureJson`을 바꾸거나 Terraform 코드를 수정하지 않는다.

### 테스트가 확인하는 것

추가 테스트:

- `apps/api/src/routes/aiPreDeploymentSuggestions.test.ts`

테스트는 `POST /api/ai/pre-deployment-check`에 SSH 전체 공개 Security Group을 넣고 아래를 확인한다.

- 응답 status가 200인가
- security Check Finding이 생기는가
- 그 Check Finding과 연결된 suggestion이 생기는가
- suggestion의 `targetResourceId`가 같은 Security Group인가
- action이 `modify_resource`인가
- `expectedImpact.security`가 `improve`인가

### 리뷰 포인트

- suggestion이 원본 Check Finding과 끊기지 않는가
- action 이름이 자동 실행처럼 오해되지 않는가
- `manual_review`가 필요한 category를 무리하게 자동 수정 방향으로 만들지 않는가
- 프론트가 `AiPreDeploymentAnalysisResult.suggestions` 추가를 놓치지 않았는가

## `ef634e0` Feat: Design Simulation API 추가

### 왜 필요한가

Pre-Deployment Check는 주로 비용, 보안, 설정 누락을 본다. 하지만 사용자는 Practice Architecture가 실제 요청을 받을 때 어디가 막힐지, 어떤 Resource 장애가 크게 번질지, 낮은 예산에서 어떤 Resource가 부담이 될지도 보고 싶다.

이 커밋은 그 역할을 `Design Simulation`으로 분리했다.

### API 계약

추가 endpoint:

- `POST /api/ai/design-simulation`

요청 타입:

- `CreateDesignSimulationRequest`

요청 필드:

- `architectureJson`: 분석할 Infrastructure Graph
- `trafficLevel`: `small` 또는 `normal`
- `budgetLevel`: `low` 또는 `normal`

route schema에서는 `trafficLevel`과 `budgetLevel`이 빠지면 기본값으로 `normal`을 쓴다.

### 응답 타입

추가된 주요 타입:

- `DesignSimulationRequestFlowStep`
- `DesignSimulationBottleneck`
- `DesignSimulationFailureScenario`
- `DesignSimulationResult`

`DesignSimulationResult`는 아래 정보를 반환한다.

- `summary`: Resource와 Edge 수 기반 요약
- `assumptions`: 실제 부하 테스트가 아니라는 전제
- `requestFlow`: Edge를 요청 흐름 문장으로 바꾼 결과
- `bottlenecks`: 단일 EC2, 단일 RDS 같은 병목 후보
- `failureScenarios`: 장애가 크게 번질 수 있는 Resource
- `costPressure`: 낮은 예산에서 비용 압박이 생길 수 있는 Resource
- `recommendations`: 다음 검토 행동

### 내부 동작

핵심 파일:

- `apps/api/src/services/aiDesignSimulation.ts`

주요 함수 흐름:

1. `createResourceLookup`
   - Resource ID로 Resource를 빠르게 찾는 Map 생성
2. `createRequestFlow`
   - Edge의 `sourceId`, `targetId`를 사용자 문장으로 변환
3. `createBottlenecks`
   - 보통 트래픽에서 EC2가 하나뿐이면 병목 후보 생성
   - RDS가 하나뿐이면 DB 연결 집중 후보 생성
4. `createFailureScenarios`
   - EC2가 하나뿐이면 API 응답 중단 가능성 생성
   - RDS가 하나뿐이면 읽기/쓰기 중단 가능성 생성
5. `createCostPressure`
   - RDS와 EC2가 비용 압박을 만들 수 있음을 설명
6. `createRecommendations`
   - 병목, 장애, 비용 결과를 다음 검토 문장으로 압축

### 안전 경계

이 커밋의 `Design Simulation`은 실제 simulation engine이 아니다. 이름은 simulation이지만 현재 MVP에서는 Infrastructure Graph 기반 추정이다.

하지 않는 일:

- AWS 접속
- 실제 트래픽 생성
- 벤치마크 실행
- Terraform validate/plan/apply 실행
- CloudWatch 같은 운영 지표 조회

그래서 응답의 `assumptions`에 “실제 부하 테스트가 아닌 ArchitectureJson 기반 추정”이라고 명시한다.

### 테스트가 확인하는 것

추가 테스트:

- `apps/api/src/routes/aiDesignSimulation.test.ts`

테스트는 SUBNET -> EC2 -> RDS 구조를 보내고 아래를 확인한다.

- 실제 부하 테스트가 아니라는 assumption 포함
- EC2에서 RDS로 가는 request flow 포함
- 단일 EC2 bottleneck 포함
- 단일 RDS failure scenario 포함
- RDS cost pressure 포함
- EC2 관련 recommendation 포함

### 리뷰 포인트

- route schema와 공유 타입이 같은 계약을 말하는가
- `Design Simulation` 문장이 실제 성능 측정처럼 보이지 않는가
- `trafficLevel`, `budgetLevel`이 결과에 반영되는가
- Edge가 끊겨 있어도 fallback 설명이 비지 않는가

## `f8d7ee0` Feat: Design Simulation public 노출 시나리오 추가

### 왜 필요한가

초기 Design Simulation은 단일 EC2, 단일 RDS처럼 availability와 capacity 중심의 위험을 봤다. 하지만 Practice Architecture에서 SSH가 전체 인터넷에 열려 있으면, 성능 문제가 아니어도 운영 부담과 Security Risk가 커질 수 있다.

이 커밋은 Security Group의 SSH 전체 공개를 Design Simulation의 failure scenario로도 보여준다.

### 탐지 조건

핵심 함수:

- `hasPublicExposure`
- `hasOpenSshRule`
- `isOpenSshRule`
- `isRecord`

현재 MVP에서 잡는 조건은 매우 좁다.

1. Resource type이 `SECURITY_GROUP`
2. `config.ingress`가 배열
3. ingress rule이 객체
4. `port`가 `22` 또는 `"22"`
5. `cidr`이 `0.0.0.0/0`

이 조건을 만족하면 아래 ID 패턴의 failure scenario가 만들어진다.

- `failure-public-exposure-{resourceId}`

### 왜 좁게 잡았나

Security Group 규칙은 실제로 훨씬 다양한 형태가 가능하다. 하지만 현재 MVP에서 모든 보안 노출을 다 잡겠다고 하면 오탐이 많아지고, 팀원에게 실제 보안 스캐너처럼 보일 수 있다.

그래서 가장 확실한 위험인 SSH 전체 공개만 잡는다.

### 반환되는 설명

사용자에게는 아래 방향으로 설명한다.

- 전체 인터넷에 열린 접근 규칙은 접속 시도와 점검 부담을 늘릴 수 있음
- 본인 IP나 팀 관리용 CIDR로 줄이는 방안을 먼저 검토

여기서도 침해 여부를 단정하지 않는다. “가능성”과 “검토”로만 표현한다.

### 테스트가 확인하는 것

`apps/api/src/routes/aiDesignSimulation.test.ts`에 public exposure 테스트가 추가됐다.

테스트 입력:

- `SECURITY_GROUP`
- ingress port `22`
- cidr `0.0.0.0/0`

확인:

- 응답의 `failureScenarios` 안에 해당 Security Group ID가 들어가는가

### 리뷰 포인트

- unknown config를 안전하게 좁힌 뒤 읽는가
- `port`가 number와 string 둘 다 처리되는가
- 현재 구현이 “전체 보안 스캔”처럼 보이지 않는가
- Pre-Deployment Check의 security finding과 Design Simulation의 failure scenario가 서로 역할을 침범하지 않는가

## `5f66ed0` Feat: Pre-Deployment Check checklist 표시 보강

### 왜 필요한가

API가 Check Finding, checklist, Architecture Suggestion을 잘 내려줘도 화면에서 연결 정보가 사라지면 리뷰가 어렵다.

예를 들어 “SSH 접근 범위 제한”이라는 suggestion만 보이면 사용자는 아래를 다시 찾아야 한다.

- 어떤 Resource를 말하는가
- 어떤 Check Finding에서 나온 제안인가
- checklist의 fail이 어떤 finding과 연결되는가

이 커밋은 그 연결 정보를 화면 문장에 같이 보여준다.

### UI 출력 변화

`PreDeploymentAnalysisPanel`에서 helper 함수를 분리했다.

- `createPreDeploymentFindingItems`
- `createPreDeploymentChecklistItems`
- `createPreDeploymentSuggestionItems`

각 함수는 `ResultList`가 받을 `{ id, label, text }` 형태로 결과를 만든다.

### 표시되는 정보

Check Finding:

- severity
- title
- description
- 연결 Resource ID

Checklist:

- status
- label
- 연결 Check Finding ID 목록

Architecture Suggestion:

- action
- title
- explanation
- 대상 Resource ID
- 연결 Check Finding ID

### 안전 경계

이 커밋은 표시만 바꾼다.

하지 않는 일:

- suggestion 클릭 적용
- Resource config 자동 수정
- checklist status 자동 해결
- Architecture Board 저장

사용자가 직접 확인해야 할 연결 정보를 더 잘 보여주는 것이 목적이다.

### 테스트가 확인하는 것

추가 테스트:

- `apps/web/features/workspace/pre-deployment-analysis-panel.test.ts`

테스트는 helper 함수별로 아래 출력을 확인한다.

- finding text에 `연결 Resource: sg-public-ssh`가 포함되는가
- checklist text에 `연결 finding: security-open-ssh-sg-public-ssh`가 포함되는가
- suggestion text에 대상 Resource와 연결 finding이 함께 포함되는가

### 리뷰 포인트

- 화면 문장이 너무 길어져도 핵심 연결 정보가 보이는가
- 연결 ID가 없는 경우 fallback 문장이 자연스러운가
- helper 함수가 순수 함수라 테스트하기 쉬운가
- suggestion이 자동 수정 버튼처럼 보이지 않는가
