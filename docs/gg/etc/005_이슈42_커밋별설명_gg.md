## `d2c5b39` Feat: Pre-Deployment Check 수정 제안 추가

핵심 커밋이다.

기존 Pre-Deployment Check는 비용, 보안, 설정 문제를 finding과 checklist로 보여주는 수준이었다. 이 커밋은 finding을 사람이 검토할 수 있는 `ArchitectureSuggestion`으로 한 단계 더 바꿔 내려준다.

추가된 타입:

- `ArchitectureSuggestion`
- `ArchitectureSuggestionAction`
- `ArchitectureSuggestionExpectedImpact`
- `AiPreDeploymentAnalysisResult.suggestions`

API 결과가 이제 이런 흐름을 가진다.

1. `ArchitectureJson`의 Resource를 검사
2. `CheckFinding` 생성
3. finding category별 checklist 생성
4. 같은 finding을 기반으로 수정 제안 생성

제안은 자동 실행 명령이 아니다. 예를 들어 Security Group의 SSH가 `0.0.0.0/0`으로 열려 있으면, 제안은 “관리용 CIDR만 남기라”는 설명과 `security: improve` 같은 예상 영향만 담는다.

이 제한이 중요하다. 현재 gg AI는 설계를 멋대로 고치거나 Terraform을 실행하지 않고, 사용자가 판단할 수 있는 검토 정보를 만든다.

확인 포인트:

- `packages/types/src/index.ts`
- `apps/api/src/services/aiPreDeploymentAnalysis.ts`
- `apps/api/src/routes/aiPreDeploymentSuggestions.test.ts`

## `ef634e0` Feat: Design Simulation API 추가

핵심 커밋이다.

`POST /api/ai/design-simulation` API를 추가했다. 입력은 `ArchitectureJson`, `trafficLevel`, `budgetLevel`이다.

반환 결과는 아래 항목으로 나뉜다.

- `summary`: Resource와 Edge 개수 기반 요약
- `assumptions`: 실제 부하 테스트가 아니라는 전제
- `requestFlow`: Edge를 사용자 문장으로 바꾼 요청 흐름
- `bottlenecks`: 단일 EC2, 단일 RDS 같은 병목 후보
- `failureScenarios`: 단일 장애 지점으로 볼 수 있는 Resource
- `costPressure`: 낮은 예산에서 비용 압박이 생길 수 있는 Resource
- `recommendations`: 다음에 검토할 행동

이 커밋에서 제일 중요한 경계는 “시뮬레이션이라는 이름이지만 실제 실행은 하지 않는다”는 점이다.

현재 구현은 `ArchitectureJson`만 보고 추정한다. AWS에 접속하지 않고, 트래픽을 만들지 않고, Terraform도 실행하지 않는다. 그래서 MVP 안에서 안전하게 데모할 수 있다.

확인 포인트:

- `packages/types/src/index.ts`
- `apps/api/src/routes/ai.ts`
- `apps/api/src/services/aiDesignSimulation.ts`
- `apps/api/src/routes/aiDesignSimulation.test.ts`

## `f8d7ee0` Feat: Design Simulation public 노출 시나리오 추가

핵심 커밋이다.

초기 Design Simulation은 단일 EC2, 단일 RDS 같은 병목과 장애 후보를 중심으로 봤다. 이 커밋은 Security Group의 SSH 전체 공개도 실패 시나리오로 보여주게 확장했다.

현재 잡는 조건:

- Resource type이 `SECURITY_GROUP`
- `config.ingress` 배열 존재
- rule의 `port`가 `22` 또는 `"22"`
- rule의 `cidr`이 `0.0.0.0/0`

이 조건을 만족하면 `failure-public-exposure-{resourceId}` 시나리오를 만든다.

이것도 실제 침해 탐지나 보안 스캔이 아니다. 보드에 들어온 설계 정보만 보고 “이런 노출은 접속 시도와 점검 부담을 늘릴 수 있다”는 검토용 문장을 만든다.

확인 포인트:

- `apps/api/src/services/aiDesignSimulation.ts`
- `apps/api/src/routes/aiDesignSimulation.test.ts`

## `5f66ed0` Feat: Pre-Deployment Check checklist 표시 보강

핵심 커밋이다.

Pre-Deployment Check 결과에서 finding, checklist, suggestion의 연결 정보가 화면에서 사라지지 않게 보강했다.

기존 화면은 설명 문장만 보면 어떤 Resource와 어떤 finding을 말하는지 바로 확인하기 어려웠다. 이 커밋은 다음 정보를 화면 문장에 같이 붙인다.

- finding의 `resourceId`
- checklist의 `relatedFindingIds`
- suggestion의 `targetResourceId`
- suggestion의 `findingId`

이렇게 해야 리뷰어가 “이 제안이 어떤 위험에서 나온 건지”, “어떤 Resource를 봐야 하는지”를 바로 따라갈 수 있다.

역시 자동 적용은 없다. 화면은 연결 정보를 보여줄 뿐, ArchitectureJson을 직접 수정하지 않는다.

확인 포인트:

- `apps/web/app/workspace/PreDeploymentAnalysisPanel.tsx`
- `apps/web/features/workspace/pre-deployment-analysis-panel.test.ts`

## 전체 흐름

이번 브랜치는 아래 순서로 읽으면 된다.

1. `/workspace/ai`에서 gg AI 데모 진입점을 연다.
2. Architecture Draft를 만든다.
3. Pre-Deployment Check가 finding, checklist, suggestion을 반환한다.
4. Design Simulation이 같은 `ArchitectureJson`을 보고 요청 흐름, 병목, 장애 시나리오, 비용 압박을 추정한다.
5. UI가 이 결과를 자동 수정 없이 사용자가 검토할 수 있는 정보로 보여준다.

## 리뷰할 때 특히 볼 부분

- `Design Simulation`이 실제 부하 테스트처럼 오해될 문장을 만들지 않는가
- `ArchitectureSuggestion`이 자동 적용처럼 보이지 않는가
- finding, checklist, suggestion이 Resource와 finding ID를 잃지 않는가
- Terraform 관련 UI가 실제 `plan/apply` 실행으로 오해되지 않는가
- `/workspace/ai` 데모 화면이 기존 `/workspace` 작업대 흐름을 방해하지 않는가
