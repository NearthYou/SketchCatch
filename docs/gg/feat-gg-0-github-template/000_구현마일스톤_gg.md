# GitHub Repository Analysis와 Template Selection 구현 마일스톤

## 목표

사용자가 프로젝트에 연결한 GitHub Source Repository를 실행하지 않고 정적으로 분석한다. monorepo 안의 frontend/backend를 Application Unit으로 식별하고, 저장소 전체에 맞는 Template 하나를 선택해 선택 근거와 함께 AI Handoff로 반환한다. 지원 Template과 맞지 않으면 fallback 없이 Template Selection Failure를 반환하며, 화면 재진입을 위해 마지막 구조화된 결과만 저장한다.

## 책임 경계

- gg는 Repository Analysis, Template Selection, 선택 근거와 실패 이유를 포함한 AI Handoff까지 책임진다.
- AI는 gg가 선택한 Template을 바꾸지 않고 부족한 요구사항만 보완한다.
- Repository 코드를 실행하거나 port/health check를 실측하지 않는다.
- 구조화된 마지막 AI Handoff와 revision·분석 시각은 RDS에 저장하고, 원본 Repository 파일은 RDS/S3에 저장하지 않는다.
- Template 후보 목록과 confidence 점수는 반환하지 않는다.
- 다른 에이전트가 작업 중인 `TemplateDefinition`, Terraform Preview, Deployment UI 변경은 이번 작업에서 수정하지 않는다.

## 완료 기준

연결된 active GitHub Source Repository에 분석 요청을 보내면 다음 중 하나를 반환해야 한다.

- 성공: Application Unit, 감지한 evidence, 감지하지 못한 evidence, 선택한 Template 하나, 선택 근거를 포함한 AI Handoff
- 실패: `templateId: null`, Template Selection Failure 상태, 불일치 이유, 감지하지 못한 evidence를 포함한 AI Handoff

응답이 생성되는 동안 Repository 파일을 실행하지 않아야 하며, 원본 파일 없이 구조화된 마지막 결과만 영속화해야 한다.

## M1. 공통 계약

대상:

- `packages/types/src/index.ts`
- `docs/data-models.md`
- 관련 API compile/route test

결과:

- evidence 종류는 repository tree, `package.json`, lockfile, `Dockerfile`, framework config, `README`로 제한한다.
- Repository Analysis 결과는 성공/실패를 구분하는 discriminated union으로 만든다.
- AI Handoff는 성공 시 Template 하나와 근거를, 실패 시 `templateId: null`과 이유를 담는다.
- candidate list와 confidence는 타입에 존재하지 않는다.

## M2. GitHub evidence reader

대상:

- `apps/api/src/source-repositories/github-app-client.ts`
- `apps/api/src/source-repositories/github-app-client.test.ts`

결과:

- GitHub App installation token으로 default branch의 recursive tree를 읽는다.
- 분석 대상 경로만 골라 UTF-8 텍스트로 가져온다.
- 없는 evidence는 누락 목록으로 남기고 404를 서비스 오류로 바꾸지 않는다.
- truncated tree, 잘못된 encoding, GitHub API 실패는 typed error로 구분한다.

## M3. Repository Analysis와 Template Selection

대상:

- 새 `apps/api/src/source-repositories/repository-analysis.ts`
- 새 `apps/api/src/source-repositories/repository-analysis.test.ts`

결과:

- root와 workspace 경로의 `package.json`, Dockerfile, framework config를 이용해 Application Unit을 찾는다.
- 명확한 evidence만 사용해 여섯 `TemplateId` 중 하나를 선택한다.
- 둘 이상의 Template이 동등하게 가능한 경우 fallback하지 않고 Template Selection Failure를 반환한다.
- 함수는 순수하며 filesystem, child process, DB, AWS SDK를 호출하지 않는다.

## M4. 분석 API와 AI Handoff

대상:

- `apps/api/src/source-repositories/source-repository-service.ts`
- `apps/api/src/routes/source-repositories.ts`
- `apps/api/src/routes/source-repositories.test.ts`

결과:

- `POST /api/projects/:projectId/source-repositories/:sourceRepositoryId/analyze`를 추가한다.
- 기존 project 접근권한과 active GitHub Source Repository 검증을 재사용한다.
- 요청할 때마다 최신 default branch를 읽고 분석하며 구조화된 마지막 AI Handoff, revision, 분석 시각만 RDS에 저장한다.
- 원본 Repository 파일과 installation repository 목록은 RDS/S3에 저장하지 않는다.
- 성공과 Template Selection Failure 모두 `200`으로 반환하고, 접근권한·GitHub 읽기 실패만 기존 오류 계약에 맞춰 반환한다.

## M5. 검증

집중 검증:

```bash
pnpm --filter @sketchcatch/types typecheck
pnpm --filter @sketchcatch/api exec tsx --test src/source-repositories/repository-analysis.test.ts src/source-repositories/github-app-client.test.ts src/routes/source-repositories.test.ts
pnpm --filter @sketchcatch/api typecheck
pnpm --filter @sketchcatch/api lint
```

전체 검증:

```bash
pnpm harness:check
pnpm lint
pnpm typecheck
pnpm build
pnpm test
git diff --check
```

런타임 API QA는 실제 Fastify route를 HTTP inject로 호출해 성공, Template Selection Failure, inactive Repository를 확인한다. 최종 QA는 3000이 아닌 포트의 실제 Web/API와 GitHub App 연결을 사용해 Project Settings에서 분석 버튼, 저장 결과 복원, Template AI Handoff를 직접 확인한다. Repository 실행, GitHub write, AWS 실행은 하지 않는다.

## 커밋 규칙

- 각 마일스톤은 집중 테스트와 typecheck가 통과한 뒤 커밋한다.
- 커밋 제목은 `Type: 한국어 제목` 형식을 사용한다.
- 커밋 본문에는 무엇을 왜 바꿨는지 쉬운 한국어로 남긴다.
- 다른 에이전트의 dirty 변경은 staging하지 않는다.
- 결과 새로고침 복원을 위해 `source_repositories` 분석 요약 migration을 별도 milestone commit으로 관리한다.

## 진행 기록

### M0

- 완료: `ac7b7212 Docs: Repository Analysis 결정 문서화`
- glossary와 ADR 0003~0008에 gg 책임, evidence 범위, 실패, 비영속 원칙을 기록했다.

### M1

- `RepositoryAnalysisAiHandoff` 성공/실패 union과 Application Unit/evidence 계약을 shared type으로 추가했다.
- candidate list와 confidence가 공통 계약에 들어가지 않도록 타입을 닫았다.
- `docs/data-models.md`에 Repository Analysis와 API 응답 계약을 반영했다.
- types `tsc --noEmit`과 변경 파일 ESLint를 통과했다.

### M2

- GitHub App installation token으로 지정 branch의 recursive tree와 commit SHA를 읽는다.
- `package.json`, framework config, `Dockerfile`, `README`만 UTF-8 evidence 내용으로 읽고 lockfile과 source file은 경로만 확인한다.
- 일부만 내려온 tree와 지원하지 않는 파일 encoding은 typed error로 중단한다.
- GitHub client 테스트 13개와 API typecheck, 변경 파일 ESLint를 통과했다.

### M3

- root workspace package를 제외하고 monorepo의 frontend, backend, fullstack Application Unit을 구분한다.
- 여섯 `TemplateId`는 명시적인 framework·배포 evidence가 정확히 하나의 규칙과 맞을 때만 선택한다.
- ECS와 EKS처럼 둘 이상의 Template 신호가 함께 있거나 배포 근거가 부족하면 fallback 없이 Template Selection Failure를 반환한다.
- 분석기 테스트 8개, API typecheck, 변경 파일 ESLint, TypeScript no-excuse 검사를 통과했다.

### M4

- `POST /api/projects/:projectId/source-repositories/:sourceRepositoryId/analyze`를 추가했다.
- 기존 project 접근권한과 active GitHub Source Repository를 확인한 뒤 요청마다 최신 default branch를 분석한다.
- 원본 파일은 저장하지 않으며 성공과 Template Selection Failure를 모두 `200`으로 반환한다.
- route 테스트 10개에서 선택 성공, 선택 실패, inactive 차단을 확인했고 API typecheck와 변경 파일 ESLint를 통과했다.

### M5

  - `pnpm --filter @sketchcatch/api lint` 통과
- HTTP inject로 active repository의 선택 성공, Template Selection Failure, inactive repository 차단, rate limit 동작을 확인했다.

- 실제 Project Settings route `/dashboard/projects/:projectId/settings`에서 GitHub App 설치 목록의 `NearthYou/sketchcatch-iac-handoff-test`를 선택해 연결했다.
- 해당 실제 repository revision `8b5d394aa5035c8ebc91e37751d5c8cccc3c6ddd`를 분석해 Template Selection Failure, 불일치 이유, missing evidence를 화면에서 확인했다.

- 성공 결과에서 frontend Application Unit `.`과 Vite, `Static Web Hosting`, `static-web-hosting`, 선택 근거, missing Dockerfile/framework config를 확인했다.
- 분석 버튼의 loading·disabled 상태를 관찰했고 성공·실패 결과 모두 새로고침 후 유지됐다.
- `선택한 Template을 AI 보완으로 넘기기`를 눌러 Workspace AI banner와 `Static Web Hosting` TemplateDefinition의 6개 resource가 보드에 그대로 생성된 것을 확인했다.
- AI에 `EC2와 RDS 기반 3계층으로 바꿔줘`라고 요청했을 때 기존 Template을 교체하지 않고 `POST /api/ai/architecture-patch-preview`가 `needs_clarification`과 교체 대상 질문을 반환하는 것을 확인했다.
- Workspace는 URL의 Template ID를 신뢰하지 않고 저장된 project/source Repository Analysis 결과를 다시 조회한다. `three-tier-web-app`으로 변조한 URL은 화면 진입 전에 한국어 오류로 차단됐다.
- `POST /api/ai/architecture-draft`도 인증된 project/source 분석 결과와 요청 Template을 서버에서 대조하며, 변조 요청은 `409 REPOSITORY_ANALYSIS_TEMPLATE_MISMATCH`로 거부됐다.
- GitHub App 설치 목록·callback·repository 연결은 로그인 사용자의 GitHub OAuth account ID와 installation account ID가 일치할 때만 허용하도록 보강했다.
- 마지막 Chrome 실행은 desktop 1440px과 mobile 390px에서 잘림 없이 통과했고 console error, failed request, HTTP 4xx/5xx가 모두 0건이었다.
- QA 중 발견한 비기본 Web 포트 CORS credential 처리, 인증 복구 전 설정 API 401 경합, 제목·메타데이터 대비와 44px touch target을 보정했다.
- GitHub write는 성공 분석 fixture PR #25 생성까지만 진행했고, 외부 private repository 기본 브랜치 merge는 실행하지 않았다.
