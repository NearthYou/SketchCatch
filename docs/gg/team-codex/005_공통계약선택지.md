# 팀장 Codex용 공통 계약 선택지

> 상태: gg AI 파트와의 호환성 확인을 위한 검토용 선택지다. 추천안은 확정이 아니라 제안이다.

너는 SketchCatch 팀장/공통 계약 담당 Codex다. 구현 전에 gg AI 파트의 DTO, API 응답, 저장 정책이 팀 공통 기준과 맞는지 확인하고 선택지를 골라라.

## 선택할 것

### 1. 공통 API 응답 wrapper

**A. 아직 공통 wrapper가 코드에 없으면 기존 Fastify route 스타일을 유지하고, wrapper 도입 시 일괄 변경한다. (추천)**

- 장점: 현재 코드와 맞고 구현을 막지 않는다.
- gg AI 파트 영향: AI route는 DTO를 명확히 반환하되 wrapper에 강하게 묶이지 않는다.

**B. 지금 AI route부터 `{ data, error }` wrapper를 새로 도입한다.**

- 장점: 공통 형식 선점 가능.
- 위험: 다른 route와 응답 형식이 달라진다.

**C. 모든 route를 먼저 wrapper로 바꾸고 AI를 붙인다.**

- 장점: 가장 일관적이다.
- 위험: gg AI 작업 전에 팀장 범위 리팩터링이 필요하다.

### 2. AI DTO 위치

**A. API/프론트가 공유하는 AI DTO는 `packages/types/src/index.ts`에 둔다. (추천)**

- 장점: 타입 출처가 명확하고 팀원 Codex가 같은 타입을 import한다.
- gg AI 파트 영향: 구현 전에 shared type을 먼저 추가한다.

**B. AI DTO는 `apps/api` 내부에만 둔다.**

- 장점: API 구현은 빠르다.
- 위험: 프론트와 보드가 응답 타입을 복제할 수 있다.

**C. AI DTO는 문서에만 두고 구현 중 각자 만든다.**

- 장점: 초기 자유도는 높다.
- 위험: Codex끼리 필드명 불일치가 생긴다.

### 3. AI 결과 저장 정책

**A. 1차 제공에서는 AI 결과를 stateless response로 두고 저장하지 않는다. (추천)**

- 장점: DB schema 충돌 없이 시작한다.
- gg AI 파트 영향: 분석 결과 저장 table을 만들지 않는다.

**B. Pre-Deployment Analysis 결과만 저장한다.**

- 장점: 배포 전 판단 근거를 재조회할 수 있다.
- 위험: schema, version, stale data 정책이 필요하다.

**C. 모든 AI 요청/응답을 저장한다.**

- 장점: 디버깅과 품질 평가에는 좋다.
- 위험: prompt/output 보안, 비용, 개인정보 관리가 필요하다.

### 4. `ArchitectureSnapshot.source`와 AI source

**A. DB 저장 source와 AI response source를 분리한다. (추천)**

- 예: `ArchitectureSnapshot.source = "imported"` 또는 `"prompt"`, AI response metadata `source = "github" | "template_fallback" | "llm_fallback"`
- 장점: 저장 출처와 AI 생성 근거를 혼동하지 않는다.

**B. `ArchitectureSnapshot.source`에 `github`, `template_fallback`, `llm_fallback`을 그대로 저장한다.**

- 장점: 저장된 스냅샷에서 AI 출처를 바로 알 수 있다.
- 위험: 기존 source 의미가 넓어지고 팀원별 해석이 달라질 수 있다.

**C. source 필드를 제거하고 별도 metadata로 통합한다.**

- 장점: 장기적으로는 유연하다.
- 위험: 현재 구현과 문서를 크게 바꿔야 한다.

### 5. `ResourceType` 확장 승인

**A. `SUBNET`, `SECURITY_GROUP`, `CLOUDFRONT`를 1차 제공 공통 타입으로 승인한다. (추천)**

- 장점: AI, 보드, Terraform, 비용/보안 룰의 최소 시나리오가 맞는다.

**B. 기존 타입만 유지하고 나머지는 `UNKNOWN`으로 둔다.**

- 장점: 변경이 적다.
- 위험: AI 초안과 Terraform 생성이 약해진다.

**C. ResourceType을 문자열 자유 입력으로 바꾼다.**

- 장점: 확장은 쉽다.
- 위험: 공통 계약이 사라진다.
