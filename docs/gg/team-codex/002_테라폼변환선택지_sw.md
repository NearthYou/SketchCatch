# sw Terraform 변환 선택지

이 문서는 sw Terraform 파트에게 묻는 문서다.

쉽게 말하면:

> 보드 설계도 JSON을 Terraform 코드로 바꿀 때, 무엇을 기준으로 할지 정하는 문서다.

## 1. Terraform 코드는 무엇을 보고 만들까?

### A. `ArchitectureJson`을 기준으로 생성. 추천

보드가 저장한 설계도 JSON을 보고 Terraform을 만든다.

```text
ArchitectureJson
→ Terraform 코드
```

가장 안전하다.

### B. Terraform 코드를 기준으로 보드를 맞춤

코드가 원본이 된다.

보드와 AI가 따라가야 해서 어렵다.

### C. AI 응답을 기준으로 생성

AI가 원본이 된다.

검증이 어려워서 위험하다.

## 2. Resource config key는 누가 정할까?

예를 들어 EC2에는 이런 값이 필요할 수 있다.

```text
instanceType
ami
subnetId
securityGroupIds
```

이 이름을 팀원마다 다르게 쓰면 깨진다.

### A. sw가 필요한 config key를 정리. 추천

Terraform 생성기가 필요한 값을 sw가 안다.

gg는 그 key에 맞춰 초안을 만든다.

### B. gg가 먼저 key를 정함

AI 초안 만들기는 빠르지만 Terraform 생성과 안 맞을 수 있다.

### C. 각자 만들고 나중에 맞춤

나중에 충돌 가능성이 높다.

## 3. Terraform 코드와 보드 노드는 어떻게 연결할까?

Terraform 코드가 만든 리소스가 보드의 어떤 노드인지 알아야 한다.

예:

```text
ArchitectureJson node id: ec2-api
Terraform resource: aws_instance.api
```

### A. Terraform 결과에 node id mapping 포함. 추천

나중에 설명과 경고를 연결하기 쉽다.

### B. 이름 규칙으로 추론

예: `ec2-api` → `aws_instance.api`

규칙이 깨지면 연결이 어렵다.

### C. 연결하지 않음

코드 설명과 보드 경고를 연결하기 어렵다.

## 4. 코드 수정과 보드 동기화는 누가 맡을까?

### A. sw가 소유. 추천

코드 ↔ 다이어그램 동기화는 Terraform 파트가 책임진다.

gg는 차이와 위험을 설명한다.

### B. gg AI가 수정까지 담당

AI가 보드와 코드를 동시에 바꾸면 위험하다.

### C. MVP에서는 동기화하지 않음

범위는 줄지만 핵심 가치가 약해질 수 있다.

## sw에게 받고 싶은 답

```text
sw 선택 결과

1. Terraform 생성 기준: A/B/C
2. Resource config key 관리: A/B/C
3. node id mapping: A/B/C
4. 코드 ↔ 보드 동기화 책임: A/B/C

ResourceType별 필수 config 초안:
- VPC:
- SUBNET:
- EC2:
- RDS:
- S3:
- SECURITY_GROUP:
- CLOUDFRONT:
```
