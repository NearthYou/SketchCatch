# gg AI 파트 1차 제공 범위 초안

> 상태: SketchCatch 서비스 상세 기획서의 1차 제공 범위에 맞춘 gg AI 파트 PR 검토용 초안이다. 실제 구현 전 shared type, API schema, 팀원 파트 계약에 한 번 더 맞춘다.

## 결론

gg 파트의 중심은 "GitHub 코드를 보고 인프라를 맞히는 기능"이 아니라 **자연어 요구사항을 인프라 그래프로 구조화하고, 비용·성능·보안 관점의 설계 판단 근거를 제공하는 기능**이다.

기존의 학습형 메시지는 줄이고, 아래 흐름을 AI 파트 핵심 가치로 둔다.

```text
Requirement Prompt
→ backend AI service
→ intent/constraint extraction
→ Architecture Draft as ArchitectureJson
→ Pre-Deployment Check / Design Simulation / IaC Preview explanation
→ Architecture Board / Terraform Preview / Validation / Simulation 화면에 표시
```

AI는 설계 의도 해석과 설명을 돕지만, Terraform 최종 생성, Apply 판단, AWS 권한 변경, 실제 배포 실행의 최종 결정자가 아니다. 실제 AWS 계정 연결과 Terraform Plan/Apply/Destroy는 상세 기획서의 2차 제공 범위이므로, 1차 gg AI 범위에서는 export 전 설계 검증과 설명을 우선한다.

## LLM 적용 범위

1차 제공에서 LLM은 자연어 요구사항 해석과 설명 생성을 맡는다. LLM 응답은 항상 검증 대상이며, 검증을 통과한 구조화 결과만 제품 흐름에 반영한다.

우선순위:

1. LLM 설명 생성
   - Resource 선택 이유 설명
   - Check Finding 설명
   - 비용·성능·보안 판단 근거 설명
   - Terraform validate/export 전 확인 항목 설명
   - 배포 전 체크리스트 문장화
2. LLM 자연어 의도 분류
   - 예산, 예상 트래픽, 런타임, DB, 가용성, 보안 우선순위를 구조화
   - 대표 템플릿 또는 지원 가능한 Resource 조합으로 매핑
   - 불확실한 조건은 `assumptions`와 `confidence`로 표시
3. LLM Architecture Draft 보조 생성
   - 결과는 jh 파트의 `ArchitectureJson` 구조와 호환되어야 한다.
   - JSON schema 검증, 지원 Resource 검증, 위험 기본값 보정을 통과해야 한다.
   - 검증 실패 시 LLM 결과를 버리고 Template 기반 Architecture Draft로 fallback한다.

LLM이 직접 하지 않는 것:

- IaC Preview 최종본 생성
- Terraform Apply 여부 판단
- AWS 권한 생성 또는 확장
- Pre-Deployment Check의 deploy-blocking 판정
- GitHub 코드만 보고 배포 인프라의 정답을 단정

LLM provider와 fallback:

- 1차 제공의 기본 provider는 OpenAI API로 한다.
- 프론트엔드는 LLM provider를 직접 호출하지 않는다. 모든 AI 요청은 backend API를 경유한다.
- OpenAI API key는 서버 환경변수로만 관리하고, 프론트 번들에 포함하지 않는다.
- provider timeout, API key 누락, rate limit, 비용 제한 초과, JSON 검증 실패가 발생하면 deterministic fallback 또는 Template 기반 결과로 돌아간다.
- 발표나 QA에서 외부 LLM provider가 실패해도 자연어 초안 생성, 비용/위험 분석, 오류 설명이 deterministic fallback으로 최소 동작해야 한다.

호출 흐름:

```text
frontend
→ backend AI service
→ LLM provider
→ schema validation
→ deterministic fallback if needed
→ frontend
```

## 1. Requirement Prompt → Architecture Draft

사용자가 자연어로 원하는 조건을 입력하면 Architecture Board가 열 수 있는 인프라 설계안 초안을 만든다.

예시 입력:

- "월 비용은 100달러 이하였으면 좋겠어."
- "초기 스타트업 서비스고 하루 사용자 1만 명 정도를 예상해."
- "백엔드는 Node.js고 데이터베이스는 PostgreSQL을 쓰고 싶어."
- "장애 대응보다는 비용 절감이 더 중요해."
- "보안을 강화하되 너무 비싼 구조는 피하고 싶어."

1차 제공 깊이:

- 입력에서 예산, 트래픽, 런타임, 데이터베이스, 가용성 우선순위, 보안 우선순위를 추출한다.
- 대표 유형은 정적 웹사이트, 단일 EC2 웹 서버, API 서버 + DB, 저비용 스타트업 API 정도로 제한한다.
- 생성 결과는 자유 텍스트가 아니라 jh 파트의 `ArchitectureJson`과 호환되는 Architecture Draft JSON이다.
- VPC, Subnet, EC2, RDS, S3, Security Group, CloudFront 정도의 제한된 Resource만 생성한다.
- 알 수 없는 요구사항은 무리해서 생성하지 않고 Template 선택 또는 수동 편집으로 유도한다.

완료 기준:

- 같은 입력에 대해 재현 가능한 초안을 만든다.
- Architecture Board에서 바로 열 수 있다.
- 생성된 Resource마다 선택 이유와 assumptions가 붙는다.
- 예산이나 트래픽 조건이 있으면 비용/성능 분석으로 이어질 수 있는 metadata를 남긴다.

AI Architecture Draft 응답 DTO:

```ts
type AiArchitectureDraftResult = {
  architectureJson: ArchitectureJson;
  title: string;
  metadata: {
    source: "prompt" | "template_fallback" | "llm_fallback" | "github";
    confidence: "low" | "medium" | "high";
    assumptions: string[];
    explanations: string[];
  };
};
```

- `architectureJson`은 기존 공통 타입을 그대로 사용한다.
- jh의 Architecture Board는 `architectureJson`만 받아도 열릴 수 있어야 한다.
- AI 전용 근거와 설명은 `metadata.source`, `metadata.assumptions`, `metadata.explanations`, `metadata.confidence`에만 담는다.
- AI 전용 `resources`, `relationships` 같은 별도 그래프 구조를 만들지 않는다.

지원할 대표 유형:

| 대표 유형 | 입력 단서 예시 | 주요 Resource | 주요 Check Finding |
| --- | --- | --- | --- |
| 정적 웹사이트 | 정적 사이트, 랜딩, frontend-only | S3, CloudFront | S3 public access, CloudFront 비용 |
| 단일 EC2 웹 서버 | 작은 서버, Docker, 단일 runtime port | VPC, Subnet, EC2, Security Group | SSH `0.0.0.0/0`, instance type 비용 |
| API 서버 + DB | Node.js API, PostgreSQL, 백엔드 서비스 | VPC, Subnet, EC2, RDS, Security Group | RDS 비용, DB public access, 삭제 계획 누락 |
| 저비용 스타트업 API | 예산 제한, 트래픽 초기 단계 | EC2 또는 Lambda 후보, RDS 또는 대체 저장소 | 비용 상한 초과, 성능 한계 |

## 2. GitHub 링크 기반 초안 생성은 후순위

GitHub 링크 기반 초안 생성은 1차 제공의 메인 챌린지에서 내린다.

이유:

- 같은 코드라도 EC2, ECS, Fargate, Kubernetes, Lambda 등 여러 배포 방식이 가능하다.
- 코드는 인프라와 독립적으로 작성되는 것이 좋기 때문에, 코드만 보고 정답 인프라를 판단하는 주장은 약하다.
- 발표에서는 "코드 분석"보다 "자연어 요구사항 해석 → 인프라 그래프 생성 → Terraform/시뮬레이션 연결"이 더 설득력 있다.

1차 제공에서 남기는 범위:

- public repository의 README, package metadata, Dockerfile, compose file을 보조 evidence로 볼 수 있다.
- 단, Source Repository는 자연어 요구사항을 보강하는 참고 자료일 뿐 원천 입력이 아니다.
- private repository OAuth, 전체 코드 분석, monorepo service 분석은 고도화로 둔다.

## 3. IaC Preview / Terraform 코드 생성 보조

Terraform 생성 자체는 `ArchitectureJson`에서 결정론적으로 만든다. AI는 그 결과를 설명하고, 누락된 설정이나 위험한 설정을 지적하는 보조 역할을 한다.

1차 제공 깊이:

- IaC Preview는 sw 파트의 생성기 또는 템플릿이 만든다.
- AI는 "이 코드가 어떤 Resource를 만들며 왜 필요한지"를 설명한다.
- AI는 위험한 변경 제안, 누락된 변수, 사용자가 이해하기 어려운 부분을 설명한다.
- 다이어그램과 Terraform 코드의 동기화 원천은 AI 응답이 아니라 `ArchitectureJson`이다.

완료 기준:

- IaC Preview와 Architecture Board의 Resource가 서로 대응된다.
- AI 설명은 Resource 단위로 볼 수 있다.
- AI가 만든 설명이 틀려도 실제 생성 코드는 흔들리지 않는다.

1차 제공에서 하지 않는 것:

- AI가 자유롭게 Terraform 최종본을 작성하지 않는다.
- AI가 생성한 코드를 바로 Apply하지 않는다.
- 코드 ↔ 다이어그램 동기화의 원천 진실을 AI 응답으로 두지 않는다.

## 4. 비용·성능·보안 분석과 설명

Pre-Deployment Check에서 Check Finding을 만들고, AI가 설계 판단 근거를 설명한다.

비용/성능 책임 경계:

- gg 파트는 비용 추정, 리소스별 비용 분석, 간단한 성능/트래픽 시뮬레이션 설명을 책임진다.
- 1차 제공의 비용 추정은 실제 청구액 보장이 아니라 Resource type, instance class, 사용 시간, traffic assumption에 기반한 추정값이다.
- 가격표에 없는 Resource나 설정은 금액을 억지로 만들지 않고 `low`, `medium`, `high` Cost Risk 등급으로 fallback한다.
- 성능 시뮬레이션은 실제 부하 테스트가 아니라 Resource별 capacity assumption과 요청 흐름 그래프를 기반으로 한 설계 검토다.

1차 제공 깊이:

- Cost Risk, Security Risk, Network Risk, missing configuration, permission concern, performance bottleneck, availability concern을 구분한다.
- 위험도는 `low`, `medium`, `high`로 제한한다.
- 룰 엔진이 먼저 비용/위험 finding을 만들고, AI는 finding과 분석 근거를 설명한다.
- 비용은 total estimate와 resource-level estimate를 모두 제공한다.
- 성능은 "현재 설계가 특정 요청량을 감당할 가능성"과 "병목 후보 Resource" 정도로 제한한다.

예시 finding:

- Security Group이 `0.0.0.0/0` SSH를 허용하면 high Security Risk
- RDS, NAT Gateway, ALB는 Cost Risk
- 배포 후 Destroy 또는 정리 계획이 없으면 Cost Risk
- 필수 region, instance type, subnet 연결이 없으면 missing configuration
- 예상 트래픽이 t3.micro 단일 EC2 capacity assumption을 넘으면 performance bottleneck

완료 기준:

- 각 finding에 이유, 영향, 수정 가이드가 있다.
- export 또는 Plan 전 화면에서 사용자가 위험을 보고 멈출 수 있다.
- 비용 금액이 표시되는 경우에는 어떤 Resource와 가정 때문에 그 금액이 나왔는지 설명한다.
- 성능 분석은 실제 운영 보장이 아니라 설계 시뮬레이션임을 표시한다.

AI Pre-Deployment Analysis 응답 DTO:

```ts
type MoneyEstimate = {
  amount: number;
  currency: "USD" | "KRW";
};

type ResourceCostEstimate = {
  resourceId: string;
  resourceType: ResourceType;
  name: string;
  monthlyEstimate: MoneyEstimate;
  costDrivers: string[];
  explanation: string;
};

type CheckFinding = {
  id: string;
  category:
    | "cost"
    | "security"
    | "configuration"
    | "permission"
    | "network"
    | "performance"
    | "availability";
  severity: "low" | "medium" | "high";
  resourceId?: string;
  title: string;
  description: string;
  recommendation: string;
};

type ChecklistItem = {
  id: string;
  label: string;
  status: "pass" | "warning" | "fail";
  relatedFindingIds: string[];
};

type AiPreDeploymentAnalysisResult = {
  summary: string;
  totalMonthlyEstimate: MoneyEstimate & {
    pricingAssumption: string;
  };
  resourceCostEstimates: ResourceCostEstimate[];
  findings: CheckFinding[];
  checklist: ChecklistItem[];
};
```

- 응답은 하나로 합치되, 내부 필드는 비용, 위험, 체크리스트 배열로 분리한다.
- `resourceId`는 `ArchitectureJson.nodes[].id`와 연결되어야 한다.
- 배포 전 검증 화면은 이 DTO 하나로 비용/위험/체크리스트를 구성한다. 2차 제공의 Plan 전 화면에서도 같은 DTO를 재사용할 수 있다.
- jh의 Architecture Board는 `findings[].resourceId`로 노드별 경고 표시를 할 수 있다.
- 이 DTO는 Apply 가능 여부의 최종 판정자가 아니라 배포 전 판단 근거를 제공하는 결과다.

성능 시뮬레이션은 1차 코드 타입이 확정되기 전까지 별도 DTO로 바로 고정하지 않고, 후속 이슈에서 `DesignSimulationResult` 또는 `AiDesignSimulationResult` 같은 이름으로 분리한다.

## 5. AI 수정 제안

검증 결과나 사용자의 자연어 수정 요청을 바탕으로 수정안을 제안한다. 다만 AI 수정 제안은 자동 적용이 아니라 사용자가 diff를 보고 승인하는 proposal이다.

1차 제공 깊이:

- Check Finding 기반 수정 제안을 만든다.
- "비용 줄여줘", "보안 강화해줘", "트래픽 대응해줘" 같은 자연어 수정 요청은 대표 유형만 처리한다.
- 수정안은 문제 요약, 영향 Resource, 수정 방향, 비용/보안/성능 영향, 적용 전 변경 요약을 포함한다.
- 수정안 적용 전에는 Architecture Board 변경 요약과 Terraform Preview 변경 요약을 보여준다.
- 수정안 적용 후에는 Pre-Deployment Check를 다시 실행해야 한다.

1차 대표 수정안:

| 문제 | 수정안 |
| --- | --- |
| SSH `0.0.0.0/0` 전체 오픈 | 접근 IP 제한 또는 Bastion 후보 안내 |
| RDS public 접근 | RDS를 private subnet으로 이동하도록 제안 |
| NAT Gateway 비용 위험 | S3 접근은 VPC Endpoint 또는 더 단순한 구조로 대체 후보 안내 |
| 단일 EC2 병목 | ALB + Auto Scaling 또는 더 큰 instance 후보 안내 |
| RDS 백업 미설정 | backup retention 활성화 제안 |

완료 기준:

- AI 수정 제안은 즉시 적용되지 않는다.
- 사용자가 변경 diff를 확인하고 적용해야 한다.
- 적용 전후 비용, 성능, 보안 영향이 설명된다.
- proposal은 `ArchitectureJson` schema와 Resource support matrix를 통과해야 한다.

## 6. 오류 설명 / 체크리스트 생성

1차 제공에서는 배포 전 확인 항목과 Terraform validate/export 전 오류를 사용자 언어로 바꾼다. Terraform Plan/Apply에서 나온 오류 설명은 2차 제공의 AWS 계정 연결과 배포 관리가 붙을 때 재사용할 수 있는 후속 연동 범위로 둔다.

1차 제공 결정:

- 오류 판정은 룰 기반 분류기가 먼저 수행한다.
- LLM은 분류된 오류와 원문, 실행 단계, 관련 Resource 맥락을 받아 설명과 다음 행동을 만든다.
- LLM 응답은 원인 확정이 아니라 "가능성이 높은 원인"과 "확인할 것"으로 표현한다.
- 분류되지 않은 오류는 `unknown`으로 처리하고, 원문을 유지한 채 팀원 확인이나 로그 공유를 안내한다.

1차 제공 깊이:

- Plan 전 단계까지 재사용 가능한 체크리스트를 생성한다.
- Terraform validate/export 전 오류 메시지를 룰 기반으로 카테고리화한다.
- Plan/Apply 오류 설명은 fixture와 DTO 호환성만 준비하고, 실제 배포 로그 연동은 2차 제공으로 둔다.
- 사용자가 다음에 해야 할 행동을 1-3개로 줄여 보여준다.
- 오류 설명은 자동 재시도, 자동 Apply, AWS 권한 변경, Terraform 코드 자동 수정까지 수행하지 않는다.

1차 오류 카테고리:

| 카테고리 | 대표 단서 | 사용자에게 설명할 핵심 |
| --- | --- | --- |
| `permission` | `AccessDenied`, `UnauthorizedOperation`, `not authorized` | 현재 AWS 권한으로 해당 작업을 할 수 없음 |
| `credential` | `NoCredentialProviders`, `InvalidClientTokenId`, `ExpiredToken` | AWS 인증 정보가 없거나 만료됨 |
| `region_or_resource` | `InvalidAMIID.NotFound`, `InvalidSubnetID.NotFound`, `not found` | region 또는 참조 Resource가 맞지 않음 |
| `quota` | `VcpuLimitExceeded`, `LimitExceeded`, `quota` | 계정 한도 때문에 Resource를 만들 수 없음 |
| `syntax` | `terraform validate` 실패, `Unsupported argument`, `Missing required argument` | IaC Preview에 문법 또는 필수 값 문제가 있음 |
| `dependency` | dependency cycle, subnet/VPC/security group 연결 오류 | Resource 간 연결 관계가 잘못되었을 가능성이 있음 |
| `unknown` | 위 규칙에 매칭되지 않음 | 자동 판단하지 않고 원문과 확인 지점을 보여줌 |

AI Terraform 오류 설명 응답 DTO:

```ts
type AiTerraformErrorExplanationResult = {
  stage: "validate" | "export" | "plan" | "apply";
  category:
    | "permission"
    | "credential"
    | "region_or_resource"
    | "quota"
    | "syntax"
    | "dependency"
    | "unknown";
  severity: "low" | "medium" | "high";
  rawMessage: string;
  summary: string;
  likelyCause: string;
  nextActions: string[];
  relatedResourceId?: string;
};
```

- `rawMessage`는 숨기지 않는다.
- `nextActions`는 1-3개로 제한한다.
- `relatedResourceId`는 오류가 특정 Resource와 연결될 때만 사용한다.
- 1차 제공에서는 `validate`와 `export` 오류를 우선 처리한다.
- 2차 제공에서 ck의 Deployment History에는 원문, 실행 단계, 성공/실패 상태가 남고, gg 파트는 그 내용을 사용자 설명으로 바꾼다.

## 구현 전 팀 호환성 체크

이 섹션은 팀원별 Codex가 서로 다른 타입, 필드명, 책임 범위를 만들어 충돌하지 않도록 구현 전에 맞출 계약이다. 공통 타입의 최종 기준은 [데이터 모델](../data-models.md)이다.

팀원별 Codex에게는 [팀원 Codex 호환성 선택 문서](../team-codex/000_팀원Codex호환성선택문서.md)를 읽히고, 담당 문서의 선택 결과를 받아온다.

현재 제안 기준:

- 보드, AI, Terraform 생성기는 모두 `ArchitectureJson.nodes`와 `ArchitectureJson.edges`를 공유한다.
- AI Architecture Draft는 별도 `resources`, `relationships` 그래프를 만들지 않고 `architectureJson`만 보드 입력으로 제공한다.
- 비용/위험 분석은 `AiPreDeploymentAnalysisResult` 하나로 반환하되 내부 배열은 `resourceCostEstimates`, `findings`, `checklist`로 나눈다.
- 노드별 경고는 `CheckFinding.resourceId`와 `ArchitectureJson.nodes[].id`를 연결해서 표시한다.
- Terraform validate/export 오류 설명은 raw output을 숨기지 않고 `AiTerraformErrorExplanationResult.rawMessage`에 보존한다. Plan/Apply raw output 연동은 2차 제공에서 같은 DTO로 확장한다.
- 실제 Apply, AWS 권한 변경, Terraform 최종본 생성은 AI 파트 책임이 아니다.

팀원 선택 결과 반영:

- jh는 모든 항목 A를 선택했다. gg는 `architectureJson` 단독 입력, 공통 `ResourceType`, `CheckFinding.resourceId` 노드 연결을 기준으로 Architecture Draft와 Finding을 만든다.
- ck는 모든 항목 A를 선택했다. gg 오류 설명 입력은 `{ stage, rawMessage, relatedResourceId? }`로 제한하고, 1차 제공에서는 `validate`, `export`를 우선 지원하며 `plan`, `apply`는 2차 배포 연동 시 같은 계약으로 확장한다.
- ys는 A / B / A / C를 선택했다. gg는 프로젝트 목록에 AI 요약을 필수 요구하지 않고, 중요한 AI 이벤트만 Activity 후보로 제공하며, 익명 workspace와 로그인 user 흐름을 모두 고려한다.
- sw는 모든 항목 A를 선택했다. gg는 `ArchitectureJson`을 Terraform 생성 원천 입력으로 유지하고, sw가 정의할 required config matrix를 따른다. IaC Preview 설명은 `resourceId` 또는 node id mapping 기준으로 연결하며, 코드 ↔ 다이어그램 동기화는 sw 소유로 둔다.
- 팀장은 C / A / B / C / A를 선택했다. gg는 공통 wrapper가 전체 route에 먼저 정리되는 흐름을 따르고, AI DTO는 `packages/types`에 둔다. Pre-Deployment Analysis는 저장 가능 대상으로 보고, AI source는 metadata 안에서 다룬다. `SUBNET`, `SECURITY_GROUP`, `CLOUDFRONT` 확장은 승인된 기준으로 본다.

구현 전에 반드시 확인해야 하는 호환 지점:

| 지점 | 왜 위험한가 | gg Codex가 할 일 | 팀원에게 확인할 것 |
| --- | --- | --- | --- |
| `ResourceType` 값 | AI가 만든 node를 보드/API/Terraform이 거부할 수 있음 | shared type과 API Zod schema는 `SUBNET`, `SECURITY_GROUP`, `CLOUDFRONT`까지 맞춘다 | jh/sw가 같은 문자열을 사용할 수 있는지 확인 |
| `ResourceNode.config` key | 같은 리소스를 팀원별로 다른 설정 이름으로 읽을 수 있음 | AI template에서 쓰는 key를 문서와 테스트 fixture에 고정한다 | sw가 Terraform 생성기에 필요한 필수 key 목록을 알려달라고 요청 |
| 자연어 제약 조건 | 예산/트래픽/가용성 조건을 AI만 이해하고 다른 파트가 못 쓰면 분석이 끊김 | requirement extraction 결과를 metadata나 후속 DTO로 구조화한다 | jh/ck/팀장에게 조건 저장 위치와 화면 표시 범위를 확인 |
| 공통 API 응답 wrapper | AI route만 다른 응답 모양이면 프론트 연결이 깨짐 | AI route만 별도 wrapper를 만들지 않고, 전체 route wrapper 정리 이후 같은 형식을 따른다 | 팀장에게 wrapper 적용 시점과 공통 helper 위치를 확인 |
| Terraform/Plan/Apply raw output | 오류 설명 입력 모양이 sw/ck 파트와 다르면 연동이 깨짐 | 1차는 `validate`/`export`, 2차는 `plan`/`apply`까지 `stage`, `rawMessage`, 선택 `relatedResourceId` 최소 입력으로 받게 한다 | sw에게 validate/export 오류 shape를, ck에게 Plan/Apply 결과와 로그 line shape를 확인 |
| 분석 결과 저장 여부 | DB schema 없이 AI 결과를 저장하려 하면 충돌함 | Pre-Deployment Analysis 저장은 공통 DB 기준을 따르고, Architecture Draft/Error Explanation은 별도 합의 전까지 응답 중심으로 둔다 | 팀장에게 Pre-Deployment Analysis 저장 테이블/컬럼 확인 |

## 5주 구현 순서

1. Week 1: `ArchitectureJson`, AI 분석 DTO, Requirement Prompt 입력 계약 확정
2. Week 2: 자연어 요구사항 기반 Architecture Draft와 Template fallback 구현
3. Week 3: 비용 추정, 리소스별 비용 분석, 위험도/보안 검증 rule engine 구현
4. Week 4: 비용·트래픽 시뮬레이션 최소 모델, Terraform Preview 설명, validate/export 오류 설명 연동
5. Week 5: 버전 비교/변경점 요약 연동, 발표/QA용 fallback fixture, 시뮬레이션 샘플 고정

## 발표/QA 지원 범위

팀 전체 발표 시나리오는 팀 공통 결정이다. gg 파트 문서는 전체 발표 순서나 실제 Plan/Apply 노출 여부를 확정하지 않고, AI 파트가 제공할 수 있는 관찰 가능한 데모 단위와 fallback fixture만 정의한다.

AI 파트가 제공할 수 있는 데모 단위:

1. 자연어 요구사항을 분석해 Architecture Draft를 제안한다.
2. 생성된 Resource와 선택 근거, assumptions, confidence를 설명한다.
3. Pre-Deployment Check 결과로 비용 추정, 리소스별 비용 분석, 보안 위험, 체크리스트를 보여준다.
4. 예상 트래픽이나 비용 조건을 기준으로 병목 후보와 비용 초과 가능성을 설명한다.
5. IaC Preview나 Terraform 코드에서 위험 지점과 누락된 설정을 설명한다.
6. Terraform validate/export 오류가 발생했을 때 원인 후보와 다음 행동을 설명한다.
7. 2차 제공에서는 Plan/Apply 오류 설명을 같은 DTO로 확장한다.

팀 협의가 필요한 것:

- 위 데모 단위 중 무엇을 메인 발표 흐름에 넣을지
- 2차 제공 전까지 실제 Plan/Apply를 보여줄지, mock 실행이나 기록 화면으로 보여줄지
- 시뮬레이션을 어느 정도 수치까지 보여줄지
