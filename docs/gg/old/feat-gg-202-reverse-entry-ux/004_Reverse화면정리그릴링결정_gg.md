# Reverse 화면 정리 그릴링 결정

이 문서는 Reverse Engineering 화면을 더 깔끔하게 고치기 전에 정한 내용을 적는 곳이다.

쉽게 말하면, 지금 화면은 너무 많은 정보를 한 번에 펼쳐서 보여준다.

그래서 사용자가 가장 먼저 해야 하는 일인 **보드 후보 선택**이 잘 안 보인다.

## 1. 왼쪽 패널 역할

왼쪽 패널은 안내문을 보여주는 자리가 아니다.

왼쪽 패널은 **보드 후보 선택 전용 영역**으로 쓴다.

선택: A

즉, 왼쪽 패널에는 아래 내용만 크게 보여준다.

- 배포 관계 기준 구조
- 연결 관계 기준 구조
- Resource 종류 기준 구조

여기서 중요한 점은 이것이다.

보드 후보는 리소스 일부를 고르는 기능이 아니다.

모든 리소스는 그대로 유지하고, 그 리소스를 어떤 기준으로 묶어서 볼지만 바꾸는 기능이다.

## 2. 왜 기존 왼쪽 안내 패널이 별로였나

사용자는 이미 `기존 AWS 가져오기`를 선택해서 Reverse 화면으로 들어왔다.

그런데 왼쪽에서 다시 긴 설명을 보여주면 화면이 답답해진다.

그리고 정작 중요한 후보 선택은 오른쪽 패널 중간에 묻힌다.

그래서 왼쪽 안내문은 빼고, 후보 선택을 왼쪽에 크게 둔다.

## 3. 오른쪽 패널에 남길 역할

오른쪽 패널은 결과를 자세히 보는 영역으로 둔다.

기본으로 크게 펼쳐둘 정보는 줄인다.

후보 선택은 왼쪽으로 옮긴다.

오른쪽에는 아래 정보들을 접어서 보여주는 방향으로 간다.

- 부분 실패

Terraform import 제안은 Reverse 화면에서 기본 정보로 보여주지 않는다.

이 정보는 Terraform 쪽에서 다루는 것이 더 자연스럽다.

## 4. 아직 정해야 할 것

지금 기준으로 큰 화면 구조는 정해졌다.

남은 것은 실제 구현하면서 문구와 배치 간격을 다듬는 일이다.

## 5. Draft 수정과 리소스 파라미터

Draft 수정은 결과 패널에서 길게 펼치지 않는다.

선택: C

즉, `Draft 수정` 영역은 없앤다.

리소스를 수정하거나 자세히 보고 싶으면 Architecture Board에서 리소스를 직접 클릭한다.

그때 오른쪽 상세 패널에서 아래 내용을 본다.

- 표시 이름
- 설명
- 위치
- AWS에서 읽어온 원본 값

여기서 중요한 기준은 이것이다.

사용자가 바꿔도 되는 값과 AWS에서 읽어온 원본 값은 분리해야 한다.

사용자가 바꿔도 되는 값은 보드 표시용 값이다.

예를 들면 아래 값이다.

- 표시 이름
- 설명
- 위치

AWS에서 읽어온 원본 값은 기본적으로 읽기 전용이다.

예를 들면 아래 값이다.

- VPC ID
- Subnet ID
- CIDR
- Region
- Security Group rule
- Provider Resource ID

이 값들을 수정 가능하게 보이면 안 된다.

사용자가 이 값을 바꾸면 실제 AWS가 바뀌는 것처럼 오해할 수 있기 때문이다.

그래서 Reverse 결과 패널은 후보 선택과 요약에 집중한다.

리소스 단위의 자세한 확인과 수정은 보드의 리소스 상세 패널로 보낸다.

## 6. 부분 실패 표시 방식

부분 실패는 기본으로 길게 펼치지 않는다.

선택: A

즉, 기본 화면에서는 한 줄 요약만 보여준다.

예를 들면 아래처럼 보여준다.

```text
못 읽은 서비스 14개
```

자세한 실패 목록은 토글을 열었을 때만 보여준다.

토글을 열면 아래처럼 실제 실패 내용을 확인할 수 있다.

- RDS 권한 없음
- Lambda 권한 없음
- Resource Explorer 권한 없음
- CloudWatch 권한 없음

여기서 중요한 점은 이것이다.

부분 실패를 숨기면 안 된다.

사용자는 이 결과가 AWS 전체 상태가 아닐 수도 있다는 사실을 알아야 한다.

하지만 처음부터 실패 목록을 길게 펼치면 화면이 너무 무겁고 무섭게 보인다.

그래서 요약은 항상 보이게 하고, 긴 실패 목록은 사용자가 필요할 때만 열게 한다.

## 7. Terraform import 제안 표시 방식

Terraform import 제안은 Reverse 화면에서 뺀다.

선택: C

즉, Reverse 화면은 기존 AWS를 읽고 보드 후보를 고르는 일에 집중한다.

Terraform import 명령어는 Terraform 쪽에서 보여준다.

이유는 간단하다.

Reverse 화면에서 import 명령어까지 길게 보여주면 화면이 너무 복잡해진다.

그리고 import 명령어는 사용자가 보드 후보를 고른 다음, Terraform으로 넘길 때 필요한 정보다.

그래서 Reverse 화면에서는 아래 정도만 알 수 있으면 된다.

- AWS에서 어떤 Resource를 읽었는지
- 어떤 기준의 보드 후보를 고를 수 있는지
- 못 읽은 서비스가 있는지
- 선택한 후보를 프로젝트로 만들 수 있는지

Terraform import 제안은 나중에 Terraform 화면에서 아래처럼 다룬다.

```text
이 Practice Architecture를 Terraform으로 관리하려면 import가 필요합니다.
```

그때 필요한 명령어 목록을 보여준다.

정리하면, Reverse 화면은 `가져오기와 후보 선택` 화면이다.

Terraform import 제안은 `Terraform 관리로 넘기는 단계`에서 보여준다.

## 8. 오른쪽 패널 첫 화면

오른쪽 패널 첫 화면에는 스캔 요약과 적용 버튼만 남긴다.

선택: B

즉, 오른쪽 패널은 사용자가 선택한 후보를 프로젝트로 만들 수 있게 도와주는 자리다.

처음부터 긴 설명이나 상세 목록을 많이 보여주지 않는다.

기본 화면에 남길 정보는 아래 정도다.

- 읽은 Resource 개수
- 못 읽은 서비스 개수
- 현재 선택한 보드 후보 이름
- 프로젝트로 만들기 버튼

여기서 중요한 점은 이것이다.

왼쪽 패널은 후보를 고르는 곳이다.

오른쪽 패널은 선택한 후보를 확인하고 적용하는 곳이다.

그래서 오른쪽 패널에는 사용자가 바로 다음 행동을 할 수 있는 버튼이 잘 보여야 한다.

예를 들면 아래 흐름이다.

```text
왼쪽에서 후보 선택
→ 오른쪽에서 요약 확인
→ 프로젝트로 만들기
```

부분 실패 같은 상세 정보는 필요하면 접힌 상태로 둘 수 있다.

하지만 오른쪽 패널 첫 화면의 주인공은 상세 정보가 아니다.

주인공은 `프로젝트로 만들기` 버튼이다.

## 9. 프로젝트 생성 확인 팝업

`프로젝트로 만들기` 버튼을 누르면 확인 팝업 없이 바로 만든다.

선택: 확인 없이

이유는 간단하다.

이 화면은 이미 사용자가 보드 후보를 보고 고르는 화면이다.

즉, 사용자는 아래 과정을 이미 거쳤다.

```text
기존 AWS 가져오기 선택
→ AWS 스캔 실행
→ 보드 후보 확인
→ 원하는 후보 선택
```

여기서 다시 확인 팝업을 띄우면 흐름이 끊긴다.

그래서 `프로젝트로 만들기` 버튼은 바로 실행한다.

대신 버튼을 누르기 전 화면에서 사용자가 아래 정보를 확실히 볼 수 있어야 한다.

- 선택한 보드 후보 이름
- 읽어온 Resource 개수
- 못 읽은 서비스 개수

정리하면, 확인 팝업으로 막는 게 아니라 버튼을 누르기 전 요약을 잘 보여주는 방식으로 간다.
