# 001. 작업 계획

## 목표

0706 피드백 이후 Reverse Engineering 후보 흐름을 정리한다.

이번 브랜치에서는 아래 두 가지를 코드 기준으로 반영한다.

```text
후보는 전체 Resource를 유지하고 묶음 방식만 다르게 만든다.
Reverse 전용 화면에서 시작 방식을 다시 고르면 /workspace/new로 돌아간다.
```

Template 모달/적용 흐름은 여기서 새로 건드리지 않는다.

이 문서의 중심은 `후보`를 정확히 정의하고 구현 기준으로 바꾸는 것이다.

## 마일스톤 2. Reverse 후보 만들기

### 목표

AWS에 여러 인프라가 섞여 있어도 사용자가 맞는 구조 해석을 고를 수 있는 후보를 만든다.

### 사용자가 보는 흐름

```text
기존 AWS 가져오기
→ AWS 리소스 읽기
→ 최대 3개의 보드 후보 표시
→ 사용자가 하나 선택
→ 적용하면 프로젝트 생성
→ Architecture Board로 이동
```

사용자가 Reverse Engineering을 그만두고 시작 방식을 다시 고르고 싶으면 `/workspace/new`로 돌아간다.

이때 버튼 문구는 아래처럼 둔다.

```text
시작 방식 다시 선택
```

`작업 화면으로 돌아가기`라고 쓰지 않는다.

Reverse Engineering 전용 화면은 기존 작업 화면에서 잠깐 들어온 화면이 아니라, 새 프로젝트 시작 흐름에서 들어온 화면이기 때문이다.

### 후보를 만드는 기준

후보는 하나만 만들지 않는다.

하지만 너무 많이 만들지도 않는다.

구조 후보는 최대 3개까지만 보여준다.

3개보다 많으면 사용자가 고르기 어렵고, 1개만 만들면 정답을 하나로 단정하는 문제가 생긴다.

후보는 Resource 하나가 아니다.

후보는 VPC 하나도 아니다.

후보는 AWS에서 읽어온 전체 Resource를 어떻게 묶어볼지에 대한 `구조 해석안`이다.

중요한 점은 각 후보에 전체 Resource가 들어간다는 것이다.

후보마다 빠지는 Resource가 달라지는 것이 아니다.

후보마다 달라지는 것은 Resource를 묶는 방식이다.

따라서 기본 후보를 아래처럼 만들면 안 된다.

```text
VPC 하나 후보
S3 하나 후보
EC2 하나 후보
```

이렇게 Resource 일부만 후보로 만들면 나중에 배포나 Terraform import로 이어질 때 필요한 Resource가 빠질 수 있다.

후보를 만들 때는 아래 기준을 같이 본다.

```text
배포에 필요한 의존 관계 기준
VPC/Subnet 포함 관계 기준
ALB, EC2, RDS 같은 연결 관계 기준
tag 기준
이름 패턴 기준
region 기준
```

우선순위는 `배포에 필요한 관계 먼저, tag는 힌트`로 둔다.

쉽게 말하면 아래와 같다.

```text
전체 AWS Resource를 먼저 읽는다.
각 후보에는 전체 Resource를 모두 넣는다.
후보마다 Resource를 묶는 방식만 다르게 만든다.
사용자는 어떤 묶음 방식이 맞는지 고른다.
```

이유는 간단하다.

실제 AWS 계정에는 tag가 잘 붙은 Resource도 있고, tag가 없는 Resource도 있다.

그래서 tag만 믿으면 VPC, Subnet, Security Group 같은 필요한 Resource를 놓칠 수 있다.

반대로 연결 관계만 믿으면 사용자가 의도한 서비스 이름을 놓칠 수 있다.

그래서 정답을 하나로 단정하지 않고 여러 구조 후보를 만든다.

예를 들면 이렇게 보여줄 수 있다.

```text
구조 후보 1:
- 쇼핑몰 백엔드
- 관리자 API
- 정적 웹사이트

구조 후보 2:
- 쇼핑몰 백엔드 + 관리자 API
- 정적 웹사이트

구조 후보 3:
- VPC 중심 묶음
- S3 정적 웹사이트 묶음
```

### 후보 표시 방식

후보는 글 목록보다 보드 미리보기를 우선한다.

사용자는 작은 보드 그림을 보고 맞는 후보를 고른다.

### 완료 기준

- 스캔 결과가 하나의 보드로 바로 고정되지 않는다.
- 여러 구조 해석 후보를 만들 수 있는 구조가 정해진다.
- 후보는 보드 미리보기 중심으로 보여준다.
- 각 후보는 전체 Resource를 포함하고, 묶음 방식만 다르게 가진다.
- 구조 후보는 최대 3개까지만 보여준다.
- Reverse Engineering 전용 화면에서 `시작 방식 다시 선택`을 누르면 `/workspace/new`로 돌아간다.
- 후보를 적용하면 프로젝트를 만들고 Architecture Board로 이동한다.

## 마일스톤 3. Resource Explorer 제한 표시

### 목표

AWS를 다 읽지 못한 상황을 숨기지 않는다.

### 기본 방향

Resource Explorer를 쓸 수 있으면 넓게 찾는다.

Resource Explorer를 못 쓰면 주요 AWS API로 가능한 만큼 읽는다.

그리고 화면에 제한을 알려준다.

### 사용자에게 보여줄 말

아래처럼 쉽게 보여준다.

```text
전체 발견 범위가 제한됐습니다.
주요 리소스는 직접 조회했지만, 일부 서비스는 누락될 수 있습니다.
```

### 완료 기준

- Resource Explorer 실패가 조용히 무시되지 않는다.
- 사용자가 전체 스캔이 제한됐다는 사실을 알 수 있다.
- 못 읽은 서비스나 권한 부족이 결과 화면에 표시된다.
