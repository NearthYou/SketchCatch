# 000. 그릴링 결정 메모

## 한 줄 결론

이번 결정은 두 가지다.

```text
Reverse Engineering은 새 프로젝트를 만들기 전에 AWS를 먼저 읽고, 사용자가 후보를 고른 뒤 프로젝트를 만든다.

Template은 보드에서 큰 모달로 고르고, 선택하면 보드를 바로 템플릿 구조로 덮어쓴다.
```

## 1. Reverse Engineering 시작 위치

Reverse Engineering은 새 프로젝트 생성 흐름에서 선택한다.

가장 큰 이유는 작업 화면 오른쪽 패널의 모드 토글을 줄이기 위해서다.

현재 작업 화면 오른쪽에는 아래 같은 모드가 한 줄에 같이 들어간다.

```text
Resource
Terraform
Issues
Deploy
Reverse Engineering
```

여기에 기능이 계속 추가되면 사용자가 어떤 버튼을 눌러야 하는지 더 헷갈린다.

그래서 Reverse Engineering은 기존 작업 화면의 오른쪽 패널 토글에서 빼고, 새 프로젝트 시작 방식으로 옮긴다.

즉, Reverse Engineering은 "작업 중에 오른쪽 패널에서 잠깐 여는 기능"이 아니라 "기존 AWS에서 새 프로젝트를 시작하는 방법"으로 본다.

사용자는 새 프로젝트를 만들 때 아래 중 하나를 고를 수 있어야 한다.

```text
AI로 시작
Reverse Engineering으로 시작
빈 보드로 시작
```

다만 화면에서 셋을 똑같은 크기로 보여주지 않는다.

메인 선택지는 `AI로 시작`과 `Reverse Engineering으로 시작`이다.

`빈 보드로 시작`은 작게 둔다.

여기서 "작게 둔다"는 말은 기능을 없앤다는 뜻이 아니다.

빈 보드로 시작할 수는 있어야 한다.

다만 카드처럼 크게 보여주지 않는다.

빈 보드는 `::after` 같은 보조 라벨 형태의 작은 버튼으로 둔다.

쉽게 말하면 사용자는 누를 수 있지만, 화면의 주인공은 아니다.

다만 이 흐름은 최소 기능만 둔다.

```text
프로젝트 이름 입력
→ 빈 Architecture Board 열기
→ 사용자가 직접 Resource 추가
```

빈 보드에 별도 추천, 자동 배치 마법, 상세 안내 화면을 크게 만들지 않는다.

이유는 SketchCatch의 핵심 가치가 사용자를 빈 화면에 던져두는 것이 아니라, AI나 기존 AWS 상태를 출발점으로 삼게 하는 것이기 때문이다.

여기서 Reverse Engineering을 고르면, 바로 기존 `/workspace` 작업 화면으로 보내지 않는다.

대신 Reverse Engineering 전용 전체 화면으로 이동한다.

이 화면은 기존 AWS를 읽고, 후보를 보여주고, 사용자가 적용을 누를 때 프로젝트를 만드는 시작 화면이다.

쉽게 말하면 아래 흐름이다.

```text
새 프로젝트 만들기
→ Reverse Engineering으로 시작
→ Reverse Engineering 전용 전체 화면
→ AWS 읽기
→ 후보 보기
→ 적용
→ 프로젝트 생성
→ Architecture Board 열기
```

이렇게 하는 이유는 Reverse Engineering 결과가 작지 않기 때문이다.

AWS 연결, 스캔 상태, 후보 보드, 누락된 리소스, 권한 부족, import 제안까지 보여줘야 한다.

이걸 오른쪽 패널이나 작은 모달에 넣으면 화면이 좁고 복잡해진다.

그래서 Reverse Engineering은 오른쪽 패널 기능이 아니라 새 프로젝트를 시작하기 전의 별도 전체 화면으로 둔다.

여기서 Reverse Engineering을 고르면, 바로 빈 프로젝트를 만들지 않는다.

먼저 AWS를 읽고, 읽어온 결과를 보드 후보로 보여준다.

## 2. Reverse Engineering 첫 화면

Reverse Engineering 전용 전체 화면에 들어가면 첫 화면은 설정 화면이 아니다.

가장 크게 보이는 것은 아래 버튼이어야 한다.

```text
기존 AWS 가져오기
```

기본값은 아래처럼 둔다.

```text
검증된 AWS Role
기본 region
전체 스캔
```

region 선택, 가져올 Resource 종류, 고급 필터는 처음부터 크게 펼치지 않는다.

필요할 때만 열 수 있는 고급 설정으로 둔다.

이유는 간단하다.

사용자가 Reverse Engineering을 누른 이유는 설정을 만지고 싶어서가 아니다.

기존 AWS를 가져오고 싶어서다.

그래서 첫 화면은 "설정"보다 "가져오기"가 먼저 보여야 한다.

## 3. AWS Role 연결이 없을 때

AWS Role 연결이 없으면 Reverse 화면 안에서 연결을 처리하지 않는다.

환경설정으로 이동시킨다.

이유는 간단하다.

```text
AWS 연결은 환경설정의 책임이다.
Reverse Engineering은 연결된 AWS를 읽어오는 책임이다.
```

두 책임을 섞으면 화면도 복잡해지고, 나중에 유지보수도 어려워진다.

## 4. 프로젝트를 만드는 시점

Reverse 스캔이 시작됐다고 바로 프로젝트를 만들지 않는다.

프로젝트는 사용자가 후보를 고르고 `적용`을 눌렀을 때 만든다.

흐름은 아래처럼 잡는다.

```text
새 프로젝트 만들기
→ Reverse Engineering 선택
→ Reverse Engineering 전용 전체 화면으로 이동
→ 시작 방식을 바꾸고 싶으면 시작 방식 다시 선택
→ AWS Role 없으면 환경설정으로 이동
→ AWS Role 있으면 기존 AWS 가져오기
→ 최대 3개의 보드 후보 보여주기
→ 사용자가 후보 하나 선택
→ 적용
→ 프로젝트 생성
→ 선택한 후보가 보드에 반영
```

이렇게 하면 실패한 스캔이나 마음에 안 드는 스캔 때문에 빈 프로젝트가 남지 않는다.

Reverse Engineering 전용 화면에서 이전 화면으로 돌아가야 할 때는 `/workspace`로 보내지 않는다.

이 화면은 기존 작업 화면에서 잠깐 들어온 화면이 아니기 때문이다.

새 프로젝트 시작 흐름에서 들어온 화면이므로 돌아갈 곳은 새 프로젝트 시작 방식 선택 화면이다.

버튼 문구는 아래처럼 둔다.

```text
시작 방식 다시 선택
```

이 버튼을 누르면 `/workspace/new`로 이동한다.

적용을 눌러 프로젝트가 만들어진 뒤에는 Architecture Board가 열리는 `/workspace?projectId=...` 흐름으로 이동한다.

## 5. 여러 인프라가 섞여 있을 때

한 AWS 계정 한 region 안에도 여러 인프라가 섞여 있을 수 있다.

예를 들면 아래처럼 섞일 수 있다.

```text
쇼핑몰 서버
테스트용 EC2
개인 S3 버킷
예전 프로젝트 RDS
```

그래서 Reverse Engineering은 정답을 하나로 단정하지 않는다.

가능한 보드 후보를 최대 3개 만든다.

3개보다 많이 만들면 사용자가 고르기 어렵다.

반대로 1개만 만들면 정답을 하나로 단정하는 문제가 생긴다.

그래서 기본은 최대 3개다.

여기서 후보는 Resource 하나가 아니다.

후보는 VPC 하나도 아니다.

후보는 AWS에서 읽어온 전체 Resource를 어떻게 묶어볼지에 대한 `구조 해석안`이다.

쉽게 말하면, 각 후보에는 읽어온 Resource가 모두 들어간다.

다만 Resource끼리 어떤 서비스로 묶이는지, 어떤 Resource가 서로 한 묶음인지가 후보마다 다를 수 있다.

예를 들어 같은 Resource 목록을 보고도 아래처럼 다르게 해석할 수 있다.

```text
구조 후보 1:
- 쇼핑몰 백엔드 묶음
- 관리자 API 묶음
- 정적 웹사이트 묶음

구조 후보 2:
- 쇼핑몰 백엔드 + 관리자 API를 하나의 백엔드 묶음으로 봄
- 정적 웹사이트 묶음

구조 후보 3:
- VPC 기준으로 묶음
- S3는 별도 묶음
```

사용자가 고르는 것은 “Resource 일부”가 아니다.

사용자가 고르는 것은 “전체 Resource를 이렇게 묶어보는 게 맞다”는 해석이다.

이렇게 해야 나중에 배포나 Terraform import로 이어질 때 필요한 Resource가 빠지지 않는다.

따라서 아래는 기본 후보로 보면 안 된다.

```text
VPC 하나 후보
S3 하나 후보
EC2 하나 후보
```

위 항목들은 Resource 일부만 보여주기 때문에 배포 가능한 구조를 놓칠 수 있다.

사용자는 후보 중에서 “전체 Resource가 이렇게 묶이는 게 맞다” 싶은 것을 고른다.

각 후보는 같은 전체 Resource를 가진다.

차이는 Resource 목록이 아니라 Resource 묶음 방식이다.

그래서 화면에서는 후보마다 “어떤 묶음으로 해석했는지”를 쉽게 보여줘야 한다.

Resource 묶음을 추정하는 기준은 하나만 쓰지 않는다.

우선순위는 아래처럼 둔다.

```text
1순위: 배포에 필요한 의존 관계
2순위: VPC/Subnet 같은 포함 관계
3순위: ALB, EC2, RDS 같은 연결 관계
4순위: tag
5순위: 이름 패턴
```

tag만 보고 묶으면 안 된다.

tag는 좋은 힌트지만, 배포 가능한 구조를 보장하지 않는다.

예를 들어 `service=shop` tag가 EC2에만 있고 Subnet, VPC, Security Group에는 없을 수 있다.

그래도 EC2를 다시 배포하거나 import하려면 Subnet, VPC, Security Group 같은 Resource가 같이 필요하다.

그래서 tag보다 배포에 필요한 관계와 포함 관계를 먼저 봐야 한다.

tag가 없거나 애매하면 Resource끼리 어떻게 연결되어 있는지 본다.

예를 들어 ALB, EC2, RDS가 연결되어 있으면 같은 묶음일 가능성이 높다.

둘이 충돌하면 정답을 하나로 단정하지 않는다.

서로 다른 구조 후보를 만들고, 사용자가 맞는 후보를 고르게 한다.

## 6. 후보는 보드 미리보기로 보여준다

후보는 글 목록만 보여주지 않는다.

작은 보드 미리보기로 보여준다.

이유는 사용자가 AWS를 잘 몰라도 그림을 보면 더 쉽게 고를 수 있기 때문이다.

```text
나열식 목록:
VPC, Subnet, EC2, RDS, S3

보드 미리보기:
VPC 안에 Subnet이 있고, 그 안에 EC2와 RDS가 있는 그림
```

우리 제품은 인프라를 글이 아니라 보드로 이해시키는 제품이다.

그래서 후보 선택도 보드 미리보기가 중심이어야 한다.

## 7. Resource Explorer를 못 쓸 때

AWS Resource Explorer는 AWS 안의 리소스를 넓게 찾는 검색 기능이다.

쉽게 말하면 AWS 리소스용 검색창에 가깝다.

하지만 권한이 없거나 계정 설정이 부족하면 Resource Explorer를 못 쓸 수 있다.

그럴 때 Reverse Engineering을 완전히 막지는 않는다.

대신 아래처럼 처리한다.

```text
주요 AWS API로 가능한 만큼 읽는다.
전체 발견 범위가 제한됐다고 화면에 보여준다.
못 읽은 서비스나 권한 부족도 숨기지 않는다.
```

중요한 점은 사용자가 “전체를 완벽히 가져왔다”고 착각하면 안 된다는 것이다.

## 8. Template을 여는 방식

보드에서 Template을 열 때는 큰 모달로 연다.

템플릿 메뉴에서 들어가면 페이지로 볼 수 있다.

하지만 안에 들어가는 목록, 검색, 필터, 정렬, 태그 UI는 같은 것을 사용한다.

쉽게 말하면 아래와 같다.

```text
템플릿 메뉴에서 열기
→ 템플릿 보관함 페이지

보드에서 열기
→ 같은 템플릿 보관함을 큰 모달로 표시
```

## 9. Template 적용 방식

Template을 선택하면 현재 보드를 템플릿 구조로 덮어쓴다.

현재 보드에 섞어 넣지 않는다.

확인 팝업도 띄우지 않는다.

대신 덮어쓰기 직전에 자동 백업을 남긴다.

흐름은 아래처럼 잡는다.

```text
보드에서 Template 열기
→ 큰 모달 표시
→ Template 선택
→ 현재 보드 자동 백업
→ Template 구조로 보드 덮어쓰기
```

이 방식은 빠르다.

그리고 실수했을 때 최소한 되돌릴 길을 남긴다.

## 이번에 깊게 정하지 않은 것

아래는 이번 문서에서 깊게 정하지 않는다.

- Template 저장 권한
- Template 공유
- Template 마켓
- 기존 프로젝트와 Reverse 결과 병합
- 여러 Reverse 후보를 합쳐서 하나의 보드로 만드는 기능

이것들은 나중에 고도화 문서나 별도 이슈에서 다룬다.
