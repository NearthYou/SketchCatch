# AI 마일스톤 3, 4, 5 그릴링 결정

이 문서는 gg AI 파트에서 Milestone 3, 4, 5를 이번 작업에 어떻게 담을지 정리한다.

## 결정 1: 이번 작업 범위

이번 작업은 Milestone 3을 완료하고, Milestone 4는 최소 세로 조각으로 시작한다.

Milestone 5는 새 범위를 크게 늘리지 않고, 이미 있는 Terraform 오류 설명 기능을 데모 화면에서 확인할 수 있게 연결하는 수준으로 제한한다.

## 이유

Milestone 3은 이미 `Pre-Deployment Check` fallback 규칙과 checklist 구조가 들어가 있어 마무리하기 좋다.

Milestone 4는 실제 부하 테스트나 AWS 지표 분석은 하지 않지만, 단순 `Check Finding` 요약으로 축소하지 않는다.

이번 MVP의 `Design Simulation`은 `ArchitectureJson`의 Resource와 Edge를 보고 요청 흐름, 병목 후보, 실패 시나리오를 계산하는 별도 세로 조각으로 둔다.

Milestone 5를 Plan/Apply 오류까지 넓히면 ck 배포 파트와 강하게 엮이므로 이번 작업에서는 확장하지 않는다.

## 이번 작업에 포함

- Pre-Deployment Check finding과 checklist 정리
- `ArchitectureSuggestion` 타입으로 구조화된 수정 제안 표시
- Design Simulation 별도 API의 최소 세로 조각
- 요청 흐름, 병목 후보, 실패 시나리오
- 병목, 비용, 보안 변화 요약
- Terraform 오류 설명 데모 연결

## 이번 작업에서 제외

- 실제 AWS 성능 측정
- 실제 부하 테스트
- Terraform CLI 실행
- Terraform Plan/Apply 전체 오류 흐름
- 사용자의 승인 없는 자동 수정
- Architecture Board 자동 변경 적용
- 외부 LLM 연결

## 결정 2: Design Simulation API 분리

`Design Simulation`은 `Pre-Deployment Check` 응답 안에 묶지 않고 별도 API로 둔다.

이유:

- `CheckFinding` 요약만으로는 Simulation이라고 부르기 어렵다.
- 사용자가 보고 싶은 것은 위험 목록이 아니라 이 Practice Architecture가 어떻게 요청을 처리하고 어디서 막히는지다.
- 나중에 트래픽 수치, 예산, 설계 버전 비교, 실제 지표와 연결될 가능성이 높다.
- 별도 API로 두면 `Pre-Deployment Check`는 배포 전 위험 점검, `Design Simulation`은 설계 흐름과 한계 분석으로 역할이 분명해진다.

MVP 입력 후보:

```ts
{
  architectureJson: ArchitectureJson;
  trafficLevel: "small" | "normal";
  budgetLevel: "low" | "normal";
}
```

MVP 응답 후보:

```ts
{
  assumptions: string[];
  requestFlow: string[];
  bottlenecks: SimulationBottleneck[];
  failureScenarios: SimulationFailureScenario[];
  costPressure: string[];
  recommendations: string[];
}
```

최소 완료 기준:

- `ArchitectureJson.nodes`와 `ArchitectureJson.edges`를 보고 요청 흐름을 만든다.
- `CheckFinding`은 참고할 수 있지만 결과의 본체가 되면 안 된다.
- 병목 후보는 Resource와 연결된다.
- 실패 시나리오는 단일 EC2, 단일 RDS, open/public Resource 같은 구조적 약점을 설명한다.
- 결과는 실제 성능 측정이 아니라 가정 기반 추정이라고 분명히 말한다.

## 결정 3: 수정 제안은 별도 Architecture Suggestion으로 분리

수정 제안은 `CheckFinding.recommendation`만 강화하지 않고, 별도 `ArchitectureSuggestion` 타입으로 둔다.

역할 분리:

- `CheckFinding.recommendation`: 사용자가 바로 읽는 짧은 권장 문장
- `ArchitectureSuggestion`: 어떤 Resource를 어떻게 바꾸자는 제안인지 표현하는 구조화된 결과

이유:

- 문장만 있으면 보드에서 적용 전/후 비교, 영향 요약, 리소스 연결을 하기 어렵다.
- Milestone 4는 단순 설명이 아니라 제품에서 다룰 수 있는 수정 제안의 시작점이어야 한다.
- 자동 적용은 위험하지만, 적용 후보를 구조화해서 보여주는 것은 MVP에서도 가치가 있다.

MVP 타입 후보:

```ts
type ArchitectureSuggestion = {
  id: string;
  findingId?: string;
  title: string;
  targetResourceId?: string;
  action: "modify_resource" | "add_resource" | "remove_resource" | "manual_review";
  expectedImpact: {
    cost: "decrease" | "increase" | "neutral" | "unknown";
    security: "improve" | "weaken" | "neutral" | "unknown";
    reliability: "improve" | "weaken" | "neutral" | "unknown";
  };
  explanation: string;
};
```

이번 작업에서 하지 않는 것:

- 제안을 Architecture Board에 자동 적용
- Resource diff를 실제로 생성
- Terraform 코드를 자동 재생성
- 사용자가 "비용 줄여줘", "보안 강화해줘"라고 말하면 설계를 직접 바꾸는 흐름

## 결정 4: Terraform 오류 설명 데모는 새 입력 영역으로 분리

Terraform 오류 설명 데모는 Terraform Preview 패널에 합치지 않고, `/workspace/ai` 안에 별도 입력 영역으로 둔다.

이유:

- Terraform Preview 설명은 "이 코드가 무엇을 만드는지"를 설명하는 기능이다.
- Terraform 오류 설명은 "실패 로그가 왜 났고 다음에 무엇을 해야 하는지"를 설명하는 기능이다.
- 두 기능을 한 패널에 합치면 UI는 짧아지지만, 사용자가 넣어야 하는 입력과 사용 순간이 섞인다.
- 발표와 팀원 연동에서도 새 입력 영역이 더 설명하기 쉽다.

MVP UI 입력:

```text
Terraform 오류 설명
- stage 선택: validate / export / plan / apply
- rawMessage 입력
- relatedResourceId 선택 입력
- 오류 설명 생성 버튼
```

이번 작업에서 하지 않는 것:

- Terraform CLI 실행
- 실제 Plan/Apply 실행 결과 수집
- ck 배포 이력과 자동 연결
- 오류 설명 저장

## 그릴링 완료 결정 요약

1. Milestone 3은 Pre-Deployment Check 마무리로 둔다.
2. Design Simulation은 별도 API로 분리하고, 요청 흐름/병목/실패 시나리오를 포함한다.
3. 수정 제안은 `ArchitectureSuggestion` 별도 타입으로 둔다.
4. Terraform 오류 설명 데모는 `/workspace/ai`의 별도 입력 영역으로 둔다.
