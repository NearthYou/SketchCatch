# jh Architecture Board 선택지

이 문서는 jh 보드 파트에게 묻는 문서다.

쉽게 말하면:

> gg AI가 만든 설계 초안을 jh 보드가 어떻게 받을지 정하는 문서다.

## 1. ResourceType을 어떻게 맞출까?

AI가 이런 리소스를 만든다.

```text
VPC
SUBNET
EC2
RDS
S3
SECURITY_GROUP
CLOUDFRONT
```

보드도 같은 이름을 알아야 한다.

### A. 공통 ResourceType 그대로 사용. 추천

보드와 AI가 같은 문자열을 쓴다.

예:

```text
AI: SECURITY_GROUP
보드: SECURITY_GROUP
```

가장 안전하다.

### B. 보드에서 따로 변환

예:

```text
AI: SECURITY_GROUP
보드 표시: Security Group
```

표시는 바꿔도 되지만 저장값은 같아야 한다.

### C. 보드 전용 타입 사용

보드가 자기 타입을 따로 만든다.

연결할 때 변환 코드가 필요해져서 위험하다.

## 2. AI 초안을 보드가 어떻게 받을까?

AI 응답에는 설계도와 설명이 같이 온다.

```ts
{
  architectureJson,
  title,
  metadata
}
```

### A. 보드는 `architectureJson`만 필수로 받기. 추천

보드는 설계도만 있으면 열릴 수 있어야 한다.

AI 설명은 없어도 된다.

### B. 보드가 metadata까지 필수로 받기

AI 설명을 같이 보여주기 쉽다.

하지만 보드가 AI에 너무 묶인다.

### C. 보드 전용 데이터로 변환해서 받기

변환 코드가 필요하다.

연결 오류가 생기기 쉽다.

## 3. 경고는 어느 노드에 붙일까?

AI가 이런 경고를 만든다.

```text
ec2-api에 SSH가 열려 있어요.
```

그러면 보드는 `ec2-api` 노드에 경고를 붙여야 한다.

### A. `resourceId`로 연결. 추천

```text
CheckFinding.resourceId === ArchitectureJson.nodes[].id
```

가장 단순하다.

### B. Resource type으로 연결

예: 모든 EC2에 경고 붙이기

같은 타입이 여러 개면 헷갈린다.

### C. 위치나 label로 연결

이름이 바뀌면 깨질 수 있다.

## 4. Resource config 검증은 누가 할까?

리소스마다 설정값이 있다.

예:

```text
EC2 instanceType
RDS engine
S3 bucketName
```

### A. 보드는 기본 표시와 빈값 체크만 담당. 추천

깊은 AWS 검증은 sw/gg/ck와 나눠서 한다.

### B. 보드가 모든 필수값을 검증

보드 일이 너무 커진다.

### C. 보드는 검증하지 않음

사용자가 빠진 값을 늦게 알게 된다.

## jh에게 받고 싶은 답

```text

3. 경고 연결: A/B/C

추가로 필요한 필드:
- ...
```
