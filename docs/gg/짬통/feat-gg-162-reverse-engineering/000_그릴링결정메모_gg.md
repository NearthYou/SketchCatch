# Reverse Engineering 그릴링 결정 메모

이 문서는 `Feat: Provider Adapter 기반 Reverse Engineering 구현` 작업을 시작하기 전에 정한 내용을 적는 곳이다.

쉽게 말하면:

> 기존 AWS 리소스를 가져올 때, 사용자가 어떤 선택을 하고 어떤 결과를 확인해야 하는지 정리한다.

## 1. 시작 화면

### 결정

Reverse Engineering 시작 화면은 아래 입력을 받는다.

```text
AWS 연결
리전
가져올 리소스 종류
```

선택한 안은 B다.

```text
B. AWS 연결, 리전, 가져올 리소스 종류를 고르고 스캔 시작
```

## 2. 왜 이렇게 하나

AWS 연결과 리전만 고르고 바로 스캔하면 너무 넓다.

사용자는 보통 이런 상황일 수 있다.

```text
VPC 구조만 보고 싶다.
EC2와 Security Group만 가져오고 싶다.
S3 bucket만 확인하고 싶다.
RDS는 비용이 무서워서 일단 제외하고 싶다.
```

그래서 제품 기준으로는 처음부터 리소스 종류 필터가 있어야 한다.

## 3. 첫 구현 단위

처음부터 모든 AWS 리소스를 다 잘 가져오는 것이 목표가 아니다.

하지만 기능을 작게 만들더라도 제품 방향은 줄이지 않는다.

첫 구현 단위는 아래 리소스 종류 필터를 지원한다.

```text
VPC
Subnet
Internet Gateway
Route Table
Security Group
EC2
RDS
S3
```

사용자는 이 중 하나 이상을 고를 수 있어야 한다.

## 4. 지켜야 할 경계

- 프론트에서 AWS SDK를 직접 호출하지 않는다.
- 스캔 결과를 사용자가 확인하기 전 기존 Practice Architecture나 ProjectDraft를 덮어쓰지 않는다.
- 실제 Terraform import는 실행하지 않는다.
- import 명령어는 제안으로만 보여준다.
- Git/CI/CD PR 생성은 자동으로 하지 않는다.
- raw AWS response나 credential은 API 응답에 그대로 내보내지 않는다.

## 5. 리소스 종류 필터 기본값

### 결정

리소스 종류 필터는 처음에 전체 선택 상태로 둔다.

선택한 안은 A다.

```text
A. 전체 선택이 기본
```

처음 화면에서 기본으로 선택되는 리소스:

```text
VPC
Subnet
Internet Gateway
Route Table
Security Group
EC2
RDS
S3
```

## 6. 왜 이렇게 하나

Reverse Engineering은 기존 AWS 상태를 복원하는 제품 기능이다.

사용자가 "가져오기"를 눌렀는데 일부 리소스가 기본으로 빠져 있으면, 처음 결과가 불완전해 보일 수 있다.

그래서 기본값은 전체 선택으로 둔다.

대신 사용자가 원하면 RDS나 S3 같은 리소스를 직접 제외할 수 있어야 한다.

## 7. 지켜야 할 점

전체 선택이 기본이라고 해서 바로 기존 프로젝트를 바꾸면 안 된다.

흐름은 반드시 이렇다.

```text
스캔
→ 결과 확인
→ Practice Architecture 후보 확인
→ 사용자가 적용 버튼 클릭
→ 그때 imported ArchitectureSnapshot 생성
```

즉 전체 선택은 "많이 읽어오기"일 뿐이고, "바로 적용"이 아니다.

## 8. 보드 적용 방식

### 결정

스캔 결과는 현재 보드와 비교한 뒤 적용한다.

선택한 안은 C다.

```text
C. 현재 보드와 비교해서 추가/변경/삭제 후보를 보여준 뒤 적용
```

## 9. 왜 이렇게 하나

Reverse Engineering의 제품 가치는 단순히 기존 AWS 리소스를 보여주는 것이 아니다.

진짜 가치는 이거다.

```text
현재 보드에 있는 설계
실제 AWS에 있는 리소스
두 상태의 차이
```

이 차이를 보여줘야 사용자가 아래 질문에 답할 수 있다.

```text
AWS에는 있는데 보드에는 없는 리소스가 뭐지?
보드에는 있는데 AWS에는 없는 리소스가 뭐지?
둘 다 있는데 설정이 다른 리소스가 뭐지?
가져온 결과를 적용하면 무엇이 바뀌지?
```

## 10. 적용 전 비교 결과

스캔 결과를 적용하기 전에 최소한 아래 후보를 보여준다.

```text
추가 후보
변경 후보
삭제 후보
주의 후보
```

뜻은 이렇다.

| 후보 | 의미 |
| --- | --- |
| 추가 후보 | AWS에는 있는데 현재 보드에는 없는 Resource |
| 변경 후보 | 둘 다 있지만 설정이나 관계가 다른 Resource |
| 삭제 후보 | 현재 보드에는 있는데 AWS에는 없는 Resource |
| 주의 후보 | 자동 비교가 애매해서 사용자가 직접 봐야 하는 Resource |

## 11. 적용 정책

- 사용자가 확인하기 전에는 현재 보드나 ProjectDraft를 바꾸지 않는다.
- 적용 버튼을 누르기 전에 무엇이 바뀌는지 보여준다.
- 전체 적용과 항목별 선택 적용을 고려한다.
- 삭제 후보는 특히 별도 확인이 필요하다.
- 적용 후에는 imported source를 가진 ArchitectureSnapshot 또는 User-Accepted Change 기록이 남아야 한다.

## 12. 첫 구현 단위

첫 구현에서는 비교 결과를 아래 수준까지 만든다.

```text
추가 후보
주의 후보
```

변경 후보와 삭제 후보도 제품 방향에는 포함한다.

다만 정확한 변경/삭제 비교는 현재 보드의 Resource identity와 AWS provider identity 연결 기준이 필요하다.

그래서 구현할 때는 다음 순서로 간다.

```text
1. scan result를 ArchitectureJson 후보로 만든다.
2. 현재 보드와 id/providerResourceId 기준으로 비교한다.
3. AWS에는 있는데 보드에 없는 것을 추가 후보로 보여준다.
4. 자동 비교가 어려운 것은 주의 후보로 보여준다.
5. 변경/삭제 후보는 같은 구조 안에서 확장 가능하게 둔다.
```

## 13. 비교 기준

### 결정

현재 보드와 AWS 스캔 결과는 두 기준을 함께 사용해 비교한다.

선택한 안은 C다.

```text
C. providerResourceId와 Terraform identity 둘 다 사용
```

## 14. 비교 우선순위

비교 순서는 아래처럼 간다.

```text
1. providerResourceId로 먼저 비교
2. 없으면 Terraform identity로 보조 비교
3. 둘 다 없거나 애매하면 주의 후보로 표시
```

## 15. 왜 이렇게 하나

Reverse Engineering은 실제 AWS 상태에서 출발한다.

그래서 실제 AWS 리소스 ID가 가장 강한 기준이다.

예:

```text
vpc-123
subnet-123
i-123
sg-123
bucket-name
```

하지만 SketchCatch는 결국 IaC Preview와 Terraform import로 이어진다.

그래서 Terraform identity도 필요하다.

예:

```text
aws_vpc.main
aws_instance.web
aws_s3_bucket.assets
```

둘 다 써야 아래 상황을 처리할 수 있다.

```text
이미 AWS에 있는 리소스를 처음 가져오는 경우
이미 Terraform으로 관리하던 리소스를 다시 비교하는 경우
보드에는 Terraform 이름만 있고 AWS ID가 아직 없는 경우
AWS ID는 있지만 Terraform 이름을 아직 정하지 않은 경우
```

## 16. 비교 결과를 만들 때 주의할 점

- `providerResourceId`가 같으면 같은 리소스로 본다.
- `providerResourceId`가 없으면 Terraform identity로 보조 비교한다.
- 둘 다 다르면 다른 리소스로 본다.
- 비교 기준이 부족하면 삭제나 변경으로 단정하지 않고 주의 후보로 둔다.
- 사용자가 원하지 않는 삭제 후보가 생기지 않도록 삭제 판단은 가장 보수적으로 한다.

## 17. Import suggestion 범위

### 결정

Import suggestion은 아래 세 가지를 만든다.

선택한 안은 C다.

```text
C. 명령어 + Terraform block 초안 + Git/CI/CD handoff 준비 데이터
```

## 18. Import suggestion에 들어갈 것

각 리소스마다 아래 정보를 만든다.

```text
terraform import 명령어
최소 Terraform block 초안
Git/CI/CD handoff에 넘길 수 있는 준비 metadata
```

예:

```text
terraform import aws_instance.web i-123
```

Terraform block 초안 예:

```hcl
resource "aws_instance" "web" {
  # import 후 실제 설정을 채워야 함
}
```

Git/CI/CD handoff 준비 metadata 예:

```text
어떤 리소스를 import할지
어떤 Terraform 파일 후보에 넣을지
어떤 risk/cost finding이 붙어 있는지
PR을 만들 준비가 되었는지
```

## 19. 절대 자동으로 하지 않는 것

아래는 자동 실행하지 않는다.

```text
terraform import 실제 실행
Terraform 파일 실제 commit
GitHub PR 실제 생성
기존 ProjectDraft 덮어쓰기
배포 실행
```

이 작업들은 전부 User-Accepted Change가 필요하다.

## 20. 왜 이렇게 하나

Reverse Engineering 결과는 단순히 보기만 하면 제품 가치가 약하다.

제품 흐름은 여기까지 이어져야 한다.

```text
기존 AWS 상태 스캔
→ Practice Architecture 후보
→ IaC Preview/import 준비
→ Git/CI/CD handoff 준비
→ 사용자가 승인하면 다음 단계 진행
```

즉 준비는 충분히 해주되, 실행은 사용자가 확인한 뒤에만 한다.

## 21. High Risk 처리

### 결정

High Risk가 있어도 Reverse Engineering 결과 적용 버튼은 막지 않는다.

선택한 안은 A다.

```text
A. 결과는 보여주되 적용 버튼은 막지 않음
```

## 22. 왜 이렇게 하나

Reverse Engineering은 위험한 상태를 숨기는 기능이 아니다.

오히려 기존 AWS 상태에 어떤 위험이 있는지 보려고 쓰는 기능이다.

예:

```text
public RDS
SSH 0.0.0.0/0
S3 public access
과도한 IAM 권한
```

이런 위험이 있다고 해서 가져오기 자체를 막으면, 사용자가 현재 AWS 상태를 보드에서 확인하고 정리할 기회도 줄어든다.

그래서 결과 적용은 막지 않는다.

## 23. 대신 반드시 보여줄 것

적용을 막지는 않지만, 위험을 조용히 넘기면 안 된다.

화면에는 최소한 아래가 보여야 한다.

```text
High Risk 개수
어떤 Resource에 붙은 위험인지
왜 위험한지
어떻게 고치면 되는지
이 상태로 운영 흐름에 넘기면 추가 확인이 필요하다는 안내
```

## 24. 안전 경계

High Risk가 있어도 아래는 가능하다.

```text
스캔 결과 보기
Practice Architecture 후보로 적용
import suggestion 확인
Git/CI/CD handoff 준비 metadata 생성
```

하지만 아래는 여전히 자동으로 하지 않는다.

```text
실제 Terraform import 실행
GitHub PR 자동 생성
배포 자동 실행
기존 AWS 리소스 변경
```

실제 PR 생성, 배포 실행, destroy 같은 상태 변경은 별도 User-Accepted Change와 Deployment Safety Gate에서 다시 확인한다.

## 25. 스캔 결과 첫 화면

### 결정

스캔 결과 화면에서는 복원된 Practice Architecture 미리보기를 가장 먼저 보여준다.

선택한 안은 A다.

```text
A. 복원된 Practice Architecture 미리보기
```

## 26. 왜 이렇게 하나

사용자가 Reverse Engineering을 누른 이유는 먼저 "현재 AWS가 어떤 구조인지" 보고 싶기 때문이다.

그래서 첫 화면의 중심은 구조 미리보기다.

```text
VPC 안에 어떤 Subnet이 있는지
EC2가 어느 Subnet에 있는지
Security Group이 어떤 리소스에 붙어 있는지
RDS나 S3가 같이 있는지
```

이걸 먼저 봐야 finding이나 비교 결과도 이해할 수 있다.

## 27. 함께 보여줄 보조 정보

Practice Architecture 미리보기가 중심이지만, 아래 정보도 같은 화면에 같이 보여준다.

```text
현재 보드와 비교 결과
High Risk / Cost Risk 요약
Import suggestion 요약
Git/CI/CD handoff 준비 상태
```

즉 화면 우선순위는 이렇게 간다.

```text
1. 복원된 Practice Architecture 미리보기
2. 현재 보드와 비교 결과
3. 위험/비용 finding
4. import suggestion
5. handoff 준비 상태
```

## 28. Import suggestion 표시 방식

### 결정

Import suggestion은 두 방식으로 보여준다.

선택한 안은 C다.

```text
C. 리소스별 카드 + 전체 복사 탭
```

## 29. 리소스별 카드

기본 화면은 리소스별 카드다.

각 카드에는 아래를 보여준다.

```text
Resource 이름
Resource 종류
AWS providerResourceId
terraform import 명령어
Terraform block 초안
연결된 risk/cost finding
Git/CI/CD handoff 포함 여부
```

리소스별 카드가 필요한 이유:

```text
사용자가 어떤 리소스를 import하려는지 이해하기 쉽다.
위험 finding을 해당 리소스와 연결해서 볼 수 있다.
필요한 리소스만 선택 적용하기 쉽다.
```

## 30. 전체 복사 탭

추가로 전체 복사 탭을 둔다.

전체 복사 탭에는 아래를 묶어서 보여준다.

```text
전체 terraform import 명령어
전체 Terraform block 초안
Git/CI/CD handoff에 넘길 import plan 요약
```

전체 복사 탭이 필요한 이유:

```text
실제 작업할 때 명령어를 한 번에 확인하기 쉽다.
리뷰나 문서 공유에 편하다.
Git/CI/CD handoff 준비 상태를 한눈에 볼 수 있다.
```

## 31. 실제 AWS adapter 구현 시점

### 결정

실제 AWS adapter는 처음 구현 범위에 포함한다.

선택한 안은 A다.

```text
A. 처음부터 실제 AWS read-only API까지 붙인다
```

## 32. 왜 이렇게 하나

Reverse Engineering은 기존 AWS 상태를 가져오는 제품 기능이다.

fixture나 mocked adapter만 있으면 흐름 검증은 가능하지만, 사용자가 느끼기에는 아직 진짜 기능이 아니다.

그래서 처음부터 실제 AWS read-only 조회까지 포함한다.

여기서 중요한 말은 `read-only`다.

```text
조회만 한다.
수정하지 않는다.
삭제하지 않는다.
생성하지 않는다.
import도 실행하지 않는다.
```

## 33. 실제 AWS adapter가 해야 할 일

AWS adapter는 아래 리소스를 조회한다.

```text
VPC
Subnet
Internet Gateway
Route Table
Security Group
EC2
RDS
S3
```

조회한 raw AWS response를 그대로 프론트로 보내지 않는다.

반드시 아래 모양으로 바꾼다.

```text
DiscoveredResource[]
```

즉 API와 화면은 AWS SDK 결과를 직접 보지 않는다.

## 34. 필요한 AWS SDK

현재 패키지에는 STS, S3, Bedrock, Amazon Q 관련 SDK는 있다.

하지만 EC2와 RDS 조회를 하려면 아래 SDK가 추가로 필요할 수 있다.

```text
@aws-sdk/client-ec2
@aws-sdk/client-rds
```

S3는 이미 `@aws-sdk/client-s3`가 있다.

dependency를 추가하면 아래를 꼭 확인한다.

```text
apps/api/package.json
pnpm-lock.yaml
```

## 35. 테스트 기준

실제 AWS API는 테스트에서 직접 호출하지 않는다.

테스트에서는 AWS gateway를 가짜로 주입한다.

```text
실제 코드
→ AWS SDK 호출

테스트
→ fake AWS gateway 결과 사용
```

이렇게 해야 테스트가 빠르고 안전하다.

## 36. 안전 경계

실제 AWS adapter를 붙여도 아래는 절대 하지 않는다.

```text

AWS 리소스 삭제
terraform import 실행
Terraform apply 실행

```

Reverse Engineering의 AWS 권한은 조회 권한 중심이어야 한다.


선택한 안은 C다.


Reverse Engineering은 실제 AWS 상태를 읽어오는 기능이다.

그래서 실패했을 때도 사용자가 아래를 알아야 한다.

```text
어떤 리소스 조회가 실패했는지
왜 실패했는지
다시 시도할 수 있는지
```

그냥 `조회 실패`라고만 보여주면 사용자는 무엇을 고쳐야 하는지 알 수 없다.

예를 들어 아래 상황은 전부 다르다.

```text
RDS 권한이 없음
잘못된 region을 고름

일시적인 AWS API 오류가 남
```

이걸 하나로 뭉개면 나중에 디버깅이 어렵다.

## 41. 구현 방향

scan 결과는 전체 성공/실패 하나로만 보지 않는다.

성공한 리소스와 실패한 조회를 분리한다.

```text
성공한 것
→ DiscoveredResource[]에 넣는다

실패한 것
→ scan error summary에 넣는다
```

실패 정보에는 최소한 아래가 있어야 한다.

```text
resourceType
stage
reason
message
retryable
```

뜻은 이렇다.

```text
resourceType
→ 어떤 리소스를 읽다 실패했는지
예: RDS, S3, EC2

stage
→ 어느 단계에서 실패했는지
예: credential, region, provider_api, normalize

reason
→ 실패 종류
예: permission_denied, invalid_region, expired_credential, throttled, unknown

message
→ 사용자가 볼 쉬운 설명

retryable
→ 다시 시도 버튼을 보여줘도 되는지
```

## 42. 화면에서 보여줄 방식

```text

3. 실패한 리소스별 원인
4. 다시 시도 가능한 항목의 retry 버튼
```

중요한 점은, 일부 실패가 있어도 성공한 결과까지 버리지 않는다는 것이다.

단, 실패한 리소스가 있으면 아래처럼 분명히 표시한다.

```text

```

## 43. 안전 경계

```text

실패한 리소스를 삭제 후보로 만들지 않는다.

```

그건 `RDS를 못 읽었다`가 맞다.

선택지는 세 가지였다.

```text
A. 수정 못 하게 하고 보기만 한다
B. 적용 전에 사용자가 수정할 수 있게 한다
C. 수정할 수 있게 하되, 원본 scan 결과와 사용자가 바꾼 내용을 따로 저장한다
```

선택한 안은 C다.

```text
C. 수정할 수 있게 하되, 원본 scan 결과와 사용자가 바꾼 내용을 따로 저장한다
```

## 69. 왜 이렇게 하나

AWS scan 결과는 원본이다.

사용자가 수정한 내용은 원본이 아니라 해석이다.

예를 들어 AWS에서 가져온 이름은 이럴 수 있다.

```text
sg-04ab1234
```

사용자는 화면에서 보기 쉽게 이렇게 바꾸고 싶을 수 있다.

```text
DB Security Group
```

이때 원본 값을 덮어쓰면 안 된다.

나중에 Terraform import나 AWS 재조회에 필요한 값은 원본 AWS 값일 수 있기 때문이다.

그래서 아래처럼 나눈다.

```text
원본 scan 결과
→ AWS에서 실제로 읽은 값

사용자 수정값
→ 화면에서 보기 좋게 고친 이름, 위치, 설명
```

## 70. 구현 방향

Reverse Engineering Draft는 수정 가능하게 둔다.

하지만 수정값은 원본 scan 결과를 덮어쓰지 않는다.

쉽게 말하면:

```text
Reverse Engineering Scan
→ 원본 AWS 값 보관

Reverse Engineering Draft
→ 사용자가 고친 후보 설계 보관
```

사용자가 바꿀 수 있는 값은 먼저 아래 정도로 제한한다.

```text
Resource 표시 이름
Resource 위치
Resource 설명
```

반대로 아래는 함부로 바꾸면 안 된다.

```text
providerResourceId
providerResourceType
region
accountId
terraform import id
```

이 값들은 AWS 원본 추적과 import에 쓰일 수 있기 때문이다.

## 71. 화면에서 보여줄 방식

사용자가 수정할 수 있는 값과 수정하면 안 되는 값을 나눠 보여준다.

예를 들면:

```text
수정 가능
→ 화면 표시 이름, 설명, 위치

읽기 전용
→ AWS Resource ID, region, resource type
```

읽기 전용 값은 숨기지 않는다.

다만 사용자가 실수로 바꾸지 못하게 한다.

## 72. 안전 경계

아래는 하지 않는다.

```text
사용자 수정값으로 AWS 원본 값을 덮어쓰기
표시 이름을 providerResourceId처럼 사용하기
사용자 수정만 보고 Terraform import 명령 만들기
```

Terraform import나 AWS 재조회는 원본 AWS 값을 기준으로 해야 한다.

사용자 수정값은 보드 이해를 돕는 값이다.

## 74. 질문 16

Reverse Engineering Draft에서 사용자가 어디까지 수정할 수 있게 할까?

선택지는 세 가지였다.

```text
A. 표시 이름, 위치, 설명만 수정
B. 표시 이름, 위치, 설명, 관계선까지 수정
C. 리소스 설정값까지 수정
```

선택한 안은 A다.

```text
A. 표시 이름, 위치, 설명만 수정
```

## 75. 왜 이렇게 하나

Reverse Engineering Draft는 AWS에서 읽은 결과를 보드에 올리기 전 후보 설계다.

여기서 사용자가 너무 많은 값을 바꿀 수 있으면 원본 AWS 상태와 사용자가 만든 설계가 섞인다.

예를 들어 사용자가 아래 값을 바꿀 수 있다고 해보자.

```text
EC2 instance type
Security Group inbound rule
RDS public access 설정
Subnet 연결
```

그러면 이 Draft가 더 이상 AWS에서 읽은 상태인지, 사용자가 새로 만든 설계인지 헷갈린다.

그래서 Reverse Engineering Draft 단계에서는 보기 좋게 정리하는 수정만 허용한다.

```text
표시 이름
위치
설명
```

## 76. 수정 가능한 값

사용자가 바꿀 수 있는 값은 아래 세 가지다.

```text
displayName
position
description
```

뜻은 이렇다.

```text
displayName
→ 화면에서 보이는 이름

position
→ Architecture Board에서 보이는 위치

description
→ 사용자가 이해하기 쉽게 적는 설명
```

이 값들은 AWS 원본 추적을 망가뜨리지 않는다.

## 77. 수정하면 안 되는 값

아래 값은 Reverse Engineering Draft에서 직접 수정하지 않는다.

```text
providerResourceId
providerResourceType
region
accountId
resource config
relationship
terraform import id
```

이 값들은 실제 AWS 상태, 관계, import 기준과 연결된다.

수정하려면 별도의 설계 편집 흐름에서 해야 한다.

## 78. 고도화로 남기는 것

나중에는 아래도 열 수 있다.

```text
관계선 수정
Resource 그룹 정리
설정값 수정 제안
scan 원본과 사용자 변경 차이 보기
```

하지만 지금은 Reverse Engineering의 의미를 지키는 것이 먼저다.

AWS에서 읽은 결과를 안전하게 복원하고, 사용자가 보기 좋게 정리하는 정도만 허용한다.

## 80. 질문 17

Reverse Engineering Draft를 적용할 때 기존 보드가 비어 있지 않으면 어떻게 할까?

선택지는 세 가지였다.

```text
A. 적용을 막고 새 프로젝트에서만 적용하게 한다
B. 새 보드로 열기와 현재 보드에 추가 중 고르게 한다
C. 자동으로 현재 보드와 합친다
```

선택한 안은 B다.

```text
B. 새 보드로 열기와 현재 보드에 추가 중 고르게 한다
```

## 81. 왜 이렇게 하나

기존 보드가 비어 있지 않다는 건 사용자가 이미 작업 중이라는 뜻이다.

그 상태에서 scan 결과를 자동으로 합치면 사용자가 원하지 않는 변경이 생길 수 있다.

그래서 적용 전에 선택지를 보여준다.

```text
이미 작업 중인 보드가 있습니다.
새 보드로 열까요, 현재 보드에 추가할까요?
```

## 82. 선택지 의미

사용자가 고르는 선택지는 두 가지다.

```text
새 보드로 열기
→ 기존 보드는 건드리지 않는다
→ scan 결과를 새 보드 상태로 연다

현재 보드에 추가
→ 기존 보드는 유지한다
→ scan 결과 중 추가 가능한 Resource만 더한다
```

여기서 `현재 보드에 추가`는 자동 병합과 다르다.

수정이나 삭제까지 마음대로 하지 않는다.

## 83. 안전 경계

아래는 하지 않는다.

```text
기존 Resource 자동 삭제
기존 Resource 설정 자동 변경
기존 Resource 이름 자동 변경
겹치는 Resource를 임의로 하나로 합치기
```

겹치는지 애매한 Resource는 바로 합치지 않는다.

사용자가 직접 확인해야 하는 항목으로 남긴다.

## 84. 화면에서 보여줄 방식

기존 보드가 비어 있지 않으면 적용 버튼을 누를 때 선택 모달을 보여준다.

예:

```text
이미 작업 중인 Practice Architecture가 있습니다.

[새 보드로 열기]
[현재 보드에 추가]
[취소]
```

선택 전에는 보드를 바꾸지 않는다.

## 85. 고도화로 남기는 것

나중에는 더 똑똑한 병합을 제공할 수 있다.

예:

```text
겹치는 Resource 자동 매칭
설정 차이 비교
관계 차이 비교
충돌만 사용자에게 질문
삭제 후보 별도 확인
```

하지만 이건 Resource identity 기준이 더 단단해야 한다.

다음으로 정해야 할 것은 `현재 보드에 추가`를 선택했을 때 중복 Resource를 어떻게 처리할지다.

## 87. 질문 18

`현재 보드에 추가`를 선택했을 때 중복 Resource는 어떻게 처리할까?

선택지는 세 가지였다.

```text
A. 중복이면 무조건 추가하지 않는다
B. providerResourceId가 같으면 중복으로 보고, 애매하면 사용자가 확인한다
C. 이름이 같으면 중복으로 본다
```

선택한 안은 B다.

```text
B. providerResourceId가 같으면 중복으로 보고, 애매하면 사용자가 확인한다
```

## 88. 왜 이렇게 하나

Resource 이름은 믿으면 안 된다.

사용자가 이름을 바꿀 수 있기 때문이다.

예를 들어 화면 이름은 둘 다 이렇게 보일 수 있다.

```text
DB Security Group
```

하지만 AWS 원본 ID는 다를 수 있다.

```text
sg-1111
sg-2222
```

이 둘은 이름이 같아도 다른 Resource다.

반대로 화면 이름은 달라도 AWS 원본 ID가 같으면 같은 Resource다.

```text
화면 이름: DB Security Group
AWS ID: sg-1111

화면 이름: Database SG
AWS ID: sg-1111
```

이 경우는 같은 Resource로 본다.

## 89. 구현 방향

중복 판단 1순위는 `providerResourceId`다.

```text
providerResourceId가 같다
→ 같은 Resource로 본다
→ 새로 추가하지 않는다
```

`providerResourceId`가 없거나 애매하면 자동으로 판단하지 않는다.

```text
providerResourceId가 없다
providerResourceId가 애매하다
Terraform identity만 비슷하다
이름만 같다
```

이런 경우는 사용자 확인 항목으로 보낸다.

## 90. 사용자 확인 항목

애매한 Resource는 이렇게 보여준다.

```text
같은 Resource인지 확인이 필요합니다.

현재 보드 Resource:
DB Security Group

scan 결과 Resource:
DB Security Group
sg-2222

[새 Resource로 추가]
[추가하지 않음]
```

사용자가 직접 고르기 전에는 자동으로 합치지 않는다.

## 91. 안전 경계

아래는 하지 않는다.

```text
이름만 같다고 중복 처리
이름만 같다고 기존 Resource 덮어쓰기
providerResourceId가 다른데 같은 Resource로 합치기
애매한 Resource를 자동 삭제 후보로 만들기
```

중복 판단은 보수적으로 한다.

확신이 없으면 사용자에게 확인시킨다.

다음으로 정해야 할 것은 scan 결과에서 지원하지 않는 Resource를 발견했을 때 어떻게 보여줄지다.

## 93. 질문 19

scan에서 우리가 아직 지원하지 않는 Resource를 발견하면 어떻게 할까?

예를 들어 AWS에 Lambda나 ALB가 있는데, 아직 Architecture Board에서 제대로 못 그리는 상황이다.

선택지는 세 가지였다.

```text
A. 무시한다
B. 지원하지 않는 Resource로 목록에만 보여준다
C. 보드에 Unknown Resource 카드로 올리고, 나중에 매핑할 수 있게 한다
```

선택한 안은 C다.

```text
C. 보드에 Unknown Resource 카드로 올리고, 나중에 매핑할 수 있게 한다
```

## 94. 왜 이렇게 하나

지원하지 않는 Resource를 숨기면 사용자가 오해할 수 있다.

예를 들어 AWS에는 ALB가 있는데 우리가 무시하면 사용자는 이렇게 생각할 수 있다.

```text
AWS에 ALB가 없구나
```

하지만 정확한 말은 이거다.

```text
AWS에서 ALB를 발견했다.
다만 SketchCatch가 아직 ALB를 정식 Resource로 그리지 못한다.
```

그래서 숨기지 않고 `Unknown Resource`로 보여준다.

## 95. Unknown Resource 뜻

`Unknown Resource`는 이상한 Resource라는 뜻이 아니다.

뜻은 이거다.

```text
AWS에서는 발견했다.
하지만 아직 SketchCatch ResourceType으로 매핑하지 못했다.
```

즉 아래 두 가지를 구분해야 한다.

```text
AWS에 없음
→ 실제로 발견되지 않음

Unknown Resource
→ 발견은 됐지만 아직 정식 지원 전
```

## 96. 구현 방향

지원하지 않는 Resource도 scan 결과에 남긴다.

보드에는 Unknown Resource 카드로 보여준다.

카드에는 최소한 아래를 표시한다.

```text
provider
providerResourceType
providerResourceId
region
displayName
지원 전 Resource라는 설명
```

예:

```text
Unknown Resource
AWS Lambda
arn:aws:lambda:ap-northeast-2:123:function:hello
```

## 97. 안전 경계

Unknown Resource는 아래에 바로 쓰면 안 된다.

```text
Terraform 생성
Terraform import 명령
배포 계획
비용 계산 확정값
보안 판단 확정값
```

아직 우리가 정확히 이해하지 못한 Resource이기 때문이다.

대신 아래처럼 보여준다.

```text
발견은 됐지만 아직 정식 지원 전입니다.
Terraform 생성이나 배포에 포함하려면 ResourceType 매핑이 필요합니다.
```

## 98. 나중에 매핑한다는 뜻

나중에 해당 ResourceType을 지원하게 되면 Unknown Resource를 정식 Resource로 바꿀 수 있어야 한다.

예:

```text
Unknown Resource
→ Lambda

Unknown Resource
→ ALB
```

이때 원본 AWS 값은 이미 남아 있어야 한다.

그래야 나중에 매핑할 수 있다.

## 100. 질문 20

Unknown Resource를 Architecture Board에서 어디에 보여줄까?

선택지는 세 가지였다.

```text
A. 일반 Resource랑 같은 위치에 그냥 보여준다
B. 보드에는 흐리게 표시하고, 별도 미지원 Resource 목록에도 보여준다
C. 보드에는 안 올리고, 오른쪽 패널 목록에만 보여준다
```

선택한 안은 B다.

```text
B. 보드에는 흐리게 표시하고, 별도 미지원 Resource 목록에도 보여준다
```

## 101. 왜 이렇게 하나

Unknown Resource도 실제 AWS에서 발견된 Resource다.

그래서 보드에서 완전히 숨기면 관계를 이해하기 어렵다.

하지만 정식 지원 Resource처럼 보이면 안 된다.

사용자가 이렇게 오해할 수 있기 때문이다.

```text
이 Resource도 Terraform 생성이 되겠지?
이 Resource도 보안/비용 분석이 정확하겠지?
이 Resource도 배포에 포함되겠지?
```

그래서 보드에는 보이게 하되, 흐리게 표시한다.

그리고 별도 목록에도 모아서 보여준다.

## 102. 보드 표시 방식

보드에서는 Unknown Resource를 일반 Resource보다 약하게 보여준다.

예:

```text
흐린 색
점선 테두리
Unknown 배지
미지원 Resource 문구
```

단, 관계선은 보일 수 있어야 한다.

예를 들어 ALB가 EC2와 연결되어 있는데 ALB가 Unknown이면, 사용자는 최소한 “뭔가 EC2 앞에 있다”는 사실은 알아야 한다.

## 103. 미지원 Resource 목록

별도 목록에는 Unknown Resource만 모아서 보여준다.

목록에는 최소한 아래를 표시한다.

```text
providerResourceType
providerResourceId
region
연결된 Resource
지원 전이라는 설명
```

예:

```text
미지원 Resource

AWS ALB
arn:...
연결: EC2 web-server
상태: 아직 SketchCatch 정식 ResourceType으로 매핑되지 않음
```

## 104. 안전 경계

Unknown Resource는 보드에 보여도 정식 지원 Resource처럼 다루지 않는다.

아래 대상에서 제외한다.

```text
Terraform 생성
Terraform import 실행 대상
배포 대상
확정 비용 계산
확정 보안 판단
```

대신 아래처럼 안내한다.

```text
이 Resource는 AWS에서 발견됐지만 아직 SketchCatch에서 정식 지원하지 않습니다.
```

Reverse Engineering 결과에서 비용/보안 finding은 어디까지 계산할까?

선택한 안은 B다.


## 108. 왜 이렇게 하나

Unknown Resource는 AWS에서 발견됐지만 아직 SketchCatch가 정확히 이해하지 못한 Resource다.

그래서 비용이나 보안을 아는 척하면 위험하다.

예를 들어 Unknown Resource가 ALB라면 비용 구조가 있다.

하지만 우리가 아직 ALB를 정식 ResourceType으로 지원하지 않는다면, 정확한 비용/보안 finding을 만들면 안 된다.

그래서 아래처럼 처리한다.

```text

→ 비용/보안 finding 계산

→ 분석 제외 표시
```

## 109. 분석 제외 표시

Unknown Resource는 finding 대상에서 완전히 사라지면 안 된다.

대신 아래처럼 별도 표시한다.

```text
분석 제외 Resource

AWS ALB
이 Resource는 발견됐지만 아직 SketchCatch 정식 ResourceType으로 매핑되지 않았습니다.
비용/보안 분석에서 제외됐습니다.
```

이 표시가 있어야 사용자가 결과를 믿을 때 조심할 수 있다.

## 110. 구현 방향

finding 결과에는 두 종류를 구분한다.

```text

분석 제외 항목
→ Unknown Resource라서 계산하지 못한 항목
```

분석 제외 항목은 high risk finding처럼 겁주지 않는다.

대신 “분석 범위 밖”이라는 것을 알려준다.

## 144. 왜 이렇게 하나

scan 기록에는 AWS 구조 정보가 들어간다.

그래서 사용자가 원하면 삭제할 수 있어야 한다.

하지만 scan 결과를 보고 사용자가 보드에 적용했다면, 그 보드는 사용자의 작업물이다.

scan 기록을 삭제했다고 해서 사용자가 저장한 보드까지 지우면 위험하다.

그래서 둘을 나눈다.

```text
scan 기록
→ AWS에서 읽은 원본 기록
→ 삭제 가능

Architecture Snapshot
→ 사용자가 적용해서 저장한 보드
→ 유지
```

## 145. 삭제 후 표시

scan 기록을 삭제한 뒤에도 연결된 보드가 남아 있을 수 있다.

이때는 이렇게 표시한다.

```text
이 보드는 Reverse Engineering scan에서 시작됐습니다.
하지만 원본 scan 기록은 삭제됐습니다.
```

즉, 출처가 있었다는 사실은 알 수 있지만 원본 내용은 볼 수 없다.

## 146. 구현 방향

삭제는 scan 기록을 대상으로 한다.

```text
DELETE /reverse-engineering/scans/:scanId
```

삭제할 때 아래는 지운다.

```text
scan raw result
scan log
discovered resource 원본
조회 실패 상세
import suggestion
```

아래는 지우지 않는다.

```text
사용자가 적용해서 저장한 Architecture Snapshot
프로젝트 자체
사용자가 직접 만든 보드 변경
```

## 147. 안전 경계

아래는 하지 않는다.

```text
scan 삭제와 함께 최종 보드 삭제
scan 삭제와 함께 프로젝트 삭제
scan 삭제 후에도 원본 AWS 상세를 계속 보여주기
삭제된 scan을 completed scan처럼 조회하게 하기
```

삭제된 scan은 상세 조회할 수 없어야 한다.

단, 연결된 보드에는 “원본 scan 삭제됨” 정도의 표시만 남길 수 있다.

## 148. 고도화로 남기는 것

나중에는 더 세밀한 삭제 정책을 둘 수 있다.

예:

```text
soft delete
삭제 복구
삭제 감사 기록
scan 로그만 삭제
scan raw result만 삭제
보존 기간 자동 만료
```

하지만 기본 기준은 scan 기록과 최종 보드를 분리하는 것이다.
