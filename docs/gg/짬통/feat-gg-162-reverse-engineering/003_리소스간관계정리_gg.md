# Reverse Engineering 리소스 관계 명세서

이 문서는 학습 문서가 아니다.

이 문서는 SketchCatch Reverse Engineering에서 AWS 리소스 간 관계를 어떻게 저장하고, 해석하고, 보드에 보여줄지 정하는 구현 규약이다.

앞으로 Reverse Engineering, Architecture Board, Terraform 변환, AI 분석, 배포 전 점검에서 리소스 관계를 다룰 때 이 문서를 기준으로 삼는다.

## 1. 문서 목적

Reverse Engineering은 기존 AWS 계정에서 리소스를 읽어와서 SketchCatch 보드가 이해할 수 있는 구조로 바꾼다.

이때 가장 중요한 문제는 이것이다.

```text
AWS에는 리소스가 많고 관계도 많다.
그 관계를 전부 같은 방식으로 보드에 그리면 화면이 망가진다.
```

그래서 이 문서는 아래를 고정한다.

```text
어떤 관계 타입을 쓸지
각 관계 타입이 무슨 뜻인지
파라미터에서 관계를 어떻게 찾을지
관계 방향을 어떻게 해석할지
보드에서 어떤 관계를 강하게 보여줄지
어떤 관계를 숨기거나 상세 패널로 보낼지
```

## 2. 적용 범위

이 문서는 아래 기능에 적용한다.

```text
Reverse Engineering Provider Adapter
Reverse Engineering ArchitectureJson 생성
Architecture Board 표시
AI 분석
Terraform import suggestion
Terraform 생성 보조
배포 전 비용/보안/위험 분석
```

이 문서는 아래를 직접 다루지 않는다.

```text
실제 AWS 리소스 생성
실제 AWS 리소스 수정
실제 AWS 리소스 삭제
실제 terraform import 실행
실제 terraform apply 실행
```

## 3. 핵심 원칙

### 3.1 AWS 원본 구조와 화면 구조는 다르다

AWS에서 읽은 모든 관계를 그대로 화면에 그리면 안 된다.

AWS 원본 구조는 기계가 읽기 좋은 구조다.

SketchCatch 보드 구조는 사람이 이해하기 좋은 구조여야 한다.

따라서 Reverse Engineering은 아래 두 단계를 거쳐야 한다.

```text
1. AWS 원본 리소스와 파라미터를 최대한 읽는다.
2. 사람이 이해할 수 있는 관계만 골라 보드에 강하게 보여준다.
```

### 3.2 관계는 전부 같은 선이 아니다

아래 세 문장은 모두 다른 관계다.

```text
Subnet은 VPC 안에 있다.
EC2는 Security Group을 사용한다.
Route Table은 Internet Gateway를 참조한다.
```

이 셋을 모두 같은 선으로 보여주면 안 된다.

각 관계는 반드시 의미에 맞는 타입으로 분류해야 한다.

### 3.3 기본 보드는 Architecture View다

기본 보드는 AWS 전체 inventory 화면이 아니다.

기본 보드는 사용자가 구조를 이해하는 화면이다.

따라서 기본 보드는 아래를 우선한다.

```text
큰 공간
그 안의 주요 실행 리소스
중요한 트래픽 흐름
비용이나 위험에 직접 관련된 리소스
```

아래는 기본 보드에서 약하게 표시하거나 상세 패널로 보내야 한다.

```text
Route Table 상세
Route Table Association 상세
Security Group 상세 규칙
IAM Role
IAM Policy
CloudWatch Log Group
CloudWatch Metric Alarm
KMS Key
Lambda Permission
그 밖의 UNKNOWN 리소스
```

## 4. 데이터 모델 규약

### 4.1 DiscoveredResource

Reverse Engineering에서 AWS 리소스를 하나 발견하면 `DiscoveredResource`로 표현한다.

현재 shared type 기준 핵심 모양은 아래와 같다.

```ts
type DiscoveredResource = {
  id: string;
  provider: "aws";
  providerResourceType: string;
  providerResourceId: string;
  region: string;
  displayName: string;
  resourceType: ResourceType;
  config: Record<string, unknown>;
  relationships?: DiscoveredResourceRelationship[];
  analysisExcluded?: boolean;
};
```

필드 의미는 아래와 같다.

| 필드 | 의미 | 규칙 |
| --- | --- | --- |
| `id` | SketchCatch 내부 리소스 ID | API 응답 안에서 관계 연결에 사용한다 |
| `provider` | cloud provider | 현재는 `aws`만 사용한다 |
| `providerResourceType` | AWS 원본 리소스 타입 | 예: `AWS::EC2::Instance` |
| `providerResourceId` | AWS 원본 리소스 ID | 예: `i-1234`, `vpc-1234`, ARN |
| `region` | 읽어온 AWS region | 예: `ap-northeast-2` |
| `displayName` | 사용자에게 보여줄 이름 | 없으면 providerResourceId 기반으로 만든다 |
| `resourceType` | SketchCatch 공통 리소스 타입 | 예: `VPC`, `SUBNET`, `EC2` |
| `config` | 리소스 설정값 | 관계 추론과 상세 패널에 사용한다 |
| `relationships` | 다른 리소스와의 관계 | 이 문서의 관계 타입 규칙을 따른다 |
| `analysisExcluded` | 분석/보드 중심 표시 제외 여부 | `UNKNOWN` 등은 true가 될 수 있다 |

### 4.2 관계 타입

현재 내부 관계 타입은 세 개만 사용한다.

```ts
type DiscoveredResourceRelationshipType =
  | "contains"
  | "connects_to"
  | "depends_on";
```

관계 객체는 아래 모양을 따른다.

```ts
type DiscoveredResourceRelationship = {
  type: "contains" | "connects_to" | "depends_on";
  targetResourceId: string;
  label?: string;
};
```

각 필드 의미는 아래와 같다.

| 필드 | 의미 | 규칙 |
| --- | --- | --- |
| `type` | 내부 관계 타입 | 반드시 세 값 중 하나여야 한다 |
| `targetResourceId` | 관계 대상의 SketchCatch 내부 ID | providerResourceId가 아니라 내부 `id`를 넣는다 |
| `label` | 원본 관계 설명 | 예: `attached_to`, `uses`, `associated_with` |

### 4.3 provider 단계 관계와 내부 관계는 다르다

AWS Provider Adapter가 AWS를 읽는 단계에서는 아직 내부 `id`를 모를 수 있다.

그래서 provider 단계에서는 아래처럼 원본 ID를 사용한다.

```ts
type AwsDiscoveredRelationship = {
  type: "contains" | "depends_on" | "attached_to";
  targetProviderResourceId: string;
};
```

그 다음 내부 `DiscoveredResourceRelationship`으로 바꾼다.

변환 규칙은 아래와 같다.

| provider 단계 타입 | 내부 타입 | label |
| --- | --- | --- |
| `contains` | `contains` | `contains` |
| `depends_on` | `depends_on` | `depends_on` |
| `attached_to` | `connects_to` | `attached_to` |

중요한 규칙:

```text
Provider Adapter 밖으로 나간 뒤에는 targetProviderResourceId를 관계 기준으로 쓰면 안 된다.
API 응답과 프론트에서는 targetResourceId, 즉 내부 id를 기준으로 연결해야 한다.
```

## 5. 관계 방향 규약

### 5.1 relationships 배열의 의미

`relationships`는 현재 리소스가 다른 리소스를 참조한다는 뜻이다.

예:

```json
{
  "id": "resource-i-1234",
  "resourceType": "EC2",
  "relationships": [
    {
      "type": "contains",
      "targetResourceId": "resource-subnet-1234",
      "label": "contains"
    },
    {
      "type": "connects_to",
      "targetResourceId": "resource-sg-1234",
      "label": "attached_to"
    }
  ]
}
```

뜻은 이렇다.

```text
EC2는 subnet-1234 안에 있다.
EC2는 sg-1234에 연결되어 있다.
```

즉 `targetResourceId`는 현재 리소스가 바라보는 대상이다.

### 5.2 ResourceEdge 방향과 의미 방향은 같지 않을 수 있다

현재 ArchitectureJson edge 생성에서는 관계 대상이 `sourceId`, 현재 리소스가 `targetId`가 된다.

예:

```text
relationships 안에서는:
EC2 → Subnet

ArchitectureJson edge에서는:
Subnet → EC2
```

이것은 보드에서 보기 좋게 그리기 위한 방향이다.

따라서 구현자는 아래 규칙을 지켜야 한다.

```text
ResourceEdge의 sourceId/targetId만 보고 실제 트래픽 방향이라고 해석하면 안 된다.
트래픽 방향은 별도 관계 타입이나 label로 판단해야 한다.
```

## 6. 관계 타입별 명세

### 6.1 contains

#### 의미

`contains`는 큰 공간 안에 작은 리소스가 들어간다는 뜻이다.

쉽게 말하면 박스 안에 배치해야 하는 관계다.

#### 반드시 지켜야 할 규칙

```text
contains는 보드 배치에 강하게 사용한다.
contains의 targetResourceId는 부모 또는 상위 공간이어야 한다.
contains는 트래픽 방향을 뜻하지 않는다.
contains는 생성 순서만 뜻하지 않는다.
```

#### 올바른 예

| 현재 리소스 | targetResourceId | 의미 |
| --- | --- | --- |
| Subnet | VPC | Subnet은 VPC 안에 있다 |
| EC2 | Subnet | EC2는 Subnet 안에 있다 |
| RDS | Subnet 또는 DB Subnet Group | RDS는 해당 네트워크 영역에 속한다 |
| Route Table | VPC | Route Table은 VPC에 속한다 |

#### 잘못된 예

| 잘못된 관계 | 이유 |
| --- | --- |
| EC2 contains Security Group | Security Group은 EC2 안에 있는 것이 아니다 |
| S3 contains VPC | S3는 VPC 안에 있는 리소스가 아니다 |
| IAM Role contains EC2 | IAM Role은 공간 리소스가 아니다 |

#### 파라미터에서 찾는 방법

| 파라미터 | 관계 후보 |
| --- | --- |
| `vpcId` | 현재 리소스가 해당 VPC에 속할 수 있다 |
| `subnetId` | 현재 리소스가 해당 Subnet에 속할 수 있다 |
| `subnetIds` | 현재 리소스가 여러 Subnet 영역에 걸칠 수 있다 |
| `dbSubnetGroupName` | RDS가 DB Subnet Group 영역에 속할 수 있다 |

#### 보드 표시 규칙

`contains`는 기본 보드에서 가장 중요하다.

보드에서는 아래처럼 표시한다.

```text
VPC는 큰 박스
Subnet은 VPC 안의 작은 박스
EC2/RDS는 Subnet 또는 네트워크 영역 안의 리소스
```

단, RDS가 여러 Subnet을 참조할 때는 같은 RDS를 여러 번 배치하면 안 된다.

규칙:

```text
하나의 providerResourceId는 보드에 한 번만 배치한다.
여러 Subnet에 걸친 리소스는 대표 위치를 하나만 선택한다.
나머지 Subnet 관계는 상세 패널에 표시한다.
```

### 6.2 connects_to

#### 의미

`connects_to`는 현재 리소스가 다른 리소스와 연결되어 있거나, 다른 리소스를 사용한다는 뜻이다.

공간 안에 들어간다는 뜻이 아니다.

#### 반드시 지켜야 할 규칙

```text
connects_to는 포함 관계로 쓰면 안 된다.
connects_to는 기본 보드에서 반드시 모두 그릴 필요가 없다.
connects_to는 label로 원본 의미를 남겨야 한다.
```

#### 올바른 예

| 현재 리소스 | 대상 리소스 | label | 의미 |
| --- | --- | --- | --- |
| Internet Gateway | VPC | `attached_to` | Internet Gateway가 VPC에 붙어 있다 |
| EC2 | Security Group | `attached_to` | EC2가 Security Group을 사용한다 |
| RDS | Security Group | `attached_to` | RDS가 Security Group을 사용한다 |
| Route Table Association | Subnet | `associated_with` | Subnet과 Route Table 연결 정보다 |
| ALB | Target Group | `routes_to` | ALB가 Target Group으로 요청을 보낸다 |

#### 잘못된 예

| 잘못된 관계 | 이유 |
| --- | --- |
| EC2 connects_to Subnet만 사용 | Subnet 소속은 contains로 표현해야 한다 |
| Security Group contains EC2 | Security Group은 보호 규칙이지 부모 박스가 아니다 |
| S3 contains EC2 | S3와 EC2는 공간 포함 관계가 아니다 |

#### 파라미터에서 찾는 방법

| 파라미터 | 관계 후보 |
| --- | --- |
| `securityGroupIds` | Security Group 연결 |
| `vpcSecurityGroupIds` | Security Group 연결 |
| `routeTableId` | Route Table 연결 |
| `internetGatewayId` | Internet Gateway 연결 또는 의존 |
| `targetGroupArn` | ALB/Target Group 연결 |
| `roleArn` | IAM Role 연결 |

#### 보드 표시 규칙

기본 보드에서 모든 `connects_to`를 선으로 그리면 안 된다.

기본 보드에 보여도 되는 연결:

```text
ALB → EC2
EC2 → RDS
EC2 → S3
API Gateway → Lambda
Lambda → RDS 또는 S3
```

기본 보드에서 숨기거나 약하게 보여야 하는 연결:

```text
EC2 → Security Group
RDS → Security Group
Subnet → Route Table
Route Table → Internet Gateway
IAM Role 연결
CloudWatch Log Group 연결
```

이 정보는 버리면 안 된다.

오른쪽 상세 패널, Inventory View, Security View, Network View에서 보여줘야 한다.

### 6.3 depends_on

#### 의미

`depends_on`은 현재 리소스가 다른 리소스를 먼저 필요로 한다는 뜻이다.

이 관계는 화면 구조보다 생성 순서, import 순서, 분석 근거에 가깝다.

#### 반드시 지켜야 할 규칙

```text
depends_on은 기본 보드에서 강한 선으로 표시하지 않는다.
depends_on은 contains 대신 쓰면 안 된다.
depends_on은 connects_to 대신 남발하면 안 된다.
```

#### 올바른 예

| 현재 리소스 | 대상 리소스 | 의미 |
| --- | --- | --- |
| Route Table | Internet Gateway | route가 gateway를 참조한다 |
| Lambda Permission | Lambda | permission이 Lambda function에 묶인다 |
| IAM Instance Profile | IAM Role | instance profile이 role을 참조한다 |
| CloudWatch Alarm | 감시 대상 리소스 | alarm이 대상 metric을 참조한다 |

#### 파라미터에서 찾는 방법

| 파라미터 | 관계 후보 |
| --- | --- |
| `roleArn` | IAM Role 의존 또는 연결 |
| `functionArn` | Lambda 의존 |
| `kmsKeyId` | KMS Key 의존 |
| `alarmActions` | Action 대상 의존 |
| `route.gatewayId` | Internet Gateway 또는 NAT Gateway 의존 |

#### 보드 표시 규칙

기본 보드에서는 숨긴다.

아래 기능에서 사용한다.

```text
Terraform 생성 순서
Terraform import 제안
배포 전 점검
오류 설명
상세 패널
```

## 7. 관계 분류 우선순위

같은 파라미터에서 여러 관계 후보가 나올 수 있다.

그럴 때는 아래 순서를 따른다.

```text
1. 공간 소속이면 contains
2. 실제로 붙어 있거나 사용하는 관계면 connects_to
3. 생성 순서나 참조만 의미하면 depends_on
4. 잘 모르겠으면 관계를 만들지 말고 config에만 남긴다
5. 그래도 사용자에게 보여야 하면 UNKNOWN 또는 manual review로 남긴다
```

예:

```text
EC2.subnetId
→ EC2가 Subnet 안에 있으므로 contains

EC2.securityGroupIds
→ EC2가 Security Group을 사용하므로 connects_to

RouteTable.route.gatewayId
→ Route Table이 Gateway를 참조하므로 depends_on 또는 connects_to
→ 기본 보드에서는 숨김
```

## 8. 리소스별 관계 규약

### 8.1 Region

현재 ResourceType에는 `REGION`이 없다.

따라서 Region은 보드 노드가 아니라 scan context로 다룬다.

규칙:

```text
region은 DiscoveredResource.region에 저장한다.
region을 별도 보드 노드로 만들지 않는다.
여러 region scan을 지원할 때는 region group 또는 filter로 확장한다.
```

### 8.2 VPC

VPC는 네트워크의 가장 큰 보드 박스다.

규칙:

```text
VPC는 area node처럼 다룬다.
Subnet, Route Table, Security Group, Internet Gateway는 VPC와 관계를 가질 수 있다.
VPC 자체를 다른 VPC 안에 넣지 않는다.
```

대표 관계:

| 현재 리소스 | 관계 | 대상 |
| --- | --- | --- |
| Subnet | contains | VPC |
| Route Table | contains | VPC |
| Security Group | contains 또는 depends_on | VPC |
| Internet Gateway | connects_to | VPC |

Security Group은 VPC 안에 속하지만 화면에서는 별도 보안 경계처럼 보일 수 있다.

따라서 Security Group은 다음 규칙을 따른다.

```text
VPC 소속 정보는 유지한다.
보드에서는 Security Group이 보호하는 EC2/RDS 근처에 배치할 수 있다.
Security Group을 VPC 아래 일반 리소스처럼 무조건 나열하지 않는다.
```

### 8.3 Subnet

Subnet은 VPC 안의 작은 네트워크 영역이다.

규칙:

```text
Subnet은 반드시 VPC 안에 배치한다.
public/private 여부, AZ, CIDR이 있으면 label에 반영한다.
Subnet 안에는 EC2, NAT Gateway 같은 실행/네트워크 리소스를 배치할 수 있다.
```

대표 관계:

| 현재 리소스 | 관계 | 대상 |
| --- | --- | --- |
| EC2 | contains | Subnet |
| NAT Gateway | contains | Subnet |
| RDS | contains | Subnet 또는 DB Subnet Group |
| Route Table Association | connects_to | Subnet |

주의:

```text
Subnet이 여러 개 반복해서 보이는 것은 AWS에서는 정상일 수 있다.
하지만 보드에서는 public/private, AZ, CIDR 없이 똑같이 보이면 안 된다.
```

### 8.4 Internet Gateway

Internet Gateway는 VPC 바깥 인터넷과 VPC를 연결한다.

규칙:

```text
Internet Gateway는 VPC 안에 들어간다기보다 VPC에 붙어 있는 리소스다.
관계 타입은 connects_to를 우선한다.
label은 attached_to를 유지할 수 있다.
```

대표 관계:

| 현재 리소스 | 관계 | 대상 |
| --- | --- | --- |
| Internet Gateway | connects_to | VPC |
| Route Table | depends_on | Internet Gateway |

### 8.5 Route Table

Route Table은 Subnet의 트래픽 경로를 정하는 리소스다.

규칙:

```text
Route Table은 기본 보드에서 크게 보여주지 않는다.
Route Table은 Network Detail View에서 자세히 보여준다.
Route Table이 VPC에 속한다는 정보는 유지한다.
Subnet과의 association 정보도 유지한다.
```

대표 관계:

| 현재 리소스 | 관계 | 대상 |
| --- | --- | --- |
| Route Table | contains | VPC |
| Route Table | depends_on | Internet Gateway 또는 NAT Gateway |
| Route Table Association | connects_to | Subnet |
| Route Table Association | connects_to | Route Table |

### 8.6 Route Table Association

Route Table Association은 독립 아키텍처 리소스라기보다 연결 정보에 가깝다.

규칙:

```text
기본 보드에서는 노드로 크게 보여주지 않는다.
Subnet 상세 패널이나 Network Detail View에서 보여준다.
보드에 노드로 표시하더라도 약한 보조 리소스로 표시한다.
```

### 8.7 Security Group

Security Group은 네트워크 보안 규칙이다.

규칙:

```text
Security Group은 EC2/RDS 안에 들어가는 리소스가 아니다.
Security Group은 EC2/RDS를 보호하거나 그 리소스가 사용하는 보안 경계다.
보드에서는 Security Group을 보호 대상 근처에 배치할 수 있다.
```

대표 관계:

| 현재 리소스 | 관계 | 대상 |
| --- | --- | --- |
| Security Group | contains 또는 depends_on | VPC |
| EC2 | connects_to | Security Group |
| RDS | connects_to | Security Group |
| Lambda | connects_to | Security Group |

Security Group rule은 기본 보드 선으로 모두 표시하지 않는다.

아래처럼 분석 finding이나 상세 패널에 표시한다.

```text
SSH 22번이 0.0.0.0/0으로 열려 있음
RDS 5432 포트가 넓게 열려 있음
egress가 전체 허용됨
```

### 8.8 EC2

EC2는 실행 리소스다.

규칙:

```text
EC2는 subnetId가 있으면 해당 Subnet 안에 배치한다.
securityGroupIds는 connects_to로 표현한다.
instanceType, imageId, publicIp 등은 config로 유지한다.
```

대표 관계:

| 현재 리소스 | 관계 | 대상 |
| --- | --- | --- |
| EC2 | contains | Subnet |
| EC2 | connects_to | Security Group |
| EC2 | connects_to 또는 depends_on | IAM Instance Profile |
| EC2 | connects_to | EBS Volume |

### 8.9 RDS

RDS는 database 리소스다.

규칙:

```text
RDS는 Subnet 하나에 딱 들어간다고 단정하지 않는다.
DB Subnet Group 또는 여러 subnetIds를 기준으로 네트워크 영역을 판단한다.
보드에는 하나의 RDS 노드만 배치한다.
여러 Subnet 관계는 상세 패널에 표시한다.
```

대표 관계:

| 현재 리소스 | 관계 | 대상 |
| --- | --- | --- |
| RDS | contains | 대표 Subnet 또는 DB Subnet Group |
| RDS | connects_to | Security Group |
| RDS | depends_on | KMS Key |

주의:

```text
RDS가 subnetIds 여러 개를 갖는다고 보드에 RDS를 여러 개 만들면 안 된다.
```

### 8.10 S3

S3 Bucket은 region 리소스이지만 VPC 안에 있는 리소스가 아니다.

규칙:

```text
S3는 VPC 안에 넣지 않는다.
S3는 독립 리소스로 배치한다.
EC2/Lambda/CloudFront 등이 S3를 사용하면 connects_to로 표현할 수 있다.
```

대표 관계:

| 현재 리소스 | 관계 | 대상 |
| --- | --- | --- |
| CloudFront | connects_to | S3 |
| Lambda | connects_to | S3 |
| EC2 | connects_to | S3 |
| IAM Policy | depends_on | S3 |

### 8.11 CloudFront

CloudFront는 CDN 리소스다.

규칙:

```text
CloudFront는 VPC 안에 넣지 않는다.
CloudFront는 public entry 또는 edge 리소스로 보여준다.
origin이 S3/ALB/API Gateway면 connects_to로 표현한다.
```

### 8.12 Lambda

Lambda는 실행 리소스다.

규칙:

```text
VPC config가 있으면 관련 Subnet/Security Group 관계를 만든다.
VPC config가 없으면 VPC 밖 독립 실행 리소스로 배치한다.
roleArn은 권한 관계이므로 기본 보드에서는 약하게 다룬다.
```

대표 관계:

| 현재 리소스 | 관계 | 대상 |
| --- | --- | --- |
| Lambda | contains | Subnet, VPC config가 있을 때 |
| Lambda | connects_to | Security Group |
| Lambda | depends_on | IAM Role |
| Lambda Permission | depends_on | Lambda |

### 8.13 IAM Role, IAM Policy, IAM Instance Profile

IAM 계열은 권한 리소스다.

규칙:

```text
기본 아키텍처 보드에서 큰 노드로 보여주지 않는다.
Security View 또는 Permission View에서 보여준다.
EC2/Lambda와의 관계는 기본 보드에서는 숨기거나 약하게 표시한다.
```

대표 관계:

| 현재 리소스 | 관계 | 대상 |
| --- | --- | --- |
| EC2 | connects_to 또는 depends_on | IAM Instance Profile |
| IAM Instance Profile | depends_on | IAM Role |
| Lambda | depends_on | IAM Role |
| IAM Policy | depends_on | IAM Role 또는 대상 리소스 |

### 8.14 KMS Key

KMS Key는 암호화 리소스다.

규칙:

```text
기본 보드에서는 숨긴다.
보안 상세와 비용/위험 분석에서 보여준다.
RDS, S3, CloudWatch Logs 등이 참조하면 depends_on으로 남긴다.
```

### 8.15 CloudWatch Log Group, Metric Alarm

CloudWatch 계열은 운영 리소스다.

규칙:

```text
기본 보드에서는 숨긴다.
Operation View에서 보여준다.
Lambda/EC2/RDS와 연결될 수 있지만 기본 화면에 모든 선을 그리지 않는다.
```

### 8.16 UNKNOWN

UNKNOWN은 AWS에서 발견했지만 SketchCatch가 아직 정식 타입으로 해석하지 못한 리소스다.

규칙:

```text
UNKNOWN은 기본 보드 중앙에 자동 배치하지 않는다.
UNKNOWN은 발견 결과와 확인 목록에는 남긴다.
UNKNOWN도 providerResourceId, providerResourceType, config는 최대한 유지한다.
UNKNOWN 관계는 확실할 때만 만든다.
```

## 9. 파라미터 사용 규약

파라미터는 리소스의 설정값이다.

파라미터는 아래 세 가지 역할을 한다.

```text
1. 화면 상세 패널에 보여줄 값
2. 리소스 관계를 추론하는 힌트
3. Terraform 생성/import/분석에 필요한 값
```

모든 파라미터가 관계는 아니다.

예:

| 파라미터 | 의미 | 관계 여부 |
| --- | --- | --- |
| `instanceType` | EC2 크기 | 관계 아님 |
| `cidrBlock` | 네트워크 주소 범위 | 관계 아님 |
| `vpcId` | VPC 소속 | contains 후보 |
| `subnetId` | Subnet 소속 | contains 후보 |
| `securityGroupIds` | Security Group 사용 | connects_to 후보 |
| `roleArn` | IAM Role 사용 | depends_on 또는 connects_to 후보 |
| `kmsKeyId` | KMS Key 사용 | depends_on 후보 |

규칙:

```text
파라미터 안에 다른 리소스 ID가 있어도 무조건 관계로 만들면 안 된다.
그 ID가 공간 소속인지, 연결인지, 의존인지 먼저 분류한다.
분류할 수 없으면 관계를 만들지 않고 config에만 남긴다.
```

## 10. nested config 관계 탐색 규약

AWS 응답은 ID가 중첩 객체 안에 들어오는 경우가 많다.

예:

```json
{
  "vpcConfig": {
    "subnetIds": ["subnet-1234"],
    "securityGroupIds": ["sg-1234"]
  }
}
```

규칙:

```text
관계 탐색은 string, array, object 안쪽을 모두 확인해야 한다.
단, 관계 타입은 key 이름과 리소스 타입을 함께 보고 결정한다.
```

예:

```text
vpcConfig.subnetIds
→ contains 후보

vpcConfig.securityGroupIds
→ connects_to 후보
```

## 11. 보드 표시 등급

모든 관계를 같은 강도로 보여주면 안 된다.

관계는 보드 표시 등급을 가진다.

| 등급 | 의미 | 예시 |
| --- | --- | --- |
| primary | 기본 화면에 강하게 표시 | VPC contains Subnet, Subnet contains EC2 |
| secondary | 기본 화면에 약하게 표시하거나 토글로 표시 | EC2 connects_to Security Group |
| detail | 기본 화면에서는 숨기고 상세 패널 표시 | Route Table Association, IAM Role |
| hidden | 보드에는 숨기고 분석/로그에서만 사용 | depends_on 대부분 |

기본 규칙:

```text
contains는 primary
중요 트래픽 connects_to는 primary 또는 secondary
Security Group connects_to는 secondary 또는 detail
Route Table 관련 connects_to는 detail
depends_on은 hidden 또는 detail
UNKNOWN 관련 관계는 detail
```

## 12. 컨테이너 리소스 규약

컨테이너 리소스는 다른 리소스를 안에 담는 큰 박스 리소스다.

보드에서 컨테이너는 단순 아이콘이 아니라 영역으로 보여야 한다.

### 12.1 컨테이너 리소스 기준

아래 조건 중 하나 이상을 만족하면 컨테이너 리소스 후보가 된다.

```text
다른 리소스를 contains 관계로 가질 수 있다.
화면에서 자식 리소스를 안에 담아야 구조가 이해된다.
크기가 자식 리소스 위치에 따라 달라질 수 있다.
```

현재 제품에서 컨테이너로 다뤄야 하는 리소스는 아래다.

| 리소스 | 컨테이너 여부 | 이유 |
| --- | --- | --- |
| VPC | 예 | Subnet, Route Table, Security Group이 VPC에 속한다 |
| Subnet | 예 | EC2, RDS, NAT Gateway 같은 리소스가 Subnet 영역에 놓인다 |
| Security Group | 조건부 | 보안 경계로 표현할 때 보호 대상 리소스를 감쌀 수 있다 |
| Region | 아직 아님 | 현재 ResourceType에 없고 scan context로만 사용한다 |
| DB Subnet Group | 고도화 후보 | RDS가 여러 Subnet에 걸칠 때 필요하다 |

현재 구현에서 반드시 컨테이너로 취급할 리소스:

```text
VPC
Subnet
```

고도화에서 컨테이너로 추가 검토할 리소스:

```text
Security Group
DB Subnet Group
Availability Zone
Region
EKS Cluster
Target Group
```

### 12.2 자식 리소스 기준

자식 리소스는 컨테이너 안에 들어가야 하는 리소스다.

자식 여부는 반드시 `contains` 관계를 우선 기준으로 판단한다.

기본 규칙:

```text
contains 관계가 있으면 자식 리소스다.
config의 vpcId, subnetId, subnetIds는 contains 후보를 찾는 힌트다.
connects_to 관계만 있다고 자식 리소스로 보면 안 된다.
depends_on 관계만 있다고 자식 리소스로 보면 안 된다.
```

예:

| 관계 | 자식으로 볼까 | 이유 |
| --- | --- | --- |
| Subnet contains VPC | 예 | Subnet은 VPC 안에 배치한다 |
| EC2 contains Subnet | 예 | EC2는 Subnet 안에 배치한다 |
| EC2 connects_to Security Group | 아니오 | Security Group을 사용하는 것이지 안에 들어가는 것이 아니다 |
| Route Table depends_on Internet Gateway | 아니오 | 생성/참조 관계이지 포함 관계가 아니다 |
| RDS contains subnet-a, subnet-b | 조건부 | RDS 노드는 하나만 배치하고 여러 Subnet 관계는 상세에 남긴다 |

### 12.3 화면 배치에 영향을 주는 관계

관계 타입별 화면 배치 영향은 아래처럼 고정한다.

| 관계 타입 | 컨테이너 배치 영향 | 선 표시 영향 | 기본 화면 표시 |
| --- | --- | --- | --- |
| `contains` | 강하게 영향 있음 | 포함 관계 선은 숨길 수 있음 | 컨테이너 안에 배치 |
| `connects_to` | 기본적으로 영향 없음 | 필요하면 선으로 표시 | 중요 관계만 표시 |
| `depends_on` | 영향 없음 | 기본 화면에서는 숨김 | 상세/분석에서 사용 |

중요한 규칙:

```text
자동 배치와 자동 크기 조절은 contains를 기준으로 동작한다.
connects_to는 선 표시 기준이지 컨테이너 안쪽 배치 기준이 아니다.
depends_on은 생성 순서와 분석 기준이지 화면 배치 기준이 아니다.
```

즉:

```text
EC2 contains Subnet
→ EC2는 Subnet 안에 배치한다.

EC2 connects_to Security Group
→ EC2와 Security Group은 선이나 상세 정보로 연결한다.
→ EC2를 Security Group 안에 넣지는 않는다.
```

### 12.4 컨테이너 중첩 기준

컨테이너는 중첩될 수 있다.

기본 중첩 구조는 아래를 따른다.

```text
VPC
└─ Subnet
   ├─ EC2
   └─ RDS
```

중첩 규칙:

```text
부모 컨테이너는 자식 컨테이너를 감싸야 한다.
자식 컨테이너는 자신의 자식 리소스를 감싸야 한다.
부모 크기 계산은 자식 컨테이너의 최종 크기까지 반영해야 한다.
같은 리소스가 여러 컨테이너 안에 동시에 들어가면 안 된다.
```

예:

```text
VPC 안에 Subnet이 있다.
Subnet 안에 EC2가 있다.
EC2 위치가 바뀌어 Subnet이 커지면,
VPC도 그 커진 Subnet을 감쌀 수 있어야 한다.
```

## 13. 컨테이너 자동 배치 규약

컨테이너 자동 배치는 `contains` 관계를 보고 자식 리소스 위치를 정리하는 기능이다.

목표는 아래다.

```text
자식 리소스가 컨테이너 안에 들어가 보이게 한다.
자식 리소스끼리 겹치지 않게 한다.
컨테이너 중첩 구조를 사람이 이해할 수 있게 한다.
사용자가 직접 옮긴 위치를 함부로 덮어쓰지 않는다.
```

### 13.1 자동 배치가 실행되는 시점

자동 배치는 아래 상황에서 실행할 수 있다.

```text
Reverse Engineering scan 결과를 처음 보드 후보로 만들 때
AI Architecture Draft를 보드 후보로 만들 때
사용자가 리소스를 컨테이너 안으로 드래그해서 넣을 때
사용자가 contains 관계를 새로 만들 때
사용자가 자동 정리 버튼을 눌렀을 때
```

자동 배치를 실행하면 안 되는 상황:

```text
사용자가 수동으로 옮긴 리소스를 저장한 직후
보드 불러오기만 하는 중
선택된 리소스를 사용자가 드래그 중인 상태
사용자가 자동 배치 잠금을 꺼둔 컨테이너
```

### 13.2 자동 배치 입력값

자동 배치는 최소한 아래 값을 입력으로 받아야 한다.

```text
전체 노드 목록
전체 관계 목록
현재 viewport
사용자가 직접 옮겼는지 여부
잠금 여부
컨테이너 최소 크기
컨테이너 안쪽 여백
자식 리소스 간 간격
```

현재 타입에 전용 필드가 없다면 고도화에서 아래 metadata를 추가하는 것을 권장한다.

```ts
type LayoutMetadata = {
  layoutMode: "auto" | "manual";
  layoutLocked: boolean;
  lastAutoLayoutAt?: string;
  lastManualMoveAt?: string;
};
```

단, metadata가 아직 없더라도 구현 규칙은 아래 기준을 따른다.

```text
처음 생성된 후보는 auto로 본다.
사용자가 드래그로 옮긴 노드는 manual로 본다.
잠긴 노드는 자동 배치 대상에서 제외한다.
```

### 13.3 자동 배치 순서

자동 배치는 아래 순서로 한다.

```text
1. contains 관계만 모은다.
2. 컨테이너와 자식 목록을 만든다.
3. 가장 안쪽 컨테이너부터 자식 위치를 정한다.
4. 자식 리소스가 겹치지 않게 격자 또는 줄 배치로 놓는다.
5. 자식 위치를 기준으로 컨테이너 크기를 계산한다.
6. 부모 컨테이너가 자식 컨테이너를 감쌀 수 있게 다시 계산한다.
7. connects_to 선은 위치 계산 뒤에 다시 연결한다.
```

중요한 규칙:

```text
contains가 없는 리소스는 컨테이너 안에 강제로 넣지 않는다.
connects_to만 있는 리소스는 자동 배치 기준에서 제외한다.
depends_on만 있는 리소스는 자동 배치 기준에서 제외한다.
```

### 13.4 자식 리소스 배치 규칙

컨테이너 안 자식은 겹치면 안 된다.

기본 배치 규칙:

```text
왼쪽 위에서 오른쪽으로 배치한다.
한 줄이 가득 차면 다음 줄로 내린다.
자식 사이에는 최소 간격을 둔다.
컨테이너 테두리와 자식 사이에는 여백을 둔다.
자식 리소스 크기가 다르면 가장 큰 높이를 기준으로 줄 높이를 정한다.
```

권장 기본값:

| 값 | 권장값 | 설명 |
| --- | --- | --- |
| 컨테이너 안쪽 여백 | 48px 이상 | 테두리와 자식 사이 간격 |
| 자식 가로 간격 | 32px 이상 | 자식끼리 좌우 간격 |
| 자식 세로 간격 | 32px 이상 | 자식끼리 상하 간격 |
| 최소 자식 영역 | 112px x 112px | 작은 아이콘도 겹치지 않게 보는 기준 |
| 최대 한 줄 자식 수 | 3개부터 시작 | 너무 길게 한 줄로 늘어나는 것 방지 |

예:

```text
Subnet 안에 EC2 3개, RDS 1개가 있다.

Subnet
┌─────────────────────────┐
│ EC2     EC2     EC2     │
│ RDS                     │
└─────────────────────────┘
```

### 13.5 사용자가 직접 옮긴 위치 처리 규칙

사용자가 직접 옮긴 위치는 기본적으로 유지한다.

자동 배치가 사용자 위치를 덮어쓸 수 있는 경우는 아래뿐이다.

| 상황 | 덮어쓸 수 있나 |
| --- | --- |
| Reverse Engineering 결과를 처음 미리보기로 생성 | 예 |
| AI Draft를 처음 보드 후보로 생성 | 예 |
| 사용자가 자동 정리 버튼을 직접 클릭 | 예 |
| 사용자가 컨테이너 안에 새 리소스를 추가 | 새 리소스만 예 |
| 사용자가 이미 옮긴 기존 리소스 | 아니오 |
| 리소스가 컨테이너 밖으로 완전히 벗어나 저장 불가능 | 확인 후 예 |
| 컨테이너 크기 변경 때문에 위치 보정이 꼭 필요 | 최소 보정만 예 |

반드시 지켜야 할 규칙:

```text
사용자가 직접 옮긴 기존 리소스는 자동 배치가 함부로 움직이면 안 된다.
자동 배치가 기존 위치를 바꿔야 하면 사용자 액션이 있어야 한다.
자동 정리 버튼을 누른 경우에만 전체 재배치가 가능하다.
```

추천 동작:

```text
새 리소스만 빈 공간에 자동 배치한다.
기존 manual 리소스는 고정한다.
auto 리소스끼리만 다시 정리한다.
겹침이 생기면 manual 리소스는 움직이지 말고 auto 리소스를 피해서 배치한다.
```

### 13.6 자동 배치 예외 상황

자동 배치가 완벽하게 처리하기 어려운 상황은 아래처럼 처리한다.

| 상황 | 처리 기준 |
| --- | --- |
| 자식이 부모보다 너무 많음 | 컨테이너를 키우고, 그래도 부족하면 스크롤/접기 후보 |
| 자식이 여러 부모를 참조함 | 대표 부모 하나만 선택하고 나머지는 상세 표시 |
| 부모가 없음 | 독립 리소스 영역에 배치 |
| 부모가 UNKNOWN임 | 기본 보드에는 강제 배치하지 않음 |
| contains 순환이 생김 | 관계를 무시하고 manual review finding 생성 |
| 사용자가 잠근 노드와 겹침 | 잠긴 노드는 유지하고 다른 노드를 피해서 배치 |
| 여러 region 리소스가 섞임 | region별 그룹 고도화 전에는 독립 영역으로 분리 |

### 13.7 contains 순환 금지

contains 관계는 순환되면 안 된다.

잘못된 예:

```text
VPC contains Subnet
Subnet contains EC2
EC2 contains VPC
```

이런 경우 자동 배치는 실행하면 안 된다.

처리 기준:

```text
순환 관계는 무시한다.
해당 리소스는 manual review 대상으로 표시한다.
사용자에게 어떤 관계가 이상한지 보여준다.
```

## 14. 컨테이너 자동 크기 조절 규약

컨테이너 자동 크기 조절은 자식 리소스 위치와 개수에 맞춰 컨테이너 크기를 바꾸는 기능이다.

목표는 아래다.

```text
컨테이너가 모든 자식 리소스를 감싸야 한다.
자식이 늘어나면 컨테이너가 커져야 한다.
자식이 줄어들면 컨테이너가 줄어들 수 있다.
하지만 너무 작아지면 안 된다.
중첩 컨테이너는 안쪽부터 계산해야 한다.
```

### 14.1 자동 크기 조절이 실행되는 시점

자동 크기 조절은 아래 상황에서 실행한다.

```text
컨테이너 안에 자식 리소스가 추가될 때
컨테이너 안의 자식 리소스가 삭제될 때
자식 리소스 위치가 바뀔 때
자식 컨테이너 크기가 바뀔 때
contains 관계가 추가 또는 삭제될 때
자동 정리 버튼을 눌렀을 때
```

실행하면 안 되는 상황:

```text
사용자가 컨테이너 크기를 잠근 경우
사용자가 크기 조절 중인 경우
보드 불러오기 중 단순 렌더링만 하는 경우
```

### 14.2 컨테이너 크기 계산 기준

컨테이너 크기는 자식 리소스의 바깥 사각형을 기준으로 계산한다.

계산 순서:

```text
1. 컨테이너 안의 자식 리소스를 모두 찾는다.
2. 자식 리소스들의 가장 왼쪽, 위쪽, 오른쪽, 아래쪽 좌표를 구한다.
3. 여백을 더한다.
4. 최소 크기보다 작으면 최소 크기를 사용한다.
5. 부모 컨테이너가 있으면 부모도 다시 계산한다.
```

공식:

```text
필요한 width =
  자식들의 오른쪽 끝 - 자식들의 왼쪽 끝 + 왼쪽 여백 + 오른쪽 여백

필요한 height =
  자식들의 아래쪽 끝 - 자식들의 위쪽 끝 + 위 여백 + 아래 여백
```

### 14.3 최소 크기 기준

컨테이너는 자식이 적어도 너무 작아지면 안 된다.

권장 최소 크기:

| 컨테이너 | 최소 크기 |
| --- | --- |
| VPC | 720px x 520px |
| Subnet | 360px x 260px |
| Security Group 경계 | 260px x 180px |
| DB Subnet Group | 480px x 260px |

현재 구현에서 최소 필수:

```text
VPC 최소 크기
Subnet 최소 크기
```

나머지는 고도화에서 추가한다.

### 14.4 여백 기준

컨테이너 여백은 자식이 테두리에 붙지 않게 하기 위한 값이다.

권장값:

| 여백 | 권장값 |
| --- | --- |
| 왼쪽 | 48px 이상 |
| 오른쪽 | 48px 이상 |
| 위쪽 | 64px 이상 |
| 아래쪽 | 48px 이상 |

위쪽 여백을 더 크게 두는 이유:

```text
컨테이너 label, badge, public/private 표시가 들어갈 수 있기 때문이다.
```

### 14.5 중첩 컨테이너 크기 계산

중첩 컨테이너는 안쪽부터 계산한다.

예:

```text
VPC
└─ Subnet
   ├─ EC2
   └─ RDS
```

계산 순서:

```text
1. EC2/RDS 위치를 기준으로 Subnet 크기를 계산한다.
2. 계산된 Subnet 크기를 기준으로 VPC 크기를 계산한다.
3. VPC 크기가 커지면 전체 보드 위치도 필요하면 보정한다.
```

중요한 규칙:

```text
부모 컨테이너는 자식 컨테이너의 최종 크기를 기준으로 계산해야 한다.
자식 컨테이너가 커졌는데 부모 컨테이너가 그대로면 안 된다.
```

### 14.6 크기 줄이기 규칙

자식이 삭제되거나 안쪽으로 이동하면 컨테이너를 줄일 수 있다.

하지만 아래 조건을 모두 만족해야 한다.

```text
사용자가 컨테이너 크기를 수동으로 키운 적이 없다.
컨테이너가 layoutLocked 상태가 아니다.
줄여도 모든 자식을 감싼다.
줄여도 최소 크기보다 작아지지 않는다.
줄였을 때 주변 컨테이너나 노드와 심하게 겹치지 않는다.
```

사용자가 직접 키운 컨테이너는 자동으로 줄이지 않는다.

추천 동작:

```text
자동 생성된 컨테이너는 자동으로 줄일 수 있다.
사용자가 수동 조절한 컨테이너는 유지한다.
사용자가 자동 맞춤 버튼을 누르면 다시 계산할 수 있다.
```

### 14.7 크기 조절 예시

예시 1: VPC 안에 Subnet 두 개가 있는 경우

```text
VPC
┌────────────────────────────────────┐
│ Public Subnet       Private Subnet │
│ ┌────────────┐      ┌────────────┐ │
│ │ EC2        │      │ RDS        │ │
│ └────────────┘      └────────────┘ │
└────────────────────────────────────┘
```

규칙:

```text
VPC는 두 Subnet을 모두 감싸야 한다.
각 Subnet은 자기 안의 EC2/RDS를 감싸야 한다.
```

예시 2: Subnet 안에 EC2가 추가되는 경우

```text
변경 전:
Subnet
┌────────────┐
│ EC2        │
└────────────┘

변경 후:
Subnet
┌──────────────────────┐
│ EC2        EC2       │
└──────────────────────┘
```

규칙:

```text
새 EC2는 기존 EC2와 겹치지 않게 배치한다.
Subnet은 두 EC2를 감싸도록 커진다.
Subnet이 커지면 VPC도 필요하면 커진다.
```

예시 3: RDS가 여러 Subnet을 참조하는 경우

```text
RDS subnetIds = [subnet-a, subnet-b]
```

규칙:

```text
RDS 노드는 하나만 만든다.
대표 Subnet 하나에만 배치한다.
다른 Subnet 관계는 상세 패널에 표시한다.
DB Subnet Group 정식 지원 전까지는 RDS를 여러 개 복제하지 않는다.
```

### 14.8 자동 크기 조절 예외 상황

| 상황 | 처리 기준 |
| --- | --- |
| 자식이 컨테이너 밖에 너무 멀리 있음 | 자동 정리 전에는 컨테이너를 과도하게 키우지 않고 경고 |
| 자식이 manual 위치임 | 위치는 유지하되 컨테이너는 감싸도록 키울 수 있음 |
| 컨테이너가 layoutLocked임 | 자동 크기 조절하지 않음 |
| 중첩 컨테이너가 겹침 | 안쪽 컨테이너부터 다시 배치하거나 manual review |
| 자식 크기를 알 수 없음 | 최소 자식 영역 112px x 112px으로 계산 |
| UNKNOWN 자식 | 기본 컨테이너 크기 계산에서는 제외하고 상세 목록에 표시 |

## 17. 중복 배치 방지 규약

Reverse Engineering 보드에서 같은 실제 AWS 리소스가 여러 번 나오면 안 된다.

같은 리소스 판단 기준:

```text
provider + region + providerResourceType + providerResourceId
```

기본 규칙:

```text
같은 providerResourceId를 가진 리소스는 보드에 한 번만 배치한다.
여러 관계가 있어도 노드는 하나만 만든다.
관계는 여러 개 만들 수 있다.
```

예:

```text
RDS가 subnet-a, subnet-b를 모두 참조한다.
→ RDS 노드는 하나만 만든다.
→ subnet 관계는 여러 개 유지할 수 있다.
→ 보드 위치는 대표 위치 하나만 선택한다.
```

## 18. 관계 생성 금지 규칙

아래 경우에는 관계를 만들지 않는다.

```text
대상 리소스를 scan 결과에서 찾지 못한 경우
파라미터 값이 문자열이지만 리소스 ID인지 확실하지 않은 경우
이름이 비슷할 뿐 ID가 일치하지 않는 경우
서로 같은 리소스인 경우
UNKNOWN 리소스의 의미를 확정할 수 없는 경우
```

특히 이름 기반 추론은 금지한다.

예:

```text
web-server라는 EC2와 web-sg라는 Security Group이 있다.
이름이 비슷하다는 이유만으로 관계를 만들면 안 된다.
```

## 19. 관계 label 규약

`type`은 내부 규약이다.

`label`은 원본 의미나 사용자 설명에 가깝다.

규칙:

```text
type은 반드시 contains/connects_to/depends_on 중 하나다.
label은 원본 AWS 관계를 보존할 수 있다.
label은 화면 표시나 디버깅에 사용한다.
label만 보고 핵심 로직을 짜면 안 된다.
```

권장 label:

| label | 의미 |
| --- | --- |
| `contains` | 공간 포함 |
| `attached_to` | 붙어 있음 |
| `associated_with` | 연결 설정으로 묶임 |
| `uses` | 사용함 |
| `routes_to` | 트래픽을 보냄 |
| `protects` | 보안 경계로 보호함 |
| `logs_to` | 로그를 보냄 |
| `encrypts_with` | 암호화 키를 사용함 |

## 20. 현재 구현과 맞춰야 하는 부분

현재 코드 기준으로 맞춰야 하는 부분은 아래와 같다.

```text
packages/types/src/index.ts
→ DiscoveredResourceRelationshipType은 contains/connects_to/depends_on이다.

apps/api/src/reverse-engineering/aws-provider-adapter.ts
→ provider 단계 attached_to는 내부 connects_to로 바뀐다.

apps/api/src/reverse-engineering/aws-provider-architecture-layout.ts
→ contains와 config 참조를 이용해 VPC/Subnet 안쪽 배치를 정한다.

apps/web/features/workspace/workspace-ai-diagram-adapter.ts
→ contains, hosts edge는 area parent 관계로 처리한다.
→ fitAreaNodesToChildren은 컨테이너가 자식 리소스를 감싸도록 크기를 보정한다.
```

이 문서와 코드가 어긋나면 코드나 문서 중 하나를 반드시 같이 수정해야 한다.

## 21. 구현자가 지켜야 할 체크리스트

새 리소스 타입이나 새 관계를 추가할 때 아래를 확인한다.

```text
[ ] 이 관계가 contains/connects_to/depends_on 중 무엇인지 정했다.
[ ] targetResourceId가 providerResourceId가 아니라 내부 id인지 확인했다.
[ ] 관계를 만들 대상 리소스가 scan 결과에 존재한다.
[ ] 이름이 비슷하다는 이유로 관계를 만들지 않았다.
[ ] 기본 보드에 보여줄 관계인지 상세 패널로 보낼 관계인지 정했다.
[ ] 같은 AWS 리소스를 보드에 여러 번 배치하지 않는다.
[ ] UNKNOWN은 기본 보드 중앙에 자동 배치하지 않는다.
[ ] 관계 방향을 트래픽 방향으로 오해하지 않는다.
[ ] contains 관계가 자동 배치에 영향을 주는지 확인했다.
[ ] connects_to와 depends_on이 컨테이너 배치를 흔들지 않게 했다.
[ ] 사용자가 직접 옮긴 manual 위치를 함부로 덮어쓰지 않는다.
[ ] 컨테이너가 자식 리소스를 모두 감싸는지 확인했다.
[ ] 중첩 컨테이너는 안쪽부터 크기를 계산한다.
[ ] 중요한 관계에는 테스트를 추가했다.
```

## 22. 팀 공통 합의가 필요한 부분

아래는 아직 제품 고도화에서 더 정해야 한다.

```text
트래픽 관계를 별도 타입으로 늘릴지
권한 관계를 별도 타입으로 늘릴지
운영 관계를 별도 타입으로 늘릴지
DB Subnet Group을 정식 ResourceType으로 추가할지
ALB, Target Group, NAT Gateway를 정식 ResourceType으로 추가할지
Route Table Association을 보드 노드로 둘지, 관계 metadata로만 둘지
사용자 수동 위치 metadata를 어떤 필드로 저장할지
컨테이너 자동 정리 버튼을 어디에 둘지
컨테이너 자동 크기 조절 잠금 UI를 둘지
Security Group을 컨테이너처럼 보여줄지, 보조 리소스로만 둘지
DB Subnet Group을 RDS 배치 기준으로 정식 지원할지
```

현재 규약에서는 타입을 늘리지 않는다.

현재는 아래 세 타입으로 먼저 통일한다.

```text
contains
connects_to
depends_on
```

필요하면 나중에 다음 타입을 추가할 수 있다.

```text
routes_to
uses_permission
logs_to
encrypts_with
monitors
```

하지만 타입을 추가하려면 `packages/types`, API, 프론트, Terraform 변환, AI 분석이 모두 같이 바뀌어야 한다.

따라서 지금은 함부로 늘리지 않는다.

