# 현재 작업 방향

이 문서는 예전 AI 전담 문서와 구분하기 위한 현재 기준 문서다.

## 1. 최종 방향

SketchCatch는 요구사항 입력부터 Practice Architecture, IaC Preview, 안전 검증, Direct Deployment 또는 Git/CI/CD handoff, Reverse Engineering, Cost Risk, Deployment History까지 이어지는 AWS-first, Terraform-first, multi-cloud-ready IaC 운영 서비스로 완성한다.

## 2. 현재 핵심 흐름

```text
Requirement Prompt
-> Architecture Draft
-> User-Accepted Board Change
-> IaC Preview
-> Diagnostics
-> Pre-Deployment Check
-> Cost Risk
-> Terraform Plan
-> Approval
-> Direct Apply 또는 Git/CI/CD Handoff
-> Deployment History
-> Cleanup
```

Reverse Engineering은 아래 흐름으로 같은 후반 단계에 합류한다.

```text
Scan
-> Practice Architecture 복원
-> IaC Preview
-> Safety/Cost
-> Git/CI/CD 또는 Direct Deployment
```

Source Repository 입력은 아래 흐름으로 Architecture Draft 단계에 합류한다.

```text
Source Repository
-> Repository Analysis
-> Template Selection
-> 선택한 Template과 분석 근거를 AI 파트에 전달
-> AI 파트가 부족한 요구사항 보완
-> Architecture Draft
```

## 3. 현재 역할 연결

| 단계 | 책임 |
| --- | --- |
| Source Repository 연결 | 사용자가 분석할 GitHub repository를 프로젝트에 연결 |
| Repository Analysis | gg가 repository 파일과 metadata에서 애플리케이션 구조와 Template 선택 근거 추출 |
| Template Selection | gg가 분석 근거에 맞는 기존 Template 선택 |
| AI 보완 | AI 파트가 선택된 Template과 분석 근거를 받아 부족한 요구사항 보완 |
| Architecture Draft 이후 | AI, Architecture Board, IaC Preview 담당 흐름으로 전달 |

## 4. 내 담당 범위

현재 내 담당 범위는 AI가 내용을 보완하는 단계가 아니라, 그 앞의 Repository Analysis와 Template Selection이다.

- 연결된 GitHub Source Repository 분석
- frontend/backend 구조와 주요 실행 단서 추출
- `package.json`, `Dockerfile` 등 Template 선택 근거 수집
- 분석 근거에 맞는 기존 Template 선택
- 선택한 Template과 분석 근거를 AI 파트에 전달

AI 파트가 맡는 일은 선택된 Template에서 부족한 요구사항을 보완하고 Architecture Draft로 발전시키는 것이다. gg는 그 보완 내용을 직접 생성하지 않는다.

## 5. old 문서 위치

예전 AI 전담 시절 문서는 `docs/gg/old-ai-work` 아래에 모아둔다.

현재 작업을 볼 때는 이 문서를 먼저 보고, 과거 결정이나 구현 배경이 필요할 때만 `old-ai-work`를 확인한다.
