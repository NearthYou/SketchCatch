# 이슈 21 그릴링 결정 구현 계획

이 문서는 `docs/gg/feat-gg-21-architecture-graph/004_이슈21_그릴링결정메모_gg.md`를 보고 만든 구현 계획이다.

목표는 단순하다.

```text
자연어 + 선택값
→ 안전한 ArchitectureJson 초안
→ 왜 그렇게 만들었는지 설명
→ 지원하지 않는 요구사항은 warning으로 표시
```

이번 계획에서 중요한 원칙:

- 새 `InfrastructureGraph` 타입은 만들지 않는다.
- 결과의 핵심은 계속 `ArchitectureJson`이다.
- AI가 아무 AWS 구조나 마음대로 만들면 안 된다.
- 지원하는 시나리오 안에서만 초안을 만든다.
- 사용자가 직접 고른 값은 자연어보다 우선한다.
- 포기하거나 미루는 것은 `docs/gg/001_고도화.md`에 남긴다.

## Milestone 1. 응답 metadata 타입 확장

목표:

초안이 왜 만들어졌는지 설명할 수 있게 metadata를 넓힌다.

지금 필요한 metadata:

```ts
selectedScenario
scenarioScores
guardrailWarnings
```

쉽게 말하면:

- `selectedScenario`: 최종으로 고른 설계 종류
- `scenarioScores`: auto일 때 각 설계 종류가 몇 점이었는지
- `guardrailWarnings`: 지원하지 않는 요구사항이나 충돌 안내

수정할 파일:

- `packages/types/src/index.ts`
- `apps/api/src/routes/ai.test.ts`
- 필요하면 `apps/web/app/workspace/AiWorkspaceClient.tsx`

할 일:

- `ArchitectureScenario` 타입 추가
- `ArchitectureScenarioScore` 타입 추가
- `ArchitectureGuardrailWarning` 타입 추가
- `AiResultMetadata`에 optional field 추가

주의할 점:

- `architectureJson`은 계속 필수다.
- metadata는 설명용이다.
- metadata가 없어도 보드가 깨지면 안 된다.

완료 조건:

- shared type에서 metadata 확장 타입을 볼 수 있다.
- API 응답 테스트가 새 metadata 모양을 검사한다.
- `packages/types` typecheck가 통과한다.

## Milestone 2. auto 점수 계산 구현

목표:

`scenarioHint: "auto"`일 때 자연어를 보고 점수로 시나리오를 고른다.

지원 시나리오:

- `static_site`
- `api_server`
- `backend_with_db`

점수 규칙:

```text
DB 단서
→ backend_with_db 점수 증가

API/서버 단서
→ api_server 점수 증가

정적 웹사이트 단서
→ static_site 점수 증가
```

예시 단어:

- DB: `db`, `database`, `데이터베이스`, `rds`, `postgres`, `mysql`, `백엔드`
- API/서버: `api`, `서버`, `server`, `ec2`, `express`, `spring`
- 정적 웹사이트: `정적`, `static`, `웹사이트`, `프론트`, `frontend`, `react`, `next`

수정할 파일:

- `apps/api/src/services/aiArchitectureDrafts.ts`
- `apps/api/src/routes/ai.test.ts`

할 일:

- 점수 계산 함수 만들기
- `auto`일 때만 점수 계산 사용하기
- DB 단서와 API 단서가 같이 있으면 `backend_with_db` 우선하기
- 모든 점수가 0이면 `static_site` fallback 선택하기
- 점수 결과를 `metadata.scenarioScores`에 넣기
- 최종 선택을 `metadata.selectedScenario`에 넣기

중요한 주석:

점수 계산 함수에는 짧은 주석을 단다.

이유:

```text
LLM이 마음대로 추론하지 않게 하기 위한 deterministic scoring이다.
```

완료 조건:

- 같은 입력은 항상 같은 결과를 낸다.
- auto 선택 테스트가 통과한다.
- `metadata.scenarioScores`에 점수와 이유가 들어간다.

## Milestone 3. 직접 선택 우선과 충돌 warning 구현

목표:

사용자가 버튼으로 고른 설계 종류를 자연어보다 우선한다.

예:

```text
입력 문장: 정적 웹사이트 만들고 싶어요
선택 버튼: DB 포함 백엔드
결과: DB 포함 백엔드
```

수정할 파일:

- `apps/api/src/services/aiArchitectureDrafts.ts`
- `apps/api/src/routes/ai.test.ts`

할 일:

- `scenarioHint`가 `auto`가 아니면 그 값을 바로 선택하기
- 그래도 prompt 점수는 계산해서 충돌 여부 확인하기
- prompt 점수 1등과 직접 선택값이 다르면 warning 추가하기
- warning code 예시:

```text
scenario_conflict
```

UI에 보여줄 수 있는 문장:

```text
입력 문장과 선택한 용도가 다릅니다.
선택한 용도를 우선해서 초안을 만들었습니다.
```

중요한 주석:

직접 선택 우선 로직에는 주석을 단다.

이유:

```text
사용자가 버튼으로 명확히 고른 값은 자연어보다 강한 입력이다.
```

완료 조건:

- 직접 선택값이 자연어보다 우선한다.
- 충돌하면 `metadata.guardrailWarnings`에 남는다.
- 테스트에서 `scenario_conflict`를 확인한다.

## Milestone 4. 지원하지 않는 요구사항 fallback warning 구현

목표:

지원 가능한 단서가 있으면 그 부분만 초안으로 만들고 warning을 돌려준다. 지원 가능한 단서나 대체 가능한 단서가 전혀 없으면 초안을 만들지 않는다.

지원하지 않는 요구사항 예:

- 멀티 리전
- EKS / Kubernetes
- 금융권, 의료 수준 규제 준수
- 대규모 트래픽 자동 설계
- CI/CD 자동화
- 실제 비용 정확도 보장
- 실제 보안 적합성 보장

수정할 파일:

- `apps/api/src/services/aiArchitectureDrafts.ts`
- `apps/api/src/routes/ai.test.ts`

할 일:

- unsupported keyword 목록 만들기
- prompt에서 unsupported keyword 찾기
- 지원 가능한 단서가 있으면 HTTP error로 실패시키지 않기
- 대체 가능한 요구사항은 지원 시나리오로 치환하기
- 단서가 전혀 없으면 HTTP error로 초안 생성을 막기
- warning code 추가하기

warning code 예시:

```text
unsupported_resource_omitted
unsupported_requirement_substituted
```

UI에 보여줄 수 있는 문장:

```text
입력에 MVP 자동 초안 범위를 벗어난 요구사항이 있습니다.
지원 가능한 부분만 초안으로 생성했습니다.
지원 가능한 아키텍처 단서가 없으면 초안을 생성하지 않습니다.
```

중요한 주석:

fallback warning 함수에는 주석을 단다.

이유:

```text
데모 흐름은 끊지 않지만, 지원하지 않는 것을 지원한다고 말하지 않기 위해서다.
```

완료 조건:

- 지원 가능한 단서가 함께 있는 지원 밖 요구사항은 200 응답을 받는다.
- 단서가 전혀 없는 요구사항은 초안을 생성하지 않고 400 응답을 받는다.
- `metadata.guardrailWarnings`에 `unsupported_resource_omitted` 또는 `unsupported_requirement_substituted`가 들어간다.
- 응답이 과장된 성공처럼 보이지 않는다.

## Milestone 5. Resource id 규칙 정리

목표:

`ArchitectureJson.nodes[].id`를 사람이 읽기 쉬운 규칙으로 맞춘다.

규칙:

```text
Resource type + 역할
```

예:

```text
vpc-main
subnet-public
subnet-app
subnet-db
ec2-api
ec2-backend
rds-primary
sg-api
sg-app
sg-db
s3-site
cloudfront-site
```

수정할 파일:

- `apps/api/src/services/aiArchitectureDrafts.ts`
- `apps/api/src/routes/ai.test.ts`
- 필요하면 Pre-Deployment Check 테스트

할 일:

- node id 이름 바꾸기
- edge의 `sourceId`, `targetId`도 같이 바꾸기
- config 안의 `subnetId`, `securityGroupIds`, `vpcId`, `originResourceId`도 같이 바꾸기
- 테스트 fixture에 들어간 id도 같이 바꾸기

주의할 점:

- template id와 resource id를 섞지 않는다.
- 랜덤 id를 쓰지 않는다.
- id를 바꾸면 finding, 비용 분석, 연결선 테스트도 같이 영향을 받는다.

완료 조건:

- 모든 node id가 사람이 읽기 쉽다.
- edge와 config가 없는 id를 가리키지 않는다.
- `CheckFinding.resourceId`가 실제 node id를 가리킨다.

## Milestone 6. 선택값이 초안에 미치는 영향 정리

목표:

예산, 트래픽, 보안 선택값이 결과에 반영되게 한다.

단, 아무 리소스나 자동 추가하지 않는다.

허용 범위:

- `ArchitectureJson` 안에서만 반영한다.
- 지원하지 않는 Resource는 만들지 않는다.
- sw Terraform 변환기가 해석하기 어려운 임의 config는 넣지 않는다.
- 변경 이유는 metadata에 남긴다.

처리 방식:

### 예산

`budgetLevel: "low"`:

- 작은 인스턴스 기준 유지
- RDS 비용 warning 강화
- metadata에 낮은 예산 기준임을 남김

`budgetLevel: "normal"`:

- 기본 fallback 유지
- 고비용 보장처럼 말하지 않음

### 트래픽

`trafficLevel: "small"`:

- 단순 구조 유지

`trafficLevel: "normal"`:

- 이번 이슈에서는 ALB 자동 추가까지 하지 않음
- 대신 metadata에 "추후 ALB/Auto Scaling 검토 가능" 정도로 남김
- ALB 자동 추가는 고도화 문서에 남김

### 보안 우선순위

`securityPriority: "high"`:

- Security Group 설명 강화
- DB 공개 접근 금지 같은 보수적인 config만 사용
- 보안을 보장한다고 말하지 않음

수정할 파일:

- `apps/api/src/services/aiArchitectureDrafts.ts`
- `apps/api/src/routes/ai.test.ts`
- 필요하면 `docs/gg/001_고도화.md`

완료 조건:

- 선택값이 metadata에 남는다.
- 가능한 범위 안에서는 `ArchitectureJson` config에도 반영된다.
- 자동 추가하지 않는 항목은 고도화 문서에 남는다.

## Milestone 7. `/workspace/ai` 결과 화면에 metadata 표시

목표:

팀원이 UI에서 결과를 바로 확인할 수 있게 한다.

보여줄 것:

- 최종 선택된 시나리오
- auto 점수
- warning 목록
- assumptions

수정할 파일:

- `apps/web/app/workspace/AiWorkspaceClient.tsx`
- 필요하면 `apps/web/app/globals.css`

할 일:

- `selectedScenario` 표시
- `scenarioScores` 표시
- `guardrailWarnings` 표시
- warning은 눈에 잘 보이게 표시

주의할 점:

- `/workspace/ai`는 팀 내부 확인용 데모 UI다.
- `/workspace`는 실제 Architecture Board 작업대 화면이다.
- 최종 Architecture Board UI는 jh/ys 파트와 연결할 때 다시 맞춘다.
- UI가 깨지거나 글자가 겹치면 안 된다.

완료 조건:

- 사용자가 버튼을 눌러 metadata를 볼 수 있다.
- warning이 숨겨지지 않는다.
- 화면에서 글자가 겹치지 않는다.

## 이번 계획에서 하지 않는 것

- 새 `InfrastructureGraph` 타입 만들기
- ALB 자동 추가
- NAT Gateway 자동 추가
- Multi-AZ RDS 자동 추가
- Kubernetes/EKS 지원
- 멀티 리전 설계
- 실제 AWS Apply
- Terraform CLI 실행
- 실제 비용 정확도 보장
- 실제 보안 적합성 보장

이 항목들은 구현하다가 다시 나오면 `docs/gg/001_고도화.md`에 남긴다.
