# AI 개발 마일스톤 PRD

이 문서는 gg AI 파트를 어떤 순서로 만들지 정리한 문서다.

이슈 #21은 **Milestone 2**를 중심으로 시작한 작업이다.

현재 기준으로는 Milestone 2의 기본 흐름이 구현되어 있고, 다음 작업은 Milestone 3을 마무리하면서 Milestone 4의 최소 범위와 Milestone 5의 데모 연결을 함께 진행하는 방향이다.

## 1. 문제

사용자는 AWS 리소스를 하나하나 고르기 어렵다.

예를 들어 사용자는 이렇게 말한다.

```text
DB가 포함된 백엔드 API 서버를 만들고 싶어요.
```

그러면 우리 서비스는 이것을 설계 초안으로 바꿔줘야 한다.

결과는 보드가 바로 열 수 있는 `ArchitectureJson`이어야 한다.

쉽게 말하면:

```text
사용자 말
→ 설계 초안
→ 보드에서 확인
→ 비용/보안 점검
→ Terraform Preview 설명
```

## 2. 큰 방향

gg AI는 실제 AWS 배포를 하지 않는다.

gg AI는 아래 역할을 한다.

1. 설계 초안 만들기
2. 설계 위험 찾기
3. 오류 쉽게 설명하기
4. 비용/보안/성능 관점 설명하기
5. 수정하면 좋은 점 제안하기

외부 LLM이 없어도 fallback으로 동작해야 한다.

## 3. Milestone 1: AI API 뼈대

목표:

- AI route 만들기
- 요청 검증하기
- 공통 타입 만들기
- 외부 LLM 없이 기본 응답 반환하기

API 후보:

- `POST /api/ai/architecture-draft`
- `POST /api/ai/github-architecture-draft`
- `POST /api/ai/pre-deployment-check`
- `POST /api/ai/terraform-error-explanation`
- `POST /api/ai/terraform-preview-explanation`

`github-architecture-draft`는 핵심 경로가 아니라 보조 경로다. Public Source Repository의 README, package metadata, Docker 관련 파일을 가벼운 evidence로만 사용한다.

완료 기준:

- 잘못된 JSON은 거절
- 응답 타입은 `packages/types`와 일치
- route는 얇게 유지
- 실제 분석 로직은 service 파일로 분리

## 4. Milestone 2: 자연어 설계 초안

이슈 #21의 초기 구현 범위다.

목표:

- 자연어와 선택값을 받기
- 시나리오를 고르기
- `ArchitectureJson` 반환하기
- 선택 이유와 warning을 metadata에 넣기
- `/workspace/ai`에서 직접 테스트 가능하게 하기

주의:

- `/workspace`는 실제 Architecture Board 작업대 화면이다.
- `/workspace/ai`는 gg AI API를 팀이 직접 눌러보기 위한 데모 화면이다.
- LLM 설명 UI는 `/workspace/ai` 데모 화면과 실제 `/workspace` 화면 양쪽에 연결한다.
- 실제 `/workspace`에서는 LLM 설명을 보여주기만 하고, Resource를 자동 수정하지 않는다.

입력:

```ts
type CreateArchitectureDraftRequest = {
  prompt: string;
  scenarioHint: "auto" | "static_site" | "api_server" | "backend_with_db";
  budgetLevel: "low" | "normal";
  trafficLevel: "small" | "normal";
  securityPriority: "basic" | "high";
};
```

지원 시나리오:

| 시나리오 | 결과 |
| --- | --- |
| `static_site` | S3 + CloudFront |
| `api_server` | VPC + Subnet + EC2 + Security Group |
| `backend_with_db` | VPC + Subnet + EC2 + RDS + Security Group |

완료 기준:

- `auto`는 점수로 시나리오 선택
- 직접 고른 시나리오는 자연어보다 우선
- 지원 밖 요구사항은 fallback + warning
- 리소스 id는 `ec2-api`, `rds-primary`처럼 사람이 읽기 쉬운 값
- metadata에 선택 이유 포함
- API 테스트와 typecheck 통과

## 5. Milestone 3: 배포 전 점검

이번 작업에서 완료할 범위다.

목표:

- 설계 초안을 보고 위험 찾기
- 비용, 보안, 설정 누락을 알려주기
- checklist 만들기

초기 규칙:

| 상황 | 위험 |
| --- | --- |
| SSH가 `0.0.0.0/0` | 보안 위험 높음 |
| RDS 포함 | 비용 주의 |
| 필수 설정 누락 | 설정 확인 필요 |
| 정리 계획 없음 | 비용 남을 수 있음 |
| 작은 EC2 + 보통 트래픽 | 병목 가능성 |

완료 기준:

- finding이 특정 리소스 id를 가리킬 수 있음
- checklist 상태는 `pass`, `warning`, `fail`
- 실제 비용 보장이 아니라 추정이라고 설명
- AI가 최종 배포 가능 여부를 단정하지 않음

## 6. Milestone 4: 수정 제안과 기본 시뮬레이션

이번 작업에서는 `Design Simulation`의 최소 세로 조각까지 진행한다.

목표:

- 위험을 보고 고칠 방향 제안
- 예상 비용/성능/보안 변화 설명
- `ArchitectureSuggestion` 타입으로 구조화된 수정 제안 반환
- Resource와 Edge를 보고 요청 흐름 보여주기
- 간단한 트래픽 가정으로 병목 후보 보여주기
- 구조적 실패 시나리오 보여주기

예:

```text
SSH 전체 공개
→ 접근 IP를 제한하세요

RDS 비용 위험
→ 비용을 확인하고 정리 계획을 세우세요
```

중요한 점:

- `CheckFinding.recommendation`은 짧은 안내 문장으로 유지
- 실제 수정 제안은 별도 `ArchitectureSuggestion` 타입으로 표현
- AI 수정안은 자동 적용하지 않음
- 사용자가 확인해야 적용
- 적용 후 다시 점검
- `Design Simulation`은 `Pre-Deployment Check` 응답의 부록이 아니라 별도 API로 둠
- `CheckFinding`은 참고할 수 있지만 Simulation의 본체가 되면 안 됨
- 실제 부하 테스트나 AWS 지표 분석은 아니며, `ArchitectureJson`의 Resource와 Edge를 근거로 한 가정 기반 추정임

## 7. Milestone 5: Terraform 오류 설명

이번 작업에서는 기존 오류 설명 기능을 `/workspace/ai`의 별도 입력 영역에서 확인할 수 있게 연결하는 수준으로 제한한다.

목표:

- Terraform 오류 원문을 쉬운 말로 바꾸기
- 다음 행동을 1-3개로 알려주기
- Terraform Preview 패널과 섞지 않고 별도 영역에서 입력받기

입력:

```ts
{
  stage: "validate" | "export" | "plan" | "apply";
  rawMessage: string;
  relatedResourceId?: string;
}
```

예:

```text
AccessDenied
→ 현재 권한으로 이 작업을 할 수 없습니다.
```

1차에서는 `validate`, `export` 중심으로 시작한다.

`export`는 IaC Preview/export 흐름에서 나온 오류를 설명하기 위한 값이다.

Plan/Apply는 ck 배포 파트와 연결될 때 본격적으로 사용한다.

이번 작업에서 하지 않는 것:

- Terraform CLI 실행
- 실제 Plan/Apply 실행 결과 수집
- ck 배포 이력과 자동 연결
- 오류 설명 저장

## 8. Milestone 6: LLM 연결

목표:

- OpenAI API를 붙일 수 있게 만들기
- OpenAI 호출이 실패해도 fallback 동작 유지
- 기존 rule 기반 분석 결과를 LLM이 쉬운 설명, 요약, 추천 문장으로 보강
- `Design Simulation` 설명 보강
- `Pre-Deployment Check` 설명과 `ArchitectureSuggestion` 문장 보강
- Terraform 오류 설명 보강
- `Architecture Draft` 해석과 선택 이유 설명 보강

원칙:

- 프론트는 LLM을 직접 호출하지 않음
- OpenAI API key는 서버 환경변수로만 관리
- OpenAI 모델명은 `OPENAI_MODEL` 환경변수로 바꿀 수 있게 함
- `OPENAI_MODEL`이 없으면 코드의 기본 모델 사용
- LLM 결과는 검증 후 사용
- 테스트는 실제 LLM 호출 금지
- LLM 테스트는 fake OpenAI client로 작성
- 각 AI API route마다 LLM 성공, fallback, invalid response 케이스 확인
- 이번 단계에서는 OpenAI 전용 구현으로 진행
- OpenAI 호출은 `fetch` 직접 호출이 아니라 공식 SDK를 사용
- `apps/api`에 OpenAI SDK runtime dependency 추가
- `openAiClient`, `createOpenAiExplanation`처럼 OpenAI 기준 이름 사용 가능
- Gemini, Claude 같은 provider 교체용 추상 계층은 이번 단계 범위 밖
- LLM은 기존 rule 기반 기능을 전체 교체하지 않음
- 비용, 보안, checklist, 지원 Resource 제한, 자동 적용 금지 같은 안전 판단은 코드가 담당
- LLM은 코드가 만든 구조화된 결과를 사용자에게 이해하기 쉬운 말로 바꾸는 역할을 담당
- LLM 설명 결과는 기존 rule 결과를 덮어쓰지 않고 별도 보강 결과로 둠
- `/workspace/ai`에서는 LLM 설명 결과를 패널 맨 위의 공통 요약 박스가 아니라 관련 결과 항목 옆의 작은 `AI 설명` 영역으로 보여줌
- 실제 `/workspace`에서도 동일한 방식으로 관련 결과 항목 근처에 `AI 설명`을 노출
- 실제 `/workspace` 연결은 표시 영역 연결까지만 포함하고, Architecture Board의 Resource 자동 변경은 포함하지 않음
- LLM 설명 대기 중에는 별도 `AI 설명 생성 중` 상태를 만들지 않고 기존 API 실행 버튼 loading만 사용
- MVP에서는 API 응답은 공통 `llmExplanation` 하나로 유지하고, 화면에서 이 값을 관련 결과 영역에 배치
- `findingId`별 LLM 설명 배열처럼 항목과 1:1로 매핑되는 응답 구조는 고도화 범위로 둠
- LLM 실패, API key 없음, JSON 파싱 불가, 응답 envelope 이상은 기존 fallback 결과를 그대로 반환
- `fallbackUsed`는 LLM 호출 전체 실패뿐 아니라 일부 필드라도 rule 기반 기본값으로 대체되면 `true`
- LLM 응답에서 일부 필드만 이상하면 전체를 버리지 않고 정상 필드는 사용, 이상한 필드만 rule 기본값으로 대체
- rule 기본값은 서버의 target별 fallback builder가 생성
- fallback builder는 `Design Simulation`, `Pre-Deployment Check`, Terraform 오류 설명, `Architecture Draft`별로 분리
- fallback builder는 원본 사용자 요청이 아니라 기존 rule 결과만 근거로 기본 문구를 생성
- fallback builder는 target 이름만 보고 고정 문구를 만들지 않음
- 프론트는 빈 필드를 임시 문구로 채우지 않고 서버가 준 `llmExplanation`를 표시
- `fallbackUsed: true`이면 화면에는 에러가 아니라 작은 `기본 설명 사용` 표시만 노출
- `fallbackReason`은 optional로 두고, 화면 필수 노출이 아니라 디버깅용 이유 확인에 사용
- `fallbackReason`은 안전한 enum 값만 반환
- 원본 OpenAI 에러 메시지는 API 응답에 노출하지 않음
- OpenAI 실패 시 서버 로그에는 `target`, `fallbackReason`, 짧은 safe message만 남김
- 서버 로그에도 원본 OpenAI 에러 메시지, 원본 요청 payload, API key, 민감할 수 있는 값은 남기지 않음
- fallback 이유는 `missing_api_key`, `timeout`, `rate_limited`, `invalid_request`, `auth_error`, `provider_error`, `invalid_response`로 구분
- rule 기반 분석이 성공했다면 LLM 설명 실패만으로 API 실패를 반환하지 않음
- LLM 설명 실패는 HTTP 성공 응답과 `fallbackUsed: true`로 표현
- rule 기반 분석 자체가 실패한 경우에만 API 실패로 처리
- LLM 설명은 rule 분석이 끝난 뒤 같은 API 요청 안에서 실행
- 최종 API 응답은 rule 결과와 `llmExplanation`를 한 번에 반환
- 프론트는 rule API와 LLM API를 따로 호출하지 않음
- 프론트 loading도 API 요청 단위로만 처리하고, LLM 전용 loading state는 추가하지 않음
- AI API 호출마다 LLM 설명을 기본으로 시도
- 요청 body에 LLM 설명 on/off 옵션을 추가하지 않음
- API key가 없으면 OpenAI 호출 전 바로 fallback
- API key 없음 fallback은 `fallbackReason: "missing_api_key"`로 표현
- OpenAI timeout은 10초로 제한
- timeout 발생 시 HTTP 성공 응답과 `fallbackUsed: true`, `fallbackReason: "timeout"`으로 처리
- OpenAI 호출 실패 시 재시도하지 않고 바로 fallback 처리
- timeout, rate limit, provider error도 MVP에서는 재시도하지 않음
- LLM 응답은 필수 필드, 길이, 개수, 최소 금지 문장을 검증
- OpenAI 응답은 SDK가 지원하는 structured output 흐름을 우선 사용해 `llmExplanation` JSON 형식으로 요청
- JSON 파싱 실패 또는 공통 응답 envelope 검증 실패 시 전체 fallback
- `summary`, `highlights`, `nextActions` 중 일부 필드만 검증 실패 시 해당 필드만 fallback
- 일부 필드 fallback도 `fallbackReason: "invalid_response"`로 표시
- `llmExplanation` JSON schema는 공통으로 유지
- prompt builder는 `Design Simulation`, `Pre-Deployment Check`, Terraform 오류 설명, `Architecture Draft`별로 분리
- 기능별 prompt builder는 같은 응답 schema를 쓰되, 각 기능에 필요한 설명 기준과 금지 기준을 다르게 넣음
- LLM prompt에는 한국어로 설명하라는 기준을 넣음
- 어려운 클라우드 용어는 필요할 때만 쓰고, 사용자가 바로 이해할 수 있는 쉬운 문장으로 요청
- OpenAI 응답이 영어 또는 한영 혼합이어도 언어만으로는 fallback 처리하지 않고 그대로 표시
- 언어가 섞였더라도 JSON schema, 길이, 금지 문장 검증을 통과하면 유효한 응답으로 사용
- OpenAI에는 원본 전체가 아니라 LLM 설명용 summary payload만 전달
- summary payload에는 사용자 설명에 필요한 Resource 요약과 rule 결과 요약만 포함
- 불필요한 metadata, 저장 정보, 내부 식별자, 민감할 수 있는 값은 전달하지 않음
- Terraform 오류의 `rawMessage`는 그대로 전달하지 않고 masking/sanitize 후 전달
- `summary`는 1~300자
- `highlights`는 최대 5개, 각 120자 이하
- `nextActions`는 최대 5개, 각 120자 이하
- 빈 문자열은 제거
- 배포 가능 보장, 비용 없음 보장, 보안 안전 보장 문장은 허용하지 않음

OpenAI SDK를 쓰는 이유:

- 나중에 streaming 응답을 붙이기 쉬움
- 나중에 tool calling / function calling을 붙이기 쉬움
- structured output 기반 JSON 응답을 안정적으로 받기 쉬움
- timeout, rate limit, invalid request 같은 OpenAI 에러 타입을 나눠 fallback 처리하기 쉬움

LLM 설명 대상:

| 대상 | LLM이 보강할 것 | 코드가 유지할 것 |
| --- | --- | --- |
| `Design Simulation` | 요청 흐름, 병목, 장애 시나리오, 비용 압박 설명 | `requestFlow`, `bottlenecks`, `failureScenarios`, `costPressure` 계산 |
| `Pre-Deployment Check` | Check Finding 요약, 우선순위, 수정 제안 문장 | 위험 탐지, checklist 상태, Check Finding 원본, Resource 연결 |
| Terraform 오류 설명 | sanitize된 오류 원문을 쉬운 말로 변환, 다음 행동 정리 | `stage`, 오류 분류, masking, fallback 설명 |
| `Architecture Draft` | 선택 이유, warning, 지원 밖 요구사항 설명 | 시나리오 선택, 지원 Resource 제한, `ArchitectureJson` 정규화 |

미래 구조:

```text
ArchitectureJson / DiagramJson
→ rule 기반 분석
→ 구조화된 결과 생성
→ LLM 설명용 summary payload 생성
→ LLM 설명/요약/추천 보강
→ 검증과 부분 fallback
→ rule 결과와 llmExplanation를 함께 응답
```

역할 구분:

| 영역 | 책임 |
| --- | --- |
| rule 코드 | 위험 탐지, checklist 판정, 지원 Resource 제한, fallback 결과 생성 |
| target별 fallback builder | 기존 rule 결과를 근거로 LLM 실패 또는 일부 필드 이상 시 사용할 기본 `summary`, `highlights`, `nextActions` 생성 |
| LLM | 쉬운 설명, 요약, 우선순위 정리, 추천 문장 보강 |

테스트 기준:

- 실제 OpenAI API 호출 금지
- fake OpenAI client로 성공 응답 재현
- fake OpenAI client로 provider 실패나 timeout 재현
- fake OpenAI client로 형식이 잘못된 응답 재현
- fake OpenAI client로 일부 필드만 잘못된 응답 재현
- 각 AI API route에서 LLM 성공, fallback, invalid response 흐름 확인
- fallback 시에도 HTTP 성공 응답과 기존 rule 결과가 유지되는지 확인

그릴링 종료 산출물:

## 11. 구현 원칙

- 원천 데이터는 `ArchitectureJson`
- AI 전용 그래프 구조를 새로 만들지 않음
- 요청 검증은 Zod 사용
- AI 결과 타입은 `packages/types`에 둠
- rule 기반 결과를 먼저 만들고 LLM은 그 결과를 보강
- LLM 비중이 커져도 최종 타입 검증과 안전 경계는 코드가 담당
- 실제 배포 가능 여부는 AI가 결정하지 않음
- 비용은 추정일 뿐 실제 청구액 보장이 아님
- Terraform 최종 실행은 gg 책임이 아님

## 12. 테스트 원칙

테스트는 API 중심으로 한다.

확인할 것:

- 잘못된 요청 거절
- 올바른 요청 응답
- auto 점수 계산
- fallback warning
- `ArchitectureJson` 모양
- Terraform 오류 설명

실제 LLM은 테스트에서 호출하지 않는다.
