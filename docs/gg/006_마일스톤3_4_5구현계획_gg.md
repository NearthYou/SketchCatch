# AI 마일스톤 3, 4, 5 구현 계획

이 문서는 `005_그릴링_3_4.md`에서 확정한 결정을 실제 구현 순서로 바꾼 계획이다.

## 구현 원칙

- 원천 데이터는 계속 `ArchitectureJson`이다.
- `Pre-Deployment Check`, `Design Simulation`, `ArchitectureSuggestion`, Terraform 오류 설명의 역할을 섞지 않는다.
- 자동 수정, 실제 Terraform 실행, 실제 AWS 성능 측정은 하지 않는다.
- gg AI는 분석과 설명을 만들고, 저장과 실제 배포 실행은 다른 파트와 합의 후 연결한다.

## Milestone 3. Pre-Deployment Check 마무리

목표:

- finding과 checklist를 팀원이 바로 연결할 수 있는 형태로 정리한다.
- `CheckFinding.recommendation`은 짧은 권장 문장으로 유지한다.
- finding은 가능하면 `resourceId`로 Resource와 연결한다.

수정 후보:

- `packages/types/src/index.ts`
- `apps/api/src/services/aiPreDeploymentAnalysis.ts`
- `apps/api/src/routes/ai.test.ts`
- `apps/web/app/workspace/PreDeploymentAnalysisPanel.tsx`

완료 기준:

- open SSH, RDS 비용, 필수 설정 누락 같은 finding이 유지된다.
- finding마다 사용자가 바로 읽을 수 있는 recommendation이 있다.
- checklist 상태는 `pass`, `warning`, `fail`로 유지된다.
- AI가 최종 배포 가능 여부를 단정하지 않는다.

검증:

- `apps/api/src/routes/ai.test.ts`에서 Pre-Deployment Check 관련 테스트 확인
- API typecheck
- `/workspace/ai`에서 Draft 생성 후 배포 전 점검 실행

## Milestone 4-1. Architecture Suggestion 타입과 결과 추가

목표:

- 수정 제안을 `CheckFinding.recommendation` 문장에만 넣지 않는다.
- 별도 `ArchitectureSuggestion` 타입으로 대상 Resource, 제안 행동, 예상 영향을 표현한다.
- 제안은 화면에 보여주기만 하고 자동 적용하지 않는다.

타입 후보:

```ts
type ArchitectureSuggestion = {
  id: string;
  findingId?: string;
  title: string;
  targetResourceId?: string;
  action: "modify_resource" | "add_resource" | "remove_resource" | "manual_review";
  expectedImpact: {
    cost: "decrease" | "increase" | "neutral" | "unknown";
    security: "improve" | "weaken" | "neutral" | "unknown";
    reliability: "improve" | "weaken" | "neutral" | "unknown";
  };
  explanation: string;
};
```

수정 후보:

- `packages/types/src/index.ts`
- `apps/api/src/services/aiPreDeploymentAnalysis.ts`
- `apps/web/app/workspace/PreDeploymentAnalysisPanel.tsx`

완료 기준:

- Pre-Deployment Check 결과에서 suggestions를 볼 수 있다.
- suggestion은 finding과 연결될 수 있다.
- 자동 적용 버튼은 만들지 않는다.

검증:

- suggestion shape 테스트 추가 또는 기존 Pre-Deployment Check 테스트 확장
- `/workspace/ai`에서 suggestion 표시 확인

## Milestone 4-2. Design Simulation 별도 API 추가

목표:

- `Design Simulation`을 `Pre-Deployment Check` 응답 부록으로 만들지 않는다.
- 별도 API에서 요청 흐름, 병목 후보, 실패 시나리오를 반환한다.
- 결과는 실제 성능 측정이 아니라 가정 기반 추정이라고 분명히 말한다.

API 후보:

```text
POST /api/ai/design-simulation
```

입력 후보:

```ts
{
  architectureJson: ArchitectureJson;
  trafficLevel: "small" | "normal";
  budgetLevel: "low" | "normal";
}
```

응답 후보:

```ts
{
  assumptions: string[];
  requestFlow: string[];
  bottlenecks: SimulationBottleneck[];
  failureScenarios: SimulationFailureScenario[];
  costPressure: string[];
  recommendations: string[];
}
```

수정 후보:

- `packages/types/src/index.ts`
- `apps/api/src/routes/ai.ts`
- `apps/api/src/services/aiDesignSimulation.ts`
- `apps/api/src/routes/ai.test.ts`
- `apps/web/app/workspace/AiWorkspaceClient.tsx`

완료 기준:

- Resource와 Edge를 보고 요청 흐름을 만든다.
- 병목 후보는 Resource와 연결된다.
- 단일 EC2, 단일 RDS, public Resource 같은 실패 시나리오를 설명한다.
- `CheckFinding`은 참고할 수 있지만 Simulation의 본체가 되지 않는다.

검증:

- `POST /api/ai/design-simulation` API 테스트
- `/workspace/ai`에서 Design Simulation 버튼과 결과 확인

## Milestone 5. Terraform 오류 설명 데모 연결

목표:

- 기존 `POST /api/ai/terraform-error-explanation`을 `/workspace/ai`에서 직접 확인할 수 있게 한다.
- Terraform Preview 패널과 합치지 않고 별도 입력 영역으로 둔다.

UI 입력:

```text
Terraform 오류 설명
- stage 선택: validate / export / plan / apply
- rawMessage 입력
- relatedResourceId 선택 입력
- 오류 설명 생성 버튼
```

수정 후보:

- `apps/web/app/workspace/AiWorkspaceClient.tsx`
- 필요하면 `apps/web/app/workspace/TerraformErrorExplanationPanel.tsx`
- 필요하면 `apps/web/app/globals.css`

완료 기준:

- 사용자가 오류 메시지를 직접 넣고 설명을 받을 수 있다.
- 결과에 summary, likelyCause, nextActions가 보인다.
- Terraform Preview 설명과 오류 설명이 UI에서 섞이지 않는다.

검증:

- `/workspace/ai`에서 AccessDenied 예시 입력 후 결과 확인
- stage `export`가 화면에서 선택 가능한지 확인

## 이번 계획에서 하지 않는 것

- 실제 AWS 성능 측정
- 실제 부하 테스트
- Terraform CLI 실행
- 실제 Plan/Apply 실행 결과 자동 수집
- ck 배포 이력 자동 연결
- 오류 설명 저장
- Architecture Board 자동 변경 적용
- Terraform 코드 자동 재생성
- 외부 LLM 연결

## 전체 검증 순서

1. API 테스트를 필요한 범위만 추가하거나 확장한다.
2. `apps/api` typecheck를 실행한다.
3. `apps/web` typecheck를 실행한다.
4. `/workspace/ai`에서 직접 버튼을 눌러 확인한다.
5. 화면에서 글자가 겹치거나 기능이 섞이지 않는지 확인한다.
