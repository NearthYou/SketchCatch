# 공통 계약 선택지

이 문서는 팀장 또는 공통 타입 담당에게 묻는 문서다.

쉽게 말하면:

> 팀 전체가 같이 써야 하는 데이터 이름과 API 모양을 정하는 문서다.

## 1. API 응답 wrapper를 쓸까?

예:

```ts
{
  data: ...,
  error: ...
}
```

### A. 모든 API에 wrapper 사용

일관성은 좋다.

하지만 기존 route가 바로 DTO를 반환한다면 수정 범위가 크다.

### B. AI route만 wrapper 사용

AI만 튀게 된다.

### C. 기존 스타일 유지. 추천

공통 wrapper가 팀 전체에 정해지기 전까지는 기존 방식을 따른다.

## 2. AI DTO는 어디에 둘까?

### A. `packages/types/src/index.ts`. 추천

API와 프론트가 같은 타입을 쓸 수 있다.

### B. API 내부에만 둠

프론트가 타입을 복사하게 될 수 있다.

### C. 프론트 내부에만 둠

백엔드와 맞추기 어렵다.

## 3. AI 결과를 저장할까?

### A. 전부 저장

나중에 다시 보기 좋다.

하지만 prompt/output 보안, 비용, 버전 문제가 있다.

### B. Pre-Deployment Analysis는 저장 가능성 고려. 추천

비용/위험 분석은 프로젝트 dashboard에서 다시 볼 수 있다.

Architecture Draft와 오류 설명은 우선 응답 중심으로 둔다.

### C. 전부 저장하지 않음

간단하지만 dashboard 연결이 약하다.

## 4. source 필드는 어떻게 나눌까?

`source`라는 말이 두 가지로 쓰일 수 있다.

1. 설계가 어디서 왔는지
2. AI 결과가 어떤 근거로 만들어졌는지

헷갈리면 안 된다.

### A. 하나의 source로 합침

간단하지만 의미가 섞인다.

### B. 이름만 다르게 함

조금 낫지만 여전히 헷갈릴 수 있다.

### C. 완전히 분리. 추천

예:

```text
ArchitectureSnapshot.source
AI metadata.source
```

서로 다른 의미로 유지한다.

## 5. ResourceType 확장은 허용할까?

AI와 보드, Terraform이 같은 ResourceType을 써야 한다.

필요한 값:

```text
SUBNET
SECURITY_GROUP
CLOUDFRONT
```

### A. 공통 타입에 포함. 추천

팀 전체가 같은 값을 쓴다.

### B. AI 내부에서만 사용

보드나 Terraform과 연결할 때 깨질 수 있다.

### C. 아직 추가하지 않음

AI 초안이 제한된다.

## 팀장에게 받고 싶은 답

```text
팀장 선택 결과

1. API 응답 wrapper: A/B/C
2. AI DTO 위치: A/B/C
3. AI 결과 저장: A/B/C
4. source 분리: A/B/C
5. ResourceType 확장: A/B/C
```

## gg 추천

추천은 C / A / B / C / A.

즉:

- API wrapper는 팀 전체 결정 전까지 기존 스타일 유지
- AI 타입은 `packages/types`
- Pre-Deployment Analysis는 저장 가능성 고려
- source 의미는 분리
- 필요한 ResourceType은 공통 타입에 추가
