# 프로젝트 학습 문서

이 문서는 내가 이 프로젝트를 계속 진행하려면 무엇을 공부해야 하는지 정리한 문서다.

내 파트만 보려고 만든 문서가 아니다.

스케치캐치 전체 흐름을 이해하고, 팀원이 질문했을 때 최소한 대답할 수 있게 만드는 것이 목표다.

---

## 1. 먼저 한 문장으로 이해하기

스케치캐치는 사용자가 말로 원하는 서버 구조를 설명하면, 그걸 설계도로 만들고, 코드로 바꾸고, 배포 전에 위험을 확인한 뒤, 사용자가 승인하면 실제 클라우드에 배포하는 서비스다.

흐름은 이렇게 생각하면 된다.

사용자 요구
→ 설계 초안
→ 보드에서 확인
→ 인프라 코드 미리보기
→ 배포 전 점검
→ 실행 계획 확인
→ 사용자 승인
→ 실제 배포
→ 배포 기록과 정리

여기서 중요한 것은 하나다.

인공지능이 전부 알아서 하는 서비스가 아니다.

인공지능은 설명과 제안을 도와주고, 실제 배포 가능한 결과는 정해진 자료 구조와 규칙을 거쳐야 한다.

---

## 2. 가장 먼저 공부할 것

### 제품 흐름

먼저 사용자가 버튼을 누른 뒤 어떤 일이 일어나는지 알아야 한다.

공부할 질문:

- 사용자는 처음에 무엇을 입력하는가?
- 설계 초안은 어디서 만들어지는가?
- 보드는 어떤 자료를 받아서 화면에 그리는가?
- 인프라 코드는 무엇을 보고 만들어지는가?
- 배포 전 점검은 무엇을 보고 위험을 찾는가?
- 실제 배포는 누가 실행하는가?
- 배포 기록은 왜 필요한가?

이걸 모르면 내가 맡은 인공지능 파트도 설명하기 어렵다.

### 역할 분리

각 파트가 어디까지 맡는지 알아야 한다.

설계 파트:

- 보드에서 자원을 보여준다.
- 자원을 배치하고 연결한다.
- 사용자가 수정한 설계를 저장한다.

인프라 코드 변환 파트:

- 보드 자료를 보고 인프라 코드를 만든다.
- 코드 문법이나 정적 문제를 검사한다.
- 코드와 보드가 어긋나지 않게 맞춘다.

배포 파트:

- 클라우드 계정 연결을 확인한다.
- 실행 계획을 만든다.
- 사용자가 승인한 뒤 실제 배포를 실행한다.
- 로그와 결과를 저장한다.
- 정리 흐름을 관리한다.

플랫폼 파트:

- 로그인과 회원가입을 맡는다.
- 프로젝트 목록과 상세 화면을 맡는다.
- 사용자 권한과 활동 내역을 관리한다.

인공지능 파트:

- 자연어를 설계 초안으로 바꾼다.
- 비용, 보안, 설정 위험을 설명한다.
- 수정하면 좋은 방향을 제안한다.
- 오류 메시지를 쉬운 말로 바꾼다.
- 단, 실제 배포와 저장 책임을 대신 가져오지 않는다.

## 3. 반드시 구분해야 하는 자료

이 프로젝트에서 제일 헷갈리기 쉬운 부분은 자료 모양이다.

### 설계도 자료

설계도 자료는 분석과 저장의 기준이다.

쉽게 말하면:

- 어떤 자원이 있는지
- 자원 이름표가 무엇인지
- 자원끼리 어떻게 연결되는지
- 각 자원에 어떤 설정이 있는지

인공지능 파트는 이 자료를 만든다.

배포 전 점검과 설계 시뮬레이션도 이 자료를 보고 판단한다.

### 보드 자료

보드 자료는 화면 편집의 기준이다.

쉽게 말하면:

- 자원이 화면 어디에 있는지
- 크기가 어떤지
- 사용자가 보드에서 어떻게 움직였는지
- 인프라 코드 변환에 필요한 값이 무엇인지

보드 파트와 인프라 코드 변환 파트가 특히 중요하게 본다.

### 둘의 차이

설계도 자료는 “무슨 구조인가”에 가깝다.

보드 자료는 “화면에서 어떻게 편집되고 코드로 바뀌는가”에 가깝다.

둘은 비슷해 보여도 같은 것이 아니다.

둘을 섞으면 이런 문제가 생긴다.

- 보드에서는 보이는데 분석이 못 읽음
- 분석은 되는데 코드 변환이 못 읽음
- 저장은 됐는데 화면 복구가 깨짐
- 자원 이름표가 서로 달라져 경고를 연결하지 못함

그래서 새 필드나 새 자원을 만들 때는 혼자 정하면 안 된다.

공통 문서와 공유 타입을 먼저 확인해야 한다.

## 4. 인공지능 파트에서 공부할 것

### 설계 초안

설계 초안은 사용자가 아직 저장하거나 확정하지 않은 제안이다.

공부할 것:

- 자연어만 믿으면 왜 위험한지
- 사용자가 고른 선택값이 왜 자연어보다 강한지
- 지원하는 시나리오를 왜 세 개 정도로 제한했는지
- 지원하지 않는 요구를 왜 실패가 아니라 경고로 처리하는지
- 외부 인공지능이 없어도 기본 결과가 왜 나와야 하는지

핵심은 이것이다.

인공지능이 자유롭게 만드는 것이 아니라, 정해진 범위 안에서 안전하게 도와주는 구조다.

### 배포 전 점검

배포 전 점검은 실제 배포 전에 위험을 찾는 단계다.

공부할 것:

- 보안 위험은 어떤 조건에서 생기는가?
- 비용 위험은 어떤 자원에서 커지는가?
- 설정 누락은 어떻게 찾는가?
- 위험 결과가 어떤 자원과 연결되는가?
- 위험을 찾는 규칙과 인공지능 설명은 왜 분리해야 하는가?

핵심은 이것이다.

위험 판단은 코드 규칙이 먼저 하고, 인공지능은 그 결과를 쉽게 설명한다.

### 수정 제안

수정 제안은 자동 수정이 아니다.

공부할 것:

- 어떤 위험에서 나온 제안인지
- 어떤 자원을 대상으로 하는지
- 비용, 보안, 안정성에 어떤 영향을 줄지
- 왜 바로 적용하지 않고 제안으로만 보여주는지

핵심은 이것이다.

사용자가 승인하기 전에는 설계를 마음대로 바꾸면 안 된다.

### 설계 시뮬레이션

설계 시뮬레이션은 실제 부하 테스트가 아니다.

공부할 것:

- 자원과 연결선을 보고 요청 흐름을 어떻게 추정하는지
- 병목 후보를 어떻게 찾는지
- 단일 서버나 단일 데이터베이스가 왜 위험할 수 있는지
- 공개된 자원이 왜 장애나 보안 위험으로 이어질 수 있는지
- 추정 결과를 실제 성능 보장처럼 말하면 왜 안 되는지

핵심은 이것이다.

설계 시뮬레이션은 “이 구조라면 이런 점을 조심하라”는 안내다.

### 오류 설명

오류 설명은 실패 로그를 쉬운 말로 바꾸는 기능이다.

공부할 것:

- 검증 단계 오류와 배포 단계 오류가 어떻게 다른지
- 권한 부족 오류는 왜 자주 생기는지
- 지역 문제와 할당량 문제는 무엇인지
- 원본 로그에 비밀값이 있을 수 있다는 점
- 오류 설명은 저장보다 즉시 이해가 먼저라는 점

핵심은 이것이다.

오류 설명은 사용자가 다음에 무엇을 확인해야 하는지 알려주는 기능이다.

### 외부 인공지능 보강

외부 인공지능은 규칙을 대체하지 않는다.

공부할 것:

- 외부 인공지능을 왜 화면에서 직접 부르면 안 되는지
- 열쇠값은 왜 서버에만 있어야 하는지
- 응답이 늦거나 실패해도 기본 결과가 나와야 하는 이유
- 외부 인공지능 응답을 왜 다시 검사해야 하는지
- 설명 보강과 실제 판단은 왜 분리해야 하는지

핵심은 이것이다.

외부 인공지능은 사용자가 읽기 쉬운 말로 보강하는 역할이다.

## 5. 인프라 코드와 배포에서 공부할 것

### 인프라 코드

인프라 코드는 클라우드 자원을 코드로 표현한 것이다.

공부할 것:

- 코드가 어떤 자원을 만드는지
- 코드와 보드가 같은 설계를 바라봐야 하는 이유
- 코드 문법 검사와 실제 배포 검사는 왜 다른지
- 실행 계획이 왜 필요한지
- 상태 파일이 왜 중요한지

핵심은 이것이다.

코드가 있다고 바로 배포하면 안 된다. 먼저 실행 계획을 확인해야 한다.

### 실행 계획

실행 계획은 실제로 무엇이 생기고, 바뀌고, 지워질지 미리 보여준다.

공부할 것:

- 생성, 수정, 삭제, 교체의 차이
- 삭제가 포함될 때 왜 더 조심해야 하는지
- 사용자가 승인한 실행 계획만 실제 적용해야 하는 이유
- 실행 계획과 실제 적용 대상이 같아야 하는 이유

핵심은 이것이다.

실행 계획은 배포 전 마지막 확인서다.

### 실제 배포

실제 배포는 클라우드에 자원을 만드는 일이다.

공부할 것:

- 왜 프론트 화면에서 직접 클라우드 작업을 하면 안 되는지
- 왜 서버나 작업 실행기가 배포를 맡아야 하는지
- 배포 로그가 왜 필요한지
- 실패 단계가 왜 기록되어야 하는지
- 배포 후 정리가 왜 필수인지

핵심은 이것이다.

실제 배포는 돈과 보안에 영향을 주기 때문에 승인, 로그, 정리 흐름이 필요하다.

### 클라우드 계정 연결

사용자의 클라우드 계정을 안전하게 연결해야 한다.

공부할 것:

- 장기 열쇠를 저장하면 왜 위험한지
- 역할 위임 방식이 왜 더 안전한지
- 연결 확인과 권한 검증이 왜 필요한지
- 계정 식별값과 연결 상태를 왜 저장하는지
- 실패한 연결을 어떻게 사용자에게 보여줘야 하는지

핵심은 이것이다.

우리는 사용자의 비밀값을 오래 저장하지 않고, 검증된 연결만 배포에 사용해야 한다.

## 6. 화면과 서버에서 공부할 것

### 화면

화면은 사용자가 흐름을 이해하게 만들어야 한다.

공부할 것:

- 입력 화면과 결과 화면을 어떻게 나누는지
- 로딩, 빈 상태, 오류 상태를 어떻게 보여주는지
- 위험 경고를 숨기지 않는 방법
- 버튼을 눌렀을 때 어떤 서버 요청이 가는지
- 임시 데모 화면과 최종 화면을 구분하는 방법

핵심은 이것이다.

화면은 기능 설명서가 아니라 사용자가 실제로 눌러보는 흐름이어야 한다.

### 서버

서버는 요청을 받고, 검사하고, 서비스 로직을 실행한다.

공부할 것:

- 화면 요청이 서버 경로로 들어오는 방식
- 요청 모양을 먼저 검사해야 하는 이유
- 경로 파일은 얇게 두고 실제 로직은 서비스 파일에 두는 이유
- 외부 호출은 왜 서버에서 해야 하는지
- 오류를 어떤 모양으로 돌려줘야 하는지

핵심은 이것이다.

서버는 바깥 입력을 안전한 내부 값으로 바꿔주는 문지기다.

### 공유 타입

공유 타입은 프론트와 서버가 같은 말을 쓰게 해준다.

공부할 것:

- 같은 필드명을 써야 하는 이유
- 자원 종류 문자열을 마음대로 만들면 왜 깨지는지
- 날짜와 상태값을 왜 통일해야 하는지
- 새 응답 모양을 만들 때 어디를 먼저 확인해야 하는지

핵심은 이것이다.

공유 타입이 어긋나면 각 파트가 따로 맞는 코드를 만들고, 연결할 때 깨진다.

## 7. 저장소와 파일 저장에서 공부할 것

### 관계형 데이터베이스

관계형 데이터베이스는 원천 자료를 저장한다.

공부할 것:

- 사용자와 프로젝트를 어떻게 연결하는지
- 설계 버전을 왜 저장하는지
- 최신 편집본과 저장된 버전은 왜 다른지
- 배포 기록과 로그를 왜 남기는지
- 비밀값 원문을 저장하면 왜 안 되는지

핵심은 이것이다.

데이터베이스에는 나중에 다시 찾아야 하는 원천 자료와 기록을 저장한다.

### 파일 저장소

파일 저장소는 파일성 산출물을 저장한다.

공부할 것:

- 인프라 코드 파일을 왜 데이터베이스 본문에 넣지 않는지
- 미리보기 파일, 실행 계획 파일, 결과 파일은 왜 따로 저장하는지
- 데이터베이스에는 파일 위치와 정보만 저장하는 이유

핵심은 이것이다.

큰 파일과 실행 산출물은 파일 저장소에 두고, 데이터베이스에는 찾기 위한 정보를 둔다.

---

## 8. 로그인과 권한에서 공부할 것

사용자 기능은 단순 로그인 화면이 아니다.

공부할 것:

- 사용자가 누구인지 확인하는 방식
- 짧게 쓰는 접근 토큰과 길게 쓰는 갱신 토큰의 차이
- 갱신 토큰 원문을 저장하지 않는 이유
- 새로고침 후 로그인 상태를 복구하는 방식
- 프로젝트 소유자를 사용자 기준으로 확인해야 하는 이유
- 요청 위조를 막기 위한 추가 확인이 필요한 이유

핵심은 이것이다.

로그인과 권한이 약하면 다른 사람이 내 프로젝트나 배포 흐름을 건드릴 수 있다.

## 9. 팀 협업에서 공부할 것

이 팀은 각자 인공지능 도구를 써서 구현한다.

그래서 사람이 서로 말만 맞추는 것보다, 각자의 도구가 같은 문서를 읽게 만드는 것이 중요하다.

공부할 것:

- 공통 문서가 왜 기준인지
- 담당자별 문서는 왜 참고 자료인지
- 새 계약을 만들 때 왜 공통 문서와 공유 타입까지 바꿔야 하는지
- 다른 파트에 물어볼 때 입력, 출력, 참조할 이름표를 어떻게 정리해야 하는지
- 내가 만들지 않을 것을 명확히 말하는 이유

핵심은 이것이다.

연결 지점이 안 맞으면 각자 기능은 돌아가도 전체 서비스는 깨진다.

## 10. 개발 절차에서 공부할 것

### 작업 시작 전

- 현재 브랜치가 맞는지 확인한다.
- 공통 문서를 읽는다.
- 내가 바꿀 자료 모양이 있는지 확인한다.
- 다른 파트와 연결되는 입력과 출력을 확인한다.

### 구현 중

- 작은 단위로 구현한다.
- 중요한 함수는 왜 있는지 설명할 수 있어야 한다.
- 새 값을 만들 때 기존 이름을 먼저 찾는다.
- 비밀값을 로그나 문서에 남기지 않는다.

### 구현 후

- 테스트를 돌린다.
- 타입 검사를 돌린다.
- 코드 규칙 검사를 돌린다.
- 화면 기능이면 직접 눌러본다.
- 실패한 검사는 숨기지 않고 말한다.

핵심은 이것이다.

코드가 맞는 것처럼 보여도, 직접 실행해서 확인하기 전까지는 끝난 게 아니다.
