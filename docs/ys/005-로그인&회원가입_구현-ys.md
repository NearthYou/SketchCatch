# 로그인/회원가입 기능 구현 계획

## 1. 목표

SketchCatch에 아이디/비밀번호 기반 로그인 및 회원가입 기능을 구현한다.

이번 작업은 단순히 로그인 화면을 만드는 것이 아니라, `userId` 기준으로 프로젝트 소유권을 연결할 수 있는 인증 기반을 만든다.

## 2. 구현 범위

### 포함

- 회원가입
- 로그인
- 로그아웃
- 현재 로그인 사용자 조회
- access token 발급
- refresh token 저장 및 재발급
- 로그인 실패 기록
- `projects.user_id` 필수 소유권 연결

### 제외

- Kakao/Naver OAuth 로그인
- 이메일 인증
- 비밀번호 찾기
- 실제 AWS 배포/삭제/수정 기능
- 관리자 계정 기능

## 3. DB 설계

### 3.1 users

사용자 계정 정보를 저장한다.

| 컬럼 | 타입 | 조건 | 설명 |
|---|---|---|---|
| id | varchar(36) | PK | 사용자 id |
| username | varchar(30) | unique, not null | 로그인 아이디 |
| email | varchar(255) | unique, not null | 이메일 |
| nickname | varchar(40) | not null | 화면 표시 이름 |
| password_hash | text | not null | 해시된 비밀번호 |
| created_at | timestamp | not null | 생성 시각 |
| updated_at | timestamp | not null | 수정 시각 |
| deleted_at | timestamp | nullable | 탈퇴/비활성화 시각 |

주의사항:

- `password_hash`는 DB 내부에서만 사용한다.
- API 응답, shared type, 프론트 상태에 `password_hash`를 노출하지 않는다.
- 회원 탈퇴는 실제 삭제보다 `deleted_at` soft delete 방식으로 처리한다.

## 3.2 refresh_tokens

로그인 세션 갱신용 refresh token을 저장한다.

| 컬럼 | 타입 | 조건 | 설명 |
|---|---|---|---|
| id | varchar(36) | PK | refresh session id |
| user_id | varchar(36) | FK users.id, not null | 사용자 id |
| token_hash | text | unique, not null | refresh token 해시값 |
| expires_at | timestamp | not null | 만료 시각 |
| revoked_at | timestamp | nullable | 로그아웃/폐기 시각 |
| user_agent | text | nullable | 접속 브라우저 정보 |
| ip_address | varchar(64) | nullable | 접속 IP |
| created_at | timestamp | not null | 생성 시각 |

주의사항:

- refresh token 원문은 DB에 저장하지 않는다.
- 로그아웃 시 `revoked_at`을 채운다.
- 만료되었거나 폐기된 token은 재사용할 수 없다.

## 3.3 login_attempts

로그인 실패 기록과 제한 처리를 위한 테이블이다.

| 컬럼 | 타입 | 조건 | 설명 |
|---|---|---|---|
| id | varchar(36) | PK | 시도 기록 id |
| username | varchar(30) | nullable | 입력한 아이디 |
| ip_address | varchar(64) | nullable | 요청 IP |
| success | boolean | not null | 성공 여부 |
| failure_reason | varchar(80) | nullable | 실패 사유 |
| locked_until | timestamp | nullable | 로그인 제한 해제 시각 |
| created_at | timestamp | not null | 시도 시각 |

주의사항:

- 실패 사유는 내부 판단용으로만 사용한다.
- 사용자에게는 “아이디 또는 비밀번호가 올바르지 않습니다”처럼 계정 존재 여부를 과하게 노출하지 않는 메시지를 반환한다.

## 3.4 projects 변경

프로젝트는 항상 로그인 사용자에게 소속된다.

| 컬럼 | 타입 | 조건 | 설명 |
|---|---|---|---|
| user_id | varchar(36) | FK users.id, not null | 프로젝트 소유자 |

정책:

- `workspace_id`는 사용하지 않는다.
- `anonymous_workspaces` 테이블은 만들지 않는다.
- 프로젝트 생성은 `Authorization: Bearer <accessToken>`이 있는 사용자만 가능하다.
- 프로젝트 조회/수정/삭제는 `user_id = currentUser.id` 조건으로만 허용한다.

## 4. Shared Type 설계

`packages/types/src/index.ts`에 인증 관련 타입을 추가한다.

```ts
export type User = {
  id: string;
  username: string;
  email: string;
  nickname: string;
  createdAt: IsoDateTimeString;
};

export type LoginRequest = {
  username: string;
  password: string;
};

export type SignupRequest = {
  username: string;
  email: string;
  nickname: string;
  password: string;
};

export type AuthSession = {
  accessToken: string;
  expiresInSeconds: number;
};

export type LoginResponse = {
  user: User;
  session: AuthSession;
};
```

금지:
- passwordHash
- tokenHash
- DB 내부용 secret 값

## 5. API 설계

## 5.1 회원가입

``` http
POST /api/auth/signup
```
Request:
``` json
{
  "username": "user",
  "email": "user@example.com",
  "nickname": "user",
  "password": "Password!123"
}
```
Response:
``` json
{
  "user": {
    "id": "user_id",
    "username": "user",
    "email": "user@example.com",
    "nickname": "user",
    "createdAt": "2026-06-23T00:00:00.000Z"
  },
  "session": {
    "accessToken": "access_token",
    "expiresInSeconds": 900
  }
}
```
완료 기준:
- username 중복 불가
- email 중복 불가
- password는 서버에서 최종 검증
- DB에는 password_hash만 저장
- refresh token 원문은 응답 body에 포함하지 않고 `HttpOnly`, `SameSite=Lax` cookie로 내려보냄

## 5.2 로그인

``` http
POST /api/auth/login
```

Request:
``` json
{
  "username": "user",
  "password": "Password!123"
}
```
Response:
``` json
{
  "user": {
    "id": "user_id",
    "username": "user",
    "email": "user@example.com",
    "nickname": "user",
    "createdAt": "2026-06-23T00:00:00.000Z"
  },
  "session": {
    "accessToken": "access_token",
    "expiresInSeconds": 900
  }
}
```
완료 기준:
- 올바른 비밀번호면 token 발급
- refresh token 원문은 응답 body에 포함하지 않고 `HttpOnly`, `SameSite=Lax` cookie로 내려보냄
- 틀린 비밀번호면 401 unauthorized
- 로그인 실패 기록 저장
- 탈퇴/비활성화 계정은 로그인 불가

## 5.3 현재 사용자 조회

``` http
GET /api/auth/me
Authorization: Bearer <accessToken>
```

Response:
``` json
{
  "user": {
    "id": "user_id",
    "username": "user",
    "email": "user@example.com",
    "nickname": "user",
    "createdAt": "2026-06-23T00:00:00.000Z"
  }
}
```
완료 기준:
- access token이 유효하면 현재 사용자 반환
- token이 없거나 만료되면 401 unauthorized

## 5.4 token 재발급
``` http
POST /api/auth/refresh
```
Request:
``` http
Cookie: sketchcatch_refresh_token=<refresh_token>
X-CSRF-Token: <csrf_token>
```
Response:
``` json
{
  "user": {
    "id": "user_id",
    "username": "user",
    "email": "user@example.com",
    "nickname": "user",
    "createdAt": "2026-06-23T00:00:00.000Z"
  },
  "session": {
    "accessToken": "new_access_token",
    "expiresInSeconds": 900
  }
}
```
완료 기준:
- refresh token이 유효하면 새 access token 발급
- 만료/폐기/위조 token은 401 unauthorized
- refresh token 원문은 DB에 저장하지 않음
- 새 refresh token 원문은 응답 body가 아니라 `Set-Cookie`의 `HttpOnly` cookie로 내려보냄
- cookie 기반 요청이므로 CSRF cookie 값과 `X-CSRF-Token` header 값이 일치해야 함

## 5.5 로그아웃
``` http
POST /api/auth/logout
```
Request:
``` http
Cookie: sketchcatch_refresh_token=<refresh_token>
X-CSRF-Token: <csrf_token>
```
완료 기준:
- 해당 refresh token의 revoked_at을 채움
- 로그아웃 이후 같은 refresh token으로 재발급 불가
- refresh cookie와 csrf cookie를 만료 처리함

## 8. 보안 기준
- 비밀번호 원문 저장 금지
- password_hash API 응답 금지
- refresh token 원문 DB 저장 금지
- refresh token 원문 API 응답 body 저장 금지
- refresh token은 `HttpOnly`, `SameSite=Lax` cookie로만 전달
- refresh/logout 같은 cookie 기반 인증 요청은 CSRF header 검증
- JWT payload에 비밀번호, token, secret 저장 금지
- 로그인 실패 시 계정 존재 여부를 직접 노출하지 않기
- 모든 보호 API는 `Authorization header` 검증
- 프로젝트 접근 시 `projectId`만 믿지 않고 `userId`를 함께 확인

- [ ] 회원가입 API가 정상 동작한다.
- [ ] username 중복 가입이 막힌다.
- [ ] email 중복 가입이 막힌다.
- [ ] 로그인 API가 정상 동작한다.
- [ ] 틀린 비밀번호로 로그인하면 `401 unauthorized`가 반환된다.

- [ ] `GET /api/auth/me`가 현재 사용자 정보를 반환한다.
- [ ] 로그아웃 후 기존 `refresh token`을 다시 사용할 수 없다.
