# ys 담당 기능 구현 계획

## 문서 목적

이 문서는 담당 범위인 다음 세 기능의 구현 방향을 정리한다.

- `Feat: Deployment Safety Gate 구현`
- `Feat: Cost Risk 분석 구현`
- `Feat: Provider Adapter 기반 Reverse Engineering 구현`

세 기능은 모두 SketchCatch의 핵심 흐름인 `Practice Architecture -> IaC Preview -> Pre-Deployment Check -> Direct Deployment Path 또는 Git/CI/CD Deployment Path`에 붙는다. MVP는 AWS-first, Terraform-first로 구현하되, 모델과 설명은 AWS-only로 만들지 않는다.

## 전체 구현 순서

권장 순서는 다음과 같다.

1. `Deployment Safety Gate`
2. `Cost Risk 분석`
3. `Provider Adapter 기반 Reverse Engineering`

이유는 `Safety Gate`와 `Cost Risk`가 같은 `CheckFinding`, `AiPreDeploymentAnalysisResult`, `Deployment.planSummary`, `Deployment.isBlocked` 흐름을 공유하기 때문이다. 이 두 기능을 먼저 안정화하면 Reverse Engineering 결과도 같은 risk/cost 분석을 재사용할 수 있다.

## 공통 원칙

- 차단 판단은 AI가 아니라 deterministic rule 기반으로 한다.
- AI는 `CheckFinding.description`, `CheckFinding.recommendation`, `ArchitectureSuggestion`, 비용 증가 원인 설명, 저비용 대안 설명에만 사용한다.
- `high` risk는 기본 block이다.
- `medium`, `low` risk는 Plan 내용을 확인한 뒤 명시 승인하면 진행 가능해야 한다.
- 실제 `apply`, `destroy`는 항상 Terraform Plan, approval snapshot, hash 검증, AWS account/region 검증, secret masking, logging, cleanup 경로 뒤에 있어야 한다.
- frontend는 AWS SDK, Terraform CLI, cloud mutation logic을 직접 실행하지 않는다.

## 1. Deployment Safety Gate 구현

### 목표

다음 항목을 high risk로 감지하고 기본 차단한다.

- public RDS
- SSH `0.0.0.0/0`
- S3 public access
- 과도한 IAM 권한
- 무승인 apply/destroy

완료 조건은 다음과 같다.

- High risk는 `risk_analysis`로 block되어 승인할 수 없다.
- Medium/Low risk는 `missing_approval` 상태로 남고, 사용자가 명시 승인한 뒤 진행할 수 있다.
- 사용자는 어떤 resource가 왜 막혔는지 `CheckFinding`과 Plan warning에서 확인할 수 있다.

### 현재 기반

이미 활용할 수 있는 코드가 있다.

- `packages/types/src/index.ts`
  - `CheckFinding`
  - `ArchitectureSuggestion`
  - `DeploymentPlanSummary`
  - `DeploymentBlockedBy`
- `apps/api/src/services/aiPreDeploymentAnalysis.ts`
  - Pre-Deployment Check 통합 진입점
- `apps/api/src/services/aiPreDeploymentSecurity.ts`
  - 현재 Security Group open SSH finding 생성
- `apps/api/src/deployments/deployment-plan-service.ts`
  - `high` finding을 Plan warning으로 바꾸고 block 처리
- `apps/api/src/deployments/deployment-approval-service.ts`
  - `blockedBy !== "missing_approval"`이면 승인 거부
- `apps/api/src/deployments/terraform-artifact-safety.ts`
  - Terraform artifact 자체의 live deployment 안전성 검사
- `apps/web/features/workspace/DeploymentPanel.tsx`
  - Plan warning, block reason, approval/apply UI 표시

### 구현 단계

1. Shared type 확인
   - `CheckFinding.category`에 필요한 값이 있는지 확인한다.
   - 현재 `security`, `permission`, `network`, `configuration`, `cost`가 있으므로 대부분 추가 없이 가능하다.
   - IAM 과권한은 `permission` category를 사용한다.

2. ArchitectureJson 기반 rule 강화
   - 위치: `apps/api/src/services/aiPreDeploymentSecurity.ts`
   - 추가할 rule:
     - `RDS` node config에서 `publiclyAccessible === true` 또는 `publicly_accessible === true`
     - `SECURITY_GROUP` ingress에서 `port === 22`와 `cidr === "0.0.0.0/0"`
     - S3 config에서 `acl: "public-read"`, `publicAccessBlock` disabled, public policy 의심 값
     - IAM 관련 `UNKNOWN` 또는 future `IAM` resource config에서 wildcard 권한 탐지
   - 각 rule은 `CheckFinding`을 만든다.
   - `high` finding은 title, description, recommendation이 사용자가 바로 수정 방향을 알 수 있어야 한다.

3. Terraform artifact 기반 rule 강화
   - 위치: `apps/api/src/deployments/terraform-artifact-safety.ts`
   - 이미 S3 public ACL, public SSH/RDP, unsupported resource, provider drift를 막는다.
   - 추가 또는 보강할 rule:
     - `aws_db_instance`의 `publicly_accessible = true`
     - `aws_s3_bucket_public_access_block`에서 public block이 꺼진 조합
     - `aws_s3_bucket_policy`에서 `Principal = "*"` 또는 public read/write policy
     - `aws_iam_policy`, `aws_iam_role_policy`, `aws_iam_policy_document`의 `Action = "*"`, `Resource = "*"`, `iam:*`, `AdministratorAccess`
   - 이 검사는 마지막 방어선이다. 사용자에게 보여줄 finding은 Pre-Deployment Check에서 별도로 만들어야 한다.

4. Deployment Plan block 정책 정리
   - 위치: `apps/api/src/deployments/deployment-plan-service.ts`
   - 기준:
     - unsupported resource type: `risk_analysis`
     - destructive change delete/replace: `risk_analysis`
     - high finding: `risk_analysis`
     - high가 없고 Plan 생성 완료: `missing_approval`
   - `createBlockedPlanSummary`가 모든 Plan을 `blocked: true`로 저장하는 것은 정상이다.
   - 승인 전에는 모두 blocked이고, 승인 성공 시 `blocked: false`가 된다.

5. Approval/Apply/Destroy 방어 테스트
   - 위치:
     - `apps/api/src/deployments/deployment-approval-service.test.ts`
     - `apps/api/src/deployments/deployment-apply-service.test.ts`
     - `apps/api/src/deployments/deployment-destroy-service.test.ts`
   - 추가 테스트:
     - `blockedBy: "risk_analysis"`인 Deployment는 승인 실패
     - approval snapshot이 없으면 apply 실패
     - approved plan hash와 현재 tfplan hash가 다르면 apply 실패
     - destroy도 destroy plan approval 없이는 실패

6. UI 표시 보강
   - 위치: `apps/web/features/workspace/DeploymentPanel.tsx`
   - 표시해야 할 상태:
     - `risk_analysis`: "수정 후 Plan 재실행 필요"
     - `missing_approval`: "Plan 확인 후 승인 가능"
     - `cost_analysis`: "비용 기준 확인 필요"
   - `PlanSummaryRows`에서 warning의 level, message, related resource를 더 명확히 보여준다.

### AI 활용 범위

AI는 다음을 생성한다.

- `CheckFinding.description` 보강
- `CheckFinding.recommendation` 보강
- `ArchitectureSuggestion` 생성
- "왜 위험한지", "어떻게 수정할지" 설명

AI는 다음을 하면 안 된다.

- block 여부 결정
- approval 자동 처리
- Terraform 수정 자동 적용
- apply/destroy 자동 실행

### 완료 체크리스트

- public RDS가 high finding으로 표시된다.
- SSH `0.0.0.0/0`이 high finding으로 표시된다.
- S3 public access가 high finding으로 표시된다.
- IAM wildcard 권한이 high finding으로 표시된다.
- high finding이 있는 Plan은 `blockedBy: "risk_analysis"`로 저장된다.
- `risk_analysis` Deployment는 승인할 수 없다.
- medium/low finding만 있으면 Plan 승인 후 apply 가능하다.
- UI에서 차단 사유와 수정 방향이 보인다.

## 2. Cost Risk 분석 구현

### 목표

다음 단위에서 비용 변화를 보여준다.

- Practice Architecture
- IaC Preview
- Deployment Plan
- Deployment History

완료 조건은 다음과 같다.

- 고비용 resource가 화면에 보인다.
- fallback estimate 산정 기준이 화면에 보인다.
- 비용 증가 원인과 저비용 대안 설명이 보인다.

### 현재 기반

이미 활용할 수 있는 코드가 있다.

- `packages/types/src/index.ts`
  - `MoneyEstimate`
  - `ResourceCostEstimate`
  - `AiPreDeploymentAnalysisResult.totalMonthlyEstimate`
  - `AiPreDeploymentAnalysisResult.resourceCostEstimates`
- `apps/api/src/services/aiPreDeploymentCost.ts`
  - RDS, NAT Gateway fallback 비용 finding
- `apps/api/src/services/aiTerraformPreviewExplanation.ts`
  - Terraform code 기반 cost finding
- `apps/web/features/workspace/DeploymentPanel.tsx`
  - Pre-Deployment Check 결과 표시
- `apps/web/app/costs/page.tsx`
  - 현재는 dashboard mock 기반 비용 관리 화면

### 구현 단계

1. 비용 계산 helper 분리
   - 위치: `apps/api/src/services/aiPreDeploymentCost.ts`
   - `createResourceCostEstimate`를 단순 if문에서 fallback table 기반으로 정리한다.
   - 예시 resource:
     - `EC2`: instance runtime, instance type
     - `RDS`: DB runtime, allocated storage
     - `S3`: storage size, request/data transfer는 fallback에서는 낮게 표시
     - `UNKNOWN service=nat_gateway`: hourly runtime, data processing
   - 실제 AWS Pricing API가 아니므로 모든 estimate에는 fallback 기준을 명시한다.

2. total 계산 수정
   - 위치: `apps/api/src/services/aiPreDeploymentAnalysis.ts`
   - 현재 `totalMonthlyEstimate.amount`가 0으로 고정되어 있다.
   - `resourceCostEstimates`의 `monthlyEstimate.amount` 합계로 변경한다.
   - `pricingAssumption`에는 fallback 기준과 한계를 적는다.

3. 고비용 finding 기준 정리
   - `medium`: RDS 같은 상시 과금 resource
   - `high`: NAT Gateway, 큰 instance type, public/managed expensive service로 보이는 resource
   - MVP에서는 정확한 가격보다 "비용 위험이 있는 resource를 사용자가 먼저 확인하는 것"을 우선한다.

4. Deployment Plan cost delta 타입 추가 검토
   - 위치: `packages/types/src/index.ts`
   - 추천 확장:
     - `CostAnalysisSummary`
     - `DeploymentPlanSummary.costSummary`
   - 예시 필드:
     - `beforeMonthlyEstimate`
     - `afterMonthlyEstimate`
     - `monthlyDelta`
     - `pricingAssumption`
     - `highCostResourceIds`
   - DB는 `deployments.plan_summary`가 JSONB라서 migration 없이 시작 가능하다.
   - 단, shared type을 바꾸면 API Zod schema와 web 소비 코드도 같이 맞춘다.

5. IaC Preview 비용 분석
   - 위치: `apps/api/src/services/aiTerraformPreviewExplanation.ts`
   - Terraform resource block에서 다음을 감지한다.
     - `aws_db_instance`
     - `aws_instance`
     - `aws_s3_bucket`
     - `aws_nat_gateway`
   - `CheckFinding`과 `ResourceCostEstimate`를 연결할 수 있으면 좋다.

6. Deployment History 비용 표시
   - 위치:
     - `apps/web/features/workspace/DeploymentPanel.tsx`
     - `apps/web/app/costs/page.tsx`
   - `Deployment.planSummary.costSummary`가 있으면 다음을 표시한다.
     - Plan 전후 예상 월 비용
     - 월 비용 delta
     - 고비용 resource
     - fallback 산정 기준

7. UI 표시
   - Pre-Deployment Check:
     - 총 월 예상 비용
     - resource별 비용 카드
     - 고비용 resource 강조
     - fallback 기준 표시
   - Plan:
     - `+3 create`, `-1 delete` 같은 변경 수와 함께 비용 delta 표시
   - History:
     - 배포별 예상 비용 변화 표시

### AI 활용 범위

AI는 다음을 설명한다.

- 비용이 증가한 주요 원인
- 비용을 줄일 수 있는 대안
- Practice session에서 cleanup이 필요한 이유
- fallback estimate의 한계

AI는 다음을 하면 안 된다.

- 실제 과금액처럼 단정
- block 여부 결정
- AWS Pricing API 대체
- 사용자 승인 없이 architecture 변경

### 완료 체크리스트

- `AiPreDeploymentAnalysisResult.totalMonthlyEstimate.amount`가 resource estimate 합계로 계산된다.
- resource별 `costDrivers`와 `explanation`이 화면에 보인다.
- 고비용 resource가 finding 또는 별도 카드로 강조된다.
- fallback estimate 기준이 화면에 보인다.
- IaC Preview에서도 RDS/NAT/EC2/S3 비용 위험을 설명한다.
- Deployment Plan 또는 History에서 비용 변화가 보인다.

## 3. Provider Adapter 기반 Reverse Engineering 구현

### 목표

AWS-first scanner를 만들되 provider-neutral resource model로 복원한다.

완료 조건은 다음과 같다.

- 기존 AWS resource가 Practice Architecture로 변환된다.
- Terraform import suggestion이 생성된다.
- Git/CI/CD handoff 흐름으로 연결할 수 있다.
- 발견 resource 구조 해석과 위험 요약이 보인다.

### 현재 기반

현재 Reverse Engineering은 문서와 타입 방향은 있지만 구현은 거의 비어 있다.

이미 활용할 수 있는 기반:

- `packages/types/src/index.ts`
  - `CloudProvider`
  - `ResourceType`
  - `ArchitectureJson`
  - `InfrastructureGraph`
- `apps/api/src/aws-connections`
  - verified AWS connection
  - STS AssumeRole 기반 credential 준비
- `apps/api/src/services/diagram-to-architecture.ts`
  - Terraform/Diagram resource type을 provider-neutral `ResourceType`으로 변환하는 매핑 참고
- `apps/api/src/services/aiPreDeploymentAnalysis.ts`
  - Reverse Engineering 결과에도 같은 risk/cost 분석 재사용 가능

### 구현 단계

1. shared contract 설계
   - 위치: `packages/types/src/index.ts`
   - 추천 타입:
     - `ReverseEngineeringScan`
     - `ReverseEngineeringScanStatus`
     - `DiscoveredResource`
     - `ProviderResourceIdentity`
     - `ImportSuggestion`
     - `ReverseEngineeringScanResult`
   - 모델은 provider-neutral로 둔다.
   - AWS 전용 값은 `provider: "aws"`, `providerResourceType`, `providerResourceId`, `region`, `accountId` 같은 metadata에 넣는다.

2. DB schema 추가
   - 위치: `apps/api/src/db/schema.ts`
   - 추천 테이블:
     - `reverse_engineering_scans`
   - 저장할 값:
     - `id`
     - `projectId`
     - `provider`
     - `status`
     - `architectureId`
     - `resultJson`
     - `errorSummary`
     - `startedAt`
     - `completedAt`
     - `createdAt`
     - `updatedAt`
   - scan 결과가 크면 RDS에는 metadata와 요약 JSON만 두고, 상세 artifact는 S3로 뺀다.

3. API route 추가
   - 위치 예시:
     - `apps/api/src/routes/reverse-engineering.ts`
     - `apps/api/src/reverse-engineering/*`
   - 최소 API:
     - `POST /api/projects/:projectId/reverse-engineering/scans`
     - `GET /api/reverse-engineering/scans/:scanId`
     - `POST /api/reverse-engineering/scans/:scanId/accept-architecture`
   - 모든 route는 인증과 project ownership을 확인한다.

4. Provider Adapter 인터페이스 추가
   - 위치 예시:
     - `apps/api/src/reverse-engineering/provider-adapter.ts`
   - 인터페이스 예:
     - `listResources(input)`
     - `normalizeResource(rawResource)`
     - `createArchitectureJson(discoveredResources)`
     - `createImportSuggestions(discoveredResources)`
     - `createRiskFindings(discoveredResources)`
   - AWS adapter는 이 인터페이스의 첫 구현체다.

5. AWS Provider Adapter 구현
   - 위치 예시:
     - `apps/api/src/reverse-engineering/aws-provider-adapter.ts`
   - read-only AWS SDK 호출만 사용한다.
   - MVP 우선순위:
     - VPC
     - Subnet
     - Internet Gateway
     - Route Table
     - Security Group
     - EC2
     - S3
     - RDS
   - AWS raw response를 frontend로 그대로 보내지 않는다.
   - secret, credential, private endpoint 등 민감 값은 저장/로그에서 제외한다.

6. Practice Architecture 복원
   - 발견 resource를 `ArchitectureJson.nodes`와 `ArchitectureJson.edges`로 변환한다.
   - `ResourceNode.type`은 provider-neutral `ResourceType`을 사용한다.
   - `ResourceNode.config`에는 배포와 분석에 필요한 최소 metadata만 넣는다.
   - 예:
     - VPC -> `VPC`
     - Subnet -> `SUBNET`
     - EC2 -> `EC2`
     - RDS -> `RDS`
     - S3 -> `S3`
     - Security Group -> `SECURITY_GROUP`

7. Terraform import suggestion 생성
   - 예시:
     - `terraform import aws_vpc.main vpc-...`
     - `terraform import aws_subnet.public subnet-...`
     - `terraform import aws_instance.web i-...`
     - `terraform import aws_s3_bucket.assets bucket-name`
   - suggestion은 자동 실행하지 않는다.
   - 사용자가 확인한 뒤 Git/CI/CD handoff 또는 IaC Preview 흐름으로 넘긴다.

8. Git/CI/CD 흐름 연결
   - MVP에서는 실제 PR 생성까지 바로 하지 않아도 된다.
   - 최소 완료 형태:
     - import suggestion 목록
     - Terraform address 후보
     - resource id
     - 위험 finding
     - "Git/CI/CD handoff로 넘길 수 있는 payload" 준비
   - 이후 `GitCicdHandoff`와 연결한다.

9. UI 추가
   - 위치 예시:
     - `apps/web/features/workspace`
     - 또는 별도 Reverse Engineering tab
   - 화면 구성:
     - AWS connection 선택
     - scan 실행
     - scan status
     - 발견 resource 목록
     - risk summary
     - Practice Architecture로 가져오기
     - import suggestion 보기
     - Git/CI/CD handoff 준비

### AI 활용 범위

AI는 다음을 생성한다.

- 발견 resource 구조 해석
- 위험 요약
- import suggestion의 의미 설명
- Practice Architecture로 가져오기 전 검토 포인트

AI는 다음을 하면 안 된다.

- AWS resource 변경
- Terraform import 자동 실행
- Practice Architecture 자동 반영
- Git commit/PR 자동 생성

### 완료 체크리스트

- AWS adapter가 read-only scan을 수행한다.
- scan 결과가 provider-neutral `DiscoveredResource`로 정규화된다.
- `ArchitectureJson`이 생성된다.
- import suggestion이 생성된다.
- scan 결과에 risk summary가 포함된다.
- 사용자가 accept하기 전에는 기존 project state가 바뀌지 않는다.
- Git/CI/CD handoff로 넘길 수 있는 데이터가 준비된다.

## 우선순위 결론

발표와 MVP 완성도를 기준으로 하면 `Deployment Safety Gate`와 `Cost Risk 분석`을 먼저 끝내야 한다. 두 기능은 Pre-Deployment Check와 Deployment 승인 흐름에 바로 보이기 때문에 사용자 여정에서 효과가 크다.

`Reverse Engineering`은 scope가 넓으므로 먼저 provider-neutral type, scan status, import suggestion까지 작게 자르고, AWS scanner는 read-only PoC로 붙이는 방식이 안전하다.
