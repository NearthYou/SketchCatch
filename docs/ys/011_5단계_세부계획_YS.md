# 5단계 세부 구현 계획 - Reverse Engineering UI와 Import Suggestion 연결

## 목적

이 문서는 `009_담당기능_구현계획_YS.md`의 5단계인 `Feat: Reverse Engineering UI와 Import Suggestion 연결`을 실제 구현 가능한 단위로 쪼갠 계획이다.

5단계의 핵심은 3~4단계에서 만든 Reverse Engineering 타입, API, AWS Provider Adapter Scanner 결과를 사용자가 볼 수 있는 화면 흐름으로 연결하는 것이다.

즉, 사용자가 기존 AWS 리소스를 스캔하고, 발견된 리소스를 확인하고, Terraform import suggestion을 검토한 뒤, 선택적으로 Practice Architecture에 반영할 수 있어야 한다.

## 전제 조건

5단계는 아래 기능이 먼저 준비되어 있다고 가정한다.

- `packages/types`에 provider-neutral resource model이 정의되어 있다.
- `apps/api`에 Reverse Engineering scan 생성, 조회, accept API가 있다.
- AWS-first scanner가 기존 AWS 리소스를 provider-neutral model로 변환한다.
- scan 결과에 resource, relationship, risk summary, import suggestion, AI explanation 또는 explanation placeholder가 포함된다.
- frontend는 AWS SDK나 Terraform CLI를 직접 실행하지 않고 API만 호출한다.

권장 선행 PR은 다음과 같다.

- `Feat: Reverse Engineering 타입과 API 기반 구현`
- `Feat: AWS Provider Adapter Scanner 구현`

## 5단계 완료 조건

- Workspace 화면에서 Reverse Engineering scan을 실행할 수 있다.
- scan 진행 상태가 `queued`, `running`, `succeeded`, `failed` 등으로 표시된다.
- 발견된 기존 AWS 리소스 목록이 provider-neutral resource model 기준으로 보인다.
- 각 리소스별 위험 요약과 비용 힌트가 표시된다.
- Terraform import suggestion이 화면에 보인다.
- import command는 복사할 수 있지만, 화면에서 자동 실행하지 않는다.
- 사용자가 명시적으로 승인해야 Practice Architecture에 반영된다.
- Practice Architecture 반영 후 Git/CI/CD 흐름으로 넘길 수 있는 다음 액션이 보인다.
- empty, error, permission denied, stale scan 상태가 모두 처리된다.

## 구현 범위

### 포함

- Reverse Engineering frontend API client
- scan 실행 UI
- scan status polling hook
- discovered resource list
- resource detail panel
- risk/cost summary 표시

- 화면 상태별 문구와 접근성 처리
- frontend helper 테스트

### 제외

- frontend에서 AWS SDK 직접 호출
- frontend에서 Terraform import 직접 실행
- frontend에서 Git commit, branch, PR 직접 생성

- scanner 신규 구현

제외 항목은 이미 앞 단계에서 처리했거나 별도 PR로 분리하는 것이 안전하다.

## 상태별 UX 기준

### no scan

사용자에게 scan 시작 폼을 보여준다.

필수 입력:

- provider
- region
- connection 또는 credential profile
- scan scope

### queued/running

사용자에게 진행 중 상태를 보여준다.

표시:

- scan status
- startedAt
- lastUpdatedAt
- 현재 단계가 있으면 currentStep

주의:

- running 중에는 중복 scan 시작을 막는다.
- 화면을 떠나도 backend scan은 계속될 수 있음을 고려한다.

### succeeded

사용자에게 결과를 보여준다.

표시:

- 발견 리소스 수
- high risk 수
- import suggestion 수
- estimated monthly cost 합계 또는 fallback summary

### failed

사용자에게 실패 이유와 재시도 버튼을 보여준다.

표시:

- 실패 코드
- 사용자에게 보여줄 수 있는 실패 설명
- 권장 다음 행동

### empty

scan은 성공했지만 발견 리소스가 없는 상태다.

표시:

- 선택 region과 scope
- permission 부족 가능성
- 다른 region으로 재시도 안내

### stale

scan 결과가 오래된 상태다.

표시:

- scan 생성 시간
- 마지막 조회 시간
- accept 전 재스캔 권장

## 데이터 흐름

1. 사용자가 Workspace에서 Reverse Engineering 탭을 연다.
2. 사용자가 provider, region, scope를 확인한다.
3. 사용자가 scan 시작 버튼을 누른다.
4. frontend가 scan 생성 API를 호출한다.
5. backend가 scanId를 반환한다.
6. frontend가 scanId로 status polling을 시작한다.
7. scan이 succeeded가 되면 discovered resource와 import suggestion을 표시한다.
8. 사용자가 리소스와 import suggestion을 검토한다.
9. 사용자가 Practice Architecture 반영을 명시 승인한다.
10. frontend가 accept API를 호출한다.
11. backend가 Practice Architecture를 갱신한다.
12. frontend가 Workspace 상태를 갱신하거나 이동 CTA를 보여준다.
13. 사용자는 Deployment Plan 또는 Git/CI/CD 흐름으로 넘어간다.

## 리뷰어에게 보여줄 포인트

- frontend가 AWS SDK나 Terraform CLI를 직접 실행하지 않는가
- import suggestion이 자동 실행처럼 보이지 않는가

- error, empty, stale 상태가 빠지지 않았는가

## 위험 요소와 대응

### API shape이 아직 확정되지 않은 경우

대응:

- frontend API client에만 API shape 의존성을 모은다.
- UI 컴포넌트는 provider-neutral view model을 받게 한다.
- backend 변경이 생기면 formatter와 adapter만 수정한다.

### scan 시간이 오래 걸리는 경우

대응:

- polling interval을 너무 짧게 잡지 않는다.
- running 상태에서 중복 요청을 막는다.
- 사용자가 화면을 떠날 수 있음을 고려한다.

### import suggestion을 실제 import 실행으로 오해하는 경우

대응:

- `제안`, `복사`, `검토` 표현을 사용한다.
- `실행` 버튼을 만들지 않는다.
- plan 확인이 필요하다는 문구를 표시한다.

### Practice Architecture 반영 범위가 불명확한 경우

대응:

- 기본은 선택 리소스만 반영한다.
- 전체 반영은 별도 확인을 둔다.
- high risk 리소스 포함 여부를 accept confirmation에서 다시 보여준다.

### 비용 정보가 없는 리소스가 많은 경우

대응:

- `unknown`과 `fallback estimate`를 구분한다.
- cost가 없는 것을 0원처럼 표시하지 않는다.
