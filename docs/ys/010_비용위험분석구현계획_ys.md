# Cost Risk 분석 구현 계획

이 문서는 `Feat: Cost Risk 분석 구현` 작업을 독립 이슈나 PR로 넘길 수 있게 정리한 참고 계획이다. 확정 계약은 `docs/product.md`, `docs/data-models.md`, `docs/architecture.md`, `docs/deployment.md`, `packages/types/src/index.ts`를 우선한다.

## 목표

Cost Analysis를 단순 가격표가 아니라 배포 전후의 Cost Risk 관리 기능으로 만든다.

완성된 흐름은 아래 기준을 만족해야 한다.

- Practice Architecture, IaC Preview, Deployment Plan, Deployment History 단위로 예상 비용과 비용 위험을 볼 수 있다.
- 고비용 리소스와 fallback estimate 산정 기준이 화면에 보인다.
- 사용량 정보가 없는 항목은 정확한 금액처럼 보이지 않고 "usage unknown" 또는 fallback 기준으로 표시된다.
- AI는 비용 증가 원인과 저비용 대안을 설명하지만, 비용 산정과 warning 생성은 deterministic rule을 우선한다.

## 핵심 관점

Cost Analysis는 월 예상 비용 카드 하나가 아니라 Cost Risk 관리 기능이다. 같은 비용 정보를 화면 위치마다 다르게 보여주는 것이 아니라, 같은 산정 기준과 finding 체계를 각 workflow 단계에 맞게 재사용해야 한다.

- Practice Architecture 단계에서는 예상 고비용 리소스를 먼저 드러낸다.
- IaC Preview 단계에서는 Terraform 기준 비용 항목과 fallback 산정 기준을 보여준다.
- Deployment Plan 단계에서는 create/update/delete/replace별 비용 변화량을 보여준다.
- Deployment History 단계에서는 실제 실행 단위의 cost snapshot을 추적한다.
- AWS Pricing API가 안정화되기 전에는 월 730시간 fallback estimate와 산정 기준을 명시한다.

## 현재 구현 기반

현재 코드에는 비용 분석의 기본 형태가 있다.

- `packages/types/src/index.ts`
  - `MoneyEstimate`
  - `ResourceCostEstimate`
  - `CheckFinding`
  - `AiPreDeploymentAnalysisResult.totalMonthlyEstimate`
- `apps/api/src/services/aiPreDeploymentCost.ts`
  - RDS와 NAT Gateway fallback 비용 finding을 만든다.
  - 현재 total은 `aiPreDeploymentAnalysis.ts`에서 0 USD로 고정된다.
- `apps/api/src/services/aiPreDeploymentAnalysis.ts`
  - resource별 cost estimate와 cost finding을 Pre-Deployment Check에 포함한다.
- `apps/web/app/costs/page.tsx`
  - 현재는 static dashboard data 기반 비용관리 페이지다.
- `apps/web/app/workspace/PreDeploymentAnalysisPanel.tsx`
  - Pre-Deployment Check finding과 suggestion을 표시한다.

현재 부족한 점은 scope별 비용 모델이 없고, total estimate가 실제 resource estimate 합계로 계산되지 않으며, 비용관리 페이지와 Deployment Plan/History가 같은 Cost Risk 결과를 공유하지 않는 것이다.

## 구현 범위

### 1. Cost Analysis service 분리

`apps/api/src/services/cost-analysis.ts`를 추가한다.

책임:

- `ArchitectureJson` 또는 변환된 `DiagramJson`을 입력으로 받는다.
- resource별 fallback estimate를 만든다.
- total monthly estimate를 resource estimate 합계로 계산한다.
- Cost Risk `CheckFinding[]`을 만든다.
- fallback 산정 기준과 사용량 한계를 함께 반환한다.

예상 입력:

```ts
type CostAnalysisScope =
  | "architecture"
  | "iac_preview"
  | "deployment_plan"
  | "deployment_history";

type AnalyzeCostInput = {
  scope: CostAnalysisScope;
  architectureJson: ArchitectureJson;
  planSummary?: DeploymentPlanSummary;
};
```

예상 출력:

```ts
type CostAnalysisResult = {
  scope: CostAnalysisScope;
  totalMonthlyEstimate: MoneyEstimate & {
    pricingAssumption: string;
  };
  resourceCostEstimates: ResourceCostEstimate[];
  findings: CheckFinding[];
  pricingAssumptions: string[];
  fallbackUsed: boolean;
};
```

### 2. 추정 단가 기준

MVP v1은 실제 AWS Pricing API를 붙이지 않는다. 비용은 월 730시간 기준 fallback으로 표시한다.

AWS Pricing API가 안정화되기 전에는 fallback estimate를 실제 청구 예측처럼 말하지 않는다. 화면과 API 응답 모두에 산정 기준, 사용량 미확정 항목, region/data transfer/storage/cleanup 변수를 함께 남긴다.

기본 정책:

- EC2
  - 작은 instance type은 낮은 fallback estimate
  - 큰 instance type은 Medium 또는 High Cost Risk
- RDS
  - 기본적으로 Medium Cost Risk
  - `multiAz=true`, 큰 `allocatedStorage`, 큰 `instanceClass`는 더 높은 risk
- NAT Gateway
  - 시간당 비용과 data processing 비용이 커질 수 있으므로 High Cost Risk
  - `ResourceType` 확정 전에는 `UNKNOWN` + `config.service === "nat_gateway"`도 처리한다.
- S3
  - 저장량과 요청량이 없으면 0 USD처럼 단정하지 않는다.
  - fallback estimate는 `usage unknown` 기준 설명으로 둔다.
- 네트워크 전송
  - 사용량 정보가 없으면 금액 산정 대신 assumption으로 표시한다.

표시 문구:

- "월 730시간 기준 fallback estimate"
- "실제 청구액은 region, 사용량, data transfer, storage, cleanup 여부에 따라 달라질 수 있음"
- "사용량 정보가 없는 항목은 금액 대신 위험 설명을 우선 표시"

### 3. Pre-Deployment Check 통합

`aiPreDeploymentAnalysis.ts`는 비용 관련 로직을 직접 갖지 않고 `cost-analysis.ts`를 호출한다.

변경 기준:

- `totalMonthlyEstimate.amount`는 resource estimate 합계로 계산한다.
- `resourceCostEstimates`는 `CostAnalysisResult.resourceCostEstimates`를 그대로 사용한다.
- `findings`에는 Cost Risk finding을 기존 security/configuration finding과 합친다.
- `suggestions`는 cost finding을 기준으로 저비용 대안을 만든다.
- LLM 설명은 fallback 결과를 사람이 이해하기 쉬운 요약으로만 보강한다.

### 4. Scope별 연결

Practice Architecture:

- 사용자가 보드에서 만든 `ArchitectureJson` 또는 `DiagramJson -> ArchitectureJson` 변환 결과를 분석한다.
- 이 단계에서는 정확한 월 청구액보다 예상 고비용 리소스와 risk reason을 먼저 보여준다.
- Workspace의 Pre-Deployment Check 패널에서 바로 보여준다.

IaC Preview:

- Terraform 생성 후 정적 diagnostics와 함께 Cost Risk를 보여준다.
- Terraform `aws_*` resource 기준으로 어떤 비용 항목을 추정했는지 표시한다.
- `DiagramJson` 기준 분석을 우선하고, Terraform만 편집된 상태는 `Terraform -> DiagramJson` sync가 성공한 경우에만 반영한다.

Deployment Plan:

- `runDeploymentPlan`에서 plan summary 생성 후 Cost Analysis를 함께 생성한다.
- create/update/delete/replace별 monthly delta를 보여주고, destroy나 cleanup으로 비용이 줄어드는 경우도 변화량으로 표시한다.
- `DeploymentPlanSummary`에 `costSummary`를 optional JSON으로 붙이는 방향을 우선한다.
- 별도 비용 테이블은 v1에서 만들지 않는다.

Deployment History:

- 성공/실패한 Deployment의 `planSummary.costSummary`를 읽어 당시 비용 스냅샷을 보여준다.
- 과거 기록이 `costSummary`를 갖지 않으면 "cost snapshot unavailable"로 표시한다.

- Deployment History의 cost value는 실제 AWS bill이 아니라 실행 시점의 cost snapshot이다. 실제 청구 추적은 별도 billing integration이 붙은 뒤 확장한다.

### 5. 비용관리 페이지 연결

`apps/web/app/costs/page.tsx`를 static data 중심에서 API 결과 중심으로 바꾼다.

MVP v1 API:

- `GET /api/deployments/recent-successful-projects`로 최근 성공 배포 프로젝트를 가져온다.
- 각 deployment의 `planSummary.costSummary`가 있으면 비용 카드에 표시한다.
- 없으면 fallback unavailable 상태를 표시한다.

후속 API가 필요하면 다음 중 하나로 확장한다.

- `GET /api/projects/:projectId/cost-analysis`
- `POST /api/ai/pre-deployment-check-from-diagram` 결과의 cost 부분 재사용

### 6. Safety Gate 연결

Cost Risk는 `CheckFinding.category === "cost"`로 통합한다.

- High cost finding은 `blockedBy: "cost_analysis"`로 block할 수 있다.
- Medium/Low cost finding은 acknowledgement 후 진행 가능하다.
- 기본 MVP에서는 비용 사고 방지를 위해 NAT Gateway 같은 고비용 리소스만 High로 둔다.
- RDS는 Cost Analysis와 Pre-Deployment Check 대상이지만 기본 live apply 대상에서 제외될 수 있다.

## API/타입 영향

예상 변경 대상:

  - API 기반 비용관리 화면

  - approval 전 cost delta와 cost finding 표시

DB migration은 v1에서 피한다. `DeploymentPlanSummary` JSON에 optional `costSummary`를 저장하는 방식으로 Deployment History를 지원한다.
