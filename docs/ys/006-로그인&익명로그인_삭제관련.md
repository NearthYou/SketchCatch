# 로그인과 익명 workspace 제거 정리

## 변경 근거

### 1. 익명 workspace를 사용하지 않기로 결정했기 때문

`clientGeneratedWorkspaceId`와 `anonymousWorkspaces`는 로그인 전 익명 사용자를 구분하기 위한 구조다.

하지만 현재 정책은 익명 사용자를 지원하지 않고, 모든 프로젝트를 로그인 사용자 기준으로 관리하는 방향이다.

따라서 익명 workspace를 전제로 한 `clientGeneratedWorkspaceId`, `anonymousWorkspaces`, `workspaceId`는 더 이상 필요하지 않다.

### 2. 프로젝트 소유권 기준이 `workspaceId`에서 `userId`로 바뀌었기 때문

기존에는 프로젝트 소유자를 이렇게 판단했다.

`projects.workspace_id = clientGeneratedWorkspaceId`

변경 후에는 이렇게 판단해야 한다.

`projects.user_id = currentUser.id`

즉, 프로젝트 생성/조회/수정/삭제 권한은 클라이언트가 보낸 workspace 값이 아니라, 서버가 인증 token에서 확인한 사용자 id로 검증해야 한다.

### 3. 클라이언트가 만든 id를 권한 기준으로 쓰면 안전하지 않기 때문

`clientGeneratedWorkspaceId`는 브라우저 쪽에서 만들어서 보내는 값이다.

이 값을 프로젝트 접근 권한의 기준으로 계속 쓰면, 사용자가 임의의 workspace id를 보내는 상황까지 고려해야 한다.

로그인 기반 서비스에서는 권한 기준을 클라이언트 입력값이 아니라 서버가 검증한 인증 정보로 잡는 것이 맞다.

### 4. DB schema와 API 코드의 기준을 맞춰야 하기 때문

DB에서 `anonymous_workspaces`와 `projects.workspace_id`를 제거하고 `projects.user_id`를 필수로 만들면, API 코드도 같은 기준으로 바뀌어야 한다.

그렇지 않으면 API 코드가 존재하지 않는 컬럼을 참조하게 된다.

예를 들면 기존 코드가 계속 아래 값을 쓰면 깨진다.

`projects.workspaceId`

변경 후에는 아래 값으로 바꿔야 한다.

`projects.userId`

### 5. 로그인 후 내 프로젝트 목록 기능과도 맞기 때문

내 프로젝트 목록은 로그인 사용자의 프로젝트만 보여줘야 한다.

따라서 조회 조건도 아래처럼 되어야 한다.

`where projects.user_id = currentUser.id`

그래야 다른 사용자의 프로젝트가 섞이지 않고, 로그인한 사용자 기준의 프로젝트 목록을 안정적으로 만들 수 있다.
