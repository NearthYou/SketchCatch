# Deployment Safety Gate 구현 계획

이 문서는 `Feat: Deployment Safety Gate 구현` 작업을 독립 이슈나 PR로 넘길 수 있게 정리한 참고 계획이다. 확정 계약은 `docs/product.md`, `docs/data-models.md`, `docs/architecture.md`, `docs/deployment.md`, `packages/types/src/index.ts`를 우선한다.

## 목표

High risk 배포를 서버에서 기본 차단하고, Medium/Low risk는 사용자가 어떤 경고를 확인했는지 명시 승인한 뒤에만 진행 가능하게 만든다.

완성된 흐름은 아래 기준을 만족해야 한다.

- public RDS, SSH `0.0.0.0/0`, S3 public access, 과도한 IAM, 무승인 apply/destroy는 High risk로 block한다.
- High risk는 UI 버튼 비활성화만이 아니라 API approval/apply/destroy 단계에서도 차단한다.
- Medium/Low risk는 `acknowledgedWarningIds` 같은 명시 승인 입력이 있어야 `missing_approval` block을 해제한다.
- AI는 `CheckFinding` 설명과 `ArchitectureSuggestion`을 보강하지만, block 여부는 deterministic rule로 결정한다.

## 핵심 관점

Deployment Safety Gate는 별도 경고 화면 하나가 아니라 `Pre-Deployment Check`, `Terraform Plan Summary`, `Cost Risk`, `Approval Snapshot`을 하나의 실행 전 gate로 묶는 기능이다.

High risk는 기본 차단한다. Medium/Low는 finding과 suggestion을 사용자에게 보여준 뒤, 사용자가 어떤 warning을 승인했는지 명시적으로 남긴 경우에만 진행한다.

Apply/destroy는 승인되었다는 사실만으로 실행하지 않는다. 승인된 `tfplan`, artifact hash, AWS account, region, operation이 현재 실행 요청과 일치할 때만 실행한다.

## 현재 코드 기준 Gap 요약

| 대상 | 현재 상태 | 부족한 점 | 이번 구현 목표 |
| --- | --- | --- | --- |
| `packages/types/src/index.ts` | `DeploymentPlanWarning`은 `level`, `message`, `relatedResourceId`만 가진다. | stable id, source, code, category, acknowledgement 여부를 추적할 수 없다. | `DeploymentPlanWarning`, `ApproveDeploymentPlanRequest`, `DeploymentSafetyGateResult` 계약을 확장한다. |
| `apps/api/src/deployments/deployment-safety-gate.ts` | 현재 체크아웃에는 파일이 없다. | plan/destroy/approval 판단이 service 내부 helper에 흩어져 있다. | AI 호출 없는 pure rule service로 추가한다. |
| `apps/api/src/services/aiPreDeploymentSecurity.ts` | Security Group의 SSH `0.0.0.0/0`만 High finding으로 만든다. | public RDS, S3 public access, 과도한 IAM을 잡지 못한다. | `CheckFinding` 기반 security rule을 확장한다. |
| `apps/api/src/deployments/deployment-plan-service.ts` | `createBlockedPlanSummary`, `createDeploymentPlanBlock`이 high finding, delete/replace, unsupported type을 판단한다. | warning metadata가 얕고 apply plan 전용 로직으로 남아 있다. | Safety Gate service 결과를 plan summary와 deployment block에 반영한다. |
| `apps/api/src/deployments/deployment-destroy-plan-service.ts` | destroy plan은 unsupported type이면 `risk_analysis`, 아니면 `missing_approval`로 저장한다. | apply plan warning과 같은 metadata shape를 쓰지 않는다. | destroy plan도 같은 `DeploymentPlanWarning` 계약과 Safety Gate result를 사용한다. |
| `apps/api/src/deployments/deployment-approval-service.ts` | `blockedBy === "missing_approval"`인 deployment만 승인한다. | Medium/Low warning별 명시 승인 입력을 검증하지 않는다. | `acknowledgedWarningIds`를 받아 required warning 누락을 409로 차단한다. |
| `apps/web/features/workspace/deployment-actions.ts` | `blockedBy === "missing_approval"`일 때만 승인 버튼을 열 수 있다. | warning checkbox 상태와 승인 가능 여부가 연결되어 있지 않다. | Medium/Low acknowledgement 완료 여부를 승인 버튼 활성 조건에 추가한다. |
| `apps/web/features/workspace/DeploymentPanel.tsx` | deployment plan summary와 blocked state를 표시한다. | warning별 source/code/승인 필요 여부를 사용자에게 명확히 보여주지 못한다. | High block banner, Medium/Low checkbox, 승인 request body 연결을 추가한다. |

## 현재 구현 기반

현재 코드에는 Safety Gate에 바로 연결할 기반이 이미 있다.

- `packages/types/src/index.ts`
  - `CheckFinding`, `ChecklistItem`, `ArchitectureSuggestion`
  - `Deployment`, `DeploymentBlock`, `DeploymentBlockedBy`
  - `DeploymentPlanSummary`, `DeploymentPlanWarning`
- `apps/api/src/services/aiPreDeploymentAnalysis.ts`
  - `ArchitectureJson`을 받아 비용, 보안, 설정 finding을 만든다.
- `apps/api/src/services/aiPreDeploymentSecurity.ts`
  - 현재는 Security Group의 SSH `0.0.0.0/0`만 High Security Risk로 찾는다.
- `apps/api/src/deployments/deployment-plan-service.ts`
  - Plan 생성 후 `preDeploymentAnalysis.findings`를 `DeploymentPlanSummary.warnings`로 붙인다.
  - high finding, delete/replace, unsupported resource type을 `risk_analysis`로 block한다.
  - 일반 plan은 `missing_approval`로 block하고 approval 후 해제한다.
- `apps/api/src/deployments/deployment-approval-service.ts`
  - approval snapshot, Terraform artifact hash, tfplan hash, AWS account/region drift를 검증한다.
- `apps/api/src/routes/deployments.ts`
  - `isBlocked`이면 apply/destroy 시작을 막는다.

현재 부족한 점은 High rule이 SSH 중심이고, Medium/Low acknowledgement 계약이 없으며, warning이 어떤 finding을 기반으로 생겼는지 추적하기 어렵다는 것이다.

### 작업 1. Shared type 확장

목적: backend, route, frontend가 같은 warning 계약을 사용하게 만든다.

수정 파일:

- `packages/types/src/index.ts`

고정할 타입:

```ts
export type DeploymentPlanWarningSource =
  | "pre_deployment_check"
  | "terraform_plan"
  | "cost_risk"
  | "approval_snapshot";

export type DeploymentPlanWarningCode =
  | "PUBLIC_RDS"
  | "PUBLIC_SSH"
  | "PUBLIC_S3"
  | "IAM_WILDCARD"
  | "DESTRUCTIVE_CHANGE"
  | "UNSUPPORTED_RESOURCE"
  | "UNKNOWN_TERRAFORM_ACTION"
  | "MISSING_APPROVAL";

export type DeploymentPlanWarning = {
  id: string;
  level: DeploymentWarningLevel;
  category?: CheckFindingCategory;
  source: DeploymentPlanWarningSource;
  code: DeploymentPlanWarningCode;
  message: string;
  relatedFindingId?: string;
  relatedResourceId?: string;
  requiresAcknowledgement: boolean;
  blocksApproval: boolean;
};

export type ApproveDeploymentPlanRequest = {
  acknowledgedWarningIds: string[];
};

export type DeploymentSafetyGateResult = {
  summary: DeploymentPlanSummary;
  block: DeploymentBlock;
  requiredAcknowledgementWarningIds: string[];
};
```

구현 기준:

- 기존 `DeploymentPlanWarning` 생성 지점을 모두 함께 수정한다.
- High warning은 기본적으로 `blocksApproval: true`, `requiresAcknowledgement: false`다.
- Medium/Low warning 중 사용자 확인이 필요한 warning은 `blocksApproval: false`, `requiresAcknowledgement: true`다.
- `id`는 같은 plan/finding/resource 조합에서 재조회해도 흔들리지 않아야 한다.

완료 기준:

- TypeScript compile 기준으로 모든 `DeploymentPlanWarning` 생성 지점이 새 필드를 채운다.
- API response에 warning source/code가 포함된다.

### 작업 2. Safety Gate pure service 추가

목적: block 판단을 route/UI/plan service에 흩어두지 않고 테스트 가능한 pure function으로 모은다.

추가 파일:

- `apps/api/src/deployments/deployment-safety-gate.ts`
- `apps/api/src/deployments/deployment-safety-gate.test.ts`

입력 계약:

```ts
type EvaluateDeploymentSafetyGateInput = {
  operation: "apply" | "destroy";
  planSummary: DeploymentPlanSummary;
  findings: CheckFinding[];
  unsupportedResourceTypes: string[];
};
```

출력 계약:

```ts
type EvaluateDeploymentSafetyGateOutput = DeploymentSafetyGateResult;
```

구현 기준:

- AI 호출, DB 호출, S3 호출, Terraform 실행을 넣지 않는다.
- 입력값만 보고 `DeploymentPlanSummary.warnings`, `DeploymentBlock`, required acknowledgement id를 만든다.
- apply와 destroy를 같은 함수에서 처리하되 operation별 정책만 분기한다.

완료 기준:

- `evaluateDeploymentSafetyGate(input)` 하나로 apply/destroy plan block 결정을 재현할 수 있다.
- 기존 `createDeploymentPlanBlock`, `createDestroyPlanBlock`이 하던 판단을 이 service로 옮길 준비가 된다.

### 작업 3. Warning id/code/source 표준화

목적: 사용자가 승인한 warning id가 plan 재조회나 UI rerender마다 바뀌지 않게 한다.

id 규칙:

| Warning | id 형식 |
| --- | --- |
| `CheckFinding` 기반 | `finding:${finding.id}` |
| unsupported resource type | `terraform_plan:UNSUPPORTED_RESOURCE:${operation}:${resourceType}` |
| destructive change | `terraform_plan:DESTRUCTIVE_CHANGE:${operation}` |
| unknown action | `terraform_plan:UNKNOWN_TERRAFORM_ACTION:${resourceAddressOrIndex}` |
| missing approval | `approval_snapshot:MISSING_APPROVAL:${operation}` |

source/code 규칙:

| 상황 | `source` | `code` |
| --- | --- | --- |
| public RDS finding | `pre_deployment_check` | `PUBLIC_RDS` |
| SSH `0.0.0.0/0` finding | `pre_deployment_check` | `PUBLIC_SSH` |
| S3 public access finding | `pre_deployment_check` | `PUBLIC_S3` |
| IAM wildcard finding | `pre_deployment_check` | `IAM_WILDCARD` |
| delete/replace | `terraform_plan` | `DESTRUCTIVE_CHANGE` |
| unsupported resource | `terraform_plan` | `UNSUPPORTED_RESOURCE` |
| unknown action | `terraform_plan` | `UNKNOWN_TERRAFORM_ACTION` |
| 승인 필요 | `approval_snapshot` | `MISSING_APPROVAL` |

완료 기준:

- 같은 input을 두 번 평가하면 warning id 배열이 같다.
- UI는 warning message 문자열을 파싱하지 않고 `source`, `code`, `level`로 표시를 결정한다.

### 작업 4. Pre-Deployment security rule 확장

목적: 현재 SSH rule 중심의 security finding을 High risk block 대상 전체로 확장한다.

수정 파일:

- `apps/api/src/services/aiPreDeploymentSecurity.ts`
- `apps/api/src/services/aiPreDeploymentSecurity.test.ts`

추가 rule:

| Rule | 탐지 기준 | `CheckFinding` |
| --- | --- | --- |
| public RDS | `ResourceNode.type === "RDS"`이고 `config.publiclyAccessible === true` | `category: "security"`, `severity: "high"` |
| SSH public open | Security Group ingress rule의 port `22`, CIDR `0.0.0.0/0` | `category: "security"`, `severity: "high"` |
| S3 public access | ACL, policy, public access block config 중 public 허용 상태 | `category: "security"`, `severity: "high"` |
| IAM wildcard | action 또는 resource에 `"*"`가 과하게 열림 | `category: "permission"`, `severity: "high"` |

구현 기준:

- raw config shape가 일정하지 않을 수 있으므로 작은 type guard를 둔다.
- 탐지할 수 없는 config는 High로 단정하지 않는다.
- finding id는 stable하게 만든다.

완료 기준:

- public RDS, S3 public access, IAM wildcard test가 추가된다.
- 기존 SSH test는 유지된다.

### 작업 5. Terraform plan rule 통합

목적: Terraform show JSON에서 나온 plan summary warning도 Safety Gate warning 계약으로 통일한다.

수정 파일:

- `apps/api/src/deployments/deployment-plan-summary.ts`
- `apps/api/src/deployments/deployment-plan-summary.test.ts`
- `apps/api/src/deployments/deployment-safety-gate.ts`

정책:

| 상황 | Level | Block |
| --- | --- | --- |
| unsupported resource type | High | `risk_analysis` |
| delete/replace apply plan | High | `risk_analysis` |
| unknown Terraform action | Medium | acknowledgement 필요 |
| no-op/read only | 없음 | block 없음 |

완료 기준:

- `DeploymentPlanSummary.warnings`에 새 warning 필드가 모두 들어간다.
- unsupported type과 destructive change가 message뿐 아니라 code로 구분된다.

### 작업 6. Apply plan 연결

목적: apply plan 저장 전에 Safety Gate 결과를 반영한다.

수정 파일:

- `apps/api/src/deployments/deployment-plan-service.ts`
- `apps/api/src/deployments/deployment-plan-service.test.ts`

구현 순서:

1. Terraform show JSON으로 기본 `DeploymentPlanSummary`를 만든다.
2. `findUnsupportedLiveApplyResourceTypesFromTerraformShowJson` 결과를 구한다.
3. `analyzePreDeployment` 결과의 `findings`를 가져온다.
4. `evaluateDeploymentSafetyGate({ operation: "apply", ... })`를 호출한다.
5. 반환된 `summary`와 `block`으로 deployment를 저장한다.
6. 기존 approval snapshot 초기화 동작을 유지한다.

완료 기준:

- High finding이 있으면 `blockedBy: "risk_analysis"`다.
- High finding이 없고 승인만 필요하면 `blockedBy: "missing_approval"`다.
- Medium/Low required warning이 있으면 `blockedBy: "missing_approval"`이고 approval 전까지 blocked 상태다.

### 작업 7. Destroy plan 연결

목적: destroy plan도 같은 Safety Gate 결과를 사용하게 한다.

수정 파일:

- `apps/api/src/deployments/deployment-destroy-plan-service.ts`
- `apps/api/src/deployments/deployment-destroy-plan-service.test.ts`

정책:

| 상황 | 결과 |
| --- | --- |
| unsupported destroy resource type 있음 | `blockedBy: "risk_analysis"` |
| unsupported destroy resource type 없음 | `blockedBy: "missing_approval"` |

- unsupported destroy warning도 `source`, `code`, `blocksApproval`을 가진다.

### 작업 8. Approval API 확장

목적: Medium/Low warning을 사용자가 명시 승인했는지 서버에서 검증한다.

수정 파일:

- `apps/api/src/routes/deployments.ts`
- `apps/api/src/deployments/deployment-approval-service.ts`
- `apps/api/src/deployments/deployment-approval-service.test.ts`
- 필요 시 `packages/types/src/index.ts`

요청 body:

```ts
type ApproveDeploymentPlanRequest = {
  acknowledgedWarningIds: string[];
};
```

검증 순서:

1. deployment가 `missing_approval` 상태인지 확인한다.
2. plan summary가 존재하는지 확인한다.
3. `blocksApproval === true` warning이 있으면 승인 실패 처리한다.
4. `requiresAcknowledgement === true` warning id가 body에 모두 있는지 확인한다.
5. Terraform artifact hash, `tfplan` hash, AWS account, region drift 검증을 기존처럼 수행한다.
6. 승인 snapshot을 저장한다.

완료 기준:

- High risk deployment는 acknowledgement body가 있어도 승인되지 않는다.
- required warning id가 누락되면 409를 반환한다.
- 모든 required warning id가 있으면 기존 snapshot 저장 흐름으로 진행한다.

### 작업 9. Apply/destroy 최종 방어선 유지

목적: UI나 approval route가 우회되어도 실행 단계에서 승인 snapshot 불일치를 막는다.

확인 파일:

- `apps/api/src/deployments/deployment-apply-service.ts`
- `apps/api/src/deployments/deployment-destroy-service.ts`
- `apps/api/src/deployments/deployment-approval-service.ts`
- `apps/api/src/aws-connections/aws-connection-runtime-credentials.test.ts`

유지할 방어선:

- `approvedAt`, `approvedPlanArtifactId`, `approvedTfplanHash` 없으면 실행 실패
- 승인된 Terraform artifact hash와 현재 artifact hash가 다르면 실행 실패
- 승인된 `tfplan` hash와 S3에서 내려받은 plan hash가 다르면 실행 실패
- 승인된 AWS account/region과 현재 connection account/region이 다르면 실행 실패
- apply는 `operation: "apply"` plan만 실행
- destroy는 `operation: "destroy"` plan만 실행

완료 기준:

- Safety Gate 변경 뒤에도 기존 drift test가 유지된다.

### 설계 요약

실제 실행은 위의 작업 0~12 순서를 따른다. 아래 1~5는 각 작업에서 유지해야 할 설계 기준을 요약한 것이다.

### 1. Safety Gate service 추가

`apps/api/src/deployments/deployment-safety-gate.ts`를 추가한다.

책임:

- `CheckFinding[]`, `DeploymentPlanSummary`, Terraform resource type, operation을 입력으로 받는다.
- Cost Risk 결과도 같은 gate 입력으로 받아 비용 기반 High/Medium/Low finding을 deployment warning으로 변환한다.
- High/Medium/Low warning을 표준 `DeploymentPlanWarning`으로 만든다.
- block 결과를 `DeploymentBlock`으로 만든다.
- AI 호출 없이 순수 함수로 동작한다.

High block rules:

| Rule | 입력 기준 | `blockedBy` |
| --- | --- | --- |
| public RDS | `ResourceNode.type === "RDS"`이고 `config.publiclyAccessible === true` | `risk_analysis` |
| SSH public open | SG ingress 또는 SG rule에서 port `22`와 CIDR `0.0.0.0/0` | `risk_analysis` |
| S3 public access | ACL, bucket policy, public access block 설정이 public 허용 상태 | `risk_analysis` |
| 과도한 IAM | policy action/resource에 `"*"`가 과하게 열림 | `risk_analysis` |
| 무승인 apply/destroy | approved snapshot 없음 | `missing_approval` |
| delete/replace change | apply plan에 delete 또는 replace 있음 | `risk_analysis` |
| unsupported live resource type | MVP whitelist 밖 Terraform resource type | `risk_analysis` |

Medium/Low rules:

- 비용 증가 가능성이 큰 instance type은 Medium으로 둔다.
- 단일 AZ, backup 불명확, tag 누락, naming convention 누락은 Medium/Low로 둔다.
- Medium/Low는 block reason을 `missing_approval`로 유지하되 acknowledgement 대상 warning으로 표시한다.

### 2. Shared type 확장

`DeploymentPlanWarning`은 finding과 acknowledgement를 추적할 수 있어야 한다.

예상 확장:

```ts
type DeploymentPlanWarningSource =
  | "pre_deployment_check"
  | "terraform_plan"
  | "cost_risk"
  | "approval_snapshot";

type DeploymentPlanWarningCode =
  | "PUBLIC_RDS"
  | "PUBLIC_SSH"
  | "PUBLIC_S3"
  | "IAM_WILDCARD"
  | "DESTRUCTIVE_CHANGE"
  | "UNSUPPORTED_RESOURCE"
  | "UNKNOWN_TERRAFORM_ACTION"
  | "MISSING_APPROVAL";

type DeploymentPlanWarning = {
  id: string;
  level: DeploymentWarningLevel;
  category?: CheckFindingCategory;
  source: DeploymentPlanWarningSource;
  code: DeploymentPlanWarningCode;
  message: string;
  relatedFindingId?: string;
  relatedResourceId?: string;
  requiresAcknowledgement: boolean;
  blocksApproval: boolean;
};

type ApproveDeploymentPlanRequest = {
  acknowledgedWarningIds: string[];
};

type DeploymentSafetyGateResult = {
  summary: DeploymentPlanSummary;
  block: DeploymentBlock;
  requiredAcknowledgementWarningIds: string[];
};
```

주의:

- 기존 warning shape를 소비하는 UI와 테스트를 함께 갱신한다.
- `id`는 stable해야 하며 같은 plan을 다시 조회해도 acknowledgement 대상이 흔들리지 않아야 한다.
- High warning은 `requiresAcknowledgement`와 관계없이 approval로 해제되지 않는다.

### 3. Pre-Deployment Check rule 보강

`apps/api/src/services/aiPreDeploymentSecurity.ts`를 보강하거나 category별 rule 파일로 분리한다.

이 단계의 목적은 security finding만 늘리는 것이 아니라, `Pre-Deployment Check` 결과가 `Terraform Plan Summary`, `Cost Risk`, `Approval Snapshot`과 같은 gate 입력으로 합쳐지도록 finding shape를 정규화하는 것이다.

추가 finding:

- RDS public 접근: `security` 또는 `network`, severity `high`
- S3 public access: `security`, severity `high`
- IAM wildcard 권한: `permission`, severity `high`
- destructive plan change: `configuration` 또는 `availability`, severity `high`

RDS는 기본 Direct Deployment live apply 대상이 아니지만, Practice Architecture, IaC Preview, Cost Analysis, Pre-Deployment Check 대상에는 포함한다.

### 4. Deployment plan/approval/apply/destroy 연결

`deployment-plan-service.ts`:

- 기존 `createBlockedPlanSummary`, `createDeploymentPlanBlock`를 Safety Gate service로 대체한다.
- unsupported resource type, high finding, destructive action을 모두 같은 Gate 결과로 저장한다.
- `planSummary.blocked`는 approval 전에는 항상 `true`다.

`deployment-destroy-plan-service.ts`:

- destroy plan도 `missing_approval` block이 기본이다.
- unsupported resource type은 `risk_analysis`로 계속 block한다.
- destroy plan은 cleanup 목적이므로 delete count 자체를 High로 막지 않는다. 대신 승인 snapshot과 operation 검증을 유지한다.

`deployment-approval-service.ts`:

- body에서 `acknowledgedWarningIds`를 받는다.
- `blockedBy !== "missing_approval"`이면 승인 실패를 유지한다.
- Medium/Low warning 중 `requiresAcknowledgement === true`인 id가 모두 들어와야 승인 성공한다.
- 승인 후에도 artifact hash, tfplan hash, AWS account/region 검증은 기존대로 유지한다.

`deployment-apply-service.ts`, `deployment-destroy-service.ts`:

- 현재처럼 승인 snapshot과 `isBlocked` 검사를 마지막 방어선으로 유지한다.
- UI가 우회되어도 API에서 실행되지 않아야 한다.

### 5. UI 연결

`apps/web/features/workspace/DeploymentPanel.tsx`에서 아래 상태를 보여준다.

- `risk_analysis`: High risk block 배너, 승인 버튼 비활성화, 수정 또는 재생성 안내
- `missing_approval`: Medium/Low acknowledgement 체크 UI, 승인 버튼 활성화 조건
- `cost_analysis`: Cost Risk가 별도 block 사유로 들어올 경우 비용 기준과 대안 표시

UI 문구는 사용자가 무엇을 승인하는지 짧고 구체적으로 보여준다. 예: "SSH 22번 포트가 전체 인터넷에 열려 있음".

## Warning 계약

`DeploymentPlanWarning`은 단순 message list가 아니라 승인 판단과 UI 표시의 공통 계약이다.

| 필드 | 필수 | 설명 |
| --- | --- | --- |
| `id` | 예 | stable warning id. approval body의 기준이다. |
| `level` | 예 | `low`, `medium`, `high` 중 하나다. |
| `category` | 아니오 | `CheckFindingCategory`와 맞춘다. |
| `source` | 예 | `pre_deployment_check`, `terraform_plan`, `cost_risk`, `approval_snapshot` 중 하나다. |
| `code` | 예 | UI와 test가 message parsing 없이 판단할 수 있는 code다. |

| `relatedResourceId` | 아니오 | 관련 resource id가 있으면 연결한다. |
| `requiresAcknowledgement` | 예 | Medium/Low 명시 승인 대상인지 나타낸다. |

## Risk 정책

| Risk | 기본 정책 | 승인 가능 여부 | `blockedBy` |
| --- | --- | --- | --- |
| High security risk | 기본 차단 | 불가 | `risk_analysis` |
| High cost risk | 기본 차단 가능 | 기본 불가, Cost Risk 정책에 따름 | `cost_analysis` |
| destructive apply change | 기본 차단 | 불가 | `risk_analysis` |
| unsupported resource type | 기본 차단 | 불가 | `risk_analysis` |
| Medium/Low warning | 사용자 확인 필요 | 가능 | `missing_approval` |
| warning 없는 일반 plan | 사용자 승인 필요 | 가능 | `missing_approval` |
| 승인 snapshot 없음 | 실행 차단 | approval 후 가능 | `missing_approval` |

정책 해석:

- High warning은 `acknowledgedWarningIds`로 해제되지 않는다.
- Medium/Low warning은 required id를 모두 승인해야 진행된다.
- `missing_approval`은 위험이 없다는 뜻이 아니라 사용자 승인 절차가 남았다는 뜻이다.
- `risk_analysis`는 설계 수정, Terraform 수정, 또는 plan 재생성이 필요한 hard block이다.
- `cost_analysis`는 `docs/ys/010_비용위험분석구현계획_ys.md`와 연결되는 비용 기반 block reason으로 예약한다.
