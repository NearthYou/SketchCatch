# 1~4단계 세부 구현 계획

## 문서 목적

이 문서는 `docs/ys/009_담당기능_구현계획_YS.md`의 추천 PR 분리 중 1번부터 4번까지를 실제 구현 가능한 작은 단위로 쪼갠 계획이다.

대상 단계:

1. `Feat: Deployment Safety Gate 구현`
2. `Feat: Cost Risk 분석 구현`
3. `Feat: Reverse Engineering 타입과 API 기반 구현`
4. `Feat: AWS Provider Adapter Scanner 구현`

5번 `Reverse Engineering UI와 Import Suggestion 연결`은 이 문서 범위에서 제외한다. 단, 3~4단계에서 API 응답과 payload는 이후 UI가 바로 붙을 수 있게 만든다.

## 공통 커밋 전략

각 커밋은 리뷰 가능한 작은 단위여야 한다.

원칙:

- 한 커밋은 하나의 rule, 하나의 type, 하나의 route, 하나의 UI 표시처럼 작은 책임만 가진다.
- 가능하면 각 커밋은 관련 테스트가 통과하는 상태로 남긴다.
- shared type 변경은 API와 web 변경보다 먼저 커밋한다.
- DB schema 변경은 migration과 schema contract test를 별도 커밋으로 분리한다.
- AWS SDK dependency 변경이 필요하면 package metadata와 lockfile 변경을 별도 커밋으로 분리한다.
- secret, AWS credential, `.env`는 절대 커밋하지 않는다.
- 실제 AWS apply/destroy를 새로 열지 않는다. scanner는 read-only 호출만 사용한다.

권장 브랜치:

```text
feature/ys/<issue-number>-deployment-safety-gate
feature/ys/<issue-number>-cost-risk-analysis
feature/ys/<issue-number>-reverse-engineering-contract-api
feature/ys/<issue-number>-aws-provider-adapter-scanner
```

권장 PR 제목:

```text
Feat: Deployment Safety Gate 구현
Feat: Cost Risk 분석 구현
Feat: Reverse Engineering 타입과 API 기반 구현
Feat: AWS Provider Adapter Scanner 구현
```

전체 검증은 각 PR 마지막에 실행한다.

```bash
pnpm lint
pnpm typecheck
pnpm build
```

개발 중 빠른 검증은 범위별로 실행한다.

```bash
pnpm --filter @sketchcatch/types typecheck
pnpm --filter @sketchcatch/api test
pnpm --filter @sketchcatch/api typecheck
pnpm --filter @sketchcatch/web typecheck
```

## 1단계: Deployment Safety Gate 구현

### 목표

Pre-Deployment Check와 Deployment Plan 단계에서 다음 high risk를 rule 기반으로 감지하고, high risk가 포함된 Plan은 `blockedBy: "risk_analysis"`로 차단한다.

- public RDS
- SSH `0.0.0.0/0`
- S3 public access
- IAM wildcard/과도한 권한
- 무승인 apply/destroy

현재 `apps/api/package.json`에는 다음 AWS SDK만 있다.

- `@aws-sdk/client-s3`
- `@aws-sdk/client-sts`
- `@aws-sdk/s3-request-presigner`

EC2, VPC, Subnet, Security Group, Route Table 조회에는 `@aws-sdk/client-ec2`가 필요하다. RDS 조회에는 `@aws-sdk/client-rds`가 필요하다. dependency를 추가한다면 `package.json`과 `pnpm-lock.yaml` 변경을 별도 커밋으로 분리한다.

### 구현 순서

1. AWS SDK gateway 인터페이스를 먼저 만든다.
2. AWS raw response를 바로 쓰지 말고 normalize helper로 감싼다.
3. resource type별로 하나씩 normalization과 import suggestion을 붙인다.
4. risk summary는 기존 Pre-Deployment Check helper를 재사용한다.
5. 실제 AWS 호출은 read-only command만 허용한다.

### 4단계 완료 조건

- AWS scanner는 read-only AWS SDK command만 사용한다.
- AWS raw response는 provider-neutral `DiscoveredResource`로 정규화된다.
- VPC/Subnet/IGW/Route Table/Security Group/EC2/S3/RDS가 `ArchitectureJson`으로 복원된다.
- import suggestion이 생성된다.
- scan 결과에 risk summary가 포함된다.
- scanner failure는 `failed` status와 secret-masked `errorSummary`로 저장된다.

## 단계별 PR 완료 체크

각 PR 마지막 커밋 또는 PR 생성 전 확인한다.

```bash
pnpm lint
pnpm typecheck
pnpm build
```

추가 확인:

- 1단계는 `apps/api/src/deployments/*` 테스트를 집중 확인한다.
- 2단계는 Pre-Deployment Check route와 web 표시를 집중 확인한다.
- 3단계는 route auth, ownership, DB schema contract를 집중 확인한다.
- 4단계는 AWS SDK gateway를 모두 mock으로 테스트하고, 실제 AWS 호출은 로컬 검증 필수로 두지 않는다.

PR별로 최소한 다음 단위는 분리하는 것이 좋다.

- type
- docs
- DB schema
- migration
- repository
- service
- route
- UI
- final verification fix

## 최종 권장 진행

가장 먼저 1단계 `Deployment Safety Gate`만 issue branch로 시작한다. 이 단계는 기존 Deployment 흐름에 바로 붙고, DB migration 없이 진행할 수 있어 실패 위험이 가장 낮다.

1단계가 통과하면 2단계 `Cost Risk 분석`을 이어서 진행한다. 2단계는 `DeploymentPlanSummary.costSummary`를 추가하는 순간 API와 web contract가 같이 바뀌므로 shared type, API, UI 커밋 순서를 지킨다.

3단계와 4단계는 Reverse Engineering 범위다. 3단계는 실제 AWS 호출 없이 contract와 저장 흐름만 끝낸다. 4단계에서만 AWS SDK dependency와 read-only scanner를 붙인다.
