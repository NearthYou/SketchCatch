# 플랫폼/품질 구현 계획

## 문서 목적

이 문서는 `5번: 플랫폼/품질 -> ys` 담당 범위의 구현 방향을 고정하기 위한 문서다.

ys 파트는 서비스 기본 플랫폼과 프로젝트/템플릿 관리 기능을 맡는다.

담당 기능:

- 로그인
- 회원가입
- JWT 인증
- 내 프로젝트 목록
- 프로젝트 확인 보드
- 템플릿 등록/수정/삭제/공유/찜
- 활동 내역
- 팝업 알림

이번 문서에서 제외하는 기능:

- 배포 실행
- AWS 계정 연결
- Terraform Apply
- 배포 로그
- 생성 리소스 저장
- 비용 관리

## 현재 구현 가능성 검증

구현 가능하다고 판단한다.

근거는 아래와 같다.

| 근거 | 현재 상태 | ys 파트에서 활용할 방향 |
| --- | --- | --- |
| Fastify API | `apps/api`에 API 서버 구조가 있다. | 인증, 프로젝트, 템플릿, 활동 내역 route를 추가할 수 있다. |
| Drizzle schema | `apps/api/src/db/schema.ts`에 DB schema 관리 구조가 있다. | `users`, `oauth_accounts`, `activities`, `templates` 같은 테이블을 추가할 수 있다. |
| 기존 프로젝트 테이블 | `projects`, `architectures`, `project_assets`가 있다. | 내 프로젝트 목록, 프로젝트 확인 보드, 템플릿 등록의 기반으로 쓴다. |
| 아키텍처 JSON | `ArchitectureJson.nodes`, `ArchitectureJson.edges`가 공통 계약이다. | 보드, AI 분석, Terraform preview/export 연결 기준으로 쓴다. |
| S3 asset metadata | `project_assets`가 S3 object metadata를 저장한다. | 썸네일, Terraform 파일, export 파일 상태를 목록/보드에서 보여준다. |
| 공통 타입 패키지 | `packages/types`가 API와 web의 타입 기준이다. | API 응답과 프론트 상태 이름을 먼저 고정한다. |

주의할 점:

- 프론트엔드는 AWS SDK, Terraform 실행, 실제 인프라 변경 로직을 직접 호출하지 않는다.
- ys 파트는 배포 실행을 구현하지 않는다.
- 프로젝트 목록 API는 가볍게 유지하고 AI 상세 결과를 넣지 않는다.
- AI 결과는 프로젝트 확인 보드에서만 optional 영역으로 소비한다.
- 저장형 notification 시스템은 만들지 않고, 화면 warning과 Toast 중심으로 처리한다.

아래 자료는 구현 판단의 근거로 참고한다. 링크를 읽을 때는 전체를 다 구현하려고 보지 말고, ys 파트에 직접 필요한 인증, 권한, 프로젝트 소유권, 보드 저장, API validation 기준만 가져온다.

| Naver 로그인 | [Naver Login API](https://developers.naver.com/docs/login/api/api.md) | Naver OAuth의 `code`, `state`, token 발급/갱신/폐기 흐름을 확인하고 `oauth_accounts` 설계에 반영한다. |
| 권한 검증 | [OWASP Authorization Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Authorization_Cheat_Sheet.html), [OWASP IDOR Prevention Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Insecure_Direct_Object_Reference_Prevention_Cheat_Sheet.html) | 프로젝트는 `userId` 또는 `workspaceId`, 템플릿은 `ownerUserId` 기준으로 매 요청마다 소유권을 검증한다. | API validation | [Fastify Validation and Serialization](https://fastify.dev/docs/latest/Reference/Validation-and-Serialization/), [Zod](https://zod.dev/) | route handler에 들어오는 body/params/query를 Zod로 검증하고, route handler는 HTTP 입력 처리와 응답 shaping에 집중한다. |
| DB migration | [Drizzle ORM Migrations](https://orm.drizzle.team/docs/migrations) | `users`, `templates`, `activities` 같은 테이블 추가 시 schema와 migration을 함께 관리한다. |
| S3 asset upload | [Amazon S3 Presigned URL Upload](https://docs.aws.amazon.com/AmazonS3/latest/userguide/PresignedUrlUploadObject.html) | 썸네일, diagram image, Terraform export 파일은 S3에 저장하고 DB에는 `objectKey`와 metadata만 저장한다. |
| 보드 저장/복원 UI | [React Flow Save and Restore](https://reactflow.dev/examples/interaction/save-and-restore) | 설계 파트의 보드 상태를 `nodes`, `edges` JSON으로 저장/복원하는 화면 흐름을 참고한다. |
| Next.js 화면 구조 | [Next.js App Router Docs](https://nextjs.org/docs/app) | `/login`, `/signup`, `/projects`, `/projects/[id]`, `/templates`, `/activity` 같은 화면 라우팅 기준으로 참고한다. |

## 전체 개발 순서

### 1단계: 공통 타입 확정

위치:

- `packages/types/src/index.ts`

추가 또는 정리할 타입:

- `AuthProvider`
- `User`
- `TokenPair`
- `AuthSession`
- `LoginRequest`
- `LoginResponse`
- `ProjectListItem`
- `ProjectDashboardSummary`
- `AiAnalysisSummary`
- `AiFinding`
- `AiChecklistItem`
- `Template`
- `TemplateVisibility`
- `TemplateFavorite`
- `Activity`
- `ActivityEventName`

원칙:

- API와 프론트에서 쓰는 필드는 `camelCase`로 둔다.
- DB 컬럼은 `snake_case`로 둔다.
- shared type에는 아래 값을 절대 넣지 않는다.
  - 비밀번호
  - `passwordHash`
  - refresh token 원문
  - private token
  - DB 접속 정보
  - cloud secret 값

### 2단계: API DTO와 Zod validation 작성

위치:

- `apps/api/src/routes/auth.ts`
- `apps/api/src/routes/projects.ts`
- `apps/api/src/routes/dashboard.ts`
- `apps/api/src/routes/templates.ts`
- `apps/api/src/routes/activities.ts`

기준:

- 모든 request body, params, query는 Zod로 검증한다.
- 인증이 필요한 API는 JWT middleware를 거친다.
- 익명 프로젝트 접근은 `workspaceId` 기준으로만 허용한다.
- 로그인 프로젝트 접근은 `userId` 기준으로만 허용한다.
- 템플릿 수정/삭제는 작성자만 허용한다.
- 위험 작업은 Blocking modal 확인값을 request body에 포함해도, 최종 권한 검증은 API가 한다.

### 3단계: DB schema와 migration 작성

위치:

- `apps/api/src/db/schema.ts`
- `apps/api/drizzle`

추가 대상:

- `users`
- `oauth_accounts`
- `email_verifications`
- `refresh_tokens`
- `password_reset_tokens`
- `login_attempts`
- `projects.user_id`
- `templates`
- `template_versions`
- `template_favorites`
- `template_share_links`
- `activities`

제외 대상:

- `notifications`
- `deployments`
- `deployment_logs`
- `deployed_resources`
- `aws_credentials`
- 비용 관련 테이블
- AI 분석 전체 저장 테이블

### 4단계: API route 구현

구현 순서:

1. 인증 API
2. JWT middleware
3. 프로젝트 목록 API
4. 프로젝트 확인 보드 API
5. 템플릿 API
6. 활동 내역 API
7. 팝업 알림용 프론트 정책 연결

이 순서가 맞는 이유:

- 로그인/JWT가 먼저 있어야 `userId` 기준 소유권 검증을 할 수 있다.
- 프로젝트 목록과 보드는 템플릿 등록의 원천 데이터가 된다.
- 템플릿 등록/수정/삭제/찜은 사용자 권한 검증이 필요하다.
- 활동 내역은 위 기능들의 결과 이벤트를 기록하는 후행 기능이다.

### 5단계: web 화면 연결

위치:

- `apps/web/app/login/page.tsx`
- `apps/web/app/signup/page.tsx`
- `apps/web/app/projects/page.tsx`
- `apps/web/app/projects/[id]/page.tsx`
- `apps/web/app/templates/page.tsx`
- `apps/web/app/activity/page.tsx`
- `apps/web/components/notifications/ToastProvider.tsx`

화면 기준:

- 로그인 성공 후 `/projects`로 이동한다.
- 프로젝트 목록은 Brainboard처럼 카드 기반으로 보여주되, 프로젝트 단위로 구성한다.
- 프로젝트 카드에는 토글 상세보기를 둔다.
- 상세보기에서 `보드 열기`를 누르면 프로젝트 확인 보드로 이동한다.
- 프로젝트 확인 보드에서 AI summary, findings, checklist를 보여줄 자리를 만든다.
- 템플릿 기능은 등록/수정/삭제/공유/찜을 각각 독립된 action으로 나눈다.
- 팝업 알림은 API가 아니라 프론트 UI 정책으로 관리한다.

### 6단계: QA와 검증

필수 확인:

- 로그인 사용자는 자기 프로젝트만 조회할 수 있다.
- 익명 사용자는 자기 `workspaceId` 프로젝트만 조회할 수 있다.
- `GET /api/projects`에 AI 상세 결과가 들어가지 않는다.
- `GET /api/projects/:id/dashboard`에서만 AI optional 영역이 제공된다.
- 템플릿 수정/삭제는 작성자만 가능하다.
- 찜은 중복 생성되지 않는다.
- 활동 내역에는 중요한 이벤트만 기록된다.
- Toast, Warning, Blocking modal이 역할별로 분리된다.

문서 변경만 할 때는 full build가 필수는 아니다.

코드 구현까지 진행하면 아래를 확인한다.

```bash
pnpm lint
pnpm typecheck
pnpm build
```

## 기능별 구현 방향

## 1. 로그인

로그인은 단순 화면 진입 기능이 아니라 `userId` 기준 프로젝트, 템플릿, 활동 내역을 연결하는 시작점이다.

기능:

- 아이디/비밀번호 로그인
- Kakao OAuth 로그인
- Naver OAuth 로그인
- 자동 로그인 유지
- 로그아웃
- 모든 기기에서 로그아웃
- 비밀번호 찾기
- 아이디 찾기
- 로그인 실패 횟수 제한
- 로그인 성공 후 내 프로젝트 목록으로 이동
- 로그인 전 만든 익명 프로젝트를 내 계정으로 가져오기

구현 기준:

- access token은 짧게 유지한다.
- refresh token은 DB에 hash로 저장한다.
- refresh token 원문은 DB에 저장하지 않는다.
- 로그인 실패 횟수는 `login_attempts`에 저장한다.
- 5회 실패 시 아이디/비밀번호 찾기 안내를 보여준다.
- 과도한 시도는 일정 시간 제한한다.

에러 처리:

- 아이디 또는 비밀번호 오류는 실제 존재 여부를 과하게 노출하지 않는다.
- OAuth 이메일이 기존 일반 계정과 같으면 계정 연결을 제안한다.
- OAuth 계정은 있는데 일반 비밀번호가 없으면 소셜 로그인 또는 비밀번호 설정 흐름으로 안내한다.

## 2. 회원가입

회원가입은 계정 생성과 개인 프로젝트 공간 연결을 위한 기능이다.

기능:

- 아이디 중복 확인
- 비밀번호 조건 검사
- 비밀번호 확인
- 이메일 인증
- 이름 또는 닉네임 입력
- 약관 동의
- 개인정보 수집 동의
- 마케팅 수신 동의
- Kakao/Naver 회원가입
- 기존 이메일과 OAuth 이메일이 같으면 계정 연결 제안

구현 기준:

- 아이디는 중복 불가다.
- 이메일은 인증 완료 후 가입 가능하다.
- 비밀번호 정책은 서버에서 최종 검증한다.
- OAuth 가입 시 이메일이 없거나 검증되지 않았으면 추가 이메일 입력을 요구한다.
- 마지막 로그인 수단은 해제할 수 없게 한다.

## 3. JWT 인증

JWT는 로그인 유지뿐 아니라 프로젝트/템플릿 권한 검증 기준이다.

기능:

- access token 발급
- refresh token 발급
- token 재발급
- refresh token rotation
- 로그아웃 시 refresh token 폐기
- 모든 기기 로그아웃
- 사용자 role 확인
- 프로젝트 소유자 검증
- 템플릿 수정/삭제 권한 검증
- 위험 작업 전 재인증

재인증 대상:

- 프로젝트 영구 삭제
- 템플릿 삭제
- 템플릿 전체 공개
- 비밀번호 변경
- 계정 탈퇴

JWT에 넣지 않는 값:

- 비밀번호
- refresh token 원문
- DB 접속 정보
- private token
- cloud secret 값

## 4. 내 프로젝트 목록

프로젝트 목록은 빠르게 훑는 화면이다. 따라서 목록 API는 가볍게 유지한다.

목록 카드에 들어갈 정보:

- 프로젝트 썸네일
- 프로젝트 이름
- 짧은 설명
- 마지막 수정 시간
- 생성일
- 상태
- 리소스 개수
- 마지막 아키텍처 버전
- Terraform export 여부
- 상세보기 토글

토글 상세에 들어갈 정보:

- 프로젝트 설명 전체
- 포함된 주요 AWS 리소스
- 최근 활동 2~3개
- 최근 아키텍처 버전
- Terraform export 여부
- 연결된 템플릿 여부
- 보드 열기
- Terraform 미리보기
- 템플릿으로 등록
- 프로젝트 복제

넣지 않는 정보:

- AI finding 상세
- AI checklist 전체
- 저장형 알림 목록
- 배포 실행 정보

## 5. 프로젝트 확인 보드

프로젝트 확인 보드는 ys 파트에서 프로젝트 상태를 통합해 보여주는 화면이다.

기능:

- 아키텍처 노드/엣지 확인
- 리소스 설정 확인
- AI summary 표시
- AI finding 목록 표시
- checklist 표시
- Terraform 미리보기 진입
- Terraform export artifact 상태 표시
- 아키텍처 버전 기록 표시
- PNG/SVG 내보내기 asset 표시
- 위험 경고 확인

기준 데이터:

- `ArchitectureJson.nodes`
- `ArchitectureJson.edges`
- `project_assets`
- gg AI 파트의 optional 분석 응답

주의:

- AI summary는 프로젝트 목록에 넣지 않는다.
- 프로젝트 확인 보드에서만 AI 결과를 보여준다.
- AI 결과는 사용자가 검토할 수 있는 형태로 보여준다.
- AI 결과가 바로 프로젝트 상태를 바꾸면 안 된다.

## 6. 템플릿

템플릿은 프로젝트를 재사용 가능한 구조로 저장하고 공유하는 기능이다.

### 템플릿 등록

기능:

- 내 프로젝트를 템플릿으로 등록
- 제목 입력
- 설명 입력
- 카테고리 선택
- 난이도 선택
- 포함 리소스 자동 계산
- 썸네일 연결
- 공개/비공개 선택
- 등록 전 위험 검토 상태 확인

### 템플릿 수정

기능:

- 제목 수정
- 설명 수정
- 카테고리 수정
- 난이도 수정
- 아키텍처 내용 수정
- 공개 상태 변경
- 새 버전 저장
- 수정 이력 기록

원칙:

- 이미 다른 사용자가 복제한 프로젝트에는 영향을 주지 않는다.
- 템플릿 수정은 새 버전으로 저장한다.

### 템플릿 삭제

기능:

- 작성자만 삭제 가능
- 삭제 전 사용 횟수 표시
- 삭제 전 찜 수 표시
- 공개 템플릿은 비공개 전환을 먼저 권장
- Blocking modal로 최종 확인

### 템플릿 공유

기능:

- 비공개
- 링크 공유
- 전체 공개
- 공유 링크 복사
- 공개 전 위험 검토
- 신고 기능은 후순위
- 관리자 숨김 처리는 후순위

### 템플릿 찜

기능:

- 찜하기
- 찜 취소
- 찜한 템플릿 목록
- 찜한 템플릿에서 프로젝트 만들기
- 카테고리별 필터

알림:

- 찜 완료는 Toast로 충분하다.
- 찜 취소도 Toast로 충분하다.

## 7. 활동 내역

활동 내역은 사용자가 중요한 작업을 되돌아볼 수 있는 감사 로그다.

기록할 이벤트:

- 프로젝트 생성
- 프로젝트 수정
- 프로젝트 삭제
- 아키텍처 저장
- Terraform export
- 템플릿 등록
- 템플릿 수정
- 템플릿 공개
- 템플릿 삭제
- AI 아키텍처 초안 생성
- AI 아키텍처 검토 완료
- AI 아키텍처 검토 실패

기록하지 않을 이벤트:

- 리소스 설명 1회 조회
- 임시 UI 클릭
- Toast 표시 자체
- 모든 AI 요청 원문
- 민감값이 포함될 수 있는 payload

## 8. 팝업 알림

저장형 알림 시스템은 MVP에서 만들지 않는다.

알림은 아래 4단계로 나눈다.

| 단계 | 용도 | 예시 |
| --- | --- | --- |
| Toast | 가벼운 완료 알림 | 저장 완료, 찜 완료, 복사 완료, AI 분석 완료 |
| 화면 내 Warning | 화면에 계속 남아야 하는 주의 | 보안 위험, 설정 누락, AI 분석 실패, 검토 필요 |
| Warning popup | 사용자가 지나치면 손해가 생기는 상황 | public 접근 위험, 과도한 권한, 삭제 예정 리소스 있음, 템플릿 공개 전 주의 |
| Blocking modal | 되돌리기 어렵거나 권한 확인이 필요한 작업 | 프로젝트 영구 삭제, 템플릿 삭제, 템플릿 전체 공개, 위험한 Terraform export |

원칙:

- 저장 성공 같은 가벼운 이벤트는 Toast로 끝낸다.
- 보안/설정 위험은 화면 내 Warning으로 남긴다.
- 삭제나 공개처럼 되돌리기 어려운 작업은 Blocking modal을 사용한다.
- 팝업 알림을 DB에 저장하지 않는다.

## 다른 파트와의 의존성

| 담당 | 사람 | ys 파트가 필요한 것 | ys 파트가 제공할 것 |
| --- | --- | --- | --- |
| 1번 설계 | jh | `ArchitectureJson.nodes/edges`, 리소스 설정 구조 | 프로젝트 저장/조회 API, 프로젝트 확인 보드 진입점 |
| 2번 Terraform 변환 | sw | Terraform preview/export 상태, artifact metadata | 프로젝트/아키텍처/asset 조회 기준, Terraform export 상태 표시 영역 |
| 3번 배포 실행 | ck | 이번 범위에서는 제외 | 이번 범위에서는 제외 |
| 4번 AI 분석 | gg | `AiAnalysisSummary`, `AiFinding`, `AiChecklistItem`, 중요 AI event 이름 | dashboard의 AI 표시 영역, 활동 내역 저장 기준 |
| 5번 플랫폼/품질 | ys(나) | 전체 플랫폼 흐름 관리 | 인증, 프로젝트 목록, 템플릿, 활동 내역, 알림 UI 정책 |

## 우선순위

1. 로그인/회원가입/JWT
2. 내 프로젝트 목록
3. 프로젝트 확인 보드
4. 템플릿 등록/수정/삭제
5. 템플릿 공유/찜
6. 활동 내역
7. 팝업 알림 polish

## 완료 기준

- 사용자가 회원가입하고 로그인할 수 있다.
- 로그인 후 내 프로젝트 목록을 볼 수 있다.
- 각 프로젝트 카드의 상세보기를 열 수 있다.
- 상세보기에서 프로젝트 확인 보드로 들어갈 수 있다.
- 프로젝트를 템플릿으로 등록할 수 있다.
- 템플릿을 수정/삭제/공유/찜할 수 있다.
- 활동 내역에서 중요한 이벤트를 확인할 수 있다.
- 저장/찜/복사 완료는 Toast로 표시된다.
- 삭제/공개 같은 위험 작업은 Blocking modal로 확인한다.
