# 배포 파트 구현 순서

## 0. 이 문서의 책임

이 문서는 CK 배포 파트를 실제 코드로 옮길 때의 **작업 순서**만 정리한다.

구현 범위와 계약은 [004_배포파트결정사항초안.md](./004_배포파트결정사항초안.md)를 기준으로 하고, 진행 상태는 [001_배포실행파트구현계획.md](./001_배포실행파트구현계획.md)에 체크한다. MVP 이후 실제 Terraform/AWS 실행 순서는 [006_실제배포실행구현순서.md](./006_실제배포실행구현순서.md)를 따른다.

## 1. 구현 전 확인

1. [004_배포파트결정사항초안.md](./004_배포파트결정사항초안.md)에서 MVP 범위와 타입/API/DB 계약을 확인한다.
2. [003_배포파트의존성정리.md](./003_배포파트의존성정리.md)에서 SW, JH, GG, YS와 맞춰야 하는 입력을 확인한다.
3. `docs/data-models.md`에서 공통 모델 이름을 확인한다.
4. 현재 branch와 작업 트리 상태를 확인한다.

## 2. 1차 작업: 공유 타입과 데이터 모델 문서

### 수정 위치

- `packages/types/src/index.ts`
- `docs/data-models.md`

### 작업 순서

1. `DeploymentStage`를 추가한다.
2. `DeploymentFailureStage`를 추가한다.
3. `DeploymentPlanSummary`를 추가한다.
4. `DeploymentLogLine` 또는 확장된 `DeploymentLog`를 추가한다.
5. 기존 `Deployment`에 004에서 결정한 필드를 반영한다.
6. `docs/data-models.md`를 같은 필드명으로 갱신한다.

### 확인 기준

- 공유 타입에 secret, raw AWS key, DB password가 없다.
- `DeploymentStatus`는 기존 다섯 상태를 유지한다.
- blocked 정보는 status가 아니라 별도 필드로 표현된다.

## 3. 2차 작업: DB schema와 migration

### 수정 위치

- `apps/api/src/db/schema.ts`
- `apps/api/drizzle/*`

### 작업 순서

1. `deployment_status` enum이 필요한지 확인한다.
2. `deployments` 테이블을 추가한다.
3. `deployment_logs` 테이블을 추가한다.
4. `projects`, `architectures`, `project_assets`와 relation을 연결한다.
5. Drizzle migration을 생성한다.
6. 생성된 SQL이 예상 범위인지 검토한다.

### 확인 기준

- Terraform 원문은 RDS에 저장하지 않는다.
- `planSummary`는 구조화된 JSON으로 저장할 수 있다.
- `deployment_logs`는 `sequence` 기준 조회가 가능하다.

## 4. 3차 작업: API route와 validation

### 수정 위치

- `apps/api/src/routes/deployments.ts`
- `apps/api/src/app.ts`
- 필요 시 `apps/api/src/deployments/*`

### 작업 순서

1. route 파일을 추가한다.
2. project route와 deployment route prefix를 연결한다.
3. params schema를 만든다.
4. create deployment body schema를 만든다.
5. approve body schema를 만든다.
6. project, workspace, architecture, artifact 소유 관계를 검증한다.
7. deployment 생성, 목록, 상세, 승인, 로그 조회 handler를 구현한다.

### 확인 기준

- route handler가 너무 길어지면 service/helper로 분리한다.
- API 응답 필드는 공유 타입 의미와 맞춘다.
- 실제 Terraform CLI나 AWS SDK를 호출하지 않는다.

## 5. 4차 작업: mock 상태 전이와 로그

### 수정 위치

- `apps/api/src/deployments/mock-deployment.ts`
- `apps/api/src/deployments/deployment-service.ts`
- 또는 초기에는 `apps/api/src/routes/deployments.ts`

### 작업 순서

1. deployment 생성 시 validation 실패 케이스를 먼저 처리한다.
2. blocked 입력이 있으면 `isBlocked`와 reason을 저장한다.
3. 정상 입력이면 `PENDING` 상태의 deployment를 만든다.
4. mock log line을 생성한다.
5. approve API에서 승인 가능 조건을 확인한다.
6. 승인 성공 시 승인 시각과 승인 대상을 저장한다.
7. mock 실행 완료 상태를 저장한다.

### 확인 기준

- 실패를 성공처럼 저장하지 않는다.
- blocked deployment는 승인되지 않는다.
- 로그 메시지는 저장 전에 마스킹한다.

## 6. 5차 작업: API 테스트

### 수정 위치

- `apps/api/src/**/*.test.ts`

### 테스트 순서

1. 정상 deployment 생성 테스트
2. 없는 project 실패 테스트
3. workspace 불일치 실패 테스트
4. 없는 architecture 실패 테스트
5. 없는 Terraform artifact 실패 테스트
6. artifact type 불일치 실패 테스트
7. blocked deployment 승인 차단 테스트
8. 정상 승인 성공 테스트
9. log sequence 조회 테스트

## 7. 6차 작업: 프론트엔드 연결

### 수정 위치

- `apps/web/app/workspace/page.tsx`
- 필요 시 `apps/web/app/workspace/*`
- 필요 시 `apps/web/app/globals.css`

### 작업 순서

1. 배포 패널을 placeholder에서 실제 상태 표시 영역으로 바꾼다.
2. 배포 이력 목록을 표시한다.
3. plan summary를 표시한다.
4. blocked 사유를 표시한다.
5. 승인 가능 상태와 승인 완료 상태를 표시한다.
6. 로그 목록을 표시한다.
7. loading, empty, error 상태를 정리한다.

### 확인 기준

- 버튼 문구가 실제 AWS 배포로 오해되지 않는다.
- 프론트엔드에서 AWS SDK나 Terraform CLI를 호출하지 않는다.
- 배포 실행 책임은 API 뒤에 남겨둔다.

## 8. 최종 검증

코드 변경 후 아래 명령을 실행한다.

```bash
pnpm lint
pnpm typecheck
pnpm build
```

로컬에 `pnpm`이 없으면 아래 방식을 쓴다.

```bash
npm exec --package=pnpm@11.8.0 -- pnpm lint
npm exec --package=pnpm@11.8.0 -- pnpm typecheck
npm exec --package=pnpm@11.8.0 -- pnpm build
```
