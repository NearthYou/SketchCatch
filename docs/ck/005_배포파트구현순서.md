# 배포 실행 파트 구현 순서

## 0. 이 문서의 책임

이 문서는 CK 배포 실행 파트를 실제 코드로 옮길 때의 **작업 순서**만 정리한다.

구현 범위와 계약은 [배포 실행 파트 결정사항 원천](./004_배포파트결정사항초안.md)을 기준으로 하고, 진행 상태는 [배포 실행 파트 작업 체크리스트](./001_배포실행파트구현계획.md)에 체크한다. Terraform 실행과 apply/result extraction 상세는 [실제 배포 실행 상세 구현 순서](./006_실제배포실행구현순서.md)를 따른다.

## 1. 구현 전 확인

1. [배포 실행 파트 결정사항 원천](./004_배포파트결정사항초안.md)에서 구현 범위와 타입/API/DB 계약을 확인한다.
2. [배포 실행 파트 입출력 의존성](./003_배포파트의존성정리.md)에서 SW, JH, GG, YS와 맞춰야 하는 입력을 확인한다.
3. `docs/data-models.md`에서 공통 모델 이름을 확인한다.
4. 현재 branch와 작업 트리 상태를 확인한다.

## 2. 1차 작업: 공유 타입과 데이터 모델 문서

### 수정 위치

- `packages/types/src/index.ts`
- `docs/data-models.md`

### 작업 순서

1. `DeploymentStage`를 추가한다.
2. `DeploymentFailureStage`를 추가한다.
3. `DeploymentPlanSummary`를 추가한다.
4. `DeploymentLogLine` 또는 확장된 `DeploymentLog`를 추가한다.
5. Role Assume 기반 `AwsConnection`, `CreateAwsConnectionRequest`, `CreateAwsConnectionResponse`를 추가한다.
6. `DeployedResource`를 추가한다.
7. `TerraformOutput`을 추가한다.
8. 기존 `Deployment`에 004에서 결정한 필드를 반영한다.
9. `docs/data-models.md`를 같은 필드명으로 갱신한다.

### 확인 기준

- 공유 타입에 secret, raw AWS key, DB password가 없다.
- `DeploymentStatus`는 기존 다섯 상태를 유지한다.
- blocked 정보는 status가 아니라 별도 필드로 표현된다.

## 3. 2차 작업: DB schema와 migration

### 수정 위치

- `apps/api/src/db/schema.ts`
- `apps/api/drizzle/*`

### 작업 순서

1. `deployment_status` enum이 필요한지 확인한다.
2. `deployments` 테이블을 추가한다.
3. `deployment_logs` 테이블을 추가한다.
4. `deployed_resources` 테이블을 추가한다.
5. `terraform_outputs` 테이블을 추가한다.
6. Role Assume 연결 metadata 저장소로 `aws_connections` 테이블을 추가한다.
7. project lock이 필요하면 lock 테이블 또는 필드를 추가한다.
8. `projects`, `architectures`, `project_assets`와 relation을 연결한다.
9. Drizzle migration을 생성한다.
10. 생성된 SQL이 예상 범위인지 검토한다.

### 확인 기준

- Terraform 원문은 RDS에 저장하지 않는다.
- `planSummary`는 구조화된 JSON으로 저장할 수 있다.
- `deployment_logs`는 `sequence` 기준 조회가 가능하다.
- `tfplan`, state 같은 파일성 산출물은 S3 object metadata로 참조한다.

## 4. 3차 작업: API route와 validation

### 수정 위치

- `apps/api/src/routes/deployments.ts`
- `apps/api/src/app.ts`
- 필요 시 `apps/api/src/deployments/*`

### 작업 순서

1. route 파일을 추가한다.
2. project route와 deployment route prefix를 연결한다.
3. params schema를 만든다.
4. create deployment body schema를 만든다.
5. AWS connection setup body schema를 만든다. 현재 생성 API는 `region`만 받는다.
6. 다음 단계의 AWS connection verify body schema를 만든다. 검증 API는 사용자 `roleArn`을 받는다.
7. approve body schema를 만든다.
8. apply body schema를 만든다.
9. project, workspace, architecture, artifact 소유 관계를 검증한다.
10. deployment 생성, 목록, 상세, 승인, apply, 로그, 리소스, output 조회 handler를 구현한다.

### 확인 기준

- route handler가 너무 길어지면 service/helper로 분리한다.
- API 응답 필드는 공유 타입 의미와 맞춘다.
- 이 단계에서는 API 계약과 소유 관계 검증을 먼저 만들고, 실제 Terraform 실행은 service/helper로 분리한다.

## 5. 4차 작업: AWS Role 연결 설정과 확인

### 수정 위치

- `apps/api/src/aws-connections/aws-connection-service.ts`
- `apps/api/src/routes/aws-connections.ts`
- `apps/api/src/config/env.ts`
- `apps/web/features/workspace/api.ts`
- 다음 단계에서 필요 시 `apps/api/src/aws-connections/aws-connection-verification-service.ts`

### 작업 순서

1. AWS 연결 방식은 Role Assume + 서버 생성 External ID로 고정한다.
2. `SKETCHCATCH_AWS_CALLER_PRINCIPAL_ARN` 환경 변수를 runtime config에 추가한다.
3. `POST /api/projects/:projectId/aws-connections`에서 project 소유권을 확인한다.
4. connection 단위 `externalId`를 생성하고 `aws_connections`에 `status = "pending"`으로 저장한다.
5. 응답으로 `callerPrincipalArn`, `externalId`, `recommendedRoleName`, `trustPolicyTemplate`을 내려준다.
6. 프론트 API helper는 이 응답을 받아 UI가 Role trust policy 안내에 사용할 수 있게 한다.
7. 다음 단계에서 사용자 `roleArn`을 받는 verify API를 추가한다.
8. verify API는 저장된 `externalId`로 `sts:AssumeRole`을 호출한다.
9. `GetCallerIdentity` 성공 시 `accountId`, `roleArn`, `lastVerifiedAt`, `status = "verified"`를 저장한다.
10. 실패 시 마스킹된 오류 메시지를 반환한다.

### 확인 기준

- raw Access Key, Secret Key, session token을 DB/API/log에 남기지 않는다.
- 프론트엔드에서 AWS SDK를 호출하지 않는다.
- pending setup API는 실제 AWS 리소스를 만들거나 Terraform을 실행하지 않는다.
- AWS 연결 확인 실패는 `failureStage = connection`으로 구분할 수 있다.

## 6. 5차 작업: Terraform init / validate / plan

### 수정 위치

- `apps/api/src/deployments/artifact-restore.ts`
- `apps/api/src/deployments/terraform-runner.ts`
- `apps/api/src/deployments/terraform-plan-parser.ts`
- `apps/api/src/deployments/log-masking.ts`
- `apps/api/src/routes/deployments.ts`

### 작업 순서

1. `POST /api/deployments/:deploymentId/plan` handler를 추가한다.
2. deployment가 존재하고 `PENDING` 또는 재시도 가능한 실패 상태인지 확인한다.
3. `awsConnectionId`가 `verified` 상태인지 확인한다.
4. deployment에 연결된 `terraformArtifactId`와 S3 `objectKey`를 조회한다.
5. deployment별 격리 작업 디렉터리를 만든다.
6. S3 artifact를 작업 디렉터리에 복원한다.
7. Terraform 명령 실행 helper를 만든다. shell string이 아니라 argv 배열로 실행한다.
8. `terraform init`을 실행하고 stdout/stderr를 마스킹해 `deployment_logs`에 저장한다.
9. `terraform validate`를 실행하고 실패 시 `FAILED`, `failureStage = validate`로 저장한다.
10. `terraform plan -out=tfplan`을 실행하고 실패 시 `FAILED`, `failureStage = plan`으로 저장한다.
11. 생성된 `tfplan` hash를 계산하고 S3에 저장한다.
12. `terraform show -json tfplan`을 실행한다.
13. `resource_changes`를 파싱해 `DeploymentPlanSummary`를 만든다.
14. plan 성공 시 `planSummary`, `tfplanObjectKey`, `tfplanHash`, `isBlocked`, `blockedReason`, `updatedAt`을 저장한다.

### 확인 기준

- 실패를 성공처럼 저장하지 않는다.
- blocked deployment는 승인되지 않는다.
- 로그 메시지는 저장 전에 마스킹한다.
- Terraform 원문, raw credential, 작업 디렉터리 경로를 API 응답으로 내려주지 않는다.
- plan 실패 시 approve/apply 단계로 넘어가지 않는다.

## 7. 6차 작업: Plan 표시와 승인

### 수정 위치

- `apps/api/src/routes/deployments.ts`
- `apps/api/src/deployments/deployment-service.ts`
- `apps/web/app/workspace/*`

### 작업 순서

1. plan summary 조회 응답에 create/update/delete/replace count를 포함한다.
2. blocked 결과와 warnings를 함께 응답한다.
3. approve API에서 blocked 여부를 확인한다.
4. approve API에서 사용자가 확인한 `terraformArtifactId`, `artifactHash`, `tfplanHash`를 저장한다.
5. 승인 후 artifact, tfplan, AWS 연결이 바뀌면 apply를 막는다.

### 확인 기준

- 사용자가 무엇을 보고 승인했는지 DB에 남는다.
- blocked 상태에서는 승인 또는 apply가 불가능하다.
- 승인 없는 apply 요청은 실패한다.

## 8. 7차 작업: Terraform apply

### 수정 위치

- `apps/api/src/deployments/terraform-apply.ts`
- `apps/api/src/deployments/terraform-runner.ts`
- `apps/api/src/deployments/log-masking.ts`
- `apps/api/src/routes/deployments.ts`

### 작업 순서

1. `POST /api/deployments/:deploymentId/apply` handler를 추가한다.
2. 승인 기록, blocked 여부, project lock을 검증한다.
3. 승인된 `tfplanHash`와 S3의 `tfplan` hash가 같은지 검증한다.
4. apply 실행 전 status를 `RUNNING`으로 바꾼다.
5. `terraform apply tfplan`을 실행한다.
6. stdout/stderr를 마스킹해 `deployment_logs`에 저장한다.
7. 성공 시 status를 `SUCCESS`, 실패 시 `FAILED`로 저장한다.
8. 실패 시 `failureStage = apply`, `errorSummary`를 저장한다.

### 확인 기준

- 임의 `terraform apply`가 아니라 승인된 `tfplan`만 apply한다.
- 같은 프로젝트에서 동시에 apply가 돌지 않는다.
- apply 로그에 secret이 남지 않는다.

## 9. 8차 작업: 생성 리소스와 output 저장

### 수정 위치

- `apps/api/src/deployments/terraform-result-extractor.ts`
- `apps/api/src/routes/deployments.ts`
- `apps/api/src/db/schema.ts`

### 작업 순서

1. apply 성공 후 `terraform show -json` 또는 state JSON을 읽는다.
2. 생성 리소스의 Terraform address, type, provider, AWS resource id, region을 추출한다.
3. `deployed_resources`에 저장한다.
4. `terraform output -json`을 실행한다.
5. output 이름, 값, sensitive 여부를 파싱한다.
6. `terraform_outputs`에 저장한다.
7. sensitive output은 API와 화면에서 마스킹한다.

### 확인 기준

- 프로젝트 상세에서 생성 리소스 목록을 볼 수 있다.
- 배포 결과 화면에서 Terraform output을 볼 수 있다.
- sensitive output은 raw value가 노출되지 않는다.

## 11. 10차 작업: 프론트엔드 연결

### 수정 위치

- `apps/web/app/workspace/page.tsx`
- 필요 시 `apps/web/app/workspace/*`
- 필요 시 `apps/web/app/globals.css`

### 작업 순서

1. 배포 패널을 placeholder에서 실제 상태 표시 영역으로 바꾼다.
2. 배포 이력 목록을 표시한다.
3. AWS Role 설정값과 연결 확인 결과를 표시한다.
4. plan summary를 표시한다.
5. blocked 사유를 표시한다.
6. 승인 가능 상태와 승인 완료 상태를 표시한다.
7. apply 실행 상태와 성공/실패 상태를 표시한다.
8. 로그 목록을 표시한다.
9. 생성 리소스 목록을 표시한다.
10. Terraform output을 표시한다.
11. loading, empty, error 상태를 정리한다.

### 확인 기준

- apply 버튼은 실제 AWS 리소스를 생성/수정/삭제할 수 있음을 명확히 알려준다.
- 프론트엔드에서 AWS SDK나 Terraform CLI를 호출하지 않는다.
- 배포 실행 책임은 API 뒤에 남겨둔다.

## 12. 최종 검증

코드 변경 후 아래 명령을 실행한다.

```bash
pnpm lint
pnpm typecheck
pnpm build
```

로컬에 `pnpm`이 없으면 아래 방식을 쓴다.

```bash
npm exec --package=pnpm@11.8.0 -- pnpm lint
npm exec --package=pnpm@11.8.0 -- pnpm typecheck
npm exec --package=pnpm@11.8.0 -- pnpm build
```
