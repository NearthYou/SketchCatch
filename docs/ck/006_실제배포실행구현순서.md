# 실제 배포 실행 상세 구현 순서

## 0. 이 문서의 책임

이 문서는 AWS Role 연결 설정/확인부터 `terraform apply`, 로그 저장, 생성 리소스/output 저장까지 실제 배포 실행을 구현할 때의 **상세 순서**만 정리한다.

작업 단위와 PR 순서는 [배포 실행 파트 구현 순서](./005_배포파트구현순서.md)를 따른다. 이 문서는 각 단계에서 어떤 검증, 명령, 저장, 실패 처리가 필요한지 더 자세히 적는다.

## 1. 시작 조건

아래 조건이 갖춰지기 전에는 실제 apply를 열지 않는다.

- `Deployment`, `DeploymentLog`, plan summary, approval, blocked 기록이 구현되어 있다.
- Terraform artifact가 S3 metadata와 `objectKey` 기준으로 저장되어 있다.
- 사용자가 본 artifact와 승인 대상 artifact를 비교할 수 있다.
- 사용자가 본 `tfplan`과 apply 대상 `tfplan`을 비교할 수 있다.
- 위험/비용 분석 결과가 blocked인지 판단할 수 있다.
- 로그 마스킹 helper가 있다.
- 실제 AWS 리소스 생성/수정/삭제가 발생할 수 있음을 사용자에게 명확히 알리는 UI가 있다.
- 실습용 AWS 계정 또는 제한된 sandbox 계정에 SketchCatch용 IAM Role을 만들 수 있다.

## 2. 전체 구현 순서

구현 흐름은 아래 순서를 따른다.

```text
1. AWS Role 연결 설정값 발급
2. AWS Role 연결 확인
3. terraform init
4. terraform validate
5. terraform plan
6. Plan 결과 표시
7. 사용자 승인
8. terraform apply
9. 로그 표시
10. 성공/실패 저장
11. 생성 리소스/output 저장
```

## 3. 0단계: 로컬 Terraform 실습

### 목적

제품 코드에 넣기 전에 Terraform 명령의 입력, 출력, 생성 파일, 실패 메시지를 직접 이해한다.

### 작업

1. 개인 실습용 AWS 계정 또는 sandbox 계정을 준비한다.
2. SketchCatch가 assume할 제한된 IAM Role을 만들고 trust policy에 `External ID` 조건을 넣는 흐름을 직접 확인한다.
3. S3 bucket 하나만 만드는 Terraform 예제를 만든다.
4. 로컬 터미널에서 아래 순서를 직접 실행한다.

```bash
terraform init
terraform validate
terraform plan -out=tfplan
terraform show -json tfplan
terraform apply tfplan
terraform output -json
terraform destroy
```

### 완료 기준

- `.terraform`, `.terraform.lock.hcl`, `tfplan`, `terraform.tfstate`가 언제 생기는지 안다.
- `stdout`, `stderr`, exit code를 확인했다.
- 실패 메시지 예시를 최소 2개 이상 확보했다.
- 이 단계의 AWS key, 임시 credential, state 파일을 저장소에 커밋하지 않았다.

## 4. 1단계: 실제 실행용 모델 확장

### 목적

배포 기록을 실제 Terraform 실행 job으로 확장할 수 있게 한다.

### 추가 검토 모델

| 모델 | 목적 |
| --- | --- |
| `AwsConnection` | 실제 AWS 연결 대상 계정과 role 정보를 기록 |
| `DeploymentLock` | 같은 프로젝트 동시 실행 차단 |
| `DeployedResource` | apply 성공 후 생성 리소스 목록 저장 |
| `TerraformOutput` | `terraform output -json` 결과 저장 |

### `Deployment` 확장 후보

- `executionMode`: `plan | apply | destroy`
- `awsConnectionId`
- `tfplanObjectKey`
- `tfplanHash`
- `workingDirectoryId`
- `cancelRequestedAt`
- `timedOutAt`
- `recoveredAt`

### 완료 기준

- 실제 실행 필드가 기존 deployment 응답과 충돌하지 않는다.
- raw AWS credential은 공유 타입과 API 응답에 포함되지 않는다.
- state, tfplan, output 같은 파일성 산출물은 S3 object metadata로 참조한다.

## 5. 2단계: AWS Role 연결 설정/확인 API

### 목적

Terraform 실행 전에 사용자가 만든 IAM Role을 SketchCatch가 안전하게 assume할 수 있는지 확인한다.

### API

| Method | Path | 목적 |
| --- | --- | --- |
| `POST` | `/api/projects/:projectId/aws-connections` | 구현됨. pending Role 연결 생성, `External ID`와 trust policy template 발급 |
| `POST` | `/api/projects/:projectId/aws-connections/:connectionId/verify` | 다음 단계. 사용자 `roleArn` 검증 |

### 구현 기준

- 장기 보관 Access Key / Secret Key 저장은 기본값으로 두지 않는다.
- Role Assume + SketchCatch 생성 `External ID`를 기본 경로로 둔다.
- `POST /api/projects/:projectId/aws-connections`는 실제 AWS를 호출하지 않고 `status = "pending"` 연결을 만든다.
- pending 생성 응답에는 `callerPrincipalArn`, `externalId`, `recommendedRoleName`, `trustPolicyTemplate`을 내려준다.
- 연결 검증은 STS `AssumeRole`에 저장된 `externalId`를 넣고, 이어서 `GetCallerIdentity`를 호출한다.
- 검증 성공 응답에는 `accountId`, `callerArn`, `roleArn`, `region`, 성공/실패 사유만 내려준다.
- Secret Key, session token, raw credential은 응답과 로그에 남기지 않는다.

### 완료 기준

- pending 연결 생성 시 사용자가 IAM Role trust policy에 넣을 값을 확인할 수 있다.
- 검증 성공 시 AWS account와 caller ARN을 확인할 수 있다.
- 실패 시 `connection` failure stage로 저장할 수 있다.
- 로그에 credential이 남지 않는다.

## 6. 3단계: 작업 디렉터리 준비와 artifact 복원

### 목적

백엔드 또는 worker에서 Terraform을 실행할 격리된 작업 디렉터리를 만든다.

### 작업

1. `Deployment` 기준 임시 작업 디렉터리를 만든다.
2. S3 `TerraformArtifact.objectKey`에서 Terraform 파일을 가져온다.
3. 단일 `.tf` 파일 또는 zip bundle을 작업 디렉터리에 배치한다.
4. 실행 전 파일 hash를 계산해 `terraformArtifactId`와 승인 기준이 맞는지 확인한다.
5. 작업 디렉터리 경로는 API 응답에 노출하지 않는다.

### 완료 기준

- 프로젝트별 작업 파일이 섞이지 않는다.
- path traversal을 막는다.
- 작업 디렉터리는 실행 완료 또는 실패 후 정리할 수 있다.
- Terraform 원문을 RDS에 저장하지 않는다.

## 7. 4단계: `terraform init`과 `terraform validate`

### 목적

Terraform provider/backend 초기화와 문법 검증을 실제로 수행한다.

### 실행 기준

- 실행 위치는 API 서버 또는 worker다.
- 프론트엔드는 Terraform CLI를 호출하지 않는다.
- 명령은 shell string 조합보다 명시적 argv 방식으로 실행한다.
- stdout/stderr는 라인 단위로 수집하고 마스킹한다.

### 상태 처리

| 명령 | 성공 시 | 실패 시 |
| --- | --- | --- |
| `terraform init` | 다음 단계 진행 | `failureStage = init` |
| `terraform validate` | plan 단계 진행 | `failureStage = validate` |

### 완료 기준

- exit code를 기준으로 성공/실패를 판단한다.
- 실패 메시지를 `Deployment.errorSummary`로 요약한다.
- 로그는 `DeploymentLog`에 순서대로 저장된다.

## 8. 5단계: `terraform plan -out=tfplan`

### 목적

사용자가 승인할 정확한 실행 계획을 파일로 고정한다.

### 작업

1. `terraform plan -out=tfplan`을 실행한다.
2. 생성된 `tfplan` 파일 hash를 계산한다.
3. `tfplan` 파일을 S3에 저장한다.
4. RDS에는 `tfplanObjectKey`, `tfplanHash`, metadata만 저장한다.
5. plan 생성 시점의 `terraformArtifactId`, `artifactHash`, `awsConnectionId`, `roleArn`, `accountId`, `region`을 함께 기록한다.

### 완료 기준

- plan 실패 시 apply 단계로 넘어가지 않는다.
- 사용자가 승인한 plan과 apply 대상 plan을 비교할 수 있다.
- `tfplan` 파일은 Git과 RDS 원문 컬럼에 저장하지 않는다.

## 9. 6단계: `terraform show -json tfplan` 파싱

### 목적

Terraform plan을 UI와 AI 분석이 사용할 수 있는 구조로 바꾼다.

### 작업

1. `terraform show -json tfplan`을 실행한다.
2. `resource_changes`를 파싱한다.
3. `create`, `update`, `delete`, `replace` count를 만든다.
4. Terraform resource address와 `ResourceNode.id`를 매핑한다.
5. 삭제, 교체, public access 같은 위험 신호를 표시한다.
6. 기존 `DeploymentPlanSummary` 구조로 저장한다.

### 완료 기준

- plan summary가 UI에서 표시 가능하다.
- GG 비용/위험 분석이 재사용할 수 있는 형태다.
- 파싱 실패 시 `failureStage = plan`으로 저장한다.

## 10. 7단계: Plan 표시와 승인 고정

### 목적

사용자가 실제로 무엇을 보고 승인했는지 증거를 남긴다.

### 작업

1. plan summary를 API로 제공한다.
2. blocked 결과와 warnings를 함께 제공한다.
3. 사용자가 승인하면 `approvedAt`, `approvedBy`, `approvedTerraformArtifactId`, `approvedArtifactHash`, `approvedTfplanHash`를 저장한다.
4. 승인 후 artifact, tfplan, AWS 연결이 바뀌면 기존 승인을 무효로 본다.

### 완료 기준

- blocked 상태에서는 apply 승인 또는 실행이 차단된다.
- 승인 API는 사용자가 본 plan 기준을 저장한다.
- 승인 없이 apply API를 호출하면 실패한다.

## 11. 8단계: 승인된 `tfplan`만 apply

### 목적

사용자가 승인한 정확한 plan만 실제 AWS에 반영한다.

### API 후보

| Method | Path | 목적 |
| --- | --- | --- |
| `POST` | `/api/deployments/:deploymentId/apply` | 승인된 `tfplan` 적용 |

### 실행 전 검증

- deployment가 blocked가 아닌지
- 승인 기록이 있는지
- 승인한 `terraformArtifactId`와 현재 artifact가 같은지
- 승인한 `tfplanHash`와 S3의 `tfplan` hash가 같은지
- 같은 project에 실행 중인 lock이 없는지
- AWS 연결 계정, role, region이 plan 생성 시점과 같은지

### 완료 기준

- `terraform apply tfplan` 외의 임의 apply를 실행하지 않는다.
- 검증 실패 시 `failureStage = approval` 또는 `apply`로 저장한다.
- apply 로그는 마스킹되어 저장된다.

## 12. 9단계: 로그와 상태 저장

### 목적

실제 실행 중 사용자가 진행 상황과 실패 원인을 볼 수 있게 한다.

### 작업

1. stdout/stderr를 라인 단위로 저장한다.
2. 각 log line에 `sequence`, `stage`, `level`, `message`를 기록한다.
3. credential, token, password, DB URL, Terraform sensitive output을 마스킹한다.
4. 실행 시작 시 `RUNNING`, 성공 시 `SUCCESS`, 실패 시 `FAILED`로 상태를 갱신한다.
5. 실시간 표시는 polling으로 시작하고, 안정화 단계에서 SSE를 붙인다.

### 완료 기준

- 로그 순서가 유지된다.
- 실패 원인이 사용자에게 짧게 표시된다.
- 로그에 secret이 남지 않는다.

## 13. 10단계: state, output, 생성 리소스 추출

### 목적

apply 성공 후 실제 생성된 리소스와 사용자에게 필요한 output을 프로젝트에 연결한다.

### 작업

1. `terraform show -json` 또는 state JSON을 읽는다.
2. 생성된 리소스의 address, type, name, provider, AWS resource id, region을 추출한다.
3. `DeployedResource`로 저장한다.
4. `terraform output -json`을 실행한다.
5. output 이름, 값, sensitive 여부를 저장한다.
6. sensitive output은 화면에 그대로 노출하지 않는다.

### 완료 기준

- 프로젝트 상세에서 생성 리소스 목록을 볼 수 있다.
- output은 배포 결과 화면에서 확인할 수 있다.
- sensitive output은 마스킹된다.

## 14. 11단계: 프로젝트 화면 연결

### 목적

실제 실행 결과를 사용자가 프로젝트 맥락에서 이해하게 한다.

### UI에 표시할 것

- 배포 이력
- plan summary
- 승인 기록
- 실행 로그
- 생성 리소스 목록
- Terraform output
- 실패 단계와 복구 안내
- cleanup/destroy 필요 여부

### 완료 기준

- 사용자가 어떤 snapshot과 artifact로 배포했는지 알 수 있다.
- 어떤 AWS 리소스가 생성됐는지 알 수 있다.
- 삭제가 필요한 리소스가 남아 있는지 알 수 있다.

## 15. 12단계: lock, timeout, cancel, recovery

### 목적

실제 Terraform 실행이 꼬이지 않게 운영 안정성을 붙인다.

### 보강 항목

| 항목 | 이유 |
| --- | --- |
| project lock | 같은 프로젝트에서 동시 apply/destroy 방지 |
| timeout | 무한 실행 방지 |
| cancel | 사용자가 실행 중단 요청 가능 |
| recovery | 서버 재시작 후 `RUNNING` job 정리 |
| idempotency key | 중복 요청으로 같은 job이 여러 번 생기는 것 방지 |

### 완료 기준

- 같은 프로젝트에서 동시에 apply가 돌지 않는다.
- timeout이 발생하면 상태와 로그에 남는다.
- 서버 재시작 후 애매한 `RUNNING` 상태가 정리된다.

## 16. 13단계: destroy와 cleanup

### 목적

실습 리소스가 남아서 비용이 계속 발생하지 않게 한다.

### API 후보

| Method | Path | 목적 |
| --- | --- | --- |
| `POST` | `/api/deployments/:deploymentId/destroy` | 별도 승인 후 리소스 삭제 |

### 작업

1. destroy 전 별도 승인 UI를 둔다.
2. project lock을 확인한다.
3. `terraform destroy` 또는 승인된 destroy plan을 실행한다.
4. 로그를 저장한다.
5. 성공 시 `DeployedResource.status`를 `deleted`로 바꾼다.
6. 실패 시 남아 있을 수 있는 리소스를 사용자에게 보여준다.

### 완료 기준

- 삭제는 apply와 별도 승인 흐름을 가진다.
- 삭제 실패를 성공처럼 표시하지 않는다.
- 남은 리소스를 사용자가 확인할 수 있다.

## 17. 추천 PR 단위

| 순서 | PR 단위 | 목표 |
| --- | --- | --- |
| 1 | 실제 실행 모델 보강 | `AwsConnection`, `tfplan`, lock, resource/output 타입 후보 정리 |
| 2 | STS 연결 확인 | Role Assume setup/verify API |
| 3 | 작업 디렉터리와 artifact 복원 | S3 artifact 복원, 격리 디렉터리, path traversal 방지 |
| 4 | init/validate/plan runner | Terraform command runner, stdout/stderr 로그 저장 |
| 5 | plan JSON parser | `terraform show -json tfplan` 파싱과 plan summary 저장 |
| 6 | Plan 표시와 승인 | blocked 확인, 승인 대상 artifact/tfplan 고정 |
| 7 | approval/apply | 승인된 `tfplan`만 apply |
| 8 | result extraction | state, output, 생성 리소스 저장 |
| 9 | reliability | lock, timeout, cancel, recovery |
| 10 | destroy/cleanup | 삭제 승인과 cleanup 결과 저장 |

## 18. 절대 건너뛰면 안 되는 기준

- 프론트엔드에서 AWS SDK나 Terraform CLI를 호출하지 않는다.
- AI가 만든 Terraform을 바로 apply하지 않는다.
- 승인되지 않은 plan은 apply하지 않는다.
- plan 생성 후 artifact가 바뀌면 다시 plan을 만든다.
- Access Key, Secret Key, session token을 저장소, 로그, 응답, 테스트 데이터에 남기지 않는다.
- 삭제 실패를 성공처럼 표시하지 않는다.
- 실습용 리소스 cleanup 경로 없이 실제 apply를 열지 않는다.
