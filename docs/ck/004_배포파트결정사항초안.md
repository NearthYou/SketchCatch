# 배포 실행 파트 결정사항 원천

## 0. 이 문서의 책임

이 문서는 CK 배포 실행 파트의 **결정사항 원천**이다.

개념 설명은 [배포 실행 파트 개념 설명서](./002_배포실행파트개념과구현흐름정리.md), 파트 간 입출력과 협의 항목은 [배포 실행 파트 입출력 의존성](./003_배포파트의존성정리.md), 구현 순서는 [배포 실행 파트 구현 순서](./005_배포파트구현순서.md), 실제 Terraform 실행 상세는 [실제 배포 실행 상세 구현 순서](./006_실제배포실행구현순서.md)를 따른다.

## 1. 한 줄 결론

배포 실행 파트는 실제 AWS 리소스를 생성, 수정, 삭제할 수 있는 기능이다.

이번 구현에서는 **AWS Role 연결 설정/확인, `terraform init`, `terraform validate`, `terraform plan`, Plan 결과 표시, 사용자 승인, `terraform apply`, 로그 표시, 성공/실패 저장, 생성 리소스/output 저장**까지 만든다.

실제 apply가 포함되므로 승인 게이트, AWS 권한 위임 보호, 로그 마스킹, project lock, timeout, 작업 디렉터리 격리를 구현 범위 안에서 다룬다.

```text
ArchitectureSnapshot
-> TerraformArtifact
-> AWS Role connection setup/check
-> artifact restore
-> terraform init / validate
-> terraform plan -out=tfplan
-> terraform show -json tfplan
-> plan summary
-> pre-deployment analysis / blocked 판단
-> user approval
-> terraform apply tfplan
-> deployment logs
-> state/resource extraction
-> terraform output -json
-> deployed resources / outputs
-> deployment history
```

## 2. 결정 요약

| 결정 항목 | 결정 내용 | 상태 |
| --- | --- | --- |
| 구현 실행 범위 | AWS Role 연결 설정/확인부터 `terraform apply`, 생성 리소스/output 저장까지 구현한다. | 결정됨 |
| 배포 기준 단위 | 최신 화면 상태가 아니라 특정 `ArchitectureSnapshot`과 특정 `TerraformArtifact`를 기준으로 한다. | 결정됨 |
| Terraform 실행 위치 | 프론트가 아니라 API 서버 또는 별도 worker에서만 실행한다. | 결정됨 |
| `DeploymentStatus` | `PENDING`, `RUNNING`, `SUCCESS`, `FAILED`, `CANCELLED`를 유지한다. | 결정됨 |
| blocked 처리 | `BLOCKED`를 status에 추가하지 않고 별도 필드로 표현한다. | 결정됨 |
| AI 오류 설명 입력 | `{ stage, rawMessage, relatedResourceId? }` 최소 payload만 보낸다. | 결정됨 |
| AI 오류 설명 stage | 사용자 설명 요청은 `validate`, `plan`, `apply` 세 값으로 시작한다. 실행 로그 stage와는 분리한다. | 결정됨 |
| AI 설명 저장 | AI 설명 결과를 DB에 저장하지 않고 stateless 응답으로 둔다. | 결정됨 |
| 로그 저장 | CK가 masked Terraform stdout/stderr와 deployment log 저장을 맡는다. | 결정됨 |
| secret redaction | 저장, 응답, AI 전달 전에 secret masking을 적용한다. | 결정됨 |
| AWS 연결 방식 | Role Assume + SketchCatch 생성 External ID를 기본 경로로 둔다. 장기 보관 Access Key/Secret Key 저장은 사용하지 않는다. | 결정됨 |
| apply 실행 | 승인된 `tfplan`만 `terraform apply tfplan`로 실행한다. | 결정됨 |
| destroy/cleanup | apply 이후 안정화 항목으로 별도 승인 흐름을 둔다. | 결정 필요 |
| 생성 리소스 목록 | apply 성공 후 `DeployedResource`로 저장한다. | 결정됨 |
| Terraform output | apply 성공 후 `terraform output -json` 결과를 저장한다. | 결정됨 |

## 3. 배포 실행 흐름 결정

배포 실행 흐름은 아래 순서를 따른다.

1. 특정 프로젝트의 `ArchitectureSnapshot`을 선택한다.
2. 해당 snapshot에서 생성된 `TerraformArtifact`를 찾는다.
3. AWS Role 연결 설정값을 발급하고, 사용자가 만든 Role을 STS로 검증한다.
4. S3 `objectKey`에서 Terraform artifact를 작업 디렉터리에 복원한다.
5. backend 또는 worker에서 `terraform init`을 실행한다.
6. backend 또는 worker에서 `terraform validate`를 실행한다.
7. backend 또는 worker에서 `terraform plan -out=tfplan`을 실행한다.
8. `terraform show -json tfplan` 결과를 파싱해 `planSummary`로 저장한다.
9. 비용/위험 분석 결과가 blocked인지 확인한다.
10. blocked가 아니면 사용자가 plan summary를 확인하고 승인한다.
11. 승인된 `tfplan`만 `terraform apply tfplan`로 실행한다.
12. stdout/stderr 로그와 성공/실패 결과를 저장한다.
13. apply 성공 후 생성 리소스 목록을 추출해 저장한다.
14. apply 성공 후 `terraform output -json` 결과를 저장한다.
15. 프로젝트 배포 이력과 결과 화면에서 조회할 수 있게 한다.

## 4. 배포 기준 단위 결정

배포는 사용자의 현재 화면 상태가 아니라 저장된 snapshot과 artifact를 기준으로 한다.

| 기준 | 설명 |
| --- | --- |
| `projectId` | 어떤 프로젝트의 배포인지 |
| `architectureId` | 어떤 `ArchitectureSnapshot` 기준인지 |
| `terraformArtifactId` | 어떤 Terraform artifact 기준인지 |
| `artifactObjectKey` | S3에 저장된 Terraform artifact 위치 |
| `artifactHash` | 선택값. 승인 후 artifact 변경 여부 확인용 |

결정 규칙:

- 배포 생성 시점에 `architectureId`와 `terraformArtifactId`를 반드시 저장한다.
- 사용자가 승인한 뒤 artifact가 바뀌면 기존 승인은 무효로 본다.
- `artifactHash` 구현이 늦어지면 초기 구현에서는 `terraformArtifactId`와 `createdAt` 기준으로 시작한다.

## 5. 상태와 blocked 결정

`DeploymentStatus`는 아래 상태를 유지한다.

```ts
type DeploymentStatus =
  | "PENDING"
  | "RUNNING"
  | "SUCCESS"
  | "FAILED"
  | "CANCELLED";
```

blocked는 status가 아니라 별도 필드로 표현한다.

```ts
type DeploymentBlock = {
  isBlocked: boolean;
  blockedReason: string | null;
  blockedBy: "risk_analysis" | "cost_analysis" | "missing_approval" | null;
};
```

이렇게 나누면 실행 상태와 차단 사유가 섞이지 않는다.

## 6. 실패 단계 결정

AI 오류 설명용 stage는 아래 세 값만 사용한다.

```ts
type TerraformErrorExplanationStage = "validate" | "plan" | "apply";
```

배포 이력 내부 실패 단계와 로그 stage는 더 넓게 둔다.

```ts
type DeploymentLogStage =
  | "connection"
  | "artifact_restore"
  | "init"
  | "validate"
  | "plan"
  | "approval"
  | "apply"
  | "output"
  | "destroy";
```

| 실패 단계 | 의미 |
| --- | --- |
| `validation` | artifact나 입력값이 부족하거나 구조가 맞지 않음 |
| `connection` | AWS 연결 확인 실패 |
| `artifact_restore` | S3 artifact 복원 실패 |
| `init` | Terraform provider/backend 초기화 실패 |
| `validate` | Terraform 문법 검증 실패 |
| `plan` | plan summary 생성 또는 해석 실패 |
| `approval` | 사용자가 승인하지 않았거나 승인 대상이 바뀜 |
| `apply` | Terraform apply 실패 |
| `output` | `terraform output -json` 추출 실패 |
| `destroy` | cleanup/destroy 실패 |

## 7. Plan/Apply summary 결정

배포 파트는 Terraform 원문이 아니라 화면 표시와 승인에 쓰기 쉬운 summary를 제공한다.

```ts
type DeploymentPlanSummary = {
  createCount: number;
  updateCount: number;
  deleteCount: number;
  replaceCount: number;
  blocked: boolean;
  warnings: Array<{
    level: "low" | "medium" | "high";
    message: string;
    relatedResourceId?: string;
  }>;
};
```

plan 단계에서는 `terraform show -json tfplan` 결과를 이 구조로 변환한다.

## 8. 승인 게이트 결정

승인 게이트는 사용자가 무엇을 보고 승인했는지 기록하는 장치다.

승인 기준:

- 사용자가 plan summary를 확인했다.
- pre-deployment analysis 결과를 확인했다.
- blocked가 아니다.
- 승인 대상 `terraformArtifactId`가 배포 생성 시점과 같다.
- 선택적으로 `artifactHash`가 배포 생성 시점과 같다.
- 승인 대상 `tfplanHash`가 apply 대상 `tfplan`과 같다.
- AWS 연결 계정과 region이 plan 생성 시점과 같다.
- 승인 시각 `approvedAt`을 저장한다.

승인 기록 필드 후보:

```ts
type DeploymentApproval = {
  approvedAt: IsoDateTimeString | null;
  approvedBy: string | null;
  approvedTerraformArtifactId: string | null;
  approvedArtifactHash?: string | null;
  approvedTfplanHash?: string | null;
  approvedAwsConnectionId?: string | null;
};
```

인증 도입 전에는 `approvedBy`를 `workspaceId` 기준으로 둘 수 있다.

## 9. 로그 결정

backend 또는 worker에서 실행한 `terraform init`, `terraform validate`, `terraform plan`, `terraform show -json`, `terraform apply`, `terraform output -json`의 masked stdout/stderr를 저장한다.

```ts
type DeploymentLogLine = {
  deploymentId: string;
  sequence: number;
  stage: DeploymentLogStage;
  level: "INFO" | "WARN" | "ERROR";
  message: string;
  relatedResourceId?: string;
  createdAt: IsoDateTimeString;
};
```

결정 규칙:

- `sequence`로 로그 순서를 보장한다.
- `message`는 화면과 API에 내려줄 수 있는 마스킹된 값이다.
- AI 오류 설명에는 전체 로그 배열을 넘기지 않는다.
- AI 오류 설명에는 `{ stage, rawMessage, relatedResourceId? }` 최소 payload만 넘긴다.
- 실시간 로그 SSE는 안정화 항목이다.

## 10. API 결정

### 배포 실행 API

| Method | Path | 목적 |
| --- | --- | --- |
| `POST` | `/api/projects/:projectId/deployments` | 배포 기록 생성 |
| `GET` | `/api/projects/:projectId/deployments` | 프로젝트 배포 이력 조회 |
| `GET` | `/api/deployments/:deploymentId` | 배포 상세 조회 |
| `POST` | `/api/projects/:projectId/aws-connections` | pending AWS Role 연결 생성, caller principal ARN과 External ID 발급 |
| `POST` | `/api/projects/:projectId/aws-connections/:connectionId/test` | 구현됨. 사용자 `roleArn`과 DB의 `externalId`, `region`으로 `AssumeRole` / `GetCallerIdentity` 연결 테스트 |
| `POST` | `/api/projects/:projectId/aws-connections/:connectionId/verify` | 구현됨. 사용자 `roleArn`과 DB의 `externalId`, `region`으로 검증하고 metadata만 저장 |
| `POST` | `/api/deployments/:deploymentId/plan` | Terraform init/validate/plan 실행 |
| `POST` | `/api/deployments/:deploymentId/approve` | plan summary 승인 기록 |
| `POST` | `/api/deployments/:deploymentId/apply` | 승인된 `tfplan` 적용 |
| `GET` | `/api/deployments/:deploymentId/logs` | 배포 로그 조회 |
| `GET` | `/api/deployments/:deploymentId/resources` | 생성 리소스 목록 조회 |
| `GET` | `/api/deployments/:deploymentId/outputs` | Terraform output 조회 |
| `POST` | `/api/ai/terraform-error-explanation` | 마스킹된 최소 오류 payload로 AI 설명 요청 |

### 안정화 API

| Method | Path | 목적 |
| --- | --- | --- |
| `POST` | `/api/deployments/:deploymentId/cancel` | 실행 중 작업 취소 |
| `POST` | `/api/deployments/:deploymentId/destroy` | 별도 승인 후 cleanup |
| `GET` | `/api/deployments/:deploymentId/logs/stream` | SSE 실시간 로그 |

안정화 API는 apply를 제품 기능으로 연 뒤 바로 이어서 붙인다.

## 12. 보안 결정

- raw AWS Secret Key를 저장하지 않는다.
- raw AWS Secret Key를 API 응답으로 내려주지 않는다.
- `External ID`는 사용자가 정하지 않고 SketchCatch가 connection 단위로 생성한다.
- `SKETCHCATCH_AWS_CALLER_PRINCIPAL_ARN`은 환경 변수로 주입하고, 사용자 trust policy 안내값으로만 내려준다.
- 로그, 오류 메시지, 테스트 데이터, 문서에 secret을 남기지 않는다.
- 프론트엔드에서 AWS SDK나 Terraform CLI를 직접 호출하지 않는다.
- Terraform 실행은 backend 또는 worker에서만 수행한다.
- `terraform apply`는 사용자 승인 후 승인된 `tfplan`으로만 실행한다.
- `terraform plan`과 `terraform apply` 로그와 오류 메시지는 저장 전에 마스킹한다.
- AWS 권한 위임은 Role Assume 정책으로 제한하고, 사용자의 raw Access Key/Secret Key/session token은 저장하지 않는다.
- 위험/비용 분석 결과가 blocked이면 실행을 차단한다.
- 같은 프로젝트에서 동시에 apply가 돌지 않게 한다.
- `tfplan`, state, output 같은 파일성 산출물은 S3에 두고 RDS에는 metadata와 `objectKey`만 저장한다.
- AI 오류 설명에는 전체 로그 배열을 전달하지 않는다.
- AI 호출 전 오류 메시지와 로그를 마스킹한다.
- IaC 파일은 S3에 두고, RDS에는 metadata와 `objectKey`만 저장한다.
- 실패한 작업은 성공처럼 기록하지 않는다.
