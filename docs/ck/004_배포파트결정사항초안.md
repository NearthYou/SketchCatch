# 배포 파트 결정사항 정리

## 0. 이 문서의 책임

이 문서는 CK 배포 실행 파트의 **결정사항 원천**이다.

개념 설명은 [002_배포실행파트개념과구현흐름정리.md](./002_배포실행파트개념과구현흐름정리.md), 파트 간 입출력과 협의 항목은 [003_배포파트의존성정리.md](./003_배포파트의존성정리.md), MVP 구현 순서는 [005_배포파트구현순서.md](./005_배포파트구현순서.md), MVP 이후 실제 실행 순서는 [006_실제배포실행구현순서.md](./006_실제배포실행구현순서.md)를 따른다.

## 1. 한 줄 결론

3주 MVP의 배포 파트는 실제 AWS 리소스를 생성하는 기능이 아니다.

MVP에서는 **Terraform artifact를 기준으로 배포 전 분석, blocked 판단, 사용자 승인, mock 실행 이력, 로그, 결과 summary를 저장하는 흐름**을 먼저 만든다.

```text
ArchitectureSnapshot
-> TerraformArtifact
-> pre-deployment analysis
-> user approval
-> mock deployment record
-> deployment logs
-> deployment history
```

## 2. 결정 요약

| 결정 항목 | 결정 내용 | 상태 |
| --- | --- | --- |
| MVP 실행 범위 | 실제 AWS apply/destroy 없이 mock/통제된 실행 이력 중심으로 간다. | 결정됨 |
| 배포 기준 단위 | 최신 화면 상태가 아니라 특정 `ArchitectureSnapshot`과 특정 `TerraformArtifact`를 기준으로 한다. | 결정됨 |
| Terraform 실행 위치 | 실제 실행이 열리더라도 프론트가 아니라 API 서버 또는 별도 worker에서만 실행한다. | 결정됨 |
| `DeploymentStatus` | MVP는 `PENDING`, `RUNNING`, `SUCCESS`, `FAILED`, `CANCELLED`를 유지한다. | 결정됨 |
| blocked 처리 | `BLOCKED`를 status에 추가하지 않고 별도 필드로 표현한다. | 결정됨 |
| AI 오류 설명 입력 | `{ stage, rawMessage, relatedResourceId? }` 최소 payload만 보낸다. | 결정됨 |
| AI 오류 설명 stage | `validate`, `plan`, `apply` 세 값으로 시작한다. | 결정됨 |
| AI 설명 저장 | MVP에서는 AI 설명 결과를 DB에 저장하지 않고 stateless 응답으로 둔다. | 결정됨 |
| 로그 저장 | CK가 masked deployment log 저장을 맡는다. | 결정됨 |
| secret redaction | 저장, 응답, AI 전달 전에 secret masking을 적용한다. | 결정됨 |
| AWS credential | MVP에서는 장기 보관 Access Key/Secret Key를 저장하지 않는다. | 결정됨 |
| destroy/cleanup | MVP에서는 실제 삭제 실행을 만들지 않는다. | 결정됨 |
| 생성 리소스 목록 | MVP에서는 mock summary까지만 두고 실제 `DeployedResource`는 후순위로 둔다. | 결정됨 |

## 3. MVP 흐름 결정

MVP 배포 흐름은 아래 정도로 제한한다.

1. 특정 프로젝트의 `ArchitectureSnapshot`을 선택한다.
2. 해당 snapshot에서 생성된 `TerraformArtifact`를 찾는다.
3. Terraform artifact metadata와 plan summary를 배포 기록에 연결한다.
4. 비용/위험 분석 결과가 blocked인지 확인한다.
5. blocked가 아니면 사용자가 plan summary를 확인하고 승인한다.
6. 실제 apply 대신 mock 실행 상태를 만든다.
7. 실행 로그와 성공/실패 결과를 저장한다.
8. 프로젝트 배포 이력에서 조회할 수 있게 한다.

## 4. 배포 기준 단위 결정

배포는 사용자의 현재 화면 상태가 아니라 저장된 snapshot과 artifact를 기준으로 한다.

| 기준 | 설명 |
| --- | --- |
| `projectId` | 어떤 프로젝트의 배포인지 |
| `architectureId` | 어떤 `ArchitectureSnapshot` 기준인지 |
| `terraformArtifactId` | 어떤 Terraform artifact 기준인지 |
| `artifactObjectKey` | S3에 저장된 Terraform artifact 위치 |
| `artifactHash` | 선택값. 승인 후 artifact 변경 여부 확인용 |

결정 규칙:

- 배포 생성 시점에 `architectureId`와 `terraformArtifactId`를 반드시 저장한다.
- 사용자가 승인한 뒤 artifact가 바뀌면 기존 승인은 무효로 본다.
- `artifactHash` 구현이 늦어지면 MVP에서는 `terraformArtifactId`와 `createdAt` 기준으로 시작한다.

## 5. 상태와 blocked 결정

`DeploymentStatus`는 MVP에서 아래 상태를 유지한다.

```ts
type DeploymentStatus =
  | "PENDING"
  | "RUNNING"
  | "SUCCESS"
  | "FAILED"
  | "CANCELLED";
```

blocked는 status가 아니라 별도 필드로 표현한다.

```ts
type DeploymentBlock = {
  isBlocked: boolean;
  blockedReason: string | null;
  blockedBy: "risk_analysis" | "cost_analysis" | "missing_approval" | null;
};
```

이렇게 나누면 실행 상태와 차단 사유가 섞이지 않는다.

## 6. 실패 단계 결정

AI 오류 설명용 stage는 아래 세 값만 사용한다.

```ts
type DeploymentStage = "validate" | "plan" | "apply";
```

배포 이력 내부 실패 단계는 더 넓게 둔다.

| 실패 단계 | 의미 |
| --- | --- |
| `validation` | artifact나 입력값이 부족하거나 구조가 맞지 않음 |
| `plan` | plan summary 생성 또는 해석 실패 |
| `approval` | 사용자가 승인하지 않았거나 승인 대상이 바뀜 |
| `mock_run` | mock 실행 중 실패 처리 |

후순위 실제 실행이 열리면 `connection`, `init`, `apply`, `output`, `destroy`, `cleanup`을 추가한다.

## 7. Plan/Apply summary 결정

배포 파트는 Terraform 원문이 아니라 화면 표시와 승인에 쓰기 쉬운 summary를 제공한다.

```ts
type DeploymentPlanSummary = {
  createCount: number;
  updateCount: number;
  deleteCount: number;
  replaceCount: number;
  blocked: boolean;
  warnings: Array<{
    level: "low" | "medium" | "high";
    message: string;
    relatedResourceId?: string;
  }>;
};
```

MVP에서는 실제 Terraform JSON이 없어도 이 구조로 mock summary를 만들 수 있다.

## 8. 승인 게이트 결정

승인 게이트는 사용자가 무엇을 보고 승인했는지 기록하는 장치다.

MVP 승인 기준:

- 사용자가 plan summary를 확인했다.
- pre-deployment analysis 결과를 확인했다.
- blocked가 아니다.
- 승인 대상 `terraformArtifactId`가 배포 생성 시점과 같다.
- 선택적으로 `artifactHash`가 배포 생성 시점과 같다.
- 승인 시각 `approvedAt`을 저장한다.

승인 기록 필드 후보:

```ts
type DeploymentApproval = {
  approvedAt: IsoDateTimeString | null;
  approvedBy: string | null;
  approvedTerraformArtifactId: string | null;
  approvedArtifactHash?: string | null;
};
```

인증 도입 전에는 `approvedBy`를 `workspaceId` 기준으로 둘 수 있다.

## 9. 로그 결정

MVP에서는 실제 Terraform CLI 로그가 아니라 mock 실행 로그를 저장한다.

```ts
type DeploymentLogLine = {
  deploymentId: string;
  sequence: number;
  stage: DeploymentStage;
  level: "INFO" | "WARN" | "ERROR";
  message: string;
  relatedResourceId?: string;
  createdAt: IsoDateTimeString;
};
```

결정 규칙:

- `sequence`로 로그 순서를 보장한다.
- `message`는 화면과 API에 내려줄 수 있는 마스킹된 값이다.
- AI 오류 설명에는 전체 로그 배열을 넘기지 않는다.
- AI 오류 설명에는 `{ stage, rawMessage, relatedResourceId? }` 최소 payload만 넘긴다.
- 실시간 로그 SSE는 후순위다.

## 10. API 결정

### MVP API

| Method | Path | 목적 |
| --- | --- | --- |
| `POST` | `/api/projects/:projectId/deployments` | mock 배포 기록 생성 |
| `GET` | `/api/projects/:projectId/deployments` | 프로젝트 배포 이력 조회 |
| `GET` | `/api/deployments/:deploymentId` | 배포 상세 조회 |
| `POST` | `/api/deployments/:deploymentId/approve` | plan summary 승인 기록 |
| `GET` | `/api/deployments/:deploymentId/logs` | 배포 로그 조회 |
| `POST` | `/api/ai/terraform-error-explanation` | 마스킹된 최소 오류 payload로 AI 설명 요청 |

### 후순위 실제 실행 API

| Method | Path | 목적 |
| --- | --- | --- |
| `POST` | `/api/deployments/:deploymentId/plan` | Terraform plan 실행 |
| `POST` | `/api/deployments/:deploymentId/apply` | 승인된 plan만 apply |
| `POST` | `/api/deployments/:deploymentId/cancel` | 실행 중 작업 취소 |
| `POST` | `/api/deployments/:deploymentId/destroy` | 별도 승인 후 cleanup |
| `GET` | `/api/deployments/:deploymentId/logs/stream` | SSE 실시간 로그 |

후순위 API는 현재 MVP 구현 대상이 아니다.
후순위 API의 실제 구현 순서는 [006_실제배포실행구현순서.md](./006_실제배포실행구현순서.md)에서 관리한다.

## 11. DB 모델 결정

MVP에서 필요한 모델:

| 모델 | 목적 |
| --- | --- |
| `Deployment` | mock/통제된 실행 이력 |
| `DeploymentLog` | 실행 로그 저장 |
| `TerraformArtifact` | S3에 저장된 Terraform file metadata 참조 |

`Deployment` 확장 필드 후보:

| 필드 | 목적 |
| --- | --- |
| `terraformArtifactId` | 어떤 Terraform artifact 기준인지 |
| `artifactObjectKey` | S3 artifact 위치 |
| `artifactHash` | 선택적 artifact 변경 확인 기준 |
| `planSummary` | 화면 표시와 승인 기준 summary |
| `isBlocked` | 위험/비용 분석으로 실행이 차단됐는지 |
| `blockedBy` | 차단 주체 |
| `blockedReason` | 차단 이유 |
| `failureStage` | 실패 단계 |
| `errorSummary` | 사용자에게 보여줄 짧은 실패 요약 |
| `approvedAt` | 승인 시각 |
| `approvedBy` | 승인 주체 |
| `approvedTerraformArtifactId` | 승인 대상 artifact |
| `approvedArtifactHash` | 승인 대상 artifact hash |

후순위 모델:

| 모델 | 후순위 이유 |
| --- | --- |
| `AwsConnection` | 실제 AWS 연결 정책과 Role Assume 설계가 필요함 |
| `DeployedResource` | 실제 apply 이후에만 의미가 있음 |
| `TerraformOutput` | 실제 output 추출 정책이 필요함 |
| `DeploymentLock` | 실제 Terraform 실행이 열릴 때 필요함 |

## 12. 보안 결정

- raw AWS Secret Key를 저장하지 않는다.
- raw AWS Secret Key를 API 응답으로 내려주지 않는다.
- 로그, 오류 메시지, 테스트 데이터, 문서에 secret을 남기지 않는다.
- 프론트엔드에서 AWS SDK나 Terraform CLI를 직접 호출하지 않는다.
- 실제 apply/destroy는 명시 승인 전까지 구현하지 않는다.
- 위험/비용 분석 결과가 blocked이면 실행을 차단한다.
- AI 오류 설명에는 전체 로그 배열을 전달하지 않는다.
- AI 호출 전 오류 메시지와 로그를 마스킹한다.
- IaC 파일은 S3에 두고, RDS에는 metadata와 `objectKey`만 저장한다.
- 실패한 작업은 성공처럼 기록하지 않는다.
