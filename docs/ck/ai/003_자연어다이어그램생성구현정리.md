## 1. 전체 구현 체크리스트

### 1.1 AI Provider와 설명 생성 경계

- [x] AI Provider 실행 대상을 `pre_deployment_analysis`, `terraform_preview_explanation`, `terraform_error_explanation`, `architecture_draft`, `architecture_patch_preview`로 구분한다.
- [x] Bedrock Runtime provider와 fallback provider를 같은 route contract 뒤에 둔다.
- [x] Amazon Q Developer provider를 별도 설정으로 연결하고, 사용할 수 없을 때 fallback 설명을 반환한다.
- [x] Terraform validate/parser 진단을 AI 설명 입력에 붙인다.
- [x] Terraform Preview 리소스와 pre-deployment finding을 AI가 설명할 수 있게 payload를 구성한다.
- [x] Architecture Draft 설명은 초안 구조와 metadata를 기반으로 만든다.
- [x] LLM 설명은 보조 설명으로 취급하고, `ArchitectureJson` 결정성 기준에는 포함하지 않는다.
- [x] AI provider 오류가 사용자 기능 전체 실패로 번지지 않도록 fallback 설명을 둔다.

### 1.2 공유 타입과 API 계약

- [x] `ArchitectureScenario`에 `static_site`, `api_server`, `backend_with_db`, `server_storage`, `serverless_function`을 둔다.
- [x] `CreateArchitectureDraftRequest`에 `prompt`, `scenarioHint`, `budgetLevel`, `trafficLevel`, `securityPriority`를 둔다.
- [x] `ArchitectureDraftScenarioHint` 기본 선택값으로 `auto`를 둔다.
- [x] `AiArchitectureDraftResult`가 `architectureJson`, `title`, `metadata`, `llmExplanation`을 포함하게 한다.
- [x] `ArchitectureGuardrailWarningCode`에 자연어 우선, 미지원 대체, 부분 생성, 비용/설정 조정, 전체 교체 경고 코드를 둔다.
- [x] `/api/ai/architecture-draft` request body를 Zod로 검증한다.
- [x] 유효하지 않은 `scenarioHint`, 예산, 방문자 규모, 보호 기준 값은 API에서 거절한다.
- [x] Source Repository 근거 기반 초안도 결국 같은 Architecture Draft 생성 흐름을 재사용한다.
- [x] `ArchitecturePatchPreview` 타입을 추가해 향후 patch preview가 사용자 승인 전 상태임을 표현한다.

### 1.3 자연어 우선 시나리오 결정

- [x] 자연어 Requirement Prompt의 구체 단서를 보조 선택보다 우선한다.
- [x] 자연어와 보조 선택이 충돌하면 자연어 기준으로 생성하고 `selection_overridden_by_prompt` 경고를 남긴다.
- [x] 같은 prompt와 같은 보조 조건은 같은 `ArchitectureJson`을 반환한다.
- [x] `웹사이트 하나 배포하고 싶어`처럼 아키텍처 대상은 있지만 단서가 부족한 입력은 곧바로 생성하지 않는다.
- [x] 보조 선택이 `api_server`, `backend_with_db`처럼 명시적인 경우에는 부족한 자연어 단서를 채우는 힌트로 사용한다.
- [x] 보조 선택도 `auto`이고 자연어 단서가 부족하면 beginner-friendly 질문 흐름으로 넘긴다.
- [x] 아키텍처와 무관한 입력이나 단서가 전혀 없는 입력은 `400 bad_request`로 거절한다.
- [x] API 오류 메시지는 프론트에서 일반 오류가 아니라 추가 질문으로 전환할 수 있게 구체적으로 반환한다.

### 1.4 지원 시나리오별 고정 템플릿

- [x] `static_site`는 S3 origin과 CloudFront CDN 구조를 만든다.
- [x] `api_server`는 VPC, Subnet, Internet Gateway, Route Table, Security Group, AMI, EC2, IAM Role/Policy/Instance Profile, CloudWatch Logs/Alarm을 포함한다.
- [x] `backend_with_db`는 App Subnet과 DB Subnet을 분리하고, EC2 backend, RDS, KMS Key, IAM, Logs, Alarm을 포함한다.
- [x] `server_storage`는 EC2 서버와 S3 Bucket, VPC/Subnet/Route Table/Internet Gateway/Security Group/AMI 관계를 포함한다.
- [x] `serverless_function`은 API Gateway, Lambda, Lambda Permission, IAM Role/Policy, KMS Key, CloudWatch Log Group, Metric Alarm을 포함한다.
- [x] 템플릿은 지원하는 `ResourceType`만 사용하고 `UNKNOWN` 리소스를 생성하지 않는다.
- [x] AMI, 라우팅, IAM, Logs, Metrics, Alarm, KMS 같은 검수 맥락을 초안에 포함한다.
- [x] EC2와 S3 같은 핵심 서비스 관계는 독립 노드만 나열하지 않고 실제 사용 관계 edge로 연결한다.

### 1.5 미지원 요구사항과 guardrail

- [x] EKS/Kubernetes 요구는 단일 EC2 API 서버 초안으로 대체하고 경고를 남긴다.
- [x] ECS/Fargate 요구는 단일 EC2 API 서버 초안으로 대체하고 경고를 남긴다.
- [x] DynamoDB/NoSQL 요구는 지원 가능한 RDS 데이터베이스 구조로 대체하고 경고를 남긴다.
- [x] ALB/Auto Scaling 요구는 단일 EC2 서버 구조로 대체하고 경고를 남긴다.
- [x] 멀티 리전 요구는 단일 리전 초안으로 대체했다는 경고를 남긴다.
- [x] Redis, SQS/SNS/EventBridge/Step Functions, CI/CD 자동 구성, 실시간 비용/보안 보장, 조직 내부 시스템 연동은 자동 생성 범위에서 제외하고 경고를 남긴다.
- [x] 지원 가능한 요구와 미지원 요구가 섞이면 지원 가능한 부분만 생성하고 `partial_generation` 경고를 남긴다.
- [x] 추가 질문으로 해결 가능한 guardrail은 미리보기를 띄우기 전에 질문으로 보류한다.
- [x] 낮은 예산과 DB 포함이 충돌하면 DB 없는 구조로 만들지 묻고, 사용자의 답변을 재생성 조건에 반영한다.

### 1.6 보조 선택과 운영 조건 반영

- [x] `budgetLevel`은 단순 설명이 아니라 EC2/RDS 크기, CloudFront price class, S3 cleanup 성격 config에 반영된다.
- [x] `trafficLevel`은 EC2 monitoring, RDS storage, Lambda timeout/memory, log retention에 반영된다.
- [x] `securityPriority`는 S3 public access block, RDS deletion protection/final snapshot, Security Group ingress, log retention에 반영된다.
- [x] 낮은 예산/작은 방문자 조건은 작은 인스턴스와 낮은 로그 보존 기간을 사용한다.
- [x] 보통 예산/보통 방문자/높은 보호 조건은 더 보수적인 운영 config를 사용한다.
- [x] 운영 조건으로 config가 조정되면 `guardrail_adjusted_config` 경고를 남긴다.
- [x] 같은 자연어라도 보조 선택이 달라지면 생성되는 지원 리소스 config가 달라진다.

### 1.7 Workspace AI 채팅 UX

- [x] `/workspace/ai` 별도 화면 대신 `/workspace` 보드 안의 오른쪽 하단 AI 채팅 dock에서 동작한다.
- [x] `/workspace/ai` route는 `/workspace`로 redirect한다.
- [x] 기존 오른쪽 AI 탭 대신 floating launcher와 하단 채팅형 dock을 제공한다.
- [x] 채팅 기록은 project 단위 localStorage key로 저장한다.
- [x] 보조 선택 컨트롤을 채팅창 위에 유지한다.
- [x] 프롬프트 작성 가이드를 짧은 예시 chip과 함께 제공한다.
- [x] `웹사이트 하나 배포하고 싶어`처럼 부족한 입력은 사이트 목적, 방문자 행동, 운영 기준을 차례로 질문한다.
- [x] 질문은 `S3 public으로 할까요?` 같은 전문 용어가 아니라 사용자가 이해할 수 있는 말로 묻는다.
- [x] 질문 답변이 끝나면 구현 리스트를 먼저 보여주고, 사용자가 승인한 뒤에만 Architecture Draft를 요청한다.
- [x] guardrail 추가 질문이 남아 있으면 초안 미리보기를 띄우지 않고 답변을 기다린다.
- [x] 추가 질문 답변은 일반 프롬프트가 아니라 대기 중인 질문 세션의 응답으로 처리한다.

### 1.8 보드 미리보기와 사용자 승인

- [x] Architecture Draft는 바로 보드에 적용하지 않고 반투명 preview diagram으로 먼저 표시한다.
- [x] preview 중 보드 노드/엣지는 읽기 전용으로 취급한다.
- [x] preview 중 사용자는 카드의 `생성`, `취소`, `다시 생성` 흐름만 사용한다.
- [x] `생성`을 누르면 현재 보드가 AI 초안으로 전체 교체된다.
- [x] `취소`를 누르면 preview만 제거하고 기존 보드는 유지한다.
- [x] `다시 생성`은 마지막 `CreateArchitectureDraftRequest`를 재사용한다.
- [x] 기존 보드에 리소스가 있으면 카드 하단에 전체 교체 경고를 표시한다.
- [x] AI 제안은 항상 User-Accepted Change 이후에만 실제 보드에 반영한다.
- [ ] 기존 보드의 일부만 바꾸는 patch preview/apply는 아직 열지 않는다.

### 1.9 ArchitectureJson과 DiagramJson 변환

- [x] Architecture Draft의 `ArchitectureJson`을 실제 보드 `DiagramJson`으로 변환한다.
- [x] 리소스 노드는 기존 Resource catalog를 거쳐 수동 배치 노드와 같은 icon, size, 기본 style을 사용한다.
- [x] Architecture config는 Terraform Preview가 읽을 수 있는 node parameter values로 옮긴다.
- [x] Architecture edge는 source/target handle을 계산해 React Flow에서 실제 화살표로 보이게 한다.
- [x] `contains`, `hosts` 같은 포함관계 edge는 area parent metadata로 바꾸고 화면 화살표는 숨긴다.
- [x] VPC, Subnet, Security Group 같은 area node는 포함관계가 드러나도록 child를 감싸게 한다.
- [x] Security Group이 area로 표시될 때 보호 대상 리소스를 내부에 배치한다.
- [x] Terraform reference 문자열을 해석해 Subnet, VPC, Security Group parent 관계를 잡는다.
- [x] area box는 child bounds를 기준으로 왼쪽/위쪽까지 포함해 확장된다.
- [x] Server Storage 템플릿은 관계를 더 읽기 쉬운 nested cloud area layout으로 보정한다.
- [x] 포함관계가 아닌 사용/호출/모니터링 edge는 화살표로 유지한다.

## 2. 안정화 체크리스트

아래 항목은 현재 브랜치의 1차 구현 이후 품질을 올리기 위한 후속 작업이다.

- [ ] Browser 수동 smoke로 채팅 dock 열기, 질문 답변, preview, 생성, 취소, 다시 생성을 확인한다.
- [ ] Representative Use Journey 수준의 자동/반자동 E2E smoke를 추가한다.
- [ ] Patch preview/apply를 추가해 전체 교체가 아니라 부분 수정도 사용자 승인 뒤 적용하게 한다.
- [ ] patch preview도 현재 생성 preview처럼 반투명으로 보여주고 승인 전에는 보드 상태를 바꾸지 않는다.
- [ ] 질문 흐름을 더 많은 일반 사용자 문장에 대해 보강한다.
- [ ] 지원 ResourceType 확대 시 guardrail 대체/제외 규칙과 템플릿을 함께 갱신한다.
- [ ] 비용 추정 문구가 실제 비용 보장을 암시하지 않도록 계속 점검한다.
- [ ] provider-neutral 확장을 위해 AWS-first 템플릿과 공통 Architecture 계약의 경계를 더 명확히 분리한다.
- [ ] 현재 `/workspace` 채팅 dock 중심 구현과 구형 `WorkspaceAiPanel`의 책임을 정리한다.
- [ ] UI 스크린샷 기반 회귀 또는 Playwright smoke를 추가한다.

## 3. 완료 기준

아래 조건이 만족되면 자연어 다이어그램 생성 1차 구현은 완료로 본다.

- 사용자가 자연어로 요구사항을 입력하면 지원 가능한 Practice Architecture 초안을 만들 수 있다.
- 같은 prompt와 같은 보조 조건은 같은 `ArchitectureJson`을 반환한다.
- 자연어 단서가 보조 선택보다 우선한다.
- 보조 선택은 자연어가 부족한 경우의 힌트와 운영 config 조건으로 실제 생성 결과에 반영된다.
- 아키텍처 단서가 부족하면 초안을 만들지 않고 쉬운 질문으로 필요한 정보를 먼저 모은다.
- 질문이 남아 있으면 preview를 띄우지 않는다.
- 사용자가 질문에 답하면 그 답이 다음 생성 조건에 반영된다.
- 지원하지 않는 리소스는 생성하지 않고, 가능한 경우 지원 리소스로 대체하거나 제외 경고를 남긴다.
- Architecture Draft는 지원 ResourceType만 사용한다.
- 초안은 실제 보드에 반투명 preview로 먼저 표시되고, 사용자가 `생성`을 승인해야만 보드에 적용된다.
- 현재 버전의 `생성`은 전체 보드 교체이며, 이 위험은 카드 하단 경고로 표시된다.
- 포함관계는 area containment로 보이고, 포함관계 edge 화살표는 숨겨진다.
- 사용/호출/모니터링 같은 관계 edge는 화살표로 보인다.
- AI/LLM 설명은 보조 설명이며 실제 Architecture Board 변경은 사용자 승인 뒤에만 일어난다.
- raw secret, AWS credential, DB password, private key가 문서, 로그, 테스트, API 응답에 노출되지 않는다.

## 5. 현재 주의할 점

- 이 기능은 AWS-first 템플릿을 제공하지만 SketchCatch 제품 자체를 AWS-only로 설명하면 안 된다.
- Redis는 내부 Runtime Cache이며 사용자 Practice Architecture Resource가 아니다.
- 실제 Terraform apply/destroy나 cloud mutation은 이 기능에서 실행하지 않는다.
- 초안 생성은 Practice Architecture 제안이며, 실제 IaC 실행은 Terraform Preview, Pre-Deployment Check, 사용자 승인된 Deployment 흐름에서 다룬다.
- 현재 patch mode는 타입과 설명 경계만 있으며, Workspace UI의 부분 적용 기능으로 열려 있지 않다.
- 다이어그램 검수 지식은 [아키텍처 다이어그램 검수 가이드](./002_아키텍처다이어그램검수가이드.md)를 기준으로 계속 보강한다.
