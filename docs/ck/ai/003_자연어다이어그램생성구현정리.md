# 자연어 다이어그램 생성 실행 흐름 정리

이 문서는 현재 브랜치에서 구현한 Natural Language Diagramming 흐름을 코드 기준으로 따라가기 위한 정리다.

링크는 현재 브랜치의 코드 라인 기준 `#L` anchor다. 코드가 이동하면 이 문서의 라인 링크도 다시 확인해야 한다.

핵심은 사용자의 문장을 고정 시나리오 목록 중 하나로 고르는 것이 아니다. 자연어 안에서 아키텍처 단서를 뽑아 [`requirementFacts`](../../../packages/types/src/index.ts#L693)로 정규화하고, 그 fact 조합으로 지원 가능한 리소스만 사용해 deterministic [`ArchitectureJson`](../../../packages/types/src/index.ts#L73) 초안을 만든다. 1차 진입점은 [`createArchitectureDraft`](../../../apps/api/src/services/aiArchitectureDrafts.ts#L8)이고, 실제 조립은 [`createDraftFromRequirementFacts`](../../../apps/api/src/services/aiArchitectureRequirementDraftBuilder.ts#L9)가 맡는다.

## 1. 목표

1. 사용자는 채팅창에 자연어 요구사항만 입력한다. 입력 분기는 [`submitUserMessage`](../../../apps/web/features/workspace/WorkspaceAiChatDock.tsx#L140)와 [`handleUserMessage`](../../../apps/web/features/workspace/WorkspaceAiChatDock.tsx#L155)에서 시작한다.
2. 요구사항이 부족하면 초안을 만들지 않고 쉬운 질문과 추천 답변을 차례대로 보여준다. frontend 1차 판정은 [`needsArchitectureClarification`](../../../apps/web/features/workspace/workspace-ai-clarification.ts#L158), backend 최종 거절은 [`isGenericWebsitePromptWithoutConcreteArchitecture`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L290)와 [`AmbiguousArchitecturePromptError`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L243)에 걸려 있다.
3. 충분한 단서가 모이면 지원 가능한 `ResourceType`만 사용해 [`ArchitectureJson`](../../../packages/types/src/index.ts#L73)을 만든다. fact 타입은 [`ArchitectureRequirementFact`](../../../packages/types/src/index.ts#L693), fact 정렬은 [`sortRequirementFacts`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L380)를 본다.
4. 초안은 workspace board에 반투명 preview로만 표시하고, 사용자가 `생성`을 눌러야 실제 board에 적용한다. preview는 [`showDraftPreview`](../../../apps/web/features/workspace/WorkspaceAiChatDock.tsx#L307), 실제 적용은 [`applyDraftToBoard`](../../../apps/web/features/workspace/WorkspaceAiChatDock.tsx#L331)다.
5. 같은 요구사항 단서 조합은 같은 node id, edge id, config, layout을 가진 같은 초안을 만든다. 검증은 [`deterministic ArchitectureJson` 테스트](../../../apps/api/src/routes/ai.test.ts#L650)와 [`equivalent wording` 테스트](../../../apps/api/src/routes/ai.test.ts#L678)를 본다.

## 2. 전체 흐름

```text
사용자 자연어 입력
-> Workspace AI Chat Dock이 모호한 요구사항인지 확인
-> 질문이 필요하면 draft API를 호출하지 않고 질문/추천 답변 표시
-> 답변이 모두 모이면 구현 리스트 확인
-> POST /api/ai/architecture-draft
-> backend가 requirementFacts, operatingProfile, guardrailWarnings 계산
-> requirementFacts 조합으로 지원 리소스만 조립
-> 운영 조건 config와 guardrail metadata 적용
-> 선택적으로 LLM 설명 보강
-> frontend가 preview diagram 표시
-> 사용자 생성 승인 후 workspace board 전체 교체
```

이 흐름에서 Terraform apply, cloud mutation, Git/CI/CD handoff는 일어나지 않는다. AI 초안은 `User-Accepted Change` 경계를 넘기 전까지 실제 프로젝트 상태를 바꾸지 않는다.

## 3. 요청 계약

[`CreateArchitectureDraftRequest`](../../../packages/types/src/index.ts#L823)는 `prompt`만 받는다.

```ts
type CreateArchitectureDraftRequest = {
  prompt: string;
};
```

route validation도 [`architectureDraftBodySchema`](../../../apps/api/src/routes/ai.ts#L84)에서 `prompt`만 받는다. `scenarioHint`, `budgetLevel`, `trafficLevel`, `securityPriority` 같은 보조 선택값은 초안 생성 계약에서 제거되었다. 예산, 방문자 규모, 보호 수준은 사용자가 자연어로 말한 단서에서 backend가 [`operatingProfile`](../../../packages/types/src/index.ts#L707)로 계산한다.

legacy client가 예전 field를 보내더라도 생성 기준으로 사용하지 않는다. 자연어 `prompt`가 우선이다. 이 경계는 [`generic website prompt에서 legacy helper field를 무시하는 테스트`](../../../apps/api/src/routes/ai.test.ts#L534), [`legacy guardrail choice field를 무시하는 테스트`](../../../apps/api/src/routes/ai.test.ts#L1064), [`Workspace AI client에 보조 field가 남아 있지 않은지 확인하는 테스트`](../../../apps/web/features/workspace/workspace-ai-guardrail-warning.test.ts#L38)로 확인한다.

## 5. API 흐름

Architecture Draft API는 [`apps/api/src/routes/ai.ts`](../../../apps/api/src/routes/ai.ts#L153)의 다음 endpoint다.

```text
POST /api/ai/architecture-draft
```

route는 [`architectureDraftBodySchema`](../../../apps/api/src/routes/ai.ts#L84)로 `prompt`를 검증한 뒤 [`createArchitectureDraft`](../../../apps/api/src/services/aiArchitectureDrafts.ts#L8)를 호출한다. route 레벨에서는 [`addArchitectureDraftLlmExplanation`](../../../apps/api/src/routes/ai.ts#L274)로 설명 레이어만 붙인다.

```text
createArchitectureDraft
-> normalizeArchitectureDraftRequest
-> resolveArchitectureRequirement
-> createDraftFromRequirementFacts
-> applyOperatingConditionConfig
-> applyGuardrailMetadata
-> AiArchitectureDraftResult 반환
```

라인 기준으로 읽으면 [`createArchitectureDraft`](../../../apps/api/src/services/aiArchitectureDrafts.ts#L8) -> [`normalizeArchitectureDraftRequest`](../../../apps/api/src/services/aiArchitectureDrafts.ts#L39) -> [`resolveArchitectureRequirement`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L235) -> [`createDraftFromRequirementFacts`](../../../apps/api/src/services/aiArchitectureRequirementDraftBuilder.ts#L9) -> [`applyOperatingConditionConfig`](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts#L8) -> [`applyGuardrailMetadata`](../../../apps/api/src/services/aiArchitectureDraftMetadata.ts#L10) 순서다.

### 5.1 Service Pipeline 한 줄씩 뜯기

아래 설명은 표 요약이 아니라 실제 코드 줄을 따라 읽는 해설이다. 중요한 점은 이 파이프라인이 구조 결정, config 보정, metadata 보강을 섞지 않고 한 줄씩 넘긴다는 것이다.

#### `createArchitectureDraft`

1. [`L8 createArchitectureDraft(input)`](../../../apps/api/src/services/aiArchitectureDrafts.ts#L8)는 service의 1차 입구다. route가 넘긴 `{ prompt }`나 내부 재사용용 문자열 입력이 여기로 들어온다.

2. [`L9 const request = normalizeArchitectureDraftRequest(input)`](../../../apps/api/src/services/aiArchitectureDrafts.ts#L9)는 입력 모양을 먼저 맞춘다. 이 줄 이후 단계들은 문자열인지 객체인지 신경 쓰지 않고 `request.prompt`만 본다.

3. [`L10 const resolution = resolveArchitectureRequirement(request)`](../../../apps/api/src/services/aiArchitectureDrafts.ts#L10)는 자연어를 해석하는 첫 실질 단계다. 여기서 fact, 운영 조건, warning이 정해지고, 너무 모호하면 초안 생성 전에 멈춘다.

4. [`L11 const draft = createDraftFromRequirementFacts(resolution)`](../../../apps/api/src/services/aiArchitectureDrafts.ts#L11)는 해석 결과를 실제 `ArchitectureJson` 초안으로 바꾼다. 이 줄이 node, edge, 기본 layout을 만드는 중심이다.

5. [`L12 const configuredDraft = applyOperatingConditionConfig(...)`](../../../apps/api/src/services/aiArchitectureDrafts.ts#L12)는 이미 만들어진 리소스의 config만 조정한다. 비용, 트래픽, 보호 수준은 여기서 수치와 옵션으로 반영된다.

6. [`L14 return applyGuardrailMetadata(...)`](../../../apps/api/src/services/aiArchitectureDrafts.ts#L14)는 마지막으로 metadata를 붙여 반환한다. 구조를 다시 만들지 않고, 사용자가 알아야 할 fact와 warning 설명을 결과에 얹는다.

#### `normalizeArchitectureDraftRequest`

1. [`L39 normalizeArchitectureDraftRequest(input)`](../../../apps/api/src/services/aiArchitectureDrafts.ts#L39)는 `createArchitectureDraft` 안에서만 쓰는 작은 adapter다. 목적은 모든 입력을 `CreateArchitectureDraftRequest`처럼 다루게 하는 것이다.

2. [`L40 if (typeof input !== "string")`](../../../apps/api/src/services/aiArchitectureDrafts.ts#L40)는 이미 객체로 들어온 요청을 구분한다. API route에서 온 요청은 보통 이 경로를 탄다.

3. [`L41 return input`](../../../apps/api/src/services/aiArchitectureDrafts.ts#L41)은 객체 요청을 그대로 통과시킨다. downstream 코드는 객체 전체가 아니라 `prompt` 계약만 실제 판단에 사용한다.

4. [`L44-L45 return { prompt: input }`](../../../apps/api/src/services/aiArchitectureDrafts.ts#L44)는 문자열 입력을 `{ prompt }`로 감싼다. GitHub evidence 재사용 흐름처럼 내부에서 문자열만 넘길 때 필요하다.

#### `resolveArchitectureRequirement`

1. [`L239 findUnsupportedRequirementMatches(request.prompt)`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L239)는 먼저 현재 자동 생성 범위 밖의 요구사항을 찾는다. EKS, Redis, CI/CD 같은 단서가 여기에 걸린다.

2. [`L240 createRequirementFacts(...)`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L240)는 prompt keyword와 unsupported 대체 규칙을 합쳐 `requirementFacts`를 만든다. 이후 리소스 조립은 이 배열을 기준으로만 움직인다.

3. [`L241 const hasPromptSignal = requirementFacts.length > 0`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L241)는 생성 가능한 단서가 하나라도 있는지 확인한다. 단서가 없으면 fallback 초안을 만들지 않는다.

4. [`L242 createUnsupportedRequirementWarnings(...)`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L242)는 대체하거나 제외한 요구사항을 사용자가 볼 수 있는 warning으로 만든다.

5. [`L244 isGenericWebsitePromptWithoutConcreteArchitecture(...)`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L244)는 “웹사이트 하나”처럼 너무 넓은 요청을 막는다. 이 경우 frontend 질문 흐름으로 돌려야 한다.

6. [`L250 if (hasPromptSignal)`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L250)는 생성 가능한 자연어 단서가 있을 때만 resolution을 반환한다. 여기서부터 초안 생성이 허용된다.

7. [`L252 selectedDraftPattern`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L252)은 대표 라벨이다. 실제 리소스 생성 기준이 아니라 UI와 설명용 이름이다.

8. [`L253-L258`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L253)는 `requirementFacts`, `operatingProfile`, `guardrailWarnings`를 한 묶음으로 돌려준다. 다음 builder/config/metadata 단계가 이 값을 나눠 쓴다.

9. [`L262 throw new AmbiguousArchitecturePromptError()`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L262)는 fact가 하나도 없을 때의 최종 방어선이다. 모르는 요청을 억지 architecture로 꾸미지 않는다.

#### `createDraftFromRequirementFacts`

1. [`L12 const factSet = new Set(...)`](../../../apps/api/src/services/aiArchitectureRequirementDraftBuilder.ts#L12)는 중복 없는 fact lookup을 만든다. 이후 `if (factSet.has(...))`로 리소스 묶음을 결정한다.

2. [`L13-L15`](../../../apps/api/src/services/aiArchitectureRequirementDraftBuilder.ts#L13)는 빈 `nodes`, `edges`, 공유 `context`를 만든다. builder helper들은 이 context에 node와 edge를 추가한다.

3. [`L17-L19`](../../../apps/api/src/services/aiArchitectureRequirementDraftBuilder.ts#L17)는 웹 화면 또는 정적 배포 단서가 있으면 S3/CloudFront 정적 사이트 구성을 넣는다.

4. [`L21-L23`](../../../apps/api/src/services/aiArchitectureRequirementDraftBuilder.ts#L21)는 서버 실행 단서가 있으면 VPC, subnet, security group, AMI, EC2, IAM, observability 묶음을 넣는다.

5. [`L25-L27`](../../../apps/api/src/services/aiArchitectureRequirementDraftBuilder.ts#L25)는 Lambda/serverless 단서가 있으면 API Gateway, Lambda, permission, log/alarm 묶음을 넣는다.

6. [`L29-L31`](../../../apps/api/src/services/aiArchitectureRequirementDraftBuilder.ts#L29)는 database fact가 있으면 DB subnet, DB security group, RDS, KMS, DB alarm을 넣는다.

7. [`L33-L35`](../../../apps/api/src/services/aiArchitectureRequirementDraftBuilder.ts#L33)는 object storage가 서버나 Lambda와 같이 필요할 때 upload bucket을 추가한다.

8. [`L37 addCrossResourceEdges(context)`](../../../apps/api/src/services/aiArchitectureRequirementDraftBuilder.ts#L37)는 따로 만들어진 리소스 묶음 사이의 관계선을 마지막에 연결한다. 서버-DB, 서버-S3 같은 edge가 여기서 붙는다.

9. [`L39-L50 return`](../../../apps/api/src/services/aiArchitectureRequirementDraftBuilder.ts#L39)는 완성된 `ArchitectureJson`과 기본 metadata/title을 반환한다. source는 rule 기반 fallback인 `template_fallback`으로 남긴다.

#### `applyOperatingConditionConfig`

1. [`L12 return { ...draft }`](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts#L12)는 기존 draft를 복사해서 새 결과를 만든다. 원본 초안 객체를 직접 mutate하지 않는다.

2. [`L14-L16 architectureJson.nodes.map(...)`](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts#L14)는 모든 node를 순회하며 운영 조건 config를 적용한다. 리소스를 추가하거나 제거하지 않고 config만 바꾼다.

3. [`L26 applySizingAndTrafficConfig(...)`](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts#L26)는 비용과 트래픽 기준을 먼저 반영한다. EC2/RDS size, CloudFront price class, log retention 같은 값이 여기서 달라진다.

4. [`L28 applyHighSecurityConfig(...)`](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts#L28)는 보호 우선순위가 높을 때만 추가 보정을 한다. S3 public block, RDS private, security group ingress 축소가 이 경로다.

#### `applyGuardrailMetadata`

1. [`L15-L18 guardrailWarnings`](../../../apps/api/src/services/aiArchitectureDraftMetadata.ts#L15)는 requirement 단계 warning과 운영 조건 warning을 합친다. 낮은 예산인데 RDS가 있는 경우 같은 경고가 여기서 더해진다.

2. [`L20-L23 return { ...draft, metadata: ... }`](../../../apps/api/src/services/aiArchitectureDraftMetadata.ts#L20)는 draft 본문은 유지하고 metadata만 확장한다. 구조 결정은 이미 끝난 상태다.

3. [`L24-L27`](../../../apps/api/src/services/aiArchitectureDraftMetadata.ts#L24)는 `selectedDraftPattern`, `requirementFacts`, `operatingProfile`, `guardrailWarnings`를 결과 metadata에 노출한다. frontend와 LLM 설명이 이 값을 읽는다.

4. [`L28-L29`](../../../apps/api/src/services/aiArchitectureDraftMetadata.ts#L28)는 assumptions와 explanations를 보강한다. “대표 패턴은 라벨이고 실제 생성은 fact 기준” 같은 설명도 이 레이어에서 붙는다.

LLM 설명은 이 service 파이프라인 밖의 route 단계에서 붙는다. [`addArchitectureDraftLlmExplanation`](../../../apps/api/src/routes/ai.ts#L274)은 rule 기반 결과를 받아 설명만 보강하며, 구조 결정에는 참여하지 않는다.

[`resolveArchitectureRequirement`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L235)는 fixed scenario score를 고르는 단계가 아니다. 자연어에서 [`requirementFacts`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L239), [`operatingProfile`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L253), 지원 불가/대체 warning을 뽑는 단계다.

[`selectedDraftPattern`](../../../packages/types/src/index.ts#L661)은 UI와 LLM 설명을 위한 대표 라벨이다. 실제 리소스 생성은 [`selectDraftPattern`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L264)이 아니라 [`requirementFacts`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L239) 기준으로 동작한다. 이 설명은 [`createGuardrailExplanations`](../../../apps/api/src/services/aiArchitectureDraftMetadata.ts#L74)에도 명시되어 있다.

## 6. 요구사항 fact 해석

[`aiArchitectureRequirementResolution.ts`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L92)는 prompt에서 다음 단서를 찾는다. 타입 정의는 [`ArchitectureRequirementFact`](../../../packages/types/src/index.ts#L693)다.

```ts
type ArchitectureRequirementFact =
  | "web_frontend"
  | "static_delivery"
  | "server_runtime"
  | "database"
  | "object_storage"
  | "file_upload"
  | "auth_or_user_data"
  | "serverless_runtime"
  | "network_boundary"
  | "iam_permissions"
  | "observability"
  | "encryption";
```

예시는 다음과 같다.

| 자연어 표현 | 해석되는 단서 |
| --- | --- |
| `소개용 랜딩 웹사이트`, `정적 사이트` | [`web_frontend`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L94), [`static_delivery`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L115), [`object_storage`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L156) |
| `서버`, `API`, `백엔드` | [`server_runtime`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L119), [`network_boundary`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L359), [`iam_permissions`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L361), [`observability`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L362) |
| `파일 업로드`, `이미지 저장` | [`file_upload`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L160), [`object_storage`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L156), [`server_runtime` 파생](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L348) |
| `로그인`, `회원`, `개인정보` | [`auth_or_user_data`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L164), [`database` 파생](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L342), [`encryption` 파생](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L345), [`server_runtime` 파생](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L344) |

`웹사이트 하나 배포하고 싶어`처럼 대상은 맞지만 화면만 필요한지, 파일 업로드가 필요한지, 로그인/데이터 저장이 필요한지 알 수 없는 문장은 초안을 생성하지 않는다. 이 경우 frontend가 [`needsArchitectureClarification`](../../../apps/web/features/workspace/workspace-ai-clarification.ts#L158)으로 먼저 질문하고, backend도 [`isGenericWebsitePromptWithoutConcreteArchitecture`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L290)로 fallback draft 생성을 막는다.

반대로 `웹사이트를 S3와 CloudFront로 배포하고 싶어`처럼 명확한 아키텍처 단서가 있으면 추가 질문 없이 초안을 요청할 수 있다. 이 동작은 [`needsArchitectureClarification` 테스트](../../../apps/web/features/workspace/workspace-ai-clarification.test.ts#L15)와 [`static website request` API 테스트](../../../apps/api/src/routes/ai.test.ts#L151)에서 확인한다.

## 8. ResourceDefinition과 Terraform identity 연결

AI 초안의 `ArchitectureJson.nodes[].type`은 `EC2`, `S3`, `IAM_ROLE` 같은 domain `ResourceType`이다. Terraform Preview identity가 아니며, 사용자가 보는 보드 node의 Terraform type도 아니다.

보드와 Terraform 경계에서는 `resource/aws_instance`, `data/aws_ami`처럼 `terraformBlockType`과 `parameters.resourceType` 조합을 쓴다. 이 변환의 단일 출처는 [`resourceDefinitions`](../../../packages/types/src/resource-definitions.ts#L32)다.

| domain `ResourceType` | Terraform identity | 사용 맥락 |
| --- | --- | --- |
| `IAM_ROLE` | [`resource/aws_iam_role`](../../../packages/types/src/resource-definitions.ts#L96) | EC2/Lambda 실행 권한 |
| `IAM_POLICY` | [`resource/aws_iam_policy`](../../../packages/types/src/resource-definitions.ts#L100) | 실행 role에 붙는 권한 정책 |
| `IAM_INSTANCE_PROFILE` | [`resource/aws_iam_instance_profile`](../../../packages/types/src/resource-definitions.ts#L105) | EC2가 role을 사용할 수 있게 하는 profile |
| `KMS_KEY` | [`resource/aws_kms_key`](../../../packages/types/src/resource-definitions.ts#L110) | DB, 로그, 저장소 암호화 단서 |
| `LAMBDA_PERMISSION` | [`resource/aws_lambda_permission`](../../../packages/types/src/resource-definitions.ts#L211) | API Gateway 같은 trigger가 Lambda를 호출하는 권한 |
| `API_GATEWAY_REST_API` | [`resource/aws_api_gateway_rest_api`](../../../packages/types/src/resource-definitions.ts#L216) | serverless 진입점 |
| `CLOUDWATCH_LOG_GROUP` | [`resource/aws_cloudwatch_log_group`](../../../packages/types/src/resource-definitions.ts#L233) | 런타임 로그 |
| `CLOUDWATCH_METRIC_ALARM` | [`resource/aws_cloudwatch_metric_alarm`](../../../packages/types/src/resource-definitions.ts#L238) | 운영 알람 |

`ArchitectureJson -> DiagramJson` 변환은 [`getDefaultResourceDefinitionByResourceType`](../../../packages/types/src/resource-definitions.ts#L276)으로 domain `ResourceType`의 대표 Terraform resource를 찾는다. Web adapter의 [`mapResourceTypeToTerraform`](../../../apps/web/features/workspace/workspace-ai-diagram-adapter.ts#L703)이 이 경로다.

반대로 board/API 분석 경로는 [`getResourceDefinitionByTerraform`](../../../packages/types/src/resource-definitions.ts#L282)으로 Terraform identity를 domain `ResourceType`으로 되돌린다. Web의 [`mapTerraformResourceType`](../../../apps/web/features/workspace/workspace-ai-diagram-adapter.ts#L714)과 API의 [`mapTerraformResourceType`](../../../apps/api/src/services/diagram-to-architecture.ts#L49)이 같은 helper를 쓴다.

`UNKNOWN`은 Web adapter에서만 [`unknown_resource`](../../../apps/web/features/workspace/workspace-ai-diagram-adapter.ts#L25) fallback으로 남긴다. 생성 builder는 지원하지 않는 요구사항을 `UNKNOWN` node로 만들지 않고 warning으로 대체/제외한다.

아이콘, label, 기본 크기 같은 presentation은 Web [`resourceCatalog`](../../../apps/web/features/resource-settings/catalog.ts#L1)의 책임이다. provider, Terraform identity, `terraformPreview`/`terraformSync`/`parameterPanel` capability는 shared `ResourceDefinition`의 책임이다.

catalog 생성/검증 스크립트는 [`loadTsModule`](../../../scripts/generate-terraform-aws-catalog.mjs#L412)에서 파일 기준 [`createRequire(filePath)`](../../../scripts/generate-terraform-aws-catalog.mjs#L424)를 쓴다. 그래서 Web catalog를 VM으로 읽을 때도 `@sketchcatch/types/resource-definitions` subpath export를 같은 기준으로 해석한다.

## 9. 운영 조건 반영

[`aiArchitectureOperatingConditions.ts`](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts#L8)는 자연어에서 계산된 [`operatingProfile`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L401)을 config에 반영한다.

| 자연어 단서 | 반영 예시 |
| --- | --- |
| `저렴하게`, `처음엔`, `연습용` | [`lowBudgetKeywords`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L405)가 `budgetLevel: "low"`를 만들고, [`selectEc2InstanceType`](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts#L104), [`selectRdsInstanceClass`](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts#L108), [`selectCloudFrontPriceClass`](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts#L112), [`S3 forceDestroy`](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts#L59)에 반영 |
| `방문자 증가`, `공개 서비스`, `트래픽` | [`growthKeywords`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L406)가 `trafficLevel: "normal"`을 만들고, [`EC2 monitoring`](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts#L41), [`RDS allocatedStorage`](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts#L51), [`Lambda timeout/memory`](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts#L80)에 반영 |
| `로그인`, `개인정보`, `보호` | [`highSecurityKeywords`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L407)가 `securityPriority: "high"`를 만들고, [`applyHighSecurityConfig`](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts#L116), [`RDS deletionProtection`](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts#L52), [`guardrail_adjusted_config`](../../../apps/api/src/services/aiArchitectureDraftMetadata.ts#L46)에 반영 |

즉 같은 “웹사이트 배포”라도 파일 업로드, 로그인, 비용 우선, 방문자 증가 같은 단서가 있으면 리소스와 config가 달라진다.

## 10. Deterministic 보장 방식

구조 결정은 LLM에 맡기지 않는다. backend rule이 다음 값을 안정적으로 고정한다.

1. keyword matching 결과: [`REQUIREMENT_FACT_KEYWORD_RULES`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L92)
2. fact 정렬 순서: [`sortRequirementFacts`](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts#L380)
3. node id와 edge id: [`addNode`](../../../apps/api/src/services/aiArchitectureRequirementDraftBuilder.ts#L604), [`addEdge`](../../../apps/api/src/services/aiArchitectureRequirementDraftBuilder.ts#L610)
4. layout 좌표: [`addStaticWebsiteDelivery`의 `positionX`/`positionY`](../../../apps/api/src/services/aiArchitectureRequirementDraftBuilder.ts#L64)처럼 builder가 고정
5. config 기본값과 운영 조건별 override: [`addStaticWebsiteDelivery` config](../../../apps/api/src/services/aiArchitectureRequirementDraftBuilder.ts#L66), [`applySizingAndTrafficConfig`](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts#L31), [`applyHighSecurityConfig`](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts#L116)

LLM은 [`llmExplanation`](../../../packages/types/src/index.ts#L831)을 보강하는 설명 레이어일 뿐이다. route는 [`addArchitectureDraftLlmExplanation`](../../../apps/api/src/routes/ai.ts#L274)에서 설명을 붙이고, LLM이 없을 때는 [`createArchitectureDraftFallbackExplanation`](../../../apps/api/src/services/aiLlmExplanationFallbacks.ts#L13)으로 metadata 기반 설명을 만든다. 이 fallback은 [`architecture-draft returns fallback llmExplanation` 테스트](../../../apps/api/src/routes/aiLlmExplanationRoutes.test.ts#L111)로 확인한다.

동등한 요구사항을 다르게 말한 prompt는 같은 fact 조합으로 정규화되어 같은 `ArchitectureJson`을 반환해야 한다. 이 조건은 [`same request deterministic` 테스트](../../../apps/api/src/routes/ai.test.ts#L650)와 [`equivalent wording` 테스트](../../../apps/api/src/routes/ai.test.ts#L678)로 확인한다.

## 11. Preview와 적용 경계

초안 생성이 성공하면 frontend는 [`convertArchitectureJsonToDiagramJson`](../../../apps/web/features/workspace/workspace-ai-diagram-adapter.ts#L47)으로 `ArchitectureJson`을 `DiagramJson`으로 변환해 [`context.setPreviewDiagram`](../../../apps/web/features/workspace/WorkspaceAiChatDock.tsx#L311)에 넣는다. 이 preview는 보기 전용이며 기존 보드를 바로 바꾸지 않는다.

```text
preview 표시
-> 사용자가 생성 클릭
-> context.applyDiagramJson(...)
-> 실제 workspace board 전체 교체
```

현재 생성 모드는 전체 교체다. 실제 변경은 [`applyDraftToBoard`](../../../apps/web/features/workspace/WorkspaceAiChatDock.tsx#L331)에서 [`context.applyDiagramJson`](../../../apps/web/features/workspace/WorkspaceAiChatDock.tsx#L336)을 호출할 때만 일어난다. 기존 보드가 있으면 [`board_replacement_required`](../../../apps/web/features/workspace/WorkspaceAiChatDock.tsx#L621) warning을 카드 하단에 표시한다. 부분 patch preview/apply는 다음 단계 작업으로 남아 있다.

## 14. 주의사항

- 시나리오 점수로 하나를 고르는 방식으로 되돌리면 안 된다.
- `selectedDraftPattern`은 대표 라벨이지 생성 기준이 아니다.
- `ResourceType -> Terraform resourceType` map을 Web/API adapter에 다시 하드코딩하지 않는다. 새 리소스는 shared `ResourceDefinition`에 먼저 추가한다.
- `ResourceDefinition.resourceType`은 domain classification이고, Terraform Preview identity는 `terraform.blockType`과 `terraform.resourceType`이다.
- icon, label, size 같은 presentation은 Web catalog 책임이고, provider/Terraform identity/capability는 shared `ResourceDefinition` 책임이다.
- Redis는 내부 Runtime Cache이며 사용자 Practice Architecture 리소스로 만들지 않는다.
- 지원하지 않는 리소스를 `UNKNOWN` node로 만들지 않는다.
- 질문이 남아 있으면 preview를 표시하지 않는다.
- 현재 preview apply는 전체 교체이며, patch 방식은 별도 구현 대상이다.
- AWS-first 리소스를 사용하지만 SketchCatch 제품 설명은 provider-neutral IaC operations service 기준을 유지한다.
