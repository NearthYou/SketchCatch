# 자연어 다이어그램 생성 실행 흐름 정리

이 문서는 `feat/ck/141-Natural-Language-Diagramming` 브랜치에서 구현한 자연어 기반 Architecture Draft 생성 흐름을 처음 보는 사람이 코드 흐름을 따라갈 수 있게 정리한 문서다.

목표는 세 가지다.

1. 사용자가 채팅창에 요구사항을 입력했을 때 어떤 프론트엔드 함수와 API, backend service가 순서대로 호출되는지 알 수 있게 한다.
2. 자연어 우선, 보조 선택, 추가 질문, 미리보기, 사용자 승인 적용의 경계를 분명히 한다.
3. 지원 가능한 리소스만 사용하고, 부족하거나 위험한 조건은 질문 또는 guardrail metadata로 처리하는 방식을 설명한다.

## 1. 자연어 다이어그램 생성 흐름은 무엇인가

자연어 다이어그램 생성은 사용자의 말에서 Practice Architecture 초안을 만드는 기능이다.

AI가 곧바로 보드를 바꾸지는 않는다. 초안은 먼저 `ArchitectureJson`으로 만들어지고, 프론트엔드는 이것을 반투명 preview diagram으로 보여준다. 실제 보드 교체는 사용자가 생성 버튼을 눌렀을 때만 일어난다.

```text
사용자 자연어 입력
-> 프론트엔드 질문 흐름 확인
-> API에 Architecture Draft 요청
-> deterministic scenario resolver가 시나리오 결정
-> 지원 템플릿으로 ArchitectureJson 생성
-> 운영 조건과 guardrail metadata 적용
-> 선택적으로 LLM 설명 보강
-> 프론트엔드가 follow-up 질문 또는 preview 표시
-> 사용자 승인 후 workspace board에 적용
```

현재 구현은 AWS-first 템플릿을 사용한다. 다만 SketchCatch 제품 방향은 provider-neutral IaC operations service이므로, 이 기능을 AWS-only 제품으로 설명하지 않는다.

## 2. 전체 구조

자연어 Architecture Draft 흐름은 5개 층으로 나뉜다.

| 층 | 위치 | 책임 |
| --- | --- | --- |
| Workspace Chat UI | [WorkspaceAiChatDock.tsx](../../../apps/web/features/workspace/WorkspaceAiChatDock.tsx) | 오른쪽 하단 채팅 dock, 채팅 기록, 질문/답변, preview action |
| Frontend helper | [workspace-ai-clarification.ts](../../../apps/web/features/workspace/workspace-ai-clarification.ts), [workspace-ai-draft-follow-up.ts](../../../apps/web/features/workspace/workspace-ai-draft-follow-up.ts) | 부족한 요구사항 질문, guardrail follow-up 질문 |
| API client | [api.ts](../../../apps/web/features/workspace/api.ts) | `POST /api/ai/architecture-draft` 호출 |
| API route | [ai.ts](../../../apps/api/src/routes/ai.ts) | request validation, draft service 호출, LLM explanation 연결 |
| Backend draft service | [aiArchitectureDrafts.ts](../../../apps/api/src/services/aiArchitectureDrafts.ts) | scenario 결정, 템플릿 생성, 운영 조건, guardrail metadata 조합 |

이 흐름에서 실제 cloud mutation은 일어나지 않는다.

Terraform apply, destroy, Git/CI/CD handoff는 별도의 사용자 승인된 Deployment 흐름에서 다룬다.

## 3. 사용자가 보는 흐름

사용자는 workspace 화면 오른쪽 하단의 AI 버튼을 눌러 채팅 dock을 연다.

채팅 dock 위에는 보조 선택이 있다.

| 선택 | 내부 값 | 역할 |
| --- | --- | --- |
| 만들 구조 | `scenarioHint` | 자연어가 부족할 때 시나리오 힌트로 사용된다. 자연어가 명확하면 자연어가 우선한다. |
| 예산 | `budgetLevel` | EC2/RDS 크기, CloudFront price class 같은 config 차이에 반영된다. |
| 방문자 규모 | `trafficLevel` | 트래픽 규모에 맞는 sizing/config 차이에 반영된다. |
| 보호 수준 | `securityPriority` | 암호화, public access 차단 같은 high security config에 반영된다. |

사용자 흐름은 다음과 같다.

```text
채팅 dock 열기
-> 자연어 요구사항 입력
-> 단서가 부족하면 쉬운 질문에 답변
-> 구현 리스트 확인
-> 진행 의사를 밝히면 draft 요청
-> 추가 follow-up이 있으면 먼저 답변
-> board 위에 preview 확인
-> 생성 버튼으로 전체 교체 적용
```

현재 생성은 patch가 아니라 전체 교체다. 그래서 preview 카드에는 기존 보드가 교체될 수 있다는 경고가 붙는다.

## 4. 프론트엔드 질문 흐름

프론트엔드는 먼저 자연어가 바로 draft로 보낼 수 있는 입력인지 확인한다.

핵심 함수는 [workspace-ai-clarification.ts](../../../apps/web/features/workspace/workspace-ai-clarification.ts)에 있다.

| 함수 | 역할 |
| --- | --- |
| `needsArchitectureClarification` | 자연어와 `scenarioHint`만으로 초안을 만들 수 있는지 판단한다. |
| `createArchitectureClarificationSession` | 초보자용 질문 세션을 시작한다. |
| `getCurrentArchitectureClarificationQuestion` | 현재 물어볼 질문을 반환한다. |
| `answerArchitectureClarification` | 사용자의 답변을 session 상태에 반영한다. |
| `createArchitectureClarificationSummaryMessage` | 질문이 끝났을 때 구현 리스트를 채팅 메시지로 정리한다. |
| `createClarifiedDraftRequest` | 답변을 `CreateArchitectureDraftRequest`로 바꾼다. |

예를 들어 `웹사이트 하나 배포하고 싶어`는 아키텍처 대상은 있지만 단서가 부족한 문장이다.

이때 곧바로 S3와 CloudFront만 있는 초안을 만들지 않고, 어떤 종류의 웹사이트인지, 데이터 저장이 필요한지, 운영 우선순위가 무엇인지 쉬운 말로 물어본다.

질문에는 `S3`, `public bucket`, `VPC` 같은 전문 용어를 직접 앞세우지 않는다. 초보자가 답할 수 있는 사용자 언어를 우선한다.

## 5. 프론트엔드 draft 요청 흐름

채팅 dock의 중심 컴포넌트는 [WorkspaceAiChatDock.tsx](../../../apps/web/features/workspace/WorkspaceAiChatDock.tsx)다.

주요 상태는 다음과 같다.

| 상태 | 의미 |
| --- | --- |
| `messages` | 사용자와 AI의 채팅 기록이다. project별 localStorage key로 저장된다. |
| `scenarioHint`, `budgetLevel`, `trafficLevel`, `securityPriority` | 보조 선택값이다. |
| `draft` | 현재 preview 가능한 `AiArchitectureDraftResult`다. |
| `clarificationSession` | 초보자용 질문이 진행 중인 상태다. |
| `draftFollowUpSession` | guardrail 때문에 preview 전에 추가 확인이 필요한 상태다. |
| `lastDraftRequest` | 다시 생성할 때 사용할 마지막 요청값이다. |

실행 순서는 다음과 같다.

```text
handleSubmit
-> clarificationSession이 있으면 답변 처리
-> 없으면 needsArchitectureClarification 확인
-> 바로 생성 가능하면 createDraftFromConversation
-> createAiArchitectureDraft 호출
-> planArchitectureDraftPreview로 follow-up 필요 여부 확인
-> showDraftPreview 또는 추가 질문 표시
```

`showDraftPreview`는 `ArchitectureJson`을 `DiagramJson`으로 변환하고 `context.setPreviewDiagram`을 호출한다.

이 시점의 preview는 보기만 가능하다. 사용자가 생성 버튼을 눌러야 `context.applyDiagramJson`이 호출된다.

## 6. API route 흐름

Architecture Draft API는 [apps/api/src/routes/ai.ts](../../../apps/api/src/routes/ai.ts)에 있다.

```text
POST /api/ai/architecture-draft
```

request body는 `architectureDraftBodySchema`로 검증된다.

| 필드 | 허용값 |
| --- | --- |
| `prompt` | 비어 있지 않은 자연어 요구사항 |
| `scenarioHint` | `auto`, `static_site`, `api_server`, `backend_with_db`, `server_storage`, `serverless_function` |
| `budgetLevel` | `low`, `normal` |
| `trafficLevel` | `small`, `normal` |
| `securityPriority` | `basic`, `high` |

route는 검증된 요청을 `createArchitectureDraft`에 넘긴다.

이후 `addArchitectureDraftLlmExplanation`이 Bedrock/fallback provider 설명을 붙인다.

중요한 경계는 여기에도 있다. LLM 설명은 보조 설명일 뿐이고, `ArchitectureJson` 결정은 deterministic service가 만든다.

## 7. Backend draft service 흐름

핵심 진입점은 [aiArchitectureDrafts.ts](../../../apps/api/src/services/aiArchitectureDrafts.ts)의 `createArchitectureDraft`다.

실행 순서는 다음과 같다.

```text
createArchitectureDraft
-> normalizeArchitectureDraftRequest
-> resolveScenario
-> createDraftByScenario
-> applyOperatingConditionConfig
-> applyGuardrailMetadata
-> AiArchitectureDraftResult 반환
```

각 단계의 책임은 분리되어 있다.

| 단계 | 파일 | 책임 |
| --- | --- | --- |
| 요청 정규화 | [aiArchitectureDrafts.ts](../../../apps/api/src/services/aiArchitectureDrafts.ts) | 문자열 입력과 request object를 같은 형태로 맞춘다. |
| 시나리오 결정 | [aiArchitectureScenarioResolution.ts](../../../apps/api/src/services/aiArchitectureScenarioResolution.ts) | 자연어, 보조 선택, 미지원 요구사항을 평가한다. |
| 템플릿 생성 | [aiArchitectureDraftTemplates.ts](../../../apps/api/src/services/aiArchitectureDraftTemplates.ts) | 지원 시나리오별 고정 `ArchitectureJson`을 만든다. |
| 운영 조건 반영 | [aiArchitectureOperatingConditions.ts](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts) | budget, traffic, security 선택을 config에 반영한다. |
| metadata 보강 | [aiArchitectureDraftMetadata.ts](../../../apps/api/src/services/aiArchitectureDraftMetadata.ts) | assumptions, explanations, guardrail warnings를 붙인다. |

이 구조 때문에 같은 prompt와 같은 보조 조건은 같은 `ArchitectureJson`을 반환한다.

## 8. 시나리오 결정 규칙

시나리오 결정은 자연어가 우선이다.

보조 선택은 자연어가 애매할 때만 빈칸을 채우는 힌트로 사용된다.

```text
자연어에 명확한 구조 단서가 있음
-> 자연어 기준 scenario 선택
-> 보조 선택과 충돌하면 selection_overridden_by_prompt 경고

자연어가 아키텍처 관련이지만 구조 단서가 부족함
-> scenarioHint가 auto가 아니면 힌트로 사용
-> scenarioHint도 auto이면 프론트 질문 흐름 또는 API 거절

아키텍처와 무관함
-> draft를 만들지 않고 bad_request
```

예를 들어 `웹사이트 하나 배포하고 싶어`는 질문 흐름으로 넘어가야 한다.

반대로 `정적 웹사이트를 전 세계에 빠르게 보여주고 싶어`처럼 정적 사이트와 CDN 단서가 있으면 `static_site`로 갈 수 있다.

## 9. 지원 시나리오별 템플릿

현재 브랜치에서 직접 생성하는 시나리오는 다섯 가지다.

| 시나리오 | 생성되는 주요 구조 |
| --- | --- |
| `static_site` | S3 origin과 CloudFront CDN으로 정적 웹사이트를 제공하는 구조 |
| `api_server` | VPC, Subnet, Internet Gateway, Route Table, Security Group, AMI, EC2, IAM, Logs, Alarm을 가진 API 서버 구조 |
| `backend_with_db` | 애플리케이션 서버와 DB subnet, RDS, KMS, IAM, Logs, Alarm을 포함한 백엔드 구조 |
| `server_storage` | EC2 서버와 S3 Bucket, VPC/Subnet/라우팅/security boundary 관계를 포함한 서버+저장소 구조 |
| `serverless_function` | API Gateway, Lambda, Lambda Permission, IAM, KMS, CloudWatch Logs, Alarm을 포함한 서버리스 구조 |

템플릿은 지원하는 `ResourceType`만 사용한다.

지원하지 않는 리소스를 노드로 억지 생성하지 않고, 가능한 경우 지원 가능한 구조로 대체하거나 guardrail warning을 남긴다.

## 10. 미지원 요구사항 처리

미지원 요구사항은 두 방식으로 처리한다.

1. 지원 가능한 대체 구조가 있으면 대체해서 생성한다.
2. 대체해도 의미가 불분명하거나 범위 밖이면 생성하지 않고 경고 또는 오류로 돌린다.

예시는 다음과 같다.

| 사용자가 말한 요구 | 현재 처리 |
| --- | --- |
| EKS, Kubernetes | 단일 EC2 API 서버 구조로 대체하고 대체 경고를 남긴다. |
| ECS, Fargate | 단일 EC2 API 서버 구조로 대체하고 대체 경고를 남긴다. |
| DynamoDB, NoSQL | 지원 가능한 RDS 구조로 대체하고 경고를 남긴다. |
| ALB, Auto Scaling | 단일 서버 구조로 낮춰서 경고를 남긴다. |
| 멀티 리전 | 단일 리전 초안으로 제한하고 경고를 남긴다. |
| Redis, Queue, CI/CD 자동 구성 | 현재 draft 자동 생성 범위에서 제외한다. |

지원 가능한 요구와 미지원 요구가 섞여 있으면 가능한 부분만 생성하고 `partial_generation` warning을 남긴다.

## 11. 운영 조건 반영

보조 선택은 단순 라벨이 아니다.

[aiArchitectureOperatingConditions.ts](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts)에서 실제 config 차이로 반영된다.

| 선택 | 반영 예시 |
| --- | --- |
| 낮은 예산 | 작은 EC2/RDS class, 제한된 CDN price class, 비용 관련 warning |
| 보통 예산 | 기본 sizing/config |
| 적은 방문자 | 작은 capacity 가정 |
| 보통 방문자 | 더 큰 instance 또는 운영 config |
| 기본 보호 | MVP 기본 보안 설정 |
| 강화된 보호 | 암호화, public access block, deletion protection 성격의 config 보강 |

이 때문에 같은 자연어라도 예산, 방문자 규모, 보호 수준이 다르면 일부 리소스 config가 달라질 수 있다.

## 12. Guardrail metadata와 follow-up 질문

Backend는 draft result의 `metadata.guardrailWarnings`에 warning을 담는다.

프론트엔드는 이 warning을 무조건 카드 하단 경고로만 띄우지 않는다.

[workspace-ai-draft-follow-up.ts](../../../apps/web/features/workspace/workspace-ai-draft-follow-up.ts)의 `planArchitectureDraftPreview`가 먼저 판단한다.

```text
fixable warning이 있음
-> preview를 띄우지 않음
-> 채팅에서 추가 질문
-> 사용자가 답변
-> resolveArchitectureDraftFollowUpAnswer
-> 수정된 request로 다시 draft 생성

바로 보여줘도 되는 warning만 있음
-> preview 표시
-> 카드 하단에 한국어 경고 표시
```

예를 들어 낮은 예산인데 DB 구조가 선택되면, 이미 preview를 만들었다는 식으로 말하지 않고 `DB 없는 구조로 만들까요?`처럼 앞으로 만들 구조를 묻는다.

## 13. ArchitectureJson에서 DiagramJson으로 변환

Backend는 `ArchitectureJson`을 반환한다.

Workspace board는 `DiagramJson`을 사용하므로 프론트엔드에서 변환이 필요하다.

변환은 [workspace-ai-diagram-adapter.ts](../../../apps/web/features/workspace/workspace-ai-diagram-adapter.ts)의 `convertArchitectureJsonToDiagramJson`이 담당한다.

여기서 중요한 처리는 세 가지다.

1. 지원 Resource Catalog에 있는 리소스만 실제 diagram node로 매핑한다.
2. Region, VPC, Subnet, Security Group처럼 영역으로 보여야 하는 것은 area containment가 드러나게 배치한다.
3. 포함관계 edge는 지저분한 화살표로 렌더링하지 않고, 사용/호출/라우팅/모니터링 관계는 edge로 표시한다.

Security Group은 단순 박스가 아니라 보호 대상 리소스를 감싸는 area로 보여야 한다.

그래서 EC2가 Security Group 안에 있다는 관계는 화살표보다 containment가 우선이다.

## 14. Preview와 적용 경계

Diagram editor는 `previewDiagram`을 별도 상태로 가진다.

```text
draft 생성 완료
-> context.setPreviewDiagram(previewDiagram)
-> 화면은 previewDiagram을 표시
-> preview 상태의 node/edge는 read-only
-> 사용자가 생성 버튼 클릭
-> context.applyDiagramJson(...)
-> workspace board 실제 교체
```

이 경계가 중요한 이유는 User-Accepted Changes 원칙 때문이다.

AI가 만든 초안이 마음에 들지 않으면 사용자는 취소하거나 다시 생성할 수 있다.

AI는 보드를 직접 바꾸지 않고, 사용자가 승인한 시점에만 보드가 바뀐다.

## 15. LLM 설명 연결

Architecture Draft 자체는 deterministic rule로 만들어진다.

그 뒤 [ai.ts](../../../apps/api/src/routes/ai.ts)의 `addArchitectureDraftLlmExplanation`이 설명을 붙인다.

```text
deterministic Architecture Draft
-> LLM explanation provider 호출
-> 성공하면 llmExplanation 추가
-> 실패하면 fallback 설명
```

LLM 설명은 사용자가 초안을 이해하도록 돕는 설명이다.

리소스 선택, 관계, config의 최종 결정권은 `ArchitectureJson`을 만든 deterministic service에 있다.

## 16. Source Repository 기반 draft와의 관계

`POST /api/ai/github-architecture-draft`도 최종적으로 같은 Architecture Draft 결과 형태를 사용한다.

route는 GitHub repository URL에서 evidence를 모은 뒤 `createArchitectureDraftFromRepositoryEvidence`를 호출한다.

즉 자연어 입력과 repository evidence 입력은 출발점은 다르지만, 사용자가 확인하는 결과는 `AiArchitectureDraftResult`라는 같은 계약을 따른다.

## 17. Patch preview와 현재 생성 흐름의 차이

현재 workspace 자연어 생성은 전체 교체 흐름이다.

`ArchitecturePatchPreview` 타입과 backend preview helper는 있지만, workspace UI에서 부분 patch를 사용자 승인 후 적용하는 기능은 아직 본격 흐름으로 열려 있지 않다.

| 구분 | 현재 구현 |
| --- | --- |
| 생성 | 전체 Architecture Draft를 만들고 preview 후 전체 교체 |
| patch preview | 타입과 service 경계는 있으나 workspace 부분 적용 UX는 후속 작업 |
| 승인 경계 | 생성과 patch 모두 사용자 승인 후 적용해야 한다는 원칙은 동일 |

## 20. 현재 주의할 점

- 이 기능은 AWS-first 템플릿을 사용하지만 SketchCatch를 AWS-only로 설명하면 안 된다.
- `Redis`는 내부 Runtime Cache이며 사용자 Practice Architecture Resource로 생성하지 않는다.
- 실제 Terraform apply/destroy나 cloud mutation은 이 기능에서 실행하지 않는다.
- 지원하지 않는 리소스는 억지로 만들지 않는다. 가능한 경우 대체하고, 그렇지 않으면 제외 또는 질문으로 처리한다.
- 질문이 남아 있으면 preview를 띄우지 않는다.
- 현재 생성은 전체 교체이므로 기존 보드가 바뀌는 위험을 사용자에게 알려야 한다.
- patch 방식은 후속 작업이다. patch도 반드시 preview와 사용자 승인 경계를 따라야 한다.

## 21. 진짜 핵심 요약

자연어 다이어그램 생성은 AI가 마음대로 보드를 그리는 기능이 아니다.

사용자의 자연어를 deterministic rule이 해석하고, 지원 가능한 리소스 템플릿으로 `ArchitectureJson` 초안을 만든다.

자연어가 명확하면 자연어가 보조 선택보다 우선한다.

자연어가 부족하면 보조 선택이나 쉬운 질문으로 정보를 채운다.

미지원 리소스는 지원 가능한 구조로 대체하거나 제외하고, 그 사실을 metadata와 UI 질문/경고로 드러낸다.

생성된 초안은 바로 적용되지 않고 preview로만 표시된다.

최종 보드 변경은 사용자가 생성 버튼을 눌렀을 때만 일어난다.
