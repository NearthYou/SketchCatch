# 자연어 다이어그램 생성 실행 흐름 정리

이 문서는 현재 브랜치에서 구현한 Natural Language Diagramming 흐름을 코드 기준으로 따라가기 위한 정리다.

핵심은 사용자의 문장을 고정 시나리오 목록 중 하나로 고르는 것이 아니다. 자연어 안에서 아키텍처 단서를 뽑아 `requirementFacts`로 정규화하고, 그 fact 조합으로 지원 가능한 리소스만 사용해 deterministic `ArchitectureJson` 초안을 만든다.

## 1. 목표

1. 사용자는 채팅창에 자연어 요구사항만 입력한다.
2. 요구사항이 부족하면 초안을 만들지 않고 쉬운 질문과 추천 답변을 차례대로 보여준다.
3. 충분한 단서가 모이면 지원 가능한 `ResourceType`만 사용해 `ArchitectureJson`을 만든다.
4. 초안은 workspace board에 반투명 preview로만 표시하고, 사용자가 `생성`을 눌러야 실제 board에 적용한다.
5. 같은 요구사항 단서 조합은 같은 node id, edge id, config, layout을 가진 같은 초안을 만든다.

## 2. 전체 흐름

```text
사용자 자연어 입력
-> Workspace AI Chat Dock이 모호한 요구사항인지 확인
-> 질문이 필요하면 draft API를 호출하지 않고 질문/추천 답변 표시
-> 답변이 모두 모이면 구현 리스트 확인
-> POST /api/ai/architecture-draft
-> backend가 requirementFacts, operatingProfile, guardrailWarnings 계산
-> requirementFacts 조합으로 지원 리소스만 조립
-> 운영 조건 config와 guardrail metadata 적용
-> 선택적으로 LLM 설명 보강
-> frontend가 preview diagram 표시
-> 사용자 생성 승인 후 workspace board 전체 교체
```

이 흐름에서 Terraform apply, cloud mutation, Git/CI/CD handoff는 일어나지 않는다. AI 초안은 `User-Accepted Change` 경계를 넘기 전까지 실제 프로젝트 상태를 바꾸지 않는다.

## 3. 요청 계약

`CreateArchitectureDraftRequest`는 `prompt`만 받는다.

```ts
type CreateArchitectureDraftRequest = {
  prompt: string;
};
```

`scenarioHint`, `budgetLevel`, `trafficLevel`, `securityPriority` 같은 보조 선택값은 초안 생성 계약에서 제거되었다. 예산, 방문자 규모, 보호 수준은 사용자가 자연어로 말한 단서에서 backend가 `operatingProfile`로 계산한다.

legacy client가 예전 field를 보내더라도 생성 기준으로 사용하지 않는다. 자연어 `prompt`가 우선이다.

## 4. Frontend 책임

| 위치 | 책임 |
| --- | --- |
| [WorkspaceAiChatDock.tsx](../../../apps/web/features/workspace/WorkspaceAiChatDock.tsx) | 오른쪽 하단 채팅 dock, 채팅 기록, 자연어 입력, 질문 흐름, preview/apply/cancel/regenerate 처리 |
| [workspace-ai-clarification.ts](../../../apps/web/features/workspace/workspace-ai-clarification.ts) | 모호한 요구사항을 초보자도 답할 수 있는 질문으로 바꾸고, 답변을 자연어 prompt로 재구성 |
| [workspace-ai-draft-follow-up.ts](../../../apps/web/features/workspace/workspace-ai-draft-follow-up.ts) | guardrail warning 중 바로 preview하지 말고 되물어야 하는 경우를 처리 |
| [workspace-ai-diagram-adapter.ts](../../../apps/web/features/workspace/workspace-ai-diagram-adapter.ts) | backend `ArchitectureJson`을 board용 `DiagramJson`으로 변환 |

채팅 dock은 프로젝트별 `localStorage` key로 최근 대화 기록을 보존한다.

사용자가 질문에 답하는 중이면 draft API를 호출하지 않는다. 모든 질문이 끝나고 구현 리스트를 확인받은 뒤에만 초안을 요청한다.

## 5. API 흐름

Architecture Draft API는 [apps/api/src/routes/ai.ts](../../../apps/api/src/routes/ai.ts)의 다음 endpoint다.

```text
POST /api/ai/architecture-draft
```

route는 `prompt`를 검증한 뒤 [aiArchitectureDrafts.ts](../../../apps/api/src/services/aiArchitectureDrafts.ts)의 `createArchitectureDraft`를 호출한다.

```text
createArchitectureDraft
-> normalizeArchitectureDraftRequest
-> resolveArchitectureRequirement
-> createDraftFromRequirementFacts
-> applyOperatingConditionConfig
-> applyGuardrailMetadata
-> AiArchitectureDraftResult 반환
```

`resolveArchitectureRequirement`는 fixed scenario score를 고르는 단계가 아니다. 자연어에서 `requirementFacts`, `operatingProfile`, 지원 불가/대체 warning을 뽑는 단계다.

`selectedDraftPattern`은 UI와 LLM 설명을 위한 대표 라벨이다. 실제 리소스 생성은 `selectedDraftPattern`이 아니라 `requirementFacts` 기준으로 동작한다.

## 6. 요구사항 fact 해석

[aiArchitectureRequirementResolution.ts](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts)는 prompt에서 다음 단서를 찾는다.

```ts
type ArchitectureRequirementFact =
  | "web_frontend"
  | "static_delivery"
  | "server_runtime"
  | "database"
  | "object_storage"
  | "file_upload"
  | "auth_or_user_data"
  | "serverless_runtime"
  | "network_boundary"
  | "iam_permissions"
  | "observability"
  | "encryption";
```

예시는 다음과 같다.

| 자연어 표현 | 해석되는 단서 |
| --- | --- |
| `소개용 랜딩 웹사이트`, `정적 사이트` | `web_frontend`, `static_delivery`, `object_storage` |
| `서버`, `API`, `백엔드` | `server_runtime`, `network_boundary`, `iam_permissions`, `observability` |
| `파일 업로드`, `이미지 저장` | `file_upload`, `object_storage`, `server_runtime` |
| `로그인`, `회원`, `개인정보` | `auth_or_user_data`, `database`, `encryption`, `server_runtime` |
| `Lambda`, `서버리스 함수` | `serverless_runtime`, `iam_permissions`, `observability` |
| `DB 없이`, `데이터베이스 없는` | database 관련 fact 제거 |

`웹사이트 하나 배포하고 싶어`처럼 대상은 맞지만 화면만 필요한지, 파일 업로드가 필요한지, 로그인/데이터 저장이 필요한지 알 수 없는 문장은 초안을 생성하지 않는다. 이 경우 frontend가 전문 용어 대신 사용자 언어로 질문한다.

반대로 `웹사이트를 S3와 CloudFront로 배포하고 싶어`처럼 명확한 아키텍처 단서가 있으면 추가 질문 없이 초안을 요청할 수 있다.

## 7. 리소스 조립

[aiArchitectureRequirementDraftBuilder.ts](../../../apps/api/src/services/aiArchitectureRequirementDraftBuilder.ts)는 fact 조합을 보고 필요한 리소스를 조립한다.

| fact | 대표 생성 리소스 |
| --- | --- |
| `static_delivery` | `S3`, `CLOUDFRONT` |
| `server_runtime` | `VPC`, `SUBNET`, `INTERNET_GATEWAY`, `ROUTE_TABLE`, `SECURITY_GROUP`, `AMI`, `EC2` |
| `database` | DB subnet, DB security group, `RDS`, `KMS_KEY`, DB alarm |
| `object_storage` / `file_upload` | upload용 `S3` bucket, runtime permission edge |
| `serverless_runtime` | `API_GATEWAY_REST_API`, `LAMBDA`, `LAMBDA_PERMISSION`, execution role, logs, alarm |
| `iam_permissions` | `IAM_ROLE`, `IAM_POLICY`, `IAM_INSTANCE_PROFILE` |
| `observability` | `CLOUDWATCH_LOG_GROUP`, `CLOUDWATCH_METRIC_ALARM` |
| `encryption` | `KMS_KEY`, encrypted storage/log config |

지원하지 않는 리소스를 `UNKNOWN` node로 꾸며내지 않는다.

가능한 경우 지원 리소스로 대체하고 `unsupported_requirement_substituted` warning을 남긴다. 대체할 수 없는 요구사항은 `unsupported_resource_omitted` warning으로 제외하고, 지원 가능한 부분만 생성한다.

## 8. 운영 조건 반영

[aiArchitectureOperatingConditions.ts](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts)는 자연어에서 계산된 `operatingProfile`을 config에 반영한다.

| 자연어 단서 | 반영 예시 |
| --- | --- |
| `저렴하게`, `처음엔`, `연습용` | 작은 EC2/RDS class, 낮은 CloudFront price class, 짧은 log retention, S3 force destroy |
| `방문자 증가`, `공개 서비스`, `트래픽` | 더 큰 instance/storage, 모니터링 강화 |
| `로그인`, `개인정보`, `보호` | public access block, deletion protection, encryption, 보수적인 security group config |

즉 같은 “웹사이트 배포”라도 파일 업로드, 로그인, 비용 우선, 방문자 증가 같은 단서가 있으면 리소스와 config가 달라진다.

## 9. Deterministic 보장 방식

구조 결정은 LLM에 맡기지 않는다. backend rule이 다음 값을 안정적으로 고정한다.

1. keyword matching 결과
2. fact 정렬 순서
3. node id와 edge id
4. layout 좌표
5. config 기본값과 운영 조건별 override

LLM은 `llmExplanation`을 보강하는 설명 레이어일 뿐이다. LLM provider가 실패해도 `ArchitectureJson`은 rule 기반 결과로 반환된다.

동등한 요구사항을 다르게 말한 prompt는 같은 fact 조합으로 정규화되어 같은 `ArchitectureJson`을 반환해야 한다. 이 조건은 [apps/api/src/routes/ai.test.ts](../../../apps/api/src/routes/ai.test.ts)의 equivalent wording 테스트로 확인한다.

## 10. Preview와 적용 경계

초안 생성이 성공하면 frontend는 `ArchitectureJson`을 `DiagramJson`으로 변환해 `context.setPreviewDiagram`에 넣는다. 이 preview는 보기 전용이며 기존 보드를 바로 바꾸지 않는다.

```text
preview 표시
-> 사용자가 생성 클릭
-> context.applyDiagramJson(...)
-> 실제 workspace board 전체 교체
```

현재 생성 모드는 전체 교체다. 기존 보드가 있으면 `board_replacement_required` warning을 카드 하단에 표시한다. 부분 patch preview/apply는 다음 단계 작업으로 남아 있다.

## 13. 주의사항

- 시나리오 점수로 하나를 고르는 방식으로 되돌리면 안 된다.
- `selectedDraftPattern`은 대표 라벨이지 생성 기준이 아니다.
- Redis는 내부 Runtime Cache이며 사용자 Practice Architecture 리소스로 만들지 않는다.
- 지원하지 않는 리소스를 `UNKNOWN` node로 만들지 않는다.
- 질문이 남아 있으면 preview를 표시하지 않는다.
- 현재 preview apply는 전체 교체이며, patch 방식은 별도 구현 대상이다.
- AWS-first 리소스를 사용하지만 SketchCatch 제품 설명은 provider-neutral IaC operations service 기준을 유지한다.
