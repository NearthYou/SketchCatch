# 브랜치 AI 다이어그램 수정 내용 정리

이 문서는 현재 브랜치 `feat/ck/152-ai-diagram-editing`에서 수정한 AI 다이어그램 생성, 수정, 미리보기, 채팅 UX 흐름을 처음 보는 사람이 코드 기준으로 따라갈 수 있게 정리한 문서다.

목표는 세 가지다.

1. 자연어 요구사항이 어떤 질문, 초안, 패치 미리보기로 이어지는지 알 수 있게 한다.
2. AI가 Board를 바로 바꾸지 않고 preview와 사용자 승인 경계를 지키는 방식을 설명한다.
3. 최소 API 서버, EC2 컨테이너 배치, area 레이어, 채팅 스크롤 같은 이번 브랜치의 UX 보정 지점을 분명히 한다.

## 1. 이번 브랜치의 큰 변화

이번 브랜치는 AI를 "다이어그램을 바로 바꾸는 자동 실행기"가 아니라, 사용자의 자연어를 검토 가능한 초안 또는 패치 preview로 바꾸는 흐름으로 정리했다.

핵심은 아래다.

```text
사용자 자연어 입력
-> 생성 요청인지 수정 요청인지 판단
-> 정보가 부족하면 쉬운 질문으로 확인
-> 충분하면 deterministic ArchitectureJson 또는 patch preview 생성
-> 보드에는 반투명 preview로 표시
-> 사용자가 생성/적용을 누를 때만 실제 Board 상태 변경
-> 적용 직후 즉시 저장
```

이 브랜치에서 중점적으로 바꾼 영역은 다섯 가지다.

| 영역 | 변경 요약 |
| --- | --- |
| 자연어 생성 | 고정 preset 선택이 아니라 자연어 단서에서 requirement fact를 뽑아 지원 가능한 리소스만 조립한다. |
| 생성 전 질문 | "웹사이트"처럼 모호하면 바로 만들지 않고 서비스 목적, 데이터 저장, 파일 업로드, 운영 기준을 쉬운 질문으로 확인한다. |
| 자연어 수정 | 기존 보드가 있을 때 자연어 수정 요청을 patch preview로 만들고, 후보가 애매하면 질문으로 되묻는다. |
| preview 적용 | 생성 초안과 수정 패치는 모두 preview로만 표시하고, 사용자가 승인해야 실제 보드에 적용한다. |
| UX 보정 | EC2 컨테이너 배치, area z-index, AI 채팅 최신 메시지 스크롤, 한국어 문구를 정리했다. |

## 3. 생성과 수정의 분기

프론트의 1차 분기는 [WorkspaceAiChatDock.tsx](../../../apps/web/features/workspace/WorkspaceAiChatDock.tsx)에서 시작한다.

사용자가 AI 채팅창에 입력하면 대략 아래 순서로 처리된다.

```text
submitUserMessage
-> handleUserMessage
-> resolveWorkspaceAiChatAction
-> draft 생성 또는 patch preview 생성
```

분기 기준은 [workspace-ai-chat-routing.ts](../../../apps/web/features/workspace/workspace-ai-chat-routing.ts)에 있다.

| 상황 | 처리 |
| --- | --- |
| 빈 보드에서 새 서비스를 만들고 싶다고 함 | Architecture Draft 생성 흐름 |
| 기존 보드에서 "여기에 DB 추가"처럼 말함 | Architecture Patch Preview 흐름 |
| 기존 보드가 있어도 "새로", "처음부터"처럼 말함 | Draft 생성 흐름 |
| patch 질문 중 새 서비스 생성 요청을 함 | 기존 patch 질문을 끊고 Draft 질문 흐름으로 전환 |

중요한 점:

기존 보드가 있다고 해서 모든 자연어를 수정으로 보지 않는다. 사용자가 새 서비스를 만들려는 말인지, 현재 다이어그램을 고치려는 말인지 먼저 나눈다.

## 4. 자연어 생성 흐름

관련 파일:

- [apps/api/src/services/aiArchitectureDrafts.ts](../../../apps/api/src/services/aiArchitectureDrafts.ts)
- [apps/api/src/services/aiArchitectureRequirementResolution.ts](../../../apps/api/src/services/aiArchitectureRequirementResolution.ts)
- [apps/api/src/services/aiArchitectureRequirementDraftBuilder.ts](../../../apps/api/src/services/aiArchitectureRequirementDraftBuilder.ts)
- [apps/api/src/services/aiArchitectureOperatingConditions.ts](../../../apps/api/src/services/aiArchitectureOperatingConditions.ts)
- [apps/api/src/services/aiArchitectureDraftMetadata.ts](../../../apps/api/src/services/aiArchitectureDraftMetadata.ts)

생성 API 흐름:

```text
POST /api/ai/architecture-draft
-> createArchitectureDraft
-> resolveArchitectureRequirement
-> createDraftFromRequirementFacts
-> applyOperatingConditionConfig
-> applyGuardrailMetadata
-> AiArchitectureDraftResult 반환
```

`resolveArchitectureRequirement`는 자연어에서 아래 단서를 뽑는다.

| fact | 자연어 예시 |
| --- | --- |
| `web_frontend` | 웹사이트, 프론트엔드, 랜딩, 소개 |
| `server_runtime` | API, 서버, 백엔드, EC2 |
| `database` | DB, 데이터베이스, 로그인, 회원, 예약 |
| `object_storage` | S3, 파일, 이미지, 업로드 |
| `serverless_runtime` | Lambda, 서버리스 |
| `observability` | 서버가 있으면 기본 운영 단서로 파생 |
| `encryption` | 개인정보, 로그인, 데이터베이스에서 파생 |

이 fact 조합이 [aiArchitectureRequirementDraftBuilder.ts](../../../apps/api/src/services/aiArchitectureRequirementDraftBuilder.ts)에서 실제 node와 edge로 바뀐다.

## 5. 생성 전 질문 흐름

프론트 질문 흐름은 [workspace-ai-clarification.ts](../../../apps/web/features/workspace/workspace-ai-clarification.ts)가 맡는다.

목적은 사용자가 "웹사이트 하나 만들어줘"처럼 말했을 때 내부 리소스 이름을 묻지 않고 쉬운 말로 확인하는 것이다.

```text
모호한 자연어 입력
-> needsArchitectureClarification
-> createArchitectureClarificationQuestionMessage
-> 사용자가 답변
-> answerArchitectureClarification
-> 모든 질문 완료 시 createClarifiedDraftRequest
```

질문 기준:

| 부족한 정보 | 질문 방향 |
| --- | --- |
| 서비스 목적 | 어떤 서비스인지 묻는다. |
| 데이터 저장 여부 | 로그인, 회원, 예약, 신청 같은 데이터가 필요한지 묻는다. |
| 파일 업로드 여부 | 이미지나 파일 저장이 필요한지 묻는다. |
| 운영 기준 | 작게 시작할지, 방문자 증가를 대비할지, 보호가 중요한지 묻는다. |

이번 브랜치에서는 "추가/삭제/교체/설정 변경 중 어디에 가까운가요?" 같은 내부자식 질문을 줄이고, 사용자가 이해할 수 있는 서비스 목적 질문으로 바꿨다.

## 6. 최소 API 서버 생성

사용자가 "최소한의 API 서버 만들어줘", "간단한 API 서버", "작게 API 서버"처럼 말하면 작은/basic API 서버로 해석한다.

이때 불필요한 리소스를 붙이지 않는다.

포함되는 리소스:

| 리소스 | 이유 |
| --- | --- |
| `VPC` | EC2를 단독으로 두지 않고 네트워크 경계 안에 배치하기 위해 필요하다. |
| `SUBNET` | EC2가 들어갈 public subnet이다. |
| `INTERNET_GATEWAY` | public subnet의 인터넷 진입을 표현한다. |
| `ROUTE_TABLE` | 인터넷 경로를 표현한다. |
| `ROUTE_TABLE_ASSOCIATION` | subnet과 route table의 연결을 표현한다. |
| `SECURITY_GROUP` | HTTP 접근 허용 경계를 표현한다. |
| `AMI` | EC2 launch image 의존성을 표현한다. |
| `EC2` | 실제 API 서버 실행 공간이다. |

포함하지 않는 리소스:

| 제외 리소스 | 제외 이유 |
| --- | --- |
| `CLOUDFRONT` | 최소 API 서버에 CDN/public entry를 과하게 붙이지 않는다. |
| `S3` | 파일 업로드나 정적 배포 요구가 없으면 만들지 않는다. |
| `IAM_ROLE`, `IAM_POLICY`, `IAM_INSTANCE_PROFILE` | S3/Logs/KMS 같은 권한 대상이 없으면 최소 구조에서는 생략한다. |
| `CLOUDWATCH_LOG_GROUP`, `CLOUDWATCH_METRIC_ALARM` | 운영 관측 요구나 확장 구조가 있을 때 붙인다. |
| `RDS` | DB, 로그인, 회원, 예약 같은 데이터 요구가 있을 때만 붙인다. |

중요한 제약:

최소 API 서버여도 EC2는 단독 아이콘으로 만들지 않는다.

```text
Region
-> VPC
-> Public Subnet
-> Security Group
-> EC2 Application Server
```

프론트 변환 단계는 [workspace-ai-diagram-adapter.ts](../../../apps/web/features/workspace/workspace-ai-diagram-adapter.ts)에서 VPC/Subnet/EC2 관계를 board area 구조로 바꾼다.

## 7. 운영형 구조 생성

사용자가 로그인, 파일 업로드, 개인정보 보호, 방문자 증가 같은 단서를 말하면 최소 구조가 아니라 운영형 구조로 확장한다.

예:

```text
로그인 있는 웹사이트를 배포하고 싶어.
사용자가 이미지도 업로드해야 하고
처음엔 저렴하게 시작하되 개인정보는 보호해줘.
```

이 경우 들어가는 흐름:

```text
web_frontend
-> static_delivery
-> server_runtime
-> object_storage
-> database
-> encryption
-> iam_permissions
-> observability
```

생성되는 주요 리소스:

| 리소스 | 의미 |
| --- | --- |
| `CLOUDFRONT` | 공개 웹 진입 또는 정적 파일 배포 진입 |
| `S3` | 정적 웹 자산 또는 사용자 업로드 파일 저장 |
| `EC2` | 애플리케이션 서버 |
| `RDS` | 로그인, 회원, 예약, 신청 같은 데이터 저장 |
| `KMS_KEY` | 데이터 암호화 |
| `IAM_ROLE`, `IAM_POLICY`, `IAM_INSTANCE_PROFILE` | EC2가 S3, Logs, KMS를 사용할 권한 |
| `CLOUDWATCH_LOG_GROUP`, `CLOUDWATCH_METRIC_ALARM` | 로그와 CPU 알람 |
| `SECURITY_GROUP` | 앱과 DB 접근 경계 |

현재 지원하지 않는 타입은 생성하지 않는다.

| 아직 생성하지 않는 타입 |
| --- |
| `ROUTE53_RECORD` |
| `WAF_WEB_ACL` |
| `LOAD_BALANCER`, `LOAD_BALANCER_LISTENER` |
| `DB_SUBNET_GROUP` |
| `SECRETS_MANAGER_SECRET` |
| `VPC_ENDPOINT` |

이 타입들은 나중에 shared `ResourceType`과 Terraform 변환, catalog, UI mapping이 함께 준비될 때 추가해야 한다.

## 8. 자연어 수정 Patch Preview

관련 파일:

- [apps/api/src/services/aiArchitecturePatchPreview.ts](../../../apps/api/src/services/aiArchitecturePatchPreview.ts)
- [apps/web/features/workspace/workspace-ai-patch-preview.ts](../../../apps/web/features/workspace/workspace-ai-patch-preview.ts)
- [apps/web/features/workspace/WorkspaceAiChatDock.tsx](../../../apps/web/features/workspace/WorkspaceAiChatDock.tsx)
- [packages/types/src/index.ts](../../../packages/types/src/index.ts)

기존 보드가 있는 상태에서 사용자가 수정 요청을 하면 바로 board를 바꾸지 않는다.

```text
자연어 수정 요청
-> createArchitecturePatchPreview
-> intent 해석
-> 대상이 애매하면 질문
-> proposedArchitectureJson 생성
-> frontend가 반투명 preview 표시
-> 사용자가 적용 버튼을 누르면 실제 board 반영
```

Patch Preview 응답은 반드시 아래 경계를 가진다.

```ts
requiresUserAcceptance: true
userAcceptedChange: null
```

즉 AI가 직접 Board를 바꾸지 않는다.

## 9. 수정 요청 해석

`aiArchitecturePatchPreview.ts`는 자연어를 크게 네 종류로 본다.

| action | 예시 |
| --- | --- |
| `add_resource` | DB 하나 추가해줘, 스토리지 붙여줘 |
| `remove_resource` | 이 S3 지워줘, DB 삭제해줘 |
| `modify_resource` | EC2 타입 바꿔줘, 보안 설정 바꿔줘 |
| `manual_review` | 더 좋게 만들어줘, 서비스처럼 고쳐줘 |

이번 브랜치에서 중요한 변화:

1. `add_resource`가 들어왔을 때 기존 리소스 전체 목록을 보여주지 않는다.
2. "새 S3를 어디에 연결할까요?" 대신 "어떤 용도로 쓸까요?"를 묻는다.
3. 사용자가 용도를 말하면 API가 적절한 연결을 추론한다.
4. `manual_review`는 "무엇을 바꿀까요?" 대신 "어떤 서비스로 만들거나 고치고 싶은지"를 묻는다.

예:

```text
사용자: 데이터베이스 하나 추가해줘
AI: 새 RDS Database를 어떤 용도로 쓸까요?
사용자: 로그인/회원 데이터를 저장할래
결과: App Server -> RDS Database edge를 포함한 preview 생성
```

예:

```text
사용자: 스토리지 버킷도 넣어줘
AI: 새 S3 Bucket을 어떤 용도로 쓸까요?
사용자: 사용자 업로드 파일을 저장할래
결과: App Server -> S3 Bucket edge를 포함한 preview 생성
```

## 11. Area 레이어 보정

다이어그램 area는 resource보다 반드시 아래에 있어야 한다.

이번 브랜치에서는 [flow-mappers.ts](../../../apps/web/features/diagram-editor/flow-mappers.ts)의 z-index 계산을 area와 resource의 다른 대역으로 나눴다.

```text
area z-index band
< resource z-index band
```

의미:

| 대상 | 레이어 |
| --- | --- |
| Region, VPC, AZ, Subnet 같은 area | 항상 아래 |
| EC2, S3, RDS, IAM, CloudWatch 같은 resource | 항상 위 |
| edge | 연결 대상 node의 z-index 근처에서 계산 |

이렇게 해서 nested area가 깊어져도 resource를 덮지 않는다.

## 12. EC2 컨테이너 배치

EC2 초안은 반드시 컨테이너 기반으로 간다.

최소 API 서버도 예외가 아니다.

```text
Region
-> VPC
-> Subnet
-> Security Group
-> EC2
```

이 흐름은 backend에서 VPC/Subnet/SG/EC2 node와 edge를 만들고, frontend adapter가 board area로 변환하면서 시각화한다.

관련 테스트:

| 테스트 | 확인 내용 |
| --- | --- |
| `keeps a minimal API server small but container-based` | 최소 API 서버가 작은 리소스 세트이면서 VPC/Subnet/SG 안에 EC2를 둔다. |
| `lays out generated EC2 drafts inside cloud container areas` | 생성된 EC2 초안이 board에서 컨테이너 area 안에 배치된다. |
| `toFlowNodes keeps every area below resources` | area가 resource 위로 올라오지 않는다. |

## 20. 남은 주의점

현재 브랜치는 MVP에서 지원 가능한 타입만 사용한다.

따라서 운영형 아키텍처 피드백 중 일부는 아직 추상화해서 표현한다.

| 피드백 | 현재 처리 |
| --- | --- |
| Route 53 진입 | 아직 `ROUTE53_RECORD`를 생성하지 않고 CloudFront entry로만 표현한다. |
| WAF | 아직 `WAF_WEB_ACL`를 생성하지 않는다. |
| ALB | 아직 `LOAD_BALANCER` 타입을 생성하지 않는다. |
| S3 VPC Endpoint | 아직 `VPC_ENDPOINT`를 생성하지 않고 App -> S3, Policy -> S3 edge로 표현한다. |
| DB Subnet Group | 아직 `DB_SUBNET_GROUP`를 생성하지 않고 RDS config의 `subnetIds`로 표현한다. |
| Secrets Manager | 아직 `SECRETS_MANAGER_SECRET`를 생성하지 않는다. |

이 타입들을 추가하려면 shared type, resource definition, catalog, Terraform 변환, API 테스트, board adapter를 같이 열어야 한다.

## 21. 진짜 핵심 요약

```text
AI 채팅 입력
-> 생성/수정 요청 분기
-> 정보가 부족하면 쉬운 질문
-> 생성이면 requirementFacts 기반 ArchitectureJson
-> 수정이면 ArchitecturePatchPreview
-> frontend board preview 표시
-> 사용자가 승인하면 실제 DiagramJson 적용
-> 적용 직후 저장
```

이번 브랜치의 가장 중요한 원칙은 두 가지다.

1. AI가 보드를 직접 바꾸지 않는다. preview와 사용자 승인이 먼저다.
2. 자연어를 리소스 이름 하나로만 보지 않는다. 서비스 목적, 최소/운영형 의도, 데이터/파일/보안 단서를 해석해서 다이어그램 크기와 연결을 조절한다.
