# 아키텍처 다이어그램 검수 가이드

이 문서는 특정 코드나 도구 상태가 아니라, 클라우드 아키텍처 다이어그램을 보고 "이 설계가 맞는가?"를 판단하기 위한 일반 개념을 정리한다.

예시는 AWS 용어를 많이 사용하지만 원칙은 Azure, GCP, Kubernetes, Terraform, Pulumi, CloudFormation에도 비슷하게 적용된다. 핵심은 리소스 이름을 많이 아는 것이 아니라, 리소스 사이의 포함관계, 의존성, 트래픽 흐름, 보안 경계를 분리해서 읽는 것이다.

## 1. 먼저 나눠서 봐야 하는 것

좋은 다이어그램은 모든 관계를 화살표 하나로 해결하지 않는다. 아래 관점을 따로 봐야 한다.

| 관점 | 질문 | 주로 표현되는 방식 |
| --- | --- | --- |
| 요구사항 | 사용자가 만들고 싶다고 한 구조가 들어 있는가? | 전체 리소스 구성과 시나리오 |
| 포함관계 | 어떤 리소스가 어떤 경계 안에 배치되는가? | Region, VPC, Availability Zone, Subnet 같은 area |
| 의존성 | 어떤 리소스가 다른 리소스의 ID, 이름, ARN, endpoint를 참조하는가? | association, attach, reference |
| 트래픽 흐름 | 요청, 데이터, 파일, 이벤트가 어디에서 어디로 이동하는가? | 방향 있는 화살표 |
| 보안 경계 | 누가 무엇에 접근할 수 있고, 어디서 차단되는가? | Security Group, IAM, firewall, policy |
| 운영 요소 | 장애, 로그, 백업, 비용, 배포 후 관리가 가능한가? | Logs, Metrics, Alarm, Backup, Lifecycle |

초보자는 리소스 아이콘만 보려고 하기 쉽지만, 실제 설계 검수에서는 "무엇이 어디 안에 있는가", "무엇이 무엇을 참조하는가", "실제 요청은 어느 방향으로 가는가"를 따로 읽어야 한다.

## 2. 리소스, 관계, 의존성의 차이

### 리소스

리소스는 클라우드나 플랫폼에서 생성하거나 참조하는 구성 요소다.

| 범주 | 예시 | 검수 질문 |
| --- | --- | --- |
| 네트워크 경계 | Region, VPC, Subnet, VNet | 어디에 배치되는가? public/private 경계가 맞는가? |
| 진입점 | DNS, CDN, Load Balancer, API Gateway | 사용자는 어디로 들어오는가? TLS는 어디서 끝나는가? |
| 컴퓨트 | EC2, VM, Container, Lambda, Pod | 어떤 코드가 실행되는가? 어느 네트워크 안에 있는가? |
| 데이터 저장소 | RDS, S3, DynamoDB, EBS, Redis | 어떤 데이터를 저장하는가? 누가 읽고 쓰는가? |
| 보안과 권한 | Security Group, IAM Role, Policy, KMS | 누가 접근할 수 있는가? 암호화와 권한은 충분한가? |
| 운영 | Log, Metric, Alarm, Backup | 장애를 어떻게 발견하고 복구하는가? |

### 관계

관계는 리소스 사이의 의미다. 같은 두 리소스 사이에도 관계 종류가 다르면 화살표 방향과 표현 방식이 달라진다.

| 관계 종류 | 예시 | 다이어그램 표현 |
| --- | --- | --- |
| 포함관계 | VPC contains Subnet, Subnet hosts EC2 | area 안에 배치한다. 중복 화살표는 보통 생략한다. |
| 설정 참조 | EC2 uses Subnet, EC2 uses AMI | 생성 설정이나 association으로 표현한다. |
| 런타임 트래픽 | Client -> API -> DB | 요청이 실제로 이동하는 방향으로 화살표를 그린다. |
| 데이터 흐름 | EC2 -> S3, App -> Queue | 읽기, 쓰기, 이벤트 발행 방향을 표시한다. |
| 보안 정책 | Security Group allows EC2 traffic | 보호 대상과 허용 방향을 명확히 한다. |
| 운영 의존성 | App -> Logs, Backup -> Storage | 운영 데이터가 어디로 흘러가는지 표시한다. |

### 의존성

의존성은 어떤 리소스가 만들어지거나 동작하기 위해 다른 리소스의 값이 필요한 관계다. IaC에서는 의존성이 매우 중요하다.

예를 들어 EC2 Instance는 보통 아래 값에 의존한다.

- 어떤 AMI로 생성되는가
- 어떤 Subnet에 배치되는가
- 어떤 Security Group을 사용하는가
- 어떤 IAM Role을 붙이는가
- 어떤 Key Pair, Volume, User Data를 사용하는가

따라서 다이어그램에 EC2 아이콘만 있으면 "서버가 있다"는 사실은 보이지만, 운영 가능한 서버인지 판단하기에는 부족하다.

## 3. 포함관계는 area로, 상호작용은 화살표로

포함관계와 트래픽 흐름을 섞으면 다이어그램이 금방 지저분해진다.

| 표현 | 언제 쓰는가 | 예시 |
| --- | --- | --- |
| area 안에 배치 | 경계 안에 들어가는 관계 | Region 안의 VPC, VPC 안의 Subnet, Subnet 안의 EC2 |
| 화살표 | 요청, 데이터, 이벤트, 설정 참조가 이동하거나 연결되는 관계 | User -> ALB, EC2 -> S3, AMI -> EC2 |
| 라벨 | 화살표 의미가 모호할 때 | `stores images`, `launch image`, `reads/writes` |
| 주석 | 보안, 제약, 미지원, 생략을 설명할 때 | `IAM Role 필요`, `private subnet 권장` |

포함관계를 area로 이미 보여줬다면 같은 의미의 화살표는 보통 빼는 것이 좋다. 예를 들어 EC2가 Subnet 안에 그려져 있다면 `Subnet -> EC2` 화살표는 대부분 중복이다. 반대로 EC2가 S3에 이미지를 저장한다면 이는 포함관계가 아니므로 `EC2 -> S3` 화살표가 필요하다.

## 4. 화살표 방향의 기본 규칙

화살표는 "원인이 되는 쪽" 또는 "사용하는 쪽"에서 "대상"으로 향하게 잡는 것이 이해하기 쉽다. 단, 관계 종류를 먼저 정해야 한다.

| 상황 | 권장 방향 | 의미 |
| --- | --- | --- |
| 사용자가 웹 서비스에 접속 | User -> CDN 또는 Load Balancer | 사용자의 요청이 진입점으로 간다. |
| CDN이 정적 파일 원본을 사용 | CloudFront/CDN -> S3 | CDN이 S3를 origin으로 읽는다. |
| API 서버가 DB를 사용 | API/Backend -> RDS | 애플리케이션이 DB에 질의한다. |
| 서버가 S3에 이미지를 저장 | EC2/App -> S3 | 서버가 객체 저장소에 쓴다. |
| AMI로 EC2를 생성 | AMI -> EC2 | 이미지가 인스턴스 생성 원본이다. 런타임 트래픽은 아니다. |
| Route Table이 Internet Gateway로 보냄 | Route Table -> Internet Gateway | 라우팅 규칙이 목적지를 지정한다. |
| Security Group이 서버 접근을 제어 | Security Group -> EC2/RDS | 보안 정책이 보호 대상에 적용된다. |
| Lambda가 이벤트를 받음 | Event Source -> Lambda | 이벤트가 함수를 트리거한다. |
| Lambda가 DB에 접근 | Lambda -> DB | 함수 실행 중 DB를 사용한다. |

중요한 점은 화살표가 항상 네트워크 패킷만 뜻하지 않는다는 것이다. `AMI -> EC2`는 네트워크 통신이 아니라 생성 의존성이다. 그래서 화살표에는 가능하면 라벨을 붙여 의미를 구분해야 한다.

## 5. 네트워크 경계 읽는 법

### Region

Region은 리소스가 위치한 지리적 권역이다. 대부분의 VPC, Subnet, EC2, RDS, Lambda는 특정 Region에 속한다. S3 같은 글로벌에 가까운 서비스도 실제 데이터 위치나 버킷 Region 개념을 가진다.

검수 질문:

- 주요 리소스가 같은 Region에 있는가?
- 멀티 리전이 필요하다면 복제, 장애 전환, DNS 전략이 표현되어 있는가?
- 단일 리전이라면 장애 범위가 명확한가?

### Availability Zone

Availability Zone은 같은 Region 안의 물리적으로 분리된 데이터센터 묶음이다.

검수 질문:

- 고가용성이 필요하면 최소 2개 이상의 AZ에 분산되어 있는가?
- 단일 EC2 연습 구조라면 단일 AZ여도 의도와 비용상 납득 가능한가?
- RDS Multi-AZ, Load Balancer, Auto Scaling 같은 요소가 필요한 요구사항인가?

### VPC

VPC는 격리된 네트워크 경계다. EC2, RDS, private subnet 같은 네트워크 리소스는 보통 VPC 안에 있다.

검수 질문:

- VPC 안에 있어야 하는 리소스가 밖에 그려져 있지 않은가?
- S3, DynamoDB 같은 관리형 서비스는 보통 VPC 안 리소스로 그리지 않아도 된다.
- VPC 밖 서비스와 통신하려면 Internet Gateway, NAT Gateway, VPC Endpoint 같은 경로가 필요한가?

### Subnet

Subnet은 VPC 안에서 IP 범위를 나눈 단위다. public subnet과 private subnet은 이름만으로 결정되지 않고 라우팅과 접근 조건으로 결정된다.

public subnet으로 보려면 보통 아래 조건이 필요하다.

- Subnet이 VPC에 속한다.
- Subnet에 연결된 Route Table이 `0.0.0.0/0`을 Internet Gateway로 보낸다.
- VPC에 Internet Gateway가 붙어 있다.
- 외부에서 직접 접근할 리소스에는 public IP 또는 Load Balancer 진입점이 있다.
- Security Group이 필요한 inbound traffic을 허용한다.

private subnet은 외부 인터넷에서 직접 들어올 수 없도록 구성하고, outbound 인터넷 접근은 NAT Gateway나 VPC Endpoint를 통해 처리하는 경우가 많다.

### Internet Gateway, NAT Gateway, VPC Endpoint

| 리소스 | 역할 | 흔한 오해 |
| --- | --- | --- |
| Internet Gateway | VPC와 인터넷 사이의 양방향 통로 | 붙어 있기만 하면 모든 리소스가 인터넷에 공개된다고 오해한다. Route Table과 public IP도 필요하다. |
| NAT Gateway | private subnet 리소스의 outbound 인터넷 접근 | 외부 사용자가 private 리소스로 들어오는 통로가 아니다. |
| VPC Endpoint | S3, DynamoDB 같은 서비스에 private 경로로 접근 | 모든 서비스에 자동으로 생기는 것이 아니다. 서비스별 endpoint가 필요하다. |

## 6. 보안 경계 읽는 법

### Security Group

Security Group은 방화벽처럼 보이지만, AWS에서는 ENI 또는 인스턴스 같은 대상에 붙는 stateful 보안 규칙이다.

검수 질문:

- 보호 대상이 무엇인지 명확한가?
- inbound 규칙이 너무 넓지 않은가?
- DB는 애플리케이션 Security Group에서만 접근 가능하게 되어 있는가?
- public inbound가 필요한 리소스와 그렇지 않은 리소스가 구분되어 있는가?

Security Group을 area처럼 그리는 것은 "이 안의 리소스가 이 보안 규칙으로 보호된다"는 시각적 설명으로는 유용하다. 다만 실제 모델에서는 Security Group이 Subnet 안에 존재하는 박스라기보다 VPC 범위의 정책 리소스에 가깝다. 따라서 area 표현은 이해를 돕는 그림이지, 실제 물리적 포함관계로 오해하면 안 된다.

### IAM Role과 Policy

네트워크 경로가 있어도 권한이 없으면 API 호출은 실패한다. 예를 들어 EC2가 S3에 이미지를 저장하려면 보통 두 종류가 필요하다.

- 네트워크 관점: EC2가 S3 endpoint에 도달할 수 있어야 한다.
- 권한 관점: EC2에 연결된 IAM Role이 S3 쓰기 권한을 가져야 한다.

다이어그램에서 `EC2 -> S3` 화살표가 있다면 "통신 의도"는 보이지만, IAM Role과 Bucket Policy까지 완성되어 있다는 뜻은 아니다. 운영 설계에서는 권한 리소스도 함께 확인해야 한다.

### 암호화와 비밀값

검수 질문:

- 저장 데이터 암호화가 필요한 저장소에 KMS나 provider-managed encryption이 있는가?
- DB 비밀번호, API key, access token이 다이어그램이나 설정에 노출되지 않는가?
- Secret Manager, Parameter Store, Key Vault 같은 비밀 관리 경로가 필요한가?

## 7. 저장소와 데이터 흐름

### S3와 객체 저장소

S3는 파일과 객체를 저장하는 관리형 저장소다. EC2나 Lambda가 S3에 접근할 때 S3를 VPC 안에 넣어 그릴 필요는 없다. 보통 VPC 밖의 관리형 서비스로 그리고, 접근 경로와 권한을 별도로 본다.

검수 질문:

- 누가 S3에 읽기/쓰기를 하는가?
- public read가 정말 필요한가?
- lifecycle, versioning, encryption, access log가 필요한가?
- private subnet에서 접근한다면 NAT Gateway 또는 VPC Endpoint가 필요한가?

### RDS와 관계형 DB

RDS는 보통 private subnet에 두고, 애플리케이션 계층에서만 접근하게 한다.

검수 질문:

- DB가 public internet에 직접 노출되지 않는가?
- app -> DB 방향의 접근만 허용되는가?
- DB subnet group, backup, encryption, parameter group이 필요한가?
- 운영 요구사항이 있으면 Multi-AZ나 read replica가 필요한가?

### EBS와 S3의 차이

EBS는 EC2에 붙는 블록 스토리지다. S3는 네트워크 API로 접근하는 객체 저장소다.

| 구분 | EBS | S3 |
| --- | --- | --- |
| 사용 방식 | EC2 디스크처럼 붙여 사용 | API로 객체를 읽고 쓴다. |
| 주 사용처 | OS disk, 애플리케이션 로컬 데이터 | 이미지, 업로드 파일, 정적 파일, 백업 |
| 다이어그램 관계 | EBS -> EC2 또는 EC2 attaches EBS | EC2/App -> S3 |

이미지 저장용 구조라면 일반적으로 EC2 내부 디스크보다 S3가 더 자연스럽다. 다만 애플리케이션이 이미지를 저장한다는 관계는 `EC2/App -> S3`로 보이는 것이 좋다.

## 8. 컴퓨트 리소스 읽는 법

### EC2 또는 VM

EC2가 제대로 표현되려면 최소한 아래 질문에 답할 수 있어야 한다.

- 어떤 이미지로 생성되는가?
- 어떤 Subnet에 배치되는가?
- 어떤 Security Group으로 보호되는가?
- public 접근이 필요한가, private로 둘 것인가?
- S3, DB, Queue 같은 외부 서비스에 접근할 권한이 있는가?

AMI는 EC2를 만드는 원본 이미지다. AMI가 EC2 옆에 떠 있는데 연결이 없다면 초보자에게는 의미가 잘 보이지 않는다. `AMI -> EC2` 같은 생성 의존성을 표시하면 "이 인스턴스가 이 이미지로 만들어진다"는 의미가 분명해진다.

### Container와 Kubernetes

컨테이너 구조는 계층이 많다.

- Container image
- Task, Pod, Deployment 같은 실행 단위
- Cluster
- Service 또는 Ingress
- Load Balancer
- IAM 또는 workload identity

다이어그램이 컨테이너를 그린다면 "사용자 요청이 어디로 들어오고", "어떤 실행 단위가 실제 코드를 돌리며", "어떤 저장소와 통신하는지"가 보여야 한다.

### Lambda와 서버리스

Lambda는 트리거와 권한이 없으면 의미가 약해진다.

검수 질문:

- 무엇이 Lambda를 호출하는가? API Gateway, S3 event, EventBridge, Queue 중 무엇인가?
- Lambda가 어떤 DB, S3, 외부 API를 사용하는가?
- IAM Role이 필요한 권한을 갖는가?
- 로그와 재시도, dead-letter queue가 필요한가?

## 9. 진입점과 외부 트래픽

사용자 요청은 보통 아래 중 하나로 들어온다.

| 진입점 | 쓰는 경우 | 검수 질문 |
| --- | --- | --- |
| DNS | 도메인 이름이 필요할 때 | 어떤 record가 어떤 endpoint를 가리키는가? |
| CDN | 정적 파일 캐싱, edge delivery | origin이 무엇인가? private origin 접근은 어떻게 제어하는가? |
| Load Balancer | 여러 서버나 private target으로 분산 | target group, health check, subnet 배치가 맞는가? |
| API Gateway | API 진입점, 인증, throttling | backend integration과 auth가 표현되는가? |

진입점이 없는 서버는 외부 사용자가 어떻게 접속하는지 판단하기 어렵다. 반대로 내부 배치 서버라면 외부 진입점이 없어도 맞을 수 있다. 항상 요구사항과 함께 봐야 한다.

## 10. 좋은 다이어그램 배치 원칙

### 왼쪽에서 오른쪽으로 읽히게 한다

일반적으로 아래 흐름이 이해하기 쉽다.

1. 사용자 또는 외부 시스템
2. DNS, CDN, Load Balancer, API Gateway 같은 진입점
3. 애플리케이션 컴퓨트
4. DB, S3, Queue 같은 데이터와 연동 대상
5. Logs, Metrics, Backup 같은 운영 리소스

### 경계는 큰 것에서 작은 것으로 둔다

권장 포함 순서:

1. Region
2. VPC
3. Availability Zone
4. Subnet
5. Security Group 또는 보호 영역
6. Compute, DB, internal resource

단, Security Group은 실제 물리 경계가 아니므로 "보호 영역"으로 읽어야 한다.

### 중복 화살표를 줄인다

area가 이미 포함관계를 설명한다면, 포함관계 화살표는 생략하는 것이 좋다.

좋은 예:

- VPC area 안에 Subnet을 넣는다.
- Subnet area 안에 EC2를 넣는다.
- EC2가 S3에 저장하면 `EC2 -> S3`만 그린다.

나쁜 예:

- VPC -> Subnet, Subnet -> EC2, Security Group -> EC2, EC2 -> Security Group 등 포함관계와 정책 관계를 모두 화살표로 겹쳐 그린다.
- 라벨 없는 화살표가 많아서 데이터 흐름인지 의존성인지 알 수 없다.

## 11. 예시 구조별 검수 포인트

### EC2 서버 하나와 이미지 저장용 S3

필수에 가까운 표현:

- Region 또는 최소 배치 경계
- VPC
- Subnet
- EC2 Instance
- Security Group
- S3 Bucket
- AMI -> EC2, `launch image`
- EC2 -> S3, `stores images`

추가로 있으면 좋은 표현:

- IAM Role -> S3 권한
- Route Table -> Internet Gateway
- 외부 접근이 필요하면 User -> EC2 또는 User -> Load Balancer
- private subnet이면 NAT Gateway 또는 VPC Endpoint

검수 판단:

- EC2가 S3와 아무 연결도 없으면 "이미지 저장용 S3를 사용한다"는 요구가 잘 드러나지 않는다.
- AMI가 EC2와 아무 연결도 없으면 "이 인스턴스가 어떤 이미지로 생성되는가"가 불분명하다.
- S3로 향하는 화살표 방향은 보통 EC2/App -> S3가 자연스럽다.

### 정적 웹사이트

대표 구조:

- User -> CDN
- CDN -> S3 origin
- DNS -> CDN
- S3 Bucket
- 필요하면 Certificate, WAF, logging

검수 판단:

- User가 S3로 직접 가는지, CDN을 거치는지 구분해야 한다.
- S3 public access를 열 것인지, CDN만 접근하게 할 것인지가 보안상 중요하다.

### 웹 API와 DB

대표 구조:

- User -> Load Balancer 또는 API Gateway
- Load Balancer/API Gateway -> App Server
- App Server -> RDS
- App Server Security Group
- DB Security Group
- private subnet의 DB

검수 판단:

- DB가 public subnet에 직접 노출되어 있으면 위험 신호다.
- `User -> DB` 화살표가 있으면 대부분 잘못된 구조다.
- app -> DB 방향의 접근과 DB Security Group 제한이 보여야 한다.

### 서버리스 API

대표 구조:

- User -> API Gateway
- API Gateway -> Lambda
- Lambda -> DB 또는 S3
- Lambda -> Logs
- Lambda IAM Role

검수 판단:

- Lambda만 있으면 "무엇이 호출하는지"가 비어 있다.
- Lambda가 S3나 DB를 사용한다면 권한과 네트워크 조건이 필요하다.

### 이벤트 처리 구조

대표 구조:

- Producer -> Queue 또는 Topic
- Queue/Topic -> Worker 또는 Lambda
- Worker -> DB/S3
- 실패 처리용 DLQ
- logs and metrics

검수 판단:

- 이벤트 방향이 producer에서 consumer로 읽히는가?
- 재시도와 실패 보관이 필요한 요구사항인가?
- consumer가 처리 결과를 어디에 저장하는가?

## 12. 자주 틀리는 패턴

| 틀린 패턴 | 왜 문제인가 | 더 나은 표현 |
| --- | --- | --- |
| 포함관계마다 화살표를 모두 그림 | 실제 트래픽 화살표와 섞여 읽기 어렵다. | area 배치로 포함을 표현하고 화살표는 줄인다. |
| S3를 VPC 안에 넣음 | S3는 보통 VPC 내부 리소스처럼 존재하지 않는다. | VPC 밖 관리형 서비스로 그리고 접근 화살표를 표시한다. |
| DB가 사용자와 직접 연결됨 | 대부분 보안상 부적절하다. | User -> App -> DB로 표현한다. |
| AMI가 떠 있지만 EC2와 연결 없음 | 생성 의존성이 보이지 않는다. | AMI -> EC2, `launch image`를 표시한다. |
| EC2와 S3가 함께 있지만 연결 없음 | 앱이 S3를 쓰는지 알 수 없다. | EC2/App -> S3, `stores images`를 표시한다. |
| Internet Gateway만 있으면 public이라고 판단 | Route Table, public IP, Security Group도 필요하다. | public 접근 조건을 함께 확인한다. |
| Security Group을 Subnet처럼 오해 | Security Group은 물리 네트워크 경계가 아니다. | 보호 대상에 적용되는 정책으로 읽는다. |
| 권한 리소스가 생략됨 | 네트워크가 열려도 API 권한이 없으면 실패한다. | IAM Role, Policy, KMS 필요 여부를 확인한다. |

## 13. 검수 체크리스트

다이어그램이 맞는지 빠르게 볼 때 아래 질문을 순서대로 확인한다.

- 요구사항에 나온 핵심 리소스가 모두 있는가?
- 지원하지 않거나 생략된 리소스가 있다면 경고나 주석으로 설명되어 있는가?
- Region, VPC, Subnet 같은 큰 경계 안에 리소스가 자연스럽게 배치되어 있는가?
- 포함관계가 area로 보이고, 중복 화살표 때문에 지저분하지 않은가?
- 실제 데이터 흐름과 요청 흐름은 방향 있는 화살표로 보이는가?
- 화살표 라벨만 보고도 통신, 저장, 생성, 권한, 라우팅 중 무엇인지 알 수 있는가?
- EC2, Lambda, Container 같은 실행 리소스가 이미지, 네트워크, 보안, 권한과 연결되어 있는가?
- 저장소는 누가 읽고 쓰는지 명확한가?
- DB나 private 리소스가 외부에 직접 노출되어 있지 않은가?
- S3 같은 관리형 서비스가 VPC 내부 물리 리소스처럼 오해되게 그려져 있지 않은가?
- public 접근에는 Internet Gateway, Route Table, public endpoint, Security Group 조건이 함께 고려되어 있는가?
- private 접근에는 NAT Gateway, VPC Endpoint, peering, VPN 같은 경로가 필요한지 판단되어 있는가?
- IAM Role, Policy, KMS, Secret 같은 권한과 보안 요소가 요구사항 수준에 맞게 표현되어 있는가?
- 운영 요구사항이 있으면 logs, metrics, alarm, backup, lifecycle이 고려되어 있는가?
- 초보자가 봐도 "누가 누구를 사용한다"는 흐름이 읽히는가?

## 14. 초보자를 위한 빠른 판단법

처음 볼 때는 아래 세 문장으로 스스로 설명해 본다.

1. "사용자는 여기로 들어와서, 이 컴퓨트가 요청을 처리한다."
2. "이 컴퓨트는 이 저장소를 읽거나 쓴다."
3. "외부에서 바로 접근하면 안 되는 것은 이 보안 경계 뒤에 있다."

이 세 문장이 다이어그램을 보고 자연스럽게 나오지 않으면, 리소스가 있어도 관계 표현이 부족한 것이다.

예를 들어 "EC2와 S3가 있다"보다 "EC2가 업로드 이미지를 S3에 저장한다"가 더 좋은 설명이다. 그래서 다이어그램도 단순 배치가 아니라 `EC2 -> S3` 관계를 보여줘야 한다.

## 15. 전문가 관점에서 추가로 보는 것

전문가는 아이콘의 존재보다 아래를 더 중요하게 본다.

- 장애 범위: 단일 AZ, 단일 인스턴스, 단일 Region 장애를 감당할 수 있는가?
- 데이터 경계: 민감 데이터가 어느 저장소에 있고 암호화와 접근 통제가 있는가?
- 네트워크 경로: public/private, NAT, endpoint, peering, routing이 실제로 성립하는가?
- 배포 가능성: IaC에서 참조 관계와 생성 순서가 모호하지 않은가?
- 최소 권한: IAM, Security Group, bucket policy가 과도하게 열려 있지 않은가?
- 운영성: 로그, 알람, 백업, 복구, 비용 추적이 가능한가?
- 변경 가능성: 나중에 scale-out, multi-AZ, private endpoint, CI/CD를 붙이기 쉬운가?

## 16. 최종 판단 문장

좋은 클라우드 아키텍처 다이어그램은 "리소스가 많이 있는 그림"이 아니다.

좋은 다이어그램은 다음을 명확히 말한다.

- 무엇을 만들었는가
- 어디에 배치했는가
- 누가 무엇을 사용하는가
- 어떤 방향으로 데이터와 요청이 흐르는가
- 어떤 보안 경계가 접근을 허용하거나 막는가
- 배포와 운영에 필요한 의존성이 빠지지 않았는가

따라서 다이어그램을 검수할 때는 "아이콘이 맞는가"보다 "관계가 읽히는가"를 먼저 봐야 한다. 특히 포함관계는 area로, 실제 사용과 데이터 흐름은 화살표로, 생성 의존성과 권한은 라벨이나 보조 리소스로 구분해서 표현해야 한다.
