# 배포 실행 파트 개념 설명서

## 0. 이 문서의 책임

이 문서는 배포 실행 파트를 이해하기 위한 **개념 설명서**다.

Terraform 배포 흐름이 왜 `connect -> init -> validate -> plan -> approve -> apply -> log/result`처럼 이어지는지 설명한다. 어떤 타입, API, DB 필드를 쓸지는 [배포 실행 파트 결정사항 원천](./004_배포파트결정사항초안.md)을 기준으로 하고, 실제 구현 순서는 [배포 실행 파트 구현 순서](./005_배포파트구현순서.md)를 따른다.

## 1. 배포 파트가 맡는 일

배포 파트는 사용자가 만든 AWS 인프라 설계를 Terraform 기반 흐름으로 검증하고, 승인된 plan만 실제 AWS에 반영하며, 실행 결과와 생성 리소스/output을 프로젝트에 남기는 영역이다.

현재 구현 목표는 AWS 연결 확인부터 `terraform apply`, 결과 저장까지 이어지는 실제 배포 실행 흐름이다.

```text
ArchitectureSnapshot
-> TerraformArtifact
-> AWS 연결 확인
-> 작업 디렉터리 복원
-> terraform init / validate
-> terraform plan -out=tfplan
-> terraform show -json tfplan
-> plan summary
-> 위험/비용 분석
-> 사용자 승인
-> terraform apply tfplan
-> 로그와 성공/실패 저장
-> 생성 리소스/output 저장
-> 로그와 이력 조회
```

## 2. AWS 연결 확인

Terraform이 AWS 리소스를 만들려면 AWS API를 호출할 권한이 있어야 한다. 그래서 실제 실행 전에는 인증 정보가 유효한지, 어느 계정과 region을 대상으로 하는지, 최소한 AWS API를 호출할 수 있는지 확인해야 한다.

### Access Key 방식

Access Key 방식은 사용자가 AWS에서 발급한 `Access Key ID`, `Secret Access Key`, `Region`을 서비스에 입력하는 방식이다. 이해하기 쉽지만 Secret Key가 노출되면 비용 사고와 보안 사고로 이어질 수 있다.

따라서 장기 저장, 평문 저장, 프론트 재노출, 로그 출력은 피해야 한다.

### Role Assume 방식

Role Assume은 사용자의 AWS 계정에 IAM Role을 만들고 서비스가 그 역할을 잠시 빌려 쓰는 방식이다. Access Key를 장기 저장하는 것보다 안전하지만, 사용자가 AWS 계정에 trust policy를 설정해야 하므로 안내와 구현 난이도는 더 높다.

### STS `GetCallerIdentity`

AWS 연결 확인에는 보통 STS `GetCallerIdentity`를 사용한다. 성공하면 현재 인증 정보가 어느 AWS 계정의 어떤 주체인지 알 수 있다.

이 호출이 성공했다는 것은 인증 정보가 유효하다는 뜻이지, EC2, RDS, VPC 같은 리소스를 만들 권한까지 충분하다는 뜻은 아니다.

## 3. Terraform init

`terraform init`은 Terraform 실행 준비 단계다.

주요 역할은 다음과 같다.

- provider 다운로드
- backend 초기화
- `.terraform` 디렉터리 준비
- `.terraform.lock.hcl` 생성 또는 갱신

AWS 리소스를 만들려면 Terraform AWS provider가 필요하다. Terraform은 provider를 통해 AWS API 호출 형식으로 리소스 생성 요청을 바꾼다.

`init`이 실패하면 `validate`, `plan`, `apply`로 넘어가면 안 된다.

## 4. Terraform validate

`terraform validate`는 Terraform 코드의 문법과 구조를 확인한다. 보통 실제 AWS 리소스를 만들지는 않는다.

잡을 수 있는 문제는 다음과 같다.

- 중괄호 누락
- 잘못된 resource 블록 구조
- 필수 속성 누락
- 알 수 없는 인자
- 잘못된 참조 문법

`validate`가 성공해도 `plan`이 실패할 수 있다. 코드 문법은 맞지만 AWS 권한, 실제 리소스 상태, 이름 중복 같은 문제가 남아 있을 수 있기 때문이다.

## 5. Terraform plan

`terraform plan`은 실제 AWS를 바꾸기 전에 무엇이 변경될지 계산하는 단계다.

plan에서 확인해야 하는 것은 다음과 같다.

- 새로 생성되는 리소스
- 수정되는 리소스
- 삭제되는 리소스
- 교체되는 리소스
- 비용이나 보안 위험이 큰 변경

안전하게 apply하려면 사용자가 확인한 plan과 실제 apply 대상이 같아야 한다. 그래서 실제 실행 단계에서는 `terraform plan -out=tfplan`로 plan 파일을 만들고, `terraform apply tfplan`처럼 승인된 plan 파일만 적용하는 방식이 필요하다.

## 6. Plan 결과 표시

Terraform plan 원문은 사용자가 읽기 어렵다. 제품에서는 화면에 표시하기 쉬운 summary로 바꿔야 한다.

사용자가 알고 싶은 정보는 보통 다음이다.

| 정보 | 예시 |
| --- | --- |
| 생성 수 | S3 bucket 1개 생성 |
| 수정 수 | EC2 instance 1개 수정 |
| 삭제 수 | Security Group 1개 삭제 |
| 교체 수 | RDS instance 교체 |
| 위험 표시 | public access, delete, replace |

SketchCatch는 다이어그램 기반 서비스이므로 plan 결과를 `ResourceNode.id`와 연결할 수 있어야 한다. 그래야 사용자가 어떤 노드가 생성, 수정, 삭제되는지 이해할 수 있다.

## 7. Terraform apply

`terraform apply`는 plan 결과를 실제 AWS에 반영하는 단계다. 이 단계에서 비용과 보안 영향이 실제로 발생한다.

따라서 apply 전에는 최소한 다음이 확인되어야 한다.

- AWS 연결이 유효함
- `init`, `validate`, `plan`이 성공함
- 비용/위험 분석이 blocked가 아님
- 사용자가 plan summary를 확인하고 승인함
- 승인된 artifact와 실제 실행 artifact가 같음
- 같은 프로젝트에서 다른 실행이 진행 중이지 않음

이 단계는 실제 AWS 리소스를 생성, 수정, 삭제할 수 있으므로 사용자가 본 plan과 apply 대상 `tfplan`이 반드시 같아야 한다.

## 8. 로그 표시

배포는 시간이 걸리는 작업이다. 사용자가 진행 상황을 볼 수 있도록 로그가 필요하다.

backend 또는 worker에서 실행한 `terraform init`, `terraform validate`, `terraform plan`, `terraform show -json`, `terraform apply`, `terraform output -json`의 stdout/stderr를 수집한다.

로그에서 중요한 기준은 다음이다.

- 단계별 진행 상황을 알 수 있어야 한다.
- 실패 원인을 찾을 수 있어야 한다.
- secret, token, password, DB URL, AWS key가 노출되면 안 된다.
- 실시간 스트리밍은 안정화 단계에서 SSE를 검토할 수 있다.

## 9. 상태와 실패 단계

배포는 요청 즉시 끝나는 작업이 아니므로 상태가 필요하다.

개념적으로는 다음 상태가 있다.

| 상태 | 의미 |
| --- | --- |
| `PENDING` | 요청 생성됨 |
| `RUNNING` | 실행 중 |
| `SUCCESS` | 성공 |
| `FAILED` | 실패 |
| `CANCELLED` | 취소됨 |

위험/비용 분석으로 막힌 경우는 실행 상태와 차단 사유가 섞이지 않게 관리하는 것이 좋다. 실제 표현 방식은 004의 결정사항을 따른다.

실패 단계도 함께 저장해야 한다. 단순히 실패라고만 하면 사용자가 무엇을 고쳐야 하는지 알기 어렵다.

| 단계 | 예시 원인 |
| --- | --- |
| connection | 인증 실패, 권한 부족 |
| init | provider 다운로드 실패 |
| validate | Terraform 문법 오류 |
| plan | 권한 부족, 리소스 설정 오류 |
| approval | 승인 대상 변경, 승인 누락 |
| apply | 이름 중복, 의존성 문제 |
| output | output 추출 실패 |
| destroy | 삭제 실패 |

## 10. state, output, 생성 리소스

실제 apply가 성공하면 Terraform state에 생성된 리소스 정보가 저장된다. state는 Terraform이 관리하는 인프라 상태 장부다.

state에는 Terraform resource address, 실제 AWS resource id, provider 정보, 속성값, 의존 관계 등이 들어간다.

Terraform output은 배포 후 사용자에게 보여줄 값이다. 예를 들어 EC2 Public IP, Load Balancer URL, RDS endpoint 같은 값이 output이 될 수 있다.

이번 구현에서는 apply 성공 후 state 또는 `terraform show -json` 결과에서 생성 리소스 목록을 추출하고, `terraform output -json` 결과를 저장한다.

## 11. 작업 디렉터리와 실행 위치

Terraform은 특정 디렉터리 안의 `.tf` 파일을 기준으로 실행된다. plan/apply 실행은 프로젝트별 작업 디렉터리에서 격리되어야 한다.

프론트엔드 컴포넌트에서 Terraform CLI나 AWS SDK를 직접 호출하면 안 된다. 실제 Terraform 실행은 API 서버 또는 별도 worker에서 안전 게이트 뒤에 둔다.

파일성 데이터는 S3에 두고, RDS에는 metadata와 `objectKey`를 저장한다.

## 12. 실서비스에서 놓치면 안 되는 것

실제 apply를 구현할 때는 아래 개념을 반드시 고려해야 한다.

| 주제 | 이유 |
| --- | --- |
| plan/apply 일치 | 사용자가 본 plan과 실제 apply가 같아야 함 |
| state 관리 | state가 깨지면 리소스 추적이 어려움 |
| lock | 같은 프로젝트에 동시 실행이 생기면 안 됨 |
| credential 보안 | AWS key 유출은 실제 사고로 이어짐 |
| 로그 마스킹 | 로그에 secret이 남으면 안 됨 |
| timeout | 작업이 무한정 돌면 안 됨 |
| recovery | 서버 재시작 후 job 상태를 정리해야 함 |
| destroy | 실습 리소스를 지우는 흐름도 필요함 |
