## 1. 전체 구현 체크리스트

### 1.1 공유 타입과 문서 계약

- [x] `packages/types/src/index.ts`의 `Deployment` 관련 타입을 004 결정사항에 맞춘다.
- [x] `DeploymentStage`, `DeploymentFailureStage`, `DeploymentPlanSummary`, `DeploymentLog` 타입을 추가한다.
- [ ] `AwsConnection`, `DeployedResource`, `TerraformOutput` 타입을 추가한다.
- [ ] 기존 `Deployment`에 승인, plan, apply, output, 실패 단계 필드를 반영한다.
- [ ] `docs/data-models.md`의 `Deployment`, `DeploymentLog` 설명을 코드 타입과 맞춘다.
- [x] 공유 타입에는 raw AWS key, secret key, session token, DB password를 넣지 않는다.

- [x] `deployments` 테이블을 추가한다.
- [x] `deployment_logs` 테이블을 추가한다.

- [ ] `deployed_resources` 테이블을 추가한다.
- [ ] `terraform_outputs` 테이블을 추가한다.

- [x] `Deployment`가 `Project`, `ArchitectureSnapshot`, `TerraformArtifact`를 참조하게 한다.

- [ ] Drizzle migration diff가 예상 범위인지 확인한다.

### 1.3 API와 validation

- [x] `POST /api/projects/:projectId/deployments`를 구현한다.
- [x] `GET /api/projects/:projectId/deployments`를 구현한다.
- [x] `GET /api/deployments/:deploymentId`를 구현한다.
- [ ] `POST /api/aws/connections/test` 또는 project 하위 AWS 연결 확인 API를 구현한다.
- [ ] `POST /api/deployments/:deploymentId/plan`을 구현한다.
- [ ] `POST /api/deployments/:deploymentId/approve`를 구현한다.
- [ ] `POST /api/deployments/:deploymentId/apply`를 구현한다.
- [x] `GET /api/deployments/:deploymentId/logs`를 구현한다.
- [ ] `GET /api/deployments/:deploymentId/resources`를 구현한다.
- [ ] `GET /api/deployments/:deploymentId/outputs`를 구현한다.
- [x] 모든 request body와 params를 Zod로 검증한다.
- [x] project, workspace, architecture, Terraform artifact 소유 관계를 확인한다.
- [ ] 승인된 artifact, 승인된 `tfplan`, AWS 연결 계정/region이 apply 시점에도 같은지 확인한다.

### 1.4 AWS 연결 확인

- [ ] AWS credential 입력 방식 또는 Role Assume 방식을 정한다.
- [ ] STS `GetCallerIdentity`로 연결 확인을 수행한다.
- [ ] 응답에는 account id, caller ARN, region, 성공/실패 사유만 내려준다.
- [ ] raw credential은 DB, 로그, API 응답에 저장하지 않는다.
- [ ] AWS 연결 실패 시 `FAILED`, `failureStage = connection`으로 저장한다.

### 1.5 Terraform init / validate / plan

- [x] Terraform artifact 없음 실패 케이스를 처리한다.
- [x] architecture와 artifact 불일치 실패 케이스를 처리한다.
- [ ] S3 `objectKey` 기준으로 Terraform artifact를 작업 디렉터리에 복원한다.
- [ ] backend 또는 worker에서 `terraform init`을 실행하고 로그를 저장한다.
- [ ] backend 또는 worker에서 `terraform validate`를 실행하고 실패 단계를 저장한다.
- [ ] backend 또는 worker에서 `terraform plan -out=tfplan`을 실행한다.
- [ ] 생성된 `tfplan` 파일 hash를 계산한다.
- [ ] `tfplan` 파일은 S3에 저장하고 RDS에는 `objectKey`와 hash만 저장한다.
- [ ] `terraform show -json tfplan` 결과를 파싱해 `planSummary`를 저장한다.
- [ ] Terraform stdout/stderr 로그를 `sequence` 순서로 저장한다.
- [ ] plan 실패 시 `FAILED`와 `failureStage = plan`을 저장하고 승인 단계로 넘어가지 않는다.

### 1.6 Plan 결과 표시와 사용자 승인

- [ ] plan summary를 API와 화면에 표시한다.
- [ ] create/update/delete/replace 개수를 표시한다.

- [ ] 승인 후 artifact, `tfplan`, AWS 연결이 바뀐 경우 apply를 막는다.

### 1.7 Terraform apply

- [ ] `POST /api/deployments/:deploymentId/apply`에서 승인 기록을 검증한다.
- [ ] 승인된 `tfplanHash`와 S3의 `tfplan` hash가 같은지 검증한다.

- [ ] `terraform apply tfplan`만 실행하고 임의 apply는 실행하지 않는다.
- [ ] apply stdout/stderr 로그를 저장 전 마스킹한다.

## 2. 안정화 체크리스트

아래 항목은 apply를 제품 기능으로 열 때 사고를 막기 위해 함께 붙이거나 바로 다음 PR에서 붙인다.

- [ ] project 단위 lock
- [ ] timeout
- [ ] cancel
- [ ] 서버 재시작 후 `RUNNING` job recovery
- [ ] SSE 기반 실시간 로그
- [ ] destroy 또는 cleanup 정책
- [ ] S3 lifecycle policy로 임시 `tfplan`, state, 작업 산출물 정리

## 3. 완료 기준

아래 조건이 만족되면 CK 배포 실행 파트의 1차 구현은 완료로 본다.

- 특정 `projectId`, `architectureId`, `terraformArtifactId` 기준으로 배포 기록을 만들 수 있다.
- AWS 연결 확인을 수행하고 결과를 저장 또는 응답할 수 있다.
- backend 또는 worker에서 Terraform artifact를 복원하고 `terraform init`, `terraform validate`, `terraform plan`을 실행할 수 있다.
- `terraform show -json tfplan` 결과에서 plan summary를 만들고 저장할 수 있다.
- 사용자가 plan summary를 확인하고 승인할 수 있다.
- 승인된 `tfplan`만 `terraform apply tfplan`로 실행할 수 있다.
- Terraform 실행 로그를 저장하고 조회할 수 있다.
- 성공/실패 상태와 실패 단계를 저장하고 화면에 표시할 수 있다.
- apply 성공 후 생성 리소스 목록과 Terraform output을 프로젝트에 연결할 수 있다.
- raw secret이 DB, 로그, API 응답, 테스트 데이터에 남지 않는다.
