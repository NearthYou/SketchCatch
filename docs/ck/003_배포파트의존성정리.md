# 배포 파트 의존성 정리

## 0. 이 문서의 책임

이 문서는 CK 배포 실행 파트가 다른 파트와 **무엇을 주고받아야 하는지**만 정리한다.

MVP 범위와 타입/API/DB 결정은 [004_배포파트결정사항초안.md](./004_배포파트결정사항초안.md), 구현 순서는 [005_배포파트구현순서.md](./005_배포파트구현순서.md)를 따른다.

## 1. 전체 연결 위치

```text
JH: 다이어그램 설계
-> SW: Terraform artifact 생성
-> GG: 비용/위험 분석
-> CK: 배포 기록, 승인, mock 실행, 로그/이력 저장
-> YS: 프로젝트 화면, 승인 UI, 활동 내역, 알림
```

CK 파트는 앞 단계 결과물을 그대로 믿고 실행하지 않는다. `ArchitectureSnapshot`, `TerraformArtifact`, 위험/비용 분석 결과, 사용자 승인 상태를 함께 확인한 뒤 배포 기록을 만든다.

## 2. CK가 받아야 하는 입력

| 입력 | 제공 파트 | 쓰는 곳 |
| --- | --- | --- |
| `projectId` | YS / 프로젝트 관리 | 배포 이력의 프로젝트 연결 |
| `workspaceId` 또는 `userId` | YS / 인증, workspace | 배포 요청자와 승인 주체 기록 |
| `architectureId` | JH / 설계 저장 | 배포 기준 snapshot 고정 |
| `ArchitectureJson` 또는 snapshot metadata | JH / 설계 저장 | 리소스 매핑, 필수 설정 확인 |
| `terraformArtifactId` | SW / Terraform 변환 | 배포 기준 Terraform artifact 고정 |
| `objectKey` | SW / S3 artifact 저장 | S3 artifact 위치 참조 |
| `artifactHash` | SW / Terraform 변환 | 승인 후 artifact 변경 여부 확인 |
| Terraform validation result | SW / Terraform 변환 | 배포 전 validation 실패 차단 |
| 비용/위험 분석 결과 | GG / AI 분석 | blocked 여부와 warning 표시 |
| 사용자 승인 결과 | YS UI 또는 CK API | 승인 시각과 승인 대상 기록 |

## 3. CK가 제공해야 하는 출력

| 출력 | 소비 파트 | 쓰는 곳 |
| --- | --- | --- |
| `Deployment` 상태 | YS | 프로젝트 보드, 배포 이력, 상세 화면 |
| plan summary | JH, GG, YS | 리소스별 변경 표시, 위험/비용 설명, 승인 UI |
| blocked 정보 | GG, YS | 차단 사유 설명, 버튼 비활성화, 알림 |
| 배포 로그 | GG, YS | 로그 화면, 오류 설명 입력 |
| 실패 단계와 사유 | GG, YS | 사용자 안내, AI 오류 설명 |
| 배포 이력 | YS | 프로젝트 상세, 활동 내역 |

## 4. 파트별 협의 항목

### 4.1 JH 설계 파트

- `architectureId`가 저장/불러오기/자동 저장 후에도 배포 기준으로 안정적인지
- `ResourceNode.id`가 Terraform resource mapping까지 유지되는지
- VPC, EC2, RDS, S3 필수 파라미터 이름이 무엇인지
- 연결선이 Terraform dependency로 해석되는 기준이 무엇인지
- plan summary를 다이어그램 노드에 표시할 때 어떤 key를 쓸지

### 4.2 SW Terraform 변환 파트

- Terraform artifact가 단일 `.tf` 파일인지 zip bundle인지
- `project_assets` metadata에 `artifactHash`를 저장할 수 있는지
- artifact의 `architectureId`가 배포 기준 snapshot과 항상 연결되는지
- Terraform validation result를 어떤 shape으로 CK에 넘길지
- plan summary를 SW가 만들지, CK가 후순위 실제 Terraform JSON에서 만들지

### 4.3 GG 비용/위험 분석 파트

- 비용/위험 분석 결과 schema
- `blocked`, `warning`, `info` 같은 severity 기준
- blocked 판단을 CK가 deterministic rule로 사용할 수 있는지
- Terraform 오류 설명 입력 payload가 `{ stage, rawMessage, relatedResourceId? }`와 맞는지
- AI 설명 결과를 저장하지 않는 MVP 방식을 UI 흐름과 맞출 수 있는지

### 4.4 YS 플랫폼/품질 파트

- 승인 UI 위치
- 배포 이력 상세 화면 구조
- blocked 상태 표시 방식
- 인증 전 `workspaceId`와 인증 후 `userId` 연결 방식
- 활동 내역과 알림 이벤트 이름
- 배포 로그 화면의 empty/loading/error 상태

### 4.5 공통 DB/API

- `Deployment`, `DeploymentLog`, `TerraformArtifact` 최종 필드명
- 공통 API 성공/실패 응답 형식
- blocked, validation error, permission error 구분 방식
- 로그 보관 범위와 마스킹 기준

## 5. 협의 우선순위

1. SW와 Terraform artifact 계약을 먼저 맞춘다.
2. JH와 `architectureId`, resource mapping 기준을 맞춘다.
3. GG와 blocked 판단 기준을 맞춘다.
4. YS와 승인 UI, 배포 이력 화면 위치를 맞춘다.
5. 공통 DB/API 필드명을 `docs/data-models.md`와 맞춘다.

## 6. 아직 열려 있는 질문

- Terraform artifact는 단일 `.tf` 파일인가, zip bundle인가?
- artifact hash는 SW가 생성해서 저장하는가?
- GG의 blocked 기준은 어떤 severity부터인가?
- YS 승인 UI는 workspace 화면 안에 둘 것인가, 별도 상세 화면에 둘 것인가?
- 인증 도입 전 `approvedBy`는 `workspaceId`로 볼 것인가?
