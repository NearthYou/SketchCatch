# Terraform 표현식 확장 설계 메모

이 문서는 현재 Terraform 동기화가 지원하지 않는 표현을 나중에 확장하려면 먼저 어떤 표현 모델과 UI 정책을 정해야 하는지 정리한다.

현재 #29 범위는 literal value sync다. 즉, 우리가 생성한 제한된 Terraform HCL subset에서 확실히 값으로 읽을 수 있는 것만 `DiagramJson.nodes[].parameters.values`에 반영한다.

나중에 `module`, `variable`, `locals`, `output`, function call, interpolation, ternary, for expression, nested block, heredoc 등을 지원하려면 단순 `Record<string, unknown>` 값 저장만으로는 부족하다.

## 현재 모델의 한계

현재 `DiagramNode.parameters.values`는 아래 형태다.

```ts
values: Record<string, unknown>;
```

이 구조는 아래 값을 표현하기 좋다.

```ts
{
  cidrBlock: "10.0.0.0/16",
  enableDnsSupport: true,
  tags: {
    Name: "main-vpc"
  }
}
```

하지만 아래 Terraform 표현들은 단순 값이 아니다.

```hcl
cidr_block = var.vpc_cidr
name = "${var.prefix}-bucket"
count = var.enabled ? 1 : 0
subnet_ids = [for subnet in var.subnets : subnet.id]
cidr_block = cidrsubnet(var.vpc_cidr, 8, 1)
```

이 값들을 그냥 string으로 저장하면 사용자가 입력한 Terraform expression인지, 일반 문자열인지, reference인지 구분하기 어렵다.

## 확장 방향

표현식을 지원하려면 parameter value를 단순 값에서 “값의 종류를 가진 구조”로 넓히는 방향을 검토한다.

예시:

```ts
type ParameterValue =
  | {
      kind: "literal";
      value: unknown;
    }
  | {
      kind: "reference";
      expression: string;
    }
  | {
      kind: "expression";
      expression: string;
      summary?: string;
    };
```

이렇게 나누면 UI와 generator가 같은 값을 다르게 다룰 수 있다.

- `literal`: 일반 입력 UI로 수정 가능
- `reference`: reference picker나 읽기 전용 칩으로 표시 가능
- `expression`: advanced mode에서만 편집 가능

## UI 정책 결정이 필요한 이유

Terraform expression은 단순 입력 필드와 다르다.

예를 들어 아래 값은 화면에 어떻게 보여줘야 할까?

```hcl
cidr_block = cidrsubnet(var.vpc_cidr, 8, 1)
```

가능한 UI 정책은 여러 가지다.

1. 읽기 전용으로 보여준다.
2. advanced expression editor에서만 수정하게 한다.
3. function call을 구조화된 폼으로 분해한다.
4. 지원하지 않는 expression으로 유지하고 diagnostic만 보여준다.

정답은 하나가 아니다. 그래서 parser를 확장하기 전에 UI 정책을 먼저 정해야 한다.

## 확장 대상별 결정 사항

### variable

예:

```hcl
variable "vpc_cidr" {
  type = string
  default = "10.0.0.0/16"
}
```

결정할 것:

- variable 정의를 어디에 저장할 것인가?
- `DiagramJson`에 저장할 것인가, 별도 `TerraformVariable` 모델을 둘 것인가?
- 사용자가 node parameter에서 `var.vpc_cidr`를 선택할 수 있어야 하는가?
- default 값 변경이 node 값 변경인가, Terraform project 설정 변경인가?

추천:

- variable 정의는 node 하나의 값이 아니므로 `parameters.values`에 직접 넣지 않는다.
- 별도 Terraform project-level 모델을 둔다.
- node parameter에는 `reference`로 `var.vpc_cidr`를 저장한다.

### locals

예:

```hcl
locals {
  name_prefix = "sketchcatch"
}
```

결정할 것:

- local 값을 전역 계산 값으로 볼 것인가?
- 어떤 node가 local을 소유하는가?
- local 변경이 여러 node에 영향을 줄 때 UI에서 어떻게 표시할 것인가?

추천:

- `locals`도 node-local parameter가 아니라 project-level expression scope로 둔다.
- node에는 local reference만 저장한다.

### output

예:

```hcl
output "bucket_name" {
  value = aws_s3_bucket.logs.bucket
}
```

결정할 것:

- output은 설계 입력인가, 배포 후 사용자에게 보여줄 결과인가?
- `Deployment` output과 어떻게 연결할 것인가?
- Architecture Board에서 output node로 보여줄 것인가?

추천:

- output은 `Deployment` 또는 `TerraformArtifact` 쪽과 연결한다.
- `DiagramNode.parameters.values`에 넣지 않는다.

### provider

예:

```hcl
provider "aws" {
  region = "ap-northeast-2"
}
```

결정할 것:

- provider region은 프로젝트 설정인가, AWS connection 설정인가?
- node별 provider alias를 지원할 것인가?
- 멀티 리전/멀티 provider는 언제 지원할 것인가?

추천:

- MVP에서는 provider 설정을 node parameter가 아니라 project/deployment 설정으로 둔다.
- `DiagramJson`에는 provider identity만 필요할 때 최소로 반영한다.

### module

예:

```hcl
module "vpc" {
  source = "terraform-aws-modules/vpc/aws"
  cidr = "10.0.0.0/16"
}
```

결정할 것:

- module을 보드 node 하나로 볼 것인가?
- module 내부 리소스를 펼쳐서 보여줄 것인가?
- module input/output을 어떻게 표현할 것인가?

추천:

- module은 일반 resource node와 분리된 `ModuleNode` 또는 `ComponentNode` 개념이 필요하다.
- 내부 리소스를 자동 생성하는 것은 별도 import/expand 기능으로 다룬다.

### function call

예:

```hcl
cidr_block = cidrsubnet(var.vpc_cidr, 8, 1)
```

결정할 것:

- function call을 string expression으로 보존할 것인가?
- UI에서 함수 이름과 인자를 분리해서 보여줄 것인가?
- 함수 결과를 계산하려고 할 것인가?

추천:

- 처음에는 계산하지 않고 `expression`으로 보존한다.
- UI에는 “고급 Terraform 표현식”으로 표시한다.

### interpolation

예:

```hcl
bucket = "${var.prefix}-logs"
```

결정할 것:

- interpolation을 일반 string과 구분할 것인가?
- 문자열 일부만 reference picker로 수정할 수 있게 할 것인가?

추천:

- 일반 string으로 저장하지 않는다.
- `expression`으로 보존한다.

### ternary

예:

```hcl
count = var.enabled ? 1 : 0
```

결정할 것:

- 조건식을 UI에서 어떻게 설명할 것인가?
- 조건식의 결과 타입을 어떻게 검증할 것인가?

추천:

- `expression`으로 보존하고 advanced editor에서만 수정한다.

### for expression

예:

```hcl
subnet_ids = [for subnet in var.subnets : subnet.id]
```

결정할 것:

- 반복식 결과가 list라는 점을 type system에 반영할 것인가?
- UI에서 반복 원본과 결과를 설명할 것인가?

추천:

- 초기 확장에서는 `expression`으로 보존한다.
- list item 편집 UI로 분해하지 않는다.

### nested block

예:

```hcl
filter {
  name = "name"
  values = ["ubuntu"]
}
```

결정할 것:

- `values` 안에 nested block 구조를 어떻게 저장할 것인가?
- 같은 nested block이 여러 번 나올 수 있는 경우 배열로 표현할 것인가?
- resource parameter catalog와 어떻게 맞출 것인가?

추천:

```ts
type NestedBlockValue = {
  kind: "nestedBlock";
  blocks: Array<{
    type: string;
    values: Record<string, ParameterValue>;
  }>;
};
```

다만 이 구조는 `DiagramNode.parameters.values` 타입 변경이 크므로 별도 이슈로 다룬다.

### heredoc

예:

```hcl
user_data = <<EOF
#!/bin/bash
echo hello
EOF
```

결정할 것:

- heredoc을 일반 multiline string으로 볼 것인가?
- 민감 정보나 script 내용을 어떻게 표시하고 검증할 것인가?
- syntax highlighting이 필요한가?

추천:

- multiline literal로 저장하되, UI는 textarea/code editor를 사용한다.
- secret-like content는 diagnostics에서 경고한다.

## 권장 확장 순서

1. `nested block`
2. `reference`와 `variable` 참조 모델
3. `locals`
4. `output`
5. `heredoc`
6. `function call`, `interpolation`, `ternary`, `for expression`
7. `module`

이 순서를 추천하는 이유는 사용자 가치와 모델 난이도 때문이다.

`nested block`은 `aws_ami` data source 같은 실제 AWS 리소스에서 바로 필요하다. 반면 `module`은 보드 모델 자체를 바꿀 가능성이 크므로 나중에 다룬다.

## 구현 전 체크리스트

확장 이슈를 시작하기 전에 아래 질문에 답해야 한다.

- 이 표현은 node 하나의 parameter인가, project-level Terraform 설정인가?
- UI에서 일반 입력으로 수정 가능한가, advanced editor가 필요한가?
- round-trip 후 원래 Terraform 표현을 보존해야 하는가?
- 이 표현을 Terraform 생성기도 다시 출력할 수 있는가?
- 이 표현이 `DiagramJson`, `ArchitectureJson`, `InfrastructureGraph` 중 어디에 들어가야 하는가?
- 실패 시 기존처럼 all-or-nothing으로 처리할 것인가?

## 결론

현재 동기화는 일부러 좁다. 부족해서가 아니라, 잘못 해석해서 보드 상태를 망가뜨리지 않기 위해서다.

확장은 가능하지만, parser부터 넓히면 안 된다. 먼저 표현 모델과 UI 정책을 정해야 한다.

좋은 확장 방향은 `literal value sync`에서 `Terraform expression-aware sync`로 가는 것이다.
