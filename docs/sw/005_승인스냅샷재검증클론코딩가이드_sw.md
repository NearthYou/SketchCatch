# Direct Deployment 승인 스냅샷 재검증 클론 코딩 가이드

이 문서는 Direct Deployment Path에서 사용자가 Terraform Plan을 승인한 뒤, apply 직전에 같은 artifact, 같은 tfplan, 같은 AWS account/region인지 다시 확인하는 흐름을 클론 코딩하기 위한 참고 문서다. SketchCatch의 배포 실행은 AWS-first로 구현되어 있지만, 승인 스냅샷 개념은 Provider Adapter가 늘어나도 유지되어야 하는 안전 경계다.

## 왜 승인 스냅샷이 필요한가

Terraform Plan은 생성 시점의 입력과 실행 대상에 묶여 있다. 사용자가 승인한 뒤 아래 값 중 하나라도 바뀌면, 사용자가 실제로 승인한 계획과 apply할 계획이 달라진다.

- Terraform artifact id 또는 artifact content hash
- Terraform binary plan 파일의 hash
- 실행 대상 AWS account
- 실행 대상 AWS region

따라서 approval은 단순한 boolean이 아니라, 승인 당시의 증거를 `deployments` 레코드에 스냅샷으로 남기는 잠금이다.

## 승인 시 저장하는 값

`approveDeploymentPlan`은 현재 deployment, plan artifact, Terraform artifact, verified AWS connection을 다시 읽고 아래 값을 저장한다.

| 필드 | 의미 |
| --- | --- |
| `approvedAt` | 사용자가 승인한 시각 |
| `approvedByUserId` | 승인한 사용자 |
| `approvedTerraformArtifactId` | 승인된 Terraform artifact id |
| `approvedPlanArtifactId` | 승인된 tfplan artifact id |
| `approvedTerraformArtifactHash` | plan 생성 시점과 승인 시점에 일치한 Terraform artifact SHA-256 |
| `approvedTfplanHash` | 승인된 binary tfplan SHA-256 |
| `approvedAwsAccountId` | 승인된 AWS account |
| `approvedAwsRegion` | 승인된 AWS region |

승인 전에 `currentPlanArtifact.terraformArtifactSha256`가 비어 있으면 plan source hash가 없으므로 승인하지 않는다. 이 경우 사용자는 Terraform Plan을 다시 생성해야 한다.

## 재검증 흐름

1. Plan 생성 시 `DeploymentPlanArtifact`에 `terraformArtifactSha256`, `sha256`, `accountId`, `region`을 저장한다.
2. 승인 시 Terraform artifact를 다시 다운로드하고 안전 검사를 통과시킨 뒤 SHA-256을 계산한다.
3. 승인 시 계산한 artifact hash가 plan artifact의 `terraformArtifactSha256`와 다르면 `Terraform artifact changed after plan`으로 막는다.
4. 승인 시 verified AWS connection의 `accountId`와 `region`이 plan artifact의 값과 다르면 `AWS connection changed after plan`으로 막는다.
5. apply 시작 시 binary tfplan을 다시 다운로드하고 SHA-256을 계산한다.
6. apply 시작 시 workspace의 Terraform artifact를 다시 읽고 SHA-256을 계산한다.
7. `assertDeploymentApplyPreconditions`가 승인 스냅샷 필드 존재, apply plan operation, artifact id, plan artifact id, artifact hash, tfplan hash, AWS account, AWS region을 모두 비교한다.
8. 재검증이 실패하면 AWS credential 준비, plan file write, `terraform init`, `terraform apply` 전에 실패한다.

Destroy도 같은 승인 스냅샷을 사용한다. 다만 `operation`은 `destroy`여야 하고, destroy 가능한 source status와 `stateObjectKey`가 추가로 필요하다.

## 클론 코딩 순서

1. `apps/api/src/deployments/deployment-approval-service.test.ts`에서 `assertDeploymentApplyPreconditions` 테스트를 먼저 추가한다.
2. artifact hash drift, tfplan hash drift, AWS account drift, AWS region drift를 각각 명시적인 케이스로 만든다.
3. `approvedAt`, `approvedByUserId`, `approvedTerraformArtifactId`, `approvedPlanArtifactId`, `approvedTerraformArtifactHash`, `approvedTfplanHash`, `approvedAwsAccountId`, `approvedAwsRegion` 중 하나가 빠지면 apply가 거절되는지 확인한다.
4. plan source hash 누락은 approval 단계에서 테스트한다. `createPlanArtifactRecord({ terraformArtifactSha256: null })`처럼 만든 뒤 `Terraform Plan must be regenerated before approval`을 기대한다.
5. `apps/api/src/deployments/deployment-apply-service.test.ts`에서 `runDeploymentApply`가 같은 drift를 apply 진입점에서도 막는지 확인한다.
6. apply-service 테스트에서는 `prepareTerraformAwsCredentialEnv`, `writePlanFile`, `runTerraformInit`이 호출되지 않았는지 함께 검증한다.
7. 테스트가 실패할 때만 production code를 수정한다. 현재 경계는 `deployment-approval-service.ts`의 approval snapshot 저장과 `assertDeploymentApplyPreconditions`에 모여 있어야 한다.
8. destroy precondition 테스트가 깨지지 않도록 `deployment-destroy-service.test.ts`도 targeted run에 포함한다.

## 확인 명령

```powershell
pnpm --filter @sketchcatch/api exec tsx --test src/deployments/deployment-approval-service.test.ts src/deployments/deployment-apply-service.test.ts src/deployments/deployment-destroy-service.test.ts
pnpm --filter @sketchcatch/api test
pnpm harness:check
```

실제 AWS apply/destroy는 이 학습 흐름에서 실행하지 않는다. Direct Deployment 실행은 승인, 로그, secret masking, 실패 상태 기록, cleanup 경계가 갖춰진 API/worker 코드 안에서만 다룬다.
