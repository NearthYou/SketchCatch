# Terraform 변환 동기화 클론 코딩 가이드

이 문서는 이슈 #29 `Terraform 코드 수정 사항을 DiagramJson에 반영`을 구현할 때 기존 Terraform 생성 코드의 구조를 그대로 따라갈 수 있게 만든 작업 가이드다.

## 목표

사용자가 생성된 Terraform HCL subset을 수정하면, 그 변경을 기존 `DiagramJson.nodes[].parameters.values`에 안전하게 반영한다.

핵심 흐름은 아래처럼 본다.

```text
DiagramJson -> InfrastructureGraph -> Terraform
Terraform -> InfrastructureGraph patch -> DiagramJson
```

`DiagramJson`과 Terraform 중 하나를 유일한 source of truth로 보지 않는다. 두 편집 표면은 `InfrastructureGraph`를 통해 같은 Resource identity와 IaC identity를 공유한다.

## 클론할 기존 패턴

먼저 아래 파일을 기준으로 같은 구조를 만든다.

| 기존 파일 | 새로 만들 파일 | 따라 할 점 |
| --- | --- | --- |
| `apps/api/src/services/terraform/diagram-to-terraform.ts` | `apps/api/src/services/terraform/terraform-to-diagram.ts` | 순수 함수, 작은 helper, mutation 없는 반환 |
| `apps/api/src/services/terraform/diagram-to-terraform.test.ts` | `apps/api/src/services/terraform/terraform-to-diagram.test.ts` | `node:test`, `assert`, `makeNode` helper |
| `apps/api/src/services/terraform/terraform-diagnostics.ts` | 필요 시 같은 파일 또는 새 helper | diagnostic code, line, resourceAddress 형식 |
| `apps/api/src/routes/terraform.ts` | 같은 파일에 sync route 추가 | Zod schema를 shared type과 맞춤 |

## 구현 순서

### 1. shared type부터 추가한다

`packages/types/src/index.ts`에 요청/응답 타입을 먼저 둔다.

```ts
export type TerraformSyncToDiagramRequest = {
  diagramJson: DiagramJson;
  terraformCode: string;
};

export type TerraformSyncToDiagramResponse = {
  diagramJson: DiagramJson;
  diagnostics: TerraformDiagnostic[];
};
```

동기화 실패 시 기존 `DiagramJson`을 변경하지 않아야 하므로, 응답의 `diagramJson`은 실패하면 입력과 같은 값이어야 한다.

### 2. 테스트 파일을 먼저 만든다

`diagram-to-terraform.test.ts`의 `makeNode` helper를 복사해서 시작한다. 최소 테스트는 아래 완료 조건과 1:1로 맞춘다.

- 생성기 output을 수정하면 같은 node의 `parameters.values`가 갱신된다.
- `(resourceType, resourceName)`으로 매칭할 수 없는 block은 거절된다.
- 지원 범위 밖 expression은 diagnostic을 반환한다.
- 동기화 실패 시 기존 `DiagramJson`은 변경되지 않는다.
- 기존 `edges`는 그대로 유지된다.
- `resource`와 `data` block을 둘 다 처리한다.

### 3. 공개 함수 시그니처를 작게 잡는다

새 service의 공개 함수는 하나로 시작한다.

```ts
export function syncTerraformToDiagramJson(
  diagramJson: DiagramJson,
  terraformCode: string
): TerraformSyncToDiagramResponse {
  // parse -> match -> patch -> return
}
```

이 함수는 DB, S3, Terraform CLI, Fastify request를 알면 안 된다. 문자열과 객체만 받아 새 객체를 반환하는 순수 함수로 둔다.

### 4. Terraform parser 범위를 의도적으로 제한한다

처음부터 모든 HCL을 파싱하지 않는다. 이슈 #29의 지원 범위만 허용한다.

허용:

- `resource "aws_vpc" "main" { ... }`
- `data "aws_ami" "ubuntu" { ... }`
- top-level 단순 attribute
- string, number, boolean, null
- list
- map/object

거절하고 diagnostic 반환:

- `module`, `provider`, `variable`, `output`, `locals`
- interpolation, function call, ternary, for expression
- nested block
- heredoc
- block header가 깨진 코드
- attribute 값 뒤에 남은 알 수 없는 토큰
- 같은 Terraform address 중복
- 기존 node와 매칭되지 않는 block

## 파싱 규칙

### block 식별

기존 diagnostics의 `BLOCK_HEADER_PATTERN`과 같은 기준을 쓴다.

```ts
/^\s*(resource|data)\s+"([^"]+)"\s+"([^"]+)"\s*\{\s*$/
```

block identity는 아래 세 값이다.

```text
terraformBlockType.resourceType.resourceName
```

예:

```text
resource.aws_vpc.main
data.aws_ami.ubuntu
```

Architecture Board에서 이미 연결된 AWS Resource의 block header 줄은 사용자가 직접 수정하지 못하게 막는 것을 기본 UX로 둔다. 예를 들어 아래 줄은 `resourceType`, `resourceName`, `terraformBlockType` identity를 담고 있으므로 Terraform editor에서는 잠금 처리한다.

```hcl
data "aws_ami" "ami" {
```

단, UI 잠금만 믿지 않는다. API/service는 외부 요청이나 붙여넣기 입력을 받을 수 있으므로 block header가 바뀌었거나 깨졌으면 반드시 diagnostic을 반환한다.

### node 매칭

기존 node는 아래 조건으로 찾는다.

```ts
node.kind === "resource"
node.parameters?.terraformBlockType ?? "resource"
node.parameters?.resourceType
node.parameters?.resourceName
```

매칭은 `(terraformBlockType, resourceType, resourceName)`을 함께 보는 것을 기본으로 한다. 이슈 본문에는 `(resourceType, resourceName)` 기준이라고 되어 있지만, `data`와 `resource`가 같은 type/name을 가질 수 있으므로 내부 key에는 block type까지 포함한다.

### attribute 이름 변환

기존 생성기는 `camelCase -> snake_case`로 출력한다. 역방향은 `snake_case -> camelCase`로 되돌린다.

예:

```text
cidr_block -> cidrBlock
enable_dns_support -> enableDnsSupport
```

map/object 내부 key는 생성기처럼 원래 이름을 유지한다. 예를 들어 `tags.Name`은 `Name` 그대로 둔다.

### 값 뒤 쓰레기 토큰 감지

attribute parser는 값을 읽은 뒤 같은 줄에 남은 문자가 있는지 확인해야 한다. 아래처럼 list나 string 값 뒤에 임의 문자열이 붙으면 Terraform subset으로 해석하지 말고 전체 sync를 실패시킨다.

```hcl
data "aws_ami" "ami" {
  owners = [
    "",
  ]
  filter_name = "cxzv"
  filter_values = [
    "",
  ]asdfasdf
}
```

이 경우 `filter_values`의 list 값은 읽을 수 있어 보이더라도 `]` 뒤에 `asdfasdf`가 남아 있으므로 `terraform.sync.trailing_tokens` diagnostic을 반환한다. 같은 규칙은 아래에도 적용한다.

```hcl
cidr_block = "10.0.0.0/16"abc
enabled = true???
tags = { Name = "main" }tail
```

허용되는 trailing token은 공백과 line comment뿐이다.

```hcl
cidr_block = "10.0.0.0/16" # ok
enabled = true // ok
```

## patch 규칙

성공 시에는 기존 node를 복사해서 `parameters.values`만 바꾼다.

유지할 것:

- `node.id`
- `node.position`
- `node.size`
- `node.locked`
- `node.zIndex`
- `node.style`
- `node.parameters.resourceType`
- `node.parameters.resourceName`
- `node.parameters.fileName`
- `edges`
- `viewport`

바꿀 것:

- `node.parameters.values`

자동으로 하지 말 것:

- 새 node 생성
- edge 생성
- node 삭제
- `resourceName` rename 추론
- Terraform reference를 edge로 변환

## diagnostic 규칙

지원하지 않는 입력을 만나면 mutation 대신 diagnostic을 반환한다. 에러가 하나라도 있으면 전체 `DiagramJson`은 입력값 그대로 반환한다.

권장 code:

| code | 상황 |
| --- | --- |
| `terraform.sync.unsupported_block` | `resource`/`data`가 아닌 block |
| `terraform.sync.block_header` | block header 형식 오류 |
| `terraform.sync.unmatched_block` | 기존 node를 찾을 수 없음 |
| `terraform.sync.unsupported_expression` | 함수, interpolation, for expression 등 |
| `terraform.sync.nested_block` | nested block 발견 |
| `terraform.sync.duplicate_address` | 같은 block identity 중복 |
| `terraform.sync.trailing_tokens` | attribute 값 뒤에 알 수 없는 토큰이 남음 |

`resourceAddress`에는 가능하면 `resource.aws_vpc.main` 같은 block identity를 넣는다. 특정 줄에서 감지한 문제는 `line`을 넣는다.

## API 연결

service 테스트가 먼저 통과하면 `apps/api/src/routes/terraform.ts`에 route를 추가한다.

권장 path:

```text
POST /terraform/sync-to-diagram
```

request:

```ts
type TerraformSyncToDiagramRequest = {
  diagramJson: DiagramJson;
  terraformCode: string;
};
```

response:

```ts
type TerraformSyncToDiagramResponse = {
  diagramJson: DiagramJson;
  diagnostics: TerraformDiagnostic[];
};
```

route도 기존 `/terraform/generate`, `/terraform/validate`처럼 `requireActiveUserId`를 먼저 호출한다.

## 처음 작성할 테스트 예시

```ts
test("updates values for a matching generated resource block", () => {
  const diagramJson: DiagramJson = {
    nodes: [
      makeNode({
        id: "node-1",
        type: "aws_vpc",
        kind: "resource",
        label: "main_vpc",
        parameters: {
          terraformBlockType: "resource",
          resourceType: "aws_vpc",
          resourceName: "main",
          fileName: "main",
          values: {
            cidrBlock: "10.0.0.0/16"
          }
        }
      })
    ],
    edges: [],
    viewport: { x: 0, y: 0, zoom: 1 }
  };

  const result = syncTerraformToDiagramJson(
    diagramJson,
    `resource "aws_vpc" "main" {
  cidr_block = "10.1.0.0/16"
}`
  );

  assert.deepEqual(result.diagnostics, []);
  assert.equal(result.diagramJson.nodes[0]?.parameters?.values.cidrBlock, "10.1.0.0/16");
});
```

이 테스트가 초록색이 되면 list, map, data block, unmatched block, unsupported expression 순서로 확장한다.
