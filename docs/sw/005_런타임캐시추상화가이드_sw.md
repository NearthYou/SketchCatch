# Runtime Cache 추상화와 메모리 fallback 가이드

이 문서는 `apps/api/src/runtime-cache`의 최소 추상화와 메모리 fallback을 이해하고, 다음 Redis adapter 작업에서 같은 계약을 유지하기 위한 SW 참고 문서다. 확정 데이터 계약은 `docs/data-models.md`, 실행 경계는 `docs/architecture.md`를 우선한다.

## 책임 경계

Runtime Cache는 Deployment, Reverse Engineering, Git/CI/CD Integration처럼 오래 걸리는 workflow의 진행 상태, 짧은 TTL status, polling 보조 metadata, log streaming 보조 데이터를 다룬다. 원천 기록이 아니며 cache miss나 Redis 재시작이 있어도 최종 결과는 RDS/S3에서 복구하거나 workflow 상태를 다시 조회할 수 있어야 한다.

Redis는 내부 Runtime Cache 인프라다. 사용자 Practice Architecture Resource가 아니므로 `ResourceType`에 추가하지 않고, 사용자가 설계한 아키텍처의 노드처럼 보여주지 않는다.

## RDS, S3, Redis 관계

| 저장소 | 책임 | 예시 |
| --- | --- | --- |
| RDS | 원천 데이터와 조회 가능한 metadata | project, architecture snapshot, deployment record, plan summary, artifact metadata |
| S3 | 파일성 산출물과 큰 artifact | Terraform 파일, `tfplan`, state/output artifact, export zip, thumbnail |
| Redis Runtime Cache | 짧은 TTL의 실행 중 상태와 streaming 보조 | deployment progress, reverse engineering scan status, Git/CI/CD handoff polling marker |

서비스가 완료되면 사용자가 다시 봐야 하는 결과는 RDS/S3에 남겨야 한다. Runtime Cache에는 "지금 처리 중인지", "마지막 polling cursor가 무엇인지", "streaming으로 보여줄 임시 로그 조각이 무엇인지"처럼 없어져도 재동기화 가능한 값만 둔다.

## Key convention sketch

코드 계약은 `namespace + key + JSON value + ttlMs`다. Redis adapter를 붙일 때는 아래처럼 환경과 namespace를 포함한 물리 key를 만들 수 있다.

```text
sketchcatch:{env}:{namespace}:{key}
```

namespace 후보:

- `deployment.status`
- `deployment.log-stream`
- `reverse-engineering.scan`
- `git-ci.handoff`

key 후보:

- `deployment:{deploymentId}`
- `scan:{scanId}`
- `handoff:{handoffId}`

운영 규칙:

- raw user input을 key에 직접 넣지 않는다.
- provider-neutral workflow id를 우선 사용한다.
- value는 JSON-serializable 값만 저장한다.
- TTL은 workflow 성격별로 짧게 둔다. 완료 후 장기 조회가 필요하면 RDS/S3에 기록한다.
- cache key 이름이 사용자 Practice Architecture Resource 이름처럼 보이지 않게 한다.

## Clone-coding steps

1. `apps/api/src/runtime-cache/runtime-cache.ts`에서 `RuntimeCache` interface를 먼저 읽는다.
2. 테스트나 로컬 fallback이 필요하면 `createInMemoryRuntimeCache()`를 주입한다.
3. TTL 테스트는 `now` 함수를 주입해서 시간 진행을 직접 제어한다.
4. 캐시 value는 JSON으로 직렬화된 복사본처럼 다룬다. set 이후 원본 객체를 바꾸거나 get 결과를 바꿔도 저장된 값이 함께 바뀌면 안 된다.
5. Redis adapter 작업에서는 package 추가와 연결 설정을 adapter 내부로 제한하고, consumer가 `RuntimeCache` interface만 보게 유지한다.
6. Deployment route 통합 작업에서는 cache hit를 원천 진실로 취급하지 말고, 최종 이력과 artifact는 RDS/S3 기록을 기준으로 응답한다.
