# GitHub · CodeBuild · ECS/Fargate 통합 배포 단계별 구현 계획

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:executing-plans` to implement this plan task-by-task. 각 작업은 아래 체크박스 순서대로 진행하고, 단계별 집중 테스트가 통과하기 전 다음 단계로 넘어가지 않는다.

**Goal:** Repository 분석으로 확정한 한 commit을 한 번만 빌드해 SketchCatch 내부 S3에 보관하고, 승인된 Terraform Apply 뒤 trusted worker가 같은 Artifact를 사용해 ECS와 S3/CloudFront에 배포한다.

**Architecture:** 사용자 AWS 계정의 CodeBuild는 Repository 코드를 실행해 배포 후보 Artifact만 만든다. SketchCatch API는 DB의 lease와 단계 기록을 관리하고 CodeBuild를 시작·중단한다. 실제 ECR/ECS/S3/CloudFront 변경은 SketchCatch의 trusted release worker가 최소 권한 세션으로 수행한다. GitHub Actions도 AWS를 직접 변경하지 않고 SketchCatch release-run API만 호출한다.

**Tech Stack:** TypeScript, Fastify, Drizzle/PostgreSQL, AWS SDK v3, ECS RunTask worker, CodeBuild, CodeConnections, S3, ECR, ECS/Fargate, ALB, CloudFront, Terraform, Next.js.

**구현 상태 (2026-07-16):** Task 1~17의 source contract와 focused test 구현은 현재 브랜치에 반영됐다.
Task 18의 source 구현과 repository checks까지 완료했다. 실제 AWS acceptance는 비용과 cloud mutation이 있어
구현 완료 체크와 분리하며, 사용자 승인 전에는 실행하지 않는다.

## 1. 구현 원칙과 고정 범위

- 현재 브랜치 `codex/demo-repository-owner-parity`에서 구현한다.
- 기존 `S3 + CloudFront + ALB + ECS/Fargate` 웹 포함 스펙을 유지한다.
- 이번 구현 대상은 Repository evidence가 API와 frontend를 모두 확인한 ECS/Fargate 프로젝트다.
- `whiskend/audience-live-check`와 `jh-9999/audience-live-check`는 Repository 소유자만 다르고 같은 분석·추천·기본값·배포 흐름을 사용한다.
- CodeBuild는 빌드 전용이다. ECR push, ECS service update, 서비스 S3 upload, CloudFront invalidation을 하지 않는다.
- preflight는 Repository 자체 unit/integration/E2E를 기본 실행하지 않는다. Docker image build, 임시 컨테이너 health, frontend production build만 수행한다.
- 사용자가 승인한 Artifact와 실제 배포 Artifact는 동일해야 한다. 승인 뒤 재빌드하지 않는다.
- 최종 Output URL은 CloudFront HTTPS URL이다. ALB URL은 내부 API origin 검증에만 사용한다.
- ECS 애플리케이션 실패는 직전 RDS `succeeded` release로 자동 rollback한다.
- frontend 실패는 ECS 성공을 되돌리지 않고 `PARTIALLY_FAILED`로 기록하며 같은 Artifact로 재시도한다.
- 이번 데모에서는 ECS와 S3의 완전한 동시 rollback을 구현하지 않는다.
- 동시 실행은 queue를 만들지 않고 두 번째 실행을 차단하고 이유를 안내한다.
- 실제 AWS Apply와 비용 발생 리소스 생성은 별도의 명시적 승인 없이는 실행하지 않는다.

## 2. 현재 코드에서 교체할 경계

기존 `apps/api/src/deployments/aws-codebuild-direct-application-release-gateway.ts`의 CodeBuild 기반 ECR push/ECS 변경 경계는 아래 구조로 교체했다.

```text
Repository commit
  -> SketchCatch API가 preflight/Plan용 project lease 획득
  -> user AWS CodeBuild: preflight-build
  -> SketchCatch internal S3: immutable ReleaseCandidate
  -> Terraform Plan 저장 후 lease 해제
  -> 사용자 승인 대기
  -> Apply 요청에서 project lease 재획득 및 승인 snapshot 재검증
  -> Terraform Apply와 output/resource inventory 저장
  -> SketchCatch trusted worker: release-activate
       -> OCI archive를 ECR에 push
       -> ECS revision 활성화 및 새 task health 검증
       -> frontend assets를 S3에 업로드
       -> index.html을 마지막에 활성화
       -> CloudFront /* invalidation 완료 확인
       -> CloudFront / commit marker, /health, /api/* 검증
  -> Deployment/ApplicationRelease terminal 상태 기록
  -> lease 해제
```

`apps/api/src/git-cicd/git-cicd-workflows.ts`의 ECS workflow도 CodeBuild/AWS CLI를 직접 실행하지 않고 다음 요청만 보낸다.

```http
POST /api/projects/:projectId/release-runs
Authorization: GitHub OIDC에서 교환한 짧은 수명의 SketchCatch token
Idempotency-Key: <repository>:<commitSha>:<workflowRunId>

{
  "source": "gitops",
  "commitSha": "<40 또는 64자리 sha>",
  "repository": { "owner": "...", "name": "..." }
}
```

- `deployment_status` enum에 `PARTIALLY_FAILED`, `PARTIALLY_CANCELED` 추가

- `deployments`에 AWS account/region/connection name snapshot과 `release_candidate_id` 추가

- `project_deployment_targets.connection_id`를 nullable, FK `ON DELETE SET NULL`로 변경

DB 불변식:

- 프로젝트당 active lease는 최대 하나다.
- `fencing_version`은 lease 재획득 시 반드시 증가한다.
- candidate는 direct deployment 또는 pipeline run 중 정확히 하나를 참조한다.
- release는 자기 프로젝트의 candidate와 baseline만 참조한다.
- large Artifact는 S3에 있고 DB에는 object key, digest, byte size, retention 시간만 둔다.

| 현재 상태 | 이벤트 | 다음 상태 | 필수 증거 |
|---|---|---|---|
| `PENDING` | lease 획득, preflight 시작 | `RUNNING` | holder, fencing version, CodeBuild ID |
| `RUNNING` | preflight 실패 | `FAILED` | preflight failure stage, sanitized log |
| `RUNNING` | Plan 승인 전 취소 | `CANCELLED` | CodeBuild/worker terminal 확인 |
| `RUNNING` | Apply 실패 | `FAILED` | Terraform state/output/resource best-effort 보존 |
| `RUNNING` | ECS 실패 후 rollback 성공 | `FAILED` | baseline revision과 복구 health evidence |
| `RUNNING` | ECS 성공, frontend 실패 | `PARTIALLY_FAILED` | ECS success evidence, frontend failure stage |
