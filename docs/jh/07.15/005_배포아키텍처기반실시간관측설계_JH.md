# 배포 Architecture 기반 실시간 관측 설계

## 1. 목적

Live Observation은 특정 Auto Scaling 데모 구조만 보여주는 기능이 아니다. 사용자가 선택한 `Deployment`가 실제로 배포한 Practice Architecture 전체를 읽기 전용으로 보여주고, AWS에서 실시간으로 확인할 수 있는 Resource에만 상태와 지표를 덧씌우는 기능이다.

이 설계는 다음 문제를 해결한다.

- 고정 용량 ECS Fargate가 `max_capacity`, `scale_out_threshold`를 갖지 않는다는 이유로 관측 시작이 차단된다.
- 현재 Live Observation UI는 배포 Architecture 전체가 아니라 미리 정한 트래픽·용량 경로를 전제로 한다.
- 한 번 `manifest_invalid`가 저장되면 구성이나 서버 설정이 정상화되어도 같은 Deployment를 다시 검증할 수 없다.
- 관측을 통과시키기 위해 사용자가 선택하지 않은 Service Auto Scaling을 Terraform에 추가할 위험이 있다.

## 2. 확정된 제품 결정

1. 관측 화면의 기준은 현재 편집 중인 `project_drafts.diagram_json`이 아니라 선택한 `Deployment.architectureId`가 가리키는 불변 `architectures.architecture_json`이다.
2. 배포된 모든 Resource와 연결을 관측 화면에 표시한다.
3. 실시간 상태를 지원하지 않는 Resource도 숨기지 않는다. 해당 Resource는 `배포됨 · 관측 데이터 없음` 상태로 표시한다.
4. ECS Fargate의 Service Auto Scaling은 선택 기능이다. Live Observation을 사용하기 위해 Auto Scaling을 강제로 추가하지 않는다.
5. 고정 용량 ECS Fargate는 `desiredCount`, `runningCount`, healthy target 수를 관측한다.
6. Service Auto Scaling이 있는 ECS Fargate만 min/max capacity와 scaling policy 정보를 추가로 표시한다.
7. `scale_out_threshold`와 `max_capacity`가 없다는 이유만으로 Deployment 전체를 관측 불가로 처리하지 않는다.
8. 실제 트래픽을 청중용 페이지로 전달하기 전의 CloudFront, S3, ALB, Target Group, ECS identity 검증은 유지한다.
9. `manifest_invalid`는 영구적인 차단 상태가 아니다. 사용자의 명시적인 `관측 시작` 요청에서 다시 검증하고, 성공하면 valid manifest로 교체한다.
10. 이번 변경은 Terraform Resource를 자동 추가하거나 현재 AWS 인프라를 변경하지 않는다.

## 3. 검토한 접근

### 3.1 채택: 배포 Architecture + 선택적 관측 capability overlay

`ArchitectureJson`은 모든 Resource와 연결을 제공한다. Provider Adapter는 실제로 조회할 수 있는 Resource에 대해서만 observation binding과 실시간 상태를 제공한다.

장점:

- 무엇을 배포하든 배포 Architecture 전체를 보여줄 수 있다.
- 지원하지 않는 Resource가 있어도 관측 화면 전체가 실패하지 않는다.
- AWS-first 구현을 유지하면서 다른 Provider Adapter로 확장할 수 있다.
- 관측을 위해 Terraform 인프라 의미를 바꾸지 않는다.

비용:

- Architecture 표시와 AWS 관측 상태를 Resource 단위로 연결하는 binding 계층이 필요하다.
- 모든 Resource에 동일한 수준의 실시간 지표를 제공할 수는 없다.

### 3.2 제외: 관측 가능한 트래픽 경로만 잘라서 표시

현재 focused path 방식과 가깝지만, 사용자가 배포한 전체 Architecture를 보여달라는 요구와 충돌한다. 경로 분석에 실패하면 전체 UI가 비어 보이는 문제도 유지된다.

### 3.3 제외: 모든 ECS Fargate에 Service Auto Scaling 강제 추가

관측 기능을 통과시키기 위해 사용자가 선택하지 않은 `aws_appautoscaling_target`과 `aws_appautoscaling_policy`를 추가하면 실제 인프라 동작과 비용이 바뀐다. 관측 기능이 Practice Architecture를 무승인으로 변경하므로 제품 안전 경계에도 맞지 않는다.

## 4. 표시 기준과 데이터 흐름

```text
사용자가 Deployment 선택
  -> API가 Deployment 소유권 확인
  -> Deployment.architectureId로 불변 ArchitectureJson 조회
  -> Web이 모든 Resource와 edge를 읽기 전용으로 표시
  -> 사용자가 관측 시작
  -> API가 AWS Connection과 실제 트래픽 topology 재검증
  -> valid manifest 생성 또는 기존 valid manifest 재사용
  -> Provider Adapter가 지원 Resource의 실시간 상태 수집
  -> Web이 Resource별 상태를 Architecture 위에 overlay
```

현재 Board를 Deployment 이후에 수정해도 과거 Deployment 관측 화면은 바뀌지 않는다. 관측 화면은 `project_drafts`를 읽지 않는다.

## 5. API와 shared type 계약

### 5.1 배포 Architecture 조회

새 인증 API를 추가한다.

```text
GET /api/deployments/:deploymentId/live-observation-architecture
```

응답:

```ts
type DeploymentLiveObservationArchitectureResponse = {
  deploymentId: string;
  architectureId: string;
  terraformArtifactSha256: string;
  architecture: ArchitectureJson;
};
```

규칙:

- Deployment와 Project 소유권을 확인한다.
- `Deployment.architectureId`가 가리키는 `architectures` row만 반환한다.
- 현재 `project_drafts.diagramJson`으로 fallback하지 않는다.
- Architecture가 없거나 소유권이 없으면 일반 404를 반환한다.
- Terraform config나 AWS 상태를 이 API에서 변경하지 않는다.

### 5.2 Live Observation manifest

DB의 `schemaVersion: 2`와 JSONB 저장 구조는 유지하고, AWS adapter에 version 4를 추가한다. SQL schema 변경과 DB migration은 만들지 않는다.

```ts
type DeploymentLiveObservationAwsAdapterV4 = {
  kind: "aws-live-observation";
  version: 4;
  payload: {
    // 기존 CloudFront, S3, ALB, Target Group evidence 유지
    capacityTarget: {
      kind: "ecs_fargate";
      clusterName: string;
      serviceName: string;
      scaling:
        | { mode: "fixed" }
        | {
            mode: "service_auto_scaling";
            minCapacity: number;
            maxCapacity: number;
            metric: string | null;
            targetValue: number | null;
          };
    };
  };
};
```

기존 adapter version 1~3 manifest는 계속 읽을 수 있어야 한다. version 4는 다음 규칙을 적용한다.

- `ecs_cluster_name`, `ecs_service_name`은 ECS Fargate 관측 필수 좌표다.
- `max_capacity`, `scale_out_threshold`는 필수가 아니다.
- 배포 Architecture에 `aws_appautoscaling_target`이 없으면 `scaling.mode="fixed"`다.
- Service Auto Scaling Resource와 연결이 검증된 경우에만 `service_auto_scaling` evidence를 저장한다.
- Live Observation의 요청 압력 표시 기준은 SketchCatch 세션 표시 기준이다. Terraform scale-out threshold로 주장하지 않는다.

### 5.3 Provider snapshot

현재 nullable인 `LiveObservationProviderSnapshot.capacity.max`를 그대로 사용한다.

고정 Fargate:

```ts
capacity: {
  desired: number;
  running: number;
  healthy: number;
  max: null;
}
```

Service Auto Scaling Fargate:

```ts
capacity: {
  desired: number;
  running: number;
  healthy: number;
  max: number;
}
```

고정 Fargate에서도 `DescribeServices`와 `DescribeTargetHealth`를 호출한다. Auto Scaling 정보가 없다는 이유로 `unavailable` snapshot을 만들지 않는다.

## 6. Resource 관측 상태

Web은 각 Resource를 다음 상태 중 하나로 표시한다.

```ts
type DeploymentResourceObservationState =
  | "observed"
  | "delayed"
  | "unavailable"
  | "not_supported";
```

- `observed`: AWS에서 최신 상태를 확인했다.
- `delayed`: CloudWatch 지표가 지연되었지만 마지막 관측값이 있다.
- `unavailable`: 지원 Resource지만 현재 AWS 조회에 실패했다.
- `not_supported`: 현재 Provider Adapter가 실시간 데이터를 제공하지 않는다.

`not_supported`는 오류가 아니다. Resource와 edge는 그대로 표시하고 중립 색상과 `관측 데이터 없음` 라벨을 사용한다.

첫 구현에서 실시간 overlay를 제공할 AWS Resource:

- CloudFront distribution
- S3 frontend bucket
- ALB
- Target Group
- ECS cluster/service
- CloudWatch Log Group

VPC, Subnet, Route Table, IAM Role처럼 세션 중 상태 변화가 없는 Resource는 Architecture에 표시하되 `not_supported` 또는 정적 배포 evidence로 표시한다.

## 7. UI/UX

관측 모달은 기존 Workspace 디자인 토큰, 폰트, 간격, 버튼 규칙을 유지한다.

1. Deployment를 선택하면 세션 시작 전에도 배포 Architecture 전체를 먼저 보여준다.
2. Architecture는 읽기 전용이며 Resource 이동·수정·삭제 동작을 제공하지 않는다.
3. 관측 전에는 모든 Resource를 `배포 구성` 상태로 표시한다.
4. 관측 시작 후 지원 Resource에 상태 색상과 지표 badge를 덧씌운다.
5. 지원하지 않는 Resource는 회색으로 지우지 않고 원래 icon과 label을 유지한다.
6. 요청 particle은 실제 검증된 CloudFront -> ALB -> ECS 경로에만 표시한다.
7. Auto Scaling이 없는 ECS에는 `고정 용량`과 `desired/running/healthy`를 표시한다.
8. Auto Scaling이 있는 ECS에만 `min/max`와 scaling policy 정보를 표시한다.
9. Architecture 전체 표시 실패와 AWS 실시간 상태 수집 실패를 서로 다른 오류로 안내한다.

현재 focused-path 전용 `LiveObservationDiagramMap`은 그대로 확대하지 않는다. 전체 Architecture 렌더링 책임과 Resource observation overlay 책임을 분리한 읽기 전용 컴포넌트로 재구성한다.

## 8. 오류 처리와 manifest 재검증

### 8.1 관측을 차단하는 조건

- Deployment가 `SUCCESS`, `PARTIALLY_FAILED`, `PARTIALLY_CANCELED` 중 하나가 아니다.
- Deployment의 AWS Connection을 검증할 수 없다.
- 승인된 AWS account/region과 현재 connection identity가 다르다.
- 안전한 HTTPS Output URL 또는 검증된 traffic endpoint가 없다.
- 실제 CloudFront/S3/ALB/Target Group/ECS identity가 승인된 Terraform Output과 다르다.

### 8.2 관측을 차단하지 않는 조건

- Service Auto Scaling이 없다.
- `max_capacity` 또는 `scale_out_threshold` Output이 없다.
- 일부 Resource에 실시간 Provider Adapter가 없다.
- CloudWatch 지표가 아직 집계되지 않았다.
- 로그가 0건이다.

### 8.3 invalid manifest 재검증

현재처럼 `manifest_invalid` row가 있다는 이유만으로 즉시 409를 반환하지 않는다.

```text
existing valid
  -> provenance와 현재 public base URL 일치 시 재사용

existing manifest_invalid
  -> 명시적 관측 시작 요청에서 topology와 materialization 재실행
  -> 성공하면 같은 deployment_id row를 valid로 upsert
  -> 실패하면 invalid 상태 유지, updated_at과 안전한 진단 code 갱신
```

내부 로그에는 구체적인 실패 단계 code를 남기되 role ARN, External ID, secret, credential은 기록하지 않는다. 사용자 메시지는 `AUTO_SCALING_NOT_CONFIGURED`를 오류로 사용하지 않는다.

## 9. 구현 경계

### packages/types

- `DeploymentLiveObservationAwsAdapterV4`
- `DeploymentLiveObservationArchitectureResponse`
- `DeploymentResourceObservationState`
- 기존 adapter version 1~3 하위 호환

### apps/api

- Deployment 소유권 기반 불변 Architecture 조회 route/service
- fixed/service-auto-scaling ECS manifest materialization 분리
- invalid manifest 재검증과 valid upsert
- ECS fixed capacity snapshot에서 `max: null` 반환
- CloudFront/S3/ALB/ECS topology 안전 검증 유지

### apps/web

- 선택 Deployment의 불변 Architecture 조회
- 전체 Resource와 edge를 표시하는 read-only observation map
- Resource observation state overlay
- 고정 Fargate와 Service Auto Scaling Fargate의 용량 문구 분리
- 기존 QR, Output URL, 세션 시작/종료 UX 유지

### Terraform

- 현재 고정 Fargate Architecture에는 변경 없음
- 관측을 통과시키기 위한 가짜 `max_capacity`, `scale_out_threshold` Output 추가 금지
- 사용자가 Service Auto Scaling을 Architecture에 추가한 경우에만 해당 Resource와 Output 생성

### DB

- migration 없음
- 기존 `deployment_live_observation_manifests` JSONB row를 adapter version 4로 upsert

## 10. 필수 검증

불필요한 전체 테스트를 반복하지 않고 변경 경계에 해당하는 검증만 우선 실행한다.

1. 고정 Fargate + CloudFront topology가 `max_capacity`, `scale_out_threshold` 없이 valid manifest를 만든다.
2. Service Auto Scaling Fargate는 실제 min/max/policy evidence를 manifest에 저장한다.
3. 고정 Fargate snapshot은 `desired/running/healthy`를 반환하고 `max=null`이다.
4. 기존 version 1~3 manifest를 계속 파싱한다.
5. 기존 `manifest_invalid` row가 재검증 성공 후 valid로 교체된다.
6. 재검증 실패는 안전한 409를 반환하고 구체적인 내부 단계 code를 남긴다.
7. 다른 사용자의 Deployment Architecture를 조회할 수 없다.
8. 선택한 Deployment의 Architecture를 사용하고 현재 project draft 변경의 영향을 받지 않는다.
9. Auto Scaling이 없는 Architecture의 모든 Resource와 edge가 관측 모달에 표시된다.
10. 지원하지 않는 Resource가 있어도 `관측 시작`을 차단하지 않는다.
11. 실제 CloudFront/S3/ALB/ECS identity 불일치는 계속 차단한다.
12. 현재 Deployment `7cce9b13-8537-462f-bbd1-013692c19c09`에서 세션 생성, snapshot 수신, 종료까지 확인한다.

## 11. 완료 기준

- 사용자가 선택한 Deployment의 전체 Architecture가 세션 시작 전부터 표시된다.
- 현재 고정 Fargate Deployment가 Terraform이나 AWS Resource 변경 없이 관측 세션을 시작한다.
- 고정 Fargate의 실제 desired/running/healthy 상태가 표시된다.
- Auto Scaling이 없는 상태를 오류나 누락으로 표시하지 않는다.
- Service Auto Scaling이 있는 경우에만 min/max/policy가 추가 표시된다.
- 지원하지 않는 Resource도 Architecture에서 사라지지 않는다.
- invalid manifest가 정상 조건 회복 후 영구적으로 409를 반환하지 않는다.
- 관측을 위해 Practice Architecture나 Terraform 코드를 자동 변경하지 않는다.
