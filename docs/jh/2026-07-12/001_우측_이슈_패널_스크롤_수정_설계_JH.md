# 우측 코드 패널 이슈 스크롤 수정 설계

## 배경

우측 코드 패널의 Issues 영역은 Architecture diagnostics와 Terraform diagnostics를 위아래로 렌더링한다. 현재 상위 컨테이너는 첫 행을 `auto`, 두 번째 행을 남은 공간으로 계산하면서 전체 초과 콘텐츠를 `overflow: hidden`으로 자른다. Architecture diagnostics의 내용이 패널 높이를 대부분 차지하면 Terraform diagnostics가 아래에 존재해도 사용자가 스크롤로 접근할 수 없다.

## 목표

- 표시된 전체 이슈 수만큼 Architecture와 Terraform 진단을 모두 확인할 수 있다.
- 더 급한 Terraform 검증 문제를 위에, Architecture 설계 문제를 아래에 표시한다.
- Issues 영역의 한 세로 스크롤로 두 진단 그룹을 순서대로 탐색한다.
- 코드와 Issues 사이의 기존 리사이즈 기능, 각 진단 카드의 버튼 동작, 진단 데이터 구조는 변경하지 않는다.

## 설계

`WorkspaceIssuesPanel`을 Issues 영역의 유일한 세로 스크롤 소유자로 둔다. Terraform과 Architecture 행은 각각 콘텐츠 높이에 맞게 배치하고, 상위 컨테이너가 두 행의 합계를 스크롤한다. Terraform 래퍼와 각 진단 섹션은 자체 세로 스크롤을 소유하지 않는다.

수정 범위는 다음과 같다.

- `WorkspaceIssuesPanel.module.css`
  - 두 행을 `max-content`로 배치해 패널 높이가 작아도 진단 카드와 다음 섹션이 겹치지 않게 한다.
  - 최상위 Issues 컨테이너에 `overflow-y: auto`와 안정적인 스크롤바 여백을 적용한다.
  - Terraform 래퍼의 `overflow: hidden`을 제거한다.
- `WorkspaceIssuesPanel.tsx`
  - Terraform 검증 문제를 Architecture 설계 문제보다 먼저 렌더링한다.
- `TerraformIssuesPanel.module.css`
  - 하위 패널의 고정 높이와 `overflow`를 제거해 중첩 스크롤을 방지한다.
- `workspace-right-panel-layout.test.ts`
  - 통합 Issues 컨테이너가 스크롤을 소유하는지 확인한다.
  - 하위 진단 패널이 자체 스크롤을 만들지 않는지 확인한다.
  - Terraform과 Architecture의 렌더링 순서를 확인한다.

컴포넌트 구조, 진단 상태, API 계약은 바꾸지 않는다.

## 대안과 판단

1. Architecture와 Terraform 영역에 각각 독립 스크롤을 둔다.
   - 두 그룹을 항상 화면에 남길 수 있지만 좁은 패널에 중첩 스크롤이 생겨 조작성이 나빠진다.
2. Architecture 영역을 접을 수 있게 만든다.
   - Terraform 접근성은 좋아지지만 별도의 상태와 인터랙션을 추가하며, 모든 정보를 스크롤로 확인하려는 현재 요구보다 범위가 크다.
3. 전체 Issues 영역에 통합 스크롤을 둔다.
   - 기존 정보 순서와 컴포넌트 구조를 유지하면서 잘림만 제거하므로 이 방식을 선택한다.

## 오류 및 경계 조건

- Architecture 진단이 없거나 Terraform 진단이 없어도 빈 상태 문구까지 정상적으로 표시한다.
- 한 그룹의 진단이 패널보다 길어도 다음 그룹까지 계속 스크롤할 수 있어야 한다.
- 코드/Issues 분할 비율의 최솟값과 최댓값에서도 스크롤 가능한 높이를 유지한다.
- 가로 넘침과 코드 패널의 가로 스크롤 동작은 변경하지 않는다.

## 검증

1. 현재 잘림 구조를 잡아내는 레이아웃 회귀 테스트가 수정 전 실패하는지 확인한다.
2. CSS를 최소 수정한 뒤 같은 테스트가 통과하는지 확인한다.
3. 관련 Workspace 테스트, lint, typecheck, build, harness 검사를 실행한다.
4. 가능하면 실제 또는 브라우저 픽스처에서 Architecture 4개와 Terraform 4개가 하나의 세로 스크롤로 모두 도달 가능한지 확인한다.
