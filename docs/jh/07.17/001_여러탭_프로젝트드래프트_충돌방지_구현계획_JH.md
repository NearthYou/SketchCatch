# 여러 탭 프로젝트 Draft 충돌 방지 구현 계획

## 1. 목적

이 작업은 같은 프로젝트를 여러 브라우저 탭에서 편집할 때 오래된 탭의 저장이 서버의 최신 `ProjectDraft`를 덮어쓰지 못하게 한다. 사용자가 새로고침하면 서버 최신 상태를 기준으로 Architecture Board를 복구하고, 오래된 revision으로 저장하면 변경 내용을 유지한 채 충돌 안내와 최신 상태 재로드 선택지를 제공한다.

이 작업은 여러 사용자의 변경을 자동 병합하거나 실시간 공동 편집을 제공하지 않는다. 서버의 최신 변경을 조용히 덮어쓰는 동작을 막는 낙관적 동시성 제어가 범위다.

## 2. 현재 문제

1. `ProjectDraft` 응답에는 `revision`이 있지만 `SaveProjectDraftRequest`에는 저장 기준 revision이 없다.
2. `PUT /api/projects/:id/draft`는 현재 row를 읽어 revision을 증가시킨 뒤 `projectId` 충돌 시 조건 없이 upsert한다.
3. 각 탭의 `ProjectWorkspaceDraftManager`는 자기 탭에서 저장 중 발생한 편집만 감지하며 다른 탭의 서버 저장은 감지하지 못한다.
4. Workspace 재진입 시 서버 draft와 IndexedDB draft를 함께 읽지만 로컬 저장 시각이 더 늦으면 서버 revision과 관계없이 로컬 draft를 선택한다.
5. 저장 충돌 전용 UI와 서버 최신 상태 재로드 동작이 없다.

## 3. 확정 인터페이스

### 3.1 저장 요청

```ts
type SaveProjectDraftRequest = {
  diagramJson: DiagramJson;
  terraformFiles?: TerraformSyncFileInput[];
  expectedRevision: number | null;
};
```

- 기존 서버 draft를 편집한 저장은 사용자가 읽은 `revision`을 `expectedRevision`으로 보낸다.
- 서버 draft가 없는 새 프로젝트의 최초 저장만 `expectedRevision: null`을 보낸다.
- 이 필드는 선택값으로 두지 않는다. 누락을 허용하면 오래된 호출 경로가 계속 무조건 덮어쓸 수 있다.

### 3.2 충돌 응답

서버의 현재 revision과 `expectedRevision`이 다르면 다음 의미의 `409 Conflict`를 반환한다.

```ts
type ProjectDraftConflictResponse = ApiErrorResponse & {
  error: "conflict";
  currentRevision: number;
  currentServerSavedAt: IsoDateTimeString;
};
```

클라이언트는 이 응답을 저장 실패와 구분해 충돌 상태로 표시한다. 최신 DiagramJson은 충돌 응답에 중복 포함하지 않고 사용자가 재로드를 선택한 시점에 `GET /api/projects/:id/draft`로 다시 읽는다.

### 3.3 로컬 복구 기준

IndexedDB의 `LocalProjectDraft`에는 로컬 편집 횟수와 별개로 편집이 시작된 서버 revision을 보관한다.

```ts
type LocalProjectDraft = {
  // 기존 필드
  baseServerRevision: number | null;
};
```

- 로컬 편집을 저장할 때 `baseServerRevision`은 증가시키지 않는다.
- 서버 저장 성공 시 서버가 반환한 revision으로 교체한다.
- 기존 IndexedDB row에 필드가 없으면 서버 최신 상태를 우선하고 자동으로 오래된 로컬 draft를 서버에 저장하지 않는다.
- 기본 IndexedDB recovery key는 탭 인스턴스마다 분리하고 탭의 `sessionStorage`에 유지해, 다른 탭의 로컬 편집과 격리하면서 같은 탭 새로고침에서는 recovery draft를 다시 찾는다. 탭 복제 시 복사된 ID는 `Web Locks` 배타 점유 실패로 감지해 새 ID로 교체한다.
- 기존 공유 key인 `project:{projectId}` recovery draft는 처음 읽을 때 탭 전용 key로 복사해 기존 로컬 복구본을 잃지 않는다.

## 4. 서버 저장 규칙

### 4.1 기존 draft 갱신

`project_id = :projectId AND revision = :expectedRevision` 조건을 포함한 단일 `UPDATE ... RETURNING`으로 저장한다. 갱신 row가 없으면 `409 Conflict`를 반환한다. revision 조회와 갱신을 분리한 비교만으로 처리하지 않는다.

### 4.2 최초 draft 생성

`expectedRevision`이 `null`일 때만 `INSERT ... ON CONFLICT DO NOTHING ... RETURNING`을 수행한다. 동시에 다른 탭이 먼저 생성해 반환 row가 없으면 `409 Conflict`를 반환한다.

### 4.3 동일 내용 저장

요청 revision이 현재 revision과 같고 DiagramJson과 Terraform files가 동일하면 revision을 증가시키지 않고 현재 draft를 반환한다. 요청 revision이 오래됐으면 내용 비교 결과와 관계없이 충돌로 처리해 호출자가 서버 기준을 다시 확정하게 한다.

### 4.4 DB 변경 여부

`project_drafts.revision`과 unique `project_id`가 이미 존재하므로 PostgreSQL schema와 Drizzle migration은 변경하지 않는다.

## 5. 새로고침과 충돌 UX

1. Project draft GET 요청과 응답에 `no-store` 정책을 적용한다.
2. Workspace 최초 로드는 서버 draft를 최신 기준으로 사용한다.
3. 서버 조회가 성공했고 서버 draft가 없는 경우에만 IndexedDB draft를 복구 fallback으로 사용한다. 서버 조회 자체가 실패하면 오래된 로컬 상태를 최신 상태로 오인하지 않도록 Workspace 로드를 실패 처리한다.
4. `409 Conflict`가 발생하면 현재 로컬 Architecture Board와 IndexedDB recovery draft를 유지한다.
5. 자동 checkpoint 저장을 반복하지 않고 서버 상태를 `conflict`로 고정한다.
6. 사용자에게 다음 안내를 표시한다.
   - 안내: `다른 탭에서 이 프로젝트가 변경되었습니다.`
   - 설명: `현재 탭의 변경은 아직 서버에 저장되지 않았습니다. 최신 상태를 불러오면 현재 화면이 서버 상태로 교체됩니다.`
   - 주요 동작: `최신 상태 불러오기`
   - 보조 동작: `현재 편집 유지`
7. `최신 상태 불러오기`를 선택하면 서버 draft를 다시 요청해 Board와 Terraform files를 함께 교체하고 revision, IndexedDB, 저장 상태를 동기화한다.
8. `현재 편집 유지`는 모달만 닫고 로컬 변경을 유지한다. 다시 저장하면 최신 상태를 불러오기 전까지 같은 충돌을 반환한다.

## 6. TDD 구현 순서

### Slice 1: API stale save 차단

- 두 요청이 같은 revision을 기준으로 저장할 때 첫 요청만 성공한다.
- 두 번째 요청은 `409`를 받고 서버 draft는 첫 요청 내용과 revision을 유지한다.
- `expectedRevision` 누락과 잘못된 값은 `400`으로 거부한다.

### Slice 2: 최초 생성 경쟁과 동일 내용 저장

- 두 탭이 `expectedRevision: null`로 최초 저장하면 한 요청만 생성한다.
- 같은 revision과 같은 내용 저장은 revision을 증가시키지 않는다.
- 오래된 revision의 동일 내용 요청도 `409`로 처리한다.

### Slice 3: Web 저장 계약

- API client가 `expectedRevision`을 요청 body에 포함한다.
- Project draft repository가 로컬 편집 기준 `baseServerRevision`을 유지한다.
- 서버 저장 성공 시 반환 revision으로 기준을 갱신한다.

### Slice 4: 서버 우선 새로고침

- 서버와 로컬 draft가 모두 있으면 서버 draft를 선택한다.
- 서버 조회가 성공했지만 서버 draft가 없을 때만 로컬 draft를 선택한다.
- 서버 조회가 실패하면 로컬 draft를 조용히 표시하지 않고 Workspace 로드를 실패 처리한다.
- Project draft GET은 브라우저와 API 응답 모두 `no-store`다.

### Slice 5: 충돌 UI와 재로드

- `409`를 일반 저장 실패가 아닌 충돌 상태로 표시한다.
- 충돌 뒤 자동 checkpoint 재시도가 중단된다.
- `최신 상태 불러오기`가 Board, Terraform files, revision, IndexedDB를 서버 값으로 교체한다.
- `현재 편집 유지`가 로컬 변경을 보존한다.

## 8. 완료 조건

1. 오래된 탭의 저장이 서버 최신 draft를 변경하지 않는다.
2. 새로고침한 Workspace가 서버 최신 draft를 표시한다.
3. 충돌한 탭의 로컬 편집은 사용자가 재로드를 선택하기 전까지 보존된다.
4. 사용자가 최신 상태 재로드를 선택하면 DiagramJson과 Terraform files가 같은 revision으로 교체된다.
5. DB migration 없이 기존 draft row와 신규 row가 모두 동작한다.
6. 집중 회귀 테스트와 `pnpm test`, `pnpm lint`, `pnpm typecheck`, `pnpm build`, `pnpm harness:check`가 통과한다. 기존 실패가 있으면 이번 변경과의 관련성을 분리해 기록한다.

## 9. 롤백 기준

- 저장 요청의 `expectedRevision` 전달이 누락되는 내부 호출 경로가 남아 정상 저장이 차단되면 배포하지 않는다.
- 충돌 재로드가 DiagramJson만 교체하고 Terraform files 또는 revision을 이전 값으로 남기면 배포하지 않는다.
- 조건부 UPDATE가 동시 요청에서 둘 이상의 성공을 허용하면 배포하지 않는다.
- 기존 IndexedDB draft 때문에 새로고침 후 서버 최신 상태가 보이지 않으면 배포하지 않는다.
