# 브랜치 리뷰와 커밋 가이드

## 문서 목적

이 문서는 이번 브랜치의 변경을 리뷰할 때 어떤 순서로 보면 좋은지 정리한다.

브랜치 이름:

```txt
Feat/jh/126-리소스-추가-시-terraform-preview-기본-파라미터-skeleton-자동-생성
```

이 브랜치는 처음에는 “리소스 추가 시 Terraform Preview 기본 skeleton 생성”에서 출발했지만, 작업 중 사용자 피드백을 반영하면서 아래 범위까지 확장되었다.

- 기본 parameter skeleton 정책 재검토
- InfrastructureGraph 기반 Terraform Preview 안정화
- Terraform editor 역동기화
- proposal 자동 반영 정책
- 리소스 아이콘/카탈로그/크기 정렬
- Terraform editor syntax color와 diagnostics UX
- leave dialog와 Issues 탭 접근성 개선

## 이번 브랜치에서 정책이 바뀐 부분

### proposal 확인 UI 제거

중간 단계에서는 Terraform editor가 create/delete/rename proposal을 별도 UI로 보여주고 사용자가 선택하게 했다.

최종 정책은 바뀌었다.

- 사용자가 Terraform editor에서 저장을 누르는 행위 자체를 명시 승인으로 본다.
- 저장 action 안에서는 proposal을 자동 반영한다.
- 저장 전에는 diagram을 자동 변경하지 않는다.

이유:

- 별도 proposal 확인 UI가 편집 흐름을 끊었다.
- 사용자는 이미 editor에서 코드를 직접 수정했고 저장을 눌렀다.
- 따라서 저장은 User-Accepted Change로 볼 수 있다.

리뷰 포인트:

- 자동 반영은 저장/배포 준비 action 안에서만 일어나야 한다.
- validation error가 있으면 proposal 반영이 일어나면 안 된다.

### 수동 리소스 아이콘 parameter 자동 채움 제거

처음에는 resource icon 생성 시 기본 Terraform parameter skeleton을 채우는 방향이었다.

최종 정책은 바뀌었다.

- 수동 resource icon 생성 시 `parameters.values`는 `{}`로 시작한다.
- Terraform identity metadata만 만든다.
- AI draft config나 Terraform editor sync처럼 명시 입력으로 들어온 값은 보존한다.

이유:

- 내부 parameter 정책이 완성되지 않았다.
- 불완전한 기본값은 유효한 IaC라는 착각을 줄 수 있다.

리뷰 포인트:

- manual drop 경로와 duplicate/paste 경로의 정책이 충돌하지 않는가?
- AI draft가 명시한 config까지 지워버리지는 않는가?

### Terraform editor 오류 UX 개선

최종 정책:

- validation error는 코드 위 빨간 물결 밑줄로 보여준다.
- leave dialog가 오류 확인을 막으면 안 된다.
- diagnostics가 있으면 Issues 탭은 dirty 상태에서도 바로 열 수 있어야 한다.
- 곧 닫힐 modal의 순간 status message는 보여주지 않는다.

리뷰 포인트:

- 오류 확인 UI는 사용자 행동을 막지 않는가?
- 저장 실패와 저장 중 상태가 서로 다른 시각 피드백을 갖는가?

## 이번 브랜치에서 실행하지 않은 것

실행하지 않은 것:

- 실제 `terraform apply`
- 실제 `terraform destroy`
- 실제 cloud resource 생성/삭제
- Git/CI/CD handoff 실행
- 실제 AWS 계정 mutation

이유:

- 이번 변경은 Preview, editor sync, UX 안정화 범위다.
- 실제 deployment path는 별도 safety gate, approval, logging, cleanup 정책 뒤에서 다뤄야 한다.

## 남은 리스크

### 브라우저 수동 smoke

자동 테스트와 build는 통과했지만, 일부 흐름은 실제 브라우저에서 한 번 더 보는 것이 좋다.

추천 smoke:

- Terraform syntax error를 만든 뒤 빨간 물결 밑줄이 보이는지
- `저장하고 나가기` 실패 시 modal이 닫히고 Terraform 탭이 보이는지
- diagnostics가 있는 상태에서 Issues 탭을 누르면 modal 없이 열리는지
- 같은 resource icon을 여러 번 추가하면 Terraform name suffix가 붙는지
- 마지막 resource icon 삭제 후 Terraform Preview가 빈 코드로 바뀌는지
- CloudFront 같은 catalog 보강 리소스가 `AWS` fallback이 아니라 실제 icon으로 보이는지

### 지원 리소스 범위

이번 브랜치는 AWS-first MVP 범위의 일부 리소스를 안정화했다.

전체 AWS catalog 지원은 아니다.

리뷰 중 unsupported resource가 보이면 “버그”인지 “아직 지원하지 않는 범위”인지 먼저 구분해야 한다.

### source metadata

`sourceFileName`은 diagnostic/proposal 위치 정보다.

절대 경로나 민감한 로컬 path가 UI/API 응답에 섞이지 않게 유지해야 한다.

## 최종 리뷰 체크리스트

- type contract와 API Zod schema가 맞는가?
- frontend가 shared type과 다른 enum/string을 임의로 만들지 않는가?
- Terraform 실행이나 AWS SDK 호출이 frontend에 들어오지 않았는가?
- 저장 전 자동 diagram mutation 경로가 없는가?
- validation error가 있는 code가 diagram을 바꾸지 않는가?
- icon size, catalog icon, fallback icon 정책이 서로 충돌하지 않는가?
- docs/data-models와 실제 코드 계약이 어긋나지 않는가?
- 이번 문서 3개가 canonical 문서를 대체한다고 오해될 표현이 없는가?
