LESSON 0003 · 약 4분

누가 AWS SSO 로그인을 해야 하나?

모든 SketchCatch 사용자가 AWS에 로그인할 필요는 없다. AWS 연결을 만들거나 갱신할 권한이 있는 담당자만 로그인하면 된다.

오늘의 한 문장
SSO는 모든 사용자의 필수 절차가 아니라, AWS 연결을 승인하는 담당자의 절차다.

사람을 세 종류로 나누기

사람SSO가 필요한 때필요한 권한
AWS Identity Center 관리자처음 사용자와 Permission Set을 배정할 때SSO 관리 권한
AWS 연결 담당자SketchCatch 연결 또는 Stack 갱신을 승인할 때CloudFormation·필요한 IAM 변경 권한
일반 SketchCatch 사용자공유 연결을 사용한다면 필요 없음SketchCatch 내부 사용 권한

AWS 연결 담당자가 반드시 최고 관리자인 것은 아니다. 관리자가 필요한 Permission Set만 배정한 별도 담당자도 Stack을 승인할 수 있다.

연결이 끝난 뒤

Reverse Engineering을 실행할 때마다 사용자가 SSO로 로그인하지 않는다. SketchCatch 서버가 이미 만들어진 전용 Role을 맡아 AWS를 조회한다.

일반 사용자 요청 → SketchCatch 권한 확인 → 전용 Role 사용 → AWS Resource 조회

다른 사용자는 AWS 비밀번호, SSO 세션, 임시 자격 증명을 공유받지 않는다. SketchCatch 안에서 허용된 기능만 사용한다.

그런데 현재 SketchCatch는?

현재 데이터 구조에서는 AWS 연결이 팀이 아니라 SketchCatch 사용자 개인에게 연결된다. 따라서 지금 구현 그대로라면 각 사용자가 자기 AWS 연결을 따로 만들어야 한다.

권장하는 제품 구조

팀에서 같은 AWS 계정을 사용한다면 AWS 연결을 Workspace나 팀이 소유하게 만드는 편이 낫다.

  1. 권한 있는 담당자 한 명이 SSO로 AWS 연결을 승인한다.
  2. 연결은 Workspace 또는 팀에 저장한다.
  3. 일반 사용자는 AWS 로그인 없이 Reverse Engineering 결과를 본다.
  4. 배포처럼 위험한 작업은 SketchCatch 안에서 별도 승인 권한을 확인한다.
`모두 AWS에 로그인`과 `모두 AWS를 마음대로 사용`은 같은 말이 아니다. SSO 로그인 여부와 SketchCatch 내부의 조회·배포 승인 권한은 별도로 관리해야 한다.

30초 확인 문제

팀의 AWS 연결을 담당자 한 명이 완료했다. 일반 팀원이 Reverse Engineering 결과를 보기 위해 필요한 것은?

A. SketchCatch 내부 사용 권한
B. AWS 관리자 로그인 권한
C. S3 Template 수정 권한
정답과 이유 보기

A. AWS 조회는 SketchCatch 전용 Role이 수행한다. 일반 팀원은 SketchCatch 안에서 해당 Workspace와 기능을 사용할 권한만 있으면 된다.

공식 자료

AWS 계정 접근 설정 공식 설명은 사용자가 배정받은 계정과 Role을 선택해 Console에 들어가는 과정을 설명한다.
