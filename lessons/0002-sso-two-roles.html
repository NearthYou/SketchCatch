LESSON 0002 · 약 6분

SSO 사용자와 SketchCatch는 서로 다른 Role을 쓴다

SSO는 사람이 AWS Console에 들어가는 입구다. SketchCatch가 나중에 AWS를 읽고 배포할 때는 별도의 Role을 사용한다.

오늘의 한 문장
사용자는 SSO Role로 Stack 생성을 승인하고, Stack은 SketchCatch가 사용할 별도의 Role을 만든다.

SSO는 무엇인가?

SSO는 한 번 로그인해서 허용된 AWS 계정과 업무 역할을 고르는 입구다. AWS에서는 IAM Identity Center가 이 일을 맡는다. 관리자가 사용자에게 Permission Set을 배정하면, Identity Center가 그 권한에 맞는 사람용 IAM Role을 준비한다.

사람 경로

사용자 → SSO Role

  1. 사용자가 SSO로 로그인한다.
  2. AWS 계정과 Permission Set을 고른다.
  3. 사람용 Role의 임시 권한으로 Console에 들어간다.
  4. CloudFormation Stack 생성이나 갱신을 승인한다.
서비스 경로

SketchCatch → 전용 Role

  1. SketchCatch가 AWS 작업을 시작한다.
  2. External ID를 제시하며 전용 Role을 맡는다.
  3. 짧은 시간만 유효한 임시 자격 증명을 받는다.
  4. 허용된 Resource 조회나 배포 작업을 실행한다.
두 경로를 연결하는 다리: CloudFormation Stack
사용자가 SSO Role로 Stack을 승인하면, Stack이 SketchCatch 전용 Role과 Policy를 만든다.

S3는 어디에 끼어드나?

SketchCatch가 만든 YAML Template을 잠시 전달하는 파일 보관함이다. 사람용 SSO Role도, SketchCatch 전용 Role도 S3 안에 저장되지 않는다.

SketchCatch YAML 생성 → S3에 보관 → SSO 사용자가 Stack 승인 → 전용 Role 생성

사용자의 SSO Permission Set에 CloudFormation 또는 IAM 변경 권한이 없으면 Stack 생성·갱신이 실패할 수 있다. 이것은 SketchCatch 전용 Role의 읽기 권한 부족과는 다른 문제다.

Reverse Engineering에서는?

평소 Resource를 가져올 때 사용자의 브라우저 SSO 세션을 빌려 쓰지 않는다. SketchCatch 서버가 전용 Role을 맡아 조회한다. 다만 전용 Role의 권한을 갱신해야 할 때는 사용자가 다시 SSO로 AWS Console에 들어가 Stack 변경을 승인해야 한다.

30초 확인 문제

Stack과 SketchCatch 전용 Role을 이미 만들었다. 그 뒤 사용자의 SSO 로그인 시간이 끝났다. 일반적으로 Reverse Engineering은 어떻게 될까?

A. 전용 Role이 있으면 계속 가능
B. SSO 세션 없으면 즉시 중단
C. S3 파일 없으면 즉시 중단
정답과 이유 보기

A. SSO 세션은 사람이 Console에서 승인할 때 사용한다. 평소 Reverse Engineering은 SketchCatch 전용 Role에서 새 임시 자격 증명을 받아 실행한다.

공식 자료

IAM Identity Center Permission Set 공식 설명은 Permission Set을 배정하면 Identity Center가 계정 안에 대응하는 IAM Role을 만들고 사용자가 그 Role을 맡는다고 설명한다.
